    B4 — Users, groups, permissions, capabilities

    Every time anything happens on a Linux machine, the kernel silently asks one question first: is whoever is asking allowed to do this? That check is the subject of this module. It sounds like a dry topic — file permissions — but it is the whole of Linux security, and it is the exact machinery Kubernetes reaches for when it decides what a container may and may not do. Get it clear here, on one plain machine, and every securityContext field you will ever read becomes obvious rather than mysterious.

    There are three layers, and this module walks up all three. First, identity: who you are is a number, and the kernel only ever compares numbers. Second, permissions: the rwx bits you have half-seen in ls -l, which mean one thing on a file and a subtly different thing on a directory — the source of most real "permission denied" tickets. Third, capabilities: the modern truth that "root" is not one all-or-nothing flag but roughly forty separate powers, which is the single fact that makes it safe to run a process as "root" inside a container at all.

    Identity

    B4.1

    Identity is numbers: uid, gid, and the groups you carry

    Why it matters

    You log in as a name — ubuntu, root, www-data — but the kernel has never once looked at that name when deciding whether to let you do something. It compares numbers. Your process carries a user id and a set of group ids, and every permission check is arithmetic on those numbers. Once you internalise that names are a convenience for humans and numbers are the reality, a whole class of confusion evaporates — including the classic "the uids don't match between the host and the container, so the file I mounted is owned by nobody".

    The concept

    A running process carries a small bundle of identity:

    • a uid (user id) — one number saying who you are;
    • a primary gid (group id) — the group new files you create will belong to;
    • a list of supplementary groups — extra group memberships that also count when permissions are checked.

    The kernel knows only these numbers. The mapping from number to name — 1000 ↔ ubuntu, 0 ↔ root — lives in ordinary files (/etc/passwd, /etc/group, the subject of the next lesson) that tools consult purely so they can show you a name. Return to the hotel from B2: a guest is a process and their room number is the uid. The front desk keeps a book mapping room numbers to guests' names, but the plumbing and the door locks only ever deal in room numbers.

    There is one more subtlety you already half-met in B3 and will need throughout this module: a process actually carries a real uid (who started it) and an effective uid (who it is being checked as right now). They are usually identical. The gap between them is the entire setuid mechanism later in this module — it is how passwd, launched by you, can be checked as root while it runs.

    processcarriesuid·primary gid·supplementary groups→ kernel checksnumbers only

    Run id and, for every number it prints, name the thing it maps to using getent. Then answer: if you created a file right now, which single group would own it, and why that one and not the others? (Your primary gid — 1000(ubuntu) — is the group stamped onto new files; the supplementary groups let you access things owned by those groups but are never the owner of what you make.)

    Gotchas

    A newly added group membership does not apply to shells you already have open. Group membership is copied into a process when it starts, so after sudo usermod -aG docker ubuntu you must log out and back in (or start a fresh session) before id shows it — the running shell is carrying the old, smaller set. This trips up almost everyone the first time.

    Names are per-machine, numbers are universal, and that mismatch is a real operational hazard. A file owned by uid 1000 on one host shows as ubuntu there and as whatever local user happens to be 1000 on another host — or as a bare 1000 if no name maps to it. When you mount a directory into a container, it is the numbers that carry across, which is exactly why "owned by 1000" and not "owned by ubuntu" is the honest way to think about it.

    This is the fact under runAsUser and fsGroup in a pod's securityContext. When you write runAsUser: 1000 you are setting the exact number this lesson is about — the process inside the container runs with uid 1000, and a file it writes to a mounted volume is owned by 1000 on the host, name or no name. The infamous "my mounted files are owned by nobody" is precisely the name-lookup gap from the gotcha: the host has no /etc/passwd entry for the uid the container chose, so ls shows the number.

    You will meet this as a deliberate control in W3 (workload security context) and again where non-root containers must be given group access to a volume via fsGroup.

    Docs

    man 1 id for the command, man 2 getuid / man 2 getgid for the real-vs-effective distinction at the syscall level, and man 2 credentials (the process credentials overview) for the full identity bundle a process carries. man 1 getent for querying the account database directly. In the exam docs the equivalent is any page mentioning runAsUser/runAsGroup/fsGroup.

    B4.2

    The account files: passwd, group, shadow

    Why it matters

    The number-to-name mapping from the last lesson has to live somewhere, and it lives in three plain text files you can read with cat. There is no database server, no magic — user accounts on a Linux box are three colon-separated files. Knowing their fields by heart means you can read an account, spot a service user, or explain a login failure without any special tooling, which is exactly the situation you are in when debugging a minimal container.

    The concept

    Three files, each a table with colon-separated fields, one row per entry:

    FileFields (in order)Holds
    /etc/passwdname : x : uid : gid : comment : home : shellthe account itself — world-readable
    /etc/groupname : x : gid : membersgroups and who is in them — world-readable
    /etc/shadowname : hash : lastchange : min : max : warn : …the password hashes — root-readable only

    The one thing everybody asks about: the x in the second field of /etc/passwd. That field used to hold the password hash directly. But /etc/passwd has to stay world-readable — every tool that turns a uid into a name reads it — so keeping hashes there meant anyone could copy them and attack them offline. The hashes were moved out into /etc/shadow, which only root can read, and the x is the leftover placeholder meaning "the real secret is over in shadow". The file is still called passwd for history; it no longer holds a password.

    A hash in /etc/shadow is not the password — it is a one-way scramble of it. When you log in, the system scrambles what you typed the same way and compares; it never stores or recovers the original. On Ubuntu 24.04 the default scheme is yescrypt, which is why the hash field starts with $y$ (older systems used $6$ for SHA-512). The leading $-delimited part names the algorithm so the system knows how to re-scramble your attempt.

    Lab
    1. Read your own account row and name every field. This is the whole account: name, the x placeholder, uid, primary gid, a human comment, home directory, and login shell. 
      getent passwd ubuntu
      ubuntu:x:1000:1000:Ubuntu:/home/ubuntu:/bin/bash
    2. Look at a group row and read its membership list — the fourth field is the supplementary members, the same names you saw in id. 
      getent group sudo
      sudo:x:27:ubuntu
    3. Now the protected file. As an ordinary user you cannot read /etc/shadow at all — that is the whole point of it. With sudo, read one line and see the hash. 
      cat /etc/shadow            # denied — you are not root
sudo getent shadow ubuntu
      cat: /etc/shadow: Permission denied
ubuntu:$y$j9T$F8s...long-hash...:19800:0:99999:7:::
    Verify

    From memory, list the seven fields of a /etc/passwd line in order. Then explain, in one sentence, why moving the hash to /etc/shadow improved security given that /etc/passwd did not change permissions. (Because /etc/passwd must remain world-readable for name lookups, leaving the hash there exposed it to every user; /etc/shadow is readable only by root, so the hashes are no longer casually copyable.)

    Gotchas

    Never edit these files with a plain text editor. A single mangled line in /etc/passwd can lock everyone out, and a race between two editors can corrupt them. Use the tools built for it — useradd/usermod/userdel, passwd, vipw and vigr (which lock the file while you edit) — for the same reason you will meet visudo in B4.4.

    A shell of /usr/sbin/nologin or /bin/false in the last field is not a broken account — it is a deliberate one. It means "this account may own files and run services but nobody may log in as it interactively", which is the subject of the next lesson and the normal state of every service account on the box.

    A minimal container image often has almost nothing in its /etc/passwd — sometimes just root, sometimes a single non-root user, sometimes no entry for the uid the pod runs as. That is why a process can run as uid 1000 with no name, and why some images ship a trick to append the running uid to /etc/passwd at start-up so tools that demand a username don't fail. Understanding that the file is just a lookup table, and that the kernel doesn't need it, is what lets you reason about a "user unknown" error inside a container instead of being mystified by it.

    Kubernetes itself never authenticates humans against these files — its user identities come from certificates and tokens (W7). These files govern the Linux users inside a node and a container; keep the two planes separate in your head.

    Docs

    man 5 passwd, man 5 group and man 5 shadow document each file field by field — the section-5 pages are the file formats, distinct from the section-1 passwd(1) command. man 3 crypt explains the hash formats and the $y$/$6$ prefixes. man 8 vipw for the safe way to edit them.

    B4.3

    System users vs login users, and why services get their own

    Why it matters

    Run getent passwd on a fresh machine and you find dozens of accounts you never created — daemon, www-data, systemd-network, _apt. They are not people. Each one is a service given its own identity on purpose, and the reason is the single most important idea in practical security: blast-radius control. Understanding why every daemon gets its own uid is understanding why runAsNonRoot exists.

    The concept

    Accounts split into two kinds, distinguished by convention, not by any hard kernel rule:

    • Login users — real people. On Debian/Ubuntu their uids start at 1000, they have a home directory, and a real login shell like /bin/bash.
    • System users — identities for services. Low uids (below 1000), usually no useful home, and a shell of /usr/sbin/nologin so nobody can log in as them.

    Why bother giving nginx its own www-data identity instead of just running it as root? Because a service is the thing most likely to be attacked — it faces the network — and its uid is the fence around what a successful attacker inherits. If www-data-owned nginx is compromised, the attacker can touch exactly what uid www-data can reach: the web files, and nothing else. They cannot read other users' data, cannot write system files, cannot touch another service's secrets, because the kernel checks their (borrowed) uid on every single operation. Run the same service as root and a compromise is a compromise of the whole machine. Least privilege — give each thing the smallest identity that lets it do its job — is the entire design principle, and a dedicated low-privilege uid per service is how Linux has always expressed it.

    service compromised→ attacker inheritsexactly that uid's accessvs. as root →the whole machine

    Point at a system user in the output and state the three tells that mark it as one (low uid, nologin shell, no real home). (If nginx is compromised the attacker is confined to what uid www-data can reach — the web content — rather than owning the entire host, which is what a root compromise would be.)

    Gotchas

    The uid-1000 boundary is a convention set in /etc/login.defs (UID_MIN/SYS_UID_MAX), not a law the kernel enforces. The kernel treats uid 500 and uid 1500 identically; the split into "system" and "login" ranges is a policy that tools like useradd follow so the two kinds don't collide. Only uid 0 is special to the kernel — that is root, and that specialness is exactly what capabilities later in this module carve up.

    A system user having nologin does not stop its service from running — services are started by systemd, which switches to the service's uid directly, no login involved. nologin only blocks the interactive path (ssh, su into an interactive shell). So "you can't log in as www-data" and "nginx runs as www-data" are both true at once.

    This is the whole argument behind runAsNonRoot: true and runAsUser in a pod's securityContext, and behind the base-image convention of shipping a dedicated non-root user like nginx (uid 101) or node (uid 1000). A container that runs as root is a container whose compromise hands the attacker root inside that container — and, absent the capability and namespace fences from later in this module, uncomfortably close to root on the node. Choosing a low-privilege uid per workload is the same least-privilege move Linux services have always made, now expressed in a pod spec.

    You will see the platform enforce this in W3 (security context) and, cluster-wide, via Pod Security admission that can reject a pod insisting on running as root.

    Docs

    man 5 login.defs for the uid-range conventions (UID_MIN, SYS_UID_MAX). man 8 useradd under --system for how a service account is created. man 8 nologin for the shell that politely refuses logins. man 5 systemd.exec under User=/DynamicUser= for how a modern service is pinned to its own identity — DynamicUser= in particular manufactures a throwaway uid per run, least privilege taken to its logical end.

    B4.4

    Becoming someone else: su, sudo, and sudoers

    Why it matters

    You rarely want to be root; you want to run one specific command as root and go back to being yourself. sudo is how, and its config file is how an administrator grants exactly that — one user, one command, no more. Learning to write a single tight sudoers rule is the practical face of least privilege, and it is the same shape of thinking as an RBAC Role: name a subject, name what they may do, grant nothing else.

    The concept

    Two tools change which identity a command runs as:

    • su ("substitute user") starts a new shell as another user, after checking their password. Plain su keeps your current environment and directory; su - (with the dash) simulates a full fresh login — it resets the environment, moves to the target's home directory, and loads their shell profile and PATH. The dash is why su - "feels like" logging in as that user and plain su often leaves you with a confusingly half-changed environment.
    • sudo runs one command as another user (root by default), after checking your own password and consulting a policy file. sudo -i gives you a full interactive root login shell (like su - but authorised by policy, not by knowing root's password). The key difference from su: sudo is governed by a rulebook, so it can grant a narrow slice of root instead of all of it.

    That rulebook is /etc/sudoers. Its grammar reads almost as a sentence: who where=(as-whom) what. A line like

    deploy  ALL=(ALL)  /usr/bin/systemctl restart nginx

    means "user deploy, on ALL hosts, may run, as ALL users (i.e. including root), exactly the command systemctl restart nginx — and nothing else." That is a keycard cut for one door. You always edit this file with visudo, never directly: visudo checks the syntax before saving and refuses to write a file that would break sudo and lock every administrator out. Modern practice is to drop small rule files into /etc/sudoers.d/ (edited with visudo -f) rather than editing the main file, so one package's rule can't corrupt another's.

    Lab

    Verify

    State the two things the dash in su - changes (environment/profile, and working directory to the target's home). Then explain why sudo asks for your password while su asks for the target's — and which of the two lets an admin grant root access without ever sharing root's password. (sudo authenticates you against a policy, so an admin adds a sudoers rule and you use your own password; su requires knowing the target account's password, which is why shared-root-password setups are a security smell that sudo exists to kill.)

    Gotchas

    Always, without exception, edit sudoers through visudo. A syntax error saved directly can make sudo refuse to run at all — and if sudo is your only path to root, you have locked yourself out of your own machine with no way back short of single-user mode. visudo's entire reason to exist is to check the file and reject a broken save before it can hurt you. Snapshot the VM before experimenting.

    A trailing wildcard in a rule is a trap. deploy ALL=(ALL) /usr/bin/systemctl restart * looks tighter than full root but is not — the argument freedom can often be bent into running unintended things, and any rule granting an editor, a shell, or a command with a shell-escape (vi, less, find -exec) is effectively a grant of full root. Grant the exact command with its exact arguments, and be suspicious of your own rules.

    The mental model of sudoers — a subject, an action, and a scope, granting the narrowest slice that does the job — is exactly the model of Kubernetes RBAC. A Role lists verbs (get, list, create) on resources; a RoleBinding attaches it to a subject. "user deploy may only restart nginx" and "ServiceAccount ci may only get pods in namespace build" are the same sentence in two dialects. The habit of granting the minimum and distrusting wildcards transfers directly.

    You will write these grants for real in W7.7 — imperative RBAC, where kubectl create role/rolebinding is the cluster-side visudo.

    Docs

    man 8 sudo for the command and its flags (-i, -u, -l), man 5 sudoers for the rule grammar (long, but the EXAMPLES section is the fast way in), and man 8 visudo for safe editing and -f. man 1 su for the substitute-user shell and what - changes. For the cluster analogue, anything under RBAC Role/RoleBinding in the exam docs.

    Permissions

    B4.5

    The permission check: owner, then group, then other — first match wins

    Why it matters

    You have seen -rw-r--r-- in ls -l a hundred times. This lesson is the exact rule the kernel applies to those nine letters, and it contains one genuine surprise that catches nearly everyone: the check is first-match, not a best-of. It is entirely possible to lock the owner of a file out of their own file while a stranger can read it — and once you know why, you will never misread a permission line again.

    The concept

    Every file carries an owner (a uid), a group (a gid), and three sets of three bits — the triads: one for the owner, one for the group, one for everyone else ("other"). Each triad is read/write/execute. When a process tries to touch the file, the kernel picks one triad and uses only that one, in this order:

    1. If your uid is the file's owner → the owner triad decides. Stop. The group and other triads are never even looked at.
    2. Else if one of your gids matches the file's group → the group triad decides. Stop.
    3. Else → the other triad decides.

    That word "stop" is the whole lesson. It is first applicable triad wins, not "combine all the permissions you might qualify for". So if the owner triad is --- (nothing) but the other triad is r--, then the owner cannot read the file while everyone else can — because for the owner, the check stops at the owner triad, which grants nothing. The kernel never falls through to "well, other can read it, and you're a person, so…". You are the owner; the owner triad is your answer, full stop.

    am I the owner?yes →owner triad, done· no, in the group? yes →group triad, done· else →other triad
    Lab
    1. Build the paradox: a file whose owner can't read it but strangers can. Set the owner triad to nothing and the other triad to read. 
      echo 'a secret' > /tmp/f
chmod 044 /tmp/f          # owner ---, group r--, other r--
ls -l /tmp/f
      ----r--r-- 1 ubuntu ubuntu 9 ... /tmp/f
    2. Try to read it as yourself — the owner. Denied, because the check stopped at the empty owner triad and never considered that "other" could read it. 
      cat /tmp/f
      cat: /tmp/f: Permission denied
    3. Now read the very same file as a completely unrelated user, who falls through to the "other" triad and is allowed. This is the first-match rule made visible. 
      sudo -u nobody cat /tmp/f
      a secret
    Verify

    Without running anything, predict: a file -r--rw-r-- owned by alice:staff. If alice (who is also in staff) tries to write it, does she succeed? (No — she is the owner, so the owner triad r-- decides and stops; the more generous group triad rw- is never consulted, even though she is in staff. This is the trap: being in the group does not help you once you match as owner.)

    Gotchas

    Root ignores this entire check. A process with uid 0 (or, more precisely, one holding CAP_DAC_OVERRIDE, which you meet in B4.10) reads and writes regardless of any triad — the rwx bits simply do not apply to it. So "root can't read the file" is almost never a permission problem; it is nearly always something else (a read-only mount, SELinux/AppArmor, an immutable attribute). Permission bits are a wall for ordinary users and a suggestion to root.

    The check is per-operation, on the file as it is now. Changing a file's mode affects only future opens, not descriptors already held — a process that opened the file while it was readable keeps reading through its fd even after you chmod 000 it, because the permission check happened once, at open time (the fd from B3.7 is the proof of a passed check). This is why "I removed the permissions but it's still being read" is not a bug.

    This is the rule behind most "permission denied" errors on mounted volumes in Kubernetes. A container process runs as some uid (from runAsUser); the files on the volume have some owner and group; the kernel applies exactly this first-match check. When a non-root container can't write a volume, the fix is to make the process match a triad that grants write — usually by setting fsGroup so the volume's group becomes one the process carries, landing it in the group triad. Knowing the check is first-match tells you which knob to reach for: change the ownership/group the process matches, not "add more permissions" blindly.

    You will see this concretely in W5 (storage) when a PersistentVolume mounts with root ownership and a non-root pod can't write until fsGroup is set.

    Docs

    man 7 path_resolution and man 2 access for how the check is performed; man 1 chmod under the permission model. man 2 open's ERRORS for the EACCES this produces. In the exam docs, anything about fsGroup and volume permissions.

    B4.6

    rwx on a directory means something different

    Why it matters

    The same three letters — r, w, x — mean one thing on a file and a genuinely different thing on a directory, and confusing the two is behind the single most common real-world permission puzzle: a file that is perfectly readable but that you cannot reach because of a missing bit on a directory three levels up. This lesson makes that failure predictable instead of maddening.

    The concept

    A directory is really just a list mapping names to inodes (the on-disk file records — you will meet inodes properly in B5). The rwx bits govern what you may do with that list, which is why they don't mean what they mean on a file:

    BitOn a fileOn a directory
    rread the contentslist the names inside (what ls needs)
    wmodify the contentscreate, delete, or rename entries in it
    xexecute it as a programtraverse into it — reach things by name and pass through

    The consequential one is x (traverse) versus r (list). They are independent. A directory with x but not r lets you reach a file if you already know its name — you can cat dir/secret — but you cannot ls dir to discover what is in there. The classic "secret dropbox" is exactly this: x so authorised programs can reach known files, no r so nobody can enumerate them.

    Two more consequences that generate real tickets. First, to open /a/b/c/file you need x on /a, on /a/b, and on /a/b/c — every directory on the path — regardless of the file's own mode. One missing traverse bit three levels up denies a file that looks perfectly readable. Second, deleting a file needs w on its directory, not on the file — because delete is really "remove this name from the directory's list". This is why you can delete a file you cannot read, and cannot delete a file in a directory you may only read.

    Lab
    1. Build the traversable-but-unlistable directory. Create it and a file inside first (you need w to create), then strip it down to execute-only. 
      mkdir /tmp/vault
echo 'the answer is 42' > /tmp/vault/secret
chmod 100 /tmp/vault        # --x------ : traverse only
ls -ld /tmp/vault
      d--x------ 2 ubuntu ubuntu 4096 ... /tmp/vault
    2. Reach the file by name — this works, because x lets you traverse in and the file itself is readable. 
      cat /tmp/vault/secret
      the answer is 42
    3. Now try to list the directory. Denied — listing needs r, which you removed, even though you just read a file inside it. 
      ls /tmp/vault
      ls: cannot open directory '/tmp/vault': Permission denied
    4. See the traverse rule bite from above. Remove x from the parent and watch the same readable file become unreachable. 
      chmod 700 /tmp/vault        # restore it, then break the path instead
chmod a-x /tmp/vault
cat /tmp/vault/secret
      cat: /tmp/vault/secret: Permission denied   ← the file is fine; the directory won't let you in
chmod 700 /tmp/vault        # put it back
    Verify

    Explain in one sentence each: why cat dir/secret can work when ls dir fails, and why you can delete a file you have no permission to read. (Reaching a known file needs only x (traverse) on the directory plus read on the file, while listing needs r on the directory; and deletion is removing a name from the directory's list, which needs w on the directory, not any permission on the file.)

    Gotchas

    The command that untangles a path problem is namei -l (or namei -m): give it a path and it prints the mode of every component from the root down, so a missing traverse bit shows up immediately as the one directory in the chain lacking x. Reaching for namei the moment a "readable" file gives EACCES turns a guessing game into a five-second diagnosis — it is the hero of the B4.15 drill.

    chmod -R 755 somedir is a common way to make things worse. Recursively forcing x onto every entry sets the execute bit on regular files too, marking data files as programs. Use the "capital X" form — chmod -R u+rwX,go+rX dir — which adds x only to directories (and files that already had some x), which is almost always what you actually meant.

    Every volume mount, subPath, and projected ConfigMap in Kubernetes is a directory tree, and the same traverse rule governs whether a container process can reach a file inside it. "The file is mode 0644, why can't the container read it?" is very often a missing x on a parent directory in the mount, or a defaultMode that set the wrong bits on the directory. Reading the whole path's permissions — the namei habit — is exactly how you debug it, whether the tree is on a plain host or inside a pod.

    This directory-vs-file distinction underlies the mount mechanics you will study in B5 and the projected-volume permission fields in W3.

    Docs

    man 7 inode for what a directory entry really is, man 1 namei for walking a path's permissions, and man 1 chmod under the symbolic X mode for the recursive-safe form. man 2 unlink confirms deletion is a directory operation.

    B4.7

    Octal on sight, and chown — who may give a file away

    Why it matters

    Permissions are written and spoken in octal — chmod 644, "make it 600", "it should be 755" — and reading those three digits fluently, in both directions, is a small skill you use constantly. It is also completely mechanical once you see it, so ten minutes here buys you a lifetime of not having to think about it.

    The concept

    Each triad of three bits (rwx) is one octal digit. Give each bit a value — r = 4, w = 2, x = 1 — and add up the ones that are set. That sum, 0 through 7, is the digit. Three triads → three digits:

    OctalBitsMeansTypical use
    7rwx4+2+1full access
    6rw-4+2read/write, no execute
    5r-x4+1read/execute, no write
    4r--4read only
    0---0nothing

    So the four you will write in your sleep: 644 (rw-r--r--, a normal data file — owner edits, everyone reads), 755 (rwxr-xr-x, a program or a directory — owner writes, everyone runs/traverses), 600 (rw-------, a private file like an SSH key — owner only), and 700 (rwx------, a private directory). Read them right to left as owner-group-other and the pattern is instant.

    Ownership is changed with chown user:group file (or chgrp for the group alone). The rule that surprises people: only root may give a file away — an ordinary user cannot chown their own file to someone else. If you could, you could sidestep disk quotas by dumping files onto another user, or plant a file you own into someone's space and have it counted as theirs. You can change a file's group to another group you belong to, but handing the file to a different user is a root-only act.

    Lab
    1. Set a mode by number and read it back both ways. stat shows the octal and the symbolic form side by side so you can check your translation. 
      touch /tmp/key
chmod 600 /tmp/key
stat -c '%a %A %U:%G' /tmp/key
      600 -rw------- ubuntu:ubuntu
    2. Translate in your head, then verify. Predict the symbolic form of 751 before running it. (rwxr-x--x.) 
      chmod 751 /tmp/key
stat -c '%a %A' /tmp/key
      751 -rwxr-x--x
    3. Hit the "give it away" wall. Try to hand your file to root as an ordinary user — refused — then do the same with sudo and watch it work. 
      chown root /tmp/key            # as yourself: not allowed
sudo chown root:root /tmp/key  # as root: fine
stat -c '%U:%G' /tmp/key
      chown: changing ownership of '/tmp/key': Operation not permitted
root:root
    Verify

    Cover the table and translate both ways: what is 640 symbolically, and what octal is rwxr-xr--? (640 = rw-r-----; rwxr-xr-- = 754.) Then say why chown-to-another-user failed for you but chgrp to a group you belong to would have worked. (Giving a file to another user is root-only to prevent quota evasion and planting files; changing to a group you are already a member of hands nothing to anyone new.)

    Gotchas

    The Operation not permitted you got from chown is EPERM, not EACCES — and the distinction (which you met in B3.14) matters here. It is not a file-permission problem that a chmod could fix; it is a privileged operation that requires a capability (CAP_CHOWN, coming in B4.10). No amount of adjusting the file's mode lets a non-root user give it away, because the file's mode was never the gatekeeper.

    A leading fourth digit changes the meaning — chmod 4755 is not 755. That extra high digit is the setuid/setgid/sticky bits of B4.9. When stat shows a four-digit mode or ls shows an s or t where you expected x, a special bit is set; don't ignore it.

    Octal is exactly how Kubernetes specifies file modes for mounted config. A ConfigMap or Secret volume takes a defaultMode (and per-key mode) written in octal — 0644 for readable config, 0400 for a secret only the owner may read. The YAML often forces you to write it as a decimal-looking 0644, and reading it correctly (owner rw, everyone r) is the skill from this lesson. Getting a Secret's mode wrong — leaving it group- or world-readable — is a real finding in a cluster security review.

    You will set these modes when you mount Secrets in W3; the 0400-for-secrets instinct comes straight from 600-for-SSH-keys here.

    Docs

    man 1 chmod for both numeric and symbolic modes, man 1 chown/man 1 chgrp for ownership and the root-only rule, and man 1 stat for reading a mode back in either notation (-c '%a %A'). man 2 chown's ERRORS for the EPERM/CAP_CHOWN connection.

    B4.8

    Where default permissions come from: umask

    Why it matters

    You never set the mode of most files you create — you touch, you redirect with >, your editor saves — and yet they come out 644 rather than wide open. Something is choosing that default, and it is umask. Understanding it turns "why did my file come out with these permissions?" from a mystery into a one-line prediction, and it is the mechanism behind a whole class of "the new files have the wrong permissions" problems.

    The concept

    When a program creates a file it requests a mode — conventionally 666 (rw-rw-rw-) for a data file, 777 for a directory. Programs deliberately do not ask for execute on data files, which is why a fresh text file is never executable. Then the kernel subtracts the process's umask — a mask of bits to remove. The result is the actual mode.

    "A mask of bits to remove" is the whole idea, and it is why umask reads backwards at first. A umask of 022 means "remove write from group and other". So:

    • a file: requested 666, minus 022 → 644 (rw-r--r--);
    • a directory: requested 777, minus 022 → 755 (rwxr-xr-x).

    That single umask of 022 — the common default — is why practically every file you create comes out 644 and every directory 755. Change the umask and you change the default: 077 removes all group and other bits, giving private 600 files and 700 directories, which is what you want on a shared machine where new files should be yours alone. The formula is exact: result = requested AND NOT umask, so you can always predict the outcome before you look.

    Lab
    1. Read your current umask, then create one of each and predict before you look. With umask 022, the file should be 644 and the directory 755. 
      umask
touch /tmp/a; mkdir /tmp/d
stat -c '%a %n' /tmp/a /tmp/d
      0022
644 /tmp/a
755 /tmp/d
    2. Change the mask and watch the defaults follow the formula. A umask of 077 strips every group/other bit — new things become owner-only. 
      umask 077
touch /tmp/a2; mkdir /tmp/d2
stat -c '%a %n' /tmp/a2 /tmp/d2
      600 /tmp/a2
700 /tmp/d2
    3. Prove where the "no execute on files" default really comes from: it is the program requesting 666, not the umask. Even umask 000 gives a file 666, never 777. 
      umask 000
touch /tmp/a3
stat -c '%a %n' /tmp/a3
umask 022             # put it back
      666 /tmp/a3            ← rw-rw-rw-, still no execute
    Verify

    With a umask of 027, predict the mode of a new file and a new directory before checking. (File: 666 − 027 = 640, rw-r-----; directory: 777 − 027 = 750, rwxr-x---.) Then explain why a brand-new text file is never executable no matter how permissive your umask. (Because the creating program requests 666, which has no execute bit to begin with — the umask can only take bits away, never add them.)

    Gotchas

    umask is per-process and inherited, not a global setting. It is set for a login by /etc/profile, PAM's pam_umask, or your shell's rc file, and children inherit it — so changing it in one shell affects only that shell and what it launches. If new files in some directory keep coming out with the wrong permissions, the umask of the process that creates them (a daemon, a cron job) is what to check, not your interactive shell's.

    Ubuntu with user-private groups may show a umask of 0002 rather than 0022 for regular users — group write is left on because each user has a private group of one, so "group writable" is harmless. That is why a file can come out 664 on one system and 644 on another; it is the umask differing, not a bug. Check with umask before assuming.

    The umask travels into containers too: a process in a pod inherits a umask (from the image's entrypoint or the runtime's default, commonly 0022), and it silently shapes the mode of every file the app writes to a volume. "Files written by the app are group-readable and shouldn't be" is often a umask inherited from the base image, fixed by setting it in the entrypoint — the same per-process, inherited mechanism you just watched, one layer up. It is not something a securityContext field controls, which is exactly why knowing where the default comes from saves you from looking in the wrong place.

    Docs

    man 2 umask for the syscall and the AND-NOT formula, man 1 bash under umask for the shell builtin, and man 8 pam_umask plus man 5 login.defs (UMASK) for where a login's default is set. man 2 open under the mode argument for how the requested mode meets the umask.

    B4.9

    The special bits: setuid, setgid, and sticky

    Why it matters

    Beyond the nine rwx bits sit three special ones, and they explain things you have seen and wondered about: how an ordinary user changes their own password (which lives in a root-only file), how a shared project directory keeps every new file in the right group automatically, and why /tmp lets everyone write but nobody delete each other's files. Each is a small mechanism with an outsized security footprint — setuid in particular is the reason a single buggy program can be a direct path to root, which is why the whole capabilities half of this module exists to replace it.

    The concept

    Three bits, set with a leading fourth octal digit or symbolically:

    • setuid (u+s, octal 4000) on an executable: the program runs with the file owner's uid as its effective uid, not the caller's. This is the real-vs-effective gap from B4.1 in action. passwd is owned by root and setuid, so when you run it, it runs as root — which it must, because it has to write /etc/shadow (mode 640 root:shadow, unwritable by you). ls -l shows the x in the owner triad replaced by s: -rwsr-xr-x.
    • setgid (g+s, octal 2000) on a directory: new files created inside inherit the directory's group instead of the creator's primary group. This is the standard shared-folder trick — make a project directory setgid to a shared group and everything anyone drops in there is automatically group-owned correctly, no manual chgrp.
    • sticky (+t, octal 1000) on a directory: only a file's owner (or root) may delete or rename it, even in a directory everyone can write. This is what makes /tmp safe: everyone needs to create files there (w on the directory), but without the sticky bit anyone could delete anyone else's, since deletion is a directory-write operation (B4.6). ls -ld /tmp shows the t: drwxrwxrwt.

    Setuid is powerful and dangerous in equal measure. A setuid-root program is a doorway to root that any user may walk through — and if it has a bug, that bug runs as root. That is why the number of setuid binaries on a system is kept small and audited, and why the modern answer is to grant a program the one capability it needs (B4.11) instead of the whole of root via setuid. When the triads and these three bits are still not expressive enough, ACLs (getfacl/setfacl) add per-user and per-group entries on top — worth knowing exist, rarely needed day to day.

    Lab
    1. Look at the canonical setuid binary and read the s. This is why an ordinary user can change their password despite /etc/shadow being root-only. 
      ls -l /usr/bin/passwd
ls -l /etc/shadow
      -rwsr-xr-x 1 root root ... /usr/bin/passwd
-rw-r----- 1 root shadow ... /etc/shadow
    2. Watch setgid keep a shared directory's group tidy. Make a directory owned by a group, set the bit, create a file, and see the file inherit the directory's group rather than your own. 
      mkdir /tmp/shared
sudo chgrp adm /tmp/shared
chmod 2775 /tmp/shared        # the 2 is setgid
touch /tmp/shared/note
stat -c '%A %G %n' /tmp/shared /tmp/shared/note
      drwxrwsr-x adm /tmp/shared          ← note the s in the group triad
-rw-rw-r-- adm /tmp/shared/note    ← inherited adm, not ubuntu
    3. See the sticky bit on /tmp — the t at the end — and understand it as "write to create, but delete only your own". 
      ls -ld /tmp
      drwxrwxrwt ... /tmp
    Verify

    Explain why passwd must be setuid given that /etc/shadow is 640 root:shadow. (An ordinary user cannot write /etc/shadow at all, so changing their own password requires the program to run as root; setuid is what lets a specific vetted program do so.) Then say what the sticky bit on /tmp prevents that plain rwxrwxrwx would allow. (Without it, since delete is a directory-write operation, any user could delete any other user's temp files; sticky restricts deletion to each file's owner.)

    Gotchas

    The Linux kernel ignores the setuid bit on scripts — a setuid #!/bin/bash script does not run as its owner, because that would be trivially exploitable. Setuid only takes effect on real compiled executables. People who set setuid on a shell script and expect elevation are quietly getting nothing; the right tool is a tiny compiled wrapper or, far better, a capability or a sudoers rule.

    Every setuid-root binary is attack surface, so audit them: find / -perm -4000 -type f 2>/dev/null lists them all. A setuid binary you don't recognise is a red flag, and the trend is to have fewer of them — Ubuntu has been steadily replacing setuid with file capabilities (that is why ping no longer needs setuid on modern systems). Fewer setuid-root binaries is a smaller target.

    This is the machinery behind allowPrivilegeEscalation: false in a pod's securityContext. That single field sets the kernel's no_new_privs flag (B4.13), which tells the kernel to ignore setuid and file capabilities on anything the process later execs — so a compromised container cannot use a stray setuid-root binary in its image to climb to root. The fsGroup field, in turn, leans on setgid-style group inheritance to make a volume's contents land in the right group. Both fields are these Linux bits, surfaced as pod configuration.

    You will set allowPrivilegeEscalation: false as standard hardening in W3, and understand it here as "turn off the setuid escape hatch".

    Docs

    man 2 execve under set-user-ID for how setuid actually takes effect (and its refusal on scripts), man 7 inode for the S_ISUID/S_ISGID/S_ISVTX bits, and man 1 chmod under the special modes. man 1 setfacl/man 1 getfacl for ACLs when the triads run out. man 5 capabilities is the forward door to the better answer, B4.10.

    B4.10

    Root is not one flag: capabilities

    Why it matters

    The whole first half of this module treated root as a single all-or-nothing thing: you are uid 0 and the rules don't apply, or you aren't and they do. That has not been the real picture since 1999. Root's power was broken into roughly forty separate capabilities, each a specific permit, and a process can hold any subset. This is the single most important idea for understanding container security, because it is what lets a container's uid-0 process hold three harmless permits instead of all of them — "root", declawed.

    The concept

    A capability is one specific privileged power, carved out of what "being root" used to grant wholesale. Instead of one master switch, the kernel checks the relevant capability for each privileged action. Reuse the keyring image from the module's spine: root is not one master key, it is a keyring of individually-issued permits, and a process can be handed exactly the permits it needs and no others. There are about forty of them (on Linux 6.8, cap_last_cap is 40, so 41 capabilities numbered 0–40). A handful recur constantly:

    CapabilityGrants the power to…
    CAP_NET_BIND_SERVICEbind a socket to a port below 1024
    CAP_NET_ADMINconfigure interfaces, routes, and the firewall
    CAP_SYS_ADMINmount filesystems and a huge grab-bag more — the "junk drawer", and the one to fear
    CAP_CHOWNchange a file's ownership (give it away — B4.7)
    CAP_DAC_OVERRIDEignore the rwx permission bits entirely (this is the reason "root can read anything")

    The practical value is subtraction. A web server that needs to bind port 80 needs exactly CAP_NET_BIND_SERVICE and nothing else. Grant it that one permit and a compromise of the server yields the ability to bind a low port — not the ability to mount filesystems, load kernel modules, override every file permission, and read the whole disk. Full root is the sum of all forty permits; real security comes from handing out one or two and withholding the rest. CAP_SYS_ADMIN is called out above because it is so broad — it gates so many unrelated operations that granting it is very nearly granting full root, which is why "just add SYS_ADMIN" is a security smell.

    Lab
    1. Ask the kernel how many capabilities it knows. The highest number plus one is the count — 41 on this kernel. 
      multipass shell sandbox

      40          ← capabilities are numbered 0–40, so 41 of them
    2. Look at what your ordinary shell holds. capsh --print shows the current sets; as a non-root user, the Current (effective) set is empty — you hold no permits at all. 
      capsh --print | head -3
      Current: =          ← the empty effective set: you hold no permits
Bounding set =cap_chown,cap_dac_override,...,cap_checkpoint_restore
Ambient set =
    3. Now look at root's full keyring for contrast — every permit at once. The compact =ep with no list means "all capabilities, effective and permitted". This is what "root" really is. 
      sudo capsh --print | head -1
      Current: =ep          ← every capability, held and switched on
    Verify

    Name, from memory, the one capability a web server needs to bind port 80, and the one that lets a process ignore file permission bits. (CAP_NET_BIND_SERVICE and CAP_DAC_OVERRIDE.) Then explain in one sentence why granting CAP_SYS_ADMIN is nearly as dangerous as granting full root. (It gates such a large and varied set of privileged operations — mounting, namespace tricks, and more — that holding it is close to holding everything, so it defeats the point of splitting root up in the first place.)

    Gotchas

    Capabilities are spelled CAP_NET_ADMIN in documentation and C, but tools and configs often want them lowercase and without the prefix — cap_net_admin for setcap, and Kubernetes drops the CAP_ prefix entirely (NET_ADMIN). Same capability, three spellings depending on where you are typing; recognise the family, not the exact casing.

    Holding a capability is not the same as it being active — the next lesson's permitted/effective/bounding distinction matters. A process can have a capability "permitted" (available to switch on) but not "effective" (switched on right now), which is why capsh --print shows several sets rather than one list. For now, "the effective set is what is actually in force this instant" is enough.

    This table is the capabilities block of a pod's securityContext. You write drop: ["ALL"] to start from an empty keyring and add: ["NET_BIND_SERVICE"] to hand back exactly the one permit the app needs — least privilege expressed as two lines of YAML. Kubernetes' RuntimeDefault and the hardened container runtimes already drop the dangerous capabilities every workload should never have; understanding the set is what lets you read such a policy and predict what it will and won't break.

    You will write drop: ["ALL"] / add: [...] as standard hardening in W3, and it will make sense because you met the keyring here.

    Docs

    man 7 capabilities is the master reference — read the list of capability names once, and the File capabilities and Thread capability sets sections for the next two lessons. man 1 capsh for --print/--decode/--drop. /proc/sys/kernel/cap_last_cap for the count on any given kernel.

    B4.11

    Attach a capability to a binary with setcap

    Why it matters

    Capabilities become concrete the moment you give one to a program and watch it do something an ordinary user cannot — bind port 80 — while holding only that one permit and running as your unprivileged self. This is setuid's modern replacement: instead of "run as root and hope the program drops privilege", you grant the single power needed and nothing else. Doing it once with your own hands is what makes the container-security lessons land.

    The concept

    File capabilities are capabilities attached to an executable, stored in its extended attributes on disk. When the file is exec'd, the kernel grants the process those capabilities — even though the user is unprivileged and the file is not setuid. Two tools manage them: getcap reads a file's capabilities, setcap sets them. The syntax setcap cap_net_bind_service=+ep file means "give this file cap_net_bind_service in the permitted and effective sets" — the +ep is "permitted + effective", i.e. present and switched on at exec.

    Why this beats setuid: a setuid-root binary runs with all of root's power and must carefully drop what it doesn't need (and a bug before it drops is a root exploit). A file capability grants only the named permit from the start, so even a completely compromised program has just that one power. The blast radius is the difference between "attacker got the ability to bind low ports" and "attacker got root". This is why Ubuntu moved ping from setuid-root to a single file capability (cap_net_raw) — same function, vastly smaller risk.

    Lab
    1. See which binaries already carry a file capability. On a desktop-flavoured install ping often shows one; on a minimal server it may find nothing, which is itself fine. 
      getcap -r /usr/bin /usr/sbin 2>/dev/null
      /usr/bin/ping cap_net_raw=ep          ← ping's one permit, replacing old setuid-root
/usr/bin/mtr-packet cap_net_raw=ep
    2. Build a tiny program that tries to bind port 80 — a privileged act — and prints who it is. Compiled, so a real executable (setuid/caps are ignored on scripts). 
      cat > /tmp/bind80.c <<'C'
#include <stdio.h>
#include <unistd.h>
#include <arpa/inet.h>
#include <sys/socket.h>
int main(void){
    int s = socket(AF_INET, SOCK_STREAM, 0);
    struct sockaddr_in a = { .sin_family=AF_INET, .sin_port=htons(80) };
    if (bind(s, (struct sockaddr*)&a, sizeof a) < 0){ perror("bind"); return 1; }
    printf("bound port 80 as uid %d\n", getuid());
    return 0;
}
C

    3. Run it as your ordinary self. It fails — binding a low port is privileged, and you hold no capability for it. 
      /tmp/bind80
      bind: Permission denied
    4. Grant the one permit and run it again — still as unprivileged you, no sudo on the run itself. It binds. 
      sudo setcap cap_net_bind_service=+ep /tmp/bind80

/tmp/bind80
      /tmp/bind80 cap_net_bind_service=ep

    Verify

    Explain why the second run succeeded even though you were the same unprivileged user with the same uid. (The capability lives on the file; at exec the kernel granted the process cap_net_bind_service, so the bind check passed — no root, no setuid, just the one permit.) Then state the security advantage over making /tmp/bind80 setuid-root in one sentence. (Setuid-root would hand the program the entire keyring, so any bug in it is a path to full root; the file capability gives only the power to bind low ports, so a compromise is contained to that.)

    Gotchas

    File capabilities live in the file's extended attributes, so they do not survive operations that don't copy xattrs — cp without --preserve=xattr drops them, and a filesystem mounted nosuid or without xattr support ignores them entirely. This is the exact cousin of the "setuid bit lost after cp" failure in the B4.15 drill: copy the binary and the capability silently vanishes, leaving you with Permission denied and no obvious cause. Re-run getcap after any copy or move.

    =ep versus =p matters. +ep makes the capability effective immediately at exec, which suits a program that isn't capability-aware; =p (permitted only) hands it over but leaves it switched off, expecting the program to raise it itself via libcap. Give a naive program =p and it will still get Permission denied, because it never turned the permit on. For these labs, +ep.

    When a Kubernetes pod needs to bind a low port or send raw packets, you don't run it as root — you add the one capability: securityContext.capabilities.add: ["NET_BIND_SERVICE"], the pod-spec form of the setcap you just ran. The container's process can then bind port 80 while running as an unprivileged uid with everything else dropped. Recognising that the YAML field and the setcap command are the same mechanism is the whole payoff: the cluster is doing exactly what you did by hand.

    You will add NET_BIND_SERVICE to a hardened pod in W3 and see it bind a privileged port with runAsNonRoot: true — root nowhere in sight.

    Docs

    man 8 setcap and man 8 getcap for attaching and reading file capabilities, man 3 cap_from_text for the +ep/=p flag grammar, and man 7 capabilities under File capabilities for how exec turns them into process capabilities. man 7 xattr for where they physically live.

    B4.12

    The sets a process carries: permitted, effective, bounding

    Why it matters

    A process does not have "a capability" as a single yes/no — it carries several sets, and the difference between them is what makes "root inside a container" possible. Learning to read those sets from /proc/PID/status turns capabilities from an abstraction into something you can inspect on any live process, which is exactly how you audit what a container is actually allowed to do.

    The concept

    Every process carries capabilities in several sets. Three matter now:

    • Permitted (CapPrm) — the capabilities the process may switch on. A permit it holds, whether or not it is currently using it.
    • Effective (CapEff) — the capabilities that are switched on right now. This is the set the kernel actually checks when the process attempts a privileged action.
    • Bounding (CapBnd) — the ceiling. A capability not in the bounding set can never be gained, even by exec'ing a setuid-root or file-capability program. It is the hard upper limit the process can never rise above.

    The bounding set is the one that does the security work. Here is the surprise that makes it click: your ordinary non-root shell has an empty permitted and effective set — it holds no capabilities — but a full bounding set. That means you hold nothing, yet the ceiling is high, which is precisely why running a setuid-root program can hand your session full root: the capabilities were always within your bounding ceiling, waiting to be granted by that exec. Now trim the bounding set — remove CAP_SYS_ADMIN from the ceiling — and no exec, no setuid binary, nothing, can ever give this process (or its children) that capability again. That is the mechanism a container runtime uses to defang root, and the next lesson puts it to work.

    These sets are exposed as hex bitmasks in /proc/PID/status (CapPrm, CapEff, CapBnd, and also CapInh/CapAmb), one bit per capability. You don't decode hex by hand — capsh --decode= turns a mask back into names.

    Lab
    1. Read your own shell's sets. Note the shape: effective and permitted are all zeros (you hold nothing), but the bounding ceiling is full.

    Gotchas

    The hex masks look intimidating and are meaningless to eyeball — always pipe them through capsh --decode= rather than trying to read bit positions. And watch the leading zeros: 000001ffffffffff (41 bits) is the current full set, but you will see older docs show 0000003fffffffff (38 bits) or other widths from kernels with fewer capabilities. The width tracks cap_last_cap, so the "full" mask is a moving target across kernel versions — decode it rather than memorising it.

    There are two more sets you'll see, CapInh (inheritable) and CapAmb (ambient), which govern what passes across an execve to a non-capability-aware program. They are usually empty and you can ignore them until you deliberately need capabilities to survive an exec; mentioning them so their zero rows in /proc/PID/status don't puzzle you.

    Reading CapEff/CapBnd from /proc/PID/status is exactly how you audit what a container can really do — regardless of what its YAML claims. A container runtime starts each container's PID 1 with a reduced bounding and effective set (the runtime default drops the dangerous ones), so uid 0 inside a normal container shows a much shorter CapEff than systemd's full mask above. When you suspect a pod is over-privileged, grep Cap /proc/PID/status on the container's process (or cat it via kubectl exec) and decode it — the truth is in the bitmask, not the manifest.

    This is the concrete backing for "the container's root is a trimmed root", which the final lesson of the group turns into the full container-security story.

    B4.13

    Why root in a container is not root on the host

    Why it matters

    This is the lesson the whole module has been building toward. Everyone learns early that "running as root in a container is bad", and everyone also hears "but container root isn't really root". Both are true, and the reconciliation is the capabilities machinery you just built. Being able to explain, precisely, why uid 0 inside a container is a defanged identity is what separates cargo-culting security advice from understanding it.

    The concept

    Two mechanisms, working together, turn a container's uid-0 process from "owns the machine" into "owns almost nothing dangerous":

    • A trimmed bounding set (B4.12). The container runtime starts the container's first process with most capabilities removed from the bounding ceiling. So even though the process is uid 0, and even if it execs a setuid-root binary, it can never gain CAP_SYS_ADMIN, CAP_NET_ADMIN, and the rest — they are above its ceiling forever. It keeps a short list (enough to chown its own files, kill its own processes) and nothing that touches the host.
    • no_new_privs. This is a one-way switch a process can set on itself that says "nothing I exec may ever gain privileges" — it makes the kernel ignore setuid bits and file capabilities on every program the process later runs. Once set, it cannot be unset, and children inherit it. It slams shut the setuid escape hatch entirely, so even a stray setuid-root binary in the image is inert.

    Put them together and "root in the container" means: uid 0, yes, but with a ceiling so low it cannot reconfigure the network, mount a filesystem, load a kernel module, or override the host's file permissions — and with the setuid path nailed shut so it cannot climb. It is a genuinely different, far weaker identity than root on the host, despite sharing the number 0. Add the user namespace of B6 — which maps container uid 0 to an ordinary unprivileged uid on the host — and the isolation becomes near-total: the container's "root" is, from the host's point of view, just another powerless user.

    uid 0 in container+ trimmed boundingcan't gain dangerous caps+ no_new_privscan't climb via setuid=root in name, harmless on the host

    Gotchas

    "Container root is safe" is only true with the defaults in place. --privileged (Docker) or securityContext.privileged: true (Kubernetes) throws all of it away — full bounding set, all capabilities, no restrictions — handing the container real host root. Likewise adding CAP_SYS_ADMIN back, or mounting the host's /, punches straight through the fence. The protection is the trimmed set and no_new_privs; remove them and uid 0 inside is uid 0 on the host again.

    Dropping capabilities is necessary but not the whole of container isolation. Namespaces (B6) separate what the process can see, cgroups (B6) cap what it can consume, and seccomp (B12) limits which syscalls it may even make. Capabilities limit which privileged operations succeed; the complete picture is all four layers together, which the beginner track builds up module by module.

    This lesson is the "why" behind the entire hardened-pod recipe you will apply in W3: runAsNonRoot: true, allowPrivilegeEscalation: false (which sets the no_new_privs you just saw), capabilities.drop: ["ALL"] (which trims the bounding set to empty), and a seccompProfile of RuntimeDefault. Every one of those fields is a Linux mechanism from this module. When a security review flags a pod as privileged or over-capable, you now know exactly what protection was thrown away and what an attacker gains — that is the difference between applying the recipe and understanding it.

    The forward path: B6 adds namespaces and cgroups to complete the isolation story, and W3 is where the whole thing becomes a pod manifest you write from memory.

    B4.14

    Port 80 without root, three ways — and their blast radius

    Why it matters

    Every idea in this module meets in one concrete engineering problem: serve HTTP on port 80 without running the app as root. Binding a port below 1024 is privileged, so the naive answer is "run it as root" — and that answer is exactly what good security spends its effort avoiding. There are three real ways to solve it, and they are not equivalent: they leave wildly different amounts of power in the hands of an attacker who compromises the server. Building all three and ranking their blast radius is the whole of practical privilege thinking, and it is precisely the decision behind every hardened pod spec.

    The concept

    One privileged act — bind() on port 80 — three ways to grant just enough to do it:

    WayHow the bind is allowedThe server process runs as…Blast radius if compromised
    1. setuid rootthe binary is owned by root and setuid, so it runs as rootuid 0 — full rooteverything: the attacker owns the host
    2. file capabilitycap_net_bind_service on the binaryyour unprivileged uid + one permitthe ability to bind low ports — nothing else
    3. high port + DNATserver binds 8080 (unprivileged); the kernel redirects 80→8080your unprivileged uid, no capabilitiesnothing privileged — the redirect lives in the kernel, out of the process's reach

    The ranking is the lesson. Way 1 solves the problem and hands an attacker the entire machine — the worst trade in security, "elevate everything to do one small thing". Way 2 grants the single permit the job needs, so a compromise yields only the power to bind low ports. Way 3 grants the server no privilege at all: it binds an ordinary high port like any other program, and a completely separate, kernel-level rule (a DNAT/REDIRECT, from the packet-mangling you'll meet in B9) sends port-80 traffic to it. An attacker who owns the process has escaped nothing, because the process never held anything. Smallest blast radius wins, and it is almost always way 2 or way 3.

    Gotchas

    The DNAT in way 3 has a subtlety worth internalising: a REDIRECT in nat/PREROUTING only fires for packets arriving on an interface, and traffic you originate to localhost never traverses PREROUTING — it goes through OUTPUT. This fragility is a feature: it is one more reason setuid is the worst of the three, not just the most powerful.

    This project is the blast-radius reasoning behind every field in a pod's securityContext. Running a container as root to bind port 80 is way 1 — the trade the platform pushes you hard to avoid. Adding capabilities.add: ["NET_BIND_SERVICE"] to a non-root container is way 2. Binding a high container port and exposing it as port 80 through a Service is way 3 — the process holds nothing, and the port mapping lives in the cluster's networking, exactly like the kernel DNAT. When you choose how a workload gets its privileged port, you are choosing a blast radius, and this lab is why you'll reach for 2 or 3.

    You'll implement way 2 (added capability) and way 3 (Service remapping) in W3 and W4; the ranking you just built is the reason those are the recommended patterns and privileged: true is not.

    B4.15

    Debugging drill: the traverse ladder and the copied setuid binary

    Why it matters

    Two failures drilled until they are reflexes, because both look like something they aren't. First: a file that is provably readable but that you cannot open — because a traverse bit is missing on a directory somewhere above it, not on the file. Second: a setuid program that worked yesterday and fails today because someone copied it, silently dropping the very bits that made it work. Both waste hours when guessed at and take thirty seconds when you know the tool and the mechanism.

    The concept

    You have every piece already. The traverse ladder: opening /a/b/c/file needs x on every directory in the path (B4.6), so a readable file can be unreachable because of a directory three levels up — and namei -l prints the mode of every component at once, making the culprit obvious. The copied setuid binary: setuid depends on both the s bit and root ownership (B4.9), and cp preserves neither by default — the copy is owned by whoever ran cp and has an ordinary x where the s was, so it runs with no elevation and fails on its first privileged act. The discipline both share: work from evidence to cause — read the actual permissions and ownership, don't guess at the fix.

    Lab

    Snapshot first (B0.9). Work each symptom, state a one-sentence cause backed by a command that proves it, then read the reveal.

    Symptom 1 — a readable file you cannot open
    1. Set up a file that is unambiguously readable, buried under a directory that isn't traversable by you. 
      sudo mkdir -p /srv/data/reports

      cat /srv/data/reports/q3.txt

    A missing traverse bit, not a file problem. /srv/data is drwxr-x--- — owner and group may traverse, "other" (you) may not, so the path is severed two levels above the file. The file's own 644 was never the issue. Fix by restoring traverse for the path: sudo chmod o+x /srv/data (or, better on a real system, add yourself to the owning group so you don't widen access to everyone). The reflex: when a "readable" file gives EACCES, run namei -l before touching the file's own mode.

    Symptom 2 — the setuid binary that broke after a copy
    1. Make a working setuid-root helper (reuse the project's httpd, or any small binary), confirm it elevates, then copy it the way a careless deploy script would. 
      gcc -o /tmp/svc /tmp/httpd.c 2>/dev/null || printf 'int main(){return 0;}' | gcc -x c -o /tmp/svc -

    2. Read the evidence, don't guess. The copy lost two things at once — the s became a plain x, and ownership went from root to you — so it can no longer run as root and fails on its first privileged act.

    cp preserved neither the setuid bit nor the root ownership. A plain cp creates a new file owned by the copier with a default mode, dropping the 4000 setuid bit and the root owner that setuid depends on. Even cp -p (or --preserve=mode,ownership) cannot restore root ownership when run by a non-root user — only root can give a file to root (B4.7). The real fix is to re-apply after the copy, as root: sudo chown root:root /tmp/svc.deployed && sudo chmod u+s /tmp/svc.deployed. The deeper lesson: setuid is fragile precisely because it rides on bits and ownership that ordinary file operations quietly discard — which is one more argument for the file capabilities of B4.11, that fail the same way for the same reason and are equally worth checking after a copy.

    Gotchas

    Both traps share a shape: the symptom points at the wrong object. Symptom 1 screams "the file!" when the fault is a directory above it; symptom 2 screams "the program is broken!" when the fault is lost metadata from a copy. Reading the real state — namei -l for the path, ls -l for owner-and-bits — is what redirects you from the loud-but-innocent object to the quiet-but-guilty one. Work from evidence to cause, the same discipline as the B2 and B3 drills.

    File capabilities fail identically to setuid after a copy and are easier to overlook because there's no visible s in ls -l — only getcap shows them. If a capability-based binary stops working after a build or deploy, getcap it before anything else; the capability almost certainly didn't survive the copy.

    The traverse ladder is a routine pod failure: a volume mounted with a restrictive mode on a parent directory, or a subPath whose intermediate directory lacks x for the container's uid, produces exactly this "the file is 0644, why can't the container read it" symptom — and the fix is the same reasoning about the whole path, not the file. The copied-setuid failure maps onto image builds: a COPY in a Dockerfile can drop setuid bits and file capabilities just as cp does, so a binary that relied on them is inert in the built image until the layer re-applies them. Both are "metadata didn't survive the copy", one layer up.

    These are the permission-shaped incidents you'll meet in W6 troubleshooting; the namei/getcap reflexes transfer directly to debugging them in a pod.

    B4 recall quiz

    Answer out loud before revealing. If you can predict the outcome of a permission check in your head, every securityContext becomes readable.

    1. The kernel is about to check whether a process may read a file. What does it actually compare, and what role do the names ubuntu/root play?
      Answer

      It compares numbers — the process's uid and gids against the file's owner uid and group gid. Names are a human convenience looked up from /etc/passwd//etc/group; the kernel never consults them. This is why a uid with no name still works and why mounted files show as "1000" rather than a username on a host that lacks that entry.

    2. Why did the password hash move from /etc/passwd to /etc/shadow when /etc/passwd's permissions never changed?
      Answer

      /etc/passwd must stay world-readable for name lookups, so leaving hashes there let any user copy and attack them offline. /etc/shadow is readable only by root, so the hashes are no longer exposed. The x in the password field is the leftover placeholder meaning "the secret is in shadow".

    3. A file is ----r--r--, owned by you. Can you read it? Can a stranger?
      Answer

      You cannot; a stranger can. The check is first-match: you are the owner, so the owner triad (---) decides and stops — the more generous "other" triad is never consulted for you. A stranger matches "other" (r--) and reads it. Matching as owner is not always an advantage.

    4. A file is mode 0644 but cat gives "Permission denied". What's the most likely cause and the one command to confirm it?
      Answer

      A missing traverse (x) bit on a directory somewhere in the path — opening a file needs x on every directory above it, whatever the file's mode. Confirm with namei -l /path/to/file, which prints every component's mode and shows the one directory lacking x.

    5. Your umask is 027. What mode does a new file get, and a new directory, and why is the file never executable?
      Answer

      File: 666 − 027 = 640 (rw-r-----); directory: 777 − 027 = 750 (rwxr-x---). The file is never executable because programs request 666 (no execute) for data files, and the umask can only remove bits, never add them.

    6. Why must /usr/bin/passwd be setuid, and why does the kernel ignore the setuid bit on a shell script?
      Answer

      passwd must write /etc/shadow, which is 640 root:shadow and unwritable by an ordinary user, so it needs to run as root — setuid provides that. The kernel ignores setuid on scripts because a setuid interpreter reading an attacker-controlled script would be trivially exploitable; only compiled executables honour setuid.

    7. "Root is one all-powerful flag." Correct it, and give the one capability that lets a process ignore file permission bits.
      Answer

      Root is roughly forty separate capabilities (41 on Linux 6.8), each a specific power, and a process can hold any subset. The capability to ignore the rwx bits entirely is CAP_DAC_OVERRIDE — it, not "being uid 0" per se, is why root can read any file.

    8. You gave a binary cap_net_bind_service=+ep and it binds port 80 as an unprivileged user. Why is this safer than making it setuid-root?
      Answer

      The file capability grants only the single power to bind low ports, so a compromise of the program yields just that. Setuid-root would run the program with the entire capability set, so any bug in it is a path to full root. Least privilege: hand over one permit, not the whole keyring.

    9. Your non-root shell shows CapEff: 0000000000000000 but CapBnd: 000001ffffffffff. Explain, and say what that combination permits.
      Answer

      You currently hold no capabilities (empty effective set), but the bounding ceiling is full. Because the capabilities are within your ceiling, exec'ing a setuid-root binary can raise them into your effective set — which is exactly how sudo/passwd give you elevated power. Trimming the bounding set would remove that possibility permanently.

    10. Explain "root in the container, harmless on the host" naming the two mechanisms that make it true.
      Answer

      The runtime starts the container's uid-0 process with a trimmed bounding set, so it can never gain the dangerous capabilities (mount, net-admin, sys-admin) even by exec; and no_new_privs makes the kernel ignore setuid/file-caps on anything it execs, closing the escalation hatch. It is uid 0 with none of root's host powers. A user namespace (B6) additionally maps that uid 0 to an unprivileged host uid.

    11. A setuid-root helper worked, then stopped after a deploy script copied it. What did the copy break, and can you fix it with cp -p as a normal user?
      Answer

      cp dropped both the setuid bit and the root ownership; the copy is owned by the copier with a plain x, so it no longer elevates. cp -p as a non-root user can restore the mode but not root ownership — only root may give a file to root — so the real fix is sudo chown root:root plus sudo chmod u+s after the copy.

    12. You must serve port 80 without running the app as root. Rank setuid-root, a file capability, and a high-port-plus-DNAT by blast radius, best first.
      Answer

      Best (smallest): high port + DNAT — the process holds no privilege at all; the redirect is kernel-side. Then the file capability — the process holds only "bind low ports". Worst (largest): setuid-root — the process is full root, so a compromise owns the host. The right instinct is DNAT or capability, never setuid.
