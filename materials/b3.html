    B3 — Kernel space, user space, system calls

    There is a hard line down the middle of every Linux machine. On one side is your program: the code you wrote, the memory it was handed, the arithmetic it does. On the other side is everything real — the disk, the network card, the memory of every other process, the clock. Your program cannot touch any of it directly. When it needs something from that side, it asks, through a single narrow doorway called a system call. That is the whole architecture, and this module is about learning to see the doorway.

    Why does it matter to someone who wants to run Kubernetes? Because when a container misbehaves and its logs say nothing, the logs are only the story the program chose to tell. The system calls are what it actually did — every file it opened, every connection it made, every permission it was denied. strace shows you that list, and by the end of this module you will read one the way you read a sentence. You will also stop treating /proc and /sys as mysterious folders and start using them as the live API to the kernel that they are — the same API every diagnostic tool you own is quietly built on.

    The boundary

    B3.1

    What the kernel exclusively owns

    Why it matters

    Before you can trace what a program does, you need to know what it is even allowed to do on its own. The answer is: almost nothing that matters. A program can do arithmetic and shuffle its own memory around, and that is the complete list of what it does without permission. Everything else — reading a file, sending a packet, checking the time, starting another program — belongs to the kernel, and the program has to ask. Getting this ownership boundary clear in your head is what makes the rest of the module, and most of Linux troubleshooting, make sense.

    The concept

    The kernel is the one program that is always running and that the hardware trusts completely. It is loaded first, at boot, and from then on it is the sole owner of five things that no ordinary program is ever allowed to touch directly:

    The kernel exclusively owns……so a program cannot, by itself
    the physical hardware — disks, network cards, the clock, the keyboardwrite a byte to a disk or send a packet without asking
    the memory map of every processread or corrupt another process's memory
    the scheduler that decides who runs on the CPU nextrefuse to yield the CPU, or starve its neighbours
    the filesystems — the whole tree, and what is mounted wheresee files outside what it has been given access to
    the entire network stack — addresses, routes, connectionsopen a socket or bind a port on its own authority

    Return to the hotel from B2. The kernel is the hotel's engineering staff — the only people allowed to touch the wiring, the boiler, the water mains. A guest who wants hot water does not walk into the plant room and open a valve; they file a work order at the desk, and engineering does it for them. The rule is not there to be annoying. It is there so that one confused or malicious guest cannot flood the building, cut everyone else's power, or read the mail in another guest's room. Isolation and safety are the whole point of the boundary.

    This is why "my program crashed" almost never takes the machine down with it. When your code dereferences a bad pointer, it is only touching its own address space, which the kernel owns the map of; the kernel notices the illegal access, kills just that one process, and the rest of the hotel never notices. The kernel itself crashing — a "kernel panic" — is the rare catastrophic case precisely because the kernel is the one thing with nobody above it to catch the fall.

    your programownsits own memory + arithmetic·the kernelownshardware · other memory · scheduling · filesystems · network

    Gotchas

    "The kernel" and "the operating system" are not the same size. Ubuntu is a distribution — a kernel plus thousands of ordinary user-space programs (the shell, systemd, apt, every daemon). Only the kernel runs on the privileged side of the boundary; everything else, including the parts that feel system-y, is a normal process asking the kernel for things like any other.

    Root does not move you to the kernel's side. A root process is still an ordinary process making system calls — it is just one the kernel grants nearly every request. That distinction matters the moment you meet capabilities in B4: root is not "being the kernel", it is "having every permit the kernel hands out".

    A Kubernetes node is this boundary at scale. Every container on the node is a user-space process on one side; the single shared kernel is on the other. This is the fact behind the sentence "containers share the host kernel" — there is exactly one engineering staff for the whole building, no matter how many guests. It is why a kernel bug is a whole-node event, and why securityContext exists: since containers cannot be given a kernel of their own, the platform instead limits which requests each one may make of the shared kernel.

    You will see the node-level consequence in W6.1 — the node isn't ready: when the kernel is unhappy, no amount of application-level fixing helps, because everything above it depends on the one thing below it.

    Docs

    man 7 kernel is thin, so the better orientation is man 2 syscalls (the full list of things a program is allowed to ask for) and man 5 proc under /proc/version. man 1 uname for the identity of the running kernel. For the conceptual split, the "user space" and "kernel space" sections of any operating-systems reference — but you will learn it faster by watching strace in B3.3 than by reading about it.

    B3.2

    User mode, kernel mode, and what crossing costs

    Why it matters

    The boundary from the last lesson is not a metaphor or a software convention — it is a state of the CPU itself, enforced by the silicon. Understanding that it is a hardware wall, and that every crossing has a real cost, is what turns a vague performance rule ("don't do too many tiny reads") into something you can reason about and measure. It is also the mechanism that makes containers possible at all.

    The concept

    A modern CPU can run in (at least) two privilege levels, and it is always in exactly one of them:

    • User mode — the restricted state. The instructions that touch hardware, change the memory map, or talk to devices are simply disabled. If code in user mode tries one, the CPU refuses and traps into the kernel. Your program runs here, always.
    • Kernel mode — the unrestricted state. Every instruction is available. Only the kernel runs here.

    A system call is the controlled, one-way door between them. It is emphatically not a function call to a library — a function call stays in user mode and just jumps to another address in your own program. A syscall executes a special CPU instruction (syscall on x86-64) that does something a normal jump cannot: it flips the CPU into kernel mode, but only at a fixed entry point the kernel chose in advance. The program cannot pick where in the kernel it lands, which is the entire safety property — you may knock, but only at the doors engineering installed.

    Because a crossing means saving the program's state, switching mode, running kernel code, switching back, and restoring state, it costs far more than a plain function call — on the order of hundreds of nanoseconds versus a couple. One crossing is nothing. A million of them is a performance bug you can see. This is the concrete reason "too many small reads" is slow: reading a megabyte in one read of a megabyte is one crossing; reading it a byte at a time is a million crossings, and the crossings, not the copying, dominate. It is why buffering exists everywhere in software — libc's stdio, your language's I/O library — all of it is machinery to turn many tiny would-be syscalls into a few big ones.

    user modesyscall instruction →trap to fixed kernel entry→kernel mode does the work→ returnuser mode
    Lab

      time dd if=/tmp/big of=/dev/null bs=1M 2>/dev/null
      real	0m0.00s
    3. Prove the difference is the syscalls themselves by counting them. strace -c (the subject of B3.8) tallies calls; the byte-at-a-time run makes ~a million reads, the block run a handful. 
      strace -c dd if=/tmp/big of=/dev/null bs=1 2>&1 | grep -E 'read|calls'

 51.20    1.010     1          1000001           read
    Verify

    Explain, in one sentence each, why a syscall is not a library call and why buffering makes I/O fast. Then predict: if you rewrote the byte-at-a-time program to read into a 4 KB buffer, roughly how many read syscalls would a one-megabyte file take? (About 250 — one per 4 KB block — down from a million, which is the entire reason stdio uses a buffer that size by default.)

    Gotchas

    Not every library function is a syscall in disguise, and not every syscall is slow. A crossing is a fixed overhead, not a catastrophe — the rule is "don't make a million needless ones", not "avoid syscalls". Reading a whole file with one big read is one crossing and completely fine.

    The mode switch here is a different thing from the process context switch (swapping one process off the CPU for another), even though people blur them. A syscall crosses into the kernel but usually stays in the same process; a context switch changes which process is on the CPU and is more expensive again. Both are kernel-owned, which is why they get confused.

    This cost is why the kernel/user boundary shapes how fast software can go, and why some performance-critical paths try to minimise crossings (batching, io_uring, eBPF running logic inside the kernel). For your purposes as an operator, the payoff is smaller and more practical: when a container is burning CPU in "system" time (%sy in top) rather than "user" time, it is spending its life crossing the boundary — usually doing an enormous number of small syscalls — and that is a specific, findable class of problem rather than a mystery.

    You will meet the system-vs-user CPU split again in node troubleshooting; keep W6 in mind as the place where "high sys time" becomes a diagnosis rather than a curiosity.

    Docs

    man 2 syscall describes the raw mechanism and the architecture-specific instruction and registers. man 2 intro frames what system calls are and how errors come back. man 7 vdso is worth a skim for the clever exception — a few "syscalls" like gettimeofday are answered without a real crossing at all, which is itself evidence of how much the crossing is worth avoiding.

    B3.3

    See the doorway: strace -f echo hi

    Why it matters

    Everything so far has been description. Now you watch it happen. strace prints every system call a program makes, in order, as it makes them — so the abstract "narrow doorway" becomes a concrete, scrolling list you can read. Once you have seen that even echo hi is a couple of dozen kernel requests, the boundary stops being theory. This one command is the most useful debugging tool in this entire module, and you will use it for the rest of your career.

    The concept

    strace ("system-call trace") runs a program and reports each syscall it makes: the call's name, its arguments, and what came back. It works by asking the kernel — through the same ptrace mechanism a debugger uses — to stop the target every time it crosses the boundary and let strace peek. That is why it can see calls with no special access to the program's source: it is watching at the doorway, which the kernel controls, not inside the program.

    The -f flag means "follow children": if the program starts other programs (via the clone/execve you learned in B2), trace those too. Reach for -f by default — without it, tracing anything that spawns helpers shows you almost nothing, and a huge amount of real software is a parent that immediately forks.

    When you trace something as small as echo hi, the output splits into a shape you will start to recognise everywhere. The first line is always execve — the kernel loading your program into a fresh address space (this is the B2.5 exec step, seen from below). Then a burst of startup: mapping the C library into memory (openat, mmap, mprotect), setting up the memory allocator (brk). Then, buried in the middle, the one line that is the actual job — for echo hi, a single write of the string. Then exit_group, the program ending. Nearly all of the output is ceremony; the real work is usually one or two lines, and learning to find them in the noise is the skill.

    Lab
    1. Trace the smallest useful program. It scrolls a screen or two; that is the point — an enormous amount of machinery underlies "print two characters". 
      strace -f echo hi
    2. Look at the very first line. This is the kernel replacing the shell's copy with the echo program — the exec from B2, from the kernel's side. Note it lists the arguments and a count of environment variables. 
      execve("/usr/bin/echo", ["echo", "hi"], 0x7fff... /* 73 vars */) = 0
    3. Find the startup burst. Every one of these is the C library being wired into the process before your program's own logic runs — the same sequence in almost every binary you will ever trace. 
      openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY|O_CLOEXEC) = 3
openat(AT_FDCWD, "/lib/x86_64-linux-gnu/libc.so.6", O_RDONLY|O_CLOEXEC) = 3
mmap(NULL, 2029760, PROT_READ, MAP_PRIVATE|MAP_DENYWRITE, 3, 0) = 0x7...
brk(NULL)                               = 0x63441ae4f000
    4. Now find the one line that is the actual work. Out of the whole trace, this single write is echo doing its job — three bytes (hi\n) to file descriptor 1, standard output. Everything else was getting ready. 
      write(1, "hi\n", 3)                     = 3
close(1)                                = 0
close(2)                                = 0
exit_group(0)                           = ?
+++ exited with 0 +++
    Verify

    Run strace -f echo hi again and, out loud, narrate the three phases: load (execve), get ready (the libc/mmap/brk burst), do the job (write), end (exit_group). Then answer: which single line would still be there if you traced echo goodbye instead, unchanged in shape? (The write — same call, same fd 1, just different bytes. The whole startup is identical because it is about loading echo, not about what you asked it to print.)

    Gotchas

    strace mixes its own output with the program's, and both go to your terminal, which is confusing at first. strace writes the trace to standard error and the program writes hi to standard output, so they interleave. To separate them, send the trace to a file with -o trace.txt — then the program's real output is clean on screen and the trace waits for you in the file. B3.13 relies on exactly this.

    Tracing slows the program down a lot — every syscall now stops the process twice. That is fine for understanding a program and a disaster for measuring its speed; never draw performance conclusions from wall-clock time under strace. Use strace -c (B3.8) for counts and a separate un-traced run for timing.

    strace may be blocked by a hardened kernel's ptrace restrictions or inside a container without the right permissions, giving "Operation not permitted". That is the ptrace mechanism itself being fenced off — reasonable on a locked-down host, and worth recognising so you do not mistake it for a broken install.

    When a container crashes with an unhelpful log, strace is how you find out what it was really trying to do — often it is one denied openat (a missing config file), one failing connect (a service it cannot reach), or one EACCES (a permission it was not granted). The Kubernetes-shaped version is kubectl debug attaching an ephemeral container that shares the target's namespaces, so you can strace a running pod's process without rebuilding its image.

    The through-line to W6 troubleshooting: application logs tell you what the program decided to say; the syscall trace tells you what it actually did. When those two disagree — the log says "connecting…" forever but the trace shows the connect returning ECONNREFUSED — the trace is the truth.

    Docs

    man 1 strace — long but navigable; the options that earn their keep are -f, -o, -c, -e trace= (filter to a set of calls, e.g. -e trace=openat,connect), -p (attach to a running PID), and -s (string length shown). man 2 ptrace if you want the mechanism strace is built on. For the "which syscalls exist" reference behind every line, man 2 syscalls.

    B3.4

    Place libc: your code vs the kernel's, seen with ltrace vs strace

    Why it matters

    People say "printf makes a system call" as if printf is one. It is not. printf, fopen and malloc are ordinary functions living inside your own process, running in user mode — they only cross the boundary sometimes, and when they do, the syscall they make has a different name than the one you called. Getting this layering right explains a whole family of confusions, including the classic "my program printed the line but the output is gone after it was killed".

    The concept

    Between your program and the kernel sits the C library (libc, on Ubuntu the GNU version, glibc). It is not part of the kernel; it is a library mapped into every process — you saw it load in the last lesson's startup burst. Its job is to give programs a comfortable, portable interface (printf, fopen, malloc, getpwnam…) and, underneath, to make the actual system calls on their behalf, handling the fiddly register setup and the architecture differences.

    So there are two layers of "calls", and two tools that see them:

    • ltrace shows library calls — the functions your program calls in user space: printf("..."), malloc(24), strlen(...). This is the layer you wrote against.
    • strace shows system calls — the requests that actually cross into the kernel: write(1, ...), brk(...), mmap(...). This is the layer the kernel sees.

    The relationship between them is not one-to-one, and that gap is the whole lesson. A hundred printf calls may produce a single write syscall, because libc keeps an internal buffer and only flushes it to the kernel when the buffer fills, when you print a newline to a terminal, or when the program exits cleanly. Your malloc calls usually make no syscall at all — libc hands you memory it already has, and only occasionally asks the kernel for more with brk or mmap.

    This buffering explains the notorious lost-output bug. If a program printfs a line and is then killed with SIGKILL (B2), the line can vanish — because it was sitting in libc's userspace buffer, on your side of the boundary, and SIGKILL gives no chance to flush it. The text never became a write syscall, so it never reached the terminal. "It printed but I lost it" means "it called printf but never reached write".

    your call: printf ×100→libc buffers in user space→ onesyscall: write ×1→kernel
    Lab
    1. Build a tiny program that prints and allocates. Nothing here is a syscall by name — it is all libc. 
      cat > /tmp/hello.c <<'C'
#include <stdio.h>
#include <stdlib.h>
int main(void) {
    char *p = malloc(24);
    printf("hello\n");
    free(p);
    return 0;

    2. Watch the library layer with ltrace. You see the functions you actually wrote — the malloc, the printf, the free.

+++ exited (status 0) +++
    3. Now watch the kernel layer with strace on the same binary. The printf has become a write; the malloc/free made no syscall at all this time (libc had the memory already); a brk appears only where libc topped up its arena. 
      strace /tmp/hello 2>&1 | grep -E 'write|brk'
      brk(NULL)                               = 0x63441ae4f000

    4. See the buffering directly. Redirect the output to a file instead of the terminal, and libc switches from line-buffered to a big block buffer — so the write now happens only at exit, and if the program were killed first the "hello" would be lost. 
      strace /tmp/hello > /tmp/out 2> /tmp/trace

      write(1, "hello\n", 6)                  = 6
    Verify

    State which tool you would reach for to answer "how many times did my program actually hit the disk" (strace — that is the kernel layer) versus "is my program calling malloc in a hot loop" (ltrace — that is the library layer). Then explain in one sentence why a program that "definitely printed the error" can show no error after a SIGKILL. (The line was in libc's user-space buffer and never became a write; SIGKILL gave no chance to flush it across the boundary.)

    Gotchas

    ltrace is not installed by default on Ubuntu 24.04 (sudo apt install -y ltrace) and, unlike strace, it can be defeated by modern binary hardening — on some statically-linked or heavily-optimised programs it shows little or nothing. It is a teaching tool and an occasional aid, not the everyday workhorse; strace is the one to trust.

    The buffering rule depends on where output goes. To a terminal, stdout is line-buffered (flushes on each newline), which is why you see interactive output promptly. To a file or a pipe, it becomes fully block-buffered (flushes only when full or at exit) — the reason a long pipeline can appear to produce nothing for a while and then dump everything at once. stderr is unbuffered, which is precisely why error messages should go there: they survive a crash that would swallow buffered stdout.

    This is the single most common reason a crashing container's logs look truncated. A pod writing logs to stdout that is being captured (a pipe, not a terminal) is block-buffered; when the container is OOM-killed or SIGKILLed, the last buffer never flushes and the most interesting lines — the ones just before death — are exactly the ones you lose. The standard fixes are the process-level ones you now understand: run the program with unbuffered output (PYTHONUNBUFFERED=1, stdbuf -oL, or the language's flush call), so each line crosses the boundary immediately.

    Keep this in view for W6: "the logs cut off right before the crash" is almost never the logging system losing data — it is user-space buffering that never got flushed, the same mechanism you just watched.

    Docs

    man 1 ltrace for the library-call view and its filters (-e). man 3 printf and man 3 malloc to confirm these are library (section 3) functions, not system (section 2) calls — the manual section number itself tells you which side of the boundary a name lives on. man 3 setvbuf for how buffering is chosen and how to change it. man 7 stdio for the buffering rules in one place.

    Reading syscalls

    B3.5

    Read one strace line completely — including errno

    Why it matters

    A trace is only useful if you can read a single line of it with confidence. The good news is that every line has the same four-part grammar, and once you know it, the most valuable line in any trace — the failing one — tells you exactly what went wrong and why. Ninety percent of "why won't this work" ends at one red line in an strace, and reading it is a five-minute skill that pays out forever.

    The concept

    Every strace line is one syscall, written in the same shape:

    call name(arguments)=return valuethen, on failureERRNO (message)

    Read left to right: the call name (openat, read, connect), the arguments in parentheses (which strace helpfully decodes — turning raw numbers into readable flags and strings), an =, and the return value. A non-negative return usually means success (often it is the useful result — a new file descriptor, or a byte count). A return of -1 means failure, and then strace prints the crucial extra: the errno.

    errno ("error number") is the kernel's standard vocabulary for why a call failed. There are only about 130 of them, they are shared across every syscall, and each has a short symbolic name and a one-line meaning. strace shows both — the symbol and, in parentheses, the human sentence. This is the single most useful thing in the tool, because the symbol is a precise diagnosis:

    errnoMeansTypical cause
    ENOENTNo such file or directorythe path is wrong or the file is missing
    EACCESPermission deniedthe file exists but your credentials aren't allowed
    ECONNREFUSEDConnection refusednothing is listening on that address/port
    EMFILEToo many open filesyou hit the per-process descriptor limit (B3.12)
    EISDIRIs a directoryyou tried to read a directory as if it were a file

    The reason this matters so much operationally: an application's own error message is often vague ("could not start", "connection error"), but the errno beneath it is exact. "Could not read config" plus openat(...) = -1 ENOENT means the file is missing; the same message plus -1 EACCES means it is there but unreadable. Those are different fixes, and only the trace distinguishes them.

    Lab
    1. Make a call fail on purpose and read the failing line in full. cat a file that does not exist; the interesting line is the openat that returns -1. 
      strace cat /nonexistent 2>&1 | grep nonexistent
      openat(AT_FDCWD, "/nonexistent", O_RDONLY) = -1 ENOENT (No such file or directory)
    2. Name each part of that line out loud: call openat; arguments the directory-relative marker AT_FDCWD, the path, and the read-only flag; return -1; errno ENOENT with its message. That is the whole grammar.
    3. Change one thing — the reason — and watch the errno change while the call stays the same. Reading a file you are not allowed to read gives EACCES, not ENOENT: the file is there, you simply can't. 
      strace cat /etc/shadow 2>&1 | grep shadow
      openat(AT_FDCWD, "/etc/shadow", O_RDONLY) = -1 EACCES (Permission denied)
    4. Now a third reason — a directory where a file was expected. The openat succeeds, but the read fails with EISDIR, which is how you learn the failure is one call later than you thought. 
      strace cat /etc 2>&1 | grep -E 'EISDIR'
      read(3, 0x704774647000, 131072)         = -1 EISDIR (Is a directory)
    Verify

    Cover the errno column and, from the symbol alone, say the fix: ENOENT, EACCES, ECONNREFUSED. (Fix the path / create the file; fix permissions or run as the right user; start the thing that should be listening or fix the address.) Then explain why EISDIR appeared on read rather than on openat. (Opening a directory read-only is legal; it is reading it as a byte stream that the kernel refuses — so the error surfaces at the next call, which is a common source of "but the file opened fine" confusion.)

    Gotchas

    Not every -1 is a real problem. Programs routinely probe for optional files and expect the ENOENT — the very first thing in most traces is access("/etc/ld.so.preload", R_OK) = -1 ENOENT, which is the loader checking for a file that normally doesn't exist. A red line matters only when it is on the path the program actually needs; learn to tell a hopeful probe from a fatal failure by whether the program carries on afterwards.

    The errno symbol is stable; its number is not always what you expect and differs on some architectures. Always speak and script in symbols (ENOENT), never numbers. When a program's own logs print a bare "errno 13", errno 13 at the shell (from the moreutils package) or the table in man 3 errno translates it back to EACCES.

    The errno vocabulary is exactly what turns an opaque container failure into a one-line diagnosis. "CreateContainerError" with openat("/var/lib/.../config") = -1 ENOENT underneath is a missing mounted file; the same with EACCES is a permissions/fsGroup problem; a probe failing with connect(...) = -1 ECONNREFUSED is "the process isn't listening yet", not "the network is broken". The Kubernetes error strings are summaries; the errno is the cause.

    This is the vocabulary W6 troubleshooting runs on. Every "permission denied", "no such file", "connection refused" you will read in pod events is one of these kernel errnos, surfaced up through several layers — and knowing the original makes the summary readable.

    Docs

    man 3 errno lists every symbol with its meaning — read it once, end to end, it is short and it is the whole vocabulary. man 2 intro explains how errors are returned from syscalls. man 2 open's ERRORS section is a good model: every syscall's manpage lists exactly which errnos it can return and why, so when a line confuses you, that call's ERRORS section is the answer key.

    B3.6

    The syscalls worth knowing on sight

    Why it matters

    A trace of a real program has hundreds of lines and dozens of distinct calls, and you will never memorise all ~350 syscalls — nor should you. But about two dozen of them account for nearly everything you will ever care about, and they cluster into five plain-language groups. Learn the groups and you can skim any trace and instantly know "this is the file part, this is the network part, here is where it starts a child" without reading every line.

    The concept

    Here is the working set, grouped by what the program is trying to do. You are learning to recognise these, not to write them:

    GroupSyscallsWhat the program is doing
    Filesopenat read write close, lseek, stat/fstatopening, reading, writing, and inspecting files
    Processesexecve clone wait4 exit_groupbecoming a program, starting children, waiting, ending — the whole of B2
    Memorymmap brk mprotect munmapgetting memory from the kernel and setting its permissions
    Networksocket connect sendto recvfrom, bind listen acceptmaking and using network connections
    Everything elseioctlthe catch-all device/terminal control call — when you don't recognise it, it's often this

    A few notes that save confusion. It is openat, not open — modern glibc uses the newer call that takes a directory reference (AT_FDCWD meaning "relative to the current directory"); they do the same job. clone is the real engine behind fork (B2 told you this); clone3 is its current form. mmap shows up constantly not because programs love memory-mapping but because that is how libraries and large allocations are loaded. And ioctl is the junk drawer — a single syscall that carries thousands of device-specific commands, so when a trace has an ioctl you don't recognise, it is usually a program poking at the terminal or a device and you can skim past it.

    The payoff is pattern recognition. A program that opens files is doing file I/O; one full of socket/connect/sendto is talking to the network; a burst of clone+execve is starting a child process. You read the shape of the trace before you read the details.

    Lab
    1. Trace something that touches every group at once — curl fetching a page does files, memory, a child, and the network. Send the trace to a file so it is readable. 
      strace -f -o /tmp/curl.txt curl -s http://example.com -o /dev/null
    2. Find the process group: the first execve loading curl, and a clone3 where it spins off a helper thread for name resolution. 
      grep -E 'execve|clone' /tmp/curl.txt | head
      execve("/usr/bin/curl", ["curl", "-s", ...], ...) = 0
clone3({flags=CLONE_VM|CLONE_FS|..., ...}) = ...
    3. Find the network group — this is the actual point of curl. A socket is created, then connected to the server's IP on port 80, then bytes are sendto/recvfrom'd. 
      grep -E 'socket|connect.*htons\(80\)|sendto|recvfrom' /tmp/curl.txt | head
      socket(AF_INET, SOCK_STREAM, ...)       = 5
connect(5, {sa_family=AF_INET, sin_port=htons(80), sin_addr=inet_addr("104.20.23.154")}, 16) = 0
sendto(5, "GET / HTTP/1.1\r\n"..., ...)  = ...
recvfrom(5, "HTTP/1.1 200 OK\r\n"..., ...) = ...
    4. Find the files group — even a network tool reads plenty of files: its libraries, the CA certificates, /etc/resolv.conf to know which DNS server to ask. 
      grep -E 'openat.*(resolv.conf|ca-cert|hosts)' /tmp/curl.txt | head
      openat(AT_FDCWD, "/etc/resolv.conf", O_RDONLY|O_CLOEXEC) = ...
    Verify

    From memory, put each of these in its group: write, connect, execve, mmap, close, socket, clone. Then, given only "the trace is almost entirely openat/read/close in a loop", say what the program is doing without seeing anything else. (Walking and reading a lot of files — a backup, a search, a config scan; there is no network and no child-spawning, so it is a pure file-I/O workload.)

    Gotchas

    The exact names drift with libc and kernel versions and can surprise you: open→openat, fork→clone/clone3, stat→newfstatat/statx, select→pselect6/epoll_wait. Recognise the family, not the exact spelling — strace's -e trace= filters even understand shorthand groups like -e trace=%file, %network, %process, which is the faster way to answer "show me only the network calls".

    ioctl lines look alarming and are almost always boring — a program asking the terminal its window size, or setting a device mode. Do not spend time decoding an ioctl unless the trace clearly hangs or fails on one; treat it as the catch-all it is.

    These same groups are what a seccomp profile (B12) allows or forbids. seccomp does not ask "may this program open this file" — that is the permission system. It asks the blunter question "may this program even call openat at all", filtering at the syscall level. Kubernetes ships a RuntimeDefault seccomp profile that blocks the couple of dozen dangerous syscalls a normal workload never needs, and understanding these groups is what lets you read such a profile and predict what it will break.

    Forward pointer: when B12 shows you a seccomp profile as a list of allowed syscall names, it will be this exact vocabulary — openat, read, write, connect — and you will already know which group each belongs to.

    Docs

    man 2 syscalls is the complete list with the section-2 page for each. The five you should be able to summon a one-line purpose for on demand: man 2 openat, man 2 read, man 2 execve, man 2 clone, man 2 connect. man 1 strace under -e trace= for the %file/%network/%process shorthand groups.

    B3.7

    The file descriptor: one handle for everything

    Why it matters

    You keep seeing small integers in traces — openat(...) = 3, then read(3, ...), then close(3). That integer is a file descriptor, and it is one of the most elegant ideas in Unix: the same little number, and the same three calls (read, write, close), work on a file, a pipe, a network socket, a terminal, or a device. Learn to follow one descriptor through a trace and you can read the plot of almost any program: it opens some things, does read/write on their numbers, and closes them.

    The concept

    A file descriptor (fd) is an index into a per-process table the kernel keeps. When a program opens something, the kernel does the real work — finds the file, checks permissions, sets up its internal bookkeeping — and hands back the smallest unused integer as a ticket. From then on the program uses that number; the kernel looks up what it really refers to. The program never touches the underlying object directly, only its number, which is the boundary from B3.1 in miniature.

    The beautiful part is uniformity. The kernel arranges that files, pipes, sockets, terminals and devices all present the same read/write interface, so a program can write(fd, ...) without caring whether fd is a file on disk, a pipe to another process, or a TCP connection to the other side of the world. This is why Unix tools compose: grep does not know whether its input is a file, a pipe, or your keyboard — it just reads fd 0.

    Which is the other half: 0, 1, 2 are already open when your program starts — standard input, standard output, standard error, the three streams from B1.5. And because the kernel always hands out the lowest free number, the very first thing you open is almost always fd 3, the next fd 4, and so on. That "lowest free" rule is not trivia — it is the exact mechanism behind shell redirection. When you write command > file, the shell closes fd 1 and then opens file; since 1 is now the lowest free number, the file lands in slot 1, and the program writes to the file while believing it is writing to standard output. It never knows anything changed.

    fd 0=stdin·fd 1=stdout·fd 2=stderr·fd 3+=whatever you open next
    Lab
    1. Follow one descriptor from birth to death. Trace cat reading a real file and watch a single fd's whole life — openat returns it, read uses it, close ends it. 
      echo hello > /tmp/f
strace cat /tmp/f 2>&1 | grep -E '/tmp/f|read\(3|close\(3'
      openat(AT_FDCWD, "/tmp/f", O_RDONLY) = 3
read(3, "hello\n", 131072)          = 6
read(3, "", 131072)                 = 0
close(3)                            = 0
    2. See the three inherited descriptors on a live process. Start a sleep, look at its fd directory — 0, 1, 2 all point at your terminal, inherited from the shell (this is B2.1, from the fd angle). 
      sleep 300 & ls -l /proc/$!/fd
      lrwx------ 1 ubuntu ubuntu 64 Jul 24 10:02 0 -> /dev/pts/0
lrwx------ 1 ubuntu ubuntu 64 Jul 24 10:02 1 -> /dev/pts/0
lrwx------ 1 ubuntu ubuntu 64 Jul 24 10:02 2 -> /dev/pts/0
    3. Watch the same fd number stand for utterly different things. A pipe end, a socket, and a file all show up in /proc/PID/fd as fds — and all accept read/write. Here fd 3 of a background nc is a socket, not a file. 
      kill %1
{ sleep 60 | cat >/dev/null; } &

    Verify

    Explain why grep pattern file and cat file | grep pattern both work even though grep is reading from different kinds of source. (In the first, grep opens the file as fd 3; in the second it reads fd 0, which the shell wired to a pipe — but read on a descriptor is identical either way, so grep needs no special case.) Then predict which fd number the first openat in a fresh program returns, and why. (3 — because 0, 1, 2 are already taken and the kernel hands out the lowest free number.)

    Gotchas

    File descriptors are a finite, per-process resource, and leaking them is a classic bug: a program that opens things in a loop and forgets to close them climbs toward its limit and then every new openat fails with EMFILE (B3.12). The symptom looks like "it works for an hour then can't open anything" — and ls /proc/PID/fd | wc -l climbing without bound is the proof. The fix is always in the program, not the limit.

    Descriptors are inherited across fork and survive exec unless marked close-on-exec (the O_CLOEXEC flag you keep seeing, and FD_CLOEXEC). This is deliberate — it is how a shell hands a pipe to a child — but it is also how a descriptor accidentally leaks into a program that should never have had it. When a child mysteriously "already has" a connection open, an un-CLOEXEC'd fd is usually why.

    The uniform read/write-on-a-number model is why a container can log to "stdout" without knowing or caring what stdout is wired to. Inside the container, the app writes fd 1; the container runtime has arranged that fd 1 is a pipe it is reading; the kubelet reads that pipe and writes the bytes to a file under /var/log/pods; kubectl logs reads that file. Four different consumers, one unbroken chain of read/write on descriptors — the app is oblivious to all of it, exactly as grep is oblivious to whether its input is a file or a pipe.

    The fd-exhaustion failure mode also scales up: nodes set fs.file-max and per-process limits precisely because leaked descriptors on one busy pod could otherwise starve the whole node — the sysctl side of this is B3.10 and the limit side is B3.12.

    Docs

    man 2 open for how a descriptor is created and the flags (O_RDONLY, O_CLOEXEC). man 5 proc under /proc/[pid]/fd for the live view you used. man 2 dup2 for the redirection primitive underneath > and 2>&1. man 8 lsof ("list open files") is the friendlier, cross-process way to ask "who has this file/socket open" — it is entirely a reader of /proc/*/fd.

    B3.8

    Make strace practical: -c and -f

    Why it matters

    A full line-by-line trace of a real program is thousands of lines and overwhelming. Two flags turn strace from a firehose into a diagnostic instrument: -c collapses the whole run into a summary table — which calls, how many, how long, how many errors — and -f makes it follow child processes so that tracing anything realistic actually shows you something. Knowing just these two is the difference between strace being a curiosity and being your first move on a slow or stuck program.

    The concept

    -c (count) suppresses the per-line output and instead prints one summary table when the program finishes: each syscall the program made, the total time spent in it, the number of calls, and — the column to look at first — the number that returned an error. This answers the two questions you most often have about a misbehaving program without reading a single trace line: what is it spending its time doing, and what is failing. A program that is slow because it makes a million reads, or failing because it makes ten thousand openats that all return ENOENT, confesses immediately in this table.

    -f (follow forks) tells strace to also trace any process the target starts. This matters far more than it sounds, because a great deal of software is a small parent that immediately forks the real work into children — a shell script, a server that forks per connection, anything that shells out. Without -f, you trace the parent, see it clone away the interesting part, and then watch it do nothing while all the action happens in a child you're not following. With -f, each line is prefixed with the PID it came from ([pid 1234]), so you can tell the processes apart.

    The two combine: strace -f -c gives you a summary that includes the children, which is usually what you want for "why is this whole command slow". Add -o file to any of them when the output is large or the program's own output would otherwise get tangled in.

    Lab
    1. Get a summary of a light command. -c replaces the scroll with a table sorted by time. Note the errors column — a couple of expected access probes fail, which is normal. 
      strace -c ls / >/dev/null
      % time     seconds  usecs/call     calls    errors syscall
------ ----------- ----------- --------- --------- ----------------

  5.92    0.000090          12         7           openat
  5.20    0.000079           8         9           close
  4.34    0.000066           8         8           fstat
  1.51    0.000023          11         2         2 access
    2. Make the workload heavier and watch the shape change. A recursive listing does far more getdents64 (directory reads) and openat/newfstatat — the table now points straight at "this is a lot of directory walking". 
      strace -c ls -R /etc >/dev/null
      % time     seconds  usecs/call     calls    errors syscall

 24.1     0.003     6              520           getdents64

    3. See why -f is not optional on a script. First trace a shell one-liner without -f — you see the shell start a child and then almost nothing, because the real work moved into a process you are not following. 
      strace -c sh -c 'ls / | wc -l' 2>&1 | tail -4

... (the ls and wc did their work invisibly)
    4. Now add -f and get the whole story — the counts now include the ls and the wc children, and each line (in non--c mode) would carry its own [pid N]. 
      strace -f -c sh -c 'ls / | wc -l' 2>&1 | grep -E 'getdents|execve|calls'
      % time     seconds  usecs/call     calls    errors syscall

  ...                                2           getdents64
    Verify

    Given a strace -c table where openat shows 40,000 calls and 39,000 errors, say what the program is almost certainly doing and where to look. (Searching a long list of paths for a file that mostly isn't there — a PATH or plugin/library search gone wrong; the 39,000 ENOENTs are the probes that missed, and the fix is the search list, not the file.) Then explain in one sentence why tracing a shell script without -f is nearly useless.

    Gotchas

    The %time column is time spent in the kernel for that call, not wall-clock time, and it is badly distorted by strace's own overhead — so use the table to compare calls against each other ("read dominates"), never as a true benchmark of the program. For real timing, run the program without strace.

    -f can produce a torrent on a program that forks heavily (a build, a busy server). Combine it with -e trace= to narrow the calls (strace -f -e trace=openat,connect …) or -c to collapse to counts, and always prefer -o file so the flood lands somewhere you can search rather than scrolling past.

    -f is essential in container-land precisely because a container's entrypoint is so often a wrapper — a shell script, an exec chain, an init that forks the real server. Trace the PID 1 of a pod without -f and you watch the wrapper do nothing while the real process runs unobserved. And strace -c -f -p <pid> attached to a running, slow container is one of the fastest ways to answer "what is it actually doing" — a wall of futex means lock contention, a wall of read/write means I/O, a wall of connect with errors means it is failing to reach a dependency.

    This is a direct tool for W6: when a pod is "running but doing nothing useful", attaching strace -f -c to its process turns a mystery into a category in about ten seconds.

    Docs

    man 1 strace under -c, -f/-ff, -e trace=, -p and -o — these five options cover the vast majority of real use. For the deeper "trace only failing calls" trick, -e trace= combined with -Z (fault filtering) is worth a look once the basics are reflex. man 2 clone to remember why -f is what stitches a child back into the story.

    Kernel interfaces

    B3.9

    /proc is an API, not a folder

    Why it matters

    You have already read a dozen files under /proc in this track without stopping to ask what /proc is. It is time to name it, because the answer reframes a huge amount of Linux: /proc is not storage. Nothing there is on any disk. Every file is generated by the kernel at the instant you read it, out of the kernel's own live state. It is a filesystem-shaped API — a way to ask the kernel questions using nothing but cat and ls. Once you see it that way, you stop needing special tools, because the tools were only ever reading these files for you.

    The concept

    /proc is a virtual filesystem (type proc) — it looks and behaves like directories and files, so every tool that reads files can read it, but there is no disk behind it. When you cat /proc/meminfo, the kernel runs code that gathers the current memory figures and hands them back formatted as text, right then. Read it again a second later and the numbers differ, because it was recomputed. The "file" is really a function call wearing a filename.

    The tell is the size. Every /proc file reports zero bytes to ls -l, because there is no stored content to measure — the content does not exist until you read it. That single detail is the fastest way to prove to yourself that /proc is an API: a real 4 KB file cannot be both zero bytes and full of text, but a generated one can.

    /proc has two neighbourhoods. The system-wide files sit at the top: /proc/cpuinfo (the CPUs), /proc/meminfo (memory), /proc/mounts (what is mounted), /proc/net/tcp (open TCP connections, in hex), /proc/loadavg (load). The per-process files live under a numbered directory for each PID: /proc/PID/status, /proc/PID/fd/, /proc/PID/maps, /proc/PID/environ — the ones you x-rayed a process with in B2.4.

    The reframing that pays off: the standard tools have no special powers. ps reads /proc/*/stat. free reads /proc/meminfo. ss and netstat read /proc/net/*. uptime reads /proc/loadavg. Anything they can tell you, you can read directly — which stops being an academic point the moment you are inside a stripped-down container image that ships none of those tools but still has /proc, because the kernel mounts it regardless.

    you: cat /proc/meminfo→kernel computes current memory now→text, 0 bytes on disk
    Lab
    1. Prove the zero-byte tell. These files are full of text yet report a size of 0, because the text is generated on read, not stored. 
      ls -l /proc/cpuinfo /proc/meminfo /proc/loadavg

-r--r--r-- 1 root root 0 Jul 24 10:10 /proc/meminfo
-r--r--r-- 1 root root 0 Jul 24 10:10 /proc/loadavg

      grep -E 'MemTotal|MemAvailable' /proc/meminfo

      cat /proc/loadavg; sleep 1; cat /proc/loadavg
      0.14 0.09 0.03 1/220 91002
0.13 0.09 0.03 2/220 91010

      grep -E 'Name|State|VmRSS' /proc/self/status

    Explain, in one sentence, why a /proc file can be 0 bytes and still print pages of text. Then answer: you are in a distroless container with no ps, no free, no top, and a process is eating memory — how do you find the culprit's memory use? (Read /proc/*/status for VmRSS, or /proc/*/smaps_rollup, directly with the shell — the kernel mounts /proc even in an image with no tools, so the data is always reachable.)

    Gotchas

    /proc/PID/environ and /proc/PID/maps are readable only by the process's owner and root — as an ordinary user, cat /proc/1/environ gives "Permission denied", which is a credentials check, not an empty or missing file. Do not conclude a /proc file is empty until you have tried it with sudo; the boundary from B3.1 applies here too.

    Some /proc "files" are formatted for the kernel's convenience, not yours — /proc/net/tcp lists addresses and ports in hex and little-endian, which is why ss exists to decode them. Reading the raw file is a valid fallback, but reach for the tool when the format is hostile; the point is that you can read the raw file, not that it is always pleasant.

    /proc is how nearly every metric you have ever seen about a pod is collected. cAdvisor and the kubelet read /proc and the cgroup files to produce CPU and memory numbers; node exporters read /proc for everything they report. When Kubernetes says a pod is using 300 MiB, that number originated as text in a /proc (or cgroup) file that the kubelet read — the same file you can read yourself. And the "no tools in the image" scenario is why kubectl debug and ephemeral containers exist: attach a container that does have tools into the target's namespaces, and read the same /proc the broken app can see.

    This connects forward to W6.9 — reading resource pressure and to B7's cgroup files: metrics are not magic, they are files, and knowing which file a number comes from is what lets you trust or challenge it.

    Docs

    man 5 proc is the reference for the whole tree — on Ubuntu 24.04 it has been split, so also try man 5 proc_meminfo, man 5 proc_pid_status, man 5 proc_loadavg for individual files. man 5 proc_pid_stat documents the exact fields ps parses. The habit to build: when a tool's number confuses you, find the /proc file it came from in man 5 proc and read the source yourself.

    B3.10

    /sys and sysctl: changing the kernel, not just reading it

    Why it matters

    /proc mostly lets you read the kernel. Its siblings /sys and the /proc/sys tree let you change it — tuning knobs you can turn on a running system to alter how the kernel routes packets, manages memory, or handles files. Two of these knobs are mandatory on every Kubernetes node, and misunderstanding where the setting lives (memory vs disk) is behind a whole category of "it worked until we rebooted" incidents. This lesson is where you learn to turn a knob, and to make the turn stick.

    The concept

    /sys (the sysfs filesystem) is like /proc — generated, zero-byte, an API-as-files — but focused on devices and the kernel's object model: network interfaces under /sys/class/net/, block devices, driver settings. Many of its files are writable, and writing to one changes the device or subsystem live.

    sysctl is the friendly front-end to one specific writable tree: /proc/sys/, which holds hundreds of kernel tunables named with dots (net.ipv4.ip_forward) that map directly onto paths (/proc/sys/net/ipv4/ip_forward). The single most important thing to internalise: sysctl -w net.ipv4.ip_forward=1 and echo 1 > /proc/sys/net/ipv4/ip_forward are the exact same act. sysctl is just a nicer way to write to a file under /proc/sys. Reading is symmetric: sysctl net.ipv4.ip_forward equals cat /proc/sys/net/ipv4/ip_forward.

    Now the part that causes real incidents — live versus persistent. Because these are generated files, a value you set with sysctl -w (or echo) lives in kernel memory and vanishes on reboot. To make a setting survive, you write it to a real file on disk that the system re-applies at boot: drop a line like net.ipv4.ip_forward = 1 into a file in /etc/sysctl.d/. At boot, systemd-sysctl reads those files and re-does the sysctl -w for you. Set it live to fix the problem now; write the file so it is still fixed after a reboot. Skipping the second step is the "worked for weeks, broke after a power cut" bug.

    sysctl -w x=1=echo 1 > /proc/sys/x= live, lost on reboot/etc/sysctl.d/*.conf= re-applied at boot
    Lab
    1. Read a tunable two equivalent ways. On a fresh sandbox IP forwarding is off (0); the sysctl name and the /proc/sys path are the same knob. 
      sysctl net.ipv4.ip_forward
cat /proc/sys/net/ipv4/ip_forward
      net.ipv4.ip_forward = 0
0
    2. Turn it on live, and confirm the change is visible through the file too — proving the two interfaces are one thing. 
      sudo sysctl -w net.ipv4.ip_forward=1
cat /proc/sys/net/ipv4/ip_forward
      net.ipv4.ip_forward = 1
1
    3. Make it persist. Writing the file does not change the live value by itself — --system re-reads all the drop-ins and applies them, which is also exactly what happens at boot. 
      echo 'net.ipv4.ip_forward = 1' | sudo tee /etc/sysctl.d/99-forward.conf
sudo sysctl --system 2>&1 | grep forward
      * Applying /etc/sysctl.d/99-forward.conf ...
net.ipv4.ip_forward = 1
    4. Peek at a /sys device file to see the same idea for hardware — here, whether a network interface is up. (Read-only-ish; don't write device files casually.) 
      cat /sys/class/net/eth0/operstate
      up
    Verify

    Explain why a colleague's sysctl -w fix "stopped working after the node rebooted", and give the two-part correct procedure. (The -w value lives only in kernel memory and is lost on reboot; the fix is to also write it to /etc/sysctl.d/ so systemd-sysctl re-applies it at boot — set live for now, persist for later.) Then state what echo 1 > /proc/sys/net/ipv4/ip_forward is equivalent to.

    Gotchas

    Writing a /proc/sys file needs root, and sudo echo 1 > /proc/sys/... does not work — the redirection is done by your unprivileged shell before sudo runs, so it fails with "Permission denied". Use sudo sysctl -w, or echo 1 | sudo tee /proc/sys/...; the tee is the part running as root. This trips up almost everyone once.

    Drop-in files are applied in lexical order, so a later file wins — 99-mine.conf overrides 10-default.conf. And not every tunable can be set live; a few require the subsystem to be reloaded or only read at boot. When sysctl -w silently doesn't take, check whether the value is one the kernel only honours at initialisation.

    Two sysctls are load-bearing on every node, and the "Under the hood" note in the tracker named them for a reason. net.ipv4.ip_forward=1 is what lets a node route traffic between pods at all — without it the node is a building whose internal phone lines don't connect, and B9 builds exactly that routing by hand. And fs.inotify.max_user_instances/max_user_watches default low enough that a busy node runs out of file watches, after which the kubelet and controllers start failing in ways that look like anything but a sysctl. kubeadm sets the first for you (W0.6); the second you raise yourself when a node gets crowded.

    Kubernetes even exposes a curated subset as pod-level securityContext.sysctls, so a workload can request a "safe" namespaced tunable — but the node-level ones like ip_forward are set on the host, exactly as you just did.

    Docs

    man 8 sysctl for -w, -a (dump everything), -p/--system (apply files). man 5 sysctl.conf and man 5 sysctl.d for the persistence files and their ordering. man 7 sysfs for the /sys layout.     B3.11

    The kernel's own log: dmesg and the ring buffer

    Why it matters

    There is a log the kernel writes to that is not journalctl for a service and not any application's log file — it is the kernel talking about itself: hardware events, out-of-memory kills, filesystem errors, dropped packets. When something dies with no explanation in its own logs, this is very often where the explanation actually is. Learning to check it reflexively, as a reflex and not a last resort, is one of the highest-value habits in this whole track.

    The concept

    The kernel writes its messages into a fixed-size ring buffer in memory — a circular log that holds the most recent entries and overwrites the oldest when it fills. It has to be in-memory and self-contained because the kernel needs to report problems that happen before any disk or logging service is available (early boot), and problems with the disk or logging service themselves. dmesg ("display message") prints this buffer; journalctl -k shows the same kernel messages through systemd's journal, with the advantage of persistence across reboots (the raw ring buffer is cleared on reboot).

    Always use dmesg -T. By default the buffer timestamps entries as "seconds since boot" ([12345.678]), which is nearly useless for correlating with a real incident; -T converts them to human clock time so you can line a kernel event up with "the app fell over at 03:14".

    The reason this log is irreplaceable: it contains verdicts that exist nowhere else. The clearest example is the OOM killer (B7). When the machine runs out of memory and the kernel kills a process to survive, the victim's own logs simply stop mid-sentence — the process was shot, it did not get to say goodbye. The only record of what happened — which process, how much memory it was using, which cgroup it was in, why it was chosen — is a block of lines in dmesg. "The app died silently" almost always means "nobody read the ring buffer". The same is true of disk I/O errors, filesystem corruption, and a network card resetting: the application sees a vague failure, and the kernel's log has the specific cause.

    Lab
    1. Read the kernel log with human timestamps. On most systems reading the buffer now requires root (a hardening default, kernel.dmesg_restrict). 
      sudo dmesg -T | tail -5
      [Wed Jul 24 09:58:01 2026] eth0: Link is Up - 1Gbps/Full
[Wed Jul 24 09:58:02 2026] IPv6: ADDRCONF(NETDEV_CHANGE): eth0: link becomes ready
    2. See the same messages via the journal, which survives reboots. -k filters to kernel messages only. 
      journalctl -k -n 5 --no-pager
      Jul 24 09:58:01 sandbox kernel: eth0: Link is Up - 1Gbps/Full
    3. Provoke a real kernel verdict and read it. Run a small program that grabs memory until the OOM killer stops it, then find the kill in the log — this is the record that exists only here. (Snapshot first; this deliberately stresses memory.) 
      tail /dev/zero &   # tail of an infinite file grows without bound
sleep 5; sudo dmesg -T | grep -iE 'killed process|out of memory' | tail -3
      [Wed Jul 24 10:15:22 2026] Out of memory: Killed process 91234 (tail) total-vm:...kB
    4. Confirm the reflex pays off: the killed process left nothing useful of its own, but the kernel logged exactly what happened and to whom. That asymmetry is the whole lesson.
    Verify

    State why the kernel log must live in a memory ring buffer rather than a normal file. (It has to work before disks/logging are up and when those very things are failing — a log on the disk is no use when the disk is the problem.) Then answer: a container vanished with exit code 137 and its application log ends mid-request with no error — where is the actual cause, and what will it tell you? (In dmesg/journalctl -k: an OOM-kill line naming the process, its memory use, and its cgroup — 137 is 128+9, SIGKILL from outside, and the kernel is the outside.)

    Gotchas

    The raw ring buffer is volatile and small — it clears on reboot and overwrites its oldest entries under a flood of messages, so a real event from an hour ago may already be gone. For anything you need to keep, rely on journalctl -k with persistent journaling enabled, not dmesg. After a crash-and-reboot, the pre-reboot dmesg is gone; journalctl -k -b -1 ("previous boot") is how you read what the kernel said before it went down.

    On Kubernetes nodes and inside containers, reading dmesg is often restricted (kernel.dmesg_restrict=1) or the container has no access to the host's buffer at all — the ring buffer is the host's, shared like the kernel itself. You typically read it on the node, not from inside the pod, which is consistent with everything else about the shared kernel.

    This is the single most under-used node-troubleshooting tool. A pod OOM-killed shows Reason: OOMKilled in kubectl describe, but the node's dmesg has the full verdict — which process in the pod, exact memory, the cgroup path (which maps back to the pod). A node with mysterious pod evictions or I/O stalls very often has the cause sitting in dmesg: a failing disk, a filesystem going read-only, a network driver resetting. When a node is "unhealthy" for no reason the Kubernetes objects explain, dmesg -T on the node is the next command, every time.

    This is a core move in W6.1 node triage and connects directly to B7's OOM lesson: the exit code tells you that the kernel killed it; the ring buffer tells you why.

    Docs

    man 1 dmesg for -T, -w (follow, like tail -f), -l (filter by level), and -H (human/pager). man 1 journalctl for -k, -b/-b -1 (this/previous boot), and -p (priority). man 5 proc_sys_kernel under dmesg_restrict for why an ordinary user may be refused. For the OOM message format specifically, the kernel's Documentation/admin-guide/mm/ notes.

    B3.12

    Per-process limits: ulimit, soft vs hard, and EMFILE

    Why it matters

    The kernel does not let a single process consume everything. Each one runs under a set of resource limits — a cap on how many files it may have open, how many processes its user may spawn, how big a core dump may be, and more. Two of these limits cause real, confusing outages, and the failure they produce (EMFILE, "too many open files") looks like a bug in your code when it is actually a limit you could have raised. Meeting these on a machine you control, and deliberately triggering the failure, means you will recognise it instantly when it happens for real.

    The concept

    Every process carries a table of rlimits (resource limits), inherited from its parent. You inspect them with the shell builtin ulimit -a. Each limit has two values:

    • The soft limit — the value actually enforced right now. A process may lower it, or raise it up to the hard limit, on itself.
    • The hard limit — the ceiling on the soft limit. Only a privileged process may raise the hard limit; an ordinary one may only lower it (a one-way ratchet, for safety).

    Two limits bite in practice. -n (open files) caps how many file descriptors the process may hold at once — and since sockets are descriptors (B3.7), a busy network server hits this as connection count, not "file" count. When a process exceeds it, the next openat or socket fails with EMFILE ("too many open files"). -u (processes) caps how many processes the user may run; exceeding it makes fork fail with EAGAIN, often seen as "resource temporarily unavailable" or a fork bomb hitting its wall.

    The trap is that EMFILE reads like an application bug ("my program can't open files!") when the file is fine and the count is the problem. Two causes: either the program is leaking descriptors (opening without closing — B3.7's gotcha), which climbs until it hits the limit; or the limit is simply too low for a legitimately busy server. The fix for the first is in the code; the fix for the second is to raise the limit. Telling them apart is watching whether ls /proc/PID/fd | wc -l climbs without bound (leak) or sits near a low ceiling (limit too low).

    Lab
    1. Read your current limits. Note the soft (first) and hard (second) columns, and the two that matter — open files and max user processes. 
      ulimit -Sn; ulimit -Hn

      1024

    2. Lower the soft limit on yourself — allowed, because you may reduce a soft limit freely. This shell now runs under a tiny file cap. 
      bash -c 'ulimit -Sn 5; ulimit -Sn; cat /etc/hostname'

      bash -c 'ulimit -Sn 4; ls -R /usr/share 2>&1 | head -2'
      ls: cannot open directory '/usr/share/...': Too many open files
    4. Confirm the ratchet: an ordinary shell may lower the hard limit but not raise it back. This fails, which is the safety property, not a bug. 
      bash -c 'ulimit -Hn 100; ulimit -Hn 200' 2>&1
      bash: ulimit: open files: cannot modify limit: Operation not permitted
    Verify

    Distinguish the soft and hard limit in one sentence each, and say who may raise which. Then diagnose: a server has run fine for weeks and now every new connection fails with "too many open files", and ls /proc/PID/fd | wc -l shows a number that has been climbing all week. Leak or limit-too-low, and what is the fix? (A descriptor leak — the count climbing without bound means the program opens sockets/files and never closes them; the fix is in the code, and raising -n only delays the crash. A steady count sitting at the ceiling would have been the other diagnosis.)

    Gotchas

    ulimit is a shell builtin and changes apply only to that shell and its children — so setting a limit in one terminal does nothing for a service started by systemd. For a daemon, the limit comes from its unit file's LimitNOFILE= (B11), not from any interactive ulimit; editing your .bashrc will never affect it. This is the most common reason "I raised the limit and nothing changed".

    The soft/hard ratchet means a process that has lowered its own hard limit cannot undo it, even to a value it started above. And the defaults have shifted over time — modern systemd sets a very high -n hard limit, so the classic 1024 you may have memorised is often no longer the ceiling, only the soft default. Always check the actual values with ulimit -Sn/-Hn rather than assuming.

    "Too many open files" is a genuinely common Kubernetes incident, and it has both a per-process and a node-wide face. Per process: a container's limit comes from the container runtime's configuration (which sets the equivalent of ulimit -n for every container's PID 1), so a leaky app hits EMFILE exactly as in the lab. Node-wide: too many pods with too many descriptors can exhaust the node's fs.file-max (the B3.10 sysctl) or fs.inotify watches, taking out the kubelet itself. The ulimit -u equivalent shows up as a node hitting its process/PID limit and refusing to start new containers.

    This ties B3 together: the limit is a per-process rule (this lesson), the node-wide ceiling is a sysctl (B3.10), the failure is an errno you can read in an strace (B3.5), and the evidence is a count in /proc/PID/fd (B3.9). The mini project and drill next put all four to work at once.

    Docs

    man 1 bash under ulimit (it is a shell builtin, so it lives in the shell's manpage, not its own). man 2 getrlimit for the underlying RLIMIT_NOFILE/RLIMIT_NPROC and the soft/hard rules. man 5 limits.conf for setting limits at login via PAM, and man 5 systemd.exec under LimitNOFILE= for the way that actually matters to services and containers. man 3 errno for EMFILE vs the system-wide ENFILE.

    B3.13

    Annotate one curl, and write "hello" three ways

    Why it matters

    Every idea in this module meets in one trace. A single curl http://example.com loads a program, resolves a name, opens a socket, connects, writes a request, reads a response, and exits — the file, process, memory and network groups all in one story. If you can take that raw trace and narrate it phase by phase, you can read any program's trace, because they are all the same handful of phases in different proportions. The second half — the same "hello" written three ways — makes the boundary vivid: three wildly different programs that all end in the identical single write syscall.

    The concept

    A network fetch decomposes into seven phases, and every syscall in the trace belongs to exactly one:

    PhaseTell-tale syscallsWhat is happening
    Loadexecve, then the libc openat/mmap burstthe kernel loads curl and its libraries

    Readrecvfrom/read of HTTP/1.1 200 OK…receiving the response

What differs enormously is everything around it: the C program drags in almost nothing (~30 calls total), the shell more (~117), Python a whole interpreter (~360, most of them openats loading modules). Same destination, three very different journeys to the doorway — a concrete picture of "the syscall is the narrow part; the runtime above it is where the weight is".

    B3.14

    Descriptor exhaustion, and the errno ladder

    Why it matters

    Two skills, drilled until they are reflexes. First: recognise file-descriptor exhaustion — the "worked for a while, then couldn't open anything" failure — and prove it from evidence rather than guessing. Second: name an error from its errno alone, before you reach for strace, because in a real incident the errno is often all you have and the four common ones point at four completely different fixes. Get these two into your fingers and a whole class of production mysteries becomes a thirty-second diagnosis.

    The concept

    You met the pieces already: descriptors are finite per process (B3.7), the cap is ulimit -n (B3.12), and exceeding it yields EMFILE (B3.5). This drill wires them together under time pressure. Then the errno ladder — four failures that all look like "can't open the file" to a user but are four different truths:

    errnoSymptom in plain wordsThe one-line fix
    ENOENTthe path isn't there at allfix the path, or create the file
    EACCESit's there, you're not allowedfix permissions / run as the right user
    EPERMthe operation is forbidden regardless of file permsyou need a capability/privilege (B4), not a mode bit
    EISDIRit's a directory, you treated it as a filepoint at a file, not the folder

    The EACCES vs EPERM distinction is the subtle, valuable one. EACCES is "the file's permission bits say no" — fixable with chmod/chown or the right user. EPERM is "this action is privileged no matter what the file allows" — binding to port 80, changing another user's file ownership, loading a kernel module — and no amount of chmod helps; you need a capability, which is the entire subject of B4. Confusing them sends people down the wrong fix for hours.

    Lab

    Snapshot first. Work each symptom, state a one-sentence cause backed by a command that proves it, then read the reveal.

    Symptom 1 — a program that opens fine, then suddenly can't
    1. Reproduce exhaustion deterministically by capping descriptors low, then asking for more than the cap. 
      bash -c 'ulimit -Sn 6; for i in $(seq 1 10); do exec {fd}</etc/hostname || { echo "failed at open #$i"; break; }; done'
      failed at open #4: Too many open files
    2. Prove it is the limit, not the file. The file opens fine on its own; only under the low cap, after a few opens, does it fail — and strace names the errno. 
      bash -c 'ulimit -Sn 4; strace -e trace=openat cat /etc/hostname /etc/hostname /etc/hostname 2>&1 | grep EMFILE | head -1'
      openat(AT_FDCWD, "/etc/hostname", O_RDONLY) = -1 EMFILE (Too many open files)
    Cause & fix

    Descriptor exhaustion. The process hit its RLIMIT_NOFILE soft cap; every further openat/socket returns EMFILE. In real life, decide between the two causes by watching ls /proc/PID/fd | wc -l: climbing without bound is a leak (fix the code — it opens without closing); pinned at a low ceiling on a busy server is a limit too low (raise -n, via the systemd unit or runtime config, not interactive ulimit). Raising the limit on a leak only delays the crash.

    Symptom 2 — four "can't open" errors, four different fixes
    1. Generate all four errnos and read each one. Do not fix by trial and error — name the errno, then apply its specific fix. 
      strace cat /no/such/path 2>&1 | grep -m1 'no/such'      # ENOENT
strace cat /etc/shadow   2>&1 | grep -m1 shadow         # EACCES
strace cat /etc          2>&1 | grep -m1 EISDIR         # EISDIR
strace -f nc -l 80       2>&1 | grep -m1 EACCES          # EACCES/EPERM binding low port as non-root
      openat(..., "/no/such/path", O_RDONLY) = -1 ENOENT (No such file or directory)
openat(..., "/etc/shadow", O_RDONLY)   = -1 EACCES (Permission denied)
read(3, ...)                           = -1 EISDIR (Is a directory)
bind(3, {..., sin_port=htons(80)}, ...) = -1 EACCES (Permission denied)
    2. Confirm the port-80 case is about a capability, not the file. As root — or with the one capability that grants it — the same bind succeeds, proving no file permission was ever involved. (B4 is the whole story.) 
      sudo nc -l 80 & sleep 1; sudo kill %1 2>/dev/null; echo "root could bind :80"
      root could bind :80
    The ladder, named

    ENOENT → the path is wrong or missing (fix the path). EACCES on a file → the permission bits deny you (fix perms / user). EISDIR → you handed a directory to something expecting a file. The low-port bind failing for a non-root user is the EACCES/EPERM family that is not a file-permission problem at all — it is a missing capability (CAP_NET_BIND_SERVICE), which is why chmod would never fix it and B4 exists.

    Gotchas

    The tempting wrong move in Symptom 1 is to raise the limit and move on. If the count is climbing, you have hidden a leak that will exhaust any limit eventually — the fix is the program. And in Symptom 2, the trap is treating every "permission denied" as a chmod problem; when the errno is really EPERM on a privileged operation, file permissions are a dead end and the answer is a capability. Reading the exact errno, not the English message, is what keeps you on the right ladder.

    Both drills reward the same discipline as B2's: work from evidence to cause, not from symptom to reflex. The errno is the evidence; the fix follows from which errno, not from how loud the failure is.

    Docs

    man 3 errno for the full ladder — reread the entries for ENOENT, EACCES, EPERM, EISDIR, EMFILE. man 2 open and man 2 bind under ERRORS for exactly which errno each returns and when. man 7 capabilities as the forward pointer for the EPERM/CAP_NET_BIND_SERVICE case — the doorway into B4.     B3 recall quiz

    Answer out loud before revealing. If you cannot see the boundary in your head, none of the tooling on top of it will make sense.

    1. Name the five things the kernel exclusively owns, and what a program can do without asking.
      Answer

      Physical hardware, the memory map of every process, CPU scheduling, the filesystems, and the network stack. On its own a program can only do arithmetic and manipulate its own memory; everything else is a request across the boundary.

    2. What is a system call, and how is it different from a function call to a library?
      Answer

      A controlled switch from user mode to kernel mode via a special CPU instruction that traps to a fixed kernel entry point. A library call stays in user mode and jumps anywhere in your own program; a syscall crosses the privilege boundary and lands only where the kernel allows. That is why it costs more and why it is safe.

    3. Why is "too many small reads" a real performance bug?
      Answer

      Each read is a boundary crossing with fixed overhead. One read of a megabyte is one crossing; a million one-byte reads is a million crossings, and the crossings — not the data movement — dominate. Buffering exists to collapse many small would-be syscalls into a few big ones.

    4. You called printf a hundred times but strace shows one write. Explain.
      Answer

      printf is a libc function running in your process; it buffers text in user space and only crosses into the kernel with a write syscall when the buffer flushes (full, newline to a terminal, or clean exit). A hundred printfs can accumulate into a single write. It is also why output can be lost on SIGKILL — it never reached the kernel.

    5. Read this line: connect(5, {sa_family=AF_INET, sin_port=htons(443), ...}) = -1 ECONNREFUSED. What happened and what do you check?
      Answer

      The program made a socket (fd 5) and tried to connect to something on port 443, and nothing was listening there — ECONNREFUSED means the port is reachable but closed. Check that the target service is actually running and bound to that address/port; it is not a DNS or routing problem (those fail differently), it is "nobody home".

    6. What is a file descriptor, and why do the first files a program opens get numbers 3, 4, 5…?
      Answer

      A small integer indexing a per-process table of open things (files, pipes, sockets, terminals), all sharing one read/write/close interface. 0, 1, 2 are already stdin/stdout/stderr, and the kernel always hands out the lowest free number — so the next open is 3. That "lowest free" rule is also how shell redirection lands a file in slot 1.

    7. Why does ls -l /proc/meminfo show 0 bytes, and what does that tell you about /proc?
      Answer

      Because there is nothing stored — the content is generated by the kernel at the moment you read it. /proc is a filesystem-shaped API onto live kernel state, not disk storage. It is why ps/free/ss have no special powers (they just read these files) and why you can diagnose a toolless container by reading /proc directly.

    8. A colleague's sysctl -w net.ipv4.ip_forward=1 fix disappeared after a reboot. Why, and what is the complete fix?
      Answer

      sysctl -w writes a generated /proc/sys file that lives only in kernel memory and is lost on reboot. The complete procedure is two parts: set it live with -w for now, and write net.ipv4.ip_forward = 1 into /etc/sysctl.d/ so systemd-sysctl re-applies it at every boot.

    9. A container exits 137, its app log ends mid-request with no error, and kubectl describe says OOMKilled. Where is the real story, and why isn't it in the app's log?
      Answer

      In the node's kernel ring buffer — dmesg -T / journalctl -k — where the OOM killer logged which process it killed, its memory, and its cgroup. The app's log has nothing because 137 is 128+9 (SIGKILL from outside); the process was shot without a chance to flush or log. The kernel is the only witness.

    10. Distinguish EACCES from EPERM on an operation, and why the difference changes the fix.
      Answer

      EACCES means a file's permission bits denied you — fixable with chmod/chown or the right user. EPERM means the operation is privileged regardless of any file (binding port 80, chowning another user's file) — no chmod helps; you need a capability. Chasing an EPERM with file permissions wastes hours, which is why reading the exact errno matters.

    11. A program "can't open files" after running for hours; ls /proc/PID/fd | wc -l has been climbing all day. Leak or limit-too-low, and what is the fix?
      Answer

      A descriptor leak — an ever-climbing count means it opens without closing. The fix is in the code; raising ulimit -n only postpones the same EMFILE. A count pinned at a low ceiling (not climbing) would instead be a limit set too low for the workload, fixed by raising it in the service/runtime config.

    12. You have strace -c output for a slow command and one syscall dominates the calls column with tens of thousands of entries and many errors. How do you turn that into a diagnosis?
      Answer

      Read the dominant call and its error count as a category: masses of openat with ENOENT = a path/search-list problem; masses of read = tiny-I/O or a data firehose; masses of futex = lock contention; masses of connect with errors = failing to reach a dependency. The one hot call with its errno names the class of problem before you read a single detailed line.

    • Name what the kernel exclusively owns, and explain why the boundary exists and why a crashing process rarely takes the machine down.
    • Describe user mode vs kernel mode as a CPU state, and why crossing it costs — and why buffering exists to minimise crossings.
