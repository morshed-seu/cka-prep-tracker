    Week 5 — Storage: PVs, PVCs & CSI

    Storage is the smallest exam domain and the most mechanical — one control loop (binding), one call sequence (CSI), a handful of YAML shapes. This week you watch the PV controller match and bind claims, install two real dynamic provisioners, trace a volume from API object to a mount on a node's disk, and collect the exact error signatures for every way a claim can get stuck. By Friday a storage task should feel like filling in a form. Ticking done here also ticks the tracker.

    ephemeral volumes→PV / PVC binding→StorageClasses→dynamic provisioning→CSI machinery

    Volume fundamentals

    5.1

    The ephemeral volumes, cold: emptyDir, hostPath, and the config projections

    Why it matters

    Most exam volume work isn't persistent at all — it's "add a scratch volume", "mount this ConfigMap at /etc/app", "share a dir between two containers". These are two-minute tasks if the shapes are memorized, and one of them (medium: Memory) hides a genuine trap.

    The concept

    An ephemeral volume lives and dies with its pod — which is longer than a container: containers restart in place, and the volume survives every restart. Only deleting or rescheduling the pod loses the data.

    TypeWhat it isThe thing to remember
    emptyDirFresh empty dir on the node, created when the pod startsSurvives container restarts, dies with the pod. Shared by all containers that mount it — the standard sidecar handoff.
    emptyDir + medium: Memorytmpfs (RAM-backed)Every byte written counts against the container's memory limit — a full tmpfs can OOM-kill the pod.
    hostPathA directory that already exists on whatever node the pod lands onData does not follow the pod: reschedule to another node and the path is empty (or different). It's a node inspection tool, not storage.
    configMap / secretKeys projected as read-only filesUpdates propagate to the mounted files (kubelet sync, up to ~a minute) — except through a subPath mount, which never updates.
    downwardAPIPod metadata (labels, annotations, resources) as filesHow an app learns its own labels without calling the API.
    projectedSeveral of the above combined into one mount pointWhat the ServiceAccount token mount uses (you'll dissect that in 7.4).
    Lab
    1. A pod with a RAM-backed scratch dir, memory limit 128Mi: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata: {name: scratch}
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep","1d"]
    resources: {limits: {memory: "128Mi"}}
    volumeMounts: [{name: cache, mountPath: /cache}]
  volumes:
  - name: cache
    emptyDir: {medium: Memory}
EOF
    2. Prove it's tmpfs, and that tmpfs sizing follows the pod's memory limit: 
      kubectl exec scratch -- df -h /cache
      Filesystem   Size   Used  Available  Use%  Mounted on
tmpfs        128.0M  0    128.0M     0%    /cache
    3. Write a file, then kill PID 1 so the container crashes and restarts: 
      kubectl exec scratch -- sh -c 'echo survives-restarts > /cache/note'
kubectl exec scratch -- kill 1
kubectl get pod scratch          # wait for RESTARTS to hit 1
kubectl exec scratch -- cat /cache/note
      NAME      READY   STATUS    RESTARTS      AGE
scratch   1/1     Running   1 (5s ago)   2m
survives-restarts

      New container, same volume: the volume belongs to the pod (the kubelet keeps it in the pod's dir on the node — 5.2), and the fresh container gets it re-mounted.

    4. Now the trap — fill the tmpfs past the memory limit: 
      kubectl exec scratch -- sh -c 'dd if=/dev/zero of=/cache/big bs=1M count=200' ; kubectl get pod scratch
      NAME      READY   STATUS      RESTARTS   AGE
scratch   0/1     OOMKilled   1          3m

      The 200Mi file is memory, memory is over the 128Mi limit, the kernel kills the container. "My pod OOMs but the app barely allocates" — check what it writes to tmpfs.

    5. Clean up: 
      kubectl delete pod scratch
    Verify

    You saw both behaviors: the file survived a container restart (volume > container lifetime), and writing past the memory limit into medium: Memory got the pod OOMKilled.

    Gotchas

    hostPath is also a scheduling trap: the pod schedules anywhere (the path isn't a constraint), then behaves differently per node. If an exam task hands you a hostPath pod that "works on one node only", that's the whole story. And remember subPath freezes ConfigMap/Secret content at mount time — the classic "I updated the ConfigMap but the file didn't change".

    Docs search: "Volumes" — one page, every type, with YAML for each. Know its layout; you'll copy from it in the exam.

    5.2

    Where volumes actually live: /var/lib/kubelet/pods/<uid>/volumes

    Why it matters

    Every volume a pod mounts is a real directory on the node, at a predictable path. Once you've found one by hand, "storage" stops being abstract — and in week 6 this path is how you rescue data from a pod that won't start and decode kubelet mount errors.

    The concept

    The kubelet's volume manager prepares each pod's volumes under a fixed layout, then the runtime bind-mounts them into containers:

    /var/lib/kubelet/pods/<pod-uid>/volumes/kubernetes.io~<plugin>/<volume-name>/
                                      kubernetes.io~empty-dir/cache/      ← plain dir
                                      kubernetes.io~secret/token/         ← tmpfs (never touches disk)
                                      kubernetes.io~nfs/pvc-…/            ← the actual NFS mount (5.12)
                                      kubernetes.io~csi/pvc-…/mount/      ← CSI publish target (5.12)

    The pod UID comes from .metadata.uid. Because the container sees the volume through a bind mount, the directory is the same inode from both sides — a write from the pod is instantly visible on the host path and vice versa.

    Lab
    1. Run a pod with an emptyDir, pinned to node01 so you know where to look, and grab its UID: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata: {name: findme}
spec:
  nodeName: node01
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep","1d"]
    volumeMounts: [{name: cache, mountPath: /cache}]
  volumes: [{name: cache, emptyDir: {}}]
EOF
kubectl exec findme -- sh -c 'echo hello-from-pod > /cache/msg'
kubectl get pod findme -o jsonpath='{.metadata.uid}{"\n"}'
      1f8b2c3d-9a4e-4f7b-b2d1-0c5e6a7f8d90
    2. On node01, walk to it (substitute your UID): 
      UID=1f8b2c3d-9a4e-4f7b-b2d1-0c5e6a7f8d90
sudo ls /var/lib/kubelet/pods/$UID/volumes/
sudo cat /var/lib/kubelet/pods/$UID/volumes/kubernetes.io~empty-dir/cache/msg
      kubernetes.io~empty-dir  kubernetes.io~projected
hello-from-pod

      The ~projected neighbor is the ServiceAccount token mount — every pod has it.

    3. Prove the bind mount works both ways — write from the host, read from the pod: 
      echo hello-from-node | sudo tee /var/lib/kubelet/pods/$UID/volumes/kubernetes.io~empty-dir/cache/reply >/dev/null

      Back on cp:

      kubectl exec findme -- cat /cache/reply
      hello-from-node
    4. See which volumes are RAM, not disk (secrets and Memory-medium emptyDirs): 
      mount | grep "$UID" | grep tmpfs
      tmpfs on /var/lib/kubelet/pods/1f8b…/volumes/kubernetes.io~projected/kube-api-access-x7k2p type tmpfs (rw,relatime,size=…)
    5. Clean up: 
      kubectl delete pod findme
      Then confirm on node01 that /var/lib/kubelet/pods/$UID is gone — the kubelet garbage-collects the whole pod dir.
    Verify

    The file you wrote inside the pod was readable at the host path, and the file you wrote at the host path appeared inside the pod — same directory, two views.

    Gotchas

    emptyDir data lives on the node's disk (under /var/lib/kubelet), so it counts against the node's ephemeral storage and the pod's ephemeral-storage limit — a runaway emptyDir triggers pod eviction, a different failure than the tmpfs OOM from 5.1. And the pod dir is deleted the moment the pod is — if you need evidence out of a crashing pod's volume, copy it before you delete/recreate.

    This layout is kubelet internals — not really documented. The exam-relevant docs page remains "Volumes".

    5.3

    The object model: PV is cluster infrastructure, PVC is a namespaced request

    Why it matters

    Every storage exam task is some slice of one chain: PV ← bound to ← PVC ← mounted by ← Pod. Get the scoping straight now and the YAML writes itself; get it wrong and you'll hunt for a namespace flag that doesn't exist.

    The concept

    The split mirrors nodes and pods:

    • A PersistentVolume is cluster-scoped — a piece of real storage (an NFS export, a cloud disk, a directory) represented as an API object, created by an admin or minted by a provisioner. Like a Node, it has no namespace.
    • A PersistentVolumeClaim is namespaced — an application's request: "I need 1Gi, ReadWriteOnce, of class X". Like a Pod, it lives with the workload.
    • Pods mount claims, never PVs: volumes: [{persistentVolumeClaim: {claimName: …}}]. There is no way to reference a PV from a pod spec.

    The indirection is the point: the app author asks for storage by size and behavior; which disk satisfies it is the cluster's problem. The PVC is the boundary between those two worlds, and the pod+PVC pair is portable across clusters whose PVs look completely different.

    Pod (ns)→claimName→PVC (ns)→bound 1:1→PV (cluster)→real storage
    Lab
    1. Ask the API itself about the scoping: 
      kubectl api-resources | grep -E '^persistentvolume|NAMESPACED'
      NAME                     SHORTNAMES   APIVERSION   NAMESPACED   KIND
persistentvolumeclaims   pvc          v1           true         PersistentVolumeClaim
persistentvolumes        pv           v1           false        PersistentVolume
    2. Confirm a pod can only name a claim: 
      kubectl explain pod.spec.volumes.persistentVolumeClaim
      FIELDS:
  claimName    <string> -required-
    claimName is the name of a PersistentVolumeClaim in the same namespace
    as the pod using this volume.
  readOnly     <boolean>

      Read that sentence twice: same namespace as the pod. That constraint decides where your PVC YAML needs a namespace: and where it must not.

    Verify

    Say it without looking: which of PV / PVC / Pod take -n <namespace>? (PVC and Pod. kubectl get pv -n anything silently ignores the flag — PVs are cluster-wide.)

    Gotchas

    Exam tasks encode the scoping in their wording: "create a PersistentVolume" (no namespace given — there isn't one) then "create a PersistentVolumeClaim in namespace project-x". If a pod can't find its claim, check the namespace before anything else — a PVC in the wrong namespace produces persistentvolumeclaim "data" not found on the pod even though kubectl get pvc -A plainly shows it.

    Docs

    Docs search: "Persistent Volumes" — the long concepts page. Storage answers live either here or on the task page "Configure a Pod to Use a PersistentVolume for Storage"; learn both layouts.

    Binding internals

    5.4

    The binding control loop: match, then an exclusive 1:1 claim

    Why it matters

    Every Pending PVC you will ever debug is this loop failing to find a match. Know the four matching rules and the failure stops being mysterious — you just check them in order.

    The concept

    The PV controller (inside kube-controller-manager) runs the standard reconcile pattern: watch unbound PVCs, and for each one scan the Available PVs for candidates. A PV is a candidate only if all four hold:

    • Capacity: PV capacity ≥ PVC request.
    • Access modes: the PV offers every mode the PVC asks for (PV modes ⊇ PVC modes).
    • storageClassName: string-equal on both sides. This is pure label-matching — the class doesn't have to exist as an object for static binding.
    • Selector: if the PVC has spec.selector, the PV's labels must match it.

    Among candidates it picks the smallest PV that satisfies the claim (to waste the least), then binds by stamping both sides: claimRef (namespace+name+UID of the PVC) onto the PV, volumeName onto the PVC. The bind is exclusive and 1:1 — even if the PV is 100× bigger than the request, no second claim can share it; a 100Gi PV bound to a 1Gi claim strands 99Gi. No candidate? The PVC just sits Pending, retried forever, and so does any pod that mounts it.

    Lab
    1. Offer two static PVs — 10Gi and 1Gi, same class manual (create the backing dirs on node01 first: sudo mkdir -p /srv/pv-big /srv/pv-small): 
      cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolume
metadata: {name: pv-big}
spec:
  capacity: {storage: 10Gi}
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  hostPath: {path: /srv/pv-big}
---
apiVersion: v1
kind: PersistentVolume
metadata: {name: pv-small}
spec:
  capacity: {storage: 1Gi}
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  hostPath: {path: /srv/pv-small}
EOF
kubectl get pv
      NAME       CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS      CLAIM
pv-big     10Gi       RWO            Retain           Available
pv-small   1Gi        RWO            Retain           Available
    2. Claim 1Gi and watch which PV the controller picks: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: claim-a}
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  resources: {requests: {storage: 1Gi}}
EOF
kubectl get pvc claim-a; kubectl get pv
      NAME      STATUS   VOLUME     CAPACITY   ACCESS MODES   STORAGECLASS
claim-a   Bound    pv-small  1Gi        RWO            manual

NAME       …   STATUS      CLAIM
pv-big     …   Available
pv-small   …   Bound       default/claim-a

      Smallest satisfying PV wins — pv-big stays free for a bigger claim.

    3. Inspect both halves of the stamp: 
      kubectl get pv pv-small -o jsonpath='{.spec.claimRef.namespace}/{.spec.claimRef.name}{"\n"}'
kubectl get pvc claim-a -o jsonpath='{.spec.volumeName}{"\n"}'
      default/claim-a
pv-small
    4. Now a claim nothing can satisfy — 5Gi would fit pv-big, but ask for a mode neither PV offers: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: claim-b}
spec:
  accessModes: [ReadWriteMany]
  storageClassName: manual
  resources: {requests: {storage: 5Gi}}
EOF
kubectl describe pvc claim-b | tail -3
      Events:
  Type     Reason              Age   From                         Message
  Warning  ProvisioningFailed  4s    persistentvolume-controller  storageclass.storage.k8s.io "manual" not found

      Decode that event: static matching found no PV (the RWX ask disqualified both), so the controller fell through to "dynamically provision, then" — and there's no StorageClass object named manual to do it. On a claim you expected to bind statically, "storageclass not found" means "no PV matched". Fix the mismatch, not the class.

    5. Keep claim-a/pv-small for 5.7; delete the experiment claim: 
      kubectl delete pvc claim-b
    Verify

    claim-a bound to pv-small (not pv-big), and you can read the bind from both directions: claimRef on the PV, volumeName on the PVC.

    Gotchas

    When a pod is stuck Pending on storage, describe the PVC, not the pod — the pod event only says "unbound PersistentVolumeClaims"; the PVC event says why. And once a default StorageClass exists (you'll install one in 5.10), a PVC that omits storageClassName entirely gets the default stamped at admission — and will then never match your storageClassName: manual PV. Static-PV tasks must set the class name explicitly on both objects.

    Docs search: "Persistent Volumes" → Binding and Phase sections.

    5.5

    The static chain from memory: PV + PVC + pod

    Why it matters

    This trio is the canonical storage exam task, nearly verbatim. It's worth over-practicing: three objects, zero thinking, under five minutes — that's points banked for the hard troubleshooting tasks.

    The concept

    Nothing new — this is 5.3's chain plus 5.4's matching rules, typed out. The fields that must agree, and where each lives:

    PV:   capacity ≥ ─────────┐   accessModes ⊇ ─┐   storageClassName = ─┐
PVC:  resources.requests ──┘   accessModes ───┘   storageClassName ───┘
Pod:  volumes[].persistentVolumeClaim.claimName = PVC name  (+ same namespace)
      containers[].volumeMounts[].name = volumes[].name
    Lab
    1. Prepare the backing dir on node01, then write all three objects from memory (peek only after trying): 
      ssh node01 'sudo mkdir -p /srv/app-data'
      cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolume
metadata: {name: app-pv}
spec:
  capacity: {storage: 2Gi}
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  persistentVolumeReclaimPolicy: Retain
  hostPath: {path: /srv/app-data}
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: app-data}
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  resources: {requests: {storage: 2Gi}}
---
apiVersion: v1
kind: Pod
metadata: {name: app}
spec:
  nodeName: node01
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep","1d"]
    volumeMounts: [{name: data, mountPath: /data}]
  volumes:
  - name: data
    persistentVolumeClaim: {claimName: app-data}
EOF
    2. Write, destroy the pod, recreate, read: 
      kubectl exec app -- sh -c 'echo persistent > /data/proof'
kubectl delete pod app
# re-apply just the Pod block from step 1, then:
kubectl exec app -- cat /data/proof
      persistent

      The data outlived the pod — that's the entire value proposition, witnessed.

    3. Now the deliberate mismatch. Delete pod + PVC (PV stays), and re-create the claim asking RWX: 
      kubectl delete pod app pvc app-data
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: app-data}
spec:
  accessModes: [ReadWriteMany]
  storageClassName: manual
  resources: {requests: {storage: 2Gi}}
EOF
kubectl get pvc app-data
      NAME       STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS
app-data   Pending                                      manual

      describe shows 5.4's storageclass "manual" not found event — the RWO-only PV was silently disqualified. Note what you did not get: no error at apply time. Binding failures are asynchronous; the objects are valid, they just never meet.

    4. Fix the claim back to RWO, watch it bind (the PV is Released from step 3's delete if it had been bound — see 5.7 for that wrinkle; here app-pv went Released when you deleted the first claim, so clear it): 
      kubectl patch pv app-pv --type=merge -p '{"spec":{"claimRef":null}}'
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: app-data}
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  resources: {requests: {storage: 2Gi}}
EOF
kubectl get pvc app-data
      NAME       STATUS   VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS
app-data   Bound    app-pv   2Gi        RWO            manual

      (You just previewed 5.7: deleting a bound PVC leaves a Retain PV Released, and clearing claimRef is the unlock.)

    5. Clean up: 
      kubectl delete pvc app-data; kubectl delete pv app-pv
    Verify

    Round 1: file survived pod deletion. Round 2: the mismatched claim sat Pending with no apply-time error, and you knew which field to check first.

    Gotchas

    A hostPath PV records no node — Kubernetes will happily run your pod on node02 against an empty /srv/app-data there. That's why this lab pins with nodeName. Real single-node storage uses local: PVs, which embed a required nodeAffinity so the scheduler follows the data; hostPath PVs are a lab convenience, not a pattern.

    5.6

    Access modes are node-level guarantees — RWO does not mean one pod

    Why it matters

    "Can two pods share a ReadWriteOnce volume?" is a classic trick question (yes — if they're on the same node), and choosing an impossible mode is a classic planted fault (the PVC just never binds, as you saw in 5.5).

    The concept
    ModeShortGuarantee unitMeaning
    ReadWriteOnceRWOnodeVolume mounts read-write on one node. Any number of pods on that node share it freely.
    ReadOnlyManyROXnodeMounts read-only on many nodes.
    ReadWriteManyRWXnodeMounts read-write on many nodes — needs a shared filesystem (NFS & friends); block-device drivers can't do it.
    ReadWriteOncePodRWOPpodThe only pod-level mode: exactly one pod in the whole cluster, enforced by the scheduler. GA since v1.29.

    Two more framing facts: on the PV, modes list what the storage can do (a PV may offer several); on the PVC, what the workload demands — matching needs PV ⊇ PVC (5.4). And the mode is an attach/mount-time rule, not a data-safety feature: nothing about RWO stops same-node writers from corrupting each other's files; the application still needs its own locking.

    Lab
    1. An RWO claim shared by two pods on the same node (reuse the 5.4 leftovers — claim-a is Bound to pv-small): 
      for n in one two; do kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata: {name: writer-$n}
spec:
  nodeName: node01
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep","1d"]
    volumeMounts: [{name: d, mountPath: /shared}]
  volumes: [{name: d, persistentVolumeClaim: {claimName: claim-a}}]
EOF
done
kubectl exec writer-one -- sh -c 'echo from-one >> /shared/log'
kubectl exec writer-two -- sh -c 'echo from-two >> /shared/log'
kubectl exec writer-one -- cat /shared/log
      from-one
from-two

      Two pods, both writing, one RWO volume — perfectly legal, because they share a node. RWO ≠ one pod. (You'll build the true multi-node RWX version in 5.10 with NFS.)

    2. Now RWOP, the one that really is single-pod. New PV+PVC pair (sudo mkdir -p /srv/pv-solo on node01 first): 
      kubectl apply -f - <<'EOF'
apiVersion: v1
kind: PersistentVolume
metadata: {name: pv-solo}
spec:
  capacity: {storage: 1Gi}
  accessModes: [ReadWriteOncePod]
  storageClassName: manual
  hostPath: {path: /srv/pv-solo}
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: claim-solo}
spec:
  accessModes: [ReadWriteOncePod]
  storageClassName: manual
  resources: {requests: {storage: 1Gi}}
EOF
    3. Point the same two-pod trick at it — the second pod must lose: 
      # same pod YAML as step 1, names solo-one / solo-two, claimName claim-solo
kubectl get pods solo-one solo-two
kubectl describe pod solo-two | tail -2
      NAME       READY   STATUS    RESTARTS   AGE
solo-one   1/1     Running   0          30s
solo-two   0/1     Pending   0          25s

  Warning  FailedScheduling  …  0/3 nodes are available: 1 node has pod using PersistentVolumeClaim
  with the same name and ReadWriteOncePod access mode, 2 node(s) had untolerated taint …

      The scheduler itself enforces RWOP — the second pod never even reaches a kubelet.

    4. Clean up everything from this group's experiments: 
      kubectl delete pod writer-one writer-two solo-one solo-two
kubectl delete pvc claim-a claim-solo
kubectl delete pv pv-big pv-small pv-solo
    Verify

    Both RWO pods wrote to the shared file; the second RWOP pod stayed Pending with the scheduler event naming the claim.

    Gotchas

    The deadly real-world version: a Deployment with an RWO PVC and the default RollingUpdate strategy on a multi-node cluster. The new replica starts on another node before the old one stops, can't attach the volume there, and the rollout wedges. Fix: strategy: Recreate, or RWX storage. Also remember kubectl get pv/pvc prints modes abbreviated (RWO, ROX, RWX, RWOP) — read them fluently.

    Docs

    Docs search: "Persistent Volumes" → Access Modes — includes the per-driver support matrix.

    5.7

    Reclaim policies by doing: Released is not Available

    Why it matters

    "Make sure the data survives PVC deletion" and "why won't this PV re-bind?" are both exam material, and both are answered by one field: persistentVolumeReclaimPolicy.

    The concept

    The policy answers: when the bound PVC is deleted, what happens to the PV and the data?

    PolicyOn PVC deleteDataDefault for
    RetainPV goes Released and staysKept — admin recovers or wipes manuallystatically created PVs
    DeletePV object and backing storage removedGonedynamically provisioned PVs (inherited from the StorageClass)

    The trap is the Released state (bottom lane of 5.4's diagram): the PV still carries claimRef pointing at the deceased PVC — including its UID — so the binder will never give it to a new claim, even an identical one. That's deliberate: the volume still holds the old claimant's data, and handing it silently to a new tenant would be a leak. Clearing claimRef is the admin saying "I've dealt with the data — back into the pool."

    Lab
    1. A Retain PV + claim (backing dir: sudo mkdir -p /srv/pv-retain on node01): 
      kubectl apply -f - <<'EOF'
apiVersion: v1
kind: PersistentVolume
metadata: {name: pv-retain}
spec:
  capacity: {storage: 1Gi}
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  persistentVolumeReclaimPolicy: Retain
  hostPath: {path: /srv/pv-retain}
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: claim-r}
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  resources: {requests: {storage: 1Gi}}
EOF
    2. Delete the claim; watch the PV's phase: 
      kubectl delete pvc claim-r
kubectl get pv pv-retain
      NAME        CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS     CLAIM
pv-retain   1Gi        RWO            Retain           Released   default/claim-r
    3. Re-create the identical claim — and watch it not bind: 
      # re-apply the PVC block from step 1
kubectl get pvc claim-r
      NAME      STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS
claim-r   Pending                                      manual

      Same name isn't same claim — the old claimRef pins the PV to the deleted PVC's UID.

    4. Do the admin move — clear the claimRef, watch both objects snap together: 
      kubectl patch pv pv-retain --type=merge -p '{"spec":{"claimRef":null}}'
kubectl get pv/pv-retain pvc/claim-r
      NAME        …   STATUS      CLAIM
pv-retain   …   Available → Bound   default/claim-r     (within a few seconds)

NAME      STATUS   VOLUME
claim-r   Bound    pv-retain
    5. For contrast, flip the policy on the live PV (a real exam micro-task): 
      kubectl patch pv pv-retain -p '{"spec":{"persistentVolumeReclaimPolicy":"Delete"}}'
kubectl get pv pv-retain -o jsonpath='{.spec.persistentVolumeReclaimPolicy}{"\n"}'
      Delete

      (Don't actually delete the claim now unless you want the PV gone — for hostPath, "Delete" removes the PV object; a real driver would also destroy the disk.) Clean up: kubectl delete pvc claim-r; kubectl delete pv pv-retain --ignore-not-found.

    Verify

    You saw the full circle: Bound → (PVC deleted) → Released → identical new claim stays Pending → clear claimRef → Available → Bound.

    Gotchas

    Clearing claimRef re-exposes whatever the previous claimant left on the volume — in production you wipe first, rebind second. Know the patch cold: kubectl patch pv NAME --type=merge -p '{"spec":{"claimRef":null}}'. And "protect this volume from deletion" in a task means: patch reclaimPolicy to Retain before touching the PVC.

    Docs

    Docs search: "Change the Reclaim Policy of a PersistentVolume" — a short task page with the exact patch command.

    StorageClasses & dynamic provisioning

    5.8

    StorageClass, field by field — and what "default class" really does

    Why it matters

    The exam cluster provisions storage dynamically — you'll be handed a StorageClass and expected to read it at a glance: does it expand? does it bind immediately? what happens to my PVC if I name no class at all?

    The concept

    A StorageClass is a named recipe for minting PVs. When a PVC references one (and no static PV matches first), a provisioner creates real storage plus a PV object to represent it, pre-bound to that claim.

    FieldWhat it controls
    provisionerWho gets called — a CSI driver name (ebs.csi.aws.com) or an external provisioner (rancher.io/local-path). The class is inert if nothing in the cluster answers to this name.
    parametersOpaque key/values passed to the provisioner (disk type, filesystem, replication). Kubernetes doesn't interpret them — the driver does.
    reclaimPolicyStamped onto every PV this class mints. Defaults to Delete.
    volumeBindingModeImmediate (provision the moment the PVC appears) or WaitForFirstConsumer (wait for a pod — 5.9).
    allowVolumeExpansionWhether PVCs of this class may be resized upward (5.13). One of the few mutable SC fields.
    mountOptions, allowedTopologiesMount flags for minted PVs; zone/node restrictions for where volumes may be created.

    The default class is marked by the annotation storageclass.kubernetes.io/is-default-class: "true". Its effect happens at admission: a PVC created with no storageClassName field at all gets the default's name written into its spec permanently. A PVC with explicit storageClassName: "" opts out — "static PVs only, never provision". Unset vs empty-string are different requests; that distinction is exam-grade. (If several classes are marked default, the newest wins — but treat that as a misconfiguration to fix.)

    Lab
    1. Your kubeadm cluster ships with no classes at all — which is why everything so far needed hand-made PVs: 
      kubectl get storageclass
      No resources found
    2. Prove a StorageClass is just a pointer — create one whose provisioner doesn't exist: 
      kubectl apply -f - <<'EOF'
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata: {name: ghost}
provisioner: example.com/does-not-exist
EOF
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: claim-ghost}
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: ghost
  resources: {requests: {storage: 1Gi}}
EOF
kubectl describe pvc claim-ghost | tail -3
      Events:
  Type    Reason                Age   From                         Message
  Normal  ExternalProvisioning  5s   persistentvolume-controller  Waiting for a volume to be created either by the external provisioner 'example.com/does-not-exist' or manually by the system administrator…

      Note it's a Normal event, and the PVC waits forever, politely. A valid class whose provisioner isn't running looks exactly like this — in week 6, this signature means "go check the provisioner's pods".

    3. Read a class the way the exam expects — columns first, YAML second: 
      kubectl get sc ghost
kubectl explain storageclass | head -20
      NAME    PROVISIONER                  RECLAIMPOLICY   VOLUMEBINDINGMODE   ALLOWVOLUMEEXPANSION   AGE
ghost   example.com/does-not-exist   Delete          Immediate           false                  1m

      Three defaults you didn't write, all visible in the headline columns.

    4. Clean up: 
      kubectl delete pvc claim-ghost; kubectl delete sc ghost
    Verify

    You can answer from the kubectl get sc line alone: will PVCs of this class expand? bind before a pod exists? leave PVs behind on delete?

    Gotchas

    Once a PVC is created, its storageClassName is effectively fixed — the default-class annotation does not retroactively adopt existing claims that already sat Pending before the default existed... with one nuance: a claim with the field truly unset may still be adopted by a later default (retroactive assignment). Keep the practical rule: decide the class at creation time, explicitly, and you never depend on the subtle path. Most SC spec fields (provisioner, parameters) are immutable — "change the parameters" means create a new class and migrate.

    Docs search: "Storage Classes"; the default-class mechanics live under "Change the default StorageClass".

    5.9

    WaitForFirstConsumer: the Pending that isn't a bug

    Why it matters

    A WFFC PVC sits Pending by design, and the exam (and real dashboards) love to show you one next to a genuinely broken one. Knowing the one event message that separates them saves you from "fixing" healthy clusters.

    The concept

    With volumeBindingMode: Immediate, provisioning happens the moment the PVC is created — before any pod exists, before anyone knows where that pod will run. For topology-constrained storage (a disk in zone A, a directory on node01) that's a gamble: the volume lands somewhere, and the scheduler is then forced to put the pod next to it, even if that node/zone can't fit the pod's CPU, affinity, or taints.

    WaitForFirstConsumer inverts the order:

    1. PVC created → nothing happens; it sits Pending with event WaitForFirstConsumer.
    2. A pod using the claim appears → the scheduler picks a node using all constraints (resources, affinity, taints, other volumes) and records it on the claim: annotation volume.kubernetes.io/selected-node.
    3. The provisioner reads the annotation and creates the volume there; PVC binds; kubelet mounts; pod starts.

    Storage placement becomes a scheduling output instead of a constraint. Local and zonal storage classes all use WFFC — including local-path, which you're about to install.

    Lab

    This lab uses the local-path class — run step 1 of 5.10 first if you haven't (one kubectl apply), then come back.

    1. Create a claim and watch it deliberately not provision: 
      kubectl apply -f - <<'EOF'

kind: PersistentVolumeClaim
metadata: {name: data-local}
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: local-path
  resources: {requests: {storage: 1Gi}}
EOF
kubectl get pvc data-local; kubectl describe pvc data-local | tail -3
      NAME         STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS
data-local   Pending                                      local-path

  Type    Reason                Age   From                         Message
  Normal  WaitForFirstConsumer  8s    persistentvolume-controller  waiting for first consumer to be created before binding

      Memorize that message. It is the "everything is fine" Pending.

    2. Give it a consumer: 
      kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata: {name: consumer}
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep","1d"]
    volumeMounts: [{name: d, mountPath: /data}]
  volumes: [{name: d, persistentVolumeClaim: {claimName: data-local}}]
EOF
kubectl get pvc data-local -w
      NAME         STATUS    VOLUME                                     CAPACITY
data-local   Pending

    Gotchas

    With WFFC, provisioning problems surface on the pod (FailedScheduling / stuck ContainerCreating), not just the PVC — describe both. And don't sit watching a WFFC claim waiting for Bound before you create the workload; it will never happen. In a timed exam that's minutes gone.

    The concept

    CSI internals

    5.11

    CSI architecture: sidecars translate API objects into four gRPC calls

    Why it matters

    When a real cluster's volume won't mount, the fix starts with knowing which of four steps died — because each step logs in a different place. Provision→Attach→Stage→Publish is the storage equivalent of week 4's packet path.

    The concept

    Kubernetes core contains no vendor storage code — it was all extracted ("CSI migration") into drivers that speak one gRPC interface, the Container Storage Interface. A driver ships two halves, and stock Kubernetes sidecar containers do all the Kubernetes-facing work, so the vendor only implements storage verbs:

    • Controller plugin — a Deployment/StatefulSet, anywhere in the cluster. Sidecars watch API objects and call the driver: external-provisioner (sees PVC → CreateVolume), external-attacher (sees VolumeAttachment → ControllerPublishVolume, i.e. "attach disk to that node"), external-resizer (sees PVC size edit → ControllerExpandVolume).
    • Node plugin — a DaemonSet on every node. The node-driver-registrar sidecar registers the driver's socket with the kubelet; from then on the kubelet calls the driver directly: NodeStageVolume (format if needed + mount once per node into a global staging dir), then NodePublishVolume (bind-mount the staged volume into each pod's volumes dir — the paths from 5.2).

    The staging step exists so a volume mounted by several pods on one node is formatted and fsck'd once; each pod then gets a cheap bind mount. Bookkeeping objects you can inspect: CSIDriver (driver's capabilities — e.g. whether it needs the attach step at all), CSINode (which drivers are registered on each node), VolumeAttachment (one per attached disk-node pair — the attacher's to-do list).

    Lab

    Your lab runs no full CSI driver (local-path and NFS are external provisioners — provision-only, no attach/stage machinery), so this is an inspection lab; you'll meet real driver pods on the exam-adjacent clusters (killer.sh) in week 7.

    1. The registration objects exist even with no drivers — one CSINode per node, empty: 
      kubectl get csinodes

kubectl get csidrivers
      NAME     DRIVERS   AGE

    2. Know where the sockets and staging dirs will appear when a driver does exist: 
      ssh node01 'sudo ls /var/lib/kubelet/plugins_registry/ /var/lib/kubelet/plugins/ 2>/dev/null'
      (empty — with a driver installed you'd see e.g. ebs.csi.aws.com-reg.sock and
 /var/lib/kubelet/plugins/kubernetes.io/csi/<driver>/<hash>/globalmount)
    3. And the attacher's queue object type is registered regardless: 
      kubectl api-resources | grep -E 'volumeattachments|csi'
      csidrivers          storage.k8s.io/v1   false   CSIDriver
csinodes            storage.k8s.io/v1   false   CSINode

    Verify

    From memory: the four calls in order, who makes each (sidecar, sidecar, kubelet, kubelet), and which API object each watches (PVC, VolumeAttachment, —, —).

    Gotchas

    The triage mapping is the exam skill: PVC Pending → provisioning → describe pvc + external-provisioner sidecar logs. Pod stuck with "FailedAttachVolume" → attach → kubectl get volumeattachment + attacher logs. Pod stuck ContainerCreating with "MountVolume.SetUp failed" / timeouts → stage/publish → journalctl -u kubelet on that node. Wrong log = wasted minutes.

    5.13

    Volume expansion: up only, class permitting — and three failure signatures

    Why it matters

    "Increase the PVC to 5Gi" is a plausible exam one-liner. It's ten seconds when allowed — and when it's not, the error messages tell you exactly why, if you've seen them before. This lab manufactures all three.

    The concept

    Rules of the resize game:

    • Up only. Shrinking is rejected by API validation, always. (One exception below.)
    • The class must allow it: allowVolumeExpansion: true, and only dynamically provisioned PVCs qualify.
    • The flow on a real CSI driver: edit spec.resources.requests.storage up → external-resizer calls ControllerExpandVolume (grow the disk) → if a filesystem grow is needed, the PVC gets condition FileSystemResizePending and the kubelet finishes with NodeExpandVolume — often online, but some drivers only resize at mount time, i.e. after a pod restart.
    • You're done when status.capacity catches up to the spec. Spec > status = expansion still in flight.
    • The exception: since v1.34 (GA), a stuck expansion may be walked back — you can lower the request again as long as you stay at or above what the volume actually allocated.
    Lab
    1. Signature 1 — shrink. Try to take data-nfs (1Gi) down to 500Mi: 
      kubectl patch pvc data-nfs -p '{"spec":{"resources":{"requests":{"storage":"500Mi"}}}}'
      The PersistentVolumeClaim "data-nfs" is invalid: spec.resources.requests.storage:
Forbidden: field can not be less than previous value

      Instant, synchronous, non-negotiable.

    2. Signature 2 — class forbids. Try to grow data-local (its class has allowVolumeExpansion: false — you read that column in 5.10): 
      kubectl patch pvc data-local -p '{"spec":{"resources":{"requests":{"storage":"2Gi"}}}}'
      Error from server (Forbidden): persistentvolumeclaims "data-local" is forbidden:
only dynamically provisioned pvcs can be resized and the storageclass that provisions
the pvc must support resize

      Also synchronous. In an exam task that hits this, the move is: kubectl patch sc <class> -p '{"allowVolumeExpansion":true}' (it's a mutable field), then retry the PVC edit.

    3. Signature 3 — allowed but nobody's listening. nfs-client ships allowVolumeExpansion: true, but the nfs-subdir provisioner has no resizer. Grow data-nfs: 
      kubectl patch pvc data-nfs -p '{"spec":{"resources":{"requests":{"storage":"2Gi"}}}}'
kubectl describe pvc data-nfs | tail -4
      persistentvolumeclaim/data-nfs patched     ← accepted!
Events:
  Type     Reason             Age   From                         Message
  Warning  ExternalExpanding  6s    volume_expand                Ignoring the PVC: didn't find a plugin
  capable of expanding the volume; waiting for an external controller to process this PVC.

      The API said yes (the class allows it); the machinery that would do the work doesn't exist. Spec vs reality:

      kubectl get pvc data-nfs -o jsonpath='request={.spec.resources.requests.storage}  actual={.status.capacity.storage}{"\n"}'
      request=2Gi  actual=1Gi

      That spec/status gap is the universal "expansion in flight (or stuck)" tell — on healthy CSI drivers it closes in seconds.

    4. Recover — walk the stuck request back down to what's actually allocated (legal since the v1.34 GA of expansion-failure recovery): 
      kubectl patch pvc data-nfs -p '{"spec":{"resources":{"requests":{"storage":"1Gi"}}}}'
kubectl get pvc data-nfs
      NAME       STATUS   VOLUME      CAPACITY   ACCESS MODES   STORAGECLASS
data-nfs   Bound    pvc-3e9f…   1Gi        RWX            nfs-client

      Spec = status again; the claim is clean. (This shrink is allowed precisely because it doesn't go below the real 1Gi allocation — rule 1 still stands.)

    Verify

    You can name, from the message alone, which rule each of the three signatures violated: validation (shrink), admission/class (no expansion), missing resizer (stuck ExternalExpanding).

    Gotchas

    On real drivers, the classic end-state confusion: PVC says 2Gi, df in the pod still says 1Gi → check kubectl get pvc -o yaml for the FileSystemResizePending condition — the disk grew, the filesystem waits for the next mount. Restart the pod and it completes. Never kubectl edit the PV's capacity to "help" — the PV size is the record of reality, not a knob.

    Docs

    Docs search: "Persistent Volumes" → Expanding Persistent Volumes Claims.

    5.14

    Timed drill: SC → PVC → pod → proof, under 7 minutes

    Why it matters

    Storage tasks are the exam's most formulaic points. The drill converts this week from knowledge into a reflex with a clock on it — the same way week 7 will drill everything else.

    The concept

    One wrinkle to internalize before the clock starts: there is no imperative generator for PVCs — no kubectl create pvc. Your two fast paths are (a) type the ~10-line YAML from memory, or (b) pull the skeleton from the docs task page ("Configure a Pod to Use a PersistentVolume for Storage") and edit three fields. Practice both; (a) is faster once it's automatic. For the pod, scaffold with the generator and splice the volume in:

    kubectl run app --image=busybox:1.36 --dry-run=client -o yaml --command -- sleep 1d > pod.yaml
# then add volumeMounts: + volumes: — 5 lines you must know cold
    Lab

    The task, exam-styled — set a timer for 7 minutes and go. Do the whole thing in namespace drill:

    1. Create namespace drill. In it, create a PVC cache-pvc: 500Mi, ReadWriteOnce, using the cluster's default StorageClass (i.e. don't set storageClassName).
    2. Create pod cache-pod (image busybox:1.36, command sleep 1d) mounting the claim at /data.
    3. Write hello into /data/f, delete the pod, recreate it, and prove the file survived.
    4. Self-check commands and the target end-state: 
      kubectl -n drill get pvc,pod
kubectl -n drill exec cache-pod -- cat /data/f
      NAME                              STATUS   VOLUME      CAPACITY   ACCESS MODES   STORAGECLASS
persistentvolumeclaim/cache-pvc   Bound    pvc-91d2…   500Mi      RWO            local-path

NAME            READY   STATUS    RESTARTS   AGE
pod/cache-pod   1/1     Running   0          40s

hello
    5. Timing budget to beat: PVC ≤ 2 min, pod ≤ 3 min, write/delete/recreate/verify ≤ 2 min. Over budget on a step → that step's YAML isn't memorized yet; redo the drill tomorrow (fresh namespace, delete drill after: kubectl delete ns drill).
    Verify

    Timer under 7:00 with hello printed at the end — including the recreate. If your PVC sat "Pending" and you waited on it before creating the pod, reread 5.9: with a WFFC default class, the pod is what unblocks the claim.

    Gotchas

    Three habit-formers, straight from the graders' pet peeves: the task says which namespace — every object goes there, not in default; don't create a PV (the class provisions — hand-made PVs on a dynamic task are wasted minutes); and finish with the verification read (8.8's habit) — an unread cat is a typo waiting to cost the task.

    Week 5 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. An emptyDir with medium: Memory — what's the hidden interaction with the container's resources, and what's the failure mode?
      Answer

      tmpfs bytes count against the container's memory limit; fill the volume past the limit and the kernel OOM-kills the container even though the app allocated little (5.1).

    2. A volume's data survives which of these: container restart? pod deletion? For emptyDir vs a PVC-backed volume?
      Answer

      Container restart: both survive (volumes belong to the pod). Pod deletion: emptyDir dies with the pod dir under /var/lib/kubelet; PVC data lives in the PV and survives — that's its whole job (5.1/5.5).

    3. Where exactly on a node do you find a running pod's emptyDir contents?
      Answer

      /var/lib/kubelet/pods/<pod-uid>/volumes/kubernetes.io~empty-dir/<volume-name>/ — UID from .metadata.uid; it's a bind mount, so writes are visible from both sides (5.2).

    4. Which of PV / PVC / StorageClass are namespaced?
      Answer

      Only the PVC. PV and StorageClass are cluster-scoped — exam wording reflects it ("create a PVC in namespace X", no namespace for the PV). Pods mount claims, never PVs (5.3).

    5. The four matching rules the PV controller checks, and which candidate it picks when several qualify?
      Answer

      Capacity ≥ request, PV's accessModes ⊇ PVC's, storageClassName string-equal, selector match (if set). It binds the smallest satisfying PV — exclusively, 1:1, via claimRef + volumeName (5.4).

    6. A PVC meant for a static PV shows storageclass.storage.k8s.io "manual" not found. What does that event actually mean?
      Answer

      Static matching found no PV (size/modes/class/selector mismatch), so the controller fell through to dynamic provisioning and found no class either. Fix the PV↔PVC mismatch — don't create a StorageClass (5.4/5.5).

    7. Can two pods write to one ReadWriteOnce volume? And what does RWOP change?
      Answer

      Yes — RWO is a node-level guarantee; any number of pods on that one node share it. ReadWriteOncePod is the only pod-level mode: the scheduler blocks a second pod cluster-wide (5.6).

    8. You delete a PVC bound to a Retain PV, then recreate the identical PVC. What state is everything in, and what's the fix?
      Answer

      PV is Released (claimRef still pins the old claim's UID); new PVC sits Pending. Fix: kubectl patch pv NAME --type=merge -p '{"spec":{"claimRef":null}}' → Available → binds. Released ≠ Available (5.7).

    9. PVC with no storageClassName field vs storageClassName: "" — what does each get?
      Answer

      Unset → the default class's name is stamped in at admission (if a default exists). Empty string → explicit opt-out: static PVs only, never provision. Different requests; static-PV tasks set the class explicitly on both sides (5.8).

    10. A PVC is Pending with event waiting for first consumer to be created before binding. What do you do?
      Answer

      Nothing — create the workload. WaitForFirstConsumer defers provisioning until the scheduler places a pod (recorded in the volume.kubernetes.io/selected-node annotation) so the volume lands where the pod can run (5.9).

    11. Provision → attach → stage → publish: who makes each call, and where do you look when each step fails?
      Answer

      external-provisioner (CreateVolume; watch PVCs) → external-attacher (ControllerPublishVolume; watches VolumeAttachments) → kubelet (NodeStageVolume, once per node into globalmount) → kubelet (NodePublishVolume, bind into the pod dir). Fail-points: describe pvc / get volumeattachment + sidecar logs / journalctl -u kubelet on the node for both node calls (5.11).

    12. Three resize attempts, three messages: shrink a PVC; grow one whose class lacks allowVolumeExpansion; grow one whose class allows it but the provisioner has no resizer. Name each signature.
      Answer

      Shrink → validation error "field can not be less than previous value". Class forbids → "only dynamically provisioned pvcs can be resized and the storageclass … must support resize" (fix: patch the SC — it's mutable). No resizer → patch accepted, then stuck with ExternalExpanding "waiting for an external controller", spec > status.capacity; recoverable by lowering back to the allocated size (GA v1.34) (5.13).

    13. A pod using an NFS PVC is stuck ContainerCreating; the kubelet log says bad option; … you might need a /sbin/mount.<type> helper program. Diagnosis?
      Answer

      That node is missing the NFS client package (nfs-common on Ubuntu) — mount helpers are per-node system packages, outside Kubernetes. Install it on that node (5.10/5.12).
