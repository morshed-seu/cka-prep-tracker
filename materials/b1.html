    B1 — Shell, files, and the filesystem tree

    The shell has a reputation for being the hard, old-fashioned way to use a computer — a worse file manager that makes you type. That is a misreading. A file manager lets you do the things somebody anticipated; the shell is a small programming language for composing a dozen tiny programs into an answer nobody anticipated, including you, ten seconds ago. "Which five processes are using the most memory, and who owns them?" is not a button anywhere. It is one line here.

    This module is the vocabulary every other module on the site speaks. Nothing after it will re-explain what a pipe is, what $? means, or why /etc and /var are different places — so take the labs literally, on the sandbox you built in B0, and run every command rather than reading it.

    Checkpoints16 Hours~4 PrerequisitesB0
    • Move to any directory on any Linux machine, by absolute or relative path, without hesitating.
    • Name the job of every top-level directory, and predict where an unfamiliar program keeps its config and its state.
    • Read every column of ls -l out loud, including the type character, and go one level deeper with stat.
    • Explain what an inode is, why deleting a file is really deleting a name, and which timestamp moves when.
    • Route a program's three streams deliberately, and chain programs into a pipeline that answers a real question.
    • Tell a shell glob from a regular expression, and say which piece of software expands each.
    • Use the working core of grep sed awk cut sort uniq wc head tail find xargs without reaching for a manual.
    • Explain the difference between a shell variable and an environment variable, and how PATH turns a word into a program.
    • Quote correctly by reflex, and know exactly what breaks when you do not.
    • Write, run and debug a small script that reports its own success or failure through an exit code.
    • Open, edit, save and quit vim on a machine where nothing else is installed.

    B0. You need the sandbox VM, the toolbelt installed in B0.6, and — this matters more here than it looks — the clean snapshot from B0.7. This module has you delete files and break your own PATH on purpose. Take a snapshot before you start: snapshot.sh.

    Paths and the tree

    B1.1

    Move around without a file manager

    Why it matters

    A path is an address, and almost everything you will type for the rest of this site is either a path or a command that takes one. Kubernetes manifests are paths, certificates are paths, a broken node is diagnosed by knowing which path to look in. Getting fluent at moving around is not a preliminary to the real work — it is a large fraction of the real work, and it takes about twenty minutes to stop being slow at it.

    The concept

    Linux has exactly one tree of files, and its root is a single character: /. There is no C: and no D:. If you plug in a second disk, it does not become a new letter — it gets grafted onto some directory inside the existing tree, and from then on it just looks like a folder. That grafting is called mounting, and B5 takes it apart in detail. For now, absorb the consequence: every file on the machine has exactly one address that starts at /.

    An absolute path is one of those addresses written out in full: /etc/systemd/system. It means the same thing typed from anywhere, by anyone, at any time. A relative path is a shortcut that is interpreted from wherever you currently are — system, typed while you are sitting in /etc/systemd, means the same directory. The rule is simply: starts with a slash → absolute; anything else → relative.

    "Wherever you currently are" is not a fiction the shell maintains for your convenience. It is a real property of the running process, held by the kernel, called the current working directory. Every process has one, inherited from whatever started it, and you can look at any process's copy of it directly — /proc/self/cwd is a symbolic link pointing at the directory of whatever process is asking. When a program opens the relative path config.yaml, the kernel resolves it against that stored directory. This is why a script that works when you run it from its own folder mysteriously fails when run from somewhere else, and it is the whole explanation of that bug.

    Four shorthands do most of the work, and they are ordinary directory entries or shell features, not magic:

    ShorthandMeansWhat it really is
    .herea real entry in every directory, pointing at itself
    ..one level upa real entry in every directory, pointing at its parent
    ~your home directoryexpanded by the shell from $HOME before the command runs
    -the directory you were in beforea cd feature, remembered in $OLDPWD

    One more thing worth knowing on day one, because it explains a confusing error later: cd is not a program. It is built into the shell itself. It has to be — a program runs as a separate process with its own working directory, so a /usr/bin/cd would change its own directory and then exit, leaving your shell exactly where it was. Anything that must change the shell's own state (cd, export, exit) is necessarily a builtin.

    you type cd system→shell changes its own cwd→kernel records it on the process→later relative paths resolve against it

    Gotchas

    which cd prints nothing and exits non-zero, which looks like cd is missing. It is not — which only searches PATH for executable files, and cd is a shell builtin with no file anywhere. type cd is the command that answers correctly, and B1.11 comes back to exactly this distinction.

    Relative paths in scripts are the classic silent bug. A script containing cat config.yaml works when you run it from its own directory and fails from anywhere else, because the path resolves against the caller's working directory, not the script's. The fix is to use an absolute path or to compute one from the script's own location — and the habit of noticing which one you have written.

    A path that goes through a symbolic link has two truthful answers for "where am I". By default cd and pwd keep the route you typed; cd -P and pwd -P resolve the links and give you the physical location. On a machine where /bin is a symlink to /usr/bin — which is every modern Ubuntu — this is the difference between pwd saying /bin and pwd -P saying /usr/bin.

    The reason "one tree, and you can graft anything anywhere onto it" is worth this much attention: it is the entire mechanism by which a container gets a filesystem. A container is handed a different /, assembled by mounting things into place, and the processes inside it navigate that tree with exactly the paths you just used — unable to tell, and not caring, that it is not the machine's real root.

    You can see the machinery from the node side in W5.2 — where volumes actually live: every volume a pod uses is a real directory under /var/lib/kubelet/pods/<uid>/volumes on the host, mounted into the container at whatever path the pod spec asked for. Two absolute paths, one graft.

    Docs

    man 7 path_resolution is the authoritative description of how the kernel turns a string into a file, and it is short — read it. Then man 1 bash and search (/) for SHELL BUILTIN COMMANDS for cd, and man 5 proc for what /proc/self/cwd is.

    B1.2

    Tour the standard directories and learn them by their job

    Why it matters

    Sit down at an unfamiliar Linux machine and you already know where to look — not because you have seen that machine before, but because the layout is a published standard that nearly everything follows. Learning the top level once buys you the ability to guess correctly for the rest of your career: where a service keeps its settings, where it writes its logs, where its data lives, and which of the three you are allowed to delete.

    The concept

    The layout has a name — the FHS, the Filesystem Hierarchy Standard — and the useful way to learn it is not as a list of names but as a set of jobs. Every directory answers one question about the data in it: did a human configure this, did the machine accumulate it, did a package install it, or is it not really a file at all?

    DirectoryIts jobWhat that means in practice
    /etcconfiguration you settext files, editable, worth backing up; nothing here is generated by running the service
    /varstate the machine accumulateslogs, databases, caches, spools — grows on its own, and is where "disk full" comes from
    /usrprograms and their datainstalled by the package manager, effectively read-only to you; /usr/bin, /usr/lib, /usr/share
    /homepeopleone directory per human user; yours is /home/ubuntu
    /tmpscratchanyone may write here, and the system deletes it out from under you
    /runruntime state since bootlives in memory, empty at every boot: process ids, unix sockets
    /devdevices as filesdisks, terminals, /dev/null — the hardware, addressable by path
    /procthe kernel, answering questionsnot on any disk; one directory per running process, invented on read
    /systhe kernel's device and tuning knobsalso invented on read; where B7 will write a memory limit
    /bootwhat is needed to startthe kernel image itself and the bootloader
    /opt, /srvthird-party software; served dataoften empty; the standard's escape hatches

    Two of those rows deserve emphasis now because whole modules stand on them. /proc and /sys are not storage. Nothing in them is on the disk. They are a conversation with the running kernel wearing the costume of a filesystem: when you read /proc/meminfo, no file is opened — the kernel generates that text at the instant you ask. This is a deliberate design choice, and it is why every tool that reports on the system (ps, free, top) turns out to be a thin wrapper around reading text files. B3 makes you watch it happen.

    And the /etc versus /var distinction is the one that pays off constantly, because it tells you which files are precious. Configuration is what you set and can rewrite; state is what the machine produced and mostly cannot be regenerated. On a Kubernetes node those two directories are exactly where everything interesting lives — /etc/kubernetes holds the manifests and certificates somebody configured, /var/lib/kubelet and /var/lib/containerd hold gigabytes the machine accumulated by running.

    One historical wrinkle you will see the moment you run the lab: on modern Ubuntu, /bin, /sbin and /lib are no longer directories at all. They are symbolic links to their counterparts inside /usr — a tidy-up known as the "merged /usr" transition, done so that everything a package installs lives under one subtree that can be shared or mounted read-only. The old names still work, which is the point.

    2. Confirm that /proc is generated rather than stored. Ask what kind of filesystem is mounted at each of three places; the answers are three different words, and only one of them is a disk.

      ext2/ext3 is how stat labels the ext4 filesystem on your virtual disk. The other three are not disks at all: proc and sysfs are the kernel answering questions, tmpfs is memory pretending to be a disk.

    Without looking, say which directory each of these belongs in: an SSH server's settings; yesterday's authentication log; the ls binary; a running process's memory statistics; a lock file that must vanish on reboot. (/etc/ssh, /var/log, /usr/bin, /proc, /run.) Then check yourself: ls /etc/ssh/sshd_config /var/log/auth.log /usr/bin/ls /proc/self/status /run/lock should print all five paths with no errors.

    Gotchas

    /tmp is genuinely temporary — systemd-tmpfiles cleans files there on a schedule and the directory is emptied at boot, so anything you leave overnight may simply be gone. That is a feature, not a bug, and it is why "I'll just put it in /tmp for a minute" is fine and "I'll keep my notes in /tmp" is not. Its permissions look alarming (drwxrwxrwt — everyone may write) but the trailing t is the sticky bit, which stops you deleting other people's files there; B4 covers it properly.

    Writing your own files into /usr works right up until the package manager overwrites or removes them, because /usr is considered owned by the packaging system. Local additions belong in /usr/local or /opt, which exist precisely so the two never collide.

    Do not assume every entry under /proc is a process. The numbered directories are (one per process id), but the named ones — meminfo, cpuinfo, mounts, sys — are system-wide, and /proc/sys in particular is a completely different thing from the /sys directory: it holds tunable kernel parameters, the ones sysctl reads and writes.

    When something on a Kubernetes node breaks badly enough that kubectl cannot help — the API server itself is down, or the kubelet never started — the only thing left is a shell on the node and the knowledge of which directory to open. That ladder is W6.2 — where logs physically live, and every rung of it is a path in the table above.

    The map is short and worth recognising now: /etc/kubernetes holds the control-plane manifests and the whole certificate tree; /var/lib/kubelet holds the kubelet's state and every pod's volumes (W5.2); /var/log/pods and /var/log/containers hold what kubectl logs normally hands you; /var/lib/containerd holds the unpacked image layers you will build by hand in B5. Configuration in /etc, accumulated state in /var — the same rule, one layer up.

    Docs

    man 7 file-hierarchy is the modern, systemd-flavoured description of this layout and is more readable than the standard itself. man 5 proc is the enormous, authoritative reference for every file under /proc — skim its table of contents, do not read it. man 8 systemd-tmpfiles explains exactly when /tmp is cleaned.

    B1.3

    Read ls -l completely, left to right

    Why it matters

    Most people learn ls -l by absorbing two of its columns — the name, and roughly whether something is a directory — and ignore the rest for years. Every column is load-bearing. The very first character alone will one day tell you why a program is hanging on something that looks like an ordinary file, and the permission triads are the answer to about a third of all "permission denied" mysteries.

    The concept

    One line of ls -l carries eight pieces of information, in a fixed order:

    - rw-r--r--  1  root  root  1024  Jul 23 14:02  /etc/hostname
1     2      3    4     5      6         7             8

1  type character        what kind of thing this is
2  permission triads     what owner / group / everyone else may do
3  link count            how many names in the tree point at this file
4  owner                 the user who owns it
5  group                 the group that owns it
6  size                  bytes of content (or major,minor for a device)
7  mtime                 when the contents last changed
8  name

    The type character is the one that repays learning today, because "everything is a file" turns out to be meant literally. Seven different kinds of thing appear in the same tree, with the same permission bits, answering the same read and write system calls — and only this character tells them apart.

    CharKindWhat it isFind one at
    -regular filebytes on a disk/etc/hostname
    ddirectorya list mapping names to inode numbers/etc
    lsymbolic linka file whose contents are another path/bin
    ccharacter devicea driver you read/write a byte at a time/dev/null, your terminal
    bblock devicea driver addressed in fixed-size blocks/dev/sda
    ssockettwo-way local communication between processes/run/systemd/private
    pnamed pipe (FIFO)a one-way pipe that has a name in the treeone you make with mkfifo

    That uniformity is not a curiosity; it is the design decision the entire operating system rests on. A program that can write to a file can write to a terminal, a disk, a network peer or another program without knowing which — same write call, different type character. It is why B3's strace can show one program talking to all four in identical syntax, and why a container runtime can hand a process a socket where it expected a file and nothing notices.

    The permission triads are three groups of rwx: what the owner may do, what members of the group may do, what everyone else may do. A - means that bit is off. The full semantics — including the surprise that x on a directory means "may traverse", not "may execute" — is B4's job; for now just read them aloud correctly.

    The link count is the field people skip and then find baffling. It is the number of directory entries pointing at this file's underlying object. For a regular file it is normally 1. For a directory it is at least 2, because a directory is named by its parent and by its own . entry — and it gains one more for every subdirectory, since each of those contains a .. pointing back. That is why ls -ld /etc shows a link count in the nineties: /etc has that many subdirectories. B1.4 takes the idea apart.

      Size 7, and usr/bin is seven characters. That is the whole file.

    5. Clean up the pipe.

    You found and can point at one example of each of - d l c b s p on your own machine.     Gotchas

    ls -l on a symbolic link describes the link, not what it points at — the permissions lrwxrwxrwx are meaningless, since access is always decided by the target. Add -L to follow the link and report the target instead. This trips people up when they check permissions on a symlinked config file, see rwxrwxrwx, and conclude the file is world-writable.

    The size column for a directory (usually 4096) is the size of the list of names, not of everything inside it. A directory holding a hundred gigabytes still reads 4096. To ask the question you meant, use du -sh.

    Names beginning with a dot are not shown unless you pass -a, and this has bitten everybody at least once: you copy a directory, or empty one with rm *, and the dotfiles quietly stay behind because the shell's * does not match them either. B1.8 explains why the glob behaves that way.

    Those type characters are not trivia in Kubernetes — they appear verbatim in the API. A hostPath volume takes an optional type: field whose allowed values are literally Directory, File, Socket, CharDevice and BlockDevice, and the kubelet refuses to start the pod if what it finds on the node is a different type from what you declared. See W5.1 — the ephemeral volumes.

    That check exists because the alternative is worse: with no type declared, the kubelet will happily create an empty directory at a path you meant to be a file, and the failure surfaces much later as an application reading nothing. The socket case is the one you will meet in real life — mounting /var/run/docker.sock or a CSI driver's socket into a container is exactly the s from the table above, handed across a namespace boundary.

    Docs

    man 1 ls for the flags, but the field descriptions live in man 1 stat and man 2 stat — the latter documents the type bits the character comes from. man 7 inode is the concise reference for the file types themselves. man 1 mkfifo and man 7 fifo for named pipes, man 4 null for the device file you will use constantly in B1.6.

    B1.4

    Go one level deeper with stat: inodes, links and the three timestamps

    Why it matters

    There is one idea in this lesson, and it quietly explains a dozen otherwise-confusing behaviours: a file is not its name. The name and the file are separate objects, and the name is the disposable one. Once you have that, "why is my disk full when I deleted the log", "why did my config change not take effect", and "why does mv across disks take a minute when across directories it is instant" all stop being mysteries and become the same fact.

    The concept

    Every file on a filesystem is an inode — a numbered record holding everything about the file except its name: the type, the permissions, the owner, the size, the timestamps, and the list of disk blocks with the actual contents. A directory is just a table mapping names to inode numbers. So the name lives in the directory, and the file lives in the inode, and they are joined only by that number.

    directory entry
    "notes.txt"    →inode 262145→permissions, owner,
    timestamps, block list    →the bytes

    Several names may point at the same inode. Each such name is a hard link, and the inode counts them — that is the link count from B1.3. The consequences fall straight out:

    • Deleting is unlinking. rm removes a name and decrements the count. The file itself is only destroyed when the count reaches zero and no process still has it open. This is exactly why a deleted log file can keep occupying disk space until the program writing it is restarted — du cannot see it because it has no name left, but the blocks are still allocated.
    • Renaming is cheap. mv within one filesystem rewrites a directory entry and touches nothing else, however large the file. Across filesystems it must genuinely copy and delete, because inode numbers are only meaningful within one filesystem.
    • Hard links are not copies. There is one file with two names; writing through either name changes the same bytes. And a hard link cannot cross filesystems or (normally) point at a directory, for the same reason.

    A symbolic link is the other thing entirely, and worth contrasting once: it is its own inode, whose contents are a path string the kernel re-resolves on every access. So a symlink can cross filesystems, can point at something that does not exist, and breaks if the target is renamed — none of which is true of a hard link.

    Then the three timestamps, which are three genuinely different questions and are constantly confused:

    FieldQuestion it answersMoved by
    atime — accesswhen were the contents last read?cat, and any read — but see the gotcha
    mtime — modifywhen did the contents last change?writing to the file; this is what ls -l shows
    ctime — changewhen did the inode last change?writing, and also chmod, chown, adding a hard link

    The trap is the name: ctime is not creation time. It is "inode change time", and it moves on permission changes that leave the contents untouched. Traditional Unix simply did not record creation time at all; modern Linux filesystems do keep a birth time, which stat will show as Birth: when the filesystem supports it.

      The name is gone from the directory, so nothing that walks the tree can find it, but the process still holds it open — link count 0, open count 1 — so the inode and its blocks live on until that process exits. Kill it with kill %1 and the space returns.

      kill %1 2>/dev/null; rm -f second-name.txt

    Explain, without looking: why chmod moves ctime but not mtime; why rm on a file with link count 3 destroys nothing; and why a full disk can show no large files under du.     Gotchas

    atime is not updated the way you would expect, and if you test it naively you will conclude your machine is broken. Linux mounts filesystems with relatime by default: the access time is only written if it was older than the modify time, or older than about a day. Updating it on every read would mean a disk write for every file read, which is an absurd price for a field almost nobody uses.

    "ctime is creation time" is such a common misreading that it is worth over-learning the correct one: c is for change, meaning a change to the inode. If you want creation time, look for Birth: in stat output — and be ready for a dash instead, because not every filesystem records it.

    Editors are where the inode idea reaches out and bites. Many of them — vim included, by default — do not write into your file; they write a new file and rename it over the top, which gives you a new inode. So a hard link you carefully made to that file no longer follows the changes, and a program holding the old file open keeps reading the old contents. Anything that must survive editing should be a symlink, or you should configure the editor to write in place.

    Kubernetes uses this exact machinery, and knowing it explains one of the platform's most confusing behaviours. When you mount a ConfigMap or Secret into a pod, the kubelet does not edit files in place. It writes the new contents into a fresh timestamped directory and then atomically swaps a symlink to point at it — the same "the name is the disposable part" trick you just used with ln, applied so that a container can never observe a half-updated set of files.

    The consequence is W3.22 — update propagation: volumes update, env never, subPath never. A ConfigMap mounted as a volume updates live, because the symlink is re-pointed under the running process. Mounted with subPath, it does not — because subPath mounts one file directly, bypassing the symlink swap, and the container is left holding the original inode. That is not a Kubernetes quirk. That is inodes.

    Docs

    man 7 inode — the concise, authoritative description of everything in this lesson, including the timestamp definitions. man 1 stat for the -c format specifiers, man 2 stat for the underlying system call, man 1 ln for hard versus symbolic (-s), and man 5 fstab if you want to see where relatime is set.

    Composing tools

    B1.5

    Every program is born with three channels, not one

    Why it matters

    Beginners think of a program as having "output" — one stream of text that appears on the screen. It has two, and they are kept deliberately separate: results on one, complaints on the other. Once you can see both channels, a whole class of confusing behaviour resolves itself: why an error still appears on screen after you sent everything to a file, why a pipeline carries clean data even when something went wrong, and why 2>/dev/null is the single most-typed incantation in shell scripting.

    The concept

    When a process opens anything — a file, a terminal, a network connection, a pipe — the kernel hands it back a small integer to refer to that thing by. The integer is called a file descriptor, and from the process's point of view it is the only handle it has. Descriptor numbers are per-process and start at 0, handed out lowest-first.

    Every process is born with the first three already open, inherited from whatever started it. It does not open them and cannot count on their being anything in particular; it just uses them:

    fdNameIts jobWhere it points by default
    0stdinwhere input comes fromyour keyboard, via the terminal
    1stdoutthe results — the answer you asked foryour screen, via the terminal
    2stderrdiagnostics — errors, warnings, progressyour screen, via the terminal

    Because all three normally point at the same terminal, they look like one stream, and that is exactly why the separation is invisible until you go looking for it. The reason it exists is worth stating plainly: results are for the next program, diagnostics are for the human. If a tool mixed its error messages into its results, then piping it into another program would feed that program garbage — and suppressing the garbage would suppress the answer too. Two channels lets a pipeline carry clean data downstream while errors still reach you.

    Remember the terminal itself is a file, of type c from B1.3. Nothing about descriptors 0, 1 and 2 knows or cares that a terminal is on the other end; that is the whole point, and it is why the same program works identically when you redirect its output to a file or pipe it into another program in the next two lessons.

    keyboard→fd 0 · stdin→your program→fd 1 · stdout — the resultsand, separately,fd 2 · stderr — the complaints
    Lab
    1. Run one command that succeeds and fails at the same time. ls is asked about two paths: one exists, one does not, so it produces both a result and an error.

      Two lines, apparently from one stream. Now prove they are not.

      Read that carefully: the error appeared immediately, on screen, from the first command — while the result went silently into the file and only showed up when you catted it. Two channels, demonstrated in four words of syntax.

    3. Now do the opposite and capture only the complaints, leaving results on screen. 2> redirects descriptor 2 by number. 
      ls /nope /etc/hostname 2> err.txt

    4. Look at your own shell's three descriptors as files. The kernel exposes every open descriptor of every process under /proc/PID/fd, as symlinks to whatever they point at. 
      ls -l /proc/$$/fd

      All three point at /dev/pts/0 — your terminal, a character device. This is the picture the next lesson rewires.

      bash -c 'ls -l /proc/$$/fd' > fds.txt; cat fds.txt

      Descriptor 1 now points at the file — and note it is l-wx, write-only, because that is all a > redirect asks for. The program was never told; the shell rewired it before the program started.

    Predict, then run: what does ls /nope /etc/hostname 2> /dev/null print, and what does ls /nope /etc/hostname > /dev/null print? (The first prints only /etc/hostname; the second prints only the error.) If both surprise you, redo steps 2 and 3 — everything in B1.6 builds directly on this.

    Gotchas

    stderr is not only for errors. Plenty of well-behaved programs write progress bars, counters and status lines there on purpose, precisely so that the results on stdout stay clean and pipeable — curl, wget, dd and tar -v all do it. So a message on stderr does not mean something failed. The exit code (B1.13) is what tells you that.

    The two streams can appear out of order when you combine them, and the reason is buffering rather than anything you did wrong. The C library buffers stdout in large blocks when it is not a terminal, while stderr is unbuffered — so a program writing alternate lines to each, with stdout going to a file or pipe, will emit all its stderr promptly and its stdout in a lump at the end. This is why interleaved output looks scrambled only after you redirect it. stdbuf -oL cmd forces line buffering when it matters.

    This is the entire contract between a container and Kubernetes for logging. A containerized program is expected to write its logs to stdout and stderr and nothing else — no log files, no syslog. The runtime captures both streams, writes them to a file on the node under /var/log/pods/, and that file is what kubectl logs reads back to you.

    Two consequences follow directly, and both are in W6.2 — where logs physically live. First, an application configured to log into a file inside the container is invisible to kubectl logs, which is a genuinely common and genuinely baffling incident. Second, kubectl logs merges the two streams — so the separation you just proved exists is real on the node and gone by the time you read it, which is why exit codes rather than log content are the reliable signal for "did it fail".

    Docs

    man 3 stdin covers all three streams and their buffering rules in one short page — the buffering paragraph is the one worth reading twice. Then man 2 open for where descriptors come from, and man 5 proc for /proc/[pid]/fd.

    B1.6

    Redirect the three streams deliberately

    Why it matters

    Redirection is how you decide where a program's input comes from and where each of its two outputs goes. It is a handful of symbols, it is used in essentially every script ever written, and the one piece of it that is genuinely counter-intuitive — 2>&1 — catches everybody exactly once. Better to be caught here, deliberately, than in a script that has been silently discarding its error messages for a month.

    The concept

    A redirection is not a feature of the program. It is the shell rewiring descriptors in the instant between forking a new process and starting your program in it. By the time the program runs, its descriptor 1 already points wherever you said, and the program has no way of knowing and no need to care. That single fact explains why redirection works uniformly on every command that has ever existed.

    SyntaxDoesNote
    > filestdout to file, replacing itthe file is emptied first — see the gotcha
    >> filestdout to file, appendingthe safe default when you are unsure
    2> filestderr onlythe number is the descriptor from B1.5
    2>> filestderr, appending
    &> fileboth streams to filebash convenience; not portable to plain sh
    2>&1make stderr go wherever stdout points right nowposition-sensitive; the whole lesson is here
    < filefile becomes stdinthe program reads it as if typed
    <<'EOF'the following lines become stdina "heredoc"; quoting EOF stops expansion
    > /dev/nulldiscard/dev/null is a device that swallows writes

    Now the one that matters. 2>&1 reads as "make descriptor 2 a copy of descriptor 1" — and the crucial word is copy. It copies where descriptor 1 is pointing at the moment the shell processes it, and the shell processes redirections strictly left to right. So:

    • cmd > f 2>&1 — first stdout is pointed at f; then stderr is made a copy of stdout, which now means f. Both land in the file. This is what you want.
    • cmd 2>&1 > f — first stderr is made a copy of stdout, which still means the terminal; then stdout alone is moved to f. Errors keep going to your screen. This is the bug.

    They differ by nothing but word order, and the second one is what people type when they reason "redirect errors to output, then send output to the file". The mental model that fixes it permanently: a redirection sets a descriptor's destination, and 2>&1 reads a destination rather than establishing a link. Rewiring stdout afterwards does not drag stderr along with it.

    Finally, /dev/null deserves a sentence of its own. It is a real character device (you listed it in B1.3) whose driver discards everything written to it and returns end-of-file to every read. It is the bin, and it is also the empty file: cmd < /dev/null is the standard way to tell a program "you have no input, stop waiting".

    Lab

    Gotchas

    The redirect happens before the command runs, and > empties the file as part of setting it up. So sort file.txt > file.txt does not sort a file in place — it truncates file.txt to nothing, and then starts sort, which reads an empty file. You have destroyed the data. Write to a different name and move it over, or use a tool with an explicit in-place flag such as sed -i.

    &> and &>> are bash extensions. A script with #!/bin/sh at the top on Ubuntu runs under dash, where &> means something else entirely (run in background, then redirect) and fails silently and weirdly. Write > file 2>&1 when portability matters, or commit to #!/usr/bin/env bash as B1.13 does.

    Redirecting to a file you lack permission to write fails in the shell, before the command starts — so sudo echo hi > /etc/protected does not work: sudo elevates echo, but your unprivileged shell is the one opening the file. The fix is to elevate the thing doing the writing: echo hi | sudo tee /etc/protected.

    The truncation gotcha above has an exact analogue you will meet with kubectl: kubectl get pod x -o yaml > pod.yaml is safe, but any pattern where you read and write the same file in one line is not. More directly useful, the <<'EOF' heredoc is the single most common way manifests get created on an exam machine — cat <<'EOF' | kubectl apply -f - writes a whole YAML document straight into kubectl's stdin with no editor involved.

    That trailing -f - is stdin, exactly as in B1.5. See W7.10 — imperative generators, where the fastest possible path to a working object is a generator's stdout piped into kubectl apply -f -, and the editor never gets opened at all.

    B1.7

    Chain programs with | and build a real answer

    Why it matters

    This is the idea the whole shell is built around, and it is the reason a command line beats a graphical tool for anything nobody anticipated. No program on your machine can answer "which five processes are using the most memory". Three of them, joined by two vertical bars, answer it in one line — and the same three recombine tomorrow to answer a different question.

    The concept

    A pipe connects one program's stdout to the next program's stdin. The shell creates a small buffer inside the kernel, points the left-hand program's descriptor 1 at the writing end and the right-hand program's descriptor 0 at the reading end, and starts both — this is the part people get wrong — at the same time.

    That concurrency is not an implementation detail. cat huge.log | grep error does not read the whole file and then search it; grep is already matching the first lines while cat is still reading the middle. The buffer between them is finite (64 KB on Linux), and when it fills, the writer simply blocks until the reader catches up. The pipeline self-regulates with no coordination code anywhere.

    It also explains what happens when the reader leaves early. In … | head -5, head prints five lines and exits. The next time the left-hand program writes, there is nobody at the other end, so the kernel sends it the SIGPIPE signal, whose default action is to kill it. This is a feature: the pipeline stops as soon as the answer is complete, rather than computing output nobody will read. B2 covers signals properly; the observable version is that yes | head -1 terminates instantly instead of running forever.

    Two more properties are worth fixing in your head now, because both cause real bugs:

    • Only stdout flows through a pipe. stderr is untouched and goes straight past to the terminal — which is exactly the design from B1.5 paying off, and why you can watch errors scroll by while clean data keeps moving down the pipeline.
    • A pipeline's exit code is the exit code of the last command. So false | true reports success. B1.13 and every script you write will want set -o pipefail to change that; bash also exposes every stage's code in the array ${PIPESTATUS[@]}.

    The design philosophy underneath all this is that the universal interface between programs is lines of text. Nobody had to make ps, sort and head aware of each other; they each read lines and write lines, so they compose. That is the trade the shell makes — text is a weaker contract than a real data structure, and in exchange every tool works with every other tool, forever.

    Lab
    1. Build the memory-users pipeline one stage at a time, so you can see what each stage receives and emits. Start with the raw material: ps aux lists every process on the machine, one per line, with a header. 
      ps aux | head -3

    Gotchas

    Each stage of a pipeline runs in its own subshell, so variables set inside one do not survive. find . | while read f; do n=$((n+1)); done; echo $n prints nothing useful, because the while ran in a child process that has since exited with its variables.     Buffering makes live pipelines look frozen.     sort -k4 does not mean what it looks like: it sorts from field 4 to the end of the line, not on field 4 alone. Usually harmless, occasionally the reason two lines with equal fourth fields come out in a surprising order. sort -k4,4 is the precise version.

    kubectl is a well-behaved Unix program, and it is designed to sit in exactly these pipelines. Its structured output modes exist so that the text flowing down the pipe is predictable rather than the human-formatted table: kubectl get pods -o json | jq …, or kubectl get pods -o jsonpath=… | xargs kubectl delete pod.

    W7.11 — output surgery: jsonpath, custom-columns, sort-by drills this as an exam speed skill, and the reason it is a speed skill is the one from this lesson: nobody built a "delete every pod that is not Running" button, so you compose one from parts. Note also that kubectl get writes its table to stdout and its warnings to stderr, so piping it never contaminates your data.

    B1.8

    Globs versus regular expressions — two pattern languages, two different programs

    Why it matters

    This is the single most common confusion among people learning the shell, and it produces bugs that look like sorcery: a grep that searches for the wrong thing, a find that matches nothing, a script that works in one directory and not another. All of it comes from merging two separate pattern languages into one imaginary language. Separating them takes ten minutes and never has to be learned again.

    The concept

    There are two systems, they use some of the same characters to mean different things, and — the part that actually matters — they are expanded by different pieces of software at different times.

    A glob is expanded by the shell, against filenames, before your program starts. When you type ls *.conf, ls never sees an asterisk. The shell looks in the current directory, finds the matching names, and replaces the word with that list, so what actually runs is ls a.conf b.conf c.conf. Globs only ever match filenames, because that is the only thing the shell has to match against.

    A regular expression is interpreted by the program — grep, sed, awk — against the text it is reading. The shell must be persuaded to keep its hands off it, which is what the quotes in grep '^a.*b$' are for. Regexes match anything, usually the contents of lines.

    glob:you type *.conf→shell expands it against filenames→program receives a.conf b.conf
    regex:you type '^a.*b$'→shell leaves it alone (you quoted it)→program receives the pattern and interprets it

    Now the character collisions, which is where the confusion actually lives:

    CharacterAs a glob (shell, filenames)As a regex (program, text)
    *any run of characters, including nonezero or more of the previous item
    ?exactly one characterzero or one of the previous item
    .nothing special — a literal dotany single character
    [abc]one of a, b, cone of a, b, c — the same, for once
    ^ $nothing specialstart of line, end of line

    The row to memorise is the first one. Glob * stands alone and means "anything"; regex * is a repetition operator applied to whatever precedes it, so it is meaningless by itself. The regex that means what glob * means is .* — "any character, repeated zero or more times". Almost every "my grep found nothing" is this.

    Two more glob behaviours to know because they surprise people. First, a glob that matches nothing is, by default in bash, left alone and passed through literally — so ls *.nope hands ls the literal string *.nope and you get a confusing "No such file" error mentioning an asterisk. Second, globs do not match leading dots: * never matches .bashrc, which is exactly why rm * leaves dotfiles behind and why cp -r dir/* elsewhere/ silently omits configuration.

    Lab
    1. Watch the shell expand a glob before any program sees it. echo just prints its arguments, so it shows you precisely what the shell handed over.

      Those words are what any command would have received. The asterisk was gone before echo ran.

    3. See the no-match behaviour that produces baffling errors.

      ls is complaining about a file literally named with an asterisk, because that is what it was given.

    4. Now the regex side. . in a regex means "any character", so counting lines that contain one is counting non-empty lines. 
      grep -c '.' /etc/hostname

    5. Anchor a regex and use .* where a glob would have used *. This finds every line in /etc/passwd that starts with r and ends with sh. 
      grep '^r.*sh$' /etc/passwd

    6. Make the classic mistake deliberately, and watch it fail quietly rather than loudly — this is the one to remember. Unquoted, the shell expands the pattern against filenames first. 
      cd ~
touch alpha.log beta.log
grep error *.log
      grep: beta.log: No such file... (or silence, or the wrong file searched)

      What ran was grep error alpha.log beta.log — the shell turned your pattern into the first filename, so grep searched beta.log for the word alpha.log. The result is not an error; it is a wrong answer. Always quote a pattern meant for a program: grep 'error' *.log.

    7. Confirm globs skip dotfiles, which is worth seeing once so that rm * never surprises you. 
      mkdir -p /tmp/globtest && cd /tmp/globtest
touch visible.txt .hidden.txt
echo *
ls -a
      visible.txt
.  ..  .hidden.txt  visible.txt
    8. Clean up. 
      cd ~; rm -f alpha.log beta.log; rm -rf /tmp/globtest
    Verify

    Answer both without running them. (a) In grep '*.conf' /etc/fstab, what does grep actually search for? (A regex meaning "zero or more of nothing, then any character, then conf" — not what you meant.) (b) Which of ls *.conf and grep 'x' *.conf uses a glob, which uses a regex, and which uses both? (The first is a glob; the second is a regex and a glob, in the same line.)

    Gotchas

    find -name takes a glob, but find expands it itself, against each filename it walks past. So the pattern must be quoted — find /etc -name '*.conf' — or the shell will expand it against your current directory first and hand find something nonsensical. This single quoting rule is most of the reason find feels unreliable to beginners.

    There are two regex dialects in common use and they disagree about which characters need backslashes. grep defaults to basic regular expressions, where +, ?, | and () are literal and must be written \+, \? and so on. grep -E (or egrep) switches to extended, where they work as you would expect. When a pattern that looks right does not match, try -E before doubting yourself.

    Globs are matched by the shell against the current directory's contents, which is why the same script behaves differently depending on where it is run from — the same class of bug as the relative paths in B1.1, wearing a different hat.

    The useful thing to carry into Kubernetes is that its selectors are neither of these. A label selector like -l app=web,tier!=cache is an exact-match language with a small set of operators (=, !=, in, notin, exists) — there are no wildcards and no regexes in it at all. People type -l app=web*, get zero results and no error, and lose ten minutes.

    The place patterns do apply is on the client side, after kubectl has printed something: kubectl get pods | grep web is a regex against text, and kubectl get pods -o name | xargs … feeds a pipeline. W7.11 — output surgery is where the boundary between "the API filtered this" and "I filtered this with text tools" gets drawn properly, and it matters under time pressure: server-side selection is exact and fast, client-side grep is neither.

    Docs

    man 7 glob — the shell's pattern language, short and complete. man 7 regex — the other one, including the basic/extended distinction. Then man 1 grep, whose REGULAR EXPRESSIONS section is the practical reference you will come back to, and man 1 bash under Pathname Expansion for nullglob and dotglob if you want to change the two surprising behaviours above.

    B1.9

    The working core of the standard toolkit

    Why it matters

    There are eleven tools that appear in nearly every pipeline anyone writes, and the goal is not to learn them exhaustively — each has a manual page longer than this lesson — but to know the two or three flags of each that carry the weight. That is genuinely enough to build any answer you need, and the rest can be looked up in the seconds it takes to type man.

    The concept

    Every one of these reads lines of text and writes lines of text, which is what makes them interchangeable parts. Learn them by the question each one answers:

    ToolAnswersThe flags that matter
    grepwhich lines match this pattern?-i ignore case · -v invert · -c count · -n line numbers · -r recurse · -E extended regex
    sedreplace this with that, on every lines/old/new/ · add g for all on a line · -i edit the file in place · -n '5p' print one line
    awkdo something per line, using its fields'{print $2}' · -F: set the separator · '$3 > 100 {print}' filter · END{} for totals
    cuttake fixed columns out-d: -f1,3 — only useful when the separator is a single exact character
    sortorder these lines-n numeric · -r reverse · -k2,2 by field · -u drop duplicates · -h human sizes
    uniqcollapse adjacent duplicates-c count them · -d only duplicates — nearly always after sort
    wchow many?-l lines · -w words · -c bytes
    head / tailthe first / last few-n 20 · tail -f follow a growing file · tail -n +2 skip a header
    findwhich files match these properties?-name '*.log' · -type f · -mtime -1 · -size +10M · -exec … \;
    xargsturn a list of lines into arguments-n1 one at a time · -0 with find -print0 · -I{} place the argument

    Three combinations do so much of the real work that they are worth learning as single units rather than as compositions:

    • sort | uniq -c | sort -rn — the frequency count. Given any list of things, this tells you which occur most often. It is the answer to "which IP is hammering us", "which error is most common", "which image are most pods using".
    • find … -print0 | xargs -0 … — the safe bulk operation. find normally separates results with newlines, which breaks the moment a filename contains a space; -print0 separates with a zero byte instead, which cannot appear in a filename, and xargs -0 reads that format.
    • awk '{print $2}' — the field extractor, and the reason cut is less useful than it looks. awk treats any run of whitespace as one separator, so it handles the column-aligned output of ps, df and ss; cut -f2 on the same input fails because it wants exactly one delimiter character between fields.
    Lab
    1. Build the frequency count on real data — which shells are actually in use on this machine. cut -d: -f7 works here because /etc/passwd is genuinely colon-separated with no padding. 
      cut -d: -f7 /etc/passwd | sort | uniq -c | sort -rn
           25 /usr/sbin/nologin
      2 /bin/sync
      1 /bin/bash
      1 /bin/false
    2. Now the drill: every listening port with the program that owns it, one line each. Start by seeing the raw material. ss is the modern replacement for netstat, which is not installed on this image at all — -t TCP, -l listening only, -n numeric (do not resolve names), -p show the process. 
      sudo ss -tlnp
      State  Recv-Q Send-Q Local Address:Port  Peer Address:Port Process
LISTEN 0      4096       127.0.0.53%lo:53         0.0.0.0:*     users:(("systemd-resolve",pid=640,fd=14))
LISTEN 0      128              0.0.0.0:22         0.0.0.0:*     users:(("sshd",pid=882,fd=3))
LISTEN 0      128                 [::]:22            [::]:*     users:(("sshd",pid=882,fd=3))

      sudo matters: without it the Process column is empty, because reading which process owns a socket requires privilege.

    3. Reduce that to one clean line per port. tail -n +2 drops the header, awk takes the address field and the process field, and sed strips the surrounding noise from the process name. 
      sudo ss -tlnp | tail -n +2 | awk '{print $4, $6}' | sed 's/users:((//; s/)).*//; s/"//g' | sort -u
      0.0.0.0:22 sshd,pid=882,fd=3
127.0.0.53%lo:53 systemd-resolve,pid=640,fd=14
[::]:22 sshd,pid=882,fd=3

      Read the pipeline back as a sentence: list listening TCP sockets → drop the header → keep the address and process columns → strip the wrapper text → deduplicate. Five tools, one answer, none of which existed as a feature anywhere.

    4. Use find with a real question — configuration changed in the last day, which is the first thing you check after "it worked yesterday". 
      sudo find /etc -type f -mtime -1 | head -5
      /etc/hostname
/etc/resolv.conf
/etc/hosts
    5. Practise the space-safe pattern, on a filename designed to break the naive version. Compare the two outputs. 
      mkdir -p /tmp/spacetest && cd /tmp/spacetest
touch 'two words.txt' single.txt
find . -type f | xargs ls -l 2>&1 | tail -3
echo '--- now safely ---'
find . -type f -print0 | xargs -0 ls -l
      ls: cannot access './two': No such file or directory
ls: cannot access 'words.txt': No such file or directory
--- now safely ---
-rw-rw-r-- 1 ubuntu ubuntu 0 Jul 23 15:10 './two words.txt'
-rw-rw-r-- 1 ubuntu ubuntu 0 Jul 23 15:10 ./single.txt

      The first form split one filename into two arguments at the space. Note also how ls prints the name in single quotes — coreutils quotes any name containing a space when writing to a terminal, which is a helpful warning that the name is not what it looks like.

    6. Use awk for something cut cannot do: filter on a numeric field and total another. Here, filesystems over 10% full. 
      df -h | awk 'NR>1 && $5+0 > 10 {print $6, $5}'
      / 12%

      NR>1 skips the header line, $5+0 forces the percentage string into a number, and only matching lines reach the {print}.

    7. Clean up. 
      cd ~; rm -rf /tmp/spacetest
    Verify

    From memory, and without looking at step 3: produce one line per listening TCP port showing the port and the owning program. Any pipeline that gets there is correct. Then answer: why is sort almost always required before uniq, and why does find … | xargs need -print0/-0 to be trustworthy?

    Gotchas

    uniq only collapses adjacent identical lines — it does not deduplicate a list. Running it on unsorted input appears to work and quietly leaves duplicates behind, which is worse than failing. Either sort first, or use sort -u when you do not need the counts.

    On Ubuntu, /usr/bin/awk is chosen by the alternatives system, and both mawk and gawk are present on this image. They agree on everything in the table above, but not on extensions — a script using gensub() or asort() is gawk-only and will fail confusingly elsewhere. readlink -f "$(command -v awk)" tells you which one you have; stick to the common subset in anything you share.

    sed -i edits in place with no backup and no undo. Run the command without -i first and read the output — always. sed -i.bak keeps a copy, which costs nothing and has saved a great many afternoons.

    netstat and ifconfig are not on this machine, and this is not a mistake to work around by installing net-tools. They have been deprecated for a decade; ss and ip are the maintained replacements, they are what Kubernetes documentation and troubleshooting guides use, and every example on this site assumes them.

    These eleven tools are what node-level debugging is. When a pod cannot reach a Service, the ladder ends in a shell on the node running ss -tlnp to see what is actually listening, grep through the kubelet journal, and awk over iptables-save output to find the rule that is or is not there.

    W6.20 — the service-unreachable checklist is that ladder in order, and its lower rungs are exactly this toolkit. The specific command from step 2 recurs constantly: "is anything actually listening on that port, and what is it" is the question that separates a misconfigured Service from a crashed application, and no kubectl command answers it.

    Docs

    Read man 1 grep and man 1 find properly once — they repay it more than the others. Skim man 1 sed's Command Synopsis and stop; skim man 1 awk's Patterns and Actions and stop. Then man 8 ss for the socket states, man 1 xargs for -0 and -I, and man 1 sort. When you cannot recall a tool's name at all, apropos from B0.8 is still the right move.
