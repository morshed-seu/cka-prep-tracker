    Week 6 — Troubleshooting: the 30% week

    The highest-weighted domain, and the week where the internals from weeks 1–5 become money. The method is deliberate: for every failure class you break your own cluster (scripts in labs/faults/ do the breaking so you can't peek), fix it against the clock, and write down the signature — symptom → first command → root cause. By Friday you'll own a one-page triage sheet built from things you've personally repaired, and a 30-minute six-fault gauntlet score to prove it. Ticking done here also ticks the tracker.

    symptom→first command→root cause→fix→verify

    Toolbox & method

    6.1

    The universal first moves: events, describe, logs, wide

    Why it matters

    Every troubleshooting task starts the same way, whatever is broken. Four commands cover the opening move of ~80% of them; making them reflexes buys you the first minute of every exam task for free.

    The concept

    Events are real API objects (kind Event, stored like everything else) that components write when something notable happens: scheduler placed a pod, kubelet pulled an image, probe failed, eviction started. Two things about them shape the commands:

    • They expire — the apiserver keeps them for 1 hour by default (--event-ttl). "No events" on an old broken pod means the evidence aged out, not that nothing happened. Recreate the pod to regenerate it.
    • Default kubectl get events ordering is effectively arbitrary — always sort: --sort-by=.metadata.creationTimestamp.

    kubectl describe is an aggregator: object spec + status + the events that reference this object, at the bottom — the single highest-value screen in Kubernetes. -o wide on pods adds the node and pod IP, which is how you spot the pattern "every broken pod is on node02" in one glance.

    Lab
    1. Manufacture two different failures to practice on: 
      kubectl run brokenimg --image=nginx:1.99
kubectl run crasher --image=busybox:1.36 -- sh -c 'echo FATAL: config missing; exit 1'
    2. Move 1 — the cluster-wide timeline (your alias from 0.x is worth it): 
      kubectl get events -A --sort-by=.metadata.creationTimestamp | tail -8
      default  Warning  Failed     pod/brokenimg  Failed to pull image "nginx:1.99": ... not found
default  Warning  BackOff    pod/crasher    Back-off restarting failed container
default  Normal   Pulled     pod/crasher    Successfully pulled image "busybox:1.36"

      One command, whole cluster, chronological. When a task says "something in namespace X is broken", this is the map.

    3. Move 2 — describe, and read from the bottom up: 
      kubectl describe pod brokenimg | tail -6
        Warning  Failed   12s   kubelet  Failed to pull image "nginx:1.99": ...
  manifest for nginx:1.99 not found: manifest unknown
  Normal   BackOff  11s   kubelet  Back-off pulling image "nginx:1.99"
    4. Move 3 — logs, and the flag that matters. Current logs of a crash-looping container are empty (that container just started); the evidence died with the previous one: 
      kubectl logs crasher --previous
      FATAL: config missing
    5. Move 4 — wide, for the "which node?" dimension: 
      kubectl get pods -o wide
      NAME        READY  STATUS            RESTARTS   IP           NODE
brokenimg   0/1    ImagePullBackOff  0          10.244.1.17  node01
crasher     0/1    CrashLoopBackOff  2 (18s ago) 10.244.2.9  node02
    6. Keep both pods — 6.2 uses them.
    Verify

    For each broken pod you can say, from these four commands alone, what is failing and where the message names it — without having read the pod spec.

    Gotchas

    kubectl get events without -A only shows the current namespace — troubleshooting tasks love hiding the broken thing in a namespace you're not looking at. And describe on a deleted-and-recreated pod shows only the new pod's events; the old pod's story is gone with it.

    6.2

    Where logs physically live — the fallback ladder for when kubectl can't help

    Why it matters

    Control-plane-failure questions have a chicken-and-egg problem: the apiserver is down, so kubectl logs is down too. Knowing where logs are as files means nothing about a dead control plane can blind you.

    The concept

    kubectl logs is served by the kubelet: the apiserver proxies your request to the kubelet on the pod's node, and the kubelet reads the container's log file that the runtime writes under /var/log/pods/. Nothing about container logs requires the control plane. The ladder, from most to least infrastructure required:

    RungWorks whenCommand
    kubectl logsapiserver + kubelet both upkubectl logs pod [-c ctr] [--previous]
    crictl on the nodekubelet may be dead; runtime upsudo crictl ps -a → sudo crictl logs <id>
    raw files on the nodeeven the runtime is dead/var/log/pods/<ns>_<pod>_<uid>/<ctr>/0.log
    journalctlalways (for system services)journalctl -u kubelet, -u containerd

    /var/log/containers/*.log is a flat directory of symlinks into /var/log/pods/ with pod+container in the filename — the grep-friendly view. The kubelet itself is not a container, so its logs are in the journal, never under /var/log/pods.

    Lab
    1. Find crasher's (from 6.1) log file on its node. Get the node and UID first: 
      kubectl get pod crasher -o wide          # note the NODE column — node02 here
kubectl get pod crasher -o jsonpath='{.metadata.uid}'
      9f1c2ab4-8e77-4d1f-b2f3-6a0c94d2e1aa
    2. On that node, read the file the kubelet reads: 
      ssh node02
sudo ls /var/log/pods/default_crasher_9f1c2ab4-8e77-4d1f-b2f3-6a0c94d2e1aa/crasher/
sudo tail -2 /var/log/pods/default_crasher_*/crasher/0.log
      0.log
2026-07-12T09:14:02.71Z stdout F FATAL: config missing

      CRI log format: timestamp, stream, tag, then the line. This file is what kubectl logs returns.

    3. Same evidence via the runtime — this is your apiserver-down lifeline: 
      sudo crictl ps -a | grep crasher
sudo crictl logs $(sudo crictl ps -a --name crasher -q | head -1)
      d41f9c2ab77e  ...  Exited  crasher  ...
FATAL: config missing
    4. And the grep-friendly symlink view: 
      ls /var/log/containers/ | grep crasher
      crasher_default_crasher-d41f9c2ab77e....log
    5. Return to cp (exit) and clean up 6.1's pods: 
      kubectl delete pod brokenimg crasher
    Verify

    You produced the same log line three ways: kubectl logs --previous, crictl logs, and tail on the file — and can say which rung survives an apiserver outage (all but the first).

    Gotchas

    crictl needs root and talks to the runtime socket configured in /etc/crictl.yaml — if crictl itself errors with "connection refused" on the socket, that's not a crictl problem, that's containerd being down (a 6.11 fault signature). Also crictl ps without -a hides exited containers — for crash forensics you always want -a.

    Docs search: "Logging Architecture" (where files live) and "crictl" (Debugging Kubernetes nodes with crictl — has a command translation table).

    6.3

    kubectl debug: ephemeral containers for shell-less pods, node debug for host access

    Why it matters

    Production images increasingly ship with no shell, no ps, no curl — kubectl exec just fails. kubectl debug is the sanctioned answer, and it doubles as a way to get a host shell on any node through the API. Both appear in exam tasks and both are unusable if you first meet them under time pressure.

    The concept

    An ephemeral container is a container appended to a running pod's spec (via the pod's ephemeralcontainers subresource — a normal update can't do it). It shares the pod's network namespace automatically; with --target=<container> it also joins that container's process namespace, so its tools can see the target's processes and filesystem (via /proc/<pid>/root). It never restarts and can't be removed — it lives until the pod dies. That's the price of debugging in place.

    kubectl debug node/<name> instead creates a normal pod pinned to that node with the host filesystem mounted at /host — a host shell obtained purely through the API. Since v1.33 every kubectl debug takes --profile (what privileges the debug container gets); in v1.35 omitting it prints a deprecation warning about the old default, so type it: --profile=general for pod debugging, --profile=sysadmin for privileged node work.

    Lab
    1. A genuinely shell-less pod — the pause image has exactly one binary: 
      kubectl run quiet --image=registry.k8s.io/pause:3.10
kubectl exec quiet -- sh
      error: ... exec: "sh": executable file not found in $PATH
    2. Attach an ephemeral busybox that targets the app container: 
      kubectl debug -it quiet --image=busybox:1.36 --target=quiet --profile=general -- sh
      Targeting container "quiet".
/ #
    3. Prove you're inside its process namespace, then poke its filesystem: 
      ps aux
ls /proc/1/root/
      PID 1 ... /pause       ← the target's process, visible to your tools
pause                    ← and its root filesystem via /proc

      Exit with exit. Run kubectl get pod quiet -o jsonpath='{.spec.ephemeralContainers[*].name}' — the debugger is now a permanent part of the spec.

    4. Now a node shell, no ssh involved: 
      kubectl debug node/node01 -it --image=busybox:1.36 --profile=sysadmin -- sh
      Creating debugging pod node-debugger-node01-x7k2q with container debugger on node node01.
/ #
    5. The node's real filesystem is under /host — enter it fully with chroot: 
      chroot /host bash
crictl ps | head -3          # you are effectively root on node01
exit; exit
    6. Node debug leaves its pod behind — clean up both test objects: 
      kubectl get pods | grep node-debugger
kubectl delete pod quiet node-debugger-node01-x7k2q
    Verify

    You got a working shell in a pod where kubectl exec failed, saw the target's PID 1 from it, and ran crictl on node01 without ssh.

    Gotchas

    Without --target, the ephemeral container shares only the network namespace — ps shows nothing of the app and people conclude "debug is broken". For a pod that crashes too fast to attach to at all, the third mode is kubectl debug <pod> --copy-to=mypod --set-image=... — a mutated copy you can hold open. And remember the node debugger pod after you exit; graders and real clusters both dislike litter.

    Docs

    Docs search: "Debug Running Pods" — covers ephemeral containers, copies, and node sessions in one page.

    Control plane failures

    6.5

    The apiserver-down ladder

    Why it matters

    "kubectl doesn't work" is the scariest symptom and the most mechanical fix. The apiserver is a static pod: a file, a kubelet, a container, a listener on :6443. The ladder walks that chain in order, and each rung has a distinct answer.

    The concept
    kubectl fails→read the error→systemctl status kubelet→manifest present/valid?→crictl ps -a | grep apiserver→crictl logs / journalctl

    Rung 0 is reading the error text — it splits the fault space before you touch the node:

    kubectl saysMeaningPoints at
    connection refusedNothing listening on :6443apiserver container down — the ladder below
    timeout / i/o timeoutPackets go nowherewrong server address/firewall — check kubeconfig first
    TLS / x509 errorsListener up, trust brokencerts (6.9)
    401/403apiserver is fineyour credentials/RBAC — not an outage at all
    etcdserver: request timed out / 500sapiserver up, storage sicketcd (6.8)

    Then the chain: is the kubelet running (it's what turns manifests into pods)? Is the manifest present in /etc/kubernetes/manifests/ and parseable (a YAML typo means the pod silently never spawns — the error lands in the kubelet journal, not in any pod log)? Did the container start, and if it keeps exiting, what do its logs say?

    Lab
    1. Break it the honest way — take the manifest away and watch the kubelet kill the pod: 
      ssh cp   # if not already there
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sleep 20; kubectl get nodes
      The connection to the server 192.168.64.4:6443 was refused - did you specify the right host or port?
    2. Walk the ladder. Rung 1 — the kubelet: 
      systemctl status kubelet | head -3
      ● kubelet.service - kubelet: The Kubernetes Node Agent
     Active: active (running)       ← kubelet fine; look downstream
    3. Rung 2 — the manifest: ls /etc/kubernetes/manifests/ → no kube-apiserver.yaml. Rung 3 — the container: 
      sudo crictl ps -a | grep kube-apiserver
      (nothing — no container exists; consistent with "no manifest")
    4. Fix = put the manifest back; verify with the readiness endpoint, not vibes: 
      sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/
sleep 30; kubectl get --raw='/readyz?verbose' | tail -3
      [+]shutdown ok
readyz check passed
    5. File the signature on your triage sheet: refused + kubelet active + no container → manifest missing/unreadable; refused + container repeatedly Exited → read crictl logs (that's 6.6).
    Verify

    kubectl get --raw=/readyz returns ok, and you can recite the ladder rungs with what each one rules out.

    Gotchas

    The kubelet rescans the manifest directory continuously but recovery still takes 20–60s — don't stack a second "fix" on top of one that's still booting; watch crictl ps instead of re-editing. And a manifest with broken YAML produces no pod and no pod logs at all — the only witness is journalctl -u kubelet | grep -i apiserver.

    Docs

    Docs search: "Troubleshooting kubeadm" and "Debug Cluster" — bookmark-worthy pages for every lesson in this group.

    6.6

    Four planted apiserver faults, four distinct fingerprints

    Why it matters

    The exam's control-plane tasks are exactly this: someone broke kube-apiserver.yaml, go fix it. The same file breaks four different ways, and each way has a different fingerprint. Learn to read the fingerprint and the fix takes two minutes.

    The concept

    Memorize this table — it's the whole lesson:

    Fault in the manifestcrictl ps -a showsEvidence lives inSignature line
    wrong image tagno new container at alljournalctl -u kubeletFailed to pull image … not found
    typo'd/unknown flagstarts, exits instantly, restartscrictl logsError: unknown flag: --… (first lines)
    wrong etcd endpointruns for a while, then dies/restartscrictl logs (the end)storm of connection refused to the bogus address
    bad cert/key pathexits instantlycrictl logsopen /etc/kubernetes/pki/…: no such file

    The discriminator logic: no container → the kubelet couldn't even create it (image, or manifest YAML) → journal. Instant exit → flag parsing or file loading, named in the first log lines. Lives then dies → it started serving and something downstream (etcd) is unreachable — read the last lines, and note the port it's dialing.

    Lab

    Four drills, scripts do the planting so you diagnose blind. Take them one per sitting, and after each one write the fingerprint into your triage sheet before reading the reveal.

    1. Fetch the scripts onto cp: 
      BASE=https://raw.githubusercontent.com/morshed-seu/cka-prep-tracker/main/labs/faults
for i in 01 02 03 04; do curl -fsSLO "$BASE/w6-apiserver-$i.sh"; done
    2. Run a drill and work it — target under 10 minutes each, then aim for 5: 
      sudo bash w6-apiserver-01.sh     # type 'break' to confirm
    3. Diagnose using 6.5's ladder, fix the manifest, verify kubectl get --raw=/readyz → ok. Only then open the reveals below.

    w6-apiserver-01.sh

    Stage 1 · Expected symptom

    kubectl → connection refused within ~30s. crictl ps -a shows kube-apiserver containers Exited with the attempt count climbing — start, die, back off, retry.

    Stage 2 · Hint

    Instant exit means it never got as far as serving. The process itself tells you why it quit — read the first lines of sudo crictl logs <newest-exited-id>.

    Stage 3 · Full fix

    Log says Error: unknown flag: --allow-privilegedd. The manifest's --allow-privileged flag was typo'd. Fix:

    sudo vim /etc/kubernetes/manifests/kube-apiserver.yaml   # repair the flag spelling
sleep 30; kubectl get --raw=/readyz

    Fingerprint filed: instant exit + "unknown flag" → flag typo.

    w6-apiserver-02.sh

    Stage 1 · Expected symptom

    Slower onset than 01. The apiserver container shows as Running in crictl for tens of seconds at a time (or kubectl even answers briefly), then it dies and restarts. kubectl mostly refused, occasionally hanging.

    Stage 2 · Hint

    "Runs a while, then dies" = it's waiting on a dependency. Read the end of crictl logs and look hard at every address it's trying to reach. Then ask: is that thing actually broken, or is the apiserver looking in the wrong place? (sudo crictl ps | grep etcd — etcd is fine.)

    Stage 3 · Full fix

    Logs storm with connection refused dialing 127.0.0.1:23790. etcd listens on 2379; the manifest's --etcd-servers got an extra digit. Fix the flag, wait, verify /readyz. Fingerprint: runs-then-dies + refused-to-an-address → check the address before blaming the target.

    w6-apiserver-03.sh

    Stage 1 · Expected symptom

    Like 01: refused within ~30s, containers exiting instantly on a restart loop.

    Stage 2 · Hint

    First lines of crictl logs again — but this time it's not a flag problem, it's a file problem. Whatever file it names: ls the directory and see what's really there.

    Stage 3 · Full fix

    Log: open /etc/kubernetes/pki/apiserver-tls.crt: no such file or directory. ls /etc/kubernetes/pki/ shows the real name is apiserver.crt — the manifest's --tls-cert-file points at a file that doesn't exist. Fix the path (never rename the cert to match the typo!), verify. Fingerprint: instant exit + "no such file" → path fault; trust the filesystem, fix the manifest.

    w6-apiserver-04.sh

    Stage 1 · Expected symptom

    Refused, and — the key observation — crictl ps -a shows no fresh kube-apiserver container at all: nothing running, nothing recently exited. There are no container logs to read because there is no container.

    Stage 2 · Hint

    When the runtime has nothing, ask the component that creates containers: sudo journalctl -u kubelet --since '-5m' | grep -iE 'pull|image'.

    Stage 3 · Full fix

    Journal: Failed to pull image "registry.k8s.io/kube-apiserver:v1.35.x-broken" … not found. The manifest's image tag was sabotaged. Correct tag: check your backup, or read the version from a healthy component (grep image: /etc/kubernetes/manifests/kube-controller-manager.yaml — control-plane images share the version). Fix, verify. Fingerprint: no container anywhere → kubelet journal, image or manifest parse error.

    Verify

    All four drills fixed with /readyz ok, and your triage sheet now has four one-line fingerprints in your own words.

    Gotchas

    Do not "fix" by restoring the backup file — that repairs the cluster but wastes the drill; diagnose to the exact line first. In the exam the equivalent shortcut doesn't exist. Also: after editing a static-pod manifest, changes apply on kubelet's next sync — if nothing happens in 60s, you probably broke the YAML itself (journal, not logs).

    Docs search: "kube-apiserver" (component reference — flag spellings live here) and "Troubleshooting kubeadm".

    6.7

    Break the scheduler and controller-manager; learn their failure smell

    Why it matters

    These two fail quietly: kubectl works, nodes are Ready, dashboards are green — and yet nothing new happens in the cluster. Recognizing "healthy-looking cluster that won't act" saves you from debugging the wrong layer entirely.

    The concept

    From week 1: the scheduler's only job is filling in spec.nodeName on pending pods; the controller-manager's loops turn desired state into objects (Deployment → ReplicaSet → Pods, plus node lifecycle, endpoints, SA tokens…). So:

    Dead componentSmellInstant check
    schedulernew pods stay Pending with no events at all (nobody even looked at them)kubectl -n kube-system get pods → scheduler crashlooping; describe pod → empty event list
    controller-managerDeployments accept changes but no ReplicaSets/pods appear; scale does nothing; nodes never go NotReady either (its node-lifecycle loop is dead too)kubectl get rs after creating a deploy → nothing; CM pod crashlooping

    The discriminator vs 6.15: Pending with events ("Insufficient cpu", "untolerated taint") means the scheduler is alive and refusing — a workload problem. Pending with an empty event list means no scheduler ran at all.

    Lab
    1. Fetch and run the scheduler drill on cp: 
      BASE=https://raw.githubusercontent.com/morshed-seu/cka-prep-tracker/main/labs/faults
curl -fsSLO "$BASE/w6-scheduler-01.sh" && sudo bash w6-scheduler-01.sh
    2. Work it: prove the smell (kubectl run smoke --image=nginx → Pending, describe → no events), find the crashlooping component, read its logs, fix the manifest, verify smoke goes Running.
    3. Then the controller-manager drill: 
      curl -fsSLO "$BASE/w6-controller-manager-01.sh" && sudo bash w6-controller-manager-01.sh
    4. Work it the same way; verify with a deployment reaching 2/2. Reveals below when done.

    w6-scheduler-01.sh

    Stage 1 · Expected symptom

    smoke sits Pending forever; kubectl describe pod smoke shows no events whatsoever. Meanwhile kubectl -n kube-system get pods shows kube-scheduler-cp in CrashLoopBackOff.

    Stage 2 · Hint

    The scheduler is a static pod but the apiserver is fine — so plain kubectl -n kube-system logs kube-scheduler-cp works. What file can't it find, and who told it to look there?

    Stage 3 · Full fix

    Logs: invalid configuration: … open /etc/kubernetes/schedu1er.conf: no such file or directory. The manifest's --kubeconfig flag path was typo'd (1 for l). Fix /etc/kubernetes/manifests/kube-scheduler.yaml, wait for the pod to go Running, and watch smoke get scheduled retroactively — pending pods are picked up as soon as a scheduler exists. Clean up smoke.

    w6-controller-manager-01.sh

    Stage 1 · Expected symptom

    kubectl create deploy smoke --image=nginx --replicas=2 succeeds, but kubectl get rs,pods shows nothing was created. kube-controller-manager-cp is crashlooping in kube-system.

    Stage 2 · Hint

    Instant-exit pattern from 6.6 — the process names its problem in the first log lines. kubectl -n kube-system logs kube-controller-manager-cp.

    Stage 3 · Full fix

    Error: unknown flag: --use-service-account-credentialz. Fix the spelling in /etc/kubernetes/manifests/kube-controller-manager.yaml. Within a minute the deployment controller catches up and smoke gets its ReplicaSet and 2/2 pods — reconciliation resumes from desired state, nothing is lost. Clean up smoke.

    Verify

    Both components back to Running 1/1, smoke workloads created and cleaned up, and two new rows on the triage sheet: Pending + zero events → scheduler; objects don't cascade → controller-manager.

    Gotchas

    Because the apiserver stays up, these faults are visible from kubectl — resist the urge to ssh and crictl first. The fallback ladder exists for when kubectl is dead; when it isn't, kubectl is faster. Also note both components write leader-election leases — kubectl -n kube-system get leases shows a stale renewTime for the dead one; a nice secondary confirmation.

    Docs search: "kube-scheduler" / "kube-controller-manager" component pages for flag spellings.

    6.8

    Stop etcd and map exactly what still works

    Why it matters

    etcd outages produce the most confusing symptom mix in Kubernetes: some commands answer, some hang, apps keep serving traffic. Having watched it once, calmly, you'll recognize the mix instantly instead of chasing ghosts.

    The concept

    The apiserver is stateless; etcd holds everything. With etcd down: every write fails (create/update/delete — including the scheduler's binding and controllers' status updates, so the control plane is fully paralyzed even where kubectl still answers). Reads are a mixed bag — the apiserver keeps an in-memory watch cache and may briefly serve list/get from it, but anything needing a fresh quorum read errors. And crucially, the data plane doesn't notice: kubelets keep running pods, kube-proxy keeps forwarding, DNS keeps answering — the cluster coasts on its last known state. That's the headline: etcd down ≠ workloads down.

    Lab
    1. Put a victim workload up first, then stop etcd by removing its manifest (on cp): 
      kubectl create deploy coast --image=nginx --replicas=2
kubectl get pods -o wide          # note a pod IP, e.g. 10.244.1.31
sudo mv /etc/kubernetes/manifests/etcd.yaml /tmp/
sleep 25
    2. Probe the API — try a write, then reads, and note the difference: 
      kubectl run probe --image=nginx
kubectl get pods
      Error from server: etcdserver: request timed out        ← writes fail, named culprit
(get: may answer from cache once, then errors/hangs)
    3. Prove the data plane is coasting — the running pod still serves: 
      curl -s --max-time 2 10.244.1.31 | grep title
      <title>Welcome to nginx!</title>       ← workloads don't need etcd
    4. Read the apiserver's side of the story: 
      sudo crictl ps | grep apiserver          # still Running — it doesn't crash
sudo tail -3 /var/log/pods/kube-system_kube-apiserver-cp_*/kube-apiserver/0.log
      ... transport: Error while dialing: dial tcp 127.0.0.1:2379: connect: connection refused

      Same "refused to :2379" storm as fault 6.6/02 — but this time etcd really is down, the address is right. The log line is identical; checking whether the target is actually healthy is what distinguishes them.

    5. Restore and verify: 
      sudo mv /tmp/etcd.yaml /etc/kubernetes/manifests/
sleep 40; kubectl get pods; kubectl delete deploy coast; kubectl delete pod probe --ignore-not-found
    Verify

    You observed all three tiers with your own commands: writes fail with an etcd-naming error, reads degrade, running pods keep serving. Triage row: kubectl 500s mentioning etcdserver → etcd, not apiserver.

    Gotchas

    Don't confuse this with 6.6/02: same log signature, opposite fix. Wrong endpoint → fix the apiserver flag; real etcd outage → fix etcd (manifest, data dir, certs — and if data is gone, the 2.16–2.18 restore drill and the etcd card are the play). Diagnose which side is lying before editing anything.

    Docs search: "Operating etcd clusters for Kubernetes".

    6.9

    The expired-certificate scenario: recognize, renew, restart

    Why it matters

    kubeadm's leaf certificates live one year. Clusters that skip upgrades hit the wall, everything TLS breaks at once, and the errors look apocalyptic. The fix is three commands — if you recognize the signature.

    The concept

    The trust map from week 2: every control-plane connection is mTLS with certs under /etc/kubernetes/pki/, client creds embedded in the *.conf kubeconfigs, and the kubelet's client cert at /var/lib/kubelet/pki/ (auto-rotating since 2.8). The failure signatures:

    Where you see itSignatureMeaning
    kubectlx509: certificate has expired or is not yet valida cert in the chain is out of date (client or server side)
    kubelet journalpart of the existing bootstrap client certificate … is expiredkubelet's client cert expired (rotation was off/blocked)
    apiserver logTLS handshake errors from peerssome client (kubelet, CM, scheduler) presenting an expired cert

    Recovery is always the same shape: inventory (kubeadm certs check-expiration) → renew (kubeadm certs renew <name>|all — re-signs with the same CA, which lives 10 years) → restart the components that loaded the old certs (static pods don't watch their cert files) → refresh admin.conf-derived kubeconfigs if renewed.

    When kubectl itself is locked out by an expired admin.conf, kubeadm certs renew still works — it needs only the local CA files, not the API.     Why it matters

    6.12

    Read node Conditions fluently: MemoryPressure, DiskPressure, PIDPressure

    Why it matters

    Pressure conditions are the kubelet warning you it's about to start killing pods (3.18's eviction machinery). Exam tasks ask you to identify why pods were evicted or won't schedule onto a node — the conditions plus their matching taints are the answer key.

    The concept

    Each condition maps to an eviction signal, a threshold, and a taint the node-lifecycle controller adds while the condition is True:

    ConditionSignal (default hard threshold)Taint while TrueKubelet's reaction
    MemoryPressurememory.available < 100Mi…/memory-pressure:NoScheduleevict pods (BestEffort first — the 3.18 QoS ladder)
    DiskPressurenodefs.available < 10% or imagefs.available < 15% or nodefs.inodesFree < 5%…/disk-pressure:NoScheduleimage/container GC first, then evict pods with local storage
    PIDPressurepid.available low…/pid-pressure:NoScheduleevict; usually means a fork bomb somewhere
    Ready=False/Unknown—not-ready/unreachable:NoExecute(6.10/6.13 territory)

    Connect the dots for the exam narrative: threshold crossed → condition True → taint blocks new pods → kubelet evicts existing pods by QoS class → evicted pods show Reason: Evicted with the signal named in their status.

      kubectl get --raw "/api/v1/nodes/node02/proxy/configz" | python3 -m json.tool | grep -A6 evictionHard

    6.16

    CrashLoopBackOff mastery: --previous, lastState, and the exit-code alphabet

    Why it matters

    CrashLoopBackOff is the most common workload state on the exam and in life. The pros' edge is knowing it isn't an error — it's the kubelet pacing restarts — and knowing the two places the actual evidence lives.

    The concept

    The kubelet restarts a dying container with exponential backoff: 10s → 20s → 40s → … capped at 5m (reset after 10 minutes of stable running). "CrashLoopBackOff" is the waiting room between attempts. Evidence lives in: (1) the previous container's logs — kubectl logs --previous; (2) the pod's lastState.terminated — reason and exit code. The exit-code alphabet:

    CodeMeaningTypical story
    0clean exitcommand finished — wrong restartPolicy or a one-shot in a Deployment
    1app errorbad config, unreachable dependency — read the logs
    126 / 127not executable / not foundtypo'd command: — fails before the app exists, logs often empty
    137SIGKILL (128+9)OOMKilled, or kubelet killed it (failed liveness) — disambiguate below
    139SIGSEGVthe app crashed hard
    143SIGTERM (128+15)graceful shutdown — usually eviction/rollout, not a bug

    The 137 disambiguation: OOMKilled → lastState.terminated.reason: OOMKilled (the kernel did it — memory limit); liveness kill → events say Liveness probe failed … Killing container (the kubelet did it). Same restart, opposite fixes: raise the limit / fix the leak, vs fix the probe or the slow startup.

    Lab
    1. An OOM crasher — 64Mi limit, then deliberately hold ~128M in memory: 
      kubectl run oom --image=busybox:1.36 --restart=Never \
  --overrides='{"spec":{"containers":[{"name":"oom","image":"busybox:1.36","command":["sh","-c","sleep 5; cat /dev/zero | head -c 128m | tail"],"resources":{"limits":{"memory":"64Mi"}}}]}}'

      oom   0/1   OOMKilled   1 (10s ago)
{"containerID":"...","exitCode":137,"finishedAt":"...","reason":"OOMKilled", ...}
    2. A liveness-kill victim — healthy app, probe aimed at a dead port: 
      kubectl run livekill --image=nginx \

    6.17

    Image failures: ErrImagePull, ImagePullBackOff, and the three root causes

    Why it matters

    Image faults are the most-planted workload fault because they're easy to inject and have a satisfyingly exact fix. Knowing the three sub-causes by their describe-text means never guessing.

    The concept

    First, the two statuses are one fault: ErrImagePull is the moment of failure, ImagePullBackOff is the kubelet waiting (same backoff idea as 6.16) before retrying — you'll see them alternate. The three root causes, distinguished by the registry's answer in the events:

    Describe saysRoot causeFix
    manifest … not found: manifest unknowntag/name typo — registry answered "no such thing"correct the image reference
    dial tcp: lookup … no such host (or timeout)wrong registry host / no network pathfix the registry part of the reference
    401 Unauthorized / pull access denied / authentication requiredprivate image, no credentialsimagePullSecrets

    The credentials wiring (memorize the two commands as a pair):

    kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com --docker-username=dev \
  --docker-password=S3cret --docker-email=dev@example.com
kubectl patch serviceaccount default -p '{"imagePullSecrets":[{"name":"regcred"}]}'
# (or per-pod: spec.imagePullSecrets: [{name: regcred}])
    Lab
    1. Cause 1 — tag typo, in a Deployment so you also drill the fix motion: 
      kubectl create deploy imgfix --image=nginx:1.99 --replicas=2
kubectl get pods -l app=imgfix
kubectl describe pod -l app=imgfix | grep -A1 Failed | head -3
      imgfix-...   0/1   ImagePullBackOff   0
  Warning  Failed ... Failed to pull image "nginx:1.99": ... manifest unknown
    2. The exam-speed fix — no YAML editing: 
      kubectl set image deploy/imgfix nginx=nginx:1.29
kubectl rollout status deploy/imgfix
      deployment "imgfix" successfully rolled out
    3. Cause 2 — bogus registry host: 
      kubectl run imghost --image=registry.nowhere.example/nginx
sleep 20; kubectl describe pod imghost | grep -o 'lookup.*no such host' | head -1
      lookup registry.nowhere.example ... no such host
    4. Cause 3 — private image (a real registry answering "prove it"): 
      kubectl run imgauth --image=ghcr.io/some-org/private-app:v1
sleep 20; kubectl describe pod imgauth | grep -iE 'denied|unauthor|forbidden' | head -1
      ... denied: ... (or 401 Unauthorized / authentication required)

      This is the cue for the regcred wiring above. In the exam the task supplies server/user/password — your job is remembering the create secret docker-registry shape and where imagePullSecrets goes.

    5. Clean up: 
      kubectl delete deploy imgfix; kubectl delete pod imghost imgauth
    Verify

    Three different registry answers seen in describe output, and the imgfix deployment repaired via kubectl set image + rollout status — the two-command fix motion.

    Gotchas

    For a bare pod, image is one of the few mutable fields — kubectl set image pod/imghost imghost=nginx fixes it in place, no delete needed; the kubelet retries immediately with the new reference. Also: a node that pulled the image before may run it from cache while other nodes fail — "works on node01, ImagePullBackOff on node02" often means credentials/network, not the tag.

    6.18

    Why it matters

    These three don't announce themselves with red statuses — the pod looks almost fine. They're favorite exam material precisely because the signal is one column or one field deep.

    The concept
    Quiet symptomWhat it meansWhere to look
    Running but 0/1 READYreadiness probe failing → endpoint pulled from every Service (4.11's chain)describe → probe failure events; then the probe spec vs reality

    6.20

    The service-unreachable checklist, in strict order

    Why it matters

    "App A can't reach service B" is guaranteed exam material. The checklist works because it follows the 4.10–4.11 machinery in order — each step isolates one component, so you never debug two layers at once.

    The concept

    ObservationMeaningComponent at fault
    slice empty + curl refused instantlykube-proxy's REJECT rule for endpoint-less servicesselector mismatch, or no Ready pods (steps 2/4)
    slice populated + curl refusedDNAT delivered to a closed porttargetPort ≠ what the container listens on (step 3)

    4. Where does kubectl logs physically read from, and what are your two fallbacks when the apiserver — or even the kubelet — is dead?
      Answer

      The kubelet serves it from /var/log/pods/<ns>_<pod>_<uid>/<ctr>/0.log (runtime writes it). apiserver dead → sudo crictl ps -a + crictl logs on the node; runtime dead too → read the files directly. Kubelet's own logs: journalctl, never /var/log/pods (6.2).

    5. Exit code 137 — the two suspects, and how you tell them apart.
      Answer

      137 = SIGKILL. Kernel OOM-kill → lastState.terminated.reason: OOMKilled (over the memory limit). Kubelet liveness-kill → events: "Liveness probe failed … Killing container". Same code, opposite fixes: memory limit/leak vs probe/startup (6.16).

    6. Node NotReady, Conditions all say Unknown with "Kubelet stopped posting node status". What happened, and what's your command sequence?
      Answer

      Heartbeats stopped — the node went silent (Unknown ≠ reporting-a-problem). describe node (taints too) → ssh → sudo -i → systemctl status kubelet → journalctl -u kubelet → fix → exit ×2 → verify Ready (6.10).

    7. Three kubelet-won't-start journal signatures: wrong --config path, swap on, containerd down. Match each to its systemctl status.
      Answer

      Wrong config path → activating (auto-restart), "failed to load kubelet config file … no such file". Swap → auto-restart, "running with swap on is not supported". containerd down → kubelet active (running)(!) but node NotReady, CRI socket refused in journal, and crictl fails identically (6.11).

    8. Recite the node-death timeline with numbers, and name two pod types that never reschedule.
      Answer

      Lease renewals every 10s stop → ~40s: NotReady + unreachable:NoExecute taint → +300s (injected toleration) → eviction, controllers make replacements; dead-node pods hang as Terminating ghosts. DaemonSet pods and static pods never move (6.13/1.22).

    9. Service unreachable, wget says "connection refused". EndpointSlice empty vs populated — what does each mean and what do you check next?
      Answer

      Empty slice → kube-proxy programmed a REJECT (no ready endpoints): check selector-vs-labels, then pod readiness. Populated + refused → DNAT lands on a closed port: targetPort vs what the container listens on. Timeout instead of refused → NetworkPolicy/CNI (6.20).

    10. A distroless pod has no shell — kubectl exec fails. Exact command to get debugging tools inside it, and why the --target flag matters?
      Answer

      kubectl debug -it <pod> --image=busybox:1.36 --target=<container> --profile=general -- sh. --target joins the target's process namespace so ps/proc see the app; without it you share only the network namespace. Ephemeral containers can't be removed — they live till the pod dies (6.3).

    11. A pod is stuck Terminating on a healthy node. Cause and fix? Same status on a node that died — different fix?
      Answer

      Healthy node → a finalizer nobody clears: kubectl patch pod X --type=merge -p '{"metadata":{"finalizers":null}}'. Dead node → the kubelet can't confirm deletion; --force --grace-period=0 is the sanctioned cleanup for exactly that case (6.18/6.13).

    12. The four-command DNS ladder, from memory.
      Answer

      (1) busybox one-shot: nslookup kubernetes.default + read its resolv.conf; (2) kube-dns svc + endpointslices; (3) CoreDNS pods (-l k8s-app=kube-dns); (4) their logs. Healthy: answers from 10.96.0.10, slice has endpoints, 2× Running, logs quiet (6.21).
