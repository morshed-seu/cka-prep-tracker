    Week 6 — Troubleshooting: the 30% week

    This week is about fixing things that are broken — reading a symptom, chasing it down to the exact root cause, and applying the smallest fix that solves it, on a clock. It's the highest-weighted domain on the exam blueprint, and the week where all the internals you built up in weeks 1–5 finally become money: you can't troubleshoot a system you don't understand, and every fault below only makes sense in terms of some mechanism from an earlier week. The method here is deliberate: for every failure class you'll break your own cluster yourself (scripts in labs/faults/ do the actual breaking, precisely so you can't peek at the answer first), fix it against the clock, and write down the signature — the reliable pattern of symptom → first command → root cause that lets you recognize the same fault instantly next time. By Friday you'll own a one-page triage sheet built entirely from failures you personally diagnosed and repaired with your own hands, plus a timed 30-minute six-fault gauntlet score to prove the reflexes actually stuck. Ticking done here also ticks the tracker.

    symptom→first command→root cause→fix→verify

    Toolbox & method

    6.1

    The universal first moves: events, describe, logs, wide

    Why it matters

    When something in your cluster is broken, you don't start by reading YAML or guessing — you start by asking the cluster what it already noticed, in the same four moves every time, whatever the failure. Every troubleshooting task starts the same way: four commands cover the opening move of ~80% of them, and making them reflexes buys you the first minute of every exam task for free.

    The concept

    An event here isn't a vague notion — it's a real API object, stored the same way a Pod or Deployment is (kind Event), that a component writes the moment something notable happens: the scheduler placed a pod, the kubelet pulled an image, a probe failed, an eviction started. Because they're just objects in the same store as everything else, you can list and read them with the commands you already know. Two things about them shape the commands you actually run:

    • They expire — the apiserver keeps them for 1 hour by default (a setting called --event-ttl). So "no events" on an old broken pod doesn't mean nothing happened; it means the evidence aged out of the 1-hour window. Recreate the pod and the story starts fresh.
    • Default kubectl get events ordering is effectively arbitrary — it's not chronological unless you ask. Always sort: --sort-by=.metadata.creationTimestamp.

    kubectl describe is best thought of as an aggregator: it pulls together an object's spec (what you asked for), its status (what's actually true), and — at the very bottom — the events that reference this object specifically. That combination makes it the single highest-value screen in Kubernetes: one command, three layers of truth. -o wide on pods adds two extra columns, the node and the pod's IP, which is how you spot a pattern like "every broken pod happens to be on node02" in one glance instead of cross-referencing separately.

    Lab
    1. Manufacture two different failures to practice on: 
      kubectl run brokenimg --image=nginx:1.99
kubectl run crasher --image=busybox:1.36 -- sh -c 'echo FATAL: config missing; exit 1'
    2. Move 1 — the cluster-wide timeline (your alias from 0.x is worth it): 
      kubectl get events -A --sort-by=.metadata.creationTimestamp | tail -8
      default  Warning  Failed     pod/brokenimg  Failed to pull image "nginx:1.99": ... not found
default  Warning  BackOff    pod/crasher    Back-off restarting failed container
default  Normal   Pulled     pod/crasher    Successfully pulled image "busybox:1.36"

      One command, whole cluster, chronological. When a task says "something in namespace X is broken", this is the map.

    3. Move 2 — describe, and read from the bottom up: 
      kubectl describe pod brokenimg | tail -6
        Warning  Failed   12s   kubelet  Failed to pull image "nginx:1.99": ...
  manifest for nginx:1.99 not found: manifest unknown
  Normal   BackOff  11s   kubelet  Back-off pulling image "nginx:1.99"
    4. Move 3 — logs, and the flag that matters. Current logs of a crash-looping container are empty (that container just started); the evidence died with the previous one: 
      kubectl logs crasher --previous
      FATAL: config missing
    5. Move 4 — wide, for the "which node?" dimension: 
      kubectl get pods -o wide
      NAME        READY  STATUS            RESTARTS   IP           NODE
brokenimg   0/1    ImagePullBackOff  0          10.244.1.17  node01
crasher     0/1    CrashLoopBackOff  2 (18s ago) 10.244.2.9  node02
    6. Keep both pods — 6.2 uses them.
    Verify

    For each broken pod you can say, from these four commands alone, what is failing and where the message names it — without having read the pod spec.

    Gotchas

    kubectl get events without -A only shows the current namespace — a real trap, since troubleshooting tasks love hiding the broken thing in a namespace you weren't even looking in. And describe on a pod that got deleted and recreated only shows the new pod's events, because it's a different object with a different history — the old pod's story is gone the moment it's replaced, not just hidden.

    6.2

    Where logs physically live — the fallback ladder for when kubectl can't help

    Why it matters

    kubectl logs feels like a single, simple command, but it secretly depends on several pieces of the control plane being alive at once — which creates a chicken-and-egg problem exactly when you need logs the most: control-plane-failure questions ask you to diagnose a broken apiserver, but a broken apiserver means kubectl logs is down too. Knowing where logs physically live as files, independent of any of that, means nothing about a dead control plane can leave you blind.

    The concept

    kubectl logs is served by the kubelet, not the apiserver directly: the apiserver just proxies your request over to the kubelet running on the pod's node, and that kubelet reads the container's log file — a plain file that the container runtime (the software actually running containers, like containerd) writes under /var/log/pods/. Nothing about that chain requires the control plane at all once you're standing on the node. That gives you a ladder of fallbacks, from most to least infrastructure required — reach for the next rung down only when the one above it isn't available:

    RungWorks whenCommand
    kubectl logsapiserver + kubelet both upkubectl logs pod [-c ctr] [--previous]
    crictl on the nodekubelet may be dead; runtime upsudo crictl ps -a → sudo crictl logs <id>
    raw files on the nodeeven the runtime is dead/var/log/pods/<ns>_<pod>_<uid>/<ctr>/0.log
    journalctlalways (for system services)journalctl -u kubelet, -u containerd

    /var/log/containers/*.log is a flat directory of symlinks into /var/log/pods/, with the pod name and container name baked into the filename — a grep-friendly shortcut view rather than a separate copy of the data. The kubelet itself is not a container (it's the process that starts and watches containers, so it can't very well be one), so its own logs live in the systemd journal (read with journalctl) and never appear under /var/log/pods — that directory is only ever for the things the kubelet runs, not the kubelet itself.

    Lab
    1. Find crasher's (from 6.1) log file on its node. Get the node and UID first: 
      kubectl get pod crasher -o wide          # note the NODE column — node02 here
kubectl get pod crasher -o jsonpath='{.metadata.uid}'
      9f1c2ab4-8e77-4d1f-b2f3-6a0c94d2e1aa
    2. On that node, read the file the kubelet reads: 
      ssh node02

      0.log

      CRI log format: timestamp, stream, tag, then the line. This file is what kubectl logs returns.

    3. Same evidence via the runtime — this is your apiserver-down lifeline: 
      sudo crictl ps -a | grep crasher
sudo crictl logs $(sudo crictl ps -a --name crasher -q | head -1)
      d41f9c2ab77e  ...  Exited  crasher  ...

    4. And the grep-friendly symlink view: 
      ls /var/log/containers/ | grep crasher

      kubectl delete pod brokenimg crasher
    Verify

    You produced the same log line three ways: kubectl logs --previous, crictl logs, and tail on the file — and can say which rung survives an apiserver outage (all but the first).

    Gotchas

    crictl needs root, and it talks to the container runtime over a Unix socket path configured in /etc/crictl.yaml — so if crictl itself errors with "connection refused" on that socket, don't blame crictl; that error means containerd is down (a signature you'll meet again in 6.11). Also worth remembering: crictl ps without -a hides exited containers by default, the same way docker ps does — for crash forensics you always want -a, or the very container you're hunting for is invisible.

    6.3

    kubectl debug: ephemeral containers for shell-less pods, node debug for host access

    Why it matters

    Production images increasingly ship stripped down to just the application binary — no shell, no ps, no curl — because smaller images are more secure and faster to pull. That's great for security, terrible for debugging: kubectl exec just fails, because there's no shell inside to exec into. kubectl debug is Kubernetes' sanctioned answer to that problem, and as a bonus the same command doubles as a way to get a shell on any node's host, entirely through the API, no ssh key required. Both show up in exam tasks, and both are genuinely confusing the first time you meet them — better that first time isn't under a countdown clock.

    The concept

    An ephemeral container is a container bolted onto a pod that's already running, added via a special API path (the pod's ephemeralcontainers subresource — a normal edit-the-pod update can't add one; it needs this dedicated mechanism). By default it only shares the pod's network namespace (so it can reach the same addresses the app can) — but add --target=<container> and it also joins that specific container's process namespace, the Linux mechanism that controls which processes a program can see and signal. Joining it means your debugging tools can now see the target container's running processes and, via /proc/<pid>/root, even browse its filesystem — without ever needing a shell to already exist inside that container. The price for this convenience: an ephemeral container never restarts and can't be removed once added — it lives until the whole pod dies, a permanent guest.

    kubectl debug node/<name> does something different: instead of touching an existing pod, it creates a brand new, ordinary pod pinned to that specific node, with the node's entire host filesystem mounted inside it at /host — effectively a host shell, obtained purely by asking the API for a pod, no ssh key needed. Since v1.33 every kubectl debug invocation takes a --profile flag that controls exactly what privileges the debug container gets; by v1.35, omitting it prints a deprecation warning about the old implicit default, so just type it explicitly: --profile=general for debugging inside a pod, --profile=sysadmin when you need genuinely privileged access to a node.

    Lab
    1. A genuinely shell-less pod — the pause image has exactly one binary: 
      kubectl run quiet --image=registry.k8s.io/pause:3.10
kubectl exec quiet -- sh
      error: ... exec: "sh": executable file not found in $PATH
    2. Attach an ephemeral busybox that targets the app container: 
      kubectl debug -it quiet --image=busybox:1.36 --target=quiet --profile=general -- sh
      Targeting container "quiet".
/ #
    3. Prove you're inside its process namespace, then poke its filesystem: 
      ps aux
ls /proc/1/root/
      PID 1 ... /pause       ← the target's process, visible to your tools
pause                    ← and its root filesystem via /proc

      Exit with exit. Run kubectl get pod quiet -o jsonpath='{.spec.ephemeralContainers[*].name}' — the debugger is now a permanent part of the spec.

    4. Now a node shell, no ssh involved: 
      kubectl debug node/node01 -it --image=busybox:1.36 --profile=sysadmin -- sh
      Creating debugging pod node-debugger-node01-x7k2q with container debugger on node node01.
/ #
    5. The node's real filesystem is under /host — enter it fully with chroot: 
      chroot /host bash
crictl ps | head -3          # you are effectively root on node01
exit; exit
    6. Node debug leaves its pod behind — clean up both test objects: 
      kubectl get pods | grep node-debugger
kubectl delete pod quiet node-debugger-node01-x7k2q
    Verify

    You got a working shell in a pod where kubectl exec failed, saw the target's PID 1 from it, and ran crictl on node01 without ssh.

    Gotchas

    Without --target, the ephemeral container only shares the network namespace — so ps inside it shows nothing of the app, and people conclude "debug is broken" when really they just forgot the flag that shares process visibility. For a pod that crashes too fast to even attach a debugger to, there's a third mode: kubectl debug <pod> --copy-to=mypod --set-image=... creates a mutated copy of the pod — same spec, different image or command — that you can hold open and poke at without racing the original's crash loop. And don't forget the node debugger pod after you exit its shell; it's a real pod sitting on that node, and both graders and real production clusters dislike litter left behind.

    6.4

    Start your one-page triage flowchart (you'll refine it all week)

    Why it matters

    A triage sheet is just a one-page cheat sheet mapping "symptom I see" to "the first few commands that get me to the root cause" — and the point of building your own, rather than reading someone else's, is what it does to you under pressure. Under exam adrenaline you will not reason carefully through a decision tree; you will do whatever is rehearsed and automatic. A one-page symptom → first-3-commands sheet, built by your own hands from faults you personally broke and fixed, is that rehearsal artifact.     6.5

    The apiserver-down ladder

    Why it matters

    The apiserver is the front desk of the whole cluster — every kubectl command, every controller, every kubelet talks to Kubernetes exclusively through it, so when it's unreachable, "kubectl doesn't work" is the scariest-feeling symptom you'll see all week. It's also, reassuringly, the most mechanical fix once you know the chain behind it: the apiserver runs as a static pod (a pod the kubelet on this one node starts directly from a file on disk, before there's even a cluster to talk to — the pattern from week 1's "building superintendent" analogy) — which means underneath the abstraction it's just a file, read by a kubelet, that starts a container, that opens a listening port at :6443. The ladder below walks that exact chain in order, and each rung has one distinct, unambiguous answer.

    The concept
    kubectl fails→read the error→systemctl status kubelet→manifest present/valid?→crictl ps -a | grep apiserver→crictl logs / journalctl

    Each arrow is a question you answer before moving to the next: is the kubelet even alive? if so, does the manifest file exist and parse? if so, did a container actually start from it? if so, what do its logs say?

    Rung 0, before you even touch the node, is reading the error text kubectl itself printed — that text alone splits the fault space into completely different directions:

    kubectl saysMeaningPoints at
    connection refusedNothing listening on :6443apiserver container down — the ladder below
    timeout / i/o timeoutPackets go nowherewrong server address/firewall — check kubeconfig first
    TLS / x509 errorsListener up, trust brokencerts (6.9)
    401/403apiserver is fineyour credentials/RBAC — not an outage at all
    etcdserver: request timed out / 500sapiserver up, storage sicketcd (6.8)

    Then walk the chain rung by rung. Is the kubelet running at all — it's the node agent that turns static-pod manifest files into actual running pods, so nothing downstream matters if it's dead? Is the manifest present in /etc/kubernetes/manifests/ and does it actually parse as valid YAML (a typo in the file means the pod silently never even gets created — and because no pod ever existed, the error lands only in the kubelet's own journal, not in any pod log you'd normally check)? Did the container itself start, and if it keeps exiting, what do its logs actually say about why?

    Lab
    1. Break it the honest way — take the manifest away and watch the kubelet kill the pod: 
      ssh cp   # if not already there
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sleep 20; kubectl get nodes
      The connection to the server 192.168.64.4:6443 was refused - did you specify the right host or port?
    2. Walk the ladder. Rung 1 — the kubelet: 
      systemctl status kubelet | head -3
      ● kubelet.service - kubelet: The Kubernetes Node Agent
     Active: active (running)       ← kubelet fine; look downstream
    3. Rung 2 — the manifest: ls /etc/kubernetes/manifests/ → no kube-apiserver.yaml. Rung 3 — the container: 
      sudo crictl ps -a | grep kube-apiserver
      (nothing — no container exists; consistent with "no manifest")
    4. Fix = put the manifest back; verify with the readiness endpoint, not vibes: 
      sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/
sleep 30; kubectl get --raw='/readyz?verbose' | tail -3
      [+]shutdown ok
readyz check passed
    5. File the signature on your triage sheet: refused + kubelet active + no container → manifest missing/unreadable; refused + container repeatedly Exited → read crictl logs (that's 6.6).
    Verify

    kubectl get --raw=/readyz returns ok, and you can recite the ladder rungs with what each one rules out.

    Gotchas

    The kubelet does rescan the manifest directory continuously, but recovery still takes 20–60 seconds even so — resist the urge to stack a second "fix" on top of one that's genuinely still booting; watch crictl ps for the container actually appearing instead of re-editing files impatiently. And remember: a manifest with broken YAML produces no pod and therefore no pod logs at all — there's nothing to crictl logs because nothing ever started — so the only witness to that failure is journalctl -u kubelet | grep -i apiserver, the kubelet complaining about the file it couldn't parse.

    6.6

    Four planted apiserver faults, four distinct fingerprints

    Why it matters

    The exam's control-plane tasks are exactly this scenario, over and over: someone (a script, or a grader) broke kube-apiserver.yaml, and your job is to fix it. The same one file breaks four genuinely different ways, and each way leaves a distinct, recognizable fingerprint in the evidence — so learning to read the fingerprint turns a potentially long investigation into a two-minute fix.

    The concept

    Memorize this table — it's the whole lesson. Each row answers the same question ("what do I see, and where do I look") for a different kind of break:

    Fault in the manifestcrictl ps -a showsEvidence lives inSignature line
    wrong image tagno new container at alljournalctl -u kubeletFailed to pull image … not found
    typo'd/unknown flagstarts, exits instantly, restartscrictl logsError: unknown flag: --… (first lines)
    wrong etcd endpointruns for a while, then dies/restartscrictl logs (the end)storm of connection refused to the bogus address
    bad cert/key pathexits instantlycrictl logsopen /etc/kubernetes/pki/…: no such file

    The discriminator logic, reasoned out rather than just memorized: no container at all means the kubelet never even got as far as creating one — that only happens for an image it can't pull, or a manifest it can't parse — so the evidence has to be in the kubelet's own journal, since no container process ever existed to log anything. Instant exit means a container process did start, ran its very first setup step, and immediately gave up — that's flag parsing or a missing file it needs on startup, and it always names the problem in the first few lines of its own logs, because that's the exact moment it failed. Lives for a while, then dies means it got far enough to start serving, and only afterward hit something it depends on that isn't reachable — etcd, almost always — so read the last lines of the log, not the first, and note carefully what address it was trying to reach when it gave up.

    w6-apiserver-01.sh

    Stage 1 · Expected symptom

    kubectl → connection refused within ~30s. crictl ps -a shows kube-apiserver containers Exited with the attempt count climbing — start, die, back off, retry.

    Stage 2 · Hint

    Instant exit means it never got as far as serving. The process itself tells you why it quit — read the first lines of sudo crictl logs <newest-exited-id>.

    Stage 3 · Full fix

    Log says Error: unknown flag: --allow-privilegedd. The manifest's --allow-privileged flag was typo'd. Fix:

    sudo vim /etc/kubernetes/manifests/kube-apiserver.yaml   # repair the flag spelling

    Fingerprint filed: instant exit + "unknown flag" → flag typo.

    w6-apiserver-02.sh

    Stage 1 · Expected symptom

    Slower onset than 01. The apiserver container shows as Running in crictl for tens of seconds at a time (or kubectl even answers briefly), then it dies and restarts. kubectl mostly refused, occasionally hanging.

    Stage 2 · Hint

    "Runs a while, then dies" = it's waiting on a dependency. Read the end of crictl logs and look hard at every address it's trying to reach. Then ask: is that thing actually broken, or is the apiserver looking in the wrong place? (sudo crictl ps | grep etcd — etcd is fine.)

    Stage 3 · Full fix

    Logs storm with connection refused dialing 127.0.0.1:23790. etcd listens on 2379; the manifest's --etcd-servers got an extra digit. Fix the flag, wait, verify /readyz. Fingerprint: runs-then-dies + refused-to-an-address → check the address before blaming the target.

    w6-apiserver-03.sh

    Stage 1 · Expected symptom

    Like 01: refused within ~30s, containers exiting instantly on a restart loop.

    Stage 2 · Hint

    First lines of crictl logs again — but this time it's not a flag problem, it's a file problem. Whatever file it names: ls the directory and see what's really there.

    Stage 3 · Full fix

    Log: open /etc/kubernetes/pki/apiserver-tls.crt: no such file or directory. ls /etc/kubernetes/pki/ shows the real name is apiserver.crt — the manifest's --tls-cert-file points at a file that doesn't exist. Fix the path (never rename the cert to match the typo!), verify. Fingerprint: instant exit + "no such file" → path fault; trust the filesystem, fix the manifest.

    w6-apiserver-04.sh

    Stage 1 · Expected symptom

    Refused, and — the key observation — crictl ps -a shows no fresh kube-apiserver container at all: nothing running, nothing recently exited. There are no container logs to read because there is no container.

    Stage 2 · Hint

    When the runtime has nothing, ask the component that creates containers: sudo journalctl -u kubelet --since '-5m' | grep -iE 'pull|image'.

    Stage 3 · Full fix

    Journal: Failed to pull image "registry.k8s.io/kube-apiserver:v1.35.x-broken" … not found. The manifest's image tag was sabotaged. Correct tag: check your backup, or read the version from a healthy component (grep image: /etc/kubernetes/manifests/kube-controller-manager.yaml — control-plane images share the version). Fix, verify. Fingerprint: no container anywhere → kubelet journal, image or manifest parse error.

    All four drills fixed with /readyz ok, and your triage sheet now has four one-line fingerprints in your own words.

    Gotchas

    Do not shortcut these drills by just restoring a backup file you happen to have — that repairs the cluster, sure, but it wastes the entire point of the exercise, which is training yourself to diagnose down to the exact broken line. In the real exam that shortcut doesn't exist anyway, since there's no backup to restore from. Also worth internalizing: after you edit a static-pod manifest, the change only takes effect on the kubelet's next sync of that directory — if nothing happens within about 60 seconds, the more likely explanation is that your edit itself broke the YAML syntax, and you should be reading the kubelet's journal (not the container's logs, since no container even started) to find out how.

    6.7

    Break the scheduler and controller-manager; learn their failure smell

    Why it matters

    Some control-plane components fail loudly — kubectl refuses to connect, and you know instantly something's wrong. The scheduler and controller-manager are the opposite: they fail quietly. kubectl works fine, nodes show Ready, every dashboard looks green — and yet nothing new ever actually happens in the cluster. Learning to recognize "this cluster looks perfectly healthy and yet refuses to act" as its own distinct symptom saves you from wasting time debugging the wrong layer entirely — chasing a workload problem when the real fault is one specific component quietly not doing its job.

    The concept

    Recall from week 1: the scheduler — the "seating host" that decides which table a new party sits at — has exactly one job, filling in spec.nodeName on pods that don't have a node yet. The controller-manager runs the reconcile loops (the "thermostats" that keep re-checking whether reality matches what you asked for) that turn desired state into real objects: a Deployment becomes a ReplicaSet becomes actual Pods, and separately it also handles node lifecycle, Service endpoints, and service-account tokens. Knock out either one, and the cluster keeps looking fine on the surface while a specific category of work silently stops:

    Dead componentSmellInstant check
    schedulernew pods stay Pending with no events at all (nobody even looked at them)kubectl -n kube-system get pods → scheduler crashlooping; describe pod → empty event list
    controller-managerDeployments accept changes but no ReplicaSets/pods appear; scale does nothing; nodes never go NotReady either (its node-lifecycle loop is dead too)kubectl get rs after creating a deploy → nothing; CM pod crashlooping

    The discriminator you'll need against 6.15's Pending-pod causes: Pending with events attached ("Insufficient cpu", "untolerated taint") means the scheduler is alive, looked at the pod, and is correctly refusing to place it — that's a workload problem, not a scheduler problem. Pending with an empty event list means no scheduler ever looked at the pod at all — that's the scheduler itself being down.

    Lab

    6.8

    Stop etcd and map exactly what still works

    Why it matters

    etcd — the cluster's filing cabinet, the one place every object's state actually lives — going down produces the single most confusing symptom mix in all of Kubernetes: some commands still answer, others hang or error, and meanwhile your actual applications keep serving traffic like nothing happened. Having watched that mix once, calmly, with your own eyes, means you'll recognize it instantly on the exam instead of chasing ghosts across three different layers.

    The concept

    The apiserver itself holds no data — it's stateless, a front desk with no filing cabinet of its own — and etcd is where every object's actual state lives. So when etcd goes down: every write fails, meaning every create, update, and delete — including things you wouldn't think of as "writes" at all, like the scheduler recording which node it picked, or a controller updating a status field — so the entire control plane is effectively paralyzed even in places where kubectl still technically answers something. Reads are a mixed bag: the apiserver keeps an in-memory watch cache (a local copy of recently-seen objects it maintains so it doesn't have to ask etcd for every single read), so it may briefly keep answering list/get requests straight from that cache — but anything that needs a guaranteed-fresh answer straight from etcd (a "quorum read", meaning enough etcd members agreeing to guarantee correctness) fails outright. And crucially — this is the headline fact worth remembering above all the rest — the data plane doesn't notice etcd is gone at all: kubelets keep running the pods they already know about, kube-proxy keeps forwarding traffic using the rules it already programmed, DNS keeps answering from what it already learned. The whole cluster just coasts along on its last known-good state. etcd down does not mean workloads down.

    Lab
    1. Put a victim workload up first, then stop etcd by removing its manifest (on cp): 
      kubectl create deploy coast --image=nginx --replicas=2

      <title>Welcome to nginx!</title>       ← workloads don't need etcd

      sudo crictl ps | grep apiserver          # still Running — it doesn't crash

If the endpoint address itself is wrong, you fix the apiserver's flag; if etcd is genuinely down, you fix etcd instead (its manifest, its data directory, its certs — and if the data itself is gone, that's the 2.16–2.18 restore drill and the etcd card). The log line alone can't tell you which situation you're in — you have to separately check whether etcd is actually running before you touch anything.

    6.9

    The expired-certificate scenario: recognize, renew, restart

    Why it matters

    A certificate, recall, is like a notarized ID card — it proves who you are to something else, and it's only trusted because the two of you already agree on who the notary (the Certificate Authority, or CA) is. kubeadm's day-to-day "leaf" certificates (the individual ID cards, as opposed to the CA's own long-lived master certificate) are only valid for one year by default. Clusters that go a year or more without an upgrade eventually hit that wall all at once: every certificate expires within days of each other, every TLS connection in the cluster starts failing simultaneously, and the resulting wall of errors looks like the whole cluster collapsed. It hasn't — the fix is three commands, once you recognize the signature for what it is instead of panicking.

    The concept

    Recall the trust map from week 2: every control-plane connection is mTLS (mutual TLS — both sides present a certificate and verify each other, not just the client verifying the server like your browser does), with the certificates themselves stored under /etc/kubernetes/pki/, client credentials embedded inside the *.conf kubeconfig files, and the kubelet's own client certificate kept separately at /var/lib/kubelet/pki/ (which has auto-renewed itself since a change covered in 2.8 — the kubelet is actually the one component that doesn't suffer from this whole problem). Here's how that expiry shows up depending on where you're looking:

    Where you see itSignatureMeaning
    kubectlx509: certificate has expired or is not yet valida cert in the chain is out of date (client or server side)
    kubelet journalpart of the existing bootstrap client certificate … is expiredkubelet's client cert expired (rotation was off/blocked)
    apiserver logTLS handshake errors from peerssome client (kubelet, CM, scheduler) presenting an expired cert

    Recovery is always the same shape, and it's worth understanding why each step exists rather than just memorizing the sequence: inventory first (kubeadm certs check-expiration — see what's actually expired or close to it, don't guess) → renew (kubeadm certs renew <name>|all — this re-signs the leaf certificate using the same CA, which itself lives ten years and essentially never expires) → restart every component that already loaded the old certificate into memory, because a static pod reads its certificate file once at startup and never watches it for changes afterward — renewing the file on disk changes nothing for a process that's already running → and finally refresh any admin.conf-derived kubeconfig you're using yourself, since renewing the server's certificates doesn't touch your local copy of your own client certificate.

    Lab

    You can't fast-forward the clock safely, so drill the full recovery motion on healthy certs — identical commands, zero risk:

      sudo kubeadm certs check-expiration

      sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/ && sleep 5 \

    Gotchas

    Even when kubectl itself is completely locked out by an expired admin.conf, kubeadm certs renew still works fine — it only ever needs the local CA files sitting on disk, never a working connection to the API, so you're never actually stuck. And kubeadm upgrade quietly renews all certificates as a side effect of upgrading — which is exactly why clusters that get upgraded regularly never run into this problem at all; that fact is also the real exam-answer to "how would you prevent this from happening again".

    6.10

    The NotReady ladder, end to end

    Why it matters

    "A node is NotReady, fix it" is a near-certain exam task, worth real points, and — like the apiserver ladder — it's 100% mechanical once you've drilled the sequence enough that it's automatic. This lesson installs that ladder; 6.11 then varies what's actually broken behind it, so you get practice applying the same sequence to five different underlying problems.

    The concept
    kubectl get nodes→describe node (Conditions + taints)→ssh + sudo -i→systemctl status kubelet→journalctl -u kubelet→fix→verify Ready

    Check the status from the control plane first, only ssh onto the node once describe tells you it's worth it, then read outward from the service to its logs.

    The mechanics behind that NotReady status, recall from 1.22: the kubelet on a healthy node renews a Lease object (a lightweight "I'm still here" heartbeat, refreshed constantly so nobody has to guess) every 10 seconds. When those renewals stop arriving, the controller-manager waits about 40 seconds and then flips the node's Ready condition to Unknown, writing the exact message you'll see in describe: "Kubelet stopped posting node status." Notice what that message does not tell you — why the kubelet stopped posting. That why only exists on the node itself, inside systemd and the journal, which is precisely what the rest of this ladder exists to go read. Once you get there, there are really only two families of answer: either the kubelet service itself is unhealthy (stopped outright, or crash-looping because of bad configuration), or the kubelet process is running just fine but something it depends on is sick underneath it (the container runtime down, or the CNI plugin missing — remember "cni plugin not initialized" from 4.8).

    Lab
    1. Break node01 by hand this once (the scripts take over in 6.11). On node01: 
      ssh node01 'sudo systemctl stop kubelet'

      kubectl get nodes -w
      node01   Ready      <none>   28d   v1.35.2

    3. Rung 2 — interrogate from cp before ssh'ing: 
      kubectl describe node node01 | grep -A6 Conditions | head -8

      Ready   Unknown   ...   NodeStatusUnknown   Kubelet stopped posting node status.
Taints: node.kubernetes.io/unreachable:NoExecute
        node.kubernetes.io/unreachable:NoSchedule

      Conditions all Unknown (not False) = the node went silent — heartbeats stopped. The taints are the controller-manager preparing the 300s eviction clock (6.13).

    4. Rungs 3–5 — onto the node, exam discipline (6.14): 
      ssh node01

           Active: inactive (dead) since Sat 2026-07-12 10:02:11
Jul 12 10:02:11 node01 systemd[1]: Stopped kubelet: The Kubernetes Node Agent.

      Diagnosis: service stopped, journal shows a clean stop (no crash). Fix and leave:

      kubectl get nodes

    node01 Ready, unreachable taints gone (kubectl describe node node01 | grep Taints), and you ran the ladder in order without skipping to ssh first.

    Gotchas

    The distinction between condition value Unknown and False is diagnostic gold, worth internalizing precisely: Unknown means the kubelet has gone silent — nobody's heard from it, which is this whole ladder. Ready=False while other conditions look otherwise healthy means the opposite: the kubelet is still talking, it's just actively reporting a problem — read its message, because it names the cause directly (runtime, CNI, disk pressure), and that's a completely different first move than chasing a silent kubelet. Also worth knowing so you don't panic mid-drill: recovery removes the unreachable taint from the node, but it does not move pods that already got rescheduled elsewhere back to this node — a workload staying lopsided across nodes after recovery is expected, not a sign something's still broken.

    6.12

    Read node Conditions fluently: MemoryPressure, DiskPressure, PIDPressure

    Why it matters

    A node's pressure conditions are the kubelet's own early-warning system — it noticing it's genuinely running low on some resource (memory, disk, process slots) and saying so, right before it starts killing pods to relieve that pressure (the eviction machinery from 3.18). Exam tasks routinely ask you to figure out why pods on a node were evicted, or why new pods refuse to schedule there — and the pressure conditions, together with the matching taints they trigger, are the complete answer key for that question.

    The concept

    Each pressure condition maps to a specific measurable signal, a threshold that triggers it, and a taint (a marker on the node saying "don't schedule new pods here unless they specifically tolerate this") that the node-lifecycle controller adds automatically for as long as the condition stays True:

    ConditionSignal (default hard threshold)Taint while TrueKubelet's reaction
    MemoryPressurememory.available < 100Mi…/memory-pressure:NoScheduleevict pods (BestEffort first — the 3.18 QoS ladder)
    DiskPressurenodefs.available < 10% or imagefs.available < 15% or nodefs.inodesFree < 5%…/disk-pressure:NoScheduleimage/container GC first, then evict pods with local storage
    PIDPressurepid.available low…/pid-pressure:NoScheduleevict; usually means a fork bomb somewhere
    Ready=False/Unknown—not-ready/unreachable:NoExecute(6.10/6.13 territory)

    Put those dots together and you get the whole exam narrative in one causal chain: a threshold gets crossed → the matching condition flips to True → the taint that appears blocks any new pod from landing here → meanwhile the kubelet starts evicting existing pods, working through them in QoS-class order (the priority ladder from 3.18 — least-guaranteed pods go first) → and any pod evicted this way shows up afterward with Reason: Evicted, with the specific signal that caused it named right there in its status.

    Lab
    1. Read the current conditions the fluent way: 
      kubectl describe node node02 | grep -A9 'Conditions:' | head -10
        MemoryPressure   False   ...   KubeletHasSufficientMemory

  PIDPressure      False   ...   KubeletHasSufficientPID
  Ready            True    ...   KubeletReady

      kubectl get --raw "/api/v1/nodes/node02/proxy/configz" | python3 -m json.tool | grep -A6 evictionHard
      "evictionHard": {
    "imagefs.available": "15%",
    "memory.available": "100Mi",
    "nodefs.available": "10%",
    "nodefs.inodesFree": "5%"
}
    3. Optional but memorable — manufacture real DiskPressure on node02 (safe: one file, deleted after). Size a file to push the root fs over 90% used: 
      ssh node02 'df -h / | tail -1'                    # note Avail, e.g. 4.2G

      node02   Ready   ...

      kubectl describe node node02 | grep -E 'DiskPressure|Taints'

      ssh node02 'sudo journalctl -u kubelet --since "-3m" | grep -iE "garbage|evict|pressure" | tail -3'

    You saw a pressure condition flip True with its taint, the kubelet's reclaim attempts in the journal, and the flip back after cleanup — and node02 stayed Ready throughout (pressure ≠ dead).

    Gotchas

    Pods that the kubelet evicts don't just vanish — they stay visible afterward as Failed/Evicted husks, so a kubectl get pods output littered with Evicted entries is itself a diagnostic signature all on its own, telling you a node hit pressure recently even before you go check conditions; describe one of those husks and it names the exact signal responsible ("The node had condition: [DiskPressure]"). Also worth remembering is a real asymmetry: the pressure taint only stops new pods from scheduling there — but DaemonSet pods (the ones meant to run on every node, like a log agent) tolerate pressure taints by default and simply stay put, which is intentional, not a bug.

    6.13

    Kill a node and narrate the whole death timeline

    Why it matters

    1.22 already gave you the raw numbers involved when a node disappears (10s / ~40s / 300s); this drill makes you actually watch those numbers tick by against real running workloads, so "node down, where are my pods now?" becomes a narration you can give confidently in your own words, rather than a fact you're trying to recall under pressure.

    The concept

    Here's the rehearsal script you're about to watch play out, and why each stage takes as long as it does: the node dies → its lease renewals (the 10-second heartbeats from 6.10) stop arriving → ~40 seconds later, the node-lifecycle controller gives up waiting and marks the node's Ready condition Unknown, adding a taint of unreachable:NoExecute → every pod on that node was quietly given a 300-second toleration for exactly this taint back when it was scheduled, so that toleration's countdown now starts → 300 seconds after that, those pods finally get a deletion timestamp — except the one component that's supposed to confirm they actually stopped, the kubelet on that node, is itself dead and can't confirm anything, so the pods hang forever as Terminating ghosts. Meanwhile their controllers — a ReplicaSet, say — don't wait around for that confirmation at all, and go ahead and create fresh replacement pods on healthy nodes immediately. Total elapsed time before your service is actually healed again: roughly 5 minutes 40 seconds of deliberately designed patience, built in so that a brief network blip doesn't cause needless pod churn. Two categories of pod never get this treatment at all: DaemonSet pods and static pods simply never move, because moving them to a different node would defeat their entire purpose.

    6.16

    CrashLoopBackOff mastery: --previous, lastState, and the exit-code alphabet

    Why it matters

    CrashLoopBackOff is the single most common workload state you'll see, both on the exam and in real production clusters. The edge experienced people have isn't some secret command — it's simply knowing that CrashLoopBackOff itself is not an error message at all, just the kubelet deliberately pacing itself between restart attempts, and knowing the two exact places where the real evidence of why it's crashing actually lives.

    The concept

    When a container keeps dying, the kubelet doesn't restart it instantly and repeatedly — it restarts it with exponential backoff, waiting a little longer each time: 10s → 20s → 40s → … capped at 5 minutes between attempts (and that backoff resets back to 10s once the container manages to stay up for 10 stable minutes). "CrashLoopBackOff" is just the name for the waiting room between one attempt and the next — it's telling you a schedule is being followed, not that something new broke. The actual evidence of what's wrong lives in exactly two places: (1) the previous attempt's container logs, fetched with kubectl logs --previous since the current container just started and hasn't logged anything relevant yet; (2) the pod's lastState.terminated field, which records the reason and exit code from that last attempt. That exit code is itself a compact vocabulary worth learning cold — an alphabet where each number tells its own story:

    CodeMeaningTypical story
    0clean exitcommand finished — wrong restartPolicy or a one-shot in a Deployment
    1app errorbad config, unreachable dependency — read the logs
    126 / 127not executable / not foundtypo'd command: — fails before the app exists, logs often empty
    137SIGKILL (128+9)OOMKilled, or kubelet killed it (failed liveness) — disambiguate below
    139SIGSEGVthe app crashed hard
    143SIGTERM (128+15)graceful shutdown — usually eviction/rollout, not a bug

    Exit code 137 deserves special attention because it has two entirely different causes that look identical at first glance — telling them apart is the actual skill here: OOMKilled means the Linux kernel itself killed the process for exceeding its memory limit, and you'll see lastState.terminated.reason: OOMKilled spelling that out directly; a liveness-probe kill means the kubelet made the decision instead, and you'll see it in the pod's events as Liveness probe failed … Killing container. Same restart behavior, same exit code, but opposite fixes: raise the memory limit or fix an actual memory leak in one case, versus fix the probe's configuration or the app's slow startup in the other — so always check which one actually happened before changing anything.

    Lab
    1. An OOM crasher — 64Mi limit, then deliberately hold ~128M in memory:

    Gotchas

    An OOM kill is triggered purely by the container's own configured memory limit, never by how much memory the node happens to have free — so a pod can get OOM-killed on a completely empty, otherwise idle node, which surprises people who assume OOM only happens under real resource contention (and recall 5.1's tmpfs trap: an emptyDir volume backed by tmpfs counts against that same memory limit too, another surprise source). And when logs --previous comes back empty, resist the urge to rerun it five times hoping for different output — empty logs is itself the data point, telling you the process likely never got far enough to log anything; go read lastState and the pod's events instead of retrying the same command.

    6.17

    Image failures: ErrImagePull, ImagePullBackOff, and the three root causes

    Why it matters

    Image faults are the most commonly planted workload fault on any exam or drill, for a simple practical reason: they're trivially easy to inject (just typo one string) and have a precise, unambiguous fix once diagnosed. Knowing the three distinct sub-causes by their exact describe-text means you never have to guess which one you're looking at.

    The concept

    First, a clarification that removes some confusion: the two statuses you'll see are really just one fault at two different moments, not two different problems. ErrImagePull is the instant of the pull actually failing; ImagePullBackOff is the kubelet waiting out a backoff delay (the exact same idea as 6.16's crash backoff) before it tries the pull again — so you'll see the pod's status alternate between the two as it keeps retrying. Underneath either status, there are three genuinely different root causes, and the registry itself tells you which one via what it says back in the pod's events:

    Describe saysRoot causeFix
    manifest … not found: manifest unknowntag/name typo — registry answered "no such thing"correct the image reference
    dial tcp: lookup … no such host (or timeout)wrong registry host / no network pathfix the registry part of the reference
    401 Unauthorized / pull access denied / authentication requiredprivate image, no credentialsimagePullSecrets

    The credentials wiring (memorize the two commands as a pair):

    kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com --docker-username=dev \
  --docker-password=S3cret --docker-email=dev@example.com
kubectl patch serviceaccount default -p '{"imagePullSecrets":[{"name":"regcred"}]}'
# (or per-pod: spec.imagePullSecrets: [{name: regcred}])
    Lab
    1. Cause 1 — tag typo, in a Deployment so you also drill the fix motion: 
      kubectl create deploy imgfix --image=nginx:1.99 --replicas=2
kubectl get pods -l app=imgfix
kubectl describe pod -l app=imgfix | grep -A1 Failed | head -3
      imgfix-...   0/1   ImagePullBackOff   0
  Warning  Failed ... Failed to pull image "nginx:1.99": ... manifest unknown
    2. The exam-speed fix — no YAML editing: 
      kubectl set image deploy/imgfix nginx=nginx:1.29
kubectl rollout status deploy/imgfix
      deployment "imgfix" successfully rolled out
    3. Cause 2 — bogus registry host: 
      kubectl run imghost --image=registry.nowhere.example/nginx
sleep 20; kubectl describe pod imghost | grep -o 'lookup.*no such host' | head -1
      lookup registry.nowhere.example ... no such host
    4. Cause 3 — private image (a real registry answering "prove it"): 
      kubectl run imgauth --image=ghcr.io/some-org/private-app:v1
sleep 20; kubectl describe pod imgauth | grep -iE 'denied|unauthor|forbidden' | head -1
      ... denied: ... (or 401 Unauthorized / authentication required)

      This is the cue for the regcred wiring above. In the exam the task supplies server/user/password — your job is remembering the create secret docker-registry shape and where imagePullSecrets goes.

    5. Clean up: 
      kubectl delete deploy imgfix; kubectl delete pod imghost imgauth
    Verify

    Three different registry answers seen in describe output, and the imgfix deployment repaired via kubectl set image + rollout status — the two-command fix motion.

    Gotchas

    Most fields on a running pod can't be changed in place — you'd have to delete and recreate it. Image is a rare exception: for a bare pod (one you created directly, not through a Deployment), image is actually one of the few mutable fields, so kubectl set image pod/imghost imghost=nginx fixes it right where it stands, no delete needed — the kubelet notices the change and retries the pull immediately with the corrected reference. Also worth knowing: a node that already pulled an image successfully in the past may keep running it from its local cache even while other nodes fail to pull it fresh — so "works fine on node01, ImagePullBackOff on node02" is a real pattern, and it usually points to credentials or network reachability being different between nodes, not a bad tag (since the tag would fail identically everywhere).

    6.18

    The quiet failures: not-Ready pods, stuck Terminating, init crashes

    Why it matters

    Everything so far in this week has announced itself with a loud, obvious status — CrashLoopBackOff, ImagePullBackOff, Pending. These three don't: the pod looks almost fine at a glance, and you have to look one column, or one field, deeper to notice anything's wrong at all. That's exactly why they're favorite exam material — the failure hides in plain sight rather than shouting for attention.

    The concept
    Quiet symptomWhat it meansWhere to look
    Running but 0/1 READYreadiness probe failing → endpoint pulled from every Service (4.11's chain)describe → probe failure events; then the probe spec vs reality
    stuck Terminatingdeletion blocked: a finalizer nobody clears, or a dead kubelet (6.13)metadata.finalizers and deletionTimestamp
    Init:CrashLoopBackOff / Init:Erroran init container fails; app containers never startlogs <pod> -c <init-name> — logs need the container name

    A word on finalizers, since they're the least familiar concept in this table: think of a finalizer as a pre-delete contract — a note attached to an object saying "before you actually remove this, some specific controller needs to do its own cleanup first." The API server honors that note by holding the object in Terminating limbo, refusing to fully delete it, until every listed controller has removed its own entry from the list to signal "I'm done." The trap is when the controller responsible for clearing its entry no longer exists — an operator got uninstalled, or a fault was deliberately planted — because then nothing will ever clear that entry, and the object hangs in Terminating forever. Clearing metadata.finalizers by hand is the break-glass move for exactly that dead end.

    Lab
    1. Quiet failure 1 — Running-but-not-Ready, with the service consequence on display: 
      kubectl run quiet1 --image=nginx -l app=quiet1 \

Verify READY 1/1 and the slice gains an endpoint.

    2. Quiet failure 2 — a pod that refuses to die: 
      kubectl run quiet2 --image=nginx

      Diagnose: kubectl get pod quiet2 -o jsonpath='{.metadata.finalizers}' → the blocker, and deletionTimestamp set. Break glass:

      kubectl patch pod quiet2 --type=merge -p '{"metadata":{"finalizers":null}}'
      pod/quiet2 patched   (and it vanishes instantly — the delete was already filed)
    3. Quiet failure 3 — init container crash: 
      kubectl apply -f - <<'EOF'

      kubectl delete pod quiet1 quiet3; kubectl delete svc quiet1

    All three quiet failures produced, diagnosed from their one-deep signal, and resolved — including one finalizer patch you can now type from memory.

    Gotchas

    Clearing finalizers by hand on real infrastructure objects — a PersistentVolume backed by real cloud storage, a namespace with cloud resources tied to it — can leave those external resources orphaned and still costing money or sitting around unmanaged, since the cleanup that finalizer was protecting never actually ran. In production, that makes it a genuine last resort you reach for only after asking why the responsible controller isn't doing its job; on the exam, in a lab environment with nothing real behind it, it's usually exactly what the task wants. And on init containers specifically: they run serially, one at a time in order, so only the first one that fails actually matters — the STATUS column's Init:1/2 notation tells you precisely how far the pod got before something stopped it.

    Network & DNS debugging

    6.20

    The service-unreachable checklist, in strict order

    Why it matters

    A Service, recall, is like a phone book entry — it gives you one stable name/address to dial, and something behind the scenes routes your call to whichever real pod should answer. "App A can't reach Service B" is guaranteed exam material, precisely because there are several genuinely different things that can go wrong between dialing that number and a pod actually answering. This checklist works because it walks the 4.10–4.11 machinery in the same order packets actually flow — each step isolates exactly one component, so you're never trying to debug two layers of the stack at once and confusing yourself about which one is actually at fault.

    The concept
    1 endpointslices non-empty?→2 selector = pod labels?→3 targetPort = containerPort?→4 pod Ready?→5 NetworkPolicy?→6 curl the ClusterIP

    Steps 1–4 are all questions you can answer without sending a single packet, just by reading API objects — only steps 5–6 involve real traffic, which is why they come last.

    Here's why this exact order matters, not just an arbitrary checklist: an empty EndpointSlice (the object recording which pod IPs currently back a Service) means kube-proxy — the component that actually programs the routing rules on every node — was told there is nobody to send traffic to, so it never even set up a path; that fault sits entirely upstream of networking, in the selector or the pod's readiness, and no amount of network debugging will find it. A populated EndpointSlice but a connection that's still refused means the routing rule fired correctly and delivered the packet — just to a port nobody's listening on there (a targetPort mismatch). Only once that whole control-plane picture checks out clean is it worth spending time chasing actual packets on the wire. The observation table below turns each of those distinct outcomes into a lookup:

    ObservationMeaningComponent at fault
    slice empty + curl refused instantlykube-proxy's REJECT rule for endpoint-less servicesselector mismatch, or no Ready pods (steps 2/4)
    slice populated + curl refusedDNAT delivered to a closed porttargetPort ≠ what the container listens on (step 3)
    curl times outpackets dropped in flightNetworkPolicy (step 5) or CNI/routing — 4.x territory

And don't forget step 4's implication: a Service can look entirely "broken" purely because its pods' readiness probes are failing (6.18) — EndpointSlices only ever include pods that are Ready, so the actual fix is on the pod, never on the Service object itself.

    6.21

    The DNS ladder as reflex: four commands from memory

    Why it matters

    DNS inside a cluster is what turns a Service's name into the address the network actually needs — you built and deliberately broke the full DNS ladder back in 4.18 to understand each layer of it. This week's job is different: pure compression. The same ladder now has to come out of your fingers in under two minutes flat, because on the real exam DNS is essentially never the whole task by itself — it's the obstacle sitting in front of some other task, something you need working before you can even attempt what was actually asked.

    The concept

    The same four commands as 4.18, but now memorized as a single block you can type without thinking, walking those rungs bottom-up:

    # 1 — is resolution broken, and what does the pod's resolv.conf say?
kubectl run tmp --rm -it --restart=Never --image=busybox:1.36 -- \
  sh -c 'nslookup kubernetes.default; cat /etc/resolv.conf'
# 2 — is anything behind the kube-dns VIP?
kubectl -n kube-system get svc kube-dns
kubectl -n kube-system get endpointslices -l k8s-app=kube-dns
# 3 — are the CoreDNS pods alive?
kubectl -n kube-system get pods -l k8s-app=kube-dns
# 4 — what do they say?
kubectl -n kube-system logs -l k8s-app=kube-dns --tail=10

    Healthy baseline to recognize in one glance: nslookup answers 10.96.0.10, resolv.conf's nameserver is 10.96.0.10 with the search default.svc.cluster.local svc.cluster.local cluster.local ladder, slice carries two endpoints, pods 2× Running, logs quiet. Anything else → you know from 4.18 which rung to dig into (empty slice → scale/readiness; CrashLoop → Corefile; cluster-names-only failing → kubernetes plugin; external-only failing → forward/upstream).

If you need the fuller dig/host tools, that's the jessie-dnsutils image from 4.17, but plain busybox is faster to type and always pullable, so default to it.     6.22

    Planted NetworkPolicy fault: app can't reach db — fix it minimally

    Why it matters

    A NetworkPolicy is a firewall rule scoped to Kubernetes labels instead of IP addresses — it says which pods are allowed to talk to which other pods. Policy faults are pure reading comprehension under time pressure: nothing crashes, nothing logs an error anywhere, packets simply vanish silently because a rule quietly didn't match what you expected. The exam version of this hands you a namespace with some policies already in place and one broken flow; the winning motion is methodically comparing selectors to actual labels, then making the smallest possible edit that fixes only what's asked.

    The concept

    The debugging chain for "X can't reach Y, and I suspect a policy is the cause" runs like this: the symptom itself is a timeout — never "connection refused", which is 4.24's signature for a completely different kind of problem, packets dropped silently rather than actively rejected; first confirm the ordinary service chain is otherwise entirely healthy (6.20's steps 1–4 all pass, so this genuinely isn't a Service problem); then run kubectl get netpol -A and read every single policy that selects the destination pod against the AND/OR rules from 4.25 — is the traffic's actual source matched by at least one from element (remembering that conditions within one element are ANDed together, while separate elements in the list are ORed)? The classic planted faults, in order of how often they show up: a label typo inside from.podSelector, a missing port in the rule, a misplaced list dash that silently changes AND into OR (or vice versa), or a policy sitting in the wrong namespace entirely — worth remembering explicitly that a NetworkPolicy can only ever select pods living in its own namespace, never reach across into another one.

    Lab prerequisite: policies need an enforcing CNI — redo 4.24's swap first (snapshot → flannel out → Calico in). The script below refuses to run without Calico present. When you're done with this lesson, restore the flannel snapshot as in 4.24 step 8.

    Week 6 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. kubectl says "connection refused … :6443". Recite the ladder — and what does each rung rule out?
      Answer

      Read the error (refused = nothing listening) → systemctl status kubelet (the manifest-runner) → manifest present/valid in /etc/kubernetes/manifests → crictl ps -a | grep apiserver (did a container start?) → crictl logs / journalctl -u kubelet. Each rung isolates one link of file → kubelet → container → listener (6.5).

    2. Four kube-apiserver.yaml faults — wrong image, unknown flag, wrong etcd endpoint, bad cert path. Give each one's fingerprint.
      Answer

      Wrong image → no container at all, evidence in the kubelet journal ("Failed to pull"). Unknown flag → instant exit, "unknown flag" in first log lines. Wrong etcd endpoint → runs a while then dies, log storm of refused-to-the-bogus-address. Bad cert path → instant exit, "no such file" (6.6).

    3. New pods Pending with zero events vs Pending with "Insufficient cpu" — two different diagnoses. Which and why?
      Answer

      Zero events = no scheduler even looked (scheduler down — check kube-system). "Insufficient cpu" = scheduler alive and correctly refusing (workload/capacity problem, 6.15). The presence of events is the discriminator (6.7).

    4. Where does kubectl logs physically read from, and what are your two fallbacks when the apiserver — or even the kubelet — is dead?
      Answer

      The kubelet serves it from /var/log/pods/<ns>_<pod>_<uid>/<ctr>/0.log (runtime writes it). apiserver dead → sudo crictl ps -a + crictl logs on the node; runtime dead too → read the files directly. Kubelet's own logs: journalctl, never /var/log/pods (6.2).

    5. Exit code 137 — the two suspects, and how you tell them apart.
      Answer

      137 = SIGKILL. Kernel OOM-kill → lastState.terminated.reason: OOMKilled (over the memory limit). Kubelet liveness-kill → events: "Liveness probe failed … Killing container". Same code, opposite fixes: memory limit/leak vs probe/startup (6.16).

    6. Node NotReady, Conditions all say Unknown with "Kubelet stopped posting node status". What happened, and what's your command sequence?
      Answer

      Heartbeats stopped — the node went silent (Unknown ≠ reporting-a-problem). describe node (taints too) → ssh → sudo -i → systemctl status kubelet → journalctl -u kubelet → fix → exit ×2 → verify Ready (6.10).

    7. Three kubelet-won't-start journal signatures: wrong --config path, swap on, containerd down. Match each to its systemctl status.
      Answer

      Wrong config path → activating (auto-restart), "failed to load kubelet config file … no such file". Swap → auto-restart, "running with swap on is not supported". containerd down → kubelet active (running)(!) but node NotReady, CRI socket refused in journal, and crictl fails identically (6.11).

    8. Recite the node-death timeline with numbers, and name two pod types that never reschedule.
      Answer

      Lease renewals every 10s stop → ~40s: NotReady + unreachable:NoExecute taint → +300s (injected toleration) → eviction, controllers make replacements; dead-node pods hang as Terminating ghosts. DaemonSet pods and static pods never move (6.13/1.22).

    9. Service unreachable, wget says "connection refused". EndpointSlice empty vs populated — what does each mean and what do you check next?
      Answer

      Empty slice → kube-proxy programmed a REJECT (no ready endpoints): check selector-vs-labels, then pod readiness. Populated + refused → DNAT lands on a closed port: targetPort vs what the container listens on. Timeout instead of refused → NetworkPolicy/CNI (6.20).

    10. A distroless pod has no shell — kubectl exec fails. Exact command to get debugging tools inside it, and why the --target flag matters?
      Answer

      kubectl debug -it <pod> --image=busybox:1.36 --target=<container> --profile=general -- sh. --target joins the target's process namespace so ps/proc see the app; without it you share only the network namespace. Ephemeral containers can't be removed — they live till the pod dies (6.3).

    11. A pod is stuck Terminating on a healthy node. Cause and fix? Same status on a node that died — different fix?
      Answer

      Healthy node → a finalizer nobody clears: kubectl patch pod X --type=merge -p '{"metadata":{"finalizers":null}}'. Dead node → the kubelet can't confirm deletion; --force --grace-period=0 is the sanctioned cleanup for exactly that case (6.18/6.13).

    12. The four-command DNS ladder, from memory.
      Answer

      (1) busybox one-shot: nslookup kubernetes.default + read its resolv.conf; (2) kube-dns svc + endpointslices; (3) CoreDNS pods (-l k8s-app=kube-dns); (4) their logs. Healthy: answers from 10.96.0.10, slice has endpoints, 2× Running, logs quiet (6.21).
