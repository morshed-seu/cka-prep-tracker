    Foundations — the machinery underneath the machinery

    Kubernetes doesn't invent its hard parts from scratch — it bolts together a handful of much older ideas from Linux, networking and cryptography. If you already know what a process, a certificate, or DNS is, skim this page once and skip straight to Week 0. If any of the eight topics below are new to you, read this page first: every week's lessons link back into a section here instead of re-explaining it, so the ten minutes you spend here pays off all eight weeks.

    Each section is short on purpose: the plain-language idea, one analogy, and exactly where it resurfaces in the lessons. The weekly lessons explain these same primitives again, in place, the moment each one first matters — this page just gathers the eight cross-cutting ones in one spot so you're not meeting all of them cold, spread across eight different weeks.

    1. Processes, and why a container isn't a tiny VM

    What's actually running when you "start a container"

    A process is just a program the kernel is running: it has memory, open files, and a process ID. Normally every process on a machine can see every other process, and they all share one filesystem and one network address. A container is an ordinary Linux process that the kernel has been told to partially blindfold:

    • Namespaces control what a process can see — its own process list, its own network interfaces, its own hostname, its own root filesystem. Six or seven of these, applied together, are what make a process feel like it's alone on a machine.
    • cgroups (control groups) control what a process can use — a CPU and memory budget the kernel enforces. This is where Kubernetes resources.limits actually gets enforced, several layers down.
    • A layered filesystem (overlayfs) is how a container gets "its own" files cheaply: a read-only image on the bottom, a thin writable layer on top, and only the top layer's changes are unique to that container.

    A process is a chef working in a shared restaurant kitchen. A container is that same chef given their own sealed-off cooking station (namespaces — they can't see the other chefs' pots), a strict ingredient ration (cgroups — they can't grab more butter than allotted), and a photocopy of the master recipe book they can scribble on without touching the original (overlayfs). It's still the same kitchen and the same kernel underneath — nothing was virtualized, just fenced off.

    This is also why a container starts in milliseconds and a VM takes seconds-to-minutes: there's no second kernel booting, no virtual hardware — just one kernel drawing more fences around an ordinary process. It's also why "the container" and "the pod" are different things: a pod is Kubernetes' idea of one or more containers that share some of those namespaces (network and IPC) on purpose, so they can talk over localhost.

    2. Talking to a server: APIs, and "declare it, don't do it"

    REST, and the declarative idea that runs the whole cluster

    Most programs you've used talk to a server over HTTP using a small, boring vocabulary: GET a thing, POST a new thing, PUT/PATCH change a thing, DELETE remove a thing. That pattern — resources identified by URLs, manipulated with those verbs — is called REST, and it's exactly what kubectl is doing under the hood: every command becomes one HTTP call to the apiserver.

    The more important idea is declarative vs imperative. Imperative is a list of instructions: "start a container, then another, then check they're up." Declarative is a statement of a desired end state: "there should be 3 replicas of this, forever" — and something else is responsible for continuously checking reality against that statement and nudging it back into line. Kubernetes is declarative almost everywhere: you write what you want in YAML, and controllers spend their whole lives comparing "what you want" against "what exists" and closing the gap. That loop — read desired state, read actual state, act, repeat — is usually called a reconcile loop or control loop, and grasping it early makes half of Kubernetes's vocabulary (controller, operator, watch, drift) click into place at once.

    Imperative is personally walking to the closet and grabbing a towel whenever the bathroom runs low. Declarative is pinning a note on the closet door — "always keep 3 towels in this bathroom" — and the housekeeping staff walk by periodically, glance at the note, count the towels, and top up if short. You never told them when to check or exactly what to carry; you only told them the rule, and they keep re-applying it forever. If a guest steals a towel, the next walk-by fixes it without you doing anything.

    3. YAML: how indentation becomes structure

    Reading (and not fearing) the files everything is written in

    Every Kubernetes object — a Pod, a Deployment, a Service — is described as a YAML document: a plain-text way of writing nested key: value data. There is no special punctuation to open or close a section like { } in JSON; indentation itself is the structure. Two rules cover almost everything you'll write:

    • key: value pairs, one per line; a value that's itself a set of keys is written on the following lines, indented further.
    • A list is a set of lines starting with - at the same indent level; each - is one list item, which can itself contain further key: value pairs indented past the dash.
    metadata:
  name: web        # nested under metadata, 2 spaces in
  labels:
    app: web        # nested one level further
spec:
  containers:
  - name: web       # first (and only) item in a list
    image: nginx

    Two consequences that explain almost every "my YAML won't apply" moment: indentation must be spaces, not tab characters (the parser rejects tabs outright), and sibling keys must line up at exactly the same column. This is why W0's vim setup turns Tab into two spaces before you write a single manifest.

    YAML is a hand-written outline in a notebook, the kind with sub-bullets indented under a heading. Nobody needs brackets to know that a sub-bullet belongs to the bullet above it — the indent alone says so. Push it in one column too far or too few, and the reader (or the parser) attaches it to the wrong parent.

    4. TCP/IP, ports and DNS in one page

    Addresses, doors, and the phone book that resolves names to addresses

    Two machines that want to talk need three things: an IP address (which machine), a port (which door on that machine — a number 0–65535, with well-known ones like 443 for HTTPS), and a protocol to structure the conversation — almost always TCP, which guarantees bytes arrive in order and nothing is silently dropped, at the cost of a short handshake before any data flows.

    Remembering IP addresses doesn't scale, so DNS exists to translate names into addresses: your machine asks a resolver "what's the address for example.com?" and gets back an IP to actually connect to. That lookup happens transparently before almost every network request you make, including ones you didn't know were happening. Kubernetes runs its own internal DNS server (CoreDNS) purely so that Service names like my-svc.my-namespace.svc.cluster.local resolve the same way — a Service's whole job is to be a stable name and address for a set of pods whose individual IPs change constantly.

    app asks: resolve my-svc→CoreDNS looks it up→returns a ClusterIP→app opens a TCP connection to that IP:port

    An IP address is a street address; a port is the apartment number at that address — same building, different doors, different tenants. DNS is directory assistance: you say a name you remember ("Alex's place"), and it hands you the street address to actually dial into your GPS. Kubernetes Services exist because pods are tenants who move apartments constantly (rescheduled, restarted, new IP every time) — the Service is the one thing with a fixed, memorable address that directory assistance always resolves correctly, no matter who's currently living behind it.

    5. Public-key crypto: certificates and TLS in plain terms

    Proving who you are without ever sharing a password

    Public-key cryptography gives every party a matching pair of keys: a private key they never share, and a public key anyone can have. Anything encrypted with the public key can only be decrypted with the private one (and vice versa for signing) — which means you can prove you hold a private key without ever revealing it, just by signing something and letting others check the signature with your public key.

    A certificate is a public key plus identity information (whose key is this, what's it allowed to be used for, when does it expire), signed by someone else's private key — a Certificate Authority (CA). If you already trust that CA's public key, and the CA's signature on the certificate checks out, you now trust the identity inside it too, without ever having met the party directly. That's the entire trust model behind TLS (what makes HTTPS "secure"): a short handshake where both sides present certificates, verify the signatures, and then derive a shared secret to encrypt everything that follows. Kubernetes runs its own tiny CA for exactly this reason: every component (apiserver, kubelet, etcd, you) gets a certificate signed by the cluster's CA, so any two of them can verify each other without a shared password anywhere.

    A certificate is a notarized ID card: it states who you are and is stamped by a notary (the CA). A stranger who has never met you can still trust the card, because they already trust the notary's stamp and can verify it wasn't forged. The TLS handshake is presenting that ID at a door, having the doorman check the stamp against a notary they recognize, and only then being let in and given a private channel to talk through.

    6. Distributed consensus: why etcd wants an odd number of members

    How several machines agree on one answer without a referee

    If only one machine holds the cluster's data, it's a single point of failure. Running several copies fixes that — but now they must all agree on the exact same sequence of changes, even if messages are delayed or a machine crashes mid-write, or the copies quietly diverge into different "truths". Consensus algorithms (etcd uses one called Raft) solve this: one member is elected leader, all writes go through it, and a write is only considered committed once a quorum — a strict majority of members — has confirmed they've stored it.

    That majority rule is exactly why cluster sizes are odd (1, 3, 5): with 3 members, quorum is 2, so the cluster survives losing 1 member and still has a majority to keep deciding. With 4 members, quorum is 3 — you'd tolerate the same single failure as a 3-member cluster but need one more machine to do it, for no extra safety. Odd sizes are simply the cheapest way to buy fault tolerance.

    Think of a small committee that must approve every decision unanimously-ish — really, by strict majority. Three members means any two agreeing is enough to act, so the committee keeps functioning even if one member is out sick. Add a fourth member and you now need three to agree — you've spent a whole extra seat without being able to survive any more absences than before. That's why committees (and etcd clusters) are kept at odd numbers.

    7. Linux networking primitives: namespaces, veth, bridges, iptables

    The plumbing every CNI plugin is quietly built from

    Kubernetes networking (Week 4) is heavy precisely because it's four Linux primitives stacked together, and nobody tells you the primitives first:

    • A network namespace (netns) is a private copy of the networking stack — its own interfaces, routes, and firewall rules, isolated from the host's. Every pod gets one; it's why a pod can have "its own" eth0 and IP.
    • A veth pair is two virtual network interfaces permanently wired together — anything sent in one instantly comes out the other. It's the cable that connects a pod's isolated netns back to the host, one end living in each.
    • A bridge is a virtual switch: plug several veth "cables" into it and it forwards traffic between whichever ones need to talk, the same way a physical switch connects several computers in an office.
    • iptables / netfilter is the kernel's rule engine for rewriting and filtering packets in flight — it's how a Service's single virtual IP gets silently rewritten into one of several real pod IPs, and how NetworkPolicies actually block traffic.

    A CNI plugin's whole job is wiring these four things together consistently across every node so that pod-to-pod traffic looks like one flat network, even though underneath it's a private room, a cable, and a switch, repeated per node.

    A network namespace is a hotel room with its own phone line — sealed off from every other room. A veth pair is the wire running from that room's phone jack out to the building's wiring closet. A bridge is the switchboard in that closet, connecting many rooms' wires so any two can be connected on demand. iptables is the switchboard operator quietly rerouting a call placed to "the front desk" (a Service IP) to whichever actual room (pod) is on duty right now — the caller never knows the room number changed.

    8. Process supervision: systemd, and the static-pod escape hatch

    Who starts the thing that starts everything else

    On a modern Linux box, systemd is process ID 1 — the first thing the kernel starts, and the thing responsible for starting (and restarting, if they crash) every long-running background program, called a service or unit. Network daemons, SSH, and — on a Kubernetes node — the kubelet are all systemd services, meaning systemd is what keeps the kubelet itself alive.

    But the kubelet is also supposed to run the rest of the control plane (apiserver, etcd, scheduler, controller-manager) — and those are normally started by the apiserver's own scheduling logic, which is a chicken-and-egg problem the very first time a cluster boots: nothing can schedule the apiserver before the apiserver exists. The escape hatch is the static pod: the kubelet watches one local directory on disk and directly runs whatever pod YAML it finds there, with no apiserver involved at all. So the real boot order on a control-plane node is: systemd starts the kubelet (the only service systemd needs to know about) → the kubelet reads the static pod manifests off disk and starts etcd, the apiserver, the scheduler, and the controller-manager itself → only after that does the rest of the cluster's usual apiserver-mediated machinery become available.

    systemd is a building superintendent who unlocks the building every morning and personally keeps the boiler and the main power switch running — those can't wait for a tenant management office to open, because the tenant office needs power too. Static pods are utility rooms the superintendent watches and restarts directly off a posted maintenance schedule, rather than through the normal tenant work-order system (the apiserver) — because that system's own office is one of the rooms being kept alive this way.
