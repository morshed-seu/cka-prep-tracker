    B2 — Processes, threads, signals

    A container is a process. Not "like" a process, not "a lightweight virtual machine" — a perfectly ordinary Linux process, with some of its settings adjusted. That single sentence is the reason this module exists, and it is the one that makes the rest of the track make sense: everything Kubernetes does to a workload is something Linux has been doing to processes since the 1970s. Start it. Watch it. Ask it politely to stop. Wait a while. Give up and kill it. Report why it died.

    So this module is where you learn to narrate all of that, precisely, on a machine you can see all of. When you later read Exit Code: 137 in a pod's status at three in the morning, you will not be looking up what 137 means — you will already know that it is 128 + 9, that 9 is SIGKILL, that nothing inside the program chose to exit, and therefore that the answer is outside the program.

    What a process is

    B2.1

    Define a process precisely — four things and a number

    Why it matters

    "Process" is one of those words everybody uses and few people can define, which is fine until you need to reason about one that is misbehaving. If your mental model is "a running program", you have no way to answer questions like why does this container see a different /etc/hosts, or how can two processes share memory, or what exactly does a container add. A four-part definition answers all three, and every later module in this track modifies exactly one of the four parts.

    The concept

    Start with the distinction that the word hides. A program is a file sitting on disk — /usr/bin/sleep is a program, and it is just bytes, doing nothing. A process is one running instance of a program: the kernel has read those bytes into memory, given them somewhere to live, and started executing them. One program can have a thousand processes running from it at once, and they do not interfere with each other in the slightest.

    In our hotel picture, a process is a guest staying in the hotel, and its PID is their room number. That gives you the two most-missed properties for free. Room numbers are unique while someone is in the room — no two guests share 412 today — but they get reused: next month a different guest is in 412, and a note addressed to "whoever is in 412" reaches the wrong person. Exactly the same is true of PIDs, and it is the reason you will see careful scripts check which process a PID is before signalling it.

    Now the definition. A process is four things plus that number:

    PartPlainlyWhere you can see it
    Address spaceits private memory — code, data, stack, and anything it mapped in/proc/PID/maps
    File descriptorsa numbered table of everything it has open: files, sockets, pipes, terminals/proc/PID/fd/
    Credentialswhich user and groups it runs as, and therefore what it may touch/proc/PID/status
    PIDits identifier while it lives, and its parent's in PPID/proc/PID/ itself

    "Private memory" is worth one extra sentence, because it is the property everything else rests on. Two processes can both have a valid address 0x400000, and those are two completely different bytes of physical memory — the kernel and the CPU together maintain a per-process translation from the addresses a process uses to the physical memory it actually occupies. A process therefore cannot read another process's memory by accident. It cannot even address it. That isolation is not a feature somebody added for security; it is how memory has worked on Linux from the start, and it is why "processes are already fairly isolated" is the honest starting point for understanding containers.

    Which brings us to the payoff. There is no container object in the Linux kernel. Nothing in the source code is called a container. What a container runtime does is start an ordinary process with all four of those parts adjusted — different credentials (B4), a different view of the filesystem so its descriptors reach different files (B5), a private set of PIDs so it cannot see the rest of the hotel (B6), and a cap on how much memory and CPU its address space may consume (B7). Strip those adjustments away and you have a plain process; that is all a container ever was.

    program on disk→process=address space+open descriptors+credentials+PID

    Gotchas

    PID reuse is real and bites scripts. A script that records a PID, sleeps, and later runs kill $pid can signal an entirely unrelated process that has since inherited that number. On a busy machine the counter wraps in minutes. The robust patterns are to check identity first (ps -p "$pid" -o comm=), or to let something that tracks processes properly do it for you — which is precisely what systemd does in B11 and what your supervisor in B2.17 will do by holding a real parent-child relationship rather than a remembered number.

    /proc/PID/environ and /proc/PID/maps are readable only by the owner and root, so cat /proc/1/environ as an ordinary user gives "Permission denied" rather than an empty file. That is a credentials check, not a missing file, and sudo answers it. Do not conclude a /proc file is empty until you have tried it with privilege.

    A process is not the same thing as a job. [1] 2411 shows both: %1 is your shell's job number, meaningful only in that shell, and 2411 is the kernel's PID, meaningful everywhere. kill %1 and kill 2411 happen to do the same thing here, but only the second one works from another terminal.

    Everything in the pod spec is an adjustment to one of those four parts, and once you see that, the API stops looking like a grab-bag. securityContext.runAsUser sets the credentials. volumeMounts changes what the descriptors can reach. resources.limits caps the address space. shareProcessNamespace changes which PIDs are visible. There is no fifth kind of thing.

    The clearest place to see this from the Kubernetes side is W3.16 — requests vs limits: scheduler bookkeeping vs cgroup enforcement: a limit is not a promise the container makes, it is a number written into a kernel control file that applies to that process. B7 is where you write that file by hand.

    Docs

    man 5 proc is the reference for every file mentioned here — it is long, so search (/) for /proc/[pid]/status and /proc/[pid]/maps rather than reading front to back. man 7 credentials covers the four uid fields. For the definition itself, man 2 fork's opening paragraph is unexpectedly good on what a process consists of, because it has to list everything a copy inherits.

    B2.2

    Read ps column by column

    Why it matters

    ps is the first command you type on any machine that is behaving strangely, and most people read three of its columns and skip the rest. The skipped ones are where the answers are. Two of them — VSZ and RSS — are also responsible for a recurring false alarm: someone sees a process "using 2 GB" on a machine with 4 GB free and starts a fire drill over memory that was never occupied.

    The concept

    ps has two personalities because it inherited two different traditions, BSD and System V. ps aux is the BSD spelling (note: no dash), ps -ef is the System V spelling. They show almost the same processes in different columns, and you should be able to read both because both appear in every set of instructions on the internet. A third form, ps -eo …, lets you ask for exactly the columns you want, and it is the one worth building a habit around.

    Here is what each column is actually telling you:

    ColumnWhat it isWhat it is good for
    PIDthis process's numberthe handle for everything else you will do
    PPIDits parent's numberwho started it — the single most useful column, and absent from ps aux
    USERthe credentials it runs aswhether it can touch what you think it can
    STATits state, plus modifier letterswhether it is running, waiting, wedged or dead (B2.7)
    %CPUCPU time used ÷ time alivean average since it started, not a live reading
    %MEMRSS as a fraction of physical RAMa quick sort key, nothing more
    VSZvirtual size — address space reserved, in KBalmost never what you want
    RSSresident set size — physical memory actually occupied, in KBthe number to look at
    STARTwhen it began"did this start when the incident started?"
    TIMEtotal CPU time consumeddistinguishes busy from merely long-lived
    COMMANDhow it was invokedarguments included — often the whole diagnosis

    The VSZ/RSS pair deserves the explanation, because the gap is enormous and alarming until you know why. VSZ is a reservation; RSS is occupancy. A process can ask the kernel for an address range without ever touching it, and the kernel is happy to agree, because agreeing costs nothing — no physical memory is committed until the process actually reads or writes a page. Mapping a 10 GB file adds 10 GB to VSZ and nothing to RSS until you read from it. A language runtime that reserves a large heap up front looks terrifying in VSZ and modest in RSS. So: VSZ is what the process asked for, RSS is what it is holding, and only RSS is a resource anyone is competing for.

    One honest caveat on RSS, so you do not over-trust it either: shared pages are counted in full for every process that maps them. Fifty copies of the same program each report the shared library's pages in their own RSS, so adding up every RSS on a machine gives a total larger than the machine has. RSS is the right per-process number and the wrong thing to sum. free -m and, later, cgroup accounting (B7) are what answer "how much memory is really in use".

    Gotchas

    ps aux has no PPID column, which is the column you most often need. Every "who launched this?" investigation stalls for a moment on that omission. ps -ef has it, and ps -eo lets you ask for it — build the habit of the third form and the problem disappears.

    Processes shown in square brackets — [kthreadd], [rcu_gp] — are kernel threads, not user programs. They have no address space of their own, which is why their VSZ and RSS are 0, and you cannot kill them or strace them meaningfully. When something in brackets is consuming CPU it is a signal about the kernel's work (I/O, memory reclaim), not about a program you can restart.

    %CPU from ps is a lifetime average and %CPU from top is a sample of the last interval. They disagree constantly and both are correct. If you want "what is hot right now", use top; if you want "what has consumed the most CPU since boot", use ps's TIME column, which is the honest cumulative figure.

    The COMMAND column is truncated by terminal width in ps aux, and the truncated part is usually the arguments that would have told you which config file the process is using. ps -ww -ef disables the truncation, or read /proc/PID/cmdline directly as in B2.4.

    The VSZ/RSS distinction is the reason kubectl top pod and a container's own memory reporting so often disagree with intuition. Kubernetes cares about a number closer to RSS — specifically the cgroup's accounting of pages actually charged to the container — because that is what competes for the node's memory and what the OOM killer acts on. A container with a huge virtual size and a small resident size is not close to its limit at all.

    Follow this into W3.17 — QoS classes: derived, not declared, where which pod gets killed first under memory pressure is decided by real usage against declared requests, and into W6.12 — reading node Conditions, where MemoryPressure is the node saying the same thing at machine scale.

    B2.3

    See the process tree, and trace yourself up to PID 1

    Why it matters

    Processes on Linux are not a flat list — they are a family tree, and the shape of that tree is frequently the whole answer. "Why is there a stray python3 eating CPU?" is unanswerable from a flat listing and obvious from a tree, where you can see it hanging off a cron job. Later, when you look at a container host, the tree is what shows you the runtime supervising the containers, and each container's processes hanging beneath it.

    The concept

    Every process except one has a parent: the process that created it. The exception is PID 1, started by the kernel at boot, which has no parent and is the root of the whole tree. On Ubuntu 24.04 that is systemd, which B11 covers properly. So from any process you can follow PPID upward, one step at a time, and you will always arrive at 1 — there is exactly one path, and no cycles.

    Continuing the hotel: PID 1 is the front desk, and every guest arrived by someone letting them in. Follow "who let you in?" back far enough and you always reach the front desk, because the building has only one entrance.

    Two things about the tree surprise people the first time. First, the parent link survives the parent's own commands — your shell is a child of the SSH daemon session, which is a child of systemd, and that chain does not change as you type. Second, the link can change: if a parent dies while its child is still running, the child is not killed. It is handed to PID 1, which becomes its new parent. That is called re-parenting, it is how orphans work, and B2.9 is entirely about the consequences.

    pstree draws this. pstree -p adds PIDs, which you almost always want. By default it collapses identical siblings into sleep(2411,2455,2456) form, which is a feature — a runaway that has forked 400 copies of itself shows as one line with 400 numbers, and the shape tells you instantly that you are looking at a fork problem rather than 400 unrelated events.

    Gotchas

    Names in curly braces — {containerd}(1877) — are threads, not child processes. pstree shows them by default and it makes a busy tree look far more populated than it is. pstree -T hides them, and B2.10 explains what they are.

    Killing a parent does not kill its children. This is the single most common wrong assumption about the tree: people kill a supervising process expecting the work to stop, and the work carries on happily, re-parented to PID 1 and now much harder to find. To stop a whole group you need something that tracks the group — kill -- -PGID for a process group, or the cgroup-based supervision systemd does in B11, which is exactly why systemd exists.

    The tree you see is the tree in your PID namespace. Inside a container, pstree shows the container's own process as PID 1 and nothing else on the machine — same kernel, different view. B6 builds that view from scratch, and until then it is worth knowing that "PID 1" is always relative to who is asking.

    On a Kubernetes node the tree is the architecture diagram. systemd at the top; kubelet and containerd as its children; under containerd, one containerd-shim per pod sandbox; and under each shim, the container's own PID 1 and its children. Running pstree -p on a node and finding your workload at the bottom of that chain is the moment "a pod is just processes on a machine" stops being a slogan.

    The supervising link at the top is the subject of W1.19 — the kubelet's syncLoop and its three pod sources: the kubelet is a process whose entire job is deciding which other processes should exist on this node, which is structurally the same job as the supervisor you will write in B2.17.

    B2.4

    X-ray a live process through /proc/PID/

    Why it matters

    Almost every diagnostic tool you will ever use on Linux — ps, top, lsof, htop — is a formatter over /proc. Knowing that is not trivia: it means when a tool is missing, gives an answer you distrust, or has no flag for what you want, you can still get the truth by reading a file. On a stripped-down container image with no tools at all, /proc is often the only thing available, and it is enough.

    The concept

    /proc is not a directory of files on a disk. It is a filesystem-shaped interface to kernel data structures — you already saw in B1.3 that its files report size 0 while plainly having contents, and the reason is that the contents do not exist until you read them. The kernel generates the text at the moment of the read. B3 takes this apart properly; here, use it.

    Every live process has a directory /proc/PID/, and a handful of entries in it answer almost every question you will have:

    EntryAnswersNote
    cmdlineexactly how it was invoked, arguments includedNUL-separated, so pipe through tr '\0' ' '
    environthe environment it was giventhe copy made at exec — see B1.13
    fd/every open file, socket and pipe, as symlinksthe answer to "who is holding this file?"
    statusstate, uids, thread count, memory, signal maskshuman-readable summary of everything
    mapsthe layout of its address spacewhich libraries it actually loaded
    cwdits current directory, as a symlinkthe thing relative paths resolve against (B1.1)
    exethe binary it is running, as a symlinkstill valid after the file is deleted

    Two of those are worth calling out because they solve real problems. fd/ is how you find the process still holding a deleted 40 GB log file: when a file is unlinked but a process still has it open, the disk space is not freed (this is the link-count rule from B1.4), and ls -l /proc/*/fd shows the symlink with (deleted) appended. Nothing else on the machine will tell you. And exe keeps pointing at the original binary even after an upgrade replaces the file on disk, which is how you discover that a service is still running last week's version.

    The shorthand /proc/self/ always refers to the process doing the reading. So cat /proc/self/cmdline describes the cat, not your shell — a small joke the kernel plays on everyone once.

    Everything in /proc/PID/ is a snapshot generated at read time, so two reads a millisecond apart can disagree, and a process can exit between your ps and your cat — leaving you with "No such file or directory" for a PID that existed a moment ago. That is not an error to debug; it is the process having ended.

    Lifecycle

    B2.5

    How a process is really created: fork, then exec, then wait

    Why it matters

    Every other operating system you have used offers one call that means "run this program". Linux does not, and the split looks like a historical accident until you notice that the gap between the two calls is where everything interesting happens. Shell redirection lives in that gap. Dropping privileges lives in that gap. Joining a cgroup, entering namespaces, setting resource limits — every one of them happens in the moment after a process has been created and before it has become the program you asked for. Container runtimes are, almost entirely, code that runs in that gap.

    The concept

    Three calls, each with one job:

    • fork() makes a near-identical copy of the calling process. Two processes now exist, both sitting at the same line of code, both about to return from the same call. They are distinguished only by what fork returns: the child gets 0, the parent gets the child's PID. That one difference is how each half knows which half it is.
    • exec() throws away the calling process's program and loads a different one in its place. This is not "start a new process" — the PID does not change, the open descriptors do not change, the credentials do not change. Only the code and memory are replaced. A successful exec never returns, because there is nothing left to return to.
    • wait() blocks the parent until a child ends, and hands back that child's exit status. Without it the parent has no way to learn whether its child succeeded, and the child's exit status has nowhere to go — which is where zombies come from (B2.9).

    Now the payoff. Between fork and exec, the child is a copy of the parent that is already running its own code and can therefore adjust itself before becoming the target program. It does not have to ask the parent to set things up, and it does not need the target program to cooperate. That is why ls > out.txt works with a program that knows nothing about redirection: the shell forks, the child points its own descriptor 1 at out.txt, and only then execs ls, which writes to descriptor 1 exactly as it always does.

    Read left to right: one process becomes two at fork, the child rearranges its own world, exec swaps in the new program without changing the PID, and the exit status travels back up the dashed line when the parent calls wait.

    One more thing worth knowing, because "makes a copy" sounds ruinously expensive: it is not a real copy. The kernel gives parent and child the same physical pages marked read-only, and only duplicates a page when one of them writes to it — copy-on-write. Since the overwhelmingly common case is fork immediately followed by exec, which discards the memory anyway, almost nothing is ever actually copied.

    Gotchas

    Forgetting fflush(stdout) before fork is a classic and produces baffling duplicate output. Standard output to a file is block-buffered, so text you "printed" may still be sitting in the process's buffer — and fork copies the buffer, so both processes flush it later and the line appears twice.     exec failing does not end the process, it just returns. That is why every correct fork/exec has an exit() immediately after the exec call — without it, a failed exec leaves you with a stray copy of the parent running the parent's code, which is a genuinely confusing bug to inherit. The conventional exit code there is 127, matching the shell's "command not found" (B2.8).

    The child inherits open descriptors, and that includes ones you did not think about. A child holding an inherited descriptor to a log file or a listening socket keeps that resource alive after the parent is gone — this is the reason a restarted service sometimes cannot bind its port ("address already in use") even though the old process is definitely dead: a forgotten child still holds the socket. The fix is the FD_CLOEXEC flag, which closes a descriptor automatically at exec.

    The gap between fork and exec is, quite literally, where a container is made. When containerd starts your workload, the low-level runtime (runc) forks, and in that gap the child enters new namespaces, joins the cgroup that will enforce the pod's limits, sets the uid from securityContext, changes its root to the image's filesystem, and drops capabilities — and only then execs the image's entrypoint. Your program starts already inside a container and never knows a container happened.

    You can see the resulting structure from the Kubernetes side in W1.19 — the kubelet's syncLoop and its supervision chain, and the consequences of the exec'd program's identity in W3.5 — graceful termination, where whichever program got exec'd is the one that must handle SIGTERM.

    B2.6

    Watch parenthood concretely, and see what a child inherits

    Why it matters

    Inheritance is where a whole class of "but it works when I run it by hand!" bugs comes from. A program run from your terminal inherits your environment, your working directory, your open terminal and your user. The same program started by a service manager inherits an almost empty environment, / as its directory, no terminal at all, and possibly a different user. Nothing about the program changed; everything about what it was handed did. Being able to list what crosses the fork boundary turns those bugs from mysteries into a checklist.

    The concept

    The parent/child relationship is not a bookkeeping detail — it carries real consequences. The parent is the only process that can wait() for this child and therefore the only one that can learn how it died. If the parent disappears, the child is re-parented to PID 1 (B2.3, and B2.9 in depth). And a great deal of a process's behaviour is decided before it ever runs a line of its own code, by what it was handed.

    What a child gets from its parent, at fork, and keeps across exec:

    InheritedWhy it matters
    environment variablesPATH, HOME, LANG, and every credential passed this way
    current working directorywhat relative paths resolve against (B1.1)
    open file descriptorsincluding 0/1/2 — this is why output goes where it goes
    credentials (uid, gid, groups)what it is allowed to touch
    umask, resource limits, signal dispositions¹quiet defaults that shape behaviour
    controlling terminal, process group, sessionwhether Ctrl-C reaches it

    ¹ With one important exception: signal handlers cannot survive exec, because the code implementing them is gone. Signals set to be ignored stay ignored; everything else resets to default. B2.14 comes back to this.

    What a child does not get is equally important: it does not share the parent's memory (copy-on-write means it gets its own copy on first write), and changes it makes to any of the above are its own. This is the deep reason a child can never change its parent's environment or working directory — it is modifying a copy. That is the whole explanation of why cd must be a shell builtin (B1.1) and why source exists.

    B2.7

    Process states — and the two that mean trouble

    Why it matters

    "The process is stuck" is not a diagnosis, it is an observation, and the state letter is what turns one into the other. A process in S is not stuck at all — it is waiting for something perfectly normal. A process in D is stuck in a way you cannot fix by killing it, and is telling you the problem is underneath it. A process in Z is not a process at all any more. Three completely different situations that all look like "it's hung" from the outside.

    The concept

    At any instant, every process is in exactly one state, and there are only a handful:

    LetterNamePlainly
    Rrunning or runnableon a CPU right now, or queued and ready to be
    Sinterruptible sleepwaiting for something — a keystroke, a packet, a timer — and a signal will wake it
    Duninterruptible sleepwaiting inside the kernel for I/O that cannot safely be interrupted; signals do not reach it
    Tstoppedsuspended by SIGSTOP or Ctrl-Z, will resume on SIGCONT
    Zzombiealready dead; only its exit status remains, waiting to be collected
    Iidlean idle kernel thread — normal, and excluded from load averages

    Most processes on a healthy machine are in S, and that is what "idle" looks like from the inside: a hundred programs each blocked in a call that says "wake me when something happens". It is not a problem to be solved.

    D is the one that matters at three in the morning. The process asked the kernel for something — read this disk block, talk to this network filesystem — and the kernel is in the middle of a sequence it cannot abandon halfway without leaving data structures inconsistent. So it refuses delivery of all signals, including SIGKILL. The consequences are worth stating bluntly: you cannot kill a D-state process. Not with -9, not as root, not ever, until the I/O it is waiting on completes or fails. A pile of processes in D means the layer beneath them is wedged — a failing disk, a hung NFS mount, a storage backend that stopped answering — and the fix is down there, not up here.

    Why does uninterruptible sleep exist at all? Because the alternative is worse. If a signal could yank a process out of the middle of a block-device write, the kernel would have to be able to unwind every partially-completed operation in every driver, correctly, always. Linux takes the simpler and safer bargain: some waits are short and cannot be interrupted. When one turns out not to be short, you get a wedged process — and an unambiguous signal about where the real fault is.

    Z, the zombie, is not stuck: it is finished. Its memory is gone, its descriptors are closed, and all that survives is a small kernel record holding the exit status until the parent calls wait(). B2.9 is about that record. For now, know that a zombie consumes no memory and no CPU, cannot be killed (there is nothing left to kill), and is a message about its parent.

    ps's STAT column appends modifier letters to the state, and they are useful: s session leader, l multi-threaded, + in the foreground process group, < high priority, N low priority. So Ssl reads as "sleeping, session leader, multi-threaded" — a completely ordinary daemon.

    B2.8

    Read an exit code as a message — including 128 + signal

    Why it matters

    An exit code is the only thing a process says on its way out, and it is one small integer. Learning to read it is the difference between "the container crashed" and "the container was killed from outside, so stop reading its application logs for a cause". This is the lesson that makes Exit Code: 137 immediately meaningful, months before you meet it in a pod status.

    The concept

    When a process ends, the kernel keeps one byte of information — a number from 0 to 255 — and hands it to the parent's wait(). In the shell, $? holds the code from the last command. The conventions:

    CodeMeans
    0success — and it is the only success value
    1–125the program's own failure codes, meaning whatever its author decided
    126found, but not executable — usually a missing execute bit
    127command not found — the shell could not find it on PATH
    128 + Nkilled by signal N — the process did not choose to exit

    The 128 + N convention is the important one, and it exists because a single byte has to carry two different kinds of news: "the program decided to exit with this code" and "something killed the program with this signal". Reserving the top range for the second kind keeps them apart. So:

    • 137 = 128 + 9 = SIGKILL. Something outside decided this process should stop existing, and gave it no chance to object. On a node, this is nearly always a memory limit and the kernel's OOM killer.
    • 143 = 128 + 15 = SIGTERM. Something asked it to stop, and it either agreed or was not listening. This is the ordinary shutdown path.
    • 139 = 128 + 11 = SIGSEGV. It touched memory it does not own — a genuine bug in the program.
    • 130 = 128 + 2 = SIGINT. Somebody pressed Ctrl-C.

    The practical value is in what each rules out. A code in the 1–125 range means the program ran, made a decision, and reported it — so its logs will contain the reason. A code above 128 means the program was interrupted, so its logs contain nothing about why, and the cause is in the environment: the OOM killer, an orchestrator, a supervisor, a person. Looking in the wrong place is the most common wasted hour in incident response, and this arithmetic tells you which place is right.

    One subtlety worth carrying: the "128 + N" encoding is the shell's presentation. At the C level, wait() gives the parent a status word from which WIFEXITED/WEXITSTATUS and WIFSIGNALED/WTERMSIG extract the two cases separately and unambiguously — which is what your supervisor in B2.17 will use, and what Kubernetes uses to fill in both exitCode and reason: OOMKilled.

    B2.9

    Orphans, zombies, reaping — and what PID 1 owes its children

    Why it matters

    This is the lesson behind one of the most-hit and least-understood container problems: a container whose PID 1 was never designed to be an init, quietly accumulating dead processes until the node runs out of PIDs. It is also where the phrase "the PID 1 problem" comes from, and once you have built a zombie by hand the whole thing stops being folklore.

    The concept

    Two things can go wrong with a parent-child pair, and they are opposites.

    The parent dies first. The child is not killed — it keeps running, now with no parent. The kernel immediately re-parents it to PID 1, so the tree stays a tree. That child is an orphan, and it is a completely normal, harmless situation: this is exactly how daemons used to be created on purpose.

    The child dies first, and the parent never asks how it went. The child's memory is freed, its descriptors closed, its threads gone — but the kernel must keep the exit status somewhere until the parent collects it, because the parent still has the right to know. So a small record survives, holding a PID and a status and nothing else. That is a zombie, and the act of collecting it with wait() is called reaping.

    Think of a zombie as a receipt left on the front desk. The guest has checked out and the room is cleaned and re-let; only the paperwork remains, waiting for whoever booked them to come and collect it. One receipt is nothing. Ten thousand uncollected receipts fill the desk, and now nobody can check in — because the "desk" here is the system's fixed table of PIDs.

    Which brings us to the obligation. When a process is re-parented to PID 1, PID 1 inherits the duty to reap it. A proper init — systemd, or the classic init — does this in a loop forever: it calls wait(), discards the status, and calls wait() again. That is one of the two jobs PID 1 exists to do, and it is why orphans are harmless on a normal machine.

    Now put a program that is not an init at PID 1, which is exactly what a container does. Your application starts as PID 1 inside the container's PID namespace (B6). It spawns a helper, the helper spawns something, that grandchild's parent exits, and the grandchild is re-parented to... your application, which has no wait() loop, because nobody writing a web server expects to be an init system. The zombies accumulate with nothing to collect them. This is the PID 1 problem, and it has exactly three fixes: write the reap loop yourself, use a tiny init as your entrypoint (tini, dumb-init), or ask the platform to provide one.

    PID 1 owes its children a second duty as well, and it is the one people forget: signal handling. The kernel gives PID 1 special treatment — signals with a default action of "terminate" are not delivered to PID 1 unless it has explicitly installed a handler for them. This is a safety measure so a stray signal cannot kill init and panic the machine. Inside a container it produces a notorious surprise: your application, running as PID 1 with no SIGTERM handler, simply ignores SIGTERM. Every shutdown then takes the full grace period and ends in SIGKILL. B2.16 is where you fix that properly.

    child exits→zombie holds exit status→parent calls wait()→record freed, PID reusable

    B2.10

    What a thread actually is on Linux

    Why it matters

    Most explanations of threading start from a language runtime and describe threads as a feature of the program. Linux's view is simpler and more useful: a thread is just another schedulable task, created by the same call that creates processes, with more of its parent shared. Holding the kernel's view means ps, top and /proc stop surprising you — a "single process" showing 40 entries under /proc/PID/task is not a bug, and the machinery you are looking at is the same machinery that makes containers.

    The concept

    Go back to B2.1's four-part definition. A thread is a schedulable task that shares its address space and its file-descriptor table with its siblings, while having its own stack, its own registers, and its own kernel-visible ID. Two threads in one program see exactly the same memory at exactly the same addresses — which is what makes them fast to communicate between, and exactly what makes them dangerous to get wrong.

    In the hotel: a process is a guest with a room. Threads are several people sharing one room. They all have the same key and the same wardrobe, so passing something between them is instant — and if two of them rearrange the wardrobe simultaneously, nobody can say afterwards what happened.

    Here is the part that matters for the rest of this track. The kernel does not have two different objects for "process" and "thread". Both are created by one system call, clone(), and the flags passed to it decide how much of the parent is shared rather than copied. fork() is a thin wrapper over clone() that shares almost nothing; creating a thread is the same call sharing almost everything. It is one dial, not two mechanisms.

    And clone() is also how namespaces are created — the flags CLONE_NEWPID, CLONE_NEWNET, CLONE_NEWNS and their siblings are more positions on that same dial, saying "give the new task its own PID numbering / network stack / mount table". So the machinery behind threading and the machinery behind containers are literally the same system call with different arguments. B6 turns those flags one at a time.

    Vocabulary, because the tools use it inconsistently: the kernel calls each schedulable task a task and gives it a TID. All threads of one program share a TGID, and the TGID is what everything in userspace calls "the PID". The first thread's TID equals the TGID, which is why a single-threaded process looks like it has a PID and nothing else.

    clone()→share nothing = fork·share memory + fds = a thread·new namespace flags = a container

    B2.11

    When the thread distinction changes your diagnosis

    Why it matters

    Most of the time you can ignore threads entirely. Three situations where ignoring them costs you an hour: a process that looks mildly busy but has one thread pinned at 100%; a process that looks stuck when only one of its threads is; and a signal that appears to be handled by the wrong part of a program. All three are invisible at process granularity and obvious one level down.

    The concept

    The core mismatch is that the tools report per-process by default and the kernel schedules per-thread. A process's %CPU is the sum across its threads, so on an 8-core machine a process showing 12.5% might be entirely idle except for one thread consuming a whole core. The average hides the exact thing you are looking for, and the fix is to stop averaging: top -H and ps -L both switch to per-thread rows.

    The same applies to state. There is no such thing as "the process is in D" when the process has 40 threads — some are in D, some are in S, and which ones matter. Per-thread State in /proc/PID/task/*/status is the honest reading, and a program that appears wedged is often one thread wedged on I/O while the rest wait on it.

    Signals are the third case, and the rule is worth stating precisely because it is genuinely surprising. A signal sent to a PID is delivered to any one thread that has not blocked it — the kernel chooses, and you do not get to know which. Multi-threaded programs deal with this by having all threads block the signals they care about and dedicating one thread to receiving them. The consequence for you as an operator: "I sent SIGTERM and a different part of the program reacted than I expected" is not a bug you can fix from outside, and reading SigBlk/SigCgt per thread is how you find out who was actually listening.

    Those fields are worth knowing how to read. /proc/PID/status has four signal masks as hexadecimal bitmaps — SigPnd (pending), SigBlk (blocked), SigIgn (ignored), SigCgt (caught, i.e. a handler is installed). Bit N−1 corresponds to signal N, so SIGTERM (15) is bit 14, which is 0x4000. An all-zero SigCgt means the program installed no handlers at all — which, as B2.9 warned, is a very interesting fact about a container's PID 1.

    In thread mode top labels the column PID even though it is showing TIDs. Do not paste those numbers into a kill without thinking: signalling a TID directly (which requires tgkill, not kill) is almost never what you want, and kill <TID> from the shell will either fail or hit an unrelated process whose PID happens to match.

    An all-zero SigCgt is not automatically a problem — sleep has no handlers and dies on SIGTERM perfectly well, because the default action does the job. It only becomes a problem when the program needs to do something before exiting, or when it is PID 1 and the default action is therefore suppressed (B2.9). Read the field together with the process's role.

    The hot-thread case is what makes CPU throttling confusing in Kubernetes. A container hitting its CPU limit shows as a pod that is slow rather than a pod that is broken, and kubectl top pod reports a single aggregated number that looks comfortably under the limit — because the limit is enforced against the sum, over a short window, and one saturated thread with periodic bursts averages down. Getting to the node and running top -H against the container's PID is the move that finds it.

    The signal-mask check is the fast answer to a pod stuck in Terminating for the full grace period: if the container's PID 1 has an all-zero SigCgt, it never installed a SIGTERM handler, and the PID 1 rule from B2.9 means the signal is being discarded. That is the mechanism behind W3.5 — graceful termination, and the diagnosis path in W6.18 — the quiet failures.

    Signals

    B2.12

    The signal table worth memorizing

    Why it matters

    Every time you stop a program, reload a daemon's config, press Ctrl-C, or see a container get killed, a signal did it. There are a few dozen signals and you need to know about eight of them — the rest you can look up. Knowing those eight cold means that "the pod got SIGTERM then SIGKILL" reads as a sentence rather than a puzzle, and that you reach for the right one instead of reflexively typing kill -9 at everything.

    The concept

    A signal is a knock on the door with a standard meaning — the hotel analogy this track uses. It is not a message: there is no body, no payload, no way to attach information. The process gets a number and nothing else, and everything the number "means" is convention agreed between programs, not something the kernel enforces. That is the single most clarifying fact about signals: SIGHUP means "reload your config" only because daemons decided it would, and the kernel has no opinion.

    The eight to memorize, with their numbers, because you will type the numbers:

    SignalNo.Knock that means
    SIGTERM15"please stop" — the polite, catchable request. The default for kill.
    SIGKILL9"stop now, no discussion" — uncatchable, handled entirely by the kernel.
    SIGINT2"interrupt" — what Ctrl-C sends.
    SIGHUP1originally "the terminal went away"; by convention now "reload config".
    SIGSTOP19"freeze" — uncatchable pause; the process goes to state T.
    SIGCONT18"resume" — undoes STOP.
    SIGCHLD17"a child of yours changed state" — the nudge that drives a reaper (B2.9).
    SIGSEGV11"you touched memory you do not own" — a bug, produces the 139 from B2.8.

    Three groupings make them easier to hold. The stoppers — TERM, KILL, INT — all end a process, differing only in politeness and whether the program gets a say. The job-control pair — STOP and CONT — pause and resume without ending anything, and are what your shell's Ctrl-Z and fg are built on. The notifiers — HUP, CHLD, and the two user-defined signals USR1/USR2 — carry no fatal intent at all; a program that wants to be told something without being killed uses these.

    The numbers are not arbitrary trivia, because of B2.8: an exit code above 128 is 128 + the signal number, so knowing 9, 15 and 11 is what lets you read 137, 143 and 139 on sight. The two you must never forget are 9 and 15 — the hard kill and the polite ask — because the entire shutdown story of every container is those two in sequence.

    Gotchas

    Reflexive kill -9 is a bad habit with real costs. SIGKILL gives the program no chance to flush its buffers, remove its lock file, or finish a request — so you get corrupted output, stale lock files that block the next start, and half-written data.     Names are portable; numbers are mostly portable but not perfectly.     SIGHUP's "reload config" meaning is pure convention, and not every program honours it. Sending SIGHUP to something that never implemented the convention triggers the default action, which is to terminate (you will prove this in B2.13). Check the program's docs before assuming HUP means reload.

    The entire pod shutdown sequence is two of these signals in order: the kubelet sends SIGTERM to the container's PID 1, waits terminationGracePeriodSeconds (30 by default), then sends SIGKILL.     SIGHUP-as-reload shows up too: many Kubernetes components and sidecars reload their configuration on SIGHUP rather than restarting, which is how a config change can take effect without dropping connections. The pod-side contract is in W3.5 — graceful termination: SIGTERM, the grace period, and SIGKILL.

    B2.13

    Every signal has a default action

    Why it matters

    A program that has never heard of signals still responds to them, because the kernel has a default action for each one. Knowing the defaults lets you predict what a signal will do before you send it — which is the difference between "I'll send HUP to reload it" and accidentally killing a daemon that never implemented HUP. It also explains the otherwise-baffling behaviour where the same signal kills one program silently and dumps a core file for another.

    The concept

    When a signal arrives, the process does one of exactly five things, and if the program has not installed a handler, the kernel picks the signal's default:

    DefaultCodeMeaningExamples
    TerminateTermprocess ends, no core fileTERM, INT, HUP, ALRM
    CoreCoreprocess ends and dumps memory to a core file for debuggingSEGV, ABRT, QUIT, ILL
    IgnoreIgnsignal is discarded, process continuesCHLD, URG, WINCH
    StopStopprocess is paused (state T)STOP, TSTP, TTIN, TTOU
    ContinueConta stopped process resumesCONT

    Two things fall straight out of this table. First, the reason a segfault leaves a core file while a Ctrl-C does not is simply that SEGV's default is Core and INT's is Terminate — same "the process ends", different default, different debris. Second, the reason SIGCHLD seems to "do nothing" is that its default is genuinely Ignore: a parent that installs no handler still has its children turn into zombies (the kernel keeps the exit status), but it is never interrupted to notice, which is exactly the reaping problem from B2.9.

    A program changes a signal's disposition in one of three ways: install a handler (run my function instead of the default), set it to ignore, or reset it to default. Those dispositions are what the SigIgn and SigCgt masks from B2.11 record. And two signals — KILL and STOP — cannot have their disposition changed at all, which is B2.15's subject and the reason they always win.

    The practical skill is prediction. Given a program that installed no handlers, you can state what any signal will do to it by reading the default column of man 7 signal. Send HUP to such a program and it terminates — not because HUP is fatal, but because HUP's default is Terminate and nobody overrode it.

    B2.14

    Catch a signal yourself

    Why it matters

    Everything so far has been about signals sent to programs. This is where you sit on the other side and write the code that receives one — because "handle SIGTERM gracefully", the single most repeated piece of advice about running software in containers, is a handler, and until you have written one it is just words. It is also only about six lines, in either language.

    The concept

    Installing a handler tells the kernel: when this signal arrives, do not take the default action — run my function instead. The function is called a signal handler, and the kernel arranges for it to run by interrupting whatever the program was doing, calling your function, and then resuming. That interruption is the source of both the power and the sharp edges.

    In C the call is sigaction() (the older signal() still works but has portability quirks — prefer sigaction). In Python it is the signal module, which hides the machinery. Both do the same thing: associate a signal number with a function.

    The sharp edge is worth stating up front, because it is the source of every subtle signal bug: a handler runs asynchronously, interrupting the main program at an arbitrary point — possibly in the middle of a malloc, a printf, or an update to a data structure. So a handler may only safely do a very short list of things. The classic-correct pattern is: the handler sets a single flag and returns; the main loop notices the flag and does the real work. Trying to do the shutdown inside the handler — closing files, writing logs — is how you get deadlocks and corrupted output. In C that flag must be of type volatile sig_atomic_t, which is the only kind of variable it is safe to touch from both places.

    There is one more rule from B2.6 that lands here: signal handlers do not survive exec. A handler is code, and exec replaces the code, so the new program starts with default dispositions (except for signals that were set to ignore, which stay ignored). This is why the handler has to be installed by whatever program is actually running when the signal arrives — you cannot set it up in a wrapper and expect the exec'd program to inherit it.

    Gotchas

    Doing real work inside a handler is the most common signal bug there is. A handler that calls printf, malloc or anything touching shared state can deadlock, because it may have interrupted the main program while it held that same lock. The list of functions guaranteed safe to call from a handler (async-signal-safe functions) is short — write is on it, printf is not. Set a flag; do the work in the main loop.

    In C, an ordinary global variable is not safe to share with a handler — the compiler may cache it in a register and never see the handler's change. volatile sig_atomic_t is the one type the standard guarantees you can read and write atomically from both sides. Using a plain int works until an optimiser decides otherwise, which is the worst kind of bug.

    A handler installed before exec is gone after it (B2.6): exec replaces the code the handler lived in. So a shell wrapper that traps SIGTERM and then execs your program does not pass its handler through — the exec'd program must install its own. Signals set to ignore are the sole exception that survives exec.

    Catching a signal does not make you immune to being stopped. Even a program that handles TERM, INT and HUP can still be paused by STOP and killed by KILL, because those two cannot be caught — which is the entire point of the next lesson.

    This handler is the literal implementation of what Kubernetes documentation means by "handle SIGTERM". When the kubelet sends SIGTERM at the start of pod termination, the code that decides whether shutdown is graceful or violent is a handler exactly like the one you just wrote — set a flag, stop accepting new work, drain, exit. A container whose main process has no such handler falls through to the default action, or, if it is PID 1, to the discard-the-signal rule from B2.9.

    The full pattern, with the drain step, is B2.16, and its Kubernetes counterpart is W3.5 — graceful termination, including the preStop hook that runs alongside the signal.

    B2.15

    The two you cannot catch, and the tools that deliver signals

    Why it matters

    You just learned to catch signals, so the obvious question is: can a program make itself unkillable? The answer — no, because two signals refuse to be caught — is what keeps the system controllable, and it is the reason kill -9 is the thing everyone reaches for when a process stops cooperating. This lesson pins down that guarantee and rounds out the small family of commands that send signals.

    The concept

    Two signals cannot be caught, ignored, or blocked, no matter what a program does: SIGKILL (9) and SIGSTOP (19). A program cannot install a handler for them, cannot set them to ignore, and cannot block them for even an instant. The kernel handles both entirely on its own, without ever running a line of the target's code — which is exactly why they always work.

    This is a deliberate design decision, not an oversight. The system needs at least one way to remove a process that has stopped cooperating and at least one way to freeze one, and those levers must not be something a runaway or a malicious program can disable. So the kernel reserves two signals it will always honour. In the hotel: SIGKILL is security removing a guest — there is no conversation, the guest does not get to argue, and it is not a knock they can decline to answer.

    The cost of that guarantee is the flip side of B2.14: because SIGKILL never runs the program's code, the program gets no chance to clean up. No flushing buffers, no removing lock files, no finishing the in-flight request. Everything graceful has to happen during the SIGTERM that you send first; SIGKILL is what you fall back to when TERM was ignored, and it always costs you whatever the program had not yet saved. "TERM then, only if needed, KILL" is not politeness for its own sake — it is the only sequence that gives the program a chance to exit cleanly while guaranteeing it exits.

    kill -0 PIDnothingsends no signal; just checks whether you could — an existence/permission test

It is how a script asks "is my child still alive?" without disturbing it, and your supervisor in B2.17 will use it.

    Gotchas

    kill -9 as a first resort is the habit this whole group exists to break. It denies the program any chance to shut down cleanly, so you inherit corrupted files, stale locks and lost work. It is the right tool for exactly two situations: a program that ignored SIGTERM, and nothing else. (And it does not even work on a D-state process — B2.7.)

    pkill and killall match by name or pattern, and a broad pattern is a foot-gun. pkill -f python on a machine running several Python services kills all of them. Always pgrep -af the same pattern first to see the exact list you are about to signal — the two commands take identical arguments precisely so you can preview.

    killall means "kill every process with this name" on Linux, but on some other Unixes it historically meant "kill all processes" (a shutdown tool). If you ever work across systems, this is a genuinely dangerous ambiguity; on Linux you are safe, but the name is a bad one.

    SIGSTOP is not a way to safely pause a service you depend on. A stopped process still holds all its resources — its locks, its open files, its network connections — it simply stops servicing them, so everything waiting on it hangs. Freezing a database mid-transaction with SIGSTOP can wedge every client. Use it for inspection, briefly, not as a control mechanism.

    SIGKILL's uncatchability is the hard backstop of pod termination: after the grace period, the kubelet sends SIGKILL, and because nothing can catch it, termination is guaranteed to complete (unless the process is wedged in D on dead storage — the one true exception, B2.7). This is why terminationGracePeriodSeconds is a deadline, not a request: it is precisely how long you have before the uncatchable signal arrives.

    The consequence — losing in-flight work on SIGKILL — is the entire reason for the graceful-shutdown contract in W3.5 — graceful termination, and for setting the grace period long enough that a well-behaved app finishes draining during the TERM phase and never meets the KILL.

    Docs

    man 7 signal under the note that SIGKILL and SIGSTOP "cannot be caught, blocked, or ignored" — the one sentence this lesson rests on. man 1 kill, man 1 pkill (shared with pgrep), man 1 killall for the delivery tools. man 2 kill for signal 0 as an existence test.     B2.16

    The graceful-shutdown pattern, end to end

    Why it matters

    This is the lesson everything in the signals group was building toward, and it is the single most valuable thing in this module for running software anywhere — Kubernetes or not. A program that shuts down gracefully loses nothing when it is asked to stop; one that does not loses whatever it was holding and takes the maximum possible time to do it. The gap between the two is one signal handler and a drain loop, and here you build both and measure the difference.

    The concept

    Graceful shutdown is a four-step contract, and every step maps to something you already have:

    1. Catch SIGTERM — the handler from B2.14. This is the "please stop" knock, and catching it is what turns a violent stop into an orderly one.
    2. Stop accepting new work — the moment TERM arrives, refuse new requests, connections, or jobs. Nothing new comes in.
    3. Finish what is in flight — let the requests already being handled complete. This is the drain, and it is why graceful shutdown takes a moment rather than being instant.
    4. Exit 0 — a clean, chosen exit (B2.8), signalling "I stopped because I was asked, and I finished everything".

    The alternative — ignore TERM — does not make a program robust. It makes every shutdown take the full grace period, because whatever is stopping it waits the whole timeout for a graceful exit that never comes, then sends the uncatchable SIGKILL (B2.15), which drops all in-flight work and reports 137. So "handles TERM" and "ignores TERM" are not "graceful vs. slightly-less-graceful"; they are "one second, nothing lost" versus "the entire timeout, everything in flight lost". That is the measurement this lab makes concrete.

    The structure from B2.14 is exactly what implements this: the handler sets a flag and returns; the main loop checks the flag, stops taking new work, drains, and exits. The handler never does the draining — it only flips the switch that tells the main loop to begin.

    Lab
    1. Write a worker that does the full pattern. It processes "items"; on SIGTERM it sets a flag, finishes the current item, and exits cleanly. 
      cat > /tmp/worker.py <<'PY'
#!/usr/bin/env python3
import signal, sys, time

running = True
def on_term(signum, frame):
    global running
    print("SIGTERM — no new work, finishing current item", flush=True)
    running = False                      # step 1+2: flag set, stop taking new

    print(f"item {item}: working", flush=True)
    time.sleep(2)                        # simulate an in-flight request
    print(f"item {item}: committed", flush=True)   # step 3: it finishes

    Gotchas

    Draining is not the same as ignoring. A correct shutdown handler finishes in-flight work and refuses new work; a program that keeps accepting new work after TERM never drains and will always hit the KILL. The distinction is step 2 — stop accepting — and it is the one people leave out.

    An unbounded drain is its own bug. If "finish in-flight work" can take longer than the grace period, SIGKILL arrives mid-drain and you lose the very work you were trying to protect. A real handler caps the drain at something shorter than the grace period, so it always finishes on its own terms. Match the two numbers deliberately.

    The PID 1 trap from B2.9 undoes all of this silently. If your carefully-written TERM handler is in a program running as PID 1 with no handler actually installed at the kernel level — for instance because a shell script entrypoint is PID 1 and it, not your app, receives the signal — the signal is discarded and none of your shutdown code runs. Make sure the process with the handler is the one receiving the signal, via exec in the entrypoint or a real init.

    A shell entrypoint that runs your app without exec (just myapp, not exec myapp) makes the shell PID 1 and your app its child. The shell receives TERM, does not forward it, and your app never hears it. exec myapp replaces the shell so your app becomes PID 1 and receives the signal directly — this one keyword is the difference between a graceful and a violent shutdown for a huge fraction of real containers.

    This lab is pod termination, built by hand. The kubelet is your stop.sh: it sends SIGTERM to the container's PID 1, waits terminationGracePeriodSeconds, and sends SIGKILL. Your worker is the container's main process. The exit-0-in-1s versus exit-137-in-10s contrast is exactly what you see in a pod's lastState.terminated depending on whether the image handles TERM — and it is why a well-behaved deployment rolls in seconds while a badly-behaved one takes grace × replicas worth of thirty-second waits.

    Everything here maps onto W3.5 — graceful termination: SIGTERM, the grace period, and SIGKILL, including the preStop hook (a way to run drain logic even for an app you cannot modify) and the interaction with readiness probes that stops new traffic — the Kubernetes spelling of step 2.

    B2.17

    A mini-supervisor — in C and Python

    Why it matters

    Every idea in this module meets here. A supervisor forks a child (B2.5), forwards a signal to it (B2.14), waits and reads the status to tell "exited" from "killed" (B2.8, B2.9), and restarts it with backoff when it dies badly. Written out, it is about fifty lines — and it is, structurally, a baby kubelet. When you later read that the kubelet "supervises" pods, this is the shape of the thing it does. B11 turns this same program into a real systemd unit; here you build the engine.

    The concept

    A supervisor is a loop around one child. Stripped to its logic:

    1. Start the child: fork, and in the child exec the target program. Remember its PID.
    2. Wait for it: block in waitpid until it ends, and read the status word.
    3. Decide from the status: exited cleanly (0) → stop; exited with an error, or was killed → restart, unless we are shutting down.
    4. Back off before restarting, so a child that crashes instantly does not become a busy-loop: wait 1s, then 2, 4, 8… capped.
    5. Meanwhile, forward SIGTERM: when the supervisor itself is asked to stop, pass that request to the child and let it shut down gracefully rather than orphaning it.

    Two subtleties are the whole reason this is a real exercise and not a toy. First, the wait can be interrupted: when SIGTERM arrives and your handler runs, the waitpid the supervisor was blocked in returns early with EINTR ("interrupted") — you must loop and retry it, or you will misread the interruption as the child ending. Second, the handler must be minimal: it forwards one signal and sets a flag (B2.14's rule), and all the real decision-making happens back in the main loop where it is safe.

    The exit-vs-signalled distinction from B2.8 is what makes the restart policy correct. A child that exits 0 asked to stop — respect it. A child that exits non-zero or was killed by a signal failed — restart it. And if we are the reason it stopped (we forwarded a TERM because we are shutting down), do not restart it at all. Those three cases are exactly what WIFEXITED/WEXITSTATUS and WIFSIGNALED let you tell apart.

    fork+exec child→waitpid (retry on EINTR)→exited 0 or shutting down? stop·else back off & restart

    Gotchas

    The EINTR retry is the single most common thing people leave out, and it is invisible until a signal arrives at exactly the wrong moment. Python papers over it for you (PEP 475 auto-retries most syscalls), which is why the Python version has no explicit retry loop and the C version must — a genuine difference between the languages worth understanding rather than memorising.

    Forwarding only SIGTERM is a deliberate simplification. A production supervisor forwards the whole termination sequence — TERM, wait for the grace period, then KILL — exactly like stop.sh from B2.16. Adding that is the natural next step, and it is precisely what systemd does for you in B11, which is why you will not have to write it again.

    This supervisor does not reap grandchildren. It waits only for its own direct child. If that child spawns its own children and dies, they re-parent to PID 1 (or to this supervisor, if it is PID 1 in a container) — which is the B2.9 problem, and the reason a real init has a general reaping loop in addition to supervising. Know the boundary of what you built.

    A child that fails during exec (bad path, not executable) exits 127 from the _exit(127) line — and your supervisor will dutifully restart it forever, backing off to the 30s cap. That is correct behaviour for a transient failure and maddening for a permanent one; real supervisors add a "give up after N failures in M seconds" rule (systemd's StartLimitBurst). Note where you would add it.

    You have just written the core of a kubelet. The kubelet forks and execs containers (through the runtime), waits on them, reads their exit status to decide restart, and applies a backoff — the CrashLoopBackOff you will meet in W6.16 is this exact 1-2-4-8 backoff, capped, under a different name. The restart-policy decision (exit 0 → stop vs. non-zero → restart) is the process-level version of a pod's restartPolicy.

    The signal-forwarding half is what a container's PID 1 must do for the processes beneath it, and its absence is the PID 1 problem from B2.9. Follow this forward into W1.19 — the kubelet's syncLoop and its three pod sources, which is the same loop scaled up to "which containers should exist on this node", and into B11, where this program becomes a systemd unit and stops being your job.

    B2.18

    Three symptoms, named from evidence

    Why it matters

    You meet faults symptom-first, never cause-first. Each of these three is one you will genuinely hit — on a laptop, a server, or a Kubernetes node — and each has the property that the obvious reaction is the wrong one. The skill being trained is walking from what you can see to what is true, using the specific evidence this module taught you to read.

    Take a snapshot first (snapshot.sh on your host), then work each symptom without reading ahead. State a one-sentence cause backed by a command that proves it, then open the reveals in order.

    Symptom 1 — the process table is filling with <defunct>
    1. Plant it. This starts a parent that forks many children and never reaps them. 
      cat > /tmp/leaky.py <<'PY'
#!/usr/bin/env python3
import os, time
while True:
    for _ in range(20):
        if os.fork() == 0:
            os._exit(0)          # child dies immediately
    time.sleep(2)                # parent never calls wait()
PY
python3 /tmp/leaky.py &
sleep 3
    2. Observe, then diagnose before reading on. 
      ps -eo stat= | grep -c '^Z'

    Stage 1 · Expected symptom

    The zombie count climbs and keeps climbing — 20, then 40, then 60. Every entry is <defunct>, all sharing one PPID. Memory is not moving (free -m is flat), which is the tell that this is not a memory leak despite looking like runaway growth. What is being consumed is PIDs, heading toward /proc/sys/kernel/pid_max, after which nothing new can start.

    Stage 2 · Hint

    A zombie is a dead child whose exit status was never collected (B2.9). It is a symptom whose cause is exactly one row up the tree — the parent that is not reaping. Find that parent by grouping the zombies: ps -eo ppid=,stat= | awk '$2 ~ /^Z/ {print $1}' | sort | uniq -c | sort -rn | head. One PID will own all of them.

    Stage 3 · Fix & fingerprint

    The named parent is leaky.py: it forks children and never calls wait(), so each exit leaves a zombie. Killing the zombies does nothing (they are already dead); the fix is the parent. Either fix its code to reap (add a SIGCHLD handler that calls os.wait() in a loop, or a wait() after each child), or, as the blunt operational fix, kill the parent so PID 1 inherits and reaps the lot:

    pkill -f leaky.py
sleep 1; ps -eo stat= | grep -c '^Z'
rm -f /tmp/leaky.py
    0

    Fingerprint: growing count of <defunct>, flat memory, one common PPID → a parent that stopped reaping. In a container this is the PID 1 problem, and it surfaces as node PIDPressure.

    Symptom 2 — a service takes exactly 30 seconds to stop, every time
    1. Plant it. This is a "service" that does its job but was written to ignore the polite stop signal. 
      cat > /tmp/stubborn-svc.py <<'PY'
#!/usr/bin/env python3

    SIGTERM does nothing — the process is still there after the wait, unbothered. The suspicious part is the exactly 30s, every time — a genuine slow shutdown varies, a fixed grace period is a constant.

    Stage 2 · Hint

    If a program does not respond to SIGTERM, ask whether it is listening. The SigCgt/SigIgn masks from B2.11 answer it directly: grep -E '^Sig(Cgt|Ign)' /proc/$svc/status.     Stage 3 · Fix & fingerprint

    SigIgn has bit 14 set (0x4000): the service explicitly ignores SIGTERM. It can only be stopped by SIGKILL, which is why every shutdown costs the full grace period and loses in-flight work. The operational fix is kill -9; the real fix is in the program — install a handler that drains and exits (B2.16), or if it is PID 1 in a container, ensure the handler is actually on the process receiving the signal.

    Symptom 3 — a process won't die, and kill -9 does nothing

    2. Observe. Sample its state repeatedly and try to kill it.

    Stage 1 · Expected symptom

    The process spends much of its time in state D (with R flickers between I/O operations), and kill -9 does not make it vanish immediately — it dies only when its current I/O operation completes.     Stage 2 · Hint

    D is uninterruptible sleep (B2.7): the process is inside a kernel call that will not accept a signal until it returns. So the question is never "why won't it die" but "what is it waiting on" — and WCHAN answers it. The wchan column names the kernel function it is asleep in; anything with blk/bio is a block device, anything with nfs/rpc is a network filesystem. That name is your real culprit.

    Stage 3 · Fix & fingerprint

    You cannot fix a D-state process by signalling it — there is no safe point to deliver the signal. The only resolutions are to let the I/O complete (as it does here once dd finishes) or to fix the layer underneath: remount or recover the failed storage, at which point the queued signals are finally delivered and the process dies. Killing is not on the menu.

    wait $dd_pid 2>/dev/null

    Fingerprint: process stuck in D, immune to kill -9, WCHAN naming a storage/network function → the layer beneath it is wedged; fix that, do not chase the process. A pile of D-state processes on a node is a storage incident, not a process incident, and shows up as a pod stuck Terminating.

    Verify

    Each drill is done when you can state its fingerprint in one sentence without looking, and when you fixed it at the cause rather than the symptom. Clean up (rm -f /tmp/leaky.py /tmp/stubborn-svc.py /tmp/dfile) and confirm no zombies remain (ps -eo stat= | grep -c '^Z').

    Gotchas

    All three tempt you to act on the symptom: kill the zombies, kill -9 the slow service without asking why, kill -9 harder at the D-state process. The first does nothing, the second hides a real bug that will keep costing you the grace period, and the third cannot work by construction. Every one of them is fixed one layer away from where it hurts — the parent, the program's signal handling, the storage.

    The reasoning generalises to the whole rest of the site: a symptom is a starting point, not a diagnosis. W6 applies exactly this discipline at cluster scale — read the evidence the system is actually giving you (ps state, signal masks, WCHAN; later, events, conditions, logs) before reaching for a fix, because the loudest symptom is rarely at the location of the cause.

    B2 recall quiz

    Answer out loud before revealing. Everything Kubernetes does to a workload is one of these facts about a process — if you cannot answer it here, it will not get easier wrapped in an API.

    1. Define a process in four parts, and name the one Linux object that a container actually is.
      Answer

      An address space (private memory), a table of open file descriptors, a set of credentials, and a PID. A container is not a distinct kernel object — it is an ordinary process with those four things adjusted (credentials, filesystem view, PID visibility, resource caps). There is nothing else there.

    2. Why is VSZ almost always far larger than RSS, and which one matters?
      Answer

      VSZ is address space reserved; RSS is physical memory occupied. Reserving costs nothing until a page is touched, so a process can map gigabytes it never uses. RSS is the number that competes for real memory — though even RSS double-counts shared pages, so it is the right per-process figure and the wrong thing to sum.

    3. What are the three system calls that create and manage a new program, and what happens in the gap between the first two?
      Answer

      fork (copy the caller), exec (replace the copy's program), wait (collect the result). In the gap between fork and exec the child is running its own code and can adjust itself — redirect descriptors, drop privileges, join a cgroup, enter namespaces — before becoming the target program. That gap is where a container is made.

    4. A process is in state D and will not respond to kill -9. What is happening and what do you do?
      Answer

      It is in uninterruptible sleep inside a kernel I/O call that will not accept a signal until it returns — almost always storage or a network filesystem. You cannot kill it; you read WCHAN to find what it is waiting on and fix that layer (the disk, the mount). A pile of D processes is an incident one level down, not a process problem.

    5. What is a zombie, what does it consume, and how do you get rid of one?
      Answer

      A dead child whose exit status has not yet been collected by its parent's wait(). It holds no memory and no CPU — only a PID and a status record. You cannot kill it (nothing is there); it is reaped when its parent calls wait, or, if the parent exits, when PID 1 inherits and reaps it. Thousands of them exhaust PIDs, not memory.

    6. Decode 137, 143 and 139. Which one tells you to read the application's logs?
      Answer

      128 + N, where N is the signal. 137 = SIGKILL (9) — killed from outside, usually the OOM killer. 143 = SIGTERM (15) — a normal shutdown. 139 = SIGSEGV (11) — the program's own memory bug. Only 139 is the application failing on its own, so only 139 makes its logs the right place to look; the other two were done to it from outside.

    7. What is a thread, in the kernel's terms, and what single system call creates both threads and containers?
      Answer

      A schedulable task that shares its address space and file-descriptor table with its siblings while keeping its own stack and kernel ID. clone() creates it — the same call that creates processes (sharing little) and, with namespace flags, containers (sharing nothing and getting private PID/network/mount views). One dial, many settings.

    8. Name the two signals that cannot be caught, and explain the operational consequence.
      Answer

      SIGKILL (9) and SIGSTOP (19). The kernel handles both without running the target's code, so a program can never block them — which guarantees you can always stop or freeze a process. The cost is that a SIGKILLed process gets no chance to clean up, so all graceful shutdown must happen during the SIGTERM you send first.

    9. Why should a signal handler set a flag rather than do the work itself?
      Answer

      A handler runs asynchronously, interrupting the main program at an arbitrary point — possibly mid-malloc or holding a lock. Doing real work there risks deadlock and corruption, because almost nothing is safe to call from a handler. The safe pattern is: handler sets a flag and returns; the main loop notices the flag and does the real work at a safe point.

    10. State the four steps of graceful shutdown, and what happens to a program that ignores SIGTERM.
      Answer

      Catch SIGTERM; stop accepting new work; finish in-flight work (drain); exit 0. A program that ignores TERM does not become robust — it forces its supervisor to wait the entire grace period and then send the uncatchable SIGKILL, which drops all in-flight work and reports 137. "Ignores TERM" means "always takes the maximum time and loses data", not "tougher".

    11. Why does a container's PID 1 need special care around both reaping and signals?
      Answer

      PID 1 inherits re-parented orphans, so it must reap them or they pile up as zombies — but an ordinary application at PID 1 has no reaping loop. And the kernel suppresses default signal actions for PID 1, so an app there with no explicit SIGTERM handler ignores TERM entirely. Both are fixed by a proper init (tini, the pause container) or by writing the handler and reap loop yourself.

    12. Structurally, what is a kubelet, in the vocabulary of this module?
      Answer

      A supervisor: it forks and execs children (containers), waits on them, reads exit status to tell "asked to stop" from "failed", and restarts failures with a capped backoff (which appears as CrashLoopBackOff). It is the mini-supervisor from B2.17 scaled up to "which processes should exist on this node", which is exactly the program you wrote.
