    B2 — Processes, threads, signals

    A container is a process. Not "like" a process, not "a lightweight virtual machine" — a perfectly ordinary Linux process, with some of its settings adjusted. That single sentence is the reason this module exists, and it is the one that makes the rest of the track make sense: everything Kubernetes does to a workload is something Linux has been doing to processes since the 1970s. Start it. Watch it. Ask it politely to stop. Wait a while. Give up and kill it. Report why it died.

    So this module is where you learn to narrate all of that, precisely, on a machine you can see all of. When you later read Exit Code: 137 in a pod's status at three in the morning, you will not be looking up what 137 means — you will already know that it is 128 + 9, that 9 is SIGKILL, that nothing inside the program chose to exit, and therefore that the answer is outside the program.

    Checkpoints18 Hours~6 PrerequisitesB1
    • Say precisely what a process is, in four parts, and point at each part on a live one.
    • Read ps aux column by column, and explain why VSZ is always far larger than RSS.
    • Read the process tree and trace any process up to PID 1.
    • X-ray a running process through /proc/PID/ without stopping it.
    • Explain why creating a process takes three system calls — fork, exec, wait — and what happens in the gap between the first two.
    • Interpret every process state, including the two that scare people: D and Z.
    • Read an exit code as a message, including the 128 + signal convention that produces 137 and 143.
    • Explain orphans, zombies, reaping, and what PID 1 owes its children.
    • Describe a thread as the kernel sees it, and know when the distinction changes your diagnosis.
    • Recall the signal table worth memorizing, and predict a signal's default action before sending it.
    • Catch a signal in your own program — and explain why two of them can never be caught.
    • Implement graceful shutdown: catch TERM, finish in-flight work, exit 0.
    • Write a mini-supervisor that forks, forwards signals, reaps, and restarts with backoff — a baby kubelet.

    B1. Every lab here reads /proc with the tools you already have — ls, cat, grep, pipes, quoting, $?. The mini project is a shell script plus a small C program and a small Python program; if gcc is missing, sudo apt install -y build-essential. Take a snapshot before you start (snapshot.sh from B0.9) — one drill deliberately wedges a process in a state you cannot kill.

    What a process is

    B2.1

    Define a process precisely — four things and a number

    Why it matters

    "Process" is one of those words everybody uses and few people can define, which is fine until you need to reason about one that is misbehaving. If your mental model is "a running program", you have no way to answer questions like why does this container see a different /etc/hosts, or how can two processes share memory, or what exactly does a container add. A four-part definition answers all three, and every later module in this track modifies exactly one of the four parts.

    The concept

    Start with the distinction that the word hides. A program is a file sitting on disk — /usr/bin/sleep is a program, and it is just bytes, doing nothing. A process is one running instance of a program: the kernel has read those bytes into memory, given them somewhere to live, and started executing them. One program can have a thousand processes running from it at once, and they do not interfere with each other in the slightest.

    In our hotel picture, a process is a guest staying in the hotel, and its PID is their room number. That gives you the two most-missed properties for free. Room numbers are unique while someone is in the room — no two guests share 412 today — but they get reused: next month a different guest is in 412, and a note addressed to "whoever is in 412" reaches the wrong person. Exactly the same is true of PIDs, and it is the reason you will see careful scripts check which process a PID is before signalling it.

    Now the definition. A process is four things plus that number:

    PartPlainlyWhere you can see it
    Address spaceits private memory — code, data, stack, and anything it mapped in/proc/PID/maps
    File descriptorsa numbered table of everything it has open: files, sockets, pipes, terminals/proc/PID/fd/
    Credentialswhich user and groups it runs as, and therefore what it may touch/proc/PID/status
    PIDits identifier while it lives, and its parent's in PPID/proc/PID/ itself

    "Private memory" is worth one extra sentence, because it is the property everything else rests on. Two processes can both have a valid address 0x400000, and those are two completely different bytes of physical memory — the kernel and the CPU together maintain a per-process translation from the addresses a process uses to the physical memory it actually occupies. A process therefore cannot read another process's memory by accident. It cannot even address it. That isolation is not a feature somebody added for security; it is how memory has worked on Linux from the start, and it is why "processes are already fairly isolated" is the honest starting point for understanding containers.

    Which brings us to the payoff. There is no container object in the Linux kernel. Nothing in the source code is called a container. What a container runtime does is start an ordinary process with all four of those parts adjusted — different credentials (B4), a different view of the filesystem so its descriptors reach different files (B5), a private set of PIDs so it cannot see the rest of the hotel (B6), and a cap on how much memory and CPU its address space may consume (B7). Strip those adjustments away and you have a plain process; that is all a container ever was.

    program on disk→process=address space+open descriptors+credentials+PID

    Verify

    Without looking, name the four parts of a process and one file under /proc/PID/ that shows each. Then answer: if a program is 2 MB on disk and you run 50 copies of it, roughly how much memory is used by the program's code? (Close to 2 MB total, not 100 MB — the read-only code pages are mapped from the same file and shared between all 50 address spaces. Private memory means each process cannot modify another's; it does not mean the kernel duplicates identical read-only pages.)

    Gotchas

    PID reuse is real and bites scripts. A script that records a PID, sleeps, and later runs kill $pid can signal an entirely unrelated process that has since inherited that number. On a busy machine the counter wraps in minutes. The robust patterns are to check identity first (ps -p "$pid" -o comm=), or to let something that tracks processes properly do it for you — which is precisely what systemd does in B11 and what your supervisor in B2.17 will do by holding a real parent-child relationship rather than a remembered number.

    /proc/PID/environ and /proc/PID/maps are readable only by the owner and root, so cat /proc/1/environ as an ordinary user gives "Permission denied" rather than an empty file. That is a credentials check, not a missing file, and sudo answers it. Do not conclude a /proc file is empty until you have tried it with privilege.

    A process is not the same thing as a job. [1] 2411 shows both: %1 is your shell's job number, meaningful only in that shell, and 2411 is the kernel's PID, meaningful everywhere. kill %1 and kill 2411 happen to do the same thing here, but only the second one works from another terminal.

    Everything in the pod spec is an adjustment to one of those four parts, and once you see that, the API stops looking like a grab-bag. securityContext.runAsUser sets the credentials. volumeMounts changes what the descriptors can reach. resources.limits caps the address space. shareProcessNamespace changes which PIDs are visible. There is no fifth kind of thing.

    The clearest place to see this from the Kubernetes side is W3.16 — requests vs limits: scheduler bookkeeping vs cgroup enforcement: a limit is not a promise the container makes, it is a number written into a kernel control file that applies to that process. B7 is where you write that file by hand.

    Docs

    man 5 proc is the reference for every file mentioned here — it is long, so search (/) for /proc/[pid]/status and /proc/[pid]/maps rather than reading front to back. man 7 credentials covers the four uid fields. For the definition itself, man 2 fork's opening paragraph is unexpectedly good on what a process consists of, because it has to list everything a copy inherits.

    B2.2

    Read ps column by column

    Why it matters

    ps is the first command you type on any machine that is behaving strangely, and most people read three of its columns and skip the rest. The skipped ones are where the answers are. Two of them — VSZ and RSS — are also responsible for a recurring false alarm: someone sees a process "using 2 GB" on a machine with 4 GB free and starts a fire drill over memory that was never occupied.

    The concept

    ps has two personalities because it inherited two different traditions, BSD and System V. ps aux is the BSD spelling (note: no dash), ps -ef is the System V spelling. They show almost the same processes in different columns, and you should be able to read both because both appear in every set of instructions on the internet. A third form, ps -eo …, lets you ask for exactly the columns you want, and it is the one worth building a habit around.

    Here is what each column is actually telling you:

    ColumnWhat it isWhat it is good for
    PIDthis process's numberthe handle for everything else you will do
    PPIDits parent's numberwho started it — the single most useful column, and absent from ps aux
    USERthe credentials it runs aswhether it can touch what you think it can
    STATits state, plus modifier letterswhether it is running, waiting, wedged or dead (B2.7)
    %CPUCPU time used ÷ time alivean average since it started, not a live reading
    %MEMRSS as a fraction of physical RAMa quick sort key, nothing more
    VSZvirtual size — address space reserved, in KBalmost never what you want
    RSSresident set size — physical memory actually occupied, in KBthe number to look at
    STARTwhen it began"did this start when the incident started?"
    TIMEtotal CPU time consumeddistinguishes busy from merely long-lived
    COMMANDhow it was invokedarguments included — often the whole diagnosis

    The VSZ/RSS pair deserves the explanation, because the gap is enormous and alarming until you know why. VSZ is a reservation; RSS is occupancy. A process can ask the kernel for an address range without ever touching it, and the kernel is happy to agree, because agreeing costs nothing — no physical memory is committed until the process actually reads or writes a page. Mapping a 10 GB file adds 10 GB to VSZ and nothing to RSS until you read from it. A language runtime that reserves a large heap up front looks terrifying in VSZ and modest in RSS. So: VSZ is what the process asked for, RSS is what it is holding, and only RSS is a resource anyone is competing for.

    One honest caveat on RSS, so you do not over-trust it either: shared pages are counted in full for every process that maps them. Fifty copies of the same program each report the shared library's pages in their own RSS, so adding up every RSS on a machine gives a total larger than the machine has. RSS is the right per-process number and the wrong thing to sum. free -m and, later, cgroup accounting (B7) are what answer "how much memory is really in use".

    From memory, produce a listing of the five biggest processes by real memory, showing who owns each and who started it — ps -eo pid,ppid,user,rss,comm --sort=-rss | head -6. Then explain to yourself why the machine is not in trouble even though one process shows a VSZ larger than the machine's total RAM.

    Gotchas

    ps aux has no PPID column, which is the column you most often need. Every "who launched this?" investigation stalls for a moment on that omission. ps -ef has it, and ps -eo lets you ask for it — build the habit of the third form and the problem disappears.

    Processes shown in square brackets — [kthreadd], [rcu_gp] — are kernel threads, not user programs. They have no address space of their own, which is why their VSZ and RSS are 0, and you cannot kill them or strace them meaningfully. When something in brackets is consuming CPU it is a signal about the kernel's work (I/O, memory reclaim), not about a program you can restart.

    %CPU from ps is a lifetime average and %CPU from top is a sample of the last interval. They disagree constantly and both are correct. If you want "what is hot right now", use top; if you want "what has consumed the most CPU since boot", use ps's TIME column, which is the honest cumulative figure.

    The COMMAND column is truncated by terminal width in ps aux, and the truncated part is usually the arguments that would have told you which config file the process is using. ps -ww -ef disables the truncation, or read /proc/PID/cmdline directly as in B2.4.

    The VSZ/RSS distinction is the reason kubectl top pod and a container's own memory reporting so often disagree with intuition. Kubernetes cares about a number closer to RSS — specifically the cgroup's accounting of pages actually charged to the container — because that is what competes for the node's memory and what the OOM killer acts on. A container with a huge virtual size and a small resident size is not close to its limit at all.

    Follow this into W3.17 — QoS classes: derived, not declared, where which pod gets killed first under memory pressure is decided by real usage against declared requests, and into W6.12 — reading node Conditions, where MemoryPressure is the node saying the same thing at machine scale.

    Docs

    man 1 ps is genuinely large; the sections to know are STANDARD FORMAT SPECIFIERS (every column name available to -o) and PROCESS STATE CODES (which B2.7 uses). man 1 free for the available column, which is the one to trust. man 5 proc under /proc/[pid]/statm shows where these numbers come from — ps is only formatting files you could read yourself.

    B2.3

    See the process tree, and trace yourself up to PID 1

    Why it matters

    Processes on Linux are not a flat list — they are a family tree, and the shape of that tree is frequently the whole answer. "Why is there a stray python3 eating CPU?" is unanswerable from a flat listing and obvious from a tree, where you can see it hanging off a cron job. Later, when you look at a container host, the tree is what shows you the runtime supervising the containers, and each container's processes hanging beneath it.

    The concept

    Every process except one has a parent: the process that created it. The exception is PID 1, started by the kernel at boot, which has no parent and is the root of the whole tree. On Ubuntu 24.04 that is systemd, which B11 covers properly. So from any process you can follow PPID upward, one step at a time, and you will always arrive at 1 — there is exactly one path, and no cycles.

    Continuing the hotel: PID 1 is the front desk, and every guest arrived by someone letting them in. Follow "who let you in?" back far enough and you always reach the front desk, because the building has only one entrance.

    Two things about the tree surprise people the first time. First, the parent link survives the parent's own commands — your shell is a child of the SSH daemon session, which is a child of systemd, and that chain does not change as you type. Second, the link can change: if a parent dies while its child is still running, the child is not killed. It is handed to PID 1, which becomes its new parent. That is called re-parenting, it is how orphans work, and B2.9 is entirely about the consequences.

    pstree draws this. pstree -p adds PIDs, which you almost always want. By default it collapses identical siblings into sleep(2411,2455,2456) form, which is a feature — a runaway that has forked 400 copies of itself shows as one line with 400 numbers, and the shape tells you instantly that you are looking at a fork problem rather than 400 unrelated events.

    Gotchas

    Names in curly braces — {containerd}(1877) — are threads, not child processes. pstree shows them by default and it makes a busy tree look far more populated than it is. pstree -T hides them, and B2.10 explains what they are.

    Killing a parent does not kill its children. This is the single most common wrong assumption about the tree: people kill a supervising process expecting the work to stop, and the work carries on happily, re-parented to PID 1 and now much harder to find. To stop a whole group you need something that tracks the group — kill -- -PGID for a process group, or the cgroup-based supervision systemd does in B11, which is exactly why systemd exists.

    The tree you see is the tree in your PID namespace. Inside a container, pstree shows the container's own process as PID 1 and nothing else on the machine — same kernel, different view. B6 builds that view from scratch, and until then it is worth knowing that "PID 1" is always relative to who is asking.

    On a Kubernetes node the tree is the architecture diagram. systemd at the top; kubelet and containerd as its children; under containerd, one containerd-shim per pod sandbox; and under each shim, the container's own PID 1 and its children. Running pstree -p on a node and finding your workload at the bottom of that chain is the moment "a pod is just processes on a machine" stops being a slogan.

    The supervising link at the top is the subject of W1.19 — the kubelet's syncLoop and its three pod sources: the kubelet is a process whose entire job is deciding which other processes should exist on this node, which is structurally the same job as the supervisor you will write in B2.17.

    Docs

    man 1 pstree — short, worth reading in full; note -T, -a (show arguments) and -s (show ancestors). man 5 proc under /proc/[pid]/stat documents the field order used in the walk-by-hand step, including why comm arrives wrapped in parentheses. man 1 ps under PROCESS SELECTION for --ppid, which lists a process's children directly.

    B2.4

    X-ray a live process through /proc/PID/

    Why it matters

    Almost every diagnostic tool you will ever use on Linux — ps, top, lsof, htop — is a formatter over /proc. Knowing that is not trivia: it means when a tool is missing, gives an answer you distrust, or has no flag for what you want, you can still get the truth by reading a file. On a stripped-down container image with no tools at all, /proc is often the only thing available, and it is enough.

    The concept

    /proc is not a directory of files on a disk. It is a filesystem-shaped interface to kernel data structures — you already saw in B1.3 that its files report size 0 while plainly having contents, and the reason is that the contents do not exist until you read them. The kernel generates the text at the moment of the read. B3 takes this apart properly; here, use it.

    Every live process has a directory /proc/PID/, and a handful of entries in it answer almost every question you will have:

    EntryAnswersNote
    cmdlineexactly how it was invoked, arguments includedNUL-separated, so pipe through tr '\0' ' '
    environthe environment it was giventhe copy made at exec — see B1.13
    fd/every open file, socket and pipe, as symlinksthe answer to "who is holding this file?"
    statusstate, uids, thread count, memory, signal maskshuman-readable summary of everything
    mapsthe layout of its address spacewhich libraries it actually loaded
    cwdits current directory, as a symlinkthe thing relative paths resolve against (B1.1)
    exethe binary it is running, as a symlinkstill valid after the file is deleted

    Two of those are worth calling out because they solve real problems. fd/ is how you find the process still holding a deleted 40 GB log file: when a file is unlinked but a process still has it open, the disk space is not freed (this is the link-count rule from B1.4), and ls -l /proc/*/fd shows the symlink with (deleted) appended. Nothing else on the machine will tell you. And exe keeps pointing at the original binary even after an upgrade replaces the file on disk, which is how you discover that a service is still running last week's version.

    The shorthand /proc/self/ always refers to the process doing the reading. So cat /proc/self/cmdline describes the cat, not your shell — a small joke the kernel plays on everyone once.

      /dev/sda1        19G  2.8G   16G  16% /

    6. The file is gone from the directory and the space is still used. Find the culprit through /proc — this is the move that no other tool gives you. 
      sudo ls -l /proc/*/fd 2>/dev/null | grep deleted
      l-wx------ 1 ubuntu ubuntu 64 Jul 23 09:31 3 -> /tmp/big.log (deleted)
    7. Release it by ending the holder, and confirm the space returns. In production this is why "I deleted the logs but the disk is still full" is answered by restarting the service, not by deleting more. 
      pkill -f 'sleep 600'
sleep 1
df -h /tmp | tail -1
      /dev/sda1        19G  2.3G   17G  13% /
    8. Finally, meet /proc/self and its small joke. 
      tr '\0' ' ' < /proc/self/cmdline; echo
pkill sleep
      tr \0
    Verify

    Pick any PID from ps -ef and, using only /proc, state: what command line it was started with, which directory it considers current, how many threads it has, and one file it currently has open. Then reproduce the deleted-file scenario from scratch and find the holder without looking at the lab above.

    Gotchas

    cat /proc/PID/cmdline looks like it prints everything joined together — sleep900--verbose — because the NUL separators are invisible in a terminal. Nothing is wrong; you are seeing the format correctly and the terminal is hiding the delimiters. Always pipe through tr.

    Reading environ, maps and fd/ of a process you do not own requires root, and the failure is a permission error rather than an empty result. This matters more than it sounds: environment variables routinely contain credentials, so the kernel protecting environ is the only thing stopping any user on the machine from harvesting every secret passed to every service. It is also why passing secrets by environment variable is weaker than passing them by file.

    Everything in /proc/PID/ is a snapshot generated at read time, so two reads a millisecond apart can disagree, and a process can exit between your ps and your cat — leaving you with "No such file or directory" for a PID that existed a moment ago. That is not an error to debug; it is the process having ended.

    lsof gives you the deleted-file answer more conveniently (sudo lsof +L1), and it is worth installing. But it is not present on minimal images, including most container images, and it is reading exactly the files you just read. Know the manual method; use the tool when it is there.

    This is the technique that survives the worst case in Kubernetes: a container with no shell, no ps, and nothing to exec into. From the node you can still find the container's PID and read everything about it through /proc, because the kernel is shared and /proc is the kernel's own interface. Distroless images make this the normal path, not the fallback.

    The Kubernetes-native version of the same move is W6.3 — kubectl debug: ephemeral containers for shell-less pods, which attaches a container full of tools into the target pod's namespaces so that its /proc view includes your workload. Same idea, wrapped in an API. And the deleted-file trap reappears at cluster scale as a node whose disk stays full after log rotation — which surfaces as DiskPressure in W6.12.

    Docs

    man 5 proc is the whole lesson in reference form — and on Ubuntu 24.04 it has been split, so also check man 5 proc_pid_status and man 5 proc_pid_maps if a subsection seems missing. man 8 lsof for +L1. man 1 tr for the \0 translation. When a tool's output confuses you, the honest next step is man 5 proc and reading the source of the number yourself.
