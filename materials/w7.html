    Week 7 — RBAC, ServiceAccounts & speed drills

    Two jobs this week. First, close out the 25% Cluster Architecture domain with the piece weeks 1–2 left open: how the apiserver decides who you are and what you may do — certificates, ServiceAccount tokens, the CSR onboarding flow, and RBAC's four objects. Second, convert everything you know into speed: imperative generators, jsonpath surgery, fast-edit patterns, timed sprints, and your first killer.sh session. Knowledge stopped being the bottleneck around week 4; from here the enemy is the clock. Ticking done here also ticks the tracker.

    request + credential→authentication — who?→authorization — may you?→admission→etcd

    Authentication & authorization internals

    7.1

    Consolidate authentication: three credentials, one rule — there is no User object

    Why it matters

    You've already met every credential type this cluster uses — client certs in week 2, the sa.key pair in the PKI map, bootstrap tokens during kubeadm join. This lesson snaps them into one picture, and gives you the 401-vs-403 discriminator that instantly classifies half of all "access denied" exam symptoms.

    The concept

    Every request hits the apiserver's authenticator chain: each configured authenticator either identifies the request or passes; the first success wins and yields a username + group list — nothing more. Kubernetes stores no User objects anywhere: a "user" exists only as whatever name a credential carries. That's why you can't kubectl get users, and why "create a user" tasks always mean "mint a credential" (a cert via the CSR API, or a ServiceAccount).

    CredentialVerified againstIdentity it yieldsYou built it in
    x509 client certcluster CA (ca.crt)CN = username, O = groups2.3
    ServiceAccount JWTsa.pub (signed by sa.key)system:serviceaccount:<ns>:<name>, group system:serviceaccounts2.5
    bootstrap tokenSecret in kube-systemsystem:bootstrap:<id>, group system:bootstrappers2.x (kubeadm join)
    no credential— (anonymous auth on by default)system:anonymous, group system:unauthenticated—

    The status codes then split cleanly: 401 Unauthorized = authentication failed (bad/expired credential — the apiserver doesn't know who you are). 403 Forbidden = authentication succeeded, authorization refused (it knows exactly who you are and the answer is no). A 403 always names the identity it resolved — read it; it tells you which credential you're actually using.

    Lab
    1. Ask the API who your kubeconfig makes you (works with any credential type): 
      kubectl auth whoami
      ATTRIBUTE   VALUE
Username    kubernetes-admin
Groups      [kubeadm:cluster-admins system:authenticated]
    2. On cp, talk to the apiserver with no credential. Health/version endpoints are open to anonymous; real resources are not: 
      curl -sk https://127.0.0.1:6443/version | grep gitVersion
curl -sk https://127.0.0.1:6443/api/v1/pods | grep -A2 message
        "gitVersion": "v1.35.1",
  "message": "pods is forbidden: User \"system:anonymous\" cannot list resource \"pods\" ...",
  "reason": "Forbidden",
  "code": 403

      No credential still produced an identity (system:anonymous) — and a 403: authn passed, authz said no.

    3. Now a broken credential — a garbage bearer token: 
      curl -sk -H 'Authorization: Bearer garbage' https://127.0.0.1:6443/api/v1/pods | grep -E 'message|code'
        "message": "Unauthorized",
  "code": 401

      Bad credential → 401, no identity at all. That's the discriminator: 401 = fix the credential, 403 = fix RBAC.

    4. Crack open a ServiceAccount JWT — it's three base64 chunks joined by dots; the middle one is the claims: 
      TOKEN=$(kubectl create token default)
echo "$TOKEN" | cut -d. -f2 | tr '_-' '/+' | base64 -d 2>/dev/null; echo
      {"aud":["https://kubernetes.default.svc.cluster.local"],
 "exp":1783742533,"iat":1783738933,
 "iss":"https://kubernetes.default.svc.cluster.local",
 "sub":"system:serviceaccount:default:default", ...}

      Audience, issue/expiry times, and the subject line that is the SA's username. The signature (third chunk) is what sa.pub verifies — 7.4 goes deeper.

    Verify

    You can state, for each row of the table, where its signing/verification key lives on cp — and you can classify 401 vs 403 without looking anything up.

    Gotchas

    Because users are just names on certificates, there is no revocation: you cannot "delete" a cert user, and Kubernetes checks no CRLs. Locking someone out means removing their RBAC bindings (or re-issuing the CA — a cluster-wide event). Short expirationSeconds on issued certs (7.3) is the mitigation. Also: system:masters in a cert's O field bypasses authorization entirely — no binding needed, no way to restrict it. That's why 2.3's super-admin.conf stays offline.

    Docs

    Docs search: "Authenticating" (Kubernetes docs — the strategies table) and "anonymous requests" on the same page.

    7.2

    Dissect a kubeconfig — clusters × users, joined by contexts

    Why it matters

    Every exam task begins with kubectl config use-context … — the one line you must run ~16 times without thinking. And "fix this broken kubeconfig" is a recurring task type: solvable in a minute if you know exactly which field lives where.

    The concept

    A kubeconfig is three parallel lists plus a pointer. clusters say where (server URL + the CA to trust it by). users say as whom (client cert+key, or a token). A context is just a named pairing of one cluster + one user, optionally with a default namespace. current-context points at one of them:

    Everything kubectl does starts by resolving current-context → (cluster, user, namespace). The classic broken-kubeconfig faults are one-field edits: wrong server port (timeout/refused), wrong certificate-authority-data ("x509: certificate signed by unknown authority" — you met this in 2.3), a user pointing at a missing key file, or a context naming a user that doesn't exist.

    Lab
    1. Read your real one — view redacts secrets, --minify shows only the current context's slice: 
      kubectl config view --minify
      clusters:
- cluster:
    certificate-authority-data: DATA+OMITTED
    server: https://192.168.64.2:6443
  name: kubernetes
contexts:
- context:
    cluster: kubernetes
    user: kubernetes-admin
  name: kubernetes-admin@kubernetes
current-context: kubernetes-admin@kubernetes
users:
- name: kubernetes-admin
  user:
    client-certificate-data: DATA+OMITTED
    client-key-data: DATA+OMITTED
    2. The two config commands you'll type all exam. List contexts, and pull one field with jsonpath (a 7.11 preview): 
      kubectl config get-contexts
kubectl config view --minify -o jsonpath='{.clusters[0].cluster.server}'
      CURRENT   NAME                          CLUSTER      AUTHINFO           NAMESPACE
*         kubernetes-admin@kubernetes   kubernetes   kubernetes-admin
https://192.168.64.2:6443
    3. Drill the namespace switch — this changes what "no -n" means for every later command: 
      kubectl config set-context --current --namespace=kube-system
kubectl get pods | head -3
kubectl config set-context --current --namespace=default
      Context "kubernetes-admin@kubernetes" modified.
NAME                         READY   STATUS    RESTARTS      AGE
coredns-7c65d6cfc9-9qhfx     1/1     Running   3 (2d ago)    21d
coredns-7c65d6cfc9-l7npk     1/1     Running   3 (2d ago)    21d

      kube-system pods with no -n flag — because the context now carries the namespace.

    4. Run each ten times until your fingers know them: kubectl config use-context <name> (switch the pointer) and kubectl config set-context --current --namespace=<ns> (edit the current pairing). Tab completion works on both.
    Verify

    Given any kubeconfig, you can answer in seconds: which server will this talk to, as whom, in which default namespace — and you can switch all three without reading help text.

    Gotchas

    Exam trap: a task's setup line sets a context whose namespace isn't default — every object you create lands there unless you notice. Make kubectl config get-contexts (the NAMESPACE column) part of your per-task ritual. Also, KUBECONFIG can hold a colon-separated list of files that kubectl merges — if a cluster you can't explain appears in config view, check that variable. And --minify fails if current-context is unset — that itself is a planted-fault signature.

    Docs

    Docs search: "Configure Access to Multiple Clusters" — the kubeconfig structure reference; "kubectl config" for subcommand syntax.

    7.3

    Onboard a human user via the CSR API — the full flow, from memory

    Why it matters

    This is a classic exam task, worth real points, and it's pure mechanics: five steps, zero improvisation. It's also 7.1 made concrete — you mint a certificate whose CN/O become a user the moment the cluster CA signs it. Drill it until the whole transcript takes under five minutes.

    The concept
    openssl key + CSR→CSR object (base64)→approve→extract signed cert→kubeconfig entry

    The CertificateSigningRequest API turns the cluster CA into a service: you submit a PEM CSR wrapped in a CSR object, an admin approves it, and the controller-manager (which holds ca.key) signs and publishes the certificate in the object's status. Three spec fields do all the work: request (your CSR, base64, one line), signerName: kubernetes.io/kube-apiserver-client (the signer for "certs that authenticate humans to the apiserver" — never auto-approved, by design), and usages: [client auth]. Optional expirationSeconds requests a lifetime (minimum 600); the signer grants the smaller of your request and its --cluster-signing-duration.

    Lab
    1. Mint the user's key and CSR — CN is the username, O the group (7.1's rule): 
      openssl genrsa -out jane.key 2048
openssl req -new -key jane.key -out jane.csr -subj "/CN=jane/O=devs"
    2. Wrap it in a CSR object. The base64 must be a single line — -w0: 
      cat <<EOF | kubectl apply -f -
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: jane
spec:
  request: $(base64 -w0 < jane.csr)
  signerName: kubernetes.io/kube-apiserver-client
  expirationSeconds: 86400
  usages:
  - client auth
EOF
kubectl get csr
      certificatesigningrequest.certificates.k8s.io/jane created
NAME   AGE   SIGNERNAME                            REQUESTOR          CONDITION
jane   4s    kubernetes.io/kube-apiserver-client   kubernetes-admin   Pending
    3. Approve — this signer always waits for a human decision: 
      kubectl certificate approve jane
kubectl get csr jane
      certificatesigningrequest.certificates.k8s.io/jane approved
NAME   AGE   SIGNERNAME                            REQUESTOR          CONDITION
jane   30s   kubernetes.io/kube-apiserver-client   kubernetes-admin   Approved,Issued
    4. The signed cert now sits in .status.certificate — extract and inspect it: 
      kubectl get csr jane -o jsonpath='{.status.certificate}' | base64 -d > jane.crt
openssl x509 -in jane.crt -noout -subject -dates
      subject=O = devs, CN = jane
notBefore=Jul 13 09:00:00 2026 GMT
notAfter=Jul 14 09:00:00 2026 GMT

      Exactly one day — the expirationSeconds you asked for, granted because it's below the signer's cap.

    5. Build her kubeconfig entry (in your existing file — same cluster, new user + context): 
      kubectl config set-credentials jane --client-certificate=jane.crt --client-key=jane.key --embed-certs
kubectl config set-context jane --cluster=kubernetes --user=jane
      User "jane" set.
Context "jane" created.
    6. Prove both halves of the story — she authenticates, and (having no RBAC yet) can do nothing: 
      kubectl --context=jane auth whoami
kubectl --context=jane get pods
      Username   jane
Groups     [devs system:authenticated]
Error from server (Forbidden): pods is forbidden: User "jane" cannot list resource "pods" ...

      Forbidden is success here — 403, not 401: the cert works, the user exists, RBAC just hasn't granted anything. The RBAC group fixes that with this exact jane.

    Verify

    kubectl --context=jane auth whoami returns jane + devs, and the whole flow — six commands — ran without you opening the docs.

    Gotchas

    The three spec traps, in order of popularity: request must be the base64 of the CSR (not the key, not the cert) with no line wraps; usages must be exactly client auth; signerName is the exact string kubernetes.io/kube-apiserver-client (the similar kubelet-serving signer rejects CN patterns like this). A CSR that sits Pending forever is normal — nothing auto-approves this signer. And if the issued cert seems to ignore your expirationSeconds, the signer's --cluster-signing-duration cap won — check the controller-manager manifest.

    Docs

    Docs search: "Certificate Signing Requests" — the "Normal user" section of that page is this lesson's transcript in official form (copy the object skeleton from there during the exam).

    7.4

    Modern ServiceAccount tokens: TokenRequest, projected volumes, rotation

    Why it matters

    Old tutorials (and old muscle memory) say "every ServiceAccount has a token Secret". That world ended in v1.24. Exam tasks and real debugging both require the current model: tokens are minted on demand, time-bound, and rotated under your pods without their knowledge.

    The concept

    A ServiceAccount object contains no credential at all — it's just a name RBAC can bind to. Tokens come from the TokenRequest API (a subresource of the SA): the apiserver signs a JWT with sa.key on the spot, scoped to an audience and a lifetime, and never stores it. Two consumers:

    • kubectl create token <sa> — the CLI front-end. Default lifetime 1 hour; --duration to change it.
    • Pods — the admission chain injects a projected volume at /var/run/secrets/kubernetes.io/serviceaccount combining three sources: a serviceAccountToken (the kubelet calls TokenRequest for it), the cluster CA bundle (a ConfigMap), and the namespace (downward API). The kubelet rotates the token when it passes 80% of its TTL (or 24h) — it rewrites the file; well-behaved clients re-read it rather than caching.

    Tokens bound to a pod carry that pod's identity in their claims and die with the pod — a stolen token from a deleted pod is useless. Legacy non-expiring Secret tokens aren't auto-created anymore; the only reason to make one today is an explicit task demanding a long-lived token (there's a sanctioned Secret type for it — see Gotchas).

    Lab
    1. Create an SA and look for its token — there isn't one: 
      kubectl create serviceaccount probe
kubectl describe sa probe | grep -A1 Tokens
kubectl get secrets
      serviceaccount/probe created
Tokens:              <none>
No resources found in default namespace.
    2. Mint one on demand and read its claims (the 7.1 trick): 
      TOKEN=$(kubectl create token probe --duration=10m)
echo "$TOKEN" | cut -d. -f2 | tr '_-' '/+' | base64 -d 2>/dev/null; echo
      {"aud":["https://kubernetes.default.svc.cluster.local"],
 "exp":1783739533,"iat":1783738933,          ← exp − iat = 600s: exactly your --duration
 "sub":"system:serviceaccount:default:probe", ...}
    3. Now the pod path. Run a pod under this SA and inspect what admission injected: 
      kubectl run tokenpod --image=busybox:1.36 \
  --overrides='{"spec":{"serviceAccountName":"probe"}}' -- sleep 3600
kubectl get pod tokenpod -o yaml | grep -A12 'projected:'
          - projected:
        sources:
        - serviceAccountToken:
            expirationSeconds: 3607
            path: token
        - configMap:
            name: kube-root-ca.crt
        - downwardAPI:
            items:
            - fieldRef: {fieldPath: metadata.namespace}
              path: namespace

      You never wrote a volume — the ServiceAccount admission plugin did. 3607s ≈ 1h is the kubelet's requested TTL.

    4. See the rotation machinery from inside — the files are symlinks through a swappable ..data dir, so the kubelet can replace all three atomically: 
      kubectl exec tokenpod -- ls -l /var/run/secrets/kubernetes.io/serviceaccount/
      ca.crt -> ..data/ca.crt
namespace -> ..data/namespace
token -> ..data/token
    5. Clean up: 
      kubectl delete pod tokenpod
    Verify

    You can explain where a pod's token comes from (kubelet → TokenRequest → projected file), why kubectl get secrets shows nothing for a new SA, and what happens at 80% of the token's TTL.

    Gotchas

    If a task really demands a long-lived token (CI systems, external clients), the sanctioned form is a Secret of type kubernetes.io/service-account-token carrying the annotation kubernetes.io/service-account.name: <sa> — the controller fills in a non-expiring token. Don't reach for it otherwise. In-pod clients that read the token once at startup break mysteriously after rotation — "worked for a day, then 401" is that signature. And automountServiceAccountToken: false (on the SA or pod) removes the volume entirely: a pod that can't find /var/run/secrets/… isn't broken, it's hardened.

    Docs

    Docs search: "Managing Service Accounts" (token mechanics, bound tokens) and "Configure Service Accounts for Pods" (the long-lived Secret recipe).

    7.5

    The authorization lineup: Node,RBAC — order matters, first decision wins

    Why it matters

    RBAC isn't alone in the gate. Knowing what --authorization-mode=Node,RBAC actually means explains one weird Forbidden message you'll meet in kubelet debugging, and answers the direct exam question "which authorization modes does this cluster run?"

    The concept

    Authorization is a chain of authorizers consulted in flag order. Each returns one of three verdicts: allow (request proceeds, chain stops), deny (request fails, chain stops), or no opinion (next authorizer). If the chain ends with nobody opining — deny. RBAC, notably, can only allow or abstain — it has no deny verdict, which is why RBAC rules are purely additive and "revoke" always means "delete a binding".

    kubeadm's lineup is Node,RBAC. The Node authorizer runs first and handles exactly one clientele: identities named system:node:<name> in group system:nodes — the kubelets. It grants each kubelet read access only to the objects related to pods bound to that node (its ConfigMaps, Secrets, PVCs…), plus writes to its own Node status and pod statuses. A generic RBAC rule can't express "only secrets your pods mount" — the Node authorizer computes that relationship graph live. Everyone else falls through to RBAC. (Other modes exist — Webhook, ABAC, AlwaysAllow/AlwaysDeny — and newer clusters can define the same chain in a structured --authorization-config file instead of the flag; kubeadm still uses the flag, and the two are mutually exclusive.)

    Lab
    1. Read the lineup off your apiserver: 
      sudo grep authorization-mode /etc/kubernetes/manifests/kube-apiserver.yaml
          - --authorization-mode=Node,RBAC
    2. Borrow the kubelet's identity (its kubeconfig from 2.3) and see who the Node authorizer thinks you are: 
      sudo kubectl --kubeconfig /etc/kubernetes/kubelet.conf auth whoami
      Username   system:node:cp
Groups     [system:nodes system:authenticated]
    3. Watch the relationship rule bite. Create a secret no pod on cp mounts, then ask for it as the kubelet: 
      kubectl create secret generic topsecret --from-literal=k=v
sudo kubectl --kubeconfig /etc/kubernetes/kubelet.conf get secret topsecret
      Error from server (Forbidden): secrets "topsecret" is forbidden:
User "system:node:cp" cannot get resource "secrets" ...: no relationship found between node 'cp' and this object

      That phrase is the Node authorizer's signature — you'll never see RBAC say it.

    4. But node-related reads sail through the same credential: 
      sudo kubectl --kubeconfig /etc/kubernetes/kubelet.conf get node cp
      NAME   STATUS   ROLES           AGE   VERSION
cp     Ready    control-plane   21d   v1.35.1
    5. Keep topsecret — 7.11's drills decode it.
    Verify

    You can recite the three verdicts and the ordering rule, say which identities the Node authorizer serves, and explain why RBAC has no deny.

    Gotchas

    Order is meaningful but rarely a fault: with Node,RBAC, non-node requests pass through Node as "no opinion" at zero cost. The trap is conceptual questions: "add a deny rule for user X" is a trick — RBAC can't; you remove/narrow bindings instead. And don't confuse the Node authorizer with the NodeRestriction admission plugin (also on by default in kubeadm): the authorizer limits what kubelets may read, NodeRestriction limits what they may modify (only their own Node and their own pods' status).

    Docs

    Docs search: "Authorization Overview" (modes + decision flow) and "Using Node Authorization".

    RBAC in practice

    7.6

    The four-object matrix: Role/ClusterRole × RoleBinding/ClusterRoleBinding

    Why it matters

    Every RBAC exam task is assembled from four object kinds and one question: does a binding connect this subject to a rule allowing this verb on this resource in this scope? Get the matrix straight once and every task becomes fill-in-the-blanks.

    The concept

    Two kinds hold rules (what can be done), two kinds hold grants (who gets the rules, where):

    RoleBinding (grants in ONE namespace)ClusterRoleBinding (grants EVERYWHERE)
    Role (rules live in a ns)classic per-ns access — rules and grant both scoped to that ns✗ impossible — a ClusterRoleBinding can't reference a Role
    ClusterRole (rules are global)the reuse pattern: one shared rule-set, granted per-namespace — the binding's ns is the boundaryfull cluster-wide grant; also the only way to grant cluster-scoped resources (nodes, PVs, namespaces) and non-resource URLs

    A rule is apiGroups + resources + verbs (optionally resourceNames). The RBAC authorizer is purely additive: it looks for any binding→rule path that allows the request; finding none means no-opinion (→ deny at chain's end, per 7.5). Two hard scope laws: a namespaced Role can never grant cluster-scoped things like nodes — writing one succeeds but grants nothing (favorite trick question); and a RoleBinding to a ClusterRole grants those rules only inside the binding's namespace.

    Lab

    Uses jane from 7.3 — recreate her if the cert expired.

    1. Quadrant 1 — Role + RoleBinding in a fresh namespace: 
      kubectl create namespace dev
kubectl -n dev create role pod-reader --verb=get,list,watch --resource=pods
kubectl -n dev create rolebinding jane-pods --role=pod-reader --user=jane
      namespace/dev created
role.rbac.authorization.k8s.io/pod-reader created
rolebinding.rbac.authorization.k8s.io/jane-pods created
    2. Prove the scope edge from jane's side — allowed in dev, Forbidden one namespace over: 
      kubectl --context=jane -n dev get pods
kubectl --context=jane get pods
      No resources found in dev namespace.        ← empty list = the read WORKED
Error from server (Forbidden): pods is forbidden: User "jane" cannot list resource "pods" in API group "" in the namespace "default"
    3. Quadrant 3, the reuse pattern — one ClusterRole, granted only in dev by a RoleBinding: 
      kubectl create clusterrole cm-reader --verb=get,list --resource=configmaps
kubectl -n dev create rolebinding jane-cms --clusterrole=cm-reader --user=jane
kubectl --context=jane -n dev get configmaps
kubectl --context=jane get configmaps -A
      clusterrole.rbac.authorization.k8s.io/cm-reader created
rolebinding.rbac.authorization.k8s.io/jane-cms created
NAME               DATA   AGE
kube-root-ca.crt   1      2m
Error from server (Forbidden): configmaps is forbidden: User "jane" cannot list resource "configmaps" ... at the cluster scope

      Global rules, namespace-sized grant. This is how the built-in view/edit/admin ClusterRoles are meant to be used.

    4. Read a binding the way the authorizer does — subject, roleRef, namespace: 
      kubectl -n dev describe rolebinding jane-cms
      Role:
  Kind:  ClusterRole
  Name:  cm-reader
Subjects:
  Kind  Name  Namespace
  User  jane
    Verify

    You can fill the 2×2 table from memory, including which quadrant is impossible, and jane's four results (works/Forbidden × pods/configmaps) match your prediction before each command.

    Gotchas

    A binding's roleRef is immutable — pointing an existing binding at a different role means delete + recreate, not edit (kubectl tells you, but only after your edit bounces). Subjects have three kinds — User, Group, ServiceAccount — and only ServiceAccounts have a namespace field; binding to group devs would cover every cert with O=devs, jane included. And before hand-writing any rule, check whether a built-in ClusterRole (view, edit, admin) already says what you mean.

    Docs

    Docs search: "Using RBAC Authorization" — the one page with every object, the built-in roles table, and copyable examples.

    7.7

    Imperative RBAC: create role / rolebinding until it's instant

    Why it matters

    RBAC YAML is verbose and easy to typo under pressure; the imperative forms are one line each and generate correct objects every time. This is the difference between a 3-minute task and an 8-minute one.

    The concept

    Four commands cover every RBAC task. The grammar to internalize:

    CommandKey flags
    kubectl create role--verb=get,list,watch --resource=pods [--resource-name=web] [-n ns]
    kubectl create clusterrolesame, plus --resource=nodes etc. works here
    kubectl create rolebinding--role=X or --clusterrole=X, then --user=jane / --group=devs / --serviceaccount=ns:name
    kubectl create clusterrolebinding--clusterrole=X + the same three subject flags

    Details that matter: --resource accepts singular, plural, and short names (deploy, cm) and infers the apiGroup; subresources use a slash (--resource=pods/log); --serviceaccount is always namespace:name; --resource-name narrows a rule to named objects (but can never gate create — the name doesn't exist yet at authorization time).

    Lab
    1. Warm up with $do — see what each one-liner generates before creating anything: 
      kubectl create role log-reader --verb=get --resource=pods/log $do
      apiVersion: rbac.authorization.k8s.io/v1
kind: Role
rules:
- apiGroups: [""]
  resources:
  - pods/log
  verbs: ["get"]
    2. The apiGroup inference — deployments land in apps without you saying so: 
      kubectl create clusterrole deploy-admin --verb=create,delete --resource=deploy $do | grep -B1 -A3 apiGroups
      rules:
- apiGroups:
  - apps
  resources:
  - deployments
    3. All three subject flags, dry-run, back to back: 
      kubectl create rolebinding b1 --role=pod-reader --user=jane -n dev $do | grep -A4 subjects
kubectl create rolebinding b2 --clusterrole=view --group=devs -n dev $do | grep -A4 subjects
kubectl create rolebinding b3 --clusterrole=view --serviceaccount=dev:app-sa -n dev $do | grep -A4 subjects
      subjects:
- kind: User,  name: jane      (apiGroup: rbac.authorization.k8s.io)
- kind: Group, name: devs      (apiGroup: rbac.authorization.k8s.io)
- kind: ServiceAccount, name: app-sa, namespace: dev

      (Output condensed — run each to see the full YAML. Note only the SA subject carries a namespace.)

    4. Now the drill: without $do, create and delete each of these five times, racing yourself — role, clusterrole, rolebinding to a role, rolebinding to a clusterrole, clusterrolebinding: 
      kubectl -n dev create role r1 --verb=get --resource=pods
kubectl -n dev create rolebinding rb1 --role=r1 --user=jane
kubectl -n dev delete rolebinding rb1 ; kubectl -n dev delete role r1

      Target: any single object in under 15 seconds, from memory.

    Verify

    Each of the four create commands comes out of your fingers correctly on the first try — including --serviceaccount=ns:name ordering — with no --help peeks.

    Gotchas

    The binding flags are --role or --clusterrole — passing a ClusterRole name to --role creates a binding to a Role that doesn't exist, and RBAC treats a dangling roleRef as "grants nothing" (no error at create time!). --serviceaccount=name without the namespace prefix is rejected — it's always ns:name. And verbs are a fixed vocabulary (get list watch create update patch delete deletecollection) — inventing read fails at create, but a misspelled resource often doesn't: RBAC rules aren't validated against real resource types.

    Docs

    Docs search: "kubectl create role" / "kubectl create rolebinding" — reference pages with every flag; the RBAC page's "kubectl create" section has the same in context.

    7.8

    Verify like the exam does: kubectl auth can-i (and impersonation)

    Why it matters

    Habit 8.8 — verify every answer — applied to RBAC. auth can-i asks the apiserver's real authorization chain, so it's proof, not opinion. It's also how you debug someone else's permissions without ever holding their credentials.

    The concept

    kubectl auth can-i VERB RESOURCE creates a SelfSubjectAccessReview — the server runs its authorizer chain (Node, RBAC, everything) for that exact request and answers yes or no. Add --as=user / --as-group=group and kubectl sends impersonation headers: the apiserver evaluates the question as that identity instead. Impersonation is itself RBAC-gated (verb impersonate) — your admin context has it via cluster-admin; jane doesn't. ServiceAccounts impersonate by their full username: --as=system:serviceaccount:<ns>:<name>. And --list flips the question from "may I do X?" to "what may I do here?" — the whole permission surface as a table.

    Lab
    1. Check jane's grants from 7.6 — no context switching, just impersonation: 
      kubectl auth can-i list pods --as=jane -n dev
kubectl auth can-i list pods --as=jane
kubectl auth can-i create deployments --as=jane -n dev
      yes
no
no
    2. The SA form — full username, namespace baked in: 
      kubectl auth can-i list pods --as=system:serviceaccount:dev:app-sa -n dev
      no        ← app-sa doesn't exist yet; 7.9 creates and binds it, then this flips
    3. The whole surface at once: 
      kubectl auth can-i --list --as=jane -n dev
      Resources     Resource Names   Verbs
configmaps    []               [get list]
pods          []               [get list watch]
selfsubjectreviews.authentication.k8s.io  []  [create]
...                                            ← system:authenticated boilerplate below
    4. It's a real yes/no exit code — scriptable verification: 
      kubectl auth can-i get nodes --as=jane; echo "exit=$?"
      no
exit=1
    5. Group membership counts — jane's cert carries O=devs, so a group grant reaches her. Predict, then test: 
      kubectl -n dev create rolebinding devs-view --clusterrole=view --group=devs
kubectl auth can-i list secrets --as=jane --as-group=devs -n dev
kubectl -n dev delete rolebinding devs-view
      rolebinding.rbac.authorization.k8s.io/devs-view created
no          ← view excludes secrets, deliberately — good trivia
rolebinding.rbac.authorization.k8s.io "devs-view" deleted

      Note the mechanics: --as=jane alone impersonates the bare user without her groups — add --as-group=devs to test group-granted access.

    Verify

    Every RBAC object you created this week has a matching can-i proof: yes where you granted, no where you didn't — and you can write the SA impersonation string without pausing.

    Gotchas

    The big one: impersonating a user does not impersonate their groups — the apiserver can't know cert groups without seeing the cert, so --as=jane tests bare-jane and you must add --as-group=devs yourself. (Testing with --context=jane uses the real cert, groups included — that's why 7.6 did both.) can-i --list includes rows every authenticated user has — don't read those as your grant working. And in scripts, a "no" answer exits 1: set -e will kill your verification script on a correct negative.

    Docs

    Docs search: "Checking API Access" (can-i, SelfSubjectAccessReview) and "user impersonation" on the Authenticating page.

    7.9

    Wire a pod to a ServiceAccount end-to-end — and prove it with curl

    Why it matters

    This is the full workload-identity story in one lab: SA → Role → binding → pod → in-pod API call. It's a complete exam task type by itself, and it welds together 7.4 (where the token lives) with 7.6–7.8 (what it may do).

    The concept

    A pod runs as exactly one ServiceAccount, set by spec.serviceAccountName (default: the namespace's default SA — every namespace has one). In-pod clients authenticate with the three projected files from 7.4: send the token as a Bearer header, trust the server via ca.crt. The apiserver is always reachable from any pod at the DNS name kubernetes.default.svc (port 443 → the real :6443 via the Service you traced in week 4). So the proof loop is: bind a Role to the SA, curl the API from inside, watch an allowed call return a resource list and a non-granted call return a 403 naming the SA.

    Lab
    1. Identity and grant (reuses dev and its pod-reader Role from 7.6): 
      kubectl -n dev create serviceaccount app-sa
kubectl -n dev create rolebinding app-sa-pods --role=pod-reader --serviceaccount=dev:app-sa
kubectl auth can-i list pods --as=system:serviceaccount:dev:app-sa -n dev
      serviceaccount/app-sa created
rolebinding.rbac.authorization.k8s.io/app-sa-pods created
yes          ← 7.8's "no" just flipped
    2. A pod that runs as it (curl-equipped image, held open with sleep): 
      kubectl -n dev run api-client --image=curlimages/curl:8.8.0 \
  --overrides='{"spec":{"serviceAccountName":"app-sa"}}' -- sleep 3600
kubectl -n dev get pod api-client -o jsonpath='{.spec.serviceAccountName}'; echo
      pod/api-client created
app-sa
    3. Inside, assemble the call from the projected files: 
      kubectl -n dev exec -it api-client -- sh
cd /var/run/secrets/kubernetes.io/serviceaccount
curl -s --cacert ca.crt -H "Authorization: Bearer $(cat token)" \
  https://kubernetes.default.svc/api/v1/namespaces/dev/pods | head -6
      {
  "kind": "PodList",
  "apiVersion": "v1",
  "metadata": {
    "resourceVersion": "48211"
  },

      A PodList, over TLS, authorized purely by the RoleBinding — no kubeconfig anywhere in this pod.

    4. Now a call you did not grant — still inside the pod: 
      curl -s --cacert ca.crt -H "Authorization: Bearer $(cat token)" \
  https://kubernetes.default.svc/api/v1/namespaces/dev/secrets | grep -E 'message|code'
exit
        "message": "secrets is forbidden: User \"system:serviceaccount:dev:app-sa\" cannot list resource \"secrets\" ...",
  "code": 403

      The 403 names the SA — the token authenticated fine; RBAC drew the line exactly where you put it.

    5. Clean up the pod (SA, Role and bindings stay for the sprint drills): 
      kubectl -n dev delete pod api-client $now
    Verify

    From inside the pod: pods list returns 200/PodList, secrets returns 403 naming system:serviceaccount:dev:app-sa — both matching your can-i predictions from the outside.

    Gotchas

    serviceAccountName is immutable on a running pod — "make this pod use SA X" means recreate the pod (or edit the Deployment's template and let the rollout do it; that's usually what the task wants). The token in the volume is bound to this pod: copying it elsewhere works only until the pod dies. And if curl gets "certificate signed by unknown authority" in-pod, you forgot --cacert ca.crt — the apiserver's cert isn't in public trust stores.

    Docs

    Docs search: "Configure Service Accounts for Pods" — includes the in-pod API access recipe; "Accessing the API from a Pod" for the endpoint details.

    7.10

    Imperative generators: scaffold everything, write YAML never

    Why it matters

    Nearly every "create X" task starts with a generator plus $do (your --dry-run=client -o yaml variable from 0.9). People who write manifests from scratch run out of exam; people who scaffold-edit-apply finish early. This drill makes the full generator vocabulary reflexive.

    The concept

    The complete set worth knowing — each emits correct YAML with $do, or creates directly without it:

    TaskGenerator
    bare Podk run web --image=nginx:1.27 --port=80 --labels=app=web
    Pod running a commandk run tester --image=busybox:1.36 -- sh -c "sleep 3600"
    Deploymentk create deploy web --image=nginx:1.27 --replicas=3 --port=80

    Secretk create secret generic app-sec --from-literal=PASS=hunter2
    Namespace / SA / quotak create ns team1 · k create sa app-sa · k create quota q1 --hard=pods=10

    Lab
    1. Run the whole table once with $do, reading each output — notice what run vs create deploy emit, where labels land, what expose copies the selector from:

    7.11

    Output surgery: jsonpath, custom-columns, sort-by — a drill set

    Why it matters

    Tasks phrased as "write the X of all Y to /path/file.txt" are pure output surgery — full points for one correct line, zero for a near-miss. And in troubleshooting, extracting exactly one field beats scrolling YAML every time.

    The concept

    Four tools, in the order you should reach for them:

    • -o jsonpath='{...}' — path into any object: {.items[*].metadata.name}. Iterate with {range .items[*]}…{end} to control separators ({"\n"} for one-per-line — files usually want this). Filter lists with [?(@.type=="InternalIP")].
    • -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName — jsonpath per column, table out. Easier than range/end when you want several fields.
    • --sort-by='{.status.containerStatuses[0].restartCount}' — server gives arbitrary order; sort-by takes a jsonpath.
    • The honest fallback: -o yaml | grep -A5 <key> — when the path won't come to you in 30 seconds, stop composing and grep. It's on the sheet because it wins points, not style contests.

    Discovery when you don't know the path: run -o yaml on one object and read the real field names — or kubectl explain pod.status --recursive | less.

    Lab

    Eight drills. For each: write the command yourself first, run it, then check against the reveal. Wrong or slow → repeat tomorrow.

    1. All pod names in kube-system, one per line.
      Solution
      kubectl -n kube-system get pods -o jsonpath='{range .items[*]}{.metadata.name}{"\n"}{end}'
    2. The InternalIP of every node (space-separated is fine).
      Solution
      kubectl get nodes -o jsonpath='{.items[*].status.addresses[?(@.type=="InternalIP")].address}'

      The filter selects which list entry; note double quotes inside single-quoted jsonpath.

    3. Every container image running in the cluster, deduplicated.
      Solution
      kubectl get pods -A -o jsonpath='{range .items[*]}{range .spec.containers[*]}{.image}{"\n"}{end}{end}' | sort -u

      Nested ranges: pods, then containers within each.

    4. A table of pod name + node it runs on, all namespaces.
      Solution
      kubectl get pods -A -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName
    5. All pods sorted by restart count — most-restarted last.
      Solution
      kubectl get pods -A --sort-by='{.status.containerStatuses[0].restartCount}'
    6. Decode the value of key k in secret topsecret (from 7.5).
      Solution
      kubectl get secret topsecret -o jsonpath='{.data.k}' | base64 -d; echo
    7. The current context's apiserver URL, from the kubeconfig.
      Solution
      kubectl config view --minify -o jsonpath='{.clusters[0].cluster.server}'
    8. Each node's name and its taints (key only), one node per line.
      Solution
      kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{": "}{.spec.taints[*].key}{"\n"}{end}'

      Untainted nodes print an empty value after the colon — that's fine.

    Verify

    Drills 1, 2 and 6 come out right on the first attempt (they're the three most exam-frequent shapes); the rest within one retry.

    Gotchas

    Always single-quote the whole jsonpath — $, * and " inside are shell food otherwise. {.items[*]} with a plain * joins values with spaces; only range/end gives you newlines. Filters compare with == (not =) and need @ for "current element". When a task says "to a file", redirect the whole command and then cat the file — a jsonpath with no trailing newline can make the file look empty on a lazy glance.

    Docs

    Docs search: "JSONPath Support" — the syntax table with examples; "kubectl Quick Reference" has the most common one-liners under "Formatting output".

    7.12

    Fast pod edits: edit where you can, replace --force where you must

    Why it matters

    "Change this field on this pod" tasks split into two speeds: mutable fields are a 20-second kubectl edit; immutable ones need the dump-edit-replace loop. Knowing which is which before you start is the speed trick — finding out via a rejected edit costs a minute each time.

    The concept

    On a live Pod, only a short list is mutable: spec.containers[*].image (and initContainers'), spec.activeDeadlineSeconds, spec.tolerations (additions only), and spec.terminationGracePeriodSeconds (limited). Everything else — env, args, volumes, probes, serviceAccountName, nodeSelector, labels-in-spec — is immutable: the apiserver rejects the update outright. The escape hatch is recreate-in-place:

    kubectl get pod X -o yaml > f.yaml→edit f.yaml→kubectl replace --force -f f.yaml

    replace --force deletes the pod and creates it again from your file in one command — new UID, brief downtime, same name. Critical fork: this is for bare pods. A pod owned by a Deployment/DaemonSet should never be surgically edited — change the controller's template (fully mutable) and let the rollout replace pods properly.

    Lab
    1. Create a pod with a wrong env value, then watch edit refuse to fix it: 
      kubectl run editme --image=nginx:1.27 --env=MODE=staging
kubectl edit pod editme        # change MODE's value to "prod", save, quit
      error: pods "editme" is invalid
# * spec: Forbidden: pod updates may not change fields other than
#   `spec.containers[*].image`, `spec.initContainers[*].image`,
#   `spec.activeDeadlineSeconds`, `spec.tolerations` (only additions...) ...
# A copy of your changes has been stored to "/tmp/kubectl-edit-1408.yaml"

      Two gifts in that failure: the canonical mutable-field list, and your rejected buffer saved to /tmp.

    2. The loop that works — under 60 seconds total: 
      kubectl get pod editme -o yaml > /tmp/editme.yaml
vim /tmp/editme.yaml           # fix MODE, ignore the status:/managedFields noise
kubectl replace --force -f /tmp/editme.yaml
      pod "editme" deleted
pod/editme replaced
    3. Confirm the change landed (and notice the pod is new — AGE reset): 
      kubectl get pod editme -o jsonpath='{.spec.containers[0].env}'; echo
kubectl get pod editme
      [{"name":"MODE","value":"prod"}]
NAME     READY   STATUS    RESTARTS   AGE
editme   1/1     Running   0          9s
    4. Contrast: the one field you never needed the loop for — 
      kubectl set image pod/editme editme=nginx:1.28
kubectl get pod editme -o jsonpath='{.spec.containers[0].image}'; echo
      pod/editme image updated
nginx:1.28
    5. Clean up: 
      kubectl delete pod editme $now
    Verify

    You can recite the mutable-fields list, and the dump-edit-replace loop runs from your fingers in under a minute.

    Gotchas

    Run the loop on a Deployment-owned pod and the controller instantly builds a replacement from the old template — your fix evaporates; edit the Deployment instead (kubectl edit deploy X — templates have no immutability problem). replace --force without --force fails on immutable fields exactly like edit did. If your edited YAML has a nodeName, the recreated pod is pinned right back to that node — usually fine, occasionally the bug. And kubectl edit's editor is $KUBE_EDITOR — you set it to vim in 0.9; on the exam, set it again first thing.

    Docs

    Docs search: "kubectl edit" / "kubectl replace" reference pages; the mutable-fields list also appears under "Pod update" in the Pods concept page.

    Mine the killer.sh solutions — a half-day that's worth more than the session

    Why it matters

    Week 7 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. 401 vs 403 — what does each prove about the request, and which one names an identity?
      Answer

      401 Unauthorized = authentication failed: the apiserver couldn't establish who you are (bad/expired credential). 403 Forbidden = authentication succeeded, authorization refused — and the message names the exact identity it resolved. 401 → fix the credential; 403 → fix RBAC (7.1).

    2. Why is there no kubectl get users — and what do "create a user" tasks actually mean?
      Answer

      Kubernetes stores no User objects; a user is just the name a credential carries (cert CN, token subject). "Create a user" means mint a credential: the CSR flow for humans (7.3) or a ServiceAccount for workloads (7.4/7.9). Corollary: no revocation — locking someone out means removing bindings (7.1).

    3. Recite the five CSR onboarding steps and the three spec fields that must be exactly right.
      Answer

      openssl key+CSR (CN=user/O=group) → CSR object → kubectl certificate approve → extract .status.certificate | base64 -d → config set-credentials + set-context. Fields: request (base64 of the CSR, one line), signerName: kubernetes.io/kube-apiserver-client, usages: [client auth] (7.3).

    4. After onboarding, the new user gets Forbidden on everything. Success or failure?
      Answer

      Success — 403 proves the cert authenticates (the server names the user); it just has no RBAC yet. Failure would be a 401 or an x509 error (7.3).

    5. Where does a pod's ServiceAccount token come from, and what happens at 80% of its TTL?
      Answer

      The kubelet calls the TokenRequest API and projects the result to /var/run/secrets/kubernetes.io/serviceaccount/token (with ca.crt + namespace). Past 80% of TTL (or 24h) the kubelet fetches a fresh one and atomically swaps the file — clients must re-read, not cache (7.4).

    6. What exactly does the Node authorizer do that RBAC can't, and what's its signature error phrase?
      Answer

      It grants each kubelet (system:node:X in system:nodes) access only to objects related to pods bound to that node — a live relationship graph no static RBAC rule can express. Signature: "no relationship found between node 'X' and this object" (7.5).

    7. With --authorization-mode=Node,RBAC: the three verdicts an authorizer can return, and the chain rule?
      Answer

      Allow, deny, or no-opinion; authorizers run in flag order and the first allow-or-deny wins; all-no-opinion → deny. RBAC can only allow or abstain — no deny verdict exists, so revoking access always means deleting/narrowing a binding (7.5/7.6).

    8. ClusterRole + RoleBinding — what does it grant, and why is the combination useful?
      Answer

      The ClusterRole's rules, but only inside the binding's namespace. One shared rule-set (like built-in view/edit), granted per-namespace — no rule duplication. The reverse (ClusterRoleBinding→Role) is impossible (7.6).

    9. Which field of a RoleBinding is immutable, and what's the trick about Roles granting nodes?
      Answer

      roleRef — repointing a binding means delete + recreate. And a namespaced Role listing cluster-scoped resources (nodes, PVs) creates fine but grants nothing — cluster-scoped access needs ClusterRole + ClusterRoleBinding (7.6).

    10. Exact command: check whether ServiceAccount app-sa in dev may list pods there — without its token.
      Answer

      kubectl auth can-i list pods --as=system:serviceaccount:dev:app-sa -n dev. Impersonation runs the real authorizer chain server-side. Caveat: --as=jane alone drops her cert groups — add --as-group=devs to test group grants (7.8).

    11. Name the mutable fields of a running Pod, and the loop for everything else.
      Answer

      Mutable: container/initContainer image, activeDeadlineSeconds, toleration additions, (limited) terminationGracePeriodSeconds. Everything else: get -o yaml > f.yaml → edit → replace --force -f f.yaml — unless a controller owns the pod, in which case edit the controller's template instead (7.12).

    12. jsonpath: all node InternalIPs, one per line — write it.
      Answer

      kubectl get nodes -o jsonpath='{range .items[*]}{.status.addresses[?(@.type=="InternalIP")].address}{"\n"}{end}' — range/end for newlines, [?(@.type=="…")] to filter the list, whole expression single-quoted (7.11).
