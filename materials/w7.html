    Week 7 — RBAC, ServiceAccounts & speed drills

    Every request that reaches your cluster has to clear two separate checkpoints, in order: first "who are you?", then "are you allowed to do this?" Answering the first question wrong gets you turned away at the door; answering the second wrong gets you let in but told no. This week builds both checkpoints from the ground up. First, close out the 25% Cluster Architecture domain with the piece weeks 1–2 left open: how the apiserver decides who you are (authentication — certificates, ServiceAccount tokens, the CSR onboarding flow) and what you may do (authorization — RBAC's four objects, which work like a keycard system: some objects define which doors a card can open, others decide who's holding which card). Second, convert everything you know into speed: imperative generators, jsonpath surgery, fast-edit patterns, timed sprints, and your first killer.sh session. Knowledge stopped being the bottleneck around week 4; from here the enemy is the clock. Ticking done here also ticks the tracker.

    request + credential→authentication — who?→authorization — may you?→admission→etcd

    Authentication & authorization internals

    7.1

    Consolidate authentication: three credentials, one rule — there is no User object

    Why it matters

    Before Kubernetes decides whether to let a request do anything, it first has to answer a much more basic question: who is even asking? That's authentication — proving identity — and it's a separate, earlier step from authorization — deciding what that identity may do (week 7's other half, starting at 7.5). You've already met every credential type this cluster uses — client certs in week 2, the sa.key pair in the PKI map, bootstrap tokens during kubeadm join. This lesson snaps them into one picture, and gives you the 401-vs-403 discriminator that instantly classifies half of all "access denied" exam symptoms.

    The concept

    Picture the apiserver's front door (the same front-desk role it plays everywhere else in this course) running down a checklist of ways to recognize a visitor — that checklist is the authenticator chain. Each configured authenticator looks at the incoming request and either recognizes the credential and identifies who sent it, or shrugs and hands the request to the next authenticator in line. The first one to succeed wins, and win or lose, all it ever produces is a username + group list — nothing more. No profile, no permissions: what that identity may actually do is a completely separate question, answered later by RBAC. Kubernetes stores no User objects anywhere — unlike a Pod or a Deployment, there's no row in etcd for "jane". A "user" exists only as whatever name a credential happens to carry. That's why you can't kubectl get users, and why "create a user" tasks always mean "mint a credential" (a cert via the CSR API, or a ServiceAccount).

A 403 always names the identity it resolved — read it; it tells you which credential you're actually using.

      curl -sk -H 'Authorization: Bearer garbage' https://127.0.0.1:6443/api/v1/pods | grep -E 'message|code'

      Bad credential → 401, no identity at all. That's the discriminator: 401 = fix the credential, 403 = fix RBAC.

      {"aud":["https://kubernetes.default.svc.cluster.local"],

      Audience, issue/expiry times, and the subject line that is the SA's username. The signature (third chunk) is what sa.pub verifies — 7.4 goes deeper.

    Verify

    You can state, for each row of the table, where its signing/verification key lives on cp — and you can classify 401 vs 403 without looking anything up.

    Gotchas

    Because a "user" is nothing but a name written into a certificate, there is no revocation: unlike a website checking a certificate revocation list (a CA's "these certs are banned" register), Kubernetes never checks one — a valid, unexpired cert authenticates its holder no matter what. You cannot "delete" a cert user because there's no user object to delete. Locking someone out means removing their RBAC bindings instead (or, in the extreme, re-issuing the whole cluster CA — a disruptive, cluster-wide event). Short expirationSeconds on issued certs (7.3) is how you limit the blast radius up front. Also worth knowing: system:masters in a cert's O field bypasses authorization entirely — no RBAC binding needed, and none can restrict it, because the authorizer short-circuits for that group. That's why 2.3's super-admin.conf stays offline.

    7.2

    Dissect a kubeconfig — clusters × users, joined by contexts

    Why it matters

    A kubeconfig is just a text file that answers three questions for kubectl every time it runs: which cluster to talk to, which credential to talk as, and which namespace to assume by default. Every exam task begins with kubectl config use-context … — the one line you must run ~16 times without thinking. And "fix this broken kubeconfig" is a recurring task type: solvable in a minute if you know exactly which field lives where.

    The concept

    Strip away the YAML and a kubeconfig is three parallel lists plus one pointer. clusters say where (a server URL, plus the CA cert needed to trust that server — the same "notarized ID card" trust relationship from week 2, just now checked in the other direction). users say as whom (a client cert+key pair, or a token — one of the credential rows from 7.1's table). A context is nothing more than a named pairing of one cluster + one user, optionally with a default namespace tacked on. current-context is the pointer: whichever context name it holds is the (cluster, user, namespace) triple every bare kubectl command actually uses.

    A cluster entry (where) and a user entry (as whom) sit in two separate lists; a context is just a labeled pairing of one from each, and current-context is the pointer that says which pairing is active right now.

    Everything kubectl does starts by resolving current-context → (cluster, user, namespace) — that's the one lookup behind every command you type, whether or not you ever think about it. Which is exactly why the classic broken-kubeconfig faults are one-field edits, each breaking exactly one leg of that triple: a wrong server port (kubectl hangs or says connection refused — the where is wrong), a wrong certificate-authority-data ("x509: certificate signed by unknown authority" — you met this exact message in 2.3, and here it means the where resolved fine but the trust check on it failed), a user entry pointing at a missing key file (the as whom can't be assembled), or a context naming a user or cluster that doesn't exist in the respective list (the pairing itself is broken). Once you can name which leg a symptom points at, the fix is usually a one-line edit.

    Given any kubeconfig, you can answer in seconds: which server will this talk to, as whom, in which default namespace — and you can switch all three without reading help text.

    Gotchas

    Exam trap: a task's setup line sets a context whose namespace isn't default — every object you create lands there silently unless you notice, because "no -n flag" doesn't mean "default namespace", it means "whatever this context's namespace field says". Make kubectl config get-contexts (the NAMESPACE column) part of your per-task ritual so you catch it before creating anything. Also, the KUBECONFIG environment variable can hold a colon-separated list of files that kubectl merges into one logical config — if a cluster you can't explain appears in config view, that variable is why; check it before assuming the file itself is wrong. And --minify fails outright if current-context is unset — since that flag works by filtering everything down to the current context's slice, an unset pointer gives it nothing to filter to, and that error is itself a signature of a planted fault.

    7.3

    Onboard a human user via the CSR API — the full flow, from memory

    Why it matters

    "Onboard a new user named jane" sounds like an account-creation task, but 7.1 already told you there's no user database to add her to — so what does onboarding actually mean? It means minting her a certificate and getting the cluster to sign it, through a self-service API instead of you hand-signing it with the CA key yourself. This is a classic exam task, worth real points, and it's pure mechanics: five steps, zero improvisation. It's also 7.1 made concrete — you mint a certificate whose CN/O become a user the moment the cluster CA signs it. Drill it until the whole transcript takes under five minutes.

    The concept
    openssl key + CSR→CSR object (base64)→approve→extract signed cert→kubeconfig entry

    The five stages above are: generate a key and a CSR with plain openssl → wrap that CSR in a Kubernetes API object → have someone with permission approve it → pull the now-signed certificate back out → wire it into a kubeconfig so you can actually use it. The CertificateSigningRequest API is what turns the cluster CA — normally a private key sitting on disk that only kubeadm init ever touched directly — into something you can drive over the API instead: you submit a PEM-encoded CSR wrapped in a CSR object, an admin approves it, and the controller-manager (the one control-plane component that actually holds ca.key) signs it and publishes the finished certificate back into the object's status field. Three spec fields do all the work: request is your CSR, base64-encoded onto a single line; signerName: kubernetes.io/kube-apiserver-client names which signer handles the request — this particular one is reserved for "certs that authenticate humans to the apiserver", and by design it is never auto-approved, so a human decision is always required; usages: [client auth] declares what the resulting cert may be used for. An optional expirationSeconds field requests a lifetime (minimum 600 seconds); the signer actually grants whichever is smaller, your request or its own configured --cluster-signing-duration cap — you can ask for less than the cap, never more.

    Lab
    1. Mint the user's key and CSR — CN is the username, O the group (7.1's rule): 
      openssl genrsa -out jane.key 2048
openssl req -new -key jane.key -out jane.csr -subj "/CN=jane/O=devs"
    2. Wrap it in a CSR object. The base64 must be a single line — -w0: 
      cat <<EOF | kubectl apply -f -
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: jane
spec:
  request: $(base64 -w0 < jane.csr)
  signerName: kubernetes.io/kube-apiserver-client
  expirationSeconds: 86400
  usages:
  - client auth
EOF
kubectl get csr
      certificatesigningrequest.certificates.k8s.io/jane created
NAME   AGE   SIGNERNAME                            REQUESTOR          CONDITION
jane   4s    kubernetes.io/kube-apiserver-client   kubernetes-admin   Pending
    3. Approve — this signer always waits for a human decision: 
      kubectl certificate approve jane
kubectl get csr jane
      certificatesigningrequest.certificates.k8s.io/jane approved
NAME   AGE   SIGNERNAME                            REQUESTOR          CONDITION
jane   30s   kubernetes.io/kube-apiserver-client   kubernetes-admin   Approved,Issued
    4. The signed cert now sits in .status.certificate — extract and inspect it: 
      kubectl get csr jane -o jsonpath='{.status.certificate}' | base64 -d > jane.crt
openssl x509 -in jane.crt -noout -subject -dates
      subject=O = devs, CN = jane
notBefore=Jul 13 09:00:00 2026 GMT
notAfter=Jul 14 09:00:00 2026 GMT

      Exactly one day — the expirationSeconds you asked for, granted because it's below the signer's cap.

    5. Build her kubeconfig entry (in your existing file — same cluster, new user + context): 
      kubectl config set-credentials jane --client-certificate=jane.crt --client-key=jane.key --embed-certs
kubectl config set-context jane --cluster=kubernetes --user=jane
      User "jane" set.
Context "jane" created.
    6. Prove both halves of the story — she authenticates, and (having no RBAC yet) can do nothing: 
      kubectl --context=jane auth whoami
kubectl --context=jane get pods
      Username   jane
Groups     [devs system:authenticated]
Error from server (Forbidden): pods is forbidden: User "jane" cannot list resource "pods" ...

      Forbidden is success here — 403, not 401: the cert works, the user exists, RBAC just hasn't granted anything. The RBAC group fixes that with this exact jane.

    Verify

    kubectl --context=jane auth whoami returns jane + devs, and the whole flow — six commands — ran without you opening the docs.

    Gotchas

    The three spec traps, in order of popularity: request must be the base64 of the CSR — not the private key, not a cert, and with no line wraps in the base64 itself (that's what -w0 in the lab prevents); usages must be exactly client auth, spelled that way; signerName must be the exact string kubernetes.io/kube-apiserver-client (a similarly-named signer, kubelet-serving, exists for a completely different purpose — issuing certs kubelets present as servers — and will reject a human's CN pattern). A CSR that sits Pending forever is normal, not broken — nothing auto-approves this particular signer, by design, so it waits until you run kubectl certificate approve yourself. And if the issued cert seems to ignore the expirationSeconds you asked for, remember the signer's own --cluster-signing-duration cap wins whenever it's smaller — check the controller-manager manifest for that flag before assuming your request field was wrong.

    Docs search: "Certificate Signing Requests" — the "Normal user" section of that page is this lesson's transcript in official form (copy the object skeleton from there during the exam).

    7.4

    Modern ServiceAccount tokens: TokenRequest, projected volumes, rotation

    Why it matters

    A ServiceAccount is how a program (a pod, a controller, a CI job) proves its identity, the same way a certificate lets a human prove theirs — but the credential itself works completely differently, and getting that difference wrong is the single most common source of stale advice you'll find online. Old tutorials (and old muscle memory) say "every ServiceAccount has a token Secret, go read it". That world ended in v1.24. Exam tasks and real debugging both require the current model: tokens are minted on demand, time-bound, and rotated out from under your pods without the pod ever noticing.

    The concept

    A ServiceAccount object itself contains no credential at all — it's just a name that RBAC can bind rules to, nothing more, the identity equivalent of an empty folder with a label on it. Tokens instead come from the TokenRequest API, a subresource of the SA that you call to ask for a fresh one: the apiserver signs a JWT with sa.key right there on the spot, scoped to a requested audience (who's allowed to accept this token) and lifetime, and — critically — never stores a copy anywhere. Ask for another token five seconds later and you get a different one. There are two consumers of this API:

    • kubectl create token <sa> — the CLI's front-end onto TokenRequest, useful for testing or handing a human a token by hand. Default lifetime 1 hour; pass --duration to change it.
    • Pods — for these, nothing calls TokenRequest by hand; the admission chain (a set of checks the apiserver runs on every object before it's allowed to exist) automatically injects a projected volume at /var/run/secrets/kubernetes.io/serviceaccount, combining three sources into one directory: a serviceAccountToken (the kubelet itself calls TokenRequest to get this, on the pod's behalf), the cluster's CA bundle (delivered as a ConfigMap, so the pod can verify the apiserver's cert), and the pod's own namespace (pulled from the downward API — the mechanism that lets a pod read facts about itself). The kubelet rotates that token once it passes 80% of its TTL (or after 24h, whichever comes first): it quietly rewrites the file in place, which is why well-behaved clients re-read the token from disk on every call rather than caching it in memory once at startup.

    Tokens bound to a pod carry that specific pod's identity in their claims and die the moment the pod does — so a token stolen off a deleted pod is worthless, it can't be replayed later. Legacy non-expiring Secret-based tokens aren't auto-created anymore for exactly this reason (a token that never expires and never dies with anything is a much bigger liability); the only reason to make one today is an explicit task demanding a long-lived token, and there's a sanctioned Secret type for that narrow case (see Gotchas).

    Lab
    1. Create an SA and look for its token — there isn't one: 
      kubectl create serviceaccount probe
kubectl describe sa probe | grep -A1 Tokens
kubectl get secrets
      serviceaccount/probe created
Tokens:              <none>
No resources found in default namespace.
    2. Mint one on demand and read its claims (the 7.1 trick): 
      TOKEN=$(kubectl create token probe --duration=10m)
echo "$TOKEN" | cut -d. -f2 | tr '_-' '/+' | base64 -d 2>/dev/null; echo
      {"aud":["https://kubernetes.default.svc.cluster.local"],
 "exp":1783739533,"iat":1783738933,          ← exp − iat = 600s: exactly your --duration
 "sub":"system:serviceaccount:default:probe", ...}
    3. Now the pod path. Run a pod under this SA and inspect what admission injected: 
      kubectl run tokenpod --image=busybox:1.36 \
  --overrides='{"spec":{"serviceAccountName":"probe"}}' -- sleep 3600
kubectl get pod tokenpod -o yaml | grep -A12 'projected:'
          - projected:
        sources:
        - serviceAccountToken:

      You never wrote a volume — the ServiceAccount admission plugin did. 3607s ≈ 1h is the kubelet's requested TTL.

    4. See the rotation machinery from inside — the files are symlinks through a swappable ..data dir, so the kubelet can replace all three atomically: 
      kubectl exec tokenpod -- ls -l /var/run/secrets/kubernetes.io/serviceaccount/

    5. Clean up: 
      kubectl delete pod tokenpod
    Verify

    You can explain where a pod's token comes from (kubelet → TokenRequest → projected file), why kubectl get secrets shows nothing for a new SA, and what happens at 80% of the token's TTL.

    Gotchas

    If a task really does demand a long-lived token (CI systems, external clients that can't easily re-fetch), the sanctioned form is a Secret of type kubernetes.io/service-account-token carrying the annotation kubernetes.io/service-account.name: <sa> — create that Secret and a controller notices the annotation and fills in a non-expiring token for you. Don't reach for it otherwise; it re-introduces the exact liability the whole TokenRequest redesign was built to remove. In-pod clients that read the token once at startup and cache it in memory break mysteriously after rotation, because the file on disk changed underneath them but their copy didn't — "worked fine for a day, then started getting 401s" is that signature, and the fix is to re-read the file per-request rather than caching. And automountServiceAccountToken: false (settable on the SA or the pod) removes the projected volume entirely on purpose: a pod that can't find anything under /var/run/secrets/… in that case isn't misconfigured, it's deliberately hardened against needing apiserver access at all.

    Docs

    Docs search: "Managing Service Accounts" (token mechanics, bound tokens) and "Configure Service Accounts for Pods" (the long-lived Secret recipe).

    7.5

    The authorization lineup: Node,RBAC — order matters, first decision wins

    Why it matters

    7.1 covered the "who are you" checkpoint; this lesson opens the second one, "what may you do" — authorization — and the twist is that RBAC isn't the only bouncer at that gate, just the last one most requests actually meet. Knowing what --authorization-mode=Node,RBAC actually means explains one weird Forbidden message you'll meet in kubelet debugging, and answers the direct exam question "which authorization modes does this cluster run?"

    The concept

    Authorization works the same shape as authentication did in 7.1 — a chain consulted in order, this time of authorizers rather than authenticators — but each link in this chain can say one of three things instead of two: allow (request proceeds, chain stops immediately), deny (request fails, chain stops immediately), or no opinion (this authorizer doesn't handle requests like this one, so move on to the next). If the chain runs out with every authorizer having shrugged — deny, by default. RBAC, notably, can only ever allow or abstain — it has no deny verdict at all in its vocabulary. That single fact is why RBAC rules are purely additive (you can only grant, never subtract with a rule) and why "revoke access" always means "delete or narrow a binding" rather than "add a deny rule" — there's no such thing to add.

    kubeadm's lineup is Node,RBAC — two links, checked in exactly that order. The Node authorizer runs first, and it has exactly one clientele: identities named system:node:<name> in the group system:nodes — in other words, kubelets, and only kubelets. For that narrow audience it grants read access only to the objects related to pods actually scheduled on that specific node (their ConfigMaps, Secrets, PVCs…), plus write access to its own Node status and its own pods' statuses. This is a job a generic RBAC rule structurally cannot do: RBAC rules are static (this role can read secrets, full stop), but "only the secrets your pods happen to mount right now" is a live relationship that changes every time a pod is scheduled or deleted — so the Node authorizer computes that relationship graph on the fly, instead of consulting a rule. Everyone who isn't a kubelet — every human, every ServiceAccount — gets "no opinion" from Node and falls straight through to RBAC. (Other authorizer modes exist — Webhook, ABAC, AlwaysAllow/AlwaysDeny — and newer clusters can define the same chain in a structured --authorization-config file instead of a comma-separated flag; kubeadm still uses the flag, and the two configuration styles are mutually exclusive.)

    Lab
    1. Read the lineup off your apiserver: 
      sudo grep authorization-mode /etc/kubernetes/manifests/kube-apiserver.yaml
          - --authorization-mode=Node,RBAC
    2. Borrow the kubelet's identity (its kubeconfig from 2.3) and see who the Node authorizer thinks you are: 
      sudo kubectl --kubeconfig /etc/kubernetes/kubelet.conf auth whoami
      Username   system:node:cp
Groups     [system:nodes system:authenticated]
    3. Watch the relationship rule bite. Create a secret no pod on cp mounts, then ask for it as the kubelet: 
      kubectl create secret generic topsecret --from-literal=k=v
sudo kubectl --kubeconfig /etc/kubernetes/kubelet.conf get secret topsecret
      Error from server (Forbidden): secrets "topsecret" is forbidden:
User "system:node:cp" cannot get resource "secrets" ...: no relationship found between node 'cp' and this object

      That phrase is the Node authorizer's signature — you'll never see RBAC say it.

    4. But node-related reads sail through the same credential: 
      sudo kubectl --kubeconfig /etc/kubernetes/kubelet.conf get node cp
      NAME   STATUS   ROLES           AGE   VERSION
cp     Ready    control-plane   21d   v1.35.1
    5. Keep topsecret — 7.11's drills decode it.
    Verify

    You can recite the three verdicts and the ordering rule, say which identities the Node authorizer serves, and explain why RBAC has no deny.

    Gotchas

    Order is meaningful but rarely the direct cause of a fault: with Node,RBAC, every non-node request simply passes through Node as "no opinion" at essentially zero cost, so in practice you can usually reason about RBAC alone for human/SA identities. The trap is conceptual questions rather than broken clusters: "add a deny rule for user X" is a trick question — RBAC has no such rule to add; the only lever is removing or narrowing the bindings that currently grant access. And don't confuse the Node authorizer covered here with the NodeRestriction admission plugin (also on by default in kubeadm, but a completely different mechanism running at a different stage): the authorizer limits what kubelets may read, while NodeRestriction limits what they may write — specifically, only their own Node object and their own pods' status, so a kubelet can't forge status for a node it isn't running.

    Docs

    Docs search: "Authorization Overview" (modes + decision flow) and "Using Node Authorization".

    RBAC in practice

    7.6

    The four-object matrix: Role/ClusterRole × RoleBinding/ClusterRoleBinding

    Why it matters

    RBAC is best pictured as a building's keycard system: somewhere there's a list of what a given kind of keycard opens (a rule), and separately, a list of who's actually holding which keycard (a grant). Kubernetes RBAC splits into exactly those two halves, each with a namespaced and a cluster-wide version, so four object kinds total. Every RBAC exam task is assembled from those four kinds and one question: does a binding connect this subject to a rule allowing this verb on this resource in this scope? Get the matrix straight once and every task becomes fill-in-the-blanks.

    The concept

    Two kinds hold rules — what a keycard opens: which verbs (get, list, create…) are allowed on which resources. Two kinds hold grants — who's holding a keycard, and where it works:

    RoleBinding (grants in ONE namespace)ClusterRoleBinding (grants EVERYWHERE)
    Role (rules live in a ns)classic per-ns access — rules and grant both scoped to that ns✗ impossible — a ClusterRoleBinding can't reference a Role
    ClusterRole (rules are global)the reuse pattern: one shared rule-set, granted per-namespace — the binding's ns is the boundaryfull cluster-wide grant; also the only way to grant cluster-scoped resources (nodes, PVs, namespaces) and non-resource URLs

    A rule itself is three or four fields: apiGroups + resources + verbs, optionally narrowed to specific object names with resourceNames. The RBAC authorizer is purely additive, exactly as 7.5 said: given a request, it looks for any binding→rule path that allows it, anywhere; finding none means "no opinion" — which, at the end of the authorization chain, becomes deny. There are two hard scope laws worth memorizing because they're both popular trick questions: a namespaced Role can never grant access to cluster-scoped resources like nodes or persistentvolumes — writing one succeeds (kubectl won't stop you) but the resulting rule grants nothing at all, because a namespace-scoped object simply has no jurisdiction over a cluster-scoped resource; and a RoleBinding pointing at a ClusterRole grants those rules only inside the binding's own namespace, never cluster-wide, no matter how broad the ClusterRole's rules look on paper.

    Why it matters

    7.6 built RBAC objects with hand-written YAML so the object model would be crystal clear — but nobody should actually type that YAML on the exam. RBAC YAML is verbose and easy to typo under pressure; the imperative kubectl create forms are one line each and generate a correct object every time, with none of the indentation or field-name risk. This is the difference between a 3-minute task and an 8-minute one.

    The concept

    Four commands, one per corner of 7.6's matrix, cover every RBAC task you'll meet. The grammar to internalize:

    CommandKey flags
    kubectl create role--verb=get,list,watch --resource=pods [--resource-name=web] [-n ns]

    7.8

    Verify like the exam does: kubectl auth can-i (and impersonation)

    Why it matters

    After building a Role and a RoleBinding, how do you actually know they work — read the YAML back and reason it through, or just ask? "Just ask" is the better habit, and it's the RBAC-specific version of the always-verify-your-answer habit this course keeps returning to: kubectl auth can-i asks the apiserver's real authorization chain the exact question you care about, so the answer is proof, not your own guess about how the rules should combine. It's also how you debug someone else's permissions without ever needing to hold their credentials or switch contexts.

    The concept

    kubectl auth can-i VERB RESOURCE works by creating a SelfSubjectAccessReview — an API object that packages up "could I do this?" as a question — and the server runs its full authorizer chain (Node, RBAC, everything from 7.5) against that exact hypothetical request, then answers plainly yes or no. Add --as=user or --as-group=group and kubectl sends impersonation headers instead of asking about yourself: the apiserver evaluates the same question as if it came from that other identity. Impersonation is itself gated by RBAC (it needs the verb impersonate on the target identity) — your admin context has that permission via cluster-admin; jane, with no such grant, doesn't. ServiceAccounts get impersonated by their full username string: --as=system:serviceaccount:<ns>:<name>, the same subject string 7.1 showed you inside a decoded token. And --list flips the whole question around, from "may I do this one specific thing?" to "what am I allowed to do here at all?" — dumping the entire permission surface as a table in one shot.

    Gotchas

    The big one: impersonating a user does not automatically impersonate their groups too — the apiserver has no way to know which groups a real credential would have carried, since impersonation headers replace the credential rather than parsing one, so --as=jane alone tests bare-jane with no groups at all, and you must add --as-group=devs yourself to test what her group membership actually grants. (Testing with --context=jane instead uses her real certificate, groups included automatically — that's why 7.6 checked results both ways.) can-i --list includes rows that every authenticated user gets for free (like creating SelfSubjectReviews) — don't mistake those baseline rows for evidence that your specific grant worked. And in scripts, a "no" answer exits with status 1 — meaning set -e will kill your verification script the moment it hits a correct negative result, not just a real error, so guard those calls accordingly.

    7.9

    Wire a pod to a ServiceAccount end-to-end — and prove it with curl

    Why it matters

    Every piece of the workload-identity story has been built separately so far — where a pod's token comes from (7.4), how to grant a ServiceAccount permissions (7.6–7.8) — and this lesson welds them into one working chain, end to end: SA → Role → binding → pod → an actual authenticated call to the API from inside a container. It's a complete exam task type by itself, not just a review.

    The concept

    A pod runs as exactly one ServiceAccount, set by spec.serviceAccountName — if you never set it, the namespace's default SA is used, since every namespace gets one automatically. Any in-pod client authenticates using the three projected files 7.4 showed you: send the contents of token as an HTTP Authorization: Bearer header, and trust the server's certificate using the bundled ca.crt rather than a public CA (the same "no public trust store, only this specific CA" fact you'll hit again in the Gotchas).     7.11

    Output surgery: jsonpath, custom-columns, sort-by — a drill set

    Why it matters

    Every Kubernetes object kubectl returns is really just a big nested tree of JSON underneath the YAML you're used to seeing — and jsonpath is a small query language for reaching into that tree and pulling out exactly one value, instead of you scanning a full printout with your eyes. Tasks phrased as "write the X of all Y to /path/file.txt" are pure output surgery — full points for one correct line, zero for a near-miss, with no partial credit for "close". And in troubleshooting, extracting exactly one field this way beats scrolling through a whole YAML dump every time.

    The concept

    Four tools, in the order you should actually reach for them:

    • -o jsonpath='{...}' — a path straight into any object's JSON tree: {.items[*].metadata.name} walks down to every item's name field. Wrap it in {range .items[*]}…{end} when you need to iterate and control what separates each result — {"\n"} inside the range gives you one result per line, which is what a file usually wants instead of one space-separated blob. Narrow which list entries you even look at with a filter like [?(@.type=="InternalIP")] — "only the entry where this field equals that value".
    • -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName — the same jsonpath idea, but one small path per column, printed as a table. Easier to read (and to write) than range/end once you want several fields side by side rather than one value at a time.
    • --sort-by='{.status.containerStatuses[0].restartCount}' — the server hands back list results in essentially arbitrary order; this flag takes a jsonpath into each item and sorts by whatever it finds there.
    • The honest fallback: -o yaml | grep -A5 <key> — when the exact path won't come to you inside 30 seconds, stop composing jsonpath and just grep the YAML dump for context around the field you want. It's on this list because it wins points under time pressure, not because it's elegant.

    When you don't already know the path by heart: run -o yaml on one single object and read the real field names straight off the dump, or run kubectl explain pod.status --recursive | less to walk the schema without needing a live object at all.

    Lab

    Eight drills. For each: write the command yourself first, run it, then check against the reveal. Wrong or slow → repeat tomorrow.

    1. All pod names in kube-system, one per line.
      Solution
      kubectl -n kube-system get pods -o jsonpath='{range .items[*]}{.metadata.name}{"\n"}{end}'
    2. The InternalIP of every node (space-separated is fine).
      Solution
      kubectl get nodes -o jsonpath='{.items[*].status.addresses[?(@.type=="InternalIP")].address}'

      The filter selects which list entry; note double quotes inside single-quoted jsonpath.

    3. Every container image running in the cluster, deduplicated.
      Solution
      kubectl get pods -A -o jsonpath='{range .items[*]}{range .spec.containers[*]}{.image}{"\n"}{end}{end}' | sort -u

      Nested ranges: pods, then containers within each.

    4. A table of pod name + node it runs on, all namespaces.
      Solution
      kubectl get pods -A -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName
    5. All pods sorted by restart count — most-restarted last.
      Solution
      kubectl get pods -A --sort-by='{.status.containerStatuses[0].restartCount}'
    6. Decode the value of key k in secret topsecret (from 7.5).
      Solution
      kubectl get secret topsecret -o jsonpath='{.data.k}' | base64 -d; echo
    7. The current context's apiserver URL, from the kubeconfig.
      Solution
      kubectl config view --minify -o jsonpath='{.clusters[0].cluster.server}'
    8. Each node's name and its taints (key only), one node per line.
      Solution
      kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{": "}{.spec.taints[*].key}{"\n"}{end}'

      Untainted nodes print an empty value after the colon — that's fine.

    Verify

    Drills 1, 2 and 6 come out right on the first attempt (they're the three most exam-frequent shapes); the rest within one retry.

    Gotchas

    Always single-quote the whole jsonpath expression — the $, * and " characters inside it mean something to your shell too, and it will happily mangle them before kubectl ever sees the string, unless quoting protects it. {.items[*]} with a plain * joins every matched value with a single space; only wrapping the expression in range/end gives you control over the separator, including real newlines. Filters compare with ==, not a single =, and need the @ symbol to mean "the current list element being tested" — both are easy slips under pressure. And when a task says "to a file", redirect the whole command's output and then actually cat the file back to confirm — a jsonpath result with no trailing newline can leave a file that looks empty in a quick glance at a file listing, when it actually has exactly the one line asked for.

    7.12

    Fast pod edits: edit where you can, replace --force where you must

    Why it matters

    A running Pod isn't as editable as it looks — most of its spec is frozen the instant it's created, on purpose, because letting fields like volumes or the container command change underneath a running container would make "what is this pod actually running" an unanswerable question. "Change this field on this pod" tasks split into two very different speeds depending on which field: mutable fields are a 20-second kubectl edit; immutable ones need a slower dump-edit-replace loop instead. Knowing which category a field falls into before you start typing is the actual speed trick here — finding out by way of a rejected edit costs you a full minute each time it happens.

    The concept

    On a live Pod, only a short, fixed list of fields is mutable: spec.containers[*].image (and the same field on initContainers), spec.activeDeadlineSeconds, spec.tolerations (only additions, never removals), and spec.terminationGracePeriodSeconds (in a limited way). Everything else on the pod — environment variables, args, volumes, probes, serviceAccountName, nodeSelector, the labels in its spec — is immutable: the apiserver simply rejects the update outright rather than applying a partial change. The escape hatch, when you genuinely need to change one of those frozen fields, is to recreate the pod in place from an edited copy of its own definition:

    kubectl get pod X -o yaml > f.yaml→edit f.yaml→kubectl replace --force -f f.yaml

    replace --force deletes the pod and creates it again from your edited file in one command — the result gets a new UID (it's genuinely a new object, not the old one patched), there's a brief moment of downtime while the old one is gone and the new one starts, but the name stays the same. One critical fork in the road before you reach for this: it's for bare, standalone pods only. A pod owned by a Deployment or DaemonSet should never be surgically edited or replaced this way — the controller notices its pod is "wrong" relative to the template and will fight you, or worse, silently recreate your fix away. For those, change the controller's template instead (which has none of a bare pod's immutability problems) and let the rollout mechanism replace pods properly on its own schedule.

    Lab
    1. Create a pod with a wrong env value, then watch edit refuse to fix it: 
      kubectl run editme --image=nginx:1.27 --env=MODE=staging
kubectl edit pod editme        # change MODE's value to "prod", save, quit
      error: pods "editme" is invalid
# * spec: Forbidden: pod updates may not change fields other than
#   `spec.containers[*].image`, `spec.initContainers[*].image`,
#   `spec.activeDeadlineSeconds`, `spec.tolerations` (only additions...) ...
# A copy of your changes has been stored to "/tmp/kubectl-edit-1408.yaml"

      Two gifts in that failure: the canonical mutable-field list, and your rejected buffer saved to /tmp.

    2. The loop that works — under 60 seconds total: 
      kubectl get pod editme -o yaml > /tmp/editme.yaml
vim /tmp/editme.yaml           # fix MODE, ignore the status:/managedFields noise
kubectl replace --force -f /tmp/editme.yaml
      pod "editme" deleted
pod/editme replaced
    3. Confirm the change landed (and notice the pod is new — AGE reset): 
      kubectl get pod editme -o jsonpath='{.spec.containers[0].env}'; echo
kubectl get pod editme
      [{"name":"MODE","value":"prod"}]
NAME     READY   STATUS    RESTARTS   AGE
editme   1/1     Running   0          9s
    4. Contrast: the one field you never needed the loop for — 
      kubectl set image pod/editme editme=nginx:1.28
kubectl get pod editme -o jsonpath='{.spec.containers[0].image}'; echo
      pod/editme image updated
nginx:1.28
    5. Clean up: 
      kubectl delete pod editme $now
    Verify

    You can recite the mutable-fields list, and the dump-edit-replace loop runs from your fingers in under a minute.

    Gotchas

    Run this loop on a Deployment-owned pod and the controller notices its pod count dropped, then instantly builds a replacement straight from the old template — your carefully edited fix evaporates within seconds, because the controller was never told about it. Edit the Deployment itself instead (kubectl edit deploy X — templates have none of a bare pod's immutability restrictions). Running plain replace without the --force flag fails on immutable fields for exactly the same reason edit did — --force is specifically what turns "reject the update" into "delete and recreate". If your edited YAML still has a nodeName set (pods usually get one written in by the scheduler once placed), the recreated pod is pinned right back onto that same node — usually harmless, but occasionally the actual bug if that node is the problem. And kubectl edit's editor comes from $KUBE_EDITOR — you set it to vim back in 0.9, but that was a shell setting in one session; set it again first thing on exam day, since a fresh terminal won't remember it.

    Docs

    Docs search: "kubectl edit" / "kubectl replace" reference pages; the mutable-fields list also appears under "Pod update" in the Pods concept page.

    7.13

    Topic sprints: ten timed tasks across weeks 2–6

    Why it matters

    Knowing how to do something and being able to do it inside the exam's time budget are two different skills, and this drill is designed to expose the gap between them. Exam tasks average ~6 minutes of budget. This sprint tells you — per topic — whether you're inside that budget, and turns "I know how" into "I know how fast". The lookups that cost you time here are your memorization list for week 8.

    The drill

    Rules: one sitting, 60-minute cap for all ten (the targets sum to 42). Docs allowed, but only kubernetes.io — search there like it's exam day. After each task, note your time and what you had to look up. Reveal a solution only after finishing or timing out; each is terse on purpose — full explanations live in the week linked.

    1. [W2 · 4 min] Snapshot etcd to /tmp/snap.db and prove the snapshot contains data.
      Solution
      sudo ETCDCTL_API=3 etcdctl snapshot save /tmp/snap.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
sudo etcdutl snapshot status /tmp/snap.db -w table

      Flags from the manifest, not memory failing you → etcd card.

    2. [W3 · 5 min] Deployment web, nginx:1.27, 3 replicas, exposed as ClusterIP :80; prove it answers from a test pod.
      Solution
      k create deploy web --image=nginx:1.27 --replicas=3 --port=80
k expose deploy web --port=80
k run t --rm -it --image=busybox:1.36 --restart=Never -- wget -qO- web | head -4
k delete deploy web; k delete svc web
    3. [W3 · 3 min] Scale a deployment to 5, roll out a bad image, then get back to the working revision.
      Solution
      k create deploy roll --image=nginx:1.27
k scale deploy roll --replicas=5
k set image deploy/roll nginx=nginx:1.99
k rollout status deploy/roll --timeout=20s   # sticks — ImagePullBackOff
k rollout undo deploy/roll
k rollout status deploy/roll                 # back to healthy
k delete deploy roll
    4. [W3 · 4 min] A static pod named sentinel (busybox, sleep) on node01.
      Solution
      ssh node01
sudo sh -c 'kubectl run sentinel --image=busybox:1.36 \
  --dry-run=client -o yaml -- sleep 3600 > /etc/kubernetes/manifests/sentinel.yaml'
exit
kubectl get pod sentinel-node01     # suffix = node name proves it's static

      No kubectl on node01? Write the manifest by hand — or generate on cp and scp it.

    5. [W4 · 6 min] In dev: only pods labeled app=frontend may reach pods labeled app=backend on :80; all other ingress to backend blocked.
      Solution
      cat <<EOF | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: {name: backend-allow, namespace: dev}
spec:
  podSelector: {matchLabels: {app: backend}}
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector: {matchLabels: {app: frontend}}
    ports:
    - {protocol: TCP, port: 80}
EOF

      Verify with two test pods (one labeled frontend, one not) + wget timeouts — W4's method.

    6. [W4/W6 · 4 min] Service broken selects nothing (create it with a wrong selector first); diagnose and fix so it has endpoints.
      Solution
      k create deploy target --image=nginx:1.27 --port=80        # labels app=target
k create service clusterip broken --tcp=80:80              # selector app=broken — wrong
k get endpointslices | grep broken                         # no endpoints
k set selector svc broken app=target
k get endpointslices | grep broken                         # endpoints appear
k delete svc broken; k delete deploy target
    7. [W5 · 6 min] hostPath PV (1Gi, RWO) + PVC + a pod mounting it at /data; prove a write lands on the node.
      Solution

      PV (hostPath /tmp/pvdata, storageClassName manual) → PVC requesting 1Gi same class → pod with volumes/volumeMounts. Then k exec ... -- sh -c 'echo hi > /data/f' and read /tmp/pvdata/f on the pod's node. Full walk: W5's PV/PVC lessons.

    8. [W2/W6 · 3 min] Drain node02 for maintenance, verify nothing schedulable remains, bring it back.
      Solution
      k drain node02 --ignore-daemonsets --delete-emptydir-data
k get nodes                        # node02 Ready,SchedulingDisabled
k get pods -A -o wide | grep node02   # only DaemonSet pods remain
k uncordon node02
    9. [W7 · 5 min] SA runner in dev that may create jobs there, nothing more; prove both halves with can-i.
      Solution
      k -n dev create sa runner
k -n dev create role job-maker --verb=create --resource=jobs
k -n dev create rolebinding runner-jobs --role=job-maker --serviceaccount=dev:runner
k auth can-i create jobs --as=system:serviceaccount:dev:runner -n dev    # yes
k auth can-i list pods   --as=system:serviceaccount:dev:runner -n dev    # no
    10. [W7 · 2 min] Write every node's InternalIP to /tmp/ips.txt, one per line.
      Solution
      kubectl get nodes -o jsonpath='{range .items[*]}{.status.addresses[?(@.type=="InternalIP")].address}{"\n"}{end}' > /tmp/ips.txt
cat /tmp/ips.txt

    Total inside 60 minutes with at most two tasks over target. Your lookup log exists and has concrete entries ("forgot etcdctl cert flag names", "NetworkPolicy from: indentation") — those go on the memorize list.

    Mine the killer.sh solutions — a half-day that's worth more than the session

    Why it matters

    Taking the killer.sh session felt like the main event, but it wasn't — the detailed answer explanations that come with it are the actual product, and the exam session you just ran was only the diagnostic that tells you which of those explanations are worth studying hard.     Steps

    Week 7 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. 401 vs 403 — what does each prove about the request, and which one names an identity?
      Answer

      401 Unauthorized = authentication failed: the apiserver couldn't establish who you are (bad/expired credential). 403 Forbidden = authentication succeeded, authorization refused — and the message names the exact identity it resolved. 401 → fix the credential; 403 → fix RBAC (7.1).

    2. Why is there no kubectl get users — and what do "create a user" tasks actually mean?
      Answer

      Kubernetes stores no User objects; a user is just the name a credential carries (cert CN, token subject). "Create a user" means mint a credential: the CSR flow for humans (7.3) or a ServiceAccount for workloads (7.4/7.9). Corollary: no revocation — locking someone out means removing bindings (7.1).

    3. Recite the five CSR onboarding steps and the three spec fields that must be exactly right.
      Answer

      openssl key+CSR (CN=user/O=group) → CSR object → kubectl certificate approve → extract .status.certificate | base64 -d → config set-credentials + set-context. Fields: request (base64 of the CSR, one line), signerName: kubernetes.io/kube-apiserver-client, usages: [client auth] (7.3).

    4. After onboarding, the new user gets Forbidden on everything. Success or failure?
      Answer

      Success — 403 proves the cert authenticates (the server names the user); it just has no RBAC yet. Failure would be a 401 or an x509 error (7.3).

    5. Where does a pod's ServiceAccount token come from, and what happens at 80% of its TTL?
      Answer

      The kubelet calls the TokenRequest API and projects the result to /var/run/secrets/kubernetes.io/serviceaccount/token (with ca.crt + namespace). Past 80% of TTL (or 24h) the kubelet fetches a fresh one and atomically swaps the file — clients must re-read, not cache (7.4).

    6. What exactly does the Node authorizer do that RBAC can't, and what's its signature error phrase?
      Answer

      It grants each kubelet (system:node:X in system:nodes) access only to objects related to pods bound to that node — a live relationship graph no static RBAC rule can express. Signature: "no relationship found between node 'X' and this object" (7.5).

    7. With --authorization-mode=Node,RBAC: the three verdicts an authorizer can return, and the chain rule?
      Answer

      Allow, deny, or no-opinion; authorizers run in flag order and the first allow-or-deny wins; all-no-opinion → deny. RBAC can only allow or abstain — no deny verdict exists, so revoking access always means deleting/narrowing a binding (7.5/7.6).

    8. ClusterRole + RoleBinding — what does it grant, and why is the combination useful?
      Answer

      The ClusterRole's rules, but only inside the binding's namespace. One shared rule-set (like built-in view/edit), granted per-namespace — no rule duplication. The reverse (ClusterRoleBinding→Role) is impossible (7.6).

    9. Which field of a RoleBinding is immutable, and what's the trick about Roles granting nodes?
      Answer

      roleRef — repointing a binding means delete + recreate. And a namespaced Role listing cluster-scoped resources (nodes, PVs) creates fine but grants nothing — cluster-scoped access needs ClusterRole + ClusterRoleBinding (7.6).

    10. Exact command: check whether ServiceAccount app-sa in dev may list pods there — without its token.
      Answer

      kubectl auth can-i list pods --as=system:serviceaccount:dev:app-sa -n dev. Impersonation runs the real authorizer chain server-side. Caveat: --as=jane alone drops her cert groups — add --as-group=devs to test group grants (7.8).

    11. Name the mutable fields of a running Pod, and the loop for everything else.
      Answer

      Mutable: container/initContainer image, activeDeadlineSeconds, toleration additions, (limited) terminationGracePeriodSeconds. Everything else: get -o yaml > f.yaml → edit → replace --force -f f.yaml — unless a controller owns the pod, in which case edit the controller's template instead (7.12).

    12. jsonpath: all node InternalIPs, one per line — write it.
      Answer

      kubectl get nodes -o jsonpath='{range .items[*]}{.status.addresses[?(@.type=="InternalIP")].address}{"\n"}{end}' — range/end for newlines, [?(@.type=="…")] to filter the list, whole expression single-quoted (7.11).
