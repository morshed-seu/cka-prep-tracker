    B5 — Files for real: VFS, storage, overlayfs

    You have been reading and writing files since B1 without ever asking where they physically live or what a directory is. This module answers both, from the metal up — and it ends somewhere that surprises most people: a container image, that thing you have heard described a hundred ways, turns out to be nothing more exotic than a stack of ordinary directories, which you will build by hand and take apart. There is no magic in a container's filesystem. There is mount, and there is overlayfs, and by the end of this module you will have run both yourself.

    The path there has three stops. First, real storage: a disk is a dumb device that speaks fixed-size chunks and knows nothing about files; a filesystem is a data structure someone wrote onto it; and mount is the act of grafting that structure into your directory tree. Second, the VFS — the kernel layer that makes ext4, a RAM disk, /proc, and a network share all answer to the same open/read/write, which is the whole reason the next stop is even possible. Third, overlayfs: stacking read-only layers under one thin writable layer, with copy-up and whiteouts — the exact mechanism a container runtime uses to turn a shared, read-only image into a private, writable root for each container.

    Checkpoints18 Hours~6 PrerequisitesB1, B4

    • Make a filesystem where there was none with mkfs, and explain that a filesystem is written into a device.
    • Explain what mount does — grafts a filesystem's root onto a directory — and read the live mount table.
    • Make a mount survive reboot via /etc/fstab, by UUID, with nofail, tested by mount -a.
    • Explain the VFS: one open/read/write interface that every filesystem implements.
    • Separate a file from its name — inode vs directory entry — and prove the name is not the file.
    • Tell a hard link from a symlink and predict which survives deleting the original.
    • Diagnose a full disk with no big files: df vs du, deleted-but-open files, inode exhaustion.
    • Mount memory as a filesystem with tmpfs and find the ones your VM already has.
    • Use a loop device to make an ordinary file behave like a disk.
    • Make one directory appear at a second path with a bind mount, and know why ro needs a remount.
    • Explain mount propagation — private, shared, rslave — and why a container runtime cares.
    • Mount an overlay by hand and name its four parts: lowerdir, upperdir, workdir, merged.
    • Trigger a copy-up and find the full copy that appears in upperdir.
    • Delete a lower-layer file and find the whiteout — a 0/0 character device — that hides it.

    B1 for paths, the filesystem tree, and reading ls -l; B4 because every mount and every file you touch here is still subject to the ownership and permission checks you just learned — a mount does not change who may read what. Every lab runs on the throwaway sandbox VM (B0.1), never on the cluster, and several of these commands (mount, mkfs, losetup) need sudo and can wedge a mount table if fumbled. Take a snapshot (B0.9) before the drill. No extra packages are needed — util-linux (which ships mount, lsblk, findmnt, losetup, namei) and e2fsprogs (mkfs.ext4) are already on the base image.

    Devices and filesystems

    B5.1

    The storage underneath: block devices and what /dev really holds

    Why it matters

    Before you can understand a filesystem you have to see the thing a filesystem sits on, and it is humbler than you think. A disk — physical or virtual — is a block device: a slab of storage addressable only in fixed-size chunks (blocks, classically 512 bytes or 4 KiB), that knows nothing about files, names, or directories. It can store block number 4,192 and hand it back later. That is the whole of its vocabulary. Everything else — files, folders, permissions — is a structure that software imposes on top. Seeing the raw device first is what stops "the disk is full" and "the filesystem is full" from blurring into one vague idea.

    The concept

    Linux exposes devices as special files under /dev. There are two flavours, and the difference is exactly the one that matters here:

    • A block device (ls -l shows a leading b) speaks in fixed-size, randomly-addressable blocks and is buffered by the kernel — disks, partitions, loop devices, USB sticks. This is what a filesystem lives on.
    • A character device (leading c) speaks a stream of bytes with no notion of blocks — terminals, /dev/null, /dev/random.

    The file in /dev is not the hardware; it is a handle to a kernel driver, identified by two numbers you will see in ls -l where a file's size would normally be: the major number (which driver) and the minor number (which specific device that driver manages). Opening /dev/sda doesn't open a file — it dials the block driver for major 8, minor 0. Return to the hotel from B2: the block device is the building's freight elevator, which only moves whole pallets between numbered floors; it has never heard of "the box labelled report.pdf". The labels are a system the front desk keeps — that system is the filesystem, and it is the next lesson.

    block devicestoresnumbered fixed-size blocks· knows nothing aboutfiles · names · directories

    Gotchas

    Device names are not stable across boots or machines. /dev/sda is "the first disk the kernel happened to enumerate", and plugging in a second disk, or booting on different hardware, can renumber them — yesterday's sdb becomes today's sda. This is precisely why the next lessons mount by UUID (a label baked into the filesystem itself) rather than by device name: the UUID follows the data, the name follows the enumeration order. Never hard-code /dev/sdb1 anywhere you expect to survive a reboot.

    "The disk is full" and "the filesystem is full" are different sentences. A block device has a fixed capacity; a filesystem written onto it can be full while the device has room (you made the filesystem smaller than the device), or the filesystem can report full for reasons that have nothing to do with block space at all — the inode-exhaustion case in B5.8. Keep the two layers distinct in your head from the start.

    A Kubernetes node is a machine with exactly these block devices, and several things you will configure map straight onto them. A local PersistentVolume or a hostPath is a directory on one of these partitions; the kubelet's working area under /var/lib/kubelet lives on the root partition you just saw; and when a cloud provider attaches a network disk to a node, it appears as a new block device (lsblk shows it) that the CSI driver then formats and mounts. "The node ran out of disk and started evicting pods" is a statement about the block device you just listed — being able to lsblk a node is the first move in diagnosing it.

    You will meet node storage as a scheduling and eviction concern in W5.1 (ephemeral volumes) and where the kubelet's disk pressure evicts pods in W6.

In the exam docs, anything about node ephemeral-storage and local/hostPath volumes.

    B5.2

    Making a filesystem: mkfs writes structure onto raw storage

    Why it matters

    A brand-new disk cannot hold a single file. It holds blocks, and nothing about a block says "this is where report.pdf begins" or "this directory contains three names". Someone has to write that bookkeeping onto the device first, and that act is mkfs — "make filesystem". Doing it once, by hand, on a disk you created from a plain file, turns "formatting a drive" from a scary ritual into an obviously mechanical step: you are stamping an empty ledger onto blank pages.

    The concept

    A filesystem is a data structure laid out inside a block device. For ext4 (the Linux default) the important parts are:

    • a superblock — the header describing the whole filesystem: block size, how many inodes and blocks exist, the UUID, where everything is;
    • an inode table — a fixed array of file records (metadata + pointers to data blocks), one per file, allocated when the filesystem is created (remember that number — it is finite);
    • directory blocks — the tables mapping names to inode numbers;
    • free-space bitmaps — which blocks and inodes are in use.

    The load-bearing idea: none of this is a property the device was born with. The device is blank; mkfs.ext4 writes all of the above onto it, deciding up front how many inodes to create (roughly one per 16 KiB of space by default). That up-front, fixed inode count is why you can later run out of inodes with gigabytes of free space — a failure that looks impossible until you know a filesystem is a structure with fixed-size tables, not an infinitely elastic bag.

    Lab
    1. Make a disk from nothing. fallocate creates a 128 MiB file; we will treat this ordinary file as if it were a small disk (the loop-device trick of B5.10, which mkfs and mount can use directly). 
      fallocate -l 128M /tmp/disk.img
ls -lh /tmp/disk.img
      -rw-rw-r-- 1 ubuntu ubuntu 128M Jul 24 10:05 /tmp/disk.img
    2. Stamp an ext4 filesystem onto it. -F forces mkfs to accept a plain file rather than a block device. Read the output — it is telling you the fixed sizes it just chose. 
      mkfs.ext4 -F /tmp/disk.img
      mke2fs 1.47.0 (5-Feb-2023)
Creating filesystem with 131072 1k blocks and 32768 inodes
Filesystem UUID: 7b3e...-...-...
...
Writing superblocks and filesystem accounting information: done
    3. Confirm the device now has an identity it did not have a moment ago. blkid reads the superblock and reports the type and UUID that mkfs wrote. 
      blkid /tmp/disk.img
      /tmp/disk.img: UUID="7b3e...-...-..." TYPE="ext4"
    Verify

    mkfs reported a specific number of inodes (32768 in the run above). Explain, in one sentence, what that fixed number means for a filesystem that later fills up with tiny files. (Each file consumes one inode, and the count was fixed at creation, so a filesystem can run out of inodes — refuse to create another file — while blocks remain free; you cannot add more inodes to an ext4 filesystem without rebuilding it.) Then say why blkid can now identify the file when it couldn't before. (mkfs wrote a superblock with a type and UUID; blkid reads that structure.)

    Gotchas

    mkfs is destructive and silent about it. It writes new bookkeeping over whatever was there, instantly making every existing file unreachable — there is no "are you sure" for a block device you name, and -F removes even the safety check that stops you formatting something that is mounted or looks like a whole disk. The classic disaster is running mkfs against /dev/sda instead of /dev/sdb1. On the sandbox VM this only ever touches /tmp/disk.img, but build the habit now: read the target device name twice before pressing enter.

    The filesystem type is a choice, not a given. ext4 is the safe default, but mkfs.xfs, mkfs.btrfs, and mkfs.vfat all exist and lay out completely different structures on the same blocks. "Formatting" always means "choosing a filesystem and writing its structure" — there is no such thing as a neutral, typeless format.

    This is exactly what a CSI (Container Storage Interface) driver does to a freshly attached cloud disk before a pod can use it. When you create a PersistentVolumeClaim and the provisioner carves out a new block device, something has to mkfs it — the StorageClass's fsType: ext4 (or xfs) parameter is literally choosing which mkfs runs. The first time a PV is attached to a node it is formatted; subsequent attaches skip that because the superblock already exists. Understanding that "provision a volume" includes a one-time mkfs explains why the very first pod to use a new PVC is a little slower and why a corrupted superblock makes a whole PV unmountable.

    You will see fsType on a StorageClass in W5, and the format-then-mount sequence is the mechanism behind dynamic provisioning there.

    Docs

    man 8 mkfs and man 8 mkfs.ext4 (a.k.a. mke2fs) for creating a filesystem and its options (block size, inode ratio via -i/-N), man 8 fallocate for making the backing file, and man 5 ext4 for the on-disk layout. man 8 tune2fs shows and adjusts superblock parameters after the fact. In the exam docs, the StorageClass fsType parameter.

    B5.3

    What mount actually does: grafting a filesystem onto a directory

    Why it matters

    "Mount the disk" is one of those phrases everyone uses and few can define precisely. Here is the precise version, and it is simpler than the mystique suggests: mounting takes a filesystem's root and grafts it onto an existing directory, so that from then on, walking into that directory leads into the filesystem instead. Nothing is copied; a pointer in the kernel's tree is redirected. Once you have watched a directory's contents disappear behind a mount and reappear when you unmount, the entire Linux filesystem tree — which is assembled this way, from many mounts — stops being a given and becomes something you can see the seams of.

    The concept

    You have one tree of directories starting at / (B1). A filesystem also has its own internal root. Mounting connects the two: mount <device> <directory> says "when anyone walks into directory, send them into the root of the filesystem on device instead." That directory is now a mount point. Three consequences follow directly:

    • Whatever was inside the directory before is not deleted — it is hidden, shadowed by the mount, and it reappears the instant you unmount.
    • The files you now see and create in that directory live on the mounted device, not on the underlying one.
    • The whole visible tree is a graft of many filesystems — /, /boot/efi, /proc, and every other mount, stitched into one seamless namespace. There is no single disk holding "everything".

    The kernel keeps a mount table — the live list of every graft: which device, at which directory, of which type, with which options. You read it with findmnt (a readable tree) or the raw /proc/mounts. That table, not any one disk, is the shape of your filesystem.

    mount device → /mnt/data→ walking into/mnt/datanow leads tothe filesystem's root· old contentshidden until umount
    Lab
    1. Prove the shadowing. Put a file in an ordinary directory first, so you can watch it vanish under the mount and return afterwards. 
      sudo mkdir -p /mnt/data

      sudo mount -o loop /tmp/disk.img /mnt/data

      sudo umount /mnt/data

    Verify

    After mounting, hello was visible and original was not; after unmounting, the reverse. Explain where original was during the mount. (Still on the underlying root filesystem, in the /mnt/data directory — never deleted, only shadowed by the mount, which is why unmounting revealed it again.) Then read your own findmnt and name the four facts the mount table records for a mount. (Target directory, source device, filesystem type, and options.)

    Gotchas

    umount fails with "target is busy" if any process has that filesystem as its working directory or has a file open on it — the kernel will not yank a filesystem out from under a running process. The fix is to find the holder (lsof /mnt/data or fuser -vm /mnt/data) and move it, not to force the unmount; a lazy unmount (umount -l) detaches the name immediately but leaves the filesystem alive until the last user lets go, which can mask the real problem. This is the same "who is holding this open" question as the deleted-but-open file in B5.8.

    Mounting onto a non-empty directory is legal and occasionally a real bug. Nothing stops you mounting over a directory that already contains files — they simply vanish from view until unmount, and someone later wonders where their data went. It is safe (the data is fine), but "my files disappeared" is often just a filesystem mounted over the top of them; findmnt on the parent path tells you immediately.

    A Kubernetes volume is a mount, full stop. When a pod declares a volume, the kubelet performs exactly this graft: it mounts the volume's filesystem onto a directory inside the container's filesystem, so the path the container sees leads into the volume instead of into the image's own layer. This is why data written to a mounted path persists (it is on the mounted device) while data written elsewhere in the container does not (that is the throwaway upper layer of B5.16), and why an emptyDir mounted over a directory that had files in the image hides those files — the exact shadowing you just watched. Every "why is this directory empty / why did my image's files disappear under a volume" question is this lesson.

    You will see the kubelet's mounts concretely in W1.23 (the kubelet's /var/lib/kubelet/pods tree) and the volume-mount mechanics throughout W5.

    Docs

    man 8 mount for mounting and its options (the FILESYSTEM-INDEPENDENT MOUNT OPTIONS section is the useful core), man 8 umount for unmounting and -l/-f, and man 8 findmnt for reading the mount table as a tree. man 5 proc under /proc/mounts and /proc/self/mountinfo for the raw form. In the exam docs, anything about pod volumes and mount paths.

    B5.4

    Mounts that survive reboot: /etc/fstab, UUIDs, and nofail

    Why it matters

    A mount command lasts until the next reboot and then it is gone. To make a disk come back automatically — the normal expectation for anything permanent — you write it into /etc/fstab, the file the system reads at boot to reassemble the filesystem tree. Getting one field wrong here is one of the few ways to make a Linux box fail to boot into a usable state, so this lesson is as much about the safe habits (mount by UUID, always nofail, always test with mount -a) as about the six fields.

    The concept

    /etc/fstab is a plain table, one mount per line, six whitespace-separated fields:

    #FieldWhat it is
    1specwhat to mount — best written as UUID=…, not /dev/sdb1
    2mount pointthe directory to graft it onto
    3typeext4, xfs, tmpfs, …
    4optionscomma-separated, e.g. defaults,nofail
    5dumpancient backup flag — almost always 0
    6passfsck order at boot — 1 for root, 2 for others, 0 to skip

    Two habits make this safe. First, mount by UUID, not device name: a UUID is baked into the filesystem's superblock (B5.2) and follows the data no matter how the kernel renumbers /dev/sda versus /dev/sdb from one boot to the next — the reordering hazard from B5.1. Second, nofail: without it, a device listed in fstab that is missing at boot (an unplugged USB disk, a cloud volume that failed to attach) can drop the whole boot into an emergency shell. nofail says "if this one isn't here, carry on without it" — which is almost always what you want for anything that isn't the root filesystem. And the golden rule: always run mount -a before you reboot, because it mounts everything in fstab exactly as boot will, surfacing a typo now, while you still have a shell, instead of after a reboot that won't complete.

    Lab
    1. Add a persistent entry. Our device is a file, so the spec field is its path plus the loop option; for a real disk you would write UUID=… here instead (shown in the note below). nofail keeps a missing image from wedging a future boot. 
      echo '/tmp/disk.img /mnt/data ext4 loop,nofail 0 2' | sudo tee -a /etc/fstab
    2. Test it the safe way — mount -a mounts everything in fstab that isn't already mounted, exactly as boot will. No error means the line is good. 
      sudo mount -a
findmnt /mnt/data
      TARGET    SOURCE      FSTYPE OPTIONS
/mnt/data /dev/loop0  ext4   rw,relatime,nofail
    3. Clean up so you don't leave a test mount behind. Unmount, then delete the line you added. 
      sudo umount /mnt/data
sudo sed -i '\#/tmp/disk.img /mnt/data#d' /etc/fstab
grep -c /tmp/disk.img /etc/fstab
      0                                       ← the test line is gone
    Verify

    For a real second disk whose partition is /dev/sdb1, write the fstab line that mounts it at /data as ext4, robustly. (Get the UUID with sudo blkid /dev/sdb1, then: UUID=<that> /data ext4 defaults,nofail 0 2.) Then say why the UUID form is safer than /dev/sdb1 and what mount -a buys you. (The UUID follows the filesystem even if the kernel renames the device across boots; mount -a tests the whole file while you still have a shell, so a typo can't first reveal itself as a failed boot.)

    Gotchas

    A broken fstab line can stop a machine booting. A wrong device, a bad option, or a missing nofail on an absent disk can drop boot into an emergency shell — and on a headless VM or cloud instance that can mean no way in at all. The two defences are non-negotiable: run mount -a after every edit (it fails loudly, now), and put nofail on anything that isn't the root filesystem. Snapshot before experimenting on a machine you care about.

    Why UUID= can't identify our loop image: a UUID is resolved by scanning block devices, and a plain file isn't one until a loop device is attached to it. That is the whole reason the lab's fstab line uses the file path with the loop option instead of a UUID — the file-backed case is the exception. For any actual disk, use the UUID.

    The kubelet and CSI drivers do the equivalent of fstab in software: when a node reboots, the volumes a pod needs must be re-mounted, and the machinery tracks each PV by a stable identifier (much like a UUID) rather than a device name, precisely because the cloud may re-attach the same disk as a different /dev/ node. The nofail instinct maps too — a robust node keeps booting and running when an optional mount is absent, rather than falling over. You rarely edit fstab on a Kubernetes node for pod storage (the kubelet owns those mounts), but the node's own disks — the root filesystem, a separate /var/lib/containerd — are still plain fstab entries, and a botched one bricks the node the same way it bricks any Linux box.

    The re-mount-on-reboot behaviour underlies PV attach/detach in W5; the "node won't come back after reboot" failure is a W6 troubleshooting scenario.

    Docs

    man 5 fstab for the six fields and their meanings, man 8 mount under -a and the nofail/defaults options, and man 8 blkid for getting the UUID to put in field one. man 8 systemd.mount notes how modern systems translate fstab into mount units. No direct exam-docs analogue — this is node-level plumbing beneath the cluster.

    The VFS view

    B5.5

    The VFS: one interface over every filesystem

    Why it matters

    You have used cat on a file in your home directory, on a file under /proc that isn't really a file at all, and (in a moment) on a file in a RAM disk — and it just worked, identically, every time. That is not luck. It is the VFS, the single most important abstraction in this whole module, because it is the thing that lets the next lessons hand a process an entirely different filesystem without changing a single line of any program. Grasp the VFS and overlayfs stops being surprising; it is just one more filesystem plugged into the same socket.

    The concept

    The VFS (virtual filesystem) is a layer inside the kernel that defines one set of operations — open, read, write, stat, readdir, and a few more — and forces every real filesystem to implement them. ext4 implements them by reading blocks off a disk. tmpfs implements them by reading pages of RAM. procfs implements read by making up the answer on the spot from kernel data structures. overlayfs implements them by consulting a stack of other filesystems. A network filesystem implements them by sending packets to another machine. The programs above — cat, your editor, every process — call the same syscalls (B3) and never know or care which implementation is underneath.

    Think of it as a standard wall socket. Every appliance has the same plug; behind the wall the power might come from the grid, a generator, or solar panels, and the appliance neither knows nor needs to. The VFS is that socket for storage: one plug shape (open/read/write), many sources behind the wall. This is precisely why B6 can put a process into a completely different root filesystem and every ordinary program keeps working — they were never talking to ext4, they were always talking to the VFS.

    cat · your editor · any process→ same syscalls →VFS→ext4 · tmpfs · procfs · overlayfs · NFS
    Lab

    Explain, in one sentence, why cat needs no special knowledge of ext4, tmpfs, or procfs to read a file on any of them. (Because cat only ever calls the VFS operations open/read; the kernel routes those to whichever filesystem implementation owns the path, so the difference is entirely below cat's view.) Then answer: how does this abstraction make overlayfs possible at all? (overlayfs is just another VFS implementation — it satisfies the same operations by consulting a stack of other filesystems, so every program works on it unchanged.)

    Gotchas

    "Everything is a file" is a VFS statement, and it has limits. /proc/uptime looks like a file and reads like one, but it has no blocks on any disk — its read runs kernel code that produces the answer on the spot, which is why its size shows as 0 and why you cannot seek around it like a real file. The uniform interface is real, but some filesystems implement only part of it or implement it with surprises; treating a /proc or /sys file as if it were an ordinary ext4 file (copying it, checking its size, memory-mapping it) is a common source of confusion.

    Not every operation is supported by every filesystem. A read-only filesystem's write returns EROFS; procfs directories can't be renamed; some virtual files reject seek. The VFS defines the interface; each filesystem chooses which parts it honours, so "the interface is uniform" does not mean "every operation always succeeds".

    The VFS is the quiet reason a container works at all. When a container runtime gives a pod its own root filesystem (an overlay mount, coming up), every binary inside the container — none of which was compiled with any knowledge of overlayfs — keeps functioning, because they all speak the VFS interface and overlayfs sits behind it. The same uniformity is why Kubernetes can project a ConfigMap as files, back an emptyDir with either disk or tmpfs (RAM), and mount a network volume, all appearing to the application as ordinary files it reads with ordinary code. "It's just a file to the app" is the VFS keeping its promise.

    The RAM-vs-disk choice for an emptyDir (W5.1) is exactly "which VFS implementation backs this mount"; the ConfigMap-as-files projection is the same idea.

    Docs

    man 2 open, man 2 read, man 2 stat for the operations the VFS defines; man 5 filesystems and cat /proc/filesystems for the registered implementations. man 8 findmnt with -o FSTYPE shows which one backs each mount. man 5 proc for how procfs fakes files convincingly. No exam-docs analogue — this is kernel architecture the cluster stands on.

    B5.6

    The file is not its name: inodes and directory entries

    Why it matters

    Almost everyone carries a wrong mental model: that a file is its name and path, the way a variable is its label. Linux does not work that way, and the gap between the two explains a whole cluster of otherwise-baffling behaviours — why you can delete a file another process is still happily reading, why a file can have two names, why "renaming" is instantaneous no matter how large the file. The fix is one clean idea: the file and its name are separate things, joined by a number.

    The concept

    Every file is really an inode — an on-disk record holding all the file's actual properties: its type, owner and group, permission bits, size, timestamps, link count, and the pointers to the data blocks. The inode has a number but no name. The name lives somewhere else entirely: a directory is just a table of (name → inode number) pairs. So "the file /home/ubuntu/notes.txt" is really "walk to the directory /home/ubuntu, look up the name notes.txt in its table, get an inode number, go read that inode." The name is a label in a table; the file is the inode the label points at.

    That indirection is the point. It means a single inode can be reachable by more than one name (two directory entries pointing at the same number — a hard link, next lesson). It means "deleting a file" is really removing one (name → number) entry, and the inode itself only disappears when its link count — how many names point at it — reaches zero and no process still has it open. And it means renaming or moving within a filesystem just rewrites a table entry; the inode and its gigabytes of data never move.

    name in a directory→ maps toinode number→ points atinode: owner · mode · size · data blocks
    Lab
    1. See the number behind the name. ls -i prints each name's inode number; stat shows the same number plus everything the inode actually stores — note there is no filename anywhere in the inode's own fields. 
      echo hello > /tmp/a.txt
ls -i /tmp/a.txt
stat /tmp/a.txt
      266241 /tmp/a.txt
  File: /tmp/a.txt
  Size: 6   Blocks: 8   IO Block: 4096   regular file
Device: 803h/2051d   Inode: 266241   Links: 1
Access: (0664/-rw-rw-r--)  Uid: (1000/ubuntu)  Gid: (1000/ubuntu)
    2. Prove the name is separate from the file by renaming and watching the inode number stay identical — same file, new label. 
      mv /tmp/a.txt /tmp/b.txt
ls -i /tmp/b.txt
      266241 /tmp/b.txt              ← same inode: only the directory entry changed
    3. Look at a directory as a table. Every entry is a name paired with an inode number, including the two the filesystem always creates: . (this directory) and .. (its parent). 
      ls -ai /tmp | head
            2 .
      2 ..
266241 b.txt
...
    Verify

    You ran mv /tmp/a.txt /tmp/b.txt and the inode number did not change. Explain what mv actually did to the directory table. (It removed the entry a.txt → 266241 and added b.txt → 266241 — the inode and its data were never touched, which is why renaming a huge file is instant.) Then answer: if two names in a directory both showed inode 266241, what would be true about them? (They would be the same file — two labels for one inode — so editing "one" changes "both", because there is only one.)

    Gotchas

    Inode numbers are unique only within one filesystem, not globally. Two files on different filesystems can share an inode number and be completely unrelated — which is why stat reports a Device as well as an Inode, and why the true identity of a file is the (device, inode) pair. Tools that de-duplicate or detect hard links compare both; comparing inode numbers alone across mount points gives false matches.

    Deleting a name is not always deleting the file. rm removes a directory entry and decrements the link count; the data blocks are freed only when the count hits zero and no process holds the inode open. This is the mechanism behind the deleted-but-open puzzle in B5.8 — the name is gone, the inode isn't, because a file descriptor is still a reference to it.

    The inode-vs-name split is why atomic config updates work the way they do in Kubernetes. When the kubelet updates a projected ConfigMap or Secret volume, it does not edit the files in place — it builds a new directory of the content and swaps a symlink to point at it, so a reader either sees the entire old version or the entire new one, never a half-written file. That trick is only possible because a name is an independent pointer to an inode, exactly what this lesson establishes. The same reasoning explains why a container that had a file open keeps reading the old content after the ConfigMap changes (its descriptor references the old inode) until it re-opens by name.

    The projected-volume atomic-swap behaviour surfaces in W5.1; the "I updated the ConfigMap but the running process still sees the old value" gotcha is this inode reference in action.

    Docs

    man 7 inode for what an inode stores (and the file-type bits), man 1 stat for reading it (Inode, Links, Device), and man 1 ls under -i. man 2 rename confirms that renaming within a filesystem is a directory-entry operation. man 3 readdir shows a directory being read as the table of entries it is.

    B5.7

    Two kinds of link: hard links and symlinks

    Why it matters

    Now that you know a name and a file are different things, links follow naturally — and there are exactly two kinds, which people constantly confuse. A hard link is a second true name for the same inode; a symbolic link is a tiny separate file that merely holds a path to somewhere else. They behave differently the moment you delete or move the original, and knowing which is which is the difference between "my link still works" and "my link is now a dangling arrow to nothing".

    The concept

    Both let one file be reachable two ways, but the mechanism is opposite:

    • A hard link (ln target newname) adds a second (name → inode number) entry pointing at the same inode. There is no "original" and "copy" — both names are equal, first-class references to one file, and the inode's link count goes up by one. Delete either name and the file survives under the other; the data vanishes only when the last name is removed. Because it references an inode number, a hard link cannot cross filesystems (numbers are per-filesystem) and, by convention, cannot point at a directory.
    • A symbolic link (ln -s target newname) creates a brand-new, separate file — its own inode — whose entire contents are a text path to the target. It is a signpost. Follow it and you are redirected to whatever that path currently names. Delete or move the target and the symlink still exists but now points at nothing — a "dangling" link. Symlinks can cross filesystems and can point at directories, precisely because they store a path, not an inode number.

    The one-line test to keep them straight: a hard link shares the file's identity (same inode); a symlink shares only the file's address (a path), and addresses can go stale.

    Lab
    1. Make a hard link and watch the link count. Both names share one inode number, and the count in stat is now 2. 
      echo 'the original' > /tmp/orig
ln /tmp/orig /tmp/hard
ls -li /tmp/orig /tmp/hard
      266300 -rw-rw-r-- 2 ubuntu ubuntu 13 ... /tmp/orig     ← same inode 266300,
266300 -rw-rw-r-- 2 ubuntu ubuntu 13 ... /tmp/hard     ← link count 2
    2. Delete the "original" and prove the hard link kept the data alive — the inode had another name, so nothing was freed; the count drops to 1. 
      rm /tmp/orig
cat /tmp/hard
stat -c 'links=%h' /tmp/hard
      the original                        ← still here: the inode outlived the name
links=1
    3. Now a symlink, and break it. The symlink is its own tiny file (note the different inode and the l type); delete its target and it dangles. 
      echo 'target data' > /tmp/real
ln -s /tmp/real /tmp/link
ls -li /tmp/link; readlink /tmp/link
rm /tmp/real
cat /tmp/link
      266322 lrwxrwxrwx 1 ubuntu ubuntu 9 ... /tmp/link -> /tmp/real
/tmp/real
cat: /tmp/link: No such file or directory   ← dangling: the path no longer resolves
    Verify

    Predict before deleting anything: you have a file, a hard link to it, and a symlink to it. You rm the file's original name. Which of the other two still gives you the data, and why? (The hard link does — it is an equal name for the same inode, so removing one name leaves the inode alive. The symlink does not — it stored the now-gone path, so it dangles.) Then say why a hard link can't span two filesystems but a symlink can. (A hard link references an inode number, which is only meaningful within one filesystem; a symlink stores a path string, which is meaningful anywhere.)

    Gotchas

    A dangling symlink is not an error until you follow it. ls -l will happily show a symlink whose target no longer exists (often highlighted in red), and creating one is legal — you can make a symlink to a path that doesn't exist yet. The failure only appears at the moment something tries to resolve it, as ENOENT. So "the file exists (I can ls the link) but I can't read it" is very often a broken symlink; readlink -f shows where it actually points.

    Editors can quietly convert a hard link into a separate file. Some editors save by writing a new file and renaming it over the old name, which breaks the hard-link relationship — after such a save, your two "hard-linked" names are now two independent files. Hard links are best for stable data, not for files that get rewritten by tools that don't preserve them; when in doubt, stat the link count before and after.

    Symlinks are load-bearing in a container's filesystem and in Kubernetes volume projection. The atomic ConfigMap/Secret update from B5.6 works by swapping a symlink (..data) to point at a new timestamped directory — the whole "consistent view during an update" guarantee rides on symlink semantics. Container images also lean on symlinks heavily (for example, /lib pointing into /usr/lib), and a symlink that points outside the intended tree is a classic container-escape and image-scanning concern. Recognising a projected volume's ..data symlink for what it is demystifies the odd directory layout you'll see inside a mounted ConfigMap.

    You'll see that ..data symlink structure when you inspect a mounted ConfigMap in W3; the atomic-update behaviour is revisited in W5.

    Docs

    man 1 ln for creating both kinds (-s for symbolic), man 7 symlink for how symlink resolution works, and man 2 link/man 2 symlink for the syscalls. man 1 readlink (-f to resolve fully) and man 1 stat (link count via %h) for inspecting them.

    B5.8

    The disk is full and nothing is big: df vs du, open-but-deleted, inodes

    Why it matters

    Three of the most common "how is this even possible" disk incidents all come from the same root: the filesystem's own accounting and the sum of the files you can see are two different numbers, and they drift apart for specific, knowable reasons. A disk reports 100% full but du finds only a few megabytes. A file you deleted didn't free any space. A filesystem refuses to create a file with gigabytes free. Each looks like a bug and each has a five-second explanation once you know where to look — and this is the exact toolkit the B5.18 drill and real W6 pages lean on.

    The concept

    Two commands answer "how much space?" from opposite directions:

    • df asks the filesystem: from the superblock's own bookkeeping, how many blocks (and, with -i, how many inodes) are used and free? This is the authoritative "how full am I" number.
    • du walks the directory tree and adds up the sizes of the files it can reach by name. This is "how much do the visible files account for".

    When these two disagree, the gap is the whole diagnosis. Three causes cover almost every case:

    1. A deleted-but-still-open file. A process opened a file, then it was deleted (its name removed) while the process still holds the descriptor. From B5.6: the name is gone so du can't see it, but the inode and its data blocks survive until the process closes the fd — so df still counts the space. Result: df says full, du says empty. The holder is visible in /proc/PID/fd as a link marked (deleted).
    2. Inode exhaustion. The fixed inode count from mkfs (B5.2) ran out — millions of tiny files each took one inode — so the filesystem can't create another file even though block space is free. df -i shows 100% inodes used while df shows blocks free.
    3. Space hidden under a mount point. Files written to a directory before something was mounted over it (B5.3) still occupy the underlying filesystem but are shadowed, so du of the mounted path can't see them.
    Lab
    1. Reproduce the deleted-but-open file. Start a process writing to a file, then delete the file's name while that process keeps it open — the space stays used with no visible file to blame. 
      ( tail -f /dev/null > /tmp/ghost.log & echo $! ) && sleep 1
# grow it, then delete the NAME while the writer holds it open:
dd if=/dev/zero of=/tmp/ghost.log bs=1M count=20 conv=notrunc 2>/dev/null
rm /tmp/ghost.log
ls /tmp/ghost.log
      1234                                    ← the writer's PID
ls: cannot access '/tmp/ghost.log': No such file or directory
    2. Find the ghost. The name is gone but a process still holds the inode; /proc/PID/fd shows the descriptor pointing at the deleted file, which is why the space isn't back. 
      ls -l /proc/1234/fd | grep deleted
      l-wx------ ... 1 -> /tmp/ghost.log (deleted)   ← space freed only when this closes
    3. See inode accounting directly. df -i reports inodes, a different budget from blocks — this is the number that hits zero in the exhaustion case even when df shows free space. (Kill the writer to release the ghost.) 
      df -i /tmp
kill 1234
      Filesystem  Inodes  IUsed  IFree IUse% Mounted on
/dev/sda1   322400   90312 232088   29% /       ← inodes, counted separately from bytes
    Verify

    df said space was used but ls found no file. Explain the discrepancy in terms of inodes and link counts. (The file's name was removed, so du/ls can't see it, but a process still holds the inode open — its link count is 0 but its open count is not — so the inode and its blocks persist, and df, which reads the filesystem's own accounting, still counts them.) Then say which command you'd reach for first when a disk is "full" but files look small, and what it would tell you. (df -i — if inodes are exhausted while blocks are free, the problem is millions of tiny files, not big ones.)

    Gotchas

    Restarting the process is the fix for a deleted-but-open file, and deleting more files is not. Since the space is held by an open descriptor, no amount of hunting for large files helps — you must make the holder close the fd, usually by restarting it (or, in a pinch, truncating through /proc/PID/fd/N). The reflex when df and du disagree by a lot: look for (deleted) in /proc/*/fd (or run lsof +L1) before anything else.

    Inode exhaustion is invisible to plain df. The default df shows only block usage, so a filesystem at 100% inodes but 40% blocks looks half-empty right up until every touch and write fails with No space left on device — the same ENOSPC error as a truly full disk, for a completely different reason. Always check df -i when ENOSPC appears on a disk that df says has room.

    Both failures are routine node incidents in Kubernetes. A container that logs heavily and has its log file rotated (renamed and deleted) while the process holds it open produces the classic "the node disk is full but I can't find what's using it" page — it is the deleted-but-open file, one layer down, and the fix is restarting the offending pod. Inode exhaustion hits nodes that run many short-lived pods or write millions of small files, and it triggers the kubelet's disk-pressure eviction with plenty of bytes free — mystifying until you run df -i on the node. The df/du/df -i/lsof reflexes transfer directly to debugging a node.

    Disk-pressure eviction and "node disk full" are W6 troubleshooting scenarios; the ephemeral-storage limits that bound a pod's disk use are in W5.

    Docs

    man 1 df (-h for readable, -i for inodes) and man 1 du (-sh, -x to stay on one filesystem) for the two views, man 8 lsof (+L1 finds open-but-unlinked files) for the holder, and man 5 proc under /proc/[pid]/fd. Re-read B5.6 — this whole lesson is the inode/link-count model made operational.
