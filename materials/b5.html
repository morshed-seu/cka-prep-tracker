    B5 — Files for real: VFS, storage, overlayfs

    You have been reading and writing files since B1 without ever asking where they physically live or what a directory is. This module answers both, from the metal up — and it ends somewhere that surprises most people: a container image, that thing you have heard described a hundred ways, turns out to be nothing more exotic than a stack of ordinary directories, which you will build by hand and take apart. There is no magic in a container's filesystem. There is mount, and there is overlayfs, and by the end of this module you will have run both yourself.

    The path there has three stops. First, real storage: a disk is a dumb device that speaks fixed-size chunks and knows nothing about files; a filesystem is a data structure someone wrote onto it; and mount is the act of grafting that structure into your directory tree. Second, the VFS — the kernel layer that makes ext4, a RAM disk, /proc, and a network share all answer to the same open/read/write, which is the whole reason the next stop is even possible. Third, overlayfs: stacking read-only layers under one thin writable layer, with copy-up and whiteouts — the exact mechanism a container runtime uses to turn a shared, read-only image into a private, writable root for each container.

    Devices and filesystems

    B5.1

    The storage underneath: block devices and what /dev really holds

    Why it matters

    Before you can understand a filesystem you have to see the thing a filesystem sits on, and it is humbler than you think. A disk — physical or virtual — is a block device: a slab of storage addressable only in fixed-size chunks (blocks, classically 512 bytes or 4 KiB), that knows nothing about files, names, or directories. It can store block number 4,192 and hand it back later. That is the whole of its vocabulary. Everything else — files, folders, permissions — is a structure that software imposes on top. Seeing the raw device first is what stops "the disk is full" and "the filesystem is full" from blurring into one vague idea.

    The concept

    Linux exposes devices as special files under /dev. There are two flavours, and the difference is exactly the one that matters here:

    • A block device (ls -l shows a leading b) speaks in fixed-size, randomly-addressable blocks and is buffered by the kernel — disks, partitions, loop devices, USB sticks. This is what a filesystem lives on.
    • A character device (leading c) speaks a stream of bytes with no notion of blocks — terminals, /dev/null, /dev/random.

    The file in /dev is not the hardware; it is a handle to a kernel driver, identified by two numbers you will see in ls -l where a file's size would normally be: the major number (which driver) and the minor number (which specific device that driver manages). Opening /dev/sda doesn't open a file — it dials the block driver for major 8, minor 0. Return to the hotel from B2: the block device is the building's freight elevator, which only moves whole pallets between numbered floors; it has never heard of "the box labelled report.pdf". The labels are a system the front desk keeps — that system is the filesystem, and it is the next lesson.

    Gotchas

    Device names are not stable across boots or machines. /dev/sda is "the first disk the kernel happened to enumerate", and plugging in a second disk, or booting on different hardware, can renumber them — yesterday's sdb becomes today's sda. This is precisely why the next lessons mount by UUID (a label baked into the filesystem itself) rather than by device name: the UUID follows the data, the name follows the enumeration order. Never hard-code /dev/sdb1 anywhere you expect to survive a reboot.

    "The disk is full" and "the filesystem is full" are different sentences. A block device has a fixed capacity; a filesystem written onto it can be full while the device has room (you made the filesystem smaller than the device), or the filesystem can report full for reasons that have nothing to do with block space at all — the inode-exhaustion case in B5.8. Keep the two layers distinct in your head from the start.

    A Kubernetes node is a machine with exactly these block devices, and several things you will configure map straight onto them. A local PersistentVolume or a hostPath is a directory on one of these partitions; the kubelet's working area under /var/lib/kubelet lives on the root partition you just saw; and when a cloud provider attaches a network disk to a node, it appears as a new block device (lsblk shows it) that the CSI driver then formats and mounts. "The node ran out of disk and started evicting pods" is a statement about the block device you just listed — being able to lsblk a node is the first move in diagnosing it.

    You will meet node storage as a scheduling and eviction concern in W5.1 (ephemeral volumes) and where the kubelet's disk pressure evicts pods in W6.

    B5.2

    Making a filesystem: mkfs writes structure onto raw storage

    Why it matters

    A brand-new disk cannot hold a single file. It holds blocks, and nothing about a block says "this is where report.pdf begins" or "this directory contains three names". Someone has to write that bookkeeping onto the device first, and that act is mkfs — "make filesystem". Doing it once, by hand, on a disk you created from a plain file, turns "formatting a drive" from a scary ritual into an obviously mechanical step: you are stamping an empty ledger onto blank pages.

    The concept

    A filesystem is a data structure laid out inside a block device. For ext4 (the Linux default) the important parts are:

    • a superblock — the header describing the whole filesystem: block size, how many inodes and blocks exist, the UUID, where everything is;
    • an inode table — a fixed array of file records (metadata + pointers to data blocks), one per file, allocated when the filesystem is created (remember that number — it is finite);
    • directory blocks — the tables mapping names to inode numbers;
    • free-space bitmaps — which blocks and inodes are in use.

    The load-bearing idea: none of this is a property the device was born with. The device is blank; mkfs.ext4 writes all of the above onto it, deciding up front how many inodes to create (roughly one per 16 KiB of space by default). That up-front, fixed inode count is why you can later run out of inodes with gigabytes of free space — a failure that looks impossible until you know a filesystem is a structure with fixed-size tables, not an infinitely elastic bag.

    Lab
    1. Make a disk from nothing. fallocate creates a 128 MiB file; we will treat this ordinary file as if it were a small disk (the loop-device trick of B5.10, which mkfs and mount can use directly).

    mkfs reported a specific number of inodes (32768 in the run above). Explain, in one sentence, what that fixed number means for a filesystem that later fills up with tiny files. (Each file consumes one inode, and the count was fixed at creation, so a filesystem can run out of inodes — refuse to create another file — while blocks remain free; you cannot add more inodes to an ext4 filesystem without rebuilding it.) Then say why blkid can now identify the file when it couldn't before. (mkfs wrote a superblock with a type and UUID; blkid reads that structure.)

    Gotchas

    mkfs is destructive and silent about it. It writes new bookkeeping over whatever was there, instantly making every existing file unreachable — there is no "are you sure" for a block device you name, and -F removes even the safety check that stops you formatting something that is mounted or looks like a whole disk. The classic disaster is running mkfs against /dev/sda instead of /dev/sdb1. On the sandbox VM this only ever touches /tmp/disk.img, but build the habit now: read the target device name twice before pressing enter.

    The filesystem type is a choice, not a given. ext4 is the safe default, but mkfs.xfs, mkfs.btrfs, and mkfs.vfat all exist and lay out completely different structures on the same blocks. "Formatting" always means "choosing a filesystem and writing its structure" — there is no such thing as a neutral, typeless format.

    This is exactly what a CSI (Container Storage Interface) driver does to a freshly attached cloud disk before a pod can use it. When you create a PersistentVolumeClaim and the provisioner carves out a new block device, something has to mkfs it — the StorageClass's fsType: ext4 (or xfs) parameter is literally choosing which mkfs runs. The first time a PV is attached to a node it is formatted; subsequent attaches skip that because the superblock already exists. Understanding that "provision a volume" includes a one-time mkfs explains why the very first pod to use a new PVC is a little slower and why a corrupted superblock makes a whole PV unmountable.

    You will see fsType on a StorageClass in W5, and the format-then-mount sequence is the mechanism behind dynamic provisioning there.

    Docs

    man 8 mkfs and man 8 mkfs.ext4 (a.k.a. mke2fs) for creating a filesystem and its options (block size, inode ratio via -i/-N), man 8 fallocate for making the backing file, and man 5 ext4 for the on-disk layout. man 8 tune2fs shows and adjusts superblock parameters after the fact. In the exam docs, the StorageClass fsType parameter.

    B5.3

    What mount actually does: grafting a filesystem onto a directory

    Why it matters

    "Mount the disk" is one of those phrases everyone uses and few can define precisely. Here is the precise version, and it is simpler than the mystique suggests: mounting takes a filesystem's root and grafts it onto an existing directory, so that from then on, walking into that directory leads into the filesystem instead. Nothing is copied; a pointer in the kernel's tree is redirected. Once you have watched a directory's contents disappear behind a mount and reappear when you unmount, the entire Linux filesystem tree — which is assembled this way, from many mounts — stops being a given and becomes something you can see the seams of.

    The concept

    You have one tree of directories starting at / (B1). A filesystem also has its own internal root. Mounting connects the two: mount <device> <directory> says "when anyone walks into directory, send them into the root of the filesystem on device instead." That directory is now a mount point. Three consequences follow directly:

    • Whatever was inside the directory before is not deleted — it is hidden, shadowed by the mount, and it reappears the instant you unmount.
    • The files you now see and create in that directory live on the mounted device, not on the underlying one.
    • The whole visible tree is a graft of many filesystems — /, /boot/efi, /proc, and every other mount, stitched into one seamless namespace. There is no single disk holding "everything".

    The kernel keeps a mount table — the live list of every graft: which device, at which directory, of which type, with which options. You read it with findmnt (a readable tree) or the raw /proc/mounts. That table, not any one disk, is the shape of your filesystem.

    mount device → /mnt/data→ walking into/mnt/datanow leads tothe filesystem's root· old contentshidden until umount

    Gotchas

    umount fails with "target is busy" if any process has that filesystem as its working directory or has a file open on it — the kernel will not yank a filesystem out from under a running process. The fix is to find the holder (lsof /mnt/data or fuser -vm /mnt/data) and move it, not to force the unmount; a lazy unmount (umount -l) detaches the name immediately but leaves the filesystem alive until the last user lets go, which can mask the real problem. This is the same "who is holding this open" question as the deleted-but-open file in B5.8.

    Mounting onto a non-empty directory is legal and occasionally a real bug. Nothing stops you mounting over a directory that already contains files — they simply vanish from view until unmount, and someone later wonders where their data went. It is safe (the data is fine), but "my files disappeared" is often just a filesystem mounted over the top of them; findmnt on the parent path tells you immediately.

    A Kubernetes volume is a mount, full stop. When a pod declares a volume, the kubelet performs exactly this graft: it mounts the volume's filesystem onto a directory inside the container's filesystem, so the path the container sees leads into the volume instead of into the image's own layer. This is why data written to a mounted path persists (it is on the mounted device) while data written elsewhere in the container does not (that is the throwaway upper layer of B5.16), and why an emptyDir mounted over a directory that had files in the image hides those files — the exact shadowing you just watched. Every "why is this directory empty / why did my image's files disappear under a volume" question is this lesson.

    You will see the kubelet's mounts concretely in W1.23 (the kubelet's /var/lib/kubelet/pods tree) and the volume-mount mechanics throughout W5.

In the exam docs, anything about pod volumes and mount paths.

    B5.4

    Mounts that survive reboot: /etc/fstab, UUIDs, and nofail

    Why it matters

    A mount command lasts until the next reboot and then it is gone. To make a disk come back automatically — the normal expectation for anything permanent — you write it into /etc/fstab, the file the system reads at boot to reassemble the filesystem tree. Getting one field wrong here is one of the few ways to make a Linux box fail to boot into a usable state, so this lesson is as much about the safe habits (mount by UUID, always nofail, always test with mount -a) as about the six fields.

    The concept

    /etc/fstab is a plain table, one mount per line, six whitespace-separated fields:

    #FieldWhat it is
    1specwhat to mount — best written as UUID=…, not /dev/sdb1
    2mount pointthe directory to graft it onto
    3typeext4, xfs, tmpfs, …
    4optionscomma-separated, e.g. defaults,nofail
    5dumpancient backup flag — almost always 0
    6passfsck order at boot — 1 for root, 2 for others, 0 to skip

    Two habits make this safe. First, mount by UUID, not device name: a UUID is baked into the filesystem's superblock (B5.2) and follows the data no matter how the kernel renumbers /dev/sda versus /dev/sdb from one boot to the next — the reordering hazard from B5.1. Second, nofail: without it, a device listed in fstab that is missing at boot (an unplugged USB disk, a cloud volume that failed to attach) can drop the whole boot into an emergency shell. nofail says "if this one isn't here, carry on without it" — which is almost always what you want for anything that isn't the root filesystem. And the golden rule: always run mount -a before you reboot, because it mounts everything in fstab exactly as boot will, surfacing a typo now, while you still have a shell, instead of after a reboot that won't complete.

    Lab
    1. Add a persistent entry. Our device is a file, so the spec field is its path plus the loop option; for a real disk you would write UUID=… here instead (shown in the note below). nofail keeps a missing image from wedging a future boot. 
      echo '/tmp/disk.img /mnt/data ext4 loop,nofail 0 2' | sudo tee -a /etc/fstab
    2. Test it the safe way — mount -a mounts everything in fstab that isn't already mounted, exactly as boot will. No error means the line is good. 
      sudo mount -a
findmnt /mnt/data
      TARGET    SOURCE      FSTYPE OPTIONS

    3. Clean up so you don't leave a test mount behind. Unmount, then delete the line you added. 
      sudo umount /mnt/data

    Verify

    For a real second disk whose partition is /dev/sdb1, write the fstab line that mounts it at /data as ext4, robustly. (Get the UUID with sudo blkid /dev/sdb1, then: UUID=<that> /data ext4 defaults,nofail 0 2.) Then say why the UUID form is safer than /dev/sdb1 and what mount -a buys you. (The UUID follows the filesystem even if the kernel renames the device across boots; mount -a tests the whole file while you still have a shell, so a typo can't first reveal itself as a failed boot.)

    Gotchas

    A broken fstab line can stop a machine booting. A wrong device, a bad option, or a missing nofail on an absent disk can drop boot into an emergency shell — and on a headless VM or cloud instance that can mean no way in at all. The two defences are non-negotiable: run mount -a after every edit (it fails loudly, now), and put nofail on anything that isn't the root filesystem. Snapshot before experimenting on a machine you care about.

    Why UUID= can't identify our loop image: a UUID is resolved by scanning block devices, and a plain file isn't one until a loop device is attached to it. That is the whole reason the lab's fstab line uses the file path with the loop option instead of a UUID — the file-backed case is the exception. For any actual disk, use the UUID.

    The kubelet and CSI drivers do the equivalent of fstab in software: when a node reboots, the volumes a pod needs must be re-mounted, and the machinery tracks each PV by a stable identifier (much like a UUID) rather than a device name, precisely because the cloud may re-attach the same disk as a different /dev/ node. The nofail instinct maps too — a robust node keeps booting and running when an optional mount is absent, rather than falling over. You rarely edit fstab on a Kubernetes node for pod storage (the kubelet owns those mounts), but the node's own disks — the root filesystem, a separate /var/lib/containerd — are still plain fstab entries, and a botched one bricks the node the same way it bricks any Linux box.

    The re-mount-on-reboot behaviour underlies PV attach/detach in W5; the "node won't come back after reboot" failure is a W6 troubleshooting scenario.

    Docs

    man 5 fstab for the six fields and their meanings, man 8 mount under -a and the nofail/defaults options, and man 8 blkid for getting the UUID to put in field one. man 8 systemd.mount notes how modern systems translate fstab into mount units. No direct exam-docs analogue — this is node-level plumbing beneath the cluster.

    The VFS view

    B5.5

    The VFS: one interface over every filesystem

    Why it matters

    You have used cat on a file in your home directory, on a file under /proc that isn't really a file at all, and (in a moment) on a file in a RAM disk — and it just worked, identically, every time. That is not luck. It is the VFS, the single most important abstraction in this whole module, because it is the thing that lets the next lessons hand a process an entirely different filesystem without changing a single line of any program. Grasp the VFS and overlayfs stops being surprising; it is just one more filesystem plugged into the same socket.

    The concept

    The VFS (virtual filesystem) is a layer inside the kernel that defines one set of operations — open, read, write, stat, readdir, and a few more — and forces every real filesystem to implement them. ext4 implements them by reading blocks off a disk. tmpfs implements them by reading pages of RAM. procfs implements read by making up the answer on the spot from kernel data structures. overlayfs implements them by consulting a stack of other filesystems. A network filesystem implements them by sending packets to another machine. The programs above — cat, your editor, every process — call the same syscalls (B3) and never know or care which implementation is underneath.

    Think of it as a standard wall socket. Every appliance has the same plug; behind the wall the power might come from the grid, a generator, or solar panels, and the appliance neither knows nor needs to. The VFS is that socket for storage: one plug shape (open/read/write), many sources behind the wall. This is precisely why B6 can put a process into a completely different root filesystem and every ordinary program keeps working — they were never talking to ext4, they were always talking to the VFS.

    cat · your editor · any process→ same syscalls →VFS→ext4 · tmpfs · procfs · overlayfs · NFS

    Gotchas

    "Everything is a file" is a VFS statement, and it has limits. /proc/uptime looks like a file and reads like one, but it has no blocks on any disk — its read runs kernel code that produces the answer on the spot, which is why its size shows as 0 and why you cannot seek around it like a real file. The uniform interface is real, but some filesystems implement only part of it or implement it with surprises; treating a /proc or /sys file as if it were an ordinary ext4 file (copying it, checking its size, memory-mapping it) is a common source of confusion.

    Not every operation is supported by every filesystem. A read-only filesystem's write returns EROFS; procfs directories can't be renamed; some virtual files reject seek. The VFS defines the interface; each filesystem chooses which parts it honours, so "the interface is uniform" does not mean "every operation always succeeds".

    The VFS is the quiet reason a container works at all. When a container runtime gives a pod its own root filesystem (an overlay mount, coming up), every binary inside the container — none of which was compiled with any knowledge of overlayfs — keeps functioning, because they all speak the VFS interface and overlayfs sits behind it. The same uniformity is why Kubernetes can project a ConfigMap as files, back an emptyDir with either disk or tmpfs (RAM), and mount a network volume, all appearing to the application as ordinary files it reads with ordinary code. "It's just a file to the app" is the VFS keeping its promise.

    The RAM-vs-disk choice for an emptyDir (W5.1) is exactly "which VFS implementation backs this mount"; the ConfigMap-as-files projection is the same idea.

    Docs

    man 2 open, man 2 read, man 2 stat for the operations the VFS defines; man 5 filesystems and cat /proc/filesystems for the registered implementations. man 8 findmnt with -o FSTYPE shows which one backs each mount. man 5 proc for how procfs fakes files convincingly. No exam-docs analogue — this is kernel architecture the cluster stands on.

    B5.6

    The file is not its name: inodes and directory entries

    Why it matters

    Almost everyone carries a wrong mental model: that a file is its name and path, the way a variable is its label. Linux does not work that way, and the gap between the two explains a whole cluster of otherwise-baffling behaviours — why you can delete a file another process is still happily reading, why a file can have two names, why "renaming" is instantaneous no matter how large the file. The fix is one clean idea: the file and its name are separate things, joined by a number.

    The concept

    Every file is really an inode — an on-disk record holding all the file's actual properties: its type, owner and group, permission bits, size, timestamps, link count, and the pointers to the data blocks. The inode has a number but no name. The name lives somewhere else entirely: a directory is just a table of (name → inode number) pairs. So "the file /home/ubuntu/notes.txt" is really "walk to the directory /home/ubuntu, look up the name notes.txt in its table, get an inode number, go read that inode." The name is a label in a table; the file is the inode the label points at.

    That indirection is the point. It means a single inode can be reachable by more than one name (two directory entries pointing at the same number — a hard link, next lesson). It means "deleting a file" is really removing one (name → number) entry, and the inode itself only disappears when its link count — how many names point at it — reaches zero and no process still has it open. And it means renaming or moving within a filesystem just rewrites a table entry; the inode and its gigabytes of data never move.

    name in a directory→ maps toinode number→ points atinode: owner · mode · size · data blocks
    Lab
    1. See the number behind the name. ls -i prints each name's inode number; stat shows the same number plus everything the inode actually stores — note there is no filename anywhere in the inode's own fields. 
      echo hello > /tmp/a.txt
ls -i /tmp/a.txt
stat /tmp/a.txt
      266241 /tmp/a.txt
  File: /tmp/a.txt
  Size: 6   Blocks: 8   IO Block: 4096   regular file
Device: 803h/2051d   Inode: 266241   Links: 1
Access: (0664/-rw-rw-r--)  Uid: (1000/ubuntu)  Gid: (1000/ubuntu)
    2. Prove the name is separate from the file by renaming and watching the inode number stay identical — same file, new label. 
      mv /tmp/a.txt /tmp/b.txt
ls -i /tmp/b.txt
      266241 /tmp/b.txt              ← same inode: only the directory entry changed
    3. Look at a directory as a table. Every entry is a name paired with an inode number, including the two the filesystem always creates: . (this directory) and .. (its parent). 
      ls -ai /tmp | head
            2 .
      2 ..
266241 b.txt
...
    Verify

    You ran mv /tmp/a.txt /tmp/b.txt and the inode number did not change. Explain what mv actually did to the directory table. (It removed the entry a.txt → 266241 and added b.txt → 266241 — the inode and its data were never touched, which is why renaming a huge file is instant.) Then answer: if two names in a directory both showed inode 266241, what would be true about them? (They would be the same file — two labels for one inode — so editing "one" changes "both", because there is only one.)

    Gotchas

    Inode numbers are unique only within one filesystem, not globally. Two files on different filesystems can share an inode number and be completely unrelated — which is why stat reports a Device as well as an Inode, and why the true identity of a file is the (device, inode) pair. Tools that de-duplicate or detect hard links compare both; comparing inode numbers alone across mount points gives false matches.

    Deleting a name is not always deleting the file. rm removes a directory entry and decrements the link count; the data blocks are freed only when the count hits zero and no process holds the inode open. This is the mechanism behind the deleted-but-open puzzle in B5.8 — the name is gone, the inode isn't, because a file descriptor is still a reference to it.

    The inode-vs-name split is why atomic config updates work the way they do in Kubernetes. When the kubelet updates a projected ConfigMap or Secret volume, it does not edit the files in place — it builds a new directory of the content and swaps a symlink to point at it, so a reader either sees the entire old version or the entire new one, never a half-written file. That trick is only possible because a name is an independent pointer to an inode, exactly what this lesson establishes. The same reasoning explains why a container that had a file open keeps reading the old content after the ConfigMap changes (its descriptor references the old inode) until it re-opens by name.

    The projected-volume atomic-swap behaviour surfaces in W5.1; the "I updated the ConfigMap but the running process still sees the old value" gotcha is this inode reference in action.

    Docs

    man 7 inode for what an inode stores (and the file-type bits), man 1 stat for reading it (Inode, Links, Device), and man 1 ls under -i. man 2 rename confirms that renaming within a filesystem is a directory-entry operation. man 3 readdir shows a directory being read as the table of entries it is.

    B5.7

    Two kinds of link: hard links and symlinks

    Why it matters

    Now that you know a name and a file are different things, links follow naturally — and there are exactly two kinds, which people constantly confuse. A hard link is a second true name for the same inode; a symbolic link is a tiny separate file that merely holds a path to somewhere else. They behave differently the moment you delete or move the original, and knowing which is which is the difference between "my link still works" and "my link is now a dangling arrow to nothing".

    The concept

    Both let one file be reachable two ways, but the mechanism is opposite:

    • A hard link (ln target newname) adds a second (name → inode number) entry pointing at the same inode. There is no "original" and "copy" — both names are equal, first-class references to one file, and the inode's link count goes up by one. Delete either name and the file survives under the other; the data vanishes only when the last name is removed. Because it references an inode number, a hard link cannot cross filesystems (numbers are per-filesystem) and, by convention, cannot point at a directory.
    • A symbolic link (ln -s target newname) creates a brand-new, separate file — its own inode — whose entire contents are a text path to the target. It is a signpost. Follow it and you are redirected to whatever that path currently names. Delete or move the target and the symlink still exists but now points at nothing — a "dangling" link. Symlinks can cross filesystems and can point at directories, precisely because they store a path, not an inode number.

    The one-line test to keep them straight: a hard link shares the file's identity (same inode); a symlink shares only the file's address (a path), and addresses can go stale.

    Lab
    1. Make a hard link and watch the link count. Both names share one inode number, and the count in stat is now 2. 
      echo 'the original' > /tmp/orig
ln /tmp/orig /tmp/hard
ls -li /tmp/orig /tmp/hard
      266300 -rw-rw-r-- 2 ubuntu ubuntu 13 ... /tmp/orig     ← same inode 266300,
266300 -rw-rw-r-- 2 ubuntu ubuntu 13 ... /tmp/hard     ← link count 2
    2. Delete the "original" and prove the hard link kept the data alive — the inode had another name, so nothing was freed; the count drops to 1. 
      rm /tmp/orig
cat /tmp/hard
stat -c 'links=%h' /tmp/hard
      the original                        ← still here: the inode outlived the name
links=1
    3. Now a symlink, and break it. The symlink is its own tiny file (note the different inode and the l type); delete its target and it dangles. 
      echo 'target data' > /tmp/real
ln -s /tmp/real /tmp/link
ls -li /tmp/link; readlink /tmp/link
rm /tmp/real
cat /tmp/link
      266322 lrwxrwxrwx 1 ubuntu ubuntu 9 ... /tmp/link -> /tmp/real
/tmp/real
cat: /tmp/link: No such file or directory   ← dangling: the path no longer resolves
    Verify

    Predict before deleting anything: you have a file, a hard link to it, and a symlink to it. You rm the file's original name. Which of the other two still gives you the data, and why? (The hard link does — it is an equal name for the same inode, so removing one name leaves the inode alive. The symlink does not — it stored the now-gone path, so it dangles.) Then say why a hard link can't span two filesystems but a symlink can. (A hard link references an inode number, which is only meaningful within one filesystem; a symlink stores a path string, which is meaningful anywhere.)

    Gotchas

    A dangling symlink is not an error until you follow it. ls -l will happily show a symlink whose target no longer exists (often highlighted in red), and creating one is legal — you can make a symlink to a path that doesn't exist yet. The failure only appears at the moment something tries to resolve it, as ENOENT. So "the file exists (I can ls the link) but I can't read it" is very often a broken symlink; readlink -f shows where it actually points.

    Editors can quietly convert a hard link into a separate file. Some editors save by writing a new file and renaming it over the old name, which breaks the hard-link relationship — after such a save, your two "hard-linked" names are now two independent files. Hard links are best for stable data, not for files that get rewritten by tools that don't preserve them; when in doubt, stat the link count before and after.

    Symlinks are load-bearing in a container's filesystem and in Kubernetes volume projection. The atomic ConfigMap/Secret update from B5.6 works by swapping a symlink (..data) to point at a new timestamped directory — the whole "consistent view during an update" guarantee rides on symlink semantics. Container images also lean on symlinks heavily (for example, /lib pointing into /usr/lib), and a symlink that points outside the intended tree is a classic container-escape and image-scanning concern. Recognising a projected volume's ..data symlink for what it is demystifies the odd directory layout you'll see inside a mounted ConfigMap.

    You'll see that ..data symlink structure when you inspect a mounted ConfigMap in W3; the atomic-update behaviour is revisited in W5.

    Docs

    man 1 ln for creating both kinds (-s for symbolic), man 7 symlink for how symlink resolution works, and man 2 link/man 2 symlink for the syscalls. man 1 readlink (-f to resolve fully) and man 1 stat (link count via %h) for inspecting them.

    B5.8

    The disk is full and nothing is big: df vs du, open-but-deleted, inodes

    Why it matters

    Three of the most common "how is this even possible" disk incidents all come from the same root: the filesystem's own accounting and the sum of the files you can see are two different numbers, and they drift apart for specific, knowable reasons. A disk reports 100% full but du finds only a few megabytes. A file you deleted didn't free any space. A filesystem refuses to create a file with gigabytes free. Each looks like a bug and each has a five-second explanation once you know where to look — and this is the exact toolkit the B5.18 drill and real W6 pages lean on.

    The concept

    Two commands answer "how much space?" from opposite directions:

    • df asks the filesystem: from the superblock's own bookkeeping, how many blocks (and, with -i, how many inodes) are used and free? This is the authoritative "how full am I" number.
    • du walks the directory tree and adds up the sizes of the files it can reach by name. This is "how much do the visible files account for".

    When these two disagree, the gap is the whole diagnosis. Three causes cover almost every case:

    1. A deleted-but-still-open file. A process opened a file, then it was deleted (its name removed) while the process still holds the descriptor. From B5.6: the name is gone so du can't see it, but the inode and its data blocks survive until the process closes the fd — so df still counts the space. Result: df says full, du says empty. The holder is visible in /proc/PID/fd as a link marked (deleted).
    2. Inode exhaustion. The fixed inode count from mkfs (B5.2) ran out — millions of tiny files each took one inode — so the filesystem can't create another file even though block space is free. df -i shows 100% inodes used while df shows blocks free.
    3. Space hidden under a mount point. Files written to a directory before something was mounted over it (B5.3) still occupy the underlying filesystem but are shadowed, so du of the mounted path can't see them.
    Lab
    1. Reproduce the deleted-but-open file. Start a process writing to a file, then delete the file's name while that process keeps it open — the space stays used with no visible file to blame. 
      ( tail -f /dev/null > /tmp/ghost.log & echo $! ) && sleep 1
# grow it, then delete the NAME while the writer holds it open:
dd if=/dev/zero of=/tmp/ghost.log bs=1M count=20 conv=notrunc 2>/dev/null
rm /tmp/ghost.log
ls /tmp/ghost.log
      1234                                    ← the writer's PID
ls: cannot access '/tmp/ghost.log': No such file or directory
    2. Find the ghost. The name is gone but a process still holds the inode; /proc/PID/fd shows the descriptor pointing at the deleted file, which is why the space isn't back. 
      ls -l /proc/1234/fd | grep deleted
      l-wx------ ... 1 -> /tmp/ghost.log (deleted)   ← space freed only when this closes
    3. See inode accounting directly. df -i reports inodes, a different budget from blocks — this is the number that hits zero in the exhaustion case even when df shows free space. (Kill the writer to release the ghost.) 
      df -i /tmp
kill 1234
      Filesystem  Inodes  IUsed  IFree IUse% Mounted on
/dev/sda1   322400   90312 232088   29% /       ← inodes, counted separately from bytes
    Verify

    df said space was used but ls found no file. Explain the discrepancy in terms of inodes and link counts. (The file's name was removed, so du/ls can't see it, but a process still holds the inode open — its link count is 0 but its open count is not — so the inode and its blocks persist, and df, which reads the filesystem's own accounting, still counts them.) Then say which command you'd reach for first when a disk is "full" but files look small, and what it would tell you. (df -i — if inodes are exhausted while blocks are free, the problem is millions of tiny files, not big ones.)

    Gotchas

    Restarting the process is the fix for a deleted-but-open file, and deleting more files is not. Since the space is held by an open descriptor, no amount of hunting for large files helps — you must make the holder close the fd, usually by restarting it (or, in a pinch, truncating through /proc/PID/fd/N). The reflex when df and du disagree by a lot: look for (deleted) in /proc/*/fd (or run lsof +L1) before anything else.

    Inode exhaustion is invisible to plain df. The default df shows only block usage, so a filesystem at 100% inodes but 40% blocks looks half-empty right up until every touch and write fails with No space left on device — the same ENOSPC error as a truly full disk, for a completely different reason. Always check df -i when ENOSPC appears on a disk that df says has room.

    Both failures are routine node incidents in Kubernetes. A container that logs heavily and has its log file rotated (renamed and deleted) while the process holds it open produces the classic "the node disk is full but I can't find what's using it" page — it is the deleted-but-open file, one layer down, and the fix is restarting the offending pod. Inode exhaustion hits nodes that run many short-lived pods or write millions of small files, and it triggers the kubelet's disk-pressure eviction with plenty of bytes free — mystifying until you run df -i on the node. The df/du/df -i/lsof reflexes transfer directly to debugging a node.

    Disk-pressure eviction and "node disk full" are W6 troubleshooting scenarios; the ephemeral-storage limits that bound a pod's disk use are in W5.

    Docs

    man 1 df (-h for readable, -i for inodes) and man 1 du (-sh, -x to stay on one filesystem) for the two views, man 8 lsof (+L1 finds open-but-unlinked files) for the holder, and man 5 proc under /proc/[pid]/fd. Re-read B5.6 — this whole lesson is the inode/link-count model made operational.

    B5.9

    Memory as a filesystem: tmpfs

    Why it matters

    Not every filesystem lives on a disk. A tmpfs is a filesystem whose storage is RAM — you mount it, write files to it, read them back with ordinary commands, and every byte is sitting in memory, gone forever on reboot. Your VM is already running several without you knowing. Understanding tmpfs is the difference between "an emptyDir with medium: Memory" being a mysterious incantation and being an obvious "put this mount in RAM", and it is your first concrete example of the VFS letting a non-disk thing be a perfectly ordinary filesystem.

    The concept

    A tmpfs ("temporary filesystem") is a VFS implementation backed by the kernel's page cache — the same memory used for caching disk data — rather than by a block device. Files you create in it occupy RAM. The consequences flow from that one fact:

    • Fast — no disk is involved, so reads and writes run at memory speed.
    • Ephemeral — RAM doesn't survive a reboot, so a tmpfs is empty every boot; it is scratch space, never storage.
    • Counts against memory — a file in tmpfs consumes RAM exactly as a running program would, so filling a tmpfs can exhaust memory and trigger the OOM killer (B2). The size= option caps how large it may grow; it can also spill to swap if the system has any.

    You already rely on several. /run (runtime state that should not persist), /dev/shm (shared-memory scratch for programs), and parts of /sys are tmpfs — which is exactly why /run is empty and rebuilt on every boot. It is the standard Linux answer to "I need a fast, disposable place to put files".

    Lab
    1. Mount a 64 MiB RAM filesystem and use it like any other. The size= option is the ceiling; write a file and it is now living in memory. 
      sudo mkdir -p /mnt/scratch
sudo mount -t tmpfs -o size=64M tmpfs /mnt/scratch
echo 'this lives in RAM' | sudo tee /mnt/scratch/note >/dev/null
findmnt /mnt/scratch
      TARGET       SOURCE FSTYPE OPTIONS
/mnt/scratch tmpfs  tmpfs  rw,relatime,size=65536k
    2. See the ones you already have. Every mount here is RAM-backed and was set up by the system at boot — none of it survives a reboot. 
      findmnt -t tmpfs -o TARGET,SIZE,USED
      TARGET               SIZE  USED
/run                 393M  1.2M
/dev/shm             994M     0
/run/lock            5.0M     0
/mnt/scratch          64M    17          ← the one you just made
    3. Confirm the cap bites. Try to write more than the size allows and it fails with "No space left" — the same ENOSPC as a full disk, but here it means "the memory budget for this mount is used up". Then unmount to release the RAM. 
      dd if=/dev/zero of=/mnt/scratch/big bs=1M count=100 2>&1 | tail -1
sudo umount /mnt/scratch
      dd: error writing '/mnt/scratch/big': No space left on device
    Verify

    Explain why a file in /mnt/scratch disappears on reboot but a file in /mnt/data (the ext4 image) does not. (tmpfs is backed by RAM, which is cleared on reboot; the ext4 filesystem is backed by a block device whose contents persist.) Then answer: your tmpfs is capped at 64 MiB — what runs out when you exceed it, and what error do you get? (Memory allocated to that mount; you get ENOSPC / "No space left on device", the same error a full disk gives, because from the filesystem's view the space budget is exhausted.)

    Gotchas

    tmpfs space is memory, and forgetting that is dangerous. A tmpfs with no size= defaults to half of RAM, and files written to it are not "free scratch" — they compete with running processes for physical memory and can drive the machine into swapping or OOM-killing. Filling a tmpfs is filling RAM. Always set an explicit size= on a tmpfs you create, sized as a fraction of available memory, not of disk.

    Don't confuse /tmp with tmpfs. On many systems /tmp is an ordinary directory on the root disk, not a tmpfs — so files there do survive a reboot (until something cleans them) and don't consume RAM. Whether /tmp is a tmpfs is a per-distro choice; check with findmnt /tmp rather than assuming. The name "temporary filesystem" refers to the RAM-backed mechanism, not to the /tmp directory.

    This is exactly emptyDir with medium: Memory in a pod spec: the volume becomes a tmpfs, so it is fast and wiped when the pod dies, and — the critical operational fact — every byte written counts against the container's memory limit and can OOM-kill the pod. A Secret mounted into a pod is also backed by tmpfs by default, precisely so the secret material never touches the node's disk. When you reason about "why did my pod get OOM-killed when its process was using little memory", a runaway medium: Memory emptyDir is a prime suspect, and it is this lesson's RAM-is-the-storage fact at work.

    You'll set emptyDir: {medium: Memory} and see the memory-limit interaction in W5.1; the tmpfs-backed Secret mount is a recurring detail in W3.

    Docs

    man 5 tmpfs for the filesystem and its options (size=, nr_inodes=, mode=), man 8 mount under -t tmpfs, and man 8 findmnt with -t tmpfs to list them. free -h and man 5 proc (/proc/meminfo) show tmpfs usage counting as memory. In the exam docs, emptyDir with medium: Memory.

    B5.10

    A file that acts like a disk: loop devices

    Why it matters

    You have already used this trick twice — mount -o loop on the image file — without seeing the machinery. A loop device is the mechanism that lets an ordinary file be a block device: it wraps /tmp/disk.img in a /dev/loopN handle that behaves, to the whole rest of the system, exactly like a disk. This is how you experiment with filesystems without a spare drive, how a downloaded .iso or cloud image gets inspected, and — one level up — a piece of how container and snapshot storage is built.

    The concept

    Recall from B5.1 that a block device is just a handle (major/minor) to a driver that serves fixed-size blocks. A loop device is a driver that serves those blocks out of a regular file: block N of /dev/loop0 is simply the bytes at offset N×blocksize in the backing file. Once losetup associates a file with a /dev/loopN, everything that works on a real disk works on the file through that device — mkfs, mount, fdisk, lsblk — because none of them can tell the difference; they only speak the block-device interface, and the loop driver answers it.

    mount -o loop file dir is a convenience that does two steps at once: it finds a free loop device, attaches the file to it with losetup, and mounts that loop device — then detaches it automatically on unmount. Doing the steps by hand (as below) is what makes the "a file pretending to be a disk" idea concrete, and it's what you need when you want to partition an image or inspect it without mounting.

    /tmp/disk.img (a file)→ losetup →/dev/loop0 (a block device)→ mount/mkfs treat it asa real disk
    Lab
    1. Attach the image to a loop device by hand. -f finds a free one, --show prints which it chose, -P would scan for partitions inside. Now lsblk lists it like any disk. 
      LOOP=$(sudo losetup -f --show /tmp/disk.img)
echo "attached as $LOOP"
lsblk "$LOOP"
      attached as /dev/loop0
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
loop0    7:0    0  128M  0 loop
    2. Mount the loop device the ordinary way — no -o loop now, because it already is a block device. This is exactly what mount -o loop was doing for you all along. 
      sudo mount "$LOOP" /mnt/data

      TARGET    SOURCE      FSTYPE OPTIONS
/mnt/data /dev/loop0  ext4   rw,relatime
/dev/loop0: [2051]:266xxx (/tmp/disk.img)   ← loop → backing file
    3. Tear it down in the right order: unmount first, then detach the loop device to free it. 
      sudo umount /mnt/data
sudo losetup -d "$LOOP"
losetup -a
                                              ← empty: the loop device is released
    Verify

    Explain what mount -o loop /tmp/disk.img /mnt/data does under the hood, in terms of the two steps you just did by hand. (It picks a free loop device, attaches the file to it with the equivalent of losetup, then mounts that loop device — and reverses both on unmount.) Then answer: why can mkfs, lsblk, and mount all operate on a loop device without any special "this is really a file" awareness? (Because the loop driver presents the same block-device interface as a real disk; those tools only ever speak that interface, so the backing being a file is invisible to them — the VFS/block-layer abstraction again.)

    Gotchas

    Loop devices leak if you forget to detach them. Every losetup -f without a matching losetup -d leaves a loop device tied up and the backing file pinned open — run enough of them and you exhaust the loop devices or can't delete the image. losetup -a lists what's attached; detach with losetup -d /dev/loopN. The mount -o loop form avoids this by auto-detaching on unmount, which is one reason to prefer it unless you specifically need the raw device.

    Order matters on teardown: unmount before detaching. Detaching a loop device that is still mounted (or forcing it) can leave the filesystem in an inconsistent state and the mount pointing at nothing. Always umount first, then losetup -d — the same "release users before removing the thing" discipline as the busy-unmount gotcha in B5.3.

    Loop devices show up under container and Kubernetes storage in a few places. Some storage drivers and CSI plugins back a volume with a filesystem image and expose it via a loop device; disk-image-based tools and certain snapshot mechanisms do the same. More practically, when you need to inspect a raw volume image, a VM disk, or a container export on a node, losetup + mount is how you open it read-only and look inside without a spare disk. The mental model — "a file the kernel is presenting as a block device" — is also the cleanest way to understand how a single node hosts dozens of independent "disks" for pods without dozens of physical drives.

    This is background for the block-and-file-backed volume mechanics you'll meet in W5; there's no direct exam task, but it's the plumbing under image-backed storage.

    Docs

    man 8 losetup for attaching/detaching and listing (-f, --show, -P, -a, -d), man 8 mount under -o loop for the convenience form, and man 4 loop for the driver itself. man 8 lsblk shows loop devices with type loop.

    B5.11

    One directory in two places: bind mounts

    Why it matters

    So far every mount has attached a device to a directory. A bind mount does something different and, once it clicks, surprisingly powerful: it attaches a directory that already exists to a second path, so the same files are reachable through two different locations — no copy, no symlink, the very same inodes seen twice. This is the mechanism a container runtime uses to inject a host directory into a container, and it is the honest answer to "how is that config file both on the host and inside the pod".

    The concept

    A bind mount (mount --bind src dst) grafts the subtree at src onto dst. After it, walking into dst lands you in the exact same inodes as src — write a file through one path and it is instantly there through the other, because there is only one set of files. It is not a copy (no duplicate data, no drift) and not a symlink (no separate signpost file that can dangle) — it is the same directory subtree with a second entry in the mount table. This is fundamentally different from the earlier mounts: nothing new is being made available, an existing part of the tree is simply being made reachable at another point.

    One sharp edge that trips everyone: a read-only bind is not read-only on the first try. mount --bind -o ro src dst quietly ignores the ro and gives you a writable bind, because the read-only flag applies to the underlying mount, not to the bind operation itself. To actually get a read-only view you must do it in two steps — bind first, then remount with ro: mount -o remount,ro,bind dst. Knowing this saves you from a bind mount you believed was locked but wasn't.

    Lab
    1. Bind a directory to a second path and prove it's the same files. Create src with a file, bind it onto dst, and note both paths show the identical inode — one file, two doors. 
      mkdir -p /tmp/src /tmp/dst
echo 'one copy, two paths' > /tmp/src/data
sudo mount --bind /tmp/src /tmp/dst
ls -i /tmp/src/data /tmp/dst/data
echo 'written via dst' >> /tmp/dst/data
cat /tmp/src/data
      266401 /tmp/src/data
266401 /tmp/dst/data                    ← same inode: not a copy

    Gotchas

    A bind mount reflects the source at bind time, not a frozen snapshot — but it does not follow the source if the source itself is later replaced by a new mount, unless propagation (next lesson) is set up to carry it. For plain data this is intuitive (same files, live), but for mount points it is the root of real confusion: bind-mounting a directory that later has something mounted inside it may or may not show that new mount through the bind, depending entirely on propagation. Keep "same files, live" for the simple case and defer the mount-inside-a-bind case to B5.12.

    The read-only-bind trap has bitten production systems. People bind-mount a directory into a sandbox with -o ro, believe it is protected, and only discover the mount was writable when something inside modifies host files. If you rely on a read-only bind for isolation, verify it with an actual write attempt (as the lab does) or check for ro in findmnt — don't trust that the flag on --bind took effect.

    Bind mounts are how a hostPath volume works: the kubelet bind-mounts a directory from the node into the container's filesystem, so the pod sees the node's files at the mount path — the same inodes, reachable inside the container, exactly as in the lab. It is also the mechanism behind mounting the container runtime's socket, host logs, or device directories into a pod. And the read-only surprise matters directly: when you mark a hostPath or volume mount readOnly: true, the container runtime is responsible for doing the remount step you just learned — a bug there (or a missing remount) is how a "read-only" host mount turns out to be writable, a real container-security finding.

    You'll use hostPath and readOnly mounts in W5.1, and the security implications of writable host mounts recur in the RBAC/hardening material of W7.

    B5.12

    Mount propagation: does a mount cross over?

    Why it matters

    This is the subtlest idea in the module, and it is the one that separates "I can use mounts" from "I understand why container mounts behave the way they do". Mount propagation answers a single question: if you mount something underneath a mount point, does that new mount also appear at its bind-mounted peers? The answer is configurable — private, shared, or slave — and the choice is exactly what lets a container see the host's mounts without the container's own throwaway mounts leaking back onto the host. Get this and the container runtime's mount setup stops looking like black magic.

    The concept

    When two paths are related by a bind mount, the kernel has to decide whether a later mount made under one of them shows up under the other. That policy is the mount's propagation type:

    • private — no crossing. A mount made under one side is invisible to the other. Each is sealed.
    • shared — two-way. The two paths form a peer group; a mount made under either appears under both. Changes propagate in both directions.
    • slave (and rslave, its recursive form) — one-way. The slave receives mounts from its master but does not send its own back. Mounts flow master→slave only.

    The one-way rslave case is the whole reason this matters, because it is what a container runtime wants. The runtime binds the host's filesystem into the container as a slave: if the host mounts a new disk or a network share, it appears inside the container (master→slave, useful) — but if the container mounts something in its own throwaway way, that mount does not propagate back and pollute the host (slave doesn't send). Private would seal the container off from useful host mounts; shared would let container mounts escape onto the host, a real hazard. rslave is the deliberate middle: the host can reach in, the container cannot reach out.

    host mounts a disk→ (rslave) →appears in container· container mounts something →stays in container
    Lab
    1. Set up a shared peer group and prove two-way propagation. Bind a directory onto itself to make it a mount point, mark it shared, then bind it to a peer — now a mount under one appears under the other. 
      sudo mkdir -p /mnt/a /mnt/b
sudo mount --bind /mnt/a /mnt/a          # make /mnt/a a mount point
sudo mount --make-shared /mnt/a
sudo mount --bind /mnt/a /mnt/b          # /mnt/b joins the peer group

    2. Now switch the same setup to private and watch propagation stop. Tear down, rebuild as private, and the mount under a does not appear under b. 
      sudo umount /mnt/a/sub; sudo umount /mnt/b; sudo umount /mnt/a

    In the shared run the tmpfs under /mnt/a appeared under /mnt/b; in the private run it did not. Explain the difference in one sentence. (Shared makes the two a peer group so mounts propagate both ways; private gives each its own sealed propagation, so a mount under one is invisible to the other.) Then answer the design question: why does a container runtime use rslave for the host filesystem rather than shared or private? (rslave is one-way — the container receives host mounts (so new host disks/shares appear inside) but the container's own mounts don't propagate back onto the host; shared would let container mounts escape to the host, and private would hide useful host mounts from the container.)

    Gotchas

    Propagation is a property of a mount point, and the default has changed under people's feet. systemd makes most mounts shared by default on a modern system, which is why the lab has to explicitly bind-and---make-shared to get predictable behaviour and why a naive experiment can give surprising results — a mount you make in one place suddenly appearing elsewhere because the tree was shared. When propagation matters, set it explicitly (--make-private/--make-shared/--make-rslave) rather than relying on the inherited default.

    This is the source of a notorious class of container bug: a mount made inside a container "leaking" onto the host, or a host mount failing to appear inside a container, almost always traces to the propagation type on the relevant mount point. When a bind mount's contents seem wrong across a container boundary — new mounts present or absent unexpectedly — check propagation with findmnt -o TARGET,PROPAGATION before suspecting anything more exotic.

    Mount propagation is a field you can actually set in a pod: volumeMounts[].mountPropagation takes None (private), HostToContainer (rslave — the container sees host mounts made under the volume), or Bidirectional (rshared — mounts made in the container propagate back to the host). Bidirectional is powerful and dangerous, restricted to privileged containers, and is exactly how components like a CSI node driver deliberately push mounts out to the node so other pods can see them. Every "the CSI driver mounted a volume but the app pod can't see it", or "my sidecar's mount isn't visible to the main container", is a mount-propagation question — this lesson is the model for all of them.

    You'll encounter mountPropagation and its privileged Bidirectional mode in the storage and security material of W5 and W7; CSI node drivers rely on it directly.

    Overlayfs

    B5.13

    Overlayfs by hand: lowerdir, upperdir, workdir, merged

    Why it matters

    This is the lesson the whole module has been climbing toward. Overlayfs stacks several directories so they appear as one, with the lower ones read-only and a single upper one absorbing all writes. Mount one by hand, with your own directories, and the phrase "container image layer" loses every ounce of mystery — because a container image is precisely this stack, and you are about to build one. Everything from here (copy-up, whiteouts, "why does my data vanish on restart") is just consequences of the four directories you're about to name.

    The concept

    overlayfs is a VFS filesystem (B5.5) that presents a union of other directories. Mounting one takes four roles:

    • lowerdir — one or more read-only directories, stacked. Written as a colon-separated list, leftmost is highest priority. These are never modified.
    • upperdir — the single writable directory. Every change — new files, edits, deletions — is recorded here, never in the lowers.
    • workdir — an empty scratch directory on the same filesystem as upperdir, which the kernel needs for internal atomic operations (like preparing a copy-up). You never touch it; it just has to exist and be empty.
    • merged — the mount point: the combined view you actually read and write through.

    The merge rule is simple: to find a name, overlayfs looks in upperdir first, then each lowerdir left to right, and the first hit wins. So a file present in two layers shows the upper one (higher layers shadow lower), and the merged view is the union of every name across all layers. Reads come from wherever the file lives; writes all land in upperdir (the next two lessons are exactly how writes to lower-layer files are handled). Think of the lowers as a stack of transparent sheets with writing on them, and upperdir as the top sheet you're allowed to draw on: you read down through the whole stack, but your pen only marks the top.

    upperdir (writable)overlower1overlower2overlower3=merged (the union)
    Lab

      cd /tmp && rm -rf ov && mkdir -p ov/{low1,low2,low3,upper,work,merged}

    Name the four directories in an overlay mount and say, in a few words, what each is for. (lowerdir — read-only stacked layers; upperdir — the one writable layer where all changes go; workdir — empty kernel scratch on the upper's filesystem; merged — the combined view you use.)     Gotchas

    workdir and upperdir must live on the same filesystem, and workdir must be empty and unshared. The kernel uses workdir to stage operations atomically (a copy-up is prepared there, then moved into upperdir), which only works if both are on one filesystem — try to put upperdir on ext4 and workdir on tmpfs and the mount fails. Give each overlay its own workdir; reusing one across two overlays corrupts them. These aren't arbitrary rules — they're what makes overlay writes crash-safe.

    The lowerdir order is the opposite of what many expect. Leftmost is highest priority (closest to the top), so in lowerdir=A:B:C, A shadows B shadows C. When container images are stacked, the topmost image layer is the leftmost lowerdir. Get this backwards and you'll predict the wrong file winning; the merged view always shows the leftmost occurrence of a name.

    This is, almost exactly, how a container's root filesystem is assembled. A container image is a series of layers; the runtime uses each image layer as a lowerdir (topmost image layer leftmost), adds a fresh empty upperdir and workdir for the container, and mounts the result as the container's / — the merged view. When you run mount | grep overlay on a Kubernetes node running containers, you see exactly the mount you just built by hand, with a long colon-separated lowerdir of image-layer paths under /var/lib/containerd (or /var/lib/docker). Every container on the node shares those read-only lower layers (one copy of the image, many containers) and each gets its own tiny upper — which is the efficiency that makes containers cheap and the fact behind the next three lessons.

    This directly underlies the image-layer material of the Intermediate track and the kubelet/runtime storage under W1.23; the shared-lowers, private-upper design is why images pull once and containers start fast.

    B5.14

    Copy-up: what happens when you write to a lower-layer file

    Why it matters

    The lowers are read-only, yet through the merged view you can happily edit a file that only exists in a lower layer. Something has to give, and what gives is copy-up: the moment you write, overlayfs copies the whole file up into the writable upper layer and edits the copy, leaving the lower untouched. This single mechanism explains a surprising performance cliff — editing one byte of a large file in a container can be slow and can consume real disk — and it is the direct reason a container's writable layer grows the way it does.

    The concept

    When you open a lower-layer file for writing through the merged view, overlayfs cannot modify the lower (it's read-only), so it performs a copy-up: it copies the entire file — all its data and metadata — from the lowerdir into the upperdir, then applies your write to that upper copy. From then on the merged view shows the upper copy (upper shadows lower, per B5.13's merge rule), and the lower original sits there unchanged, still shared with every other overlay using that layer.

    Two facts fall straight out and both bite in practice. First, the cost is paid on first write and is proportional to the file's size, not your edit's: appending one line to a 2 GB log triggers a 2 GB copy the first time, because the whole file must be materialised in the upper layer before it can be modified. Second, the upper layer grows by the full size of every lower file you touch — a container that edits large files from its image inflates its writable layer accordingly, even if each edit is tiny. Copy-up is why "just change one config value" inside a big image file is not always free.

    Gotchas

    Copy-up happens on the first write-open, and even some metadata changes can trigger it. Opening a lower file for writing, changing its permissions or owner, or truncating it can all cause a copy-up — it is not only content edits. This is why a container image built to be "read-mostly" can still balloon its writable layer if it chmods or touches many files from lower layers at startup. If a container's upper layer is growing unexpectedly, look for processes that open-for-write or modify metadata on large image files.

    Copy-up breaks the sharing that makes overlays cheap. Before the write, one lower file was shared by every container using that image; after copy-up, the writing container has its own full private copy in its upper layer, consuming that much extra disk. The design assumes writes are rare and localised; a workload that rewrites large chunks of its image is fighting the model and should use a volume (B5.16) for that data instead of the container's writable layer.

    Copy-up is behind two real Kubernetes behaviours. First, the "container writable layer" that counts against a pod's ephemeral-storage and can trigger eviction grows exactly by these copy-ups plus genuinely new files — a pod that rewrites large files from its image can be evicted for ephemeral-storage pressure even though the image itself is modest. Second, it's a core argument for mounting a volume over any path a container writes heavily: a volume is a separate mount (not part of the overlay), so writes to it skip copy-up entirely and don't inflate the throwaway upper layer. "Write your data to a mounted volume, not into the container filesystem" is copy-up avoidance stated as a best practice.

    Ephemeral-storage limits and eviction appear in W5.1 and W6; the volume-over-writable-path pattern is the practical takeaway there.

    B5.15

    Whiteouts: deleting a file that lives in a read-only layer

    Why it matters

    Copy-up handled editing a lower-layer file. But what about deleting one? You can't remove it from the read-only lower, yet rm through the merged view clearly has to make it disappear. The trick is a whiteout: overlayfs drops a special marker in the upper layer that says "hide the thing below". It's a small, elegant hack, and seeing it directly — a phantom device file standing in for a deletion — is the last piece you need before the "an image is just layers" conclusion clicks fully into place.

    The concept

    Deleting a lower-layer file can't actually remove it (the lower is read-only and shared with other overlays), so overlayfs instead records the deletion in the upper layer as a whiteout: a character device with device number 0/0, created in upperdir at the deleted file's name. When overlayfs resolves that name and finds a 0/0 character device in the upper layer, it treats the name as deleted and stops looking down the stack — so the lower file, though still physically present, vanishes from the merged view. The lower copy is completely untouched; only the merged view is affected, and only because the whiteout masks it.

    This is the deletion counterpart to copy-up's edit: writes to a lower file copy it up, deletes of a lower file white it out, and in both cases the read-only lower is preserved and the change is recorded entirely in the writable upper. A whole-directory deletion uses a related marker (an "opaque" directory) with the same idea — record in upper that the lower should be hidden. The upshot: the upper layer holds not just new and edited files but also a set of "tombstones" for everything the container deleted from its image.

(Both preserve the read-only lower and record the change only in the writable upper — copy-up for edits, whiteout for deletions.)

    Gotchas

    A whiteout does not reclaim the lower file's space, and that surprises people auditing disk use. The lower layer still holds the "deleted" file in full; the whiteout is a tiny marker. So deleting files inside a container to free space in the image's layers accomplishes nothing — the bytes are still in the read-only lowers, shared and immutable. This is the container analogue of the Docker-image lesson "you can't shrink a layer by deleting files in a later layer"; the deletion is recorded, not applied downward.

    The 0/0 character-device representation is the classic default, but not the only one. With certain mount options (e.g. userxattr, used for rootless/unprivileged overlays) whiteouts are recorded as extended attributes instead of device files, because an unprivileged process can't create device nodes. On the standard privileged overlay over ext4 you'll see the 0/0 device this lab shows; if you ever find a deletion that didn't leave a visible whiteout device, suspect an xattr-based one and check with getfattr.

    Whiteouts explain a persistent source of container-image confusion and a real security nuance. Deleting a secret file in a later image layer does not remove it from the image — the whiteout hides it from the running container's view, but the file is still present in the lower layer and recoverable by anyone who can read the image layers, which is why "we deleted the credentials in a cleanup layer" is not a real remediation (you must avoid adding them to a layer in the first place). At runtime, the whiteouts and copy-ups together are the entire content of a container's writable layer, and understanding that lets you reason precisely about what a container has changed relative to its image — the basis of tools that diff a running container against its image.

    The "can't delete secrets from a lower layer" fact underlies image-hardening guidance in the Intermediate track and the supply-chain/secret-handling concerns touched in W7.

    B5.16

    The conclusion: an image is layers, a container is layers plus a thin writable top

    Why it matters

    Everything in this module converges here, into a sentence you can now defend from first principles rather than accept on faith: a container image is a stack of read-only lowerdirs, and a running container is that stack plus one thin writable upperdir that is discarded when the container exits. That single fact explains — with no hand-waving — why data written inside a container disappears on restart, why a volume fixes it, why many containers from one image are cheap, and why deleting a file from an image doesn't shrink it. If you can derive those four from the sentence, you understand container storage.

    The concept

    Assemble what you've built. An image is a series of layers, each just a directory of files — the base OS, then a layer that added a runtime, then one that added your app. The container runtime stacks them as overlayfs lowerdirs (topmost layer leftmost, B5.13), all read-only and shared across every container made from that image. To run a container it adds one fresh, empty upperdir (and workdir), mounts the merged view as the container's /, and starts the process. Reads come from the shared lowers; every write — new files, copy-ups (B5.14), whiteouts (B5.15) — lands in that private upper. When the container is removed, its upperdir is deleted. That's the entire storage model. Four consequences follow with no extra machinery:

    • Data written in a container vanishes on restart — because "restart" often means a new container with a new, empty upperdir; the old upper (with all your writes) was thrown away.
    • A volume fixes it — a volume is a separate mount grafted over a path (B5.3), outside the overlay entirely, so its data lives on its own persistent filesystem and is untouched by the upper layer being discarded.
    • Many containers from one image are cheap — they all share the same read-only lowerdirs; only the tiny per-container upper is duplicated.
    • Deleting a file from an image doesn't shrink it — a deletion is a whiteout in an upper layer, and the lower bytes remain (B5.15).

    Gotchas

    "Restart" is ambiguous, and the difference decides whether data survives. A process restarting inside the same container keeps the same upperdir, so its writes persist; a container being recreated (a new container object, as happens when a pod is rescheduled or its image changes) gets a brand-new empty upperdir, so its writes are gone. This is exactly why "my data survived a crash but vanished after I redeployed" is not a contradiction — the first kept the upper, the second replaced it. When reasoning about container data, always ask "same container, or new one?"

    The writable layer is not durable storage, even while it exists. It's on the node's local disk, it counts against ephemeral-storage limits, it's lost if the container moves to another node, and it offers none of the guarantees (replication, snapshots, backup) real storage does. Treating the container filesystem as a place to keep anything you care about is the single most common storage mistake; the upper layer is scratch space with the lifetime of one container instance, and anything valuable belongs on a volume.

    This is the Kubernetes storage story, stated at the layer beneath the API. A pod's container filesystem is the overlay you built; its writes go to a throwaway upper that dies with the container, which is why every stateful workload needs a volume (an emptyDir for pod-lifetime scratch, a PersistentVolume for real durability) mounted over the paths it writes. It's why a rescheduled pod starts with a clean filesystem, why kubectl exec changes don't survive a pod restart, and why "just write it to a file in the container" is the classic beginner bug that loses data on the first reschedule. Every volume type in W5 exists to put data outside this disposable upper layer.

    You'll choose among emptyDir, hostPath, and PersistentVolume for exactly this reason in W5.1 onward; the ephemeral-vs-persistent distinction there is this lesson's upper-vs-volume distinction in cluster terms.

For the cluster picture, the exam docs on pod volumes, emptyDir, and PersistentVolumes are where this reasoning is applied. Re-read B5.13–B5.15 as one arc — this lesson is their sum.

    Project & drill

    B5.17

    Mini project: build an "image" by hand

    Why it matters

    You've mounted an overlay, triggered a copy-up, and found a whiteout in separate lessons. This project puts all three into one script you keep — three stacked lowerdirs under a single upper, mounted merged, then edited and deleted and reset so you watch every mechanism fire in sequence. When you're done you'll have built, by hand, the exact structure a container runtime builds for every container, and a script you'll reuse: B6 boots a shell inside this rootfs with chroot/pivot_root, and B14's from-scratch container builds on both.

    The concept

    Nothing new — this is B5.13 through B5.16 executed end to end. Three read-only lowerdirs play the role of image layers; one upperdir is the container's writable layer; merged is the container's root filesystem. You'll (1) confirm the union and the shadowing rule, (2) modify a file that exists only in the bottom layer and locate the full copy that appears in upper — copy-up, (3) delete a file from a lower layer and locate the 0/0 whiteout that hides it, and (4) empty the upper directory and remount, watching every original reappear untouched — the proof that the lowers were never modified and the writable layer is disposable. Keep the script; it is the seed of a container.

    Lab

    Snapshot first (B0.9). Save this as ~/build-image.sh — you'll extend it in B6 and B14.

    1. Write the builder. It lays out three layers, an upper and work dir, and mounts the merged view. 
      cat > ~/build-image.sh <<'SH'
#!/usr/bin/env bash
set -euo pipefail
ROOT=${1:-/tmp/handimage}
sudo umount "$ROOT/merged" 2>/dev/null || true
rm -rf "$ROOT"; mkdir -p "$ROOT"/{low1,low2,low3,upper,work,merged}
# three "image layers": base, runtime, app
echo 'base OS file'      > "$ROOT/low3/os-release"
echo 'runtime v1'        > "$ROOT/low2/runtime"
echo 'from the app layer'> "$ROOT/low1/app.conf"
echo 'BASE greeting'     > "$ROOT/low3/greeting"   # same name in low1+low3...
echo 'APP greeting'      > "$ROOT/low1/greeting"   # ...app layer must win
sudo mount -t overlay overlay \
  -o lowerdir="$ROOT/low1:$ROOT/low2:$ROOT/low3",upperdir="$ROOT/upper",workdir="$ROOT/work" \
  "$ROOT/merged"
echo "mounted rootfs at $ROOT/merged"
SH
chmod +x ~/build-image.sh
~/build-image.sh
      mounted rootfs at /tmp/handimage/merged
    2. Confirm the union and shadowing: every layer's files are present, and the app layer's greeting wins over the base layer's. 
      ls /tmp/handimage/merged
cat /tmp/handimage/merged/greeting
      app.conf  greeting  os-release  runtime
APP greeting                          ← low1 (app) shadows low3 (base)
    3. Trigger a copy-up on a bottom-layer file, then find the full copy in upper. 
      echo 'patched at runtime' >> /tmp/handimage/merged/os-release
ls /tmp/handimage/upper
cat /tmp/handimage/low3/os-release
      os-release                              ← copied up into the writable layer
base OS file                            ← the bottom layer is untouched
    4. Delete a lower-layer file and find its whiteout — the 0/0 character device in upper. 
      rm /tmp/handimage/merged/runtime
ls -l /tmp/handimage/upper/runtime
      c--------- 1 root root 0, 0 ... /tmp/handimage/upper/runtime
    5. Reset the writable layer and remount — every original reappears, proving the lowers survived every change and the upper is disposable. 
      sudo umount /tmp/handimage/merged
rm -rf /tmp/handimage/upper/* /tmp/handimage/work/*
sudo mount -t overlay overlay \
  -o lowerdir=/tmp/handimage/low1:/tmp/handimage/low2:/tmp/handimage/low3,upperdir=/tmp/handimage/upper,workdir=/tmp/handimage/work \
  /tmp/handimage/merged
ls /tmp/handimage/merged; cat /tmp/handimage/merged/os-release
      app.conf  greeting  os-release  runtime   ← runtime is back, patch is gone
base OS file                              ← original restored: lowers never changed
    Verify

    The project is complete when, from the running overlay, you can point at the copied-up file in upper/, point at the whiteout device, and explain why emptying upper/ restored every original. (The copy-up is the full lower file materialised in upper on first write; the whiteout is a 0/0 char device masking a lower file; and the lowers are read-only and were never modified, so discarding the upper — the only place changes were recorded — returns the merged view to the pristine stack.) Keep ~/build-image.sh: B6 runs a process inside merged as its root.

    Gotchas

    Re-running the script cleanly requires unmounting merged first, which the script does — but if a shell is cd'd into merged (or B6's chroot is still running), the unmount fails "target is busy" and the rebuild leaves a stale overlay. If build-image.sh errors on the umount, find the holder with lsof /tmp/handimage/merged and move out of it — the same busy-unmount discipline as B5.3.

    The whiteout and copy-up files in upper/ are owned by root (the overlay was mounted with sudo), so you'll need sudo to inspect or remove some of them. That's expected: the writable layer of a real container is likewise root-owned on the node. Don't chown them to work around it — just use sudo ls -l.

    You have now built the storage half of a container. What's missing before /tmp/handimage/merged is a real container is isolation: a process whose root is this merged directory and which can't see the host's processes, network, or mounts. That's B6 (namespaces, chroot, pivot_root) — it takes this exact rootfs and runs a shell inside it. Kubernetes' runtime does precisely this at scale: assemble the image layers as an overlay (this project), then wrap a process in namespaces and cgroups rooted at the merged view. Every pod on a node is this script plus the isolation you'll add next.

    The rootfs you built feeds directly into B6's isolation labs and the from-scratch container of B14; on the cluster it's the per-container overlay under W1.23's kubelet storage.

    Docs

    man 8 mount under -t overlay and the kernel's Documentation/filesystems/overlayfs.rst for the mechanism you've now automated. man 1 lsof for the busy-unmount case. Forward pointer: man 2 chroot and man 8 pivot_root are what B6 adds on top of this rootfs.

    B5.18

    Debugging drill: the impossible-full disk, the mount that won't cross, the stale handle

    Why it matters

    Three failures, each of which looks like something it isn't, drilled until the right command is a reflex. A disk that is 100% full holding a few megabytes of files — not a space problem at all. A mount you made that a bind-mounted peer stubbornly refuses to show — not a mount failure, a propagation setting. A "Stale file handle" error out of nowhere — not corruption, a process holding onto a filesystem that was pulled away. Each wastes an afternoon when guessed at and takes under a minute with the tool and the mechanism from this module.

    The concept

    You have every piece. Inode exhaustion (B5.2, B5.8): the fixed inode count set at mkfs time runs out, so df shows blocks free while df -i shows inodes at 100% and every new file fails with ENOSPC. Propagation (B5.12): a mount made under a private mount point doesn't appear at its peers; only shared/slave carry it across. Stale file handle (ESTALE): a process holds a reference (open fd or cwd) to an inode on a filesystem that has since been unmounted or detached, so the kernel can no longer resolve the reference. The discipline all three share: read the evidence — df -i, findmnt -o PROPAGATION, /proc/PID/ — before touching the obvious-but-wrong object.

    Lab

    Snapshot first (B0.9). Work each symptom, name a one-sentence cause backed by a command that proves it, then read the reveal.

    Symptom 1 — a disk that is full with almost no data
    1. Build a tiny filesystem with deliberately few inodes, then fill it with empty files until it "runs out of space". 
      fallocate -l 20M /tmp/tiny.img
mkfs.ext4 -F -N 256 /tmp/tiny.img          # only 256 inodes
sudo mkdir -p /mnt/tiny && sudo mount -o loop /tmp/tiny.img /mnt/tiny
i=0; while sudo touch /mnt/tiny/f$i 2>/dev/null; do i=$((i+1)); done; echo "made $i files"
      made 241                                 ← ran out well before filling 20M
    2. Look at the contradiction, then resolve it with the right view. Blocks are nearly empty; the next touch fails with the same error a truly full disk gives. 
      sudo touch /mnt/tiny/onemore
df -h /mnt/tiny
df -i /mnt/tiny
      touch: cannot touch '/mnt/tiny/onemore': No space left on device
Filesystem  Size Used Avail Use% Mounted on
/dev/loop0   19M 464K   17M   3% /mnt/tiny     ← blocks: 3% used
Filesystem Inodes IUsed IFree IUse% Mounted on
/dev/loop0    256   256     0 100% /mnt/tiny     ← inodes: exhausted
    Cause & fix

    Inode exhaustion, not block exhaustion. The filesystem was created with only 256 inodes; each file — however small — consumes one, so it refused new files at 3% block usage. df alone hides this because it shows blocks; df -i reveals the 100%. The fix is to delete unneeded files to free inodes, or, if the filesystem genuinely needs millions of small files, recreate it with more inodes (mkfs.ext4 -N or a smaller -i bytes-per-inode ratio) — you cannot add inodes to an existing ext4. The reflex: when ENOSPC hits a disk df says has room, run df -i before anything else. Clean up: sudo umount /mnt/tiny; rm /tmp/tiny.img.

    Symptom 2 — a mount that a bind peer won't show
    1. Reproduce it: a private mount point, a bind peer, and a new mount underneath that fails to appear across the bind — the container-mount-not-visible scenario in miniature. 
      sudo mkdir -p /mnt/host /mnt/view
sudo mount --bind /mnt/host /mnt/host && sudo mount --make-private /mnt/host
sudo mount --bind /mnt/host /mnt/view      # /mnt/view is a "container's" view
sudo mkdir -p /mnt/host/disk
sudo mount -t tmpfs tmpfs /mnt/host/disk   # "attach a disk" on the host side
ls /mnt/view/disk; findmnt /mnt/view/disk || echo 'not visible in the view'
      not visible in the view               ← the tmpfs didn't cross the bind
    2. Diagnose from the propagation column instead of re-trying the mount. private is the culprit; the fix is to make the relationship carry mounts. 
      findmnt -o TARGET,PROPAGATION /mnt/host /mnt/view
      TARGET     PROPAGATION
/mnt/host  private                       ← nothing propagates out of here
/mnt/view  private
    Cause & fix

    Private propagation, not a broken mount. The mount under /mnt/host succeeded fine — it just didn't propagate to the bind peer because the mount point is private. Make the source shared (both directions) or, for the real container case, rslave on the view (host→container only): sudo mount --make-shared /mnt/host, then re-mount the tmpfs and it appears in /mnt/view. The reflex: when a mount "isn't showing up" across a bind or container boundary, check findmnt -o TARGET,PROPAGATION before suspecting the mount itself. Clean up: sudo umount /mnt/host/disk 2>/dev/null; sudo umount /mnt/view; sudo umount /mnt/host.

    Symptom 3 — "Stale file handle" from nowhere
    1. Reproduce it: sit inside a mounted filesystem, have it pulled out from under you with a lazy unmount, then touch your surroundings. 
      sudo mount -o loop /tmp/disk.img /mnt/data
cd /mnt/data
sudo umount -l /mnt/data          # lazy unmount: detaches the name now
sudo losetup -D                   # detach the backing loop device
ls -l .
      ls: cannot open directory '.': Stale file handle
    2. Confirm what your shell is still holding — a cwd on a filesystem that no longer exists — then recover. 
      ls -l /proc/$$/cwd
cd /; ls -l /proc/$$/cwd
      lrwxrwxrwx ... /proc/NNNN/cwd -> /mnt/data (deleted)   ← rooted in a gone fs
lrwxrwxrwx ... /proc/NNNN/cwd -> /                        ← recovered after cd /
    Cause & fix

    A reference to a detached filesystem, not corruption. Your shell's working directory was an inode on the loop-backed filesystem; the lazy unmount detached the mount from the tree and the loop teardown removed the backing, so the kernel can no longer resolve your cwd — hence ESTALE, "Stale file handle". Nothing is broken; the process is simply anchored to something that was pulled away. The fix is to move the process off it: cd / (or restart the process), then re-mount if needed. The reflex: ESTALE means "what I'm holding got unmounted/replaced" — check /proc/PID/cwd and open fds, don't run fsck in a panic. This is why the busy-unmount refusal in B5.3 exists — a non-lazy umount would have prevented this by refusing while you were inside.

    Gotchas

    All three share the drill's recurring shape: the loud symptom names the wrong object. "No space" points at big files when the fault is inode count; "the mount failed" points at the mount when the fault is propagation; "stale file handle" points at corruption when the fault is a detached filesystem. Reading the real state — df -i, findmnt -o PROPAGATION, /proc/PID/ — is what redirects you from the innocent-but-loud object to the guilty-but-quiet one, the same evidence-first discipline as the B2, B3, and B4 drills.

    umount -l (lazy) is a trap that creates symptom 3. It detaches the mount name immediately but leaves the filesystem half-alive for anyone still using it, turning a clean "target is busy" refusal into a delayed ESTALE somewhere else. Prefer a plain umount and fix the busy holder (B5.3); reach for -l only when you understand you're deferring the problem, not solving it.

    All three are node incidents you'll meet through Kubernetes. Inode exhaustion triggers kubelet disk-pressure eviction with bytes to spare — a node "out of disk" that df says is fine until you run df -i. Propagation bugs are exactly why a CSI driver's mount sometimes isn't visible to an app pod, or a sidecar's mount doesn't reach the main container — mountPropagation is the field, and this drill is the diagnosis. And ESTALE shows up when a volume is detached or a node loses a network filesystem out from under a running container. The reflexes — df -i, propagation, /proc — are the same whether the filesystem is on a laptop or under a pod.

    These map onto the storage-failure scenarios in W6 troubleshooting; the propagation case connects to mountPropagation in W5.

    B5 recall quiz

    Answer out loud before revealing. If you can explain what a container image physically is from these, the storage half of Kubernetes is yours.

    1. What is a "block device", and how does it differ from the filesystem that lives on it?
      Answer

      A block device is raw storage addressable only in fixed-size chunks (blocks); it knows nothing about files, names, or directories. A filesystem is a data structure — superblock, inode table, directory blocks, free-space maps — written onto that device by mkfs, which is what gives the blocks the meaning of "files". The device is dumb; the filesystem is the bookkeeping layered on top.

    2. A filesystem reports "No space left on device" but df shows 60% of blocks free. What happened, and which command confirms it?
      Answer

      Inode exhaustion: the fixed number of inodes chosen at mkfs time is used up (millions of tiny files, one inode each), so no new file can be created even though block space remains. df -i shows inodes at 100% while df shows blocks free. You can't add inodes to an existing ext4 — delete files or recreate with more.

    3. Explain what mount /dev/loop0 /mnt/data actually does to the directory tree, and where the directory's previous contents go.
      Answer

      It grafts the root of the filesystem on /dev/loop0 onto /mnt/data, so walking into that directory now leads into the mounted filesystem. The directory's previous contents are not deleted — they're shadowed (hidden) by the mount and reappear the moment you unmount.

    4. Why is the VFS the thing that makes overlayfs (and containers) possible?
      Answer

      The VFS defines one set of operations — open/read/write/stat — that every filesystem implements, so programs call the same syscalls regardless of what's underneath. overlayfs is just another implementation behind that interface, which is why a process can be handed an overlay (or any) root filesystem and every ordinary program keeps working unchanged.

    5. A file has one inode but two names in the same directory listing, both showing link count 2. What are they, and what happens if you rm one?
      Answer

      They're hard links — two equal directory entries pointing at the same inode. Removing one decrements the link count to 1 and removes only that name; the file's data survives under the other name. The inode and its blocks are freed only when the link count reaches 0 and no process holds it open.

    6. df says a disk is full; du finds only a few MB of files. Give the most likely cause and how to find the culprit.
      Answer

      A deleted-but-still-open file: a process deleted a file's name while holding it open, so du/ls can't see it but its blocks are still allocated (the inode outlives the name until the fd closes). Find it with lsof +L1 or by grepping /proc/*/fd for (deleted); fix it by restarting the holding process, not by deleting more files.

    7. You run mount --bind -o ro /src /dst and then successfully write a file in /dst. Why, and how do you actually get read-only?
      Answer

      The ro flag on the initial bind is ignored — read-only is a property of the underlying mount, not of the bind operation — so the bind comes up writable. To get a truly read-only view you bind first, then remount: mount -o remount,ro,bind /dst. Always verify a "read-only" bind with an actual write attempt.

    8. Name overlayfs's four directories and what each does.
      Answer

      lowerdir — one or more read-only layers, stacked (leftmost highest priority); upperdir — the single writable layer where all changes land; workdir — empty scratch on the upper's filesystem for the kernel's atomic operations; merged — the mount point showing the combined union you read and write through.

    9. You append one line to a 2 GB file that exists only in a lower layer of an overlay. What happens, and why might it be slow?
      Answer

      A copy-up: overlayfs can't modify the read-only lower, so it copies the entire 2 GB file into the upper layer first, then applies your one-line append to the copy. The cost is proportional to the file's size, not the edit's, so the first write is slow and consumes 2 GB in the writable layer; the lower original is left untouched.

    10. You delete a file that lives in a lower layer of an overlay. What appears in the upper layer, and is the lower file gone?
      Answer

      A whiteout appears — a character device with device number 0/0 at the deleted file's name — which masks the name so overlayfs stops descending to the lower layer. The lower file is not gone; it's still physically present and untouched, just hidden. This is why deleting files from an image's lower layers doesn't reclaim their space.

    11. State what a container image physically is and what a running container adds, then derive why in-container data is lost on recreate.
      Answer

      An image is a stack of read-only lowerdirs (one per layer), shared across all containers using it. A running container adds one fresh, empty writable upperdir (plus workdir), mounted as its root. Recreating the container gives it a new empty upperdir and discards the old one — and all in-container writes lived in that upper, so they're gone. A volume (a separate mount outside the overlay) is what survives.

    12. A process gets "Stale file handle" (ESTALE). What does that mean and what's the fix — is fsck the right move?
      Answer

      It means the process holds a reference (open fd or cwd) to an inode on a filesystem that was unmounted, detached, or replaced, so the kernel can no longer resolve it. It's not corruption, so fsck is wrong. Fix by moving the process off the dead reference — cd / or restart it, checking /proc/PID/cwd and /proc/PID/fd — then re-mount if needed. Lazy unmounts (umount -l) are a common cause.
