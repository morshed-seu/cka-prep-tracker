    Week 0 — Set up your lab & logistics

    Everything in the next eight weeks happens on the cluster you build here. The exam runs on kubeadm clusters on Ubuntu, so that's exactly what you'll build — by hand, once, so every prerequisite the exam can break is something you've configured yourself. Ticking done here also ticks the checkpoint on the tracker.

    Logistics

    0.1

    Buy the exam and schedule it ~8 weeks out

    Why it matters

    A booked date turns "someday" into a training plan. Everything on this site is paced backwards from that date.

    Steps
    1. Buy the CKA on the Linux Foundation training site (training.linuxfoundation.org). Look for bundle discounts and coupon codes before paying list price — they are frequent.
    2. Your purchase includes two killer.sh exam simulator sessions and one free retake. Don't touch killer.sh yet — this plan spends session 1 in week 7 and session 2 in week 8.
    3. Schedule through the portal (exams are proctored by PSI). Pick a slot ~8 weeks out, at a time of day when you're sharp. You can reschedule up to 24h before, so booking early costs nothing.
    4. Set the same date in this site's tracker sidebar so the countdown is honest.
    Gotchas

    The exam eligibility window is 12 months from purchase, and the retake must also fall inside it. Don't buy the exam a year before you intend to study.

    Docs

    Exam handbook & policies: the Linux Foundation candidate handbook (linked from your purchase page). Not exam-allowed material — read it once now.

    0.3

    Run the PSI system check on your exam machine

    Why it matters

    People lose exams to check-in problems more often than to etcd questions. The proctoring stack is picky; find out on day 0, not exam day.

    Steps
    1. From your exam registration page, run the PSI online proctoring compatibility check on the exact machine, network, and room you'll use.
    2. Verify: webcam (they'll pan the room with it), microphone, stable bandwidth, and permission to install the PSI Secure Browser (corporate laptops often block it — use a personal machine).
    3. Plan the physical setup now: one monitor only works best, clear desk, government ID that matches your registration name, nobody else entering the room.
    Verify

    The compatibility check passes on the machine you will actually test on — not a similar one.

    Gotchas

    The exam desktop is a remote XFCE Linux VM streamed through the PSI browser. Your local OS doesn't matter, but latency does — prefer wired ethernet, and close every bandwidth-hungry app.

    0.4

    Memorize which documentation the exam allows

    Why it matters

    The CKA is open-book for exactly three sites. Training yourself to answer everything from those three sites is exam training — do it from today.

    The rules
    • Allowed: kubernetes.io/docs, kubernetes.io/blog, and helm.sh/docs (plus their subdomains).
    • Only inside the exam environment's own Firefox, in one additional tab. No personal bookmarks, no Google, no Stack Overflow, no GitHub.
    • The docs' built-in search box is allowed — it's your fastest navigation tool in there.
    Steps
    1. From now on, study with only these sites open. When you catch yourself googling, redo the lookup via kubernetes.io/docs search instead.
    2. Each week, note which doc pages you keep returning to — by week 8 you'll know their titles, which is what you type into the docs search during the exam.
    Gotchas

    Deep-link muscle memory beats bookmarks you won't have: e.g. searching "persistent volumes", "ingress", "upgrade kubeadm" lands on the right page in one try. The lessons on this site end with the exact doc pages worth knowing by title.

    Lab environment

    0.5

    Create 3 Ubuntu VMs: one control plane, two workers

    Why it matters

    You need a cluster you can break. Managed clusters (EKS/GKE) and minikube hide the control plane; the exam expects you to SSH into nodes, read journalctl, and edit static pod manifests. Only your own kubeadm cluster gives you that.

    🧱 Foundations

    These three machines are full virtual machines — each boots its own separate Linux kernel, unlike the containers you'll create inside them later on, which instead share one kernel via namespaces and cgroups (see Foundations §1). You need real, separate kernels here because kubeadm and the kubelet expect to own a whole machine — its own systemd, its own network interfaces — not a sandboxed slice of someone else's.

    The concept

    Three nodes is the minimum interesting cluster: a control plane to break, a worker to drain, and a second worker for pods to reschedule onto. 2 vCPU / 4 GB per node matches kubeadm's preflight minimums for the control plane and is plenty for workers.

    OptionGood forWatch out
    Multipass (shown below)Fastest local start on any OSVM IPs can change after a host reboot — recheck /etc/hosts
    Vagrant + VirtualBox/libvirtReproducible via Vagrantfile, static IPsHeavier; VirtualBox networking quirks
    Cloud VMs (any provider)Real network, snapshots, always-onCosts money — shut down between sessions; open required ports between nodes
    Lab
    1. Launch the three VMs (Ubuntu LTS): 
      multipass launch 24.04 --name cp     --cpus 2 --memory 4G --disk 20G
multipass launch 24.04 --name node01 --cpus 2 --memory 4G --disk 20G
multipass launch 24.04 --name node02 --cpus 2 --memory 4G --disk 20G
multipass list
      Name      State    IPv4          Image
cp        Running  10.211.64.10  Ubuntu 24.04 LTS
node01    Running  10.211.64.11  Ubuntu 24.04 LTS
node02    Running  10.211.64.12  Ubuntu 24.04 LTS
    2. Give every node name-resolution for the others. On each VM (multipass shell cp, then repeat for node01/node02), append your actual IPs: 
      sudo tee -a /etc/hosts <<'EOF'
10.211.64.10 cp
10.211.64.11 node01
10.211.64.12 node02
EOF
    3. Confirm the nodes see each other: 
      ping -c1 node01 && ping -c1 node02
      64 bytes from node01 (10.211.64.11): icmp_seq=1 ttl=64 time=0.35 ms
64 bytes from node02 (10.211.64.12): icmp_seq=1 ttl=64 time=0.40 ms
    Verify

    multipass list shows all three Running, and each VM can ping the other two by name.

    Gotchas

    Unique hostnames and unique MAC/product-uuids are kubeadm preflight requirements — cloning one VM three times without resetting identity fails kubeadm join in confusing ways. Multipass/Vagrant handle this for you.

    0.6

    Install containerd + kubeadm/kubelet/kubectl on all nodes, by hand

    Why it matters

    Every step below is a node prerequisite the exam can plant as a fault: swap on, missing kernel module, forwarding off, wrong cgroup driver, stopped runtime. Installing manually once means you'll recognize each failure later. Run this whole lab on all three nodes.

    🧱 Foundations

    A cgroup is the kernel mechanism that caps how much CPU/memory a process may use — every Kubernetes resources.limits you'll ever write ultimately becomes a cgroup setting on some process, which is why kubelet and containerd must agree on which cgroup "driver" (dialect) they speak. The CRI is just the plumbing between them: the kubelet never touches containers directly, it asks containerd to, over a socket. See Foundations §1 for the bigger container/process picture this sits on.

    The concept

    The kubelet doesn't run containers itself — it drives a runtime (containerd) over the CRI socket. Two independent programs then manage the same kernel cgroup tree, which is why their cgroup drivers must match: on systemd distros both must use the systemd driver. Mismatched drivers is the classic "pods randomly fail on one node" fault. Modern kubeadm defaults the kubelet to systemd, so the containerd side (SystemdCgroup = true) is the one you must set yourself.

    Lab
    1. Disable swap (kubelet refuses to start with swap on, by default): 
      sudo swapoff -a
sudo sed -i '/\sswap\s/ s/^/#/' /etc/fstab
free -h | grep -i swap
      Swap:            0B          0B          0B
    2. Load the required kernel modules now and on every boot: 
      cat <<'EOF' | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF
sudo modprobe overlay && sudo modprobe br_netfilter

      overlay is the layered filesystem container images run on; br_netfilter lets iptables see traffic crossing Linux bridges — without it, Service traffic between pods on the same node can bypass kube-proxy's rules.

    3. Enable forwarding sysctls: 
      cat <<'EOF' | sudo tee /etc/sysctl.d/k8s.conf
net.ipv4.ip_forward                 = 1
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
EOF
sudo sysctl --system | tail -4
      * Applying /etc/sysctl.d/k8s.conf ...
net.ipv4.ip_forward = 1
net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1

      A node is a router for its pods; with ip_forward=0 nothing leaves the pod network. This is a favorite planted fault.

    4. Install and configure containerd with the systemd cgroup driver: 
      sudo apt-get update && sudo apt-get install -y containerd
sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml >/dev/null
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml
sudo systemctl restart containerd && systemctl is-active containerd
      active
    5. Add the Kubernetes package repo. The repo URL pins a minor version — set it to the version the exam currently runs (check your exam registration page; the example below is a placeholder): 
      K8S_MINOR=v1.35   # ← set to the current exam minor version
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://pkgs.k8s.io/core:/stable:/${K8S_MINOR}/deb/Release.key \
  | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg
echo "deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/${K8S_MINOR}/deb/ /" \
  | sudo tee /etc/apt/sources.list.d/kubernetes.list
    6. Install and pin the three packages, plus cri-tools (the same repo publishes it, matching the v1.35 minor) for crictl — the node-level CRI debugging tool weeks 1 and 6 rely on: 
      sudo apt-get update
sudo apt-get install -y kubelet kubeadm kubectl cri-tools
sudo apt-mark hold kubelet kubeadm kubectl cri-tools
      kubelet set on hold.
kubeadm set on hold.
kubectl set on hold.
cri-tools set on hold.

      apt-mark hold stops apt upgrade from bumping Kubernetes behind your back — cluster upgrades must be deliberate (that's the week-2 kubeadm upgrade dance, and the unhold/hold steps are part of the exam task).

    Verify
    kubeadm version -o short && systemctl is-active containerd && lsmod | grep br_netfilter && crictl --version
    v1.35.x
active
br_netfilter           32768  0
crictl version v1.35.x
    Gotchas

    It's normal that kubelet is crash-looping right now (systemctl status kubelet shows restarts) — it has no config until kubeadm init/join writes one. Also note the repo pattern: to upgrade a minor version later you must edit /etc/apt/sources.list.d/kubernetes.list first — pkgs.k8s.io repos only carry their own minor. crictl works with no further config once /run/containerd/containerd.sock exists (1.20 has you point it there explicitly anyway).

    Docs

    During the exam, search the docs for "Installing kubeadm" and "Container runtimes" — the repo commands above are copy-pasteable from there; never type them from memory.

    0.7

    kubeadm init, install a CNI, join both workers

    Why it matters

    This is the cluster you keep for eight weeks. The init/join flow is also itself exam material (week 2 dissects every phase).

    🧱 Foundations

    Notice below that "kubeadm starts nothing itself — the kubelet does." That's the static-pod escape hatch: normally the apiserver decides what runs where, but the apiserver has to start somehow first. Foundations §8 walks through why systemd only ever needs to know about one service — the kubelet — and how the kubelet bootstraps the rest of the control plane straight from files on disk.

    The concept

    kubeadm init generates a CA and certificates, writes kubeconfigs, drops static pod manifests for the control plane into /etc/kubernetes/manifests/ (the kubelet starts them — kubeadm starts nothing itself), then installs CoreDNS and kube-proxy. Nodes report NotReady until a CNI plugin gives pods a network. --pod-network-cidr must match what your CNI expects: Flannel's default is 10.244.0.0/16.

    Lab
    1. On cp only: 
      sudo kubeadm init --pod-network-cidr=10.244.0.0/16
      [init] Using Kubernetes version: v1.35.x
[preflight] Running pre-flight checks
[certs] Generating "ca" certificate and key
...
Your Kubernetes control-plane has initialized successfully!
...
kubeadm join 10.211.64.10:6443 --token <token> \
        --discovery-token-ca-cert-hash sha256:<hash>

      Copy the kubeadm join line it prints — you need it in step 4 (and you can always regenerate it with kubeadm token create --print-join-command).

    2. Set up your kubeconfig on cp (this is admin credentials — week 2 explains what's inside): 
      mkdir -p $HOME/.kube
sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config
kubectl get nodes
      NAME   STATUS     ROLES           AGE   VERSION
cp     NotReady   control-plane   1m    v1.35.x

      NotReady is expected — no CNI yet.

    3. Install Flannel (or Calico if you prefer — then use its CIDR in step 1): 
      kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
kubectl get nodes
      NAME   STATUS   ROLES           AGE   VERSION
cp     Ready    control-plane   3m    v1.35.x
    4. On node01 and node02, paste the join command from step 1: 
      sudo kubeadm join 10.211.64.10:6443 --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>
      This node has joined the cluster:
* Certificate signing request was sent to apiserver and a response was received.
* The Kubelet was informed of the new secure connection details.
    Verify
    kubectl get nodes -o wide && kubectl get pods -n kube-system
    NAME     STATUS   ROLES           AGE   VERSION
cp       Ready    control-plane   10m   v1.35.x
node01   Ready    <none>          2m    v1.35.x
node02   Ready    <none>          1m    v1.35.x
All kube-system pods Running, including coredns-* (2 replicas)
    Gotchas

    CoreDNS pods stay Pending until the CNI is up — that ordering (CNI before DNS) is a troubleshooting signature worth remembering. If you forgot --pod-network-cidr, Flannel crash-loops with "failed to acquire lease: node ... pod cidr not assigned"; fastest lab fix is kubeadm reset and re-init (week 2 covers surgical fixes).

    Docs

    Docs search: "Creating a cluster with kubeadm". The CNI manifest URL is not in the docs — but installing a CNI from a URL given in the task is a known exam pattern.

    0.8

    Snapshot the VMs so you can break things fearlessly

    Why it matters

    Weeks 1–6 repeatedly tell you to break this cluster. A 2-minute restore path is what makes that psychologically free.

    Lab
    1. Snapshot all three VMs in their fresh, healthy state (Multipass snapshots need the VM stopped): 
      for vm in cp node01 node02; do
  multipass stop $vm
  multipass snapshot $vm --name fresh-cluster
  multipass start $vm
done
multipass list --snapshots
      Instance   Snapshot        Parent   Comment
cp         fresh-cluster   --
node01     fresh-cluster   --
node02     fresh-cluster   --
    2. Rehearse a restore once, so you trust it: 
      multipass stop node02
multipass restore node02.fresh-cluster --destructive
multipass start node02
    3. Cloud/Vagrant users: use provider snapshots or keep a rebuild script (steps 0.6–0.7 in a file) — tearing down and rebuilding by script is also excellent kubeadm practice.
    Verify

    After the rehearsal restore, kubectl get nodes on cp shows all three nodes Ready again within a couple of minutes.

    Gotchas

    Take a new snapshot after big milestones (e.g. end of week 2 with upgrades done) — restoring an 8-week-old image and replaying everything is the expensive way to fix a broken lab. Note: restoring only the control plane while workers keep running can confuse node leases for a minute or two; restore whole-cluster when in doubt.

    Shell & editor speed setup

    0.9

    Set up aliases, completion and the two magic variables

    Why it matters

    The exam is ~15–20 tasks in 120 minutes. Shaving 5 seconds off every kubectl invocation buys you a whole extra task. These four lines are the standard kit — start using them today so they're reflexes, not notes.

    Lab
    1. On cp (and any machine you practice from), append to ~/.bashrc: 
      cat <<'EOF' >> ~/.bashrc
alias k=kubectl
source <(kubectl completion bash)
complete -o default -F __start_kubectl k
export do="--dry-run=client -o yaml"
export now="--force --grace-period 0"
export KUBE_EDITOR=vim
EOF
source ~/.bashrc
    2. Test each one: 
      k get no                              # alias + completion (try TAB on 'k get pod<TAB>')
k run web --image=nginx $do           # scaffold YAML without creating anything
k run tmp --image=busybox -- sleep 1d
k delete pod tmp $now                 # no 30s graceful wait
      NAME     STATUS   ROLES           AGE   VERSION
cp       Ready    control-plane   1d    v1.35.x
...
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: web
  name: web
...
    The concept

    This is the declarative idea from Foundations §2 in miniature: instead of hand-typing YAML from a blank file, you ask the API server to render valid YAML for you and edit only the two or three fields the task actually needs. $do is the single highest-value habit: --dry-run=client -o yaml makes the API server render a valid manifest without creating anything — you scaffold, redirect to a file, edit the 2 fields the task needs, and apply. Writing YAML from scratch is how people run out of time.

    Verify

    k get pod<TAB> completes; k run x --image=nginx $do prints YAML and creates nothing (k get pods stays empty).

    Gotchas

    The exam terminal comes with k and completion mostly pre-configured — but your variables like $do are not there. Practice typing the setup block fast once; spending 60 seconds on it at exam start pays for itself.

    Docs

    Docs search: "kubectl Quick Reference" — this exact page contains the alias/completion/dry-run recipes during the exam.

    0.10

    Make vim safe for YAML and learn five motions

    Why it matters

    YAML dies on tab characters, and vim inserts tabs by default. Two lines of ~/.vimrc prevent an entire class of "my manifest won't apply" disasters.

    🧱 Foundations

    If YAML's indentation-only structure is new to you, read Foundations §3 first — it's two rules and one page, and from here on almost every lesson hands you a YAML file to edit.

    Lab
    1. Write the config: 
      cat <<'EOF' > ~/.vimrc
set tabstop=2 shiftwidth=2 expandtab
set number
syntax on
EOF

      expandtab turns Tab presses into spaces; tabstop/shiftwidth=2 match Kubernetes YAML convention.

    2. Drill these five motions until automatic (scaffold a deployment YAML with $do and practice on it):
      MotionWhat it doesExam use
      V then j/k then >visual-select lines, indentfixing indentation of a pasted block (. repeats)
      dd / yy / pcut / copy / paste a lineduplicating a container or volumeMount entry
      /image then nsearch, next matchjumping straight to the field to change
      ciwchange the word under cursorswapping an image tag or a name
      u / Ctrl-rundo / redorecovering from a bad edit fast
    Verify

    Open a scaffolded manifest, indent a 3-line block with V j j >, and confirm kubectl apply --dry-run=client -f accepts the file.

    Gotchas

    If you paste YAML and it cascades into stair-step indentation, that's vim auto-indent fighting the paste: use :set paste, paste, :set nopaste. This one trick saves minutes in the exam.

    0.11

    Learn kubectl explain — the docs that live in your terminal

    Why it matters

    For "what is this field called and where does it go", kubectl explain is faster than the website — and it's always the right API version for your cluster, because the data comes from your apiserver's OpenAPI schema.

    Lab
    1. Find a field's exact name and type: 
      k explain pod.spec.containers.livenessProbe | head -15
      KIND:       Pod
VERSION:    v1

FIELD: livenessProbe <Probe>

DESCRIPTION:
     Periodic probe of container liveness. Container will be restarted if the
     probe fails. ...
    2. See a whole subtree with structure: 
      k explain deployment.spec.strategy --recursive
      FIELDS:
  rollingUpdate <RollingUpdateDeployment>
    maxSurge      <IntOrString>
    maxUnavailable        <IntOrString>
  type  <string>
    3. Answer three questions with explain only (answers in the quiz below): where does terminationGracePeriodSeconds live? what type is readinessProbe.httpGet.port? which field of a Service selects NodePort vs ClusterIP?
    Verify

    You answered the three questions without opening a browser.

    Gotchas

    Resource names in explain are singular or plural, case-insensitive (pod, deploy, svc all work). For fields in CRDs, explain works too if the CRD publishes a schema — worth remembering for the operator questions.

    Week 0 recall quiz

    Answer out loud before revealing. If any answer surprises you, redo that lesson's lab.

    1. Which three documentation sites are allowed during the exam, and in which browser?
      Answer

      kubernetes.io/docs, kubernetes.io/blog, helm.sh/docs — only in the Firefox inside the exam environment's remote desktop, one extra tab.

    2. Why must swap be off on every node (by default)?
      Answer

      The kubelet refuses to start when it detects swap (default failSwapOn: true). Kubelet-won't-start with swap enabled is a classic planted fault: swapoff -a + comment the /etc/fstab line.

    3. What breaks if kubelet and containerd use different cgroup drivers?
      Answer

      Two managers fight over the same cgroup tree — pods fail or the node goes unstable in hard-to-diagnose ways. On systemd distros, both must use the systemd driver (SystemdCgroup = true in containerd's config).

    4. Why did nodes show NotReady right after kubeadm init, and what fixed it?
      Answer

      No CNI plugin yet — the kubelet reports the network not ready, and CoreDNS stays Pending. Installing the CNI (Flannel/Calico) flipped nodes to Ready.

    5. Why must --pod-network-cidr match the CNI's expected range?
      Answer

      The controller-manager assigns each node a pod CIDR from that range, and the CNI plugin reads it to configure node subnets. Flannel's manifest defaults to 10.244.0.0/16 — mismatch and Flannel crash-loops with "pod cidr not assigned".

    6. What does apt-mark hold kubelet kubeadm kubectl protect you from?
      Answer

      Unplanned version bumps from apt upgrade. Kubernetes upgrades must be deliberate and ordered (kubeadm first, control plane, then kubelets) — week 2 drills the real procedure, including unhold/hold.

    7. What does $do expand to, and why is it the highest-value habit?
      Answer

      --dry-run=client -o yaml — renders a valid manifest without creating the object. Scaffold → edit two fields → apply beats hand-writing YAML every time.

    8. From 0.11: where does terminationGracePeriodSeconds live, and what type is readinessProbe.httpGet.port?
      Answer

      pod.spec.terminationGracePeriodSeconds (pod level, not per-container); the probe port is IntOrString — a number or a named container port. Service type lives at service.spec.type.
