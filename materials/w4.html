    Week 4 — Services & networking, from veth up

    Networking is where most candidates guess; this week removes the guessing. You build a pod network by hand with ip commands, read the actual iptables rules a Service becomes, walk DNS resolution line by line, and wire Ingress and Gateway API against a real controller. Each layer sits on the previous one — once you can name the layer a packet died in, every networking task becomes mechanical. Ticking done here also ticks the tracker.

    Linux primitives→CNI→Services / kube-proxy→DNS→Ingress · Gateway API→NetworkPolicy

    Linux networking foundations

    4.1

    The network model: two rules, everything else is implementation

    Why it matters

    Every networking exam task is an instance of "one of the two rules is being violated — find where". If you know what the model guarantees, you know what a healthy cluster must be able to do, and every check this week tests one of those guarantees.

    The concept

    Kubernetes imposes exactly two requirements on whatever network plugin you run:

    • Every pod gets its own IP address. Containers in one pod share that IP (they share the pod's network namespace) and talk to each other over localhost.
    • All pods can reach all pods, on every node, without NAT. The IP a pod sees as its own is the same IP everyone else uses to reach it. Node agents (kubelet probes, node-local daemons) can reach every pod on their node the same way.

    That's the whole contract. Overlay tunnels, BGP route programming, eBPF — those are just different ways plugins deliver the same two guarantees (4.7). Everything above the model — Services, DNS names, Ingress — is machinery built on top of these flat, NAT-free pod IPs: a ClusterIP is a rewrite to a pod IP (4.9), a DNS record is a name for one (4.17). NAT only appears at the cluster edge: traffic leaving to the outside world is masqueraded, and NodePort traffic gets rewritten on the way in.

    Verify

    State from memory: what does a pod on node01 see as the source IP of a packet from a pod on node02? (The other pod's real pod IP — no NAT in between. If you see the node's IP instead, the CNI is misconfigured or the traffic left the pod network.)

    Gotchas

    The model says nothing about Service IPs — those are virtual, exist in no namespace, and very much do involve rewriting (4.9–4.10). Keep "pod IP = real, service IP = rules" as separate mental buckets; mixing them is the root of most DNS-vs-Service confusion.

    Docs

    Docs search: "Cluster Networking" — the model statement is at the top of that page.

    4.2

    Build a pod network by hand: netns, veth pair, bridge

    Why it matters

    This 15-minute exercise is what the bridge CNI plugin does for every pod. After it, ip addr, ip route and ip netns exec stop being incantations and become debugging tools you reach for on instinct.

    The concept

    Three Linux primitives make a pod network:

    • A network namespace is an isolated copy of the kernel's network stack — own interfaces, own routing table, own iptables. A pod is a netns (created and held open by the pause container; app containers join it).
    • A veth pair is a virtual patch cable: two interfaces, always created together; a packet in one end comes out the other. One end is moved into the pod's netns and renamed eth0; the other end stays in the node's root namespace.
    • A bridge is a virtual L2 switch in the root namespace. All the host-side veth ends plug into it, so pods on one node reach each other through the bridge, and the node reaches pods via the bridge's own IP.
    Lab

    Run this on a scratch VM, or on node01 — the names below don't clash with anything Kubernetes created, and step 7 removes everything.

    1. Create two "pods" (namespaces) and the "node bridge": 
      sudo ip netns add red
sudo ip netns add blue
sudo ip link add br-lab type bridge
sudo ip addr add 10.42.0.1/24 dev br-lab
sudo ip link set br-lab up
ip netns list
      blue
red
    2. Make a veth pair for red and push one end into the namespace: 
      sudo ip link add veth-red type veth peer name ceth-red
sudo ip link set ceth-red netns red
sudo ip link set veth-red master br-lab
sudo ip link set veth-red up
    3. Inside red: rename the interface to eth0 (exactly what CNI does), give it an IP, bring it up: 
      sudo ip netns exec red ip link set ceth-red name eth0
sudo ip netns exec red ip addr add 10.42.0.2/24 dev eth0
sudo ip netns exec red ip link set eth0 up
sudo ip netns exec red ip link set lo up
sudo ip netns exec red ip route add default via 10.42.0.1
    4. Repeat for blue with 10.42.0.3: 
      sudo ip link add veth-blue type veth peer name ceth-blue
sudo ip link set ceth-blue netns blue
sudo ip link set veth-blue master br-lab
sudo ip link set veth-blue up
sudo ip netns exec blue ip link set ceth-blue name eth0
sudo ip netns exec blue ip addr add 10.42.0.3/24 dev eth0
sudo ip netns exec blue ip link set eth0 up
sudo ip netns exec blue ip link set lo up
sudo ip netns exec blue ip route add default via 10.42.0.1
    5. Ping "pod to pod" through the bridge, and "pod to node": 
      sudo ip netns exec red ping -c1 10.42.0.3
sudo ip netns exec red ping -c1 10.42.0.1
      64 bytes from 10.42.0.3: icmp_seq=1 ttl=64 time=0.08 ms
64 bytes from 10.42.0.1: icmp_seq=1 ttl=64 time=0.05 ms
    6. Look at what you built with the same commands you'll use on real pods: 
      ip link show master br-lab
sudo ip netns exec red ip addr show eth0
      5: veth-red@if2: <BROADCAST,MULTICAST,UP,LOWER_UP> ... master br-lab
7: veth-blue@if2: <BROADCAST,MULTICAST,UP,LOWER_UP> ... master br-lab
2: eth0@if5: <BROADCAST,MULTICAST,UP,LOWER_UP> ... link-netnsid 0
    inet 10.42.0.2/24 scope global eth0

      Read eth0@if5 as "my peer is host interface index 5" — that @ifN notation is how you'll match pods to veths in 4.3.

    7. Clean up (deleting a netns deletes the veth ends inside it; the pair dies with it): 
      sudo ip netns del red
sudo ip netns del blue
sudo ip link del br-lab
    Verify

    The first ping in step 5 succeeds — you routed a packet between two isolated namespaces through a bridge you built. If it fails: check both veth host-ends are UP and master br-lab, and that the netns eth0 has its IP.

    Gotchas

    If pings between namespaces fail even though everything is UP, suspect br_netfilter + firewall rules: with net.bridge.bridge-nf-call-iptables=1 (which Kubernetes needs, W0.6), bridged traffic is run through iptables — a restrictive FORWARD chain policy can drop it. That exact interaction is why the sysctl exists: without it, same-node pod traffic would bypass kube-proxy and NetworkPolicy rules.

    Docs

    Not in the k8s docs — this is man ip-netns territory. The k8s side is docs search "Network Plugins".

    4.3

    Find a real pod's veth peer on the node

    Why it matters

    "Which host interface belongs to this pod?" is the move that lets you tcpdump a single pod's traffic from the node — the sharpest tool you have when a pod's network "just doesn't work".

    The concept

    Every veth end carries the interface index of its peer. Inside the pod, eth0@if12 means "my other end is host ifindex 12"; the matching host interface shows vethXXXX@if2 pointing back at the pod's ifindex. Match the numbers and you've matched pod to veth. Most images have no ip binary, but /sys/class/net/eth0/iflink holds the same peer index and needs no tools.

    Lab
    1. Run a pod and see where it landed: 
      kubectl run web4 --image=nginx
kubectl get pod web4 -o wide
      NAME   READY   STATUS    RESTARTS   AGE   IP           NODE
web4   1/1     Running   0          20s   10.244.1.8   node01
    2. Ask the pod for its peer's host-side index: 
      kubectl exec web4 -- cat /sys/class/net/eth0/iflink
      12
    3. On node01, find host interface 12: 
      ip link | grep '^12:'
      12: veth1a2b3c4d@if2: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 ... master cni0

      @if2 points back at the pod's eth0 (ifindex 2 inside the pod); master cni0 is flannel's bridge — the real-world br-lab from 4.2.

    4. Prove it's the right one — capture the pod's traffic from the node: 
      sudo tcpdump -ni veth1a2b3c4d -c2 icmp &
kubectl exec web4 -- ping -c1 10.244.1.1
      10.244.1.8 > 10.244.1.1: ICMP echo request
10.244.1.1 > 10.244.1.8: ICMP echo reply
    Verify

    The iflink number from inside the pod equals the ifindex of a veth* interface on the node, and tcpdump on that veth shows the pod's ping.

    Gotchas

    Interface indexes are per-namespace and not stable across pod restarts — never script against them. And if kubectl exec is unavailable (crashed pod), you can still get the netns from the runtime: sudo crictl inspectp <sandbox-id> | grep netns, then sudo nsenter --net=<path> ip addr.

    Docs

    Docs search: "Debug Running Pods" — the ephemeral-container variant (kubectl debug) for distroless images lives there.

    4.4

    Know your three CIDRs and where each is configured

    Why it matters

    Half of network triage is "which range is this IP from?" — answer that and you know which subsystem to blame. A 10.244.x.x address that won't connect is a CNI problem; a 10.96.x.x address is a kube-proxy/Service problem; a node IP is plain Linux.

    You can answer instantly: which range is 10.96.130.5 from and which component assigned it? (Service CIDR, apiserver.) Which node hosts 10.244.2.9? (node02.)

    The three ranges must never overlap — an exam-grade fault is a service CIDR overlapping the node network, which makes traffic to some ClusterIPs route out the physical NIC instead of hitting DNAT rules. Also: changing the pod CIDR after install is effectively a cluster rebuild; nodes keep their allocated podCIDR forever.

    CNI

    4.5

    The CNI contract: exec a binary, JSON in, JSON out

    Why it matters

    The 2025 curriculum added CNI understanding explicitly. More practically: "node NotReady, cni plugin not initialized" is a staple planted fault, and it only takes seconds to fix if you know the two directories involved.

    The concept

    CNI is deliberately dumb. It is not a daemon or an API — it's a convention for exec-ing a binary once per pod network event:

    • When containerd creates a pod sandbox, it reads the first config file (lexicographic order) from /etc/cni/net.d/.
    • It executes the named plugin binary from /opt/cni/bin/ with the verb in the CNI_COMMAND env var — ADD (wire this pod), DEL (unwire it), CHECK, VERSION — the network config as JSON on stdin, and details like the netns path and container ID in more env vars.
    • The plugin does its work (create veth, program routes) and prints a JSON result — the IPs it assigned — on stdout.

    Plugins chain: a .conflist file holds a list, each plugin's result feeding the next. The typical split: a main plugin wires the interface (bridge, or flannel/calico/cilium's own), an IPAM plugin picks the IP (host-local = a directory of allocated-IP files), and meta plugins add extras (portmap for hostPort, bandwidth). Flannel's plugin is famously thin: it reads the node's subnet from /run/flannel/subnet.env and delegates to the standard bridge plugin — the thing you built in 4.2, verbatim.

    containerd: create sandbox netns→read /etc/cni/net.d→exec plugin: CNI_COMMAND=ADD, JSON on stdin→plugin wires veth + routes→IPAM assigns IP→result JSON on stdout → pod IP
    Lab
    1. Read your cluster's CNI config (any node): 
      sudo cat /etc/cni/net.d/10-flannel.conflist
      {
  "name": "cbr0",
  "cniVersion": "0.3.1",
  "plugins": [
    { "type": "flannel", "delegate": { "hairpinMode": true, "isDefaultGateway": true } },
    { "type": "portmap", "capabilities": { "portMappings": true } }
  ]
}

      Two chained plugins: flannel (which delegates to bridge + host-local), then portmap.

    2. List the binaries the runtime can exec: 
      ls /opt/cni/bin
      bandwidth  bridge  dhcp  firewall  flannel  host-device  host-local  ipvlan
loopback  macvlan  portmap  ptp  sbr  static  tuning  vlan  vrf
    3. See host-local IPAM's "database" — one file per allocated IP, named by IP, containing the container ID: 
      sudo ls /var/lib/cni/networks/cbr0/ | head -4
sudo cat /var/lib/cni/networks/cbr0/10.244.1.8
      10.244.1.7
10.244.1.8
last_reserved_ip.0
lock
3f9a1c…  (the pod sandbox's container ID)  eth0
    Verify

    You can name, without looking: the config directory, the binary directory, the four verbs, and where the JSON goes in and comes out. (/etc/cni/net.d, /opt/cni/bin, ADD/DEL/CHECK/VERSION, stdin in / stdout out.)

    Gotchas

    Both directories are per node — a CNI broken on one node breaks only that node's pods (and only new pods: existing ones keep their already-wired veths). That asymmetry — old pods fine, new pods stuck in ContainerCreating on one node — is the CNI-fault signature, drilled in 4.8.

    Docs

    Docs search: "Network Plugins" (the CNI spec itself lives at github.com/containernetworking — not available in the exam, but the k8s page covers what you need).

    4.6

    Inspect your actual CNI: flannel's moving parts

    Why it matters

    Generic CNI knowledge (4.5) plus one concrete, fully-traced implementation is what makes the knowledge stick — and your lab's flannel is the simplest real CNI there is.

    The concept

    Flannel has exactly three moving parts per node: the flanneld DaemonSet pod (watches the apiserver, learns every node's pod slice, programs the VXLAN device and routes), the subnet.env file it writes (so the CNI plugin knows the node's slice without talking to the apiserver), and the CNI plugin binary (thin delegator to bridge+host-local, 4.5). The bridge cni0 and the VXLAN device flannel.1 are what they create.

    Lab
    1. The daemon — one flannel pod per node: 
      kubectl -n kube-flannel get pods -o wide
      NAME                    READY   STATUS    NODE
kube-flannel-ds-7xk2p   1/1     Running   cp
kube-flannel-ds-9rl4m   1/1     Running   node01
kube-flannel-ds-tw8qn   1/1     Running   node02
    2. The handoff file on node01 — flanneld writes it, the CNI plugin reads it: 
      cat /run/flannel/subnet.env
      FLANNEL_NETWORK=10.244.0.0/16
FLANNEL_SUBNET=10.244.1.1/24
FLANNEL_MTU=1450
FLANNEL_IPMASQ=true

      Note FLANNEL_SUBNET matches node01's spec.podCIDR from 4.4 — that's the chain: controller-manager allocates → flanneld reads the Node object → writes this file → bridge plugin's IPAM uses it. MTU 1450 = 1500 minus 50 bytes of VXLAN header.

    3. The interfaces it built, in detail: 
      ip -d link show flannel.1 | head -3
ip -d link show cni0 | head -3
      4: flannel.1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 ...
    vxlan id 1 local 10.211.64.11 dev ens3 srcport 0 0 dstport 8472 nolearning
5: cni0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 ...
    bridge forward_delay 1500 hello_time 200 ...
    4. The routing table that ties it together: 
      ip route | grep 10.244
      10.244.0.0/24 via 10.244.0.0 dev flannel.1 onlink
10.244.1.0/24 dev cni0 proto kernel scope link src 10.244.1.1
10.244.2.0/24 via 10.244.2.0 dev flannel.1 onlink

      Read it aloud: "my own slice → local bridge; every other node's slice → into the VXLAN tunnel." That one routing table is flannel's whole datapath decision.

    Verify

    For a pod IP on another node, you can trace the full path from memory: pod eth0 → veth → cni0 → route via flannel.1 → VXLAN over ens3 → peer flannel.1 → peer cni0 → veth → pod.

    Gotchas

    The MTU chain matters: pods inherit 1450 from cni0. If something resets an interface to 1500 (or an outer network drops to less), large packets blackhole while pings work — "small requests fine, big responses hang" is an MTU signature, not a policy one.

    Docs

    Docs search: "Cluster Networking". Flannel specifics are on its GitHub (not exam-available — internalize the shape now).

    4.7

    Datapath families: overlay vs routed vs eBPF

    Why it matters

    The exam asks for understanding, not vendor trivia: given a cluster, can you tell how pod traffic actually crosses between nodes, and what that implies for debugging?

    The concept

    Lab

    Prove your cluster is an overlay by catching the encapsulation in the act.

    1. Start a cross-node ping: web4 runs on node01 (4.3); ping it from a pod pinned to node02: 
      kubectl run pinger --image=busybox:1.36 \
  --overrides='{"spec":{"nodeName":"node02"}}' \
  --restart=Never -- ping -i0.5 10.244.1.8
    2. On node01, capture on the physical NIC, filtering for the VXLAN port: 
      sudo tcpdump -ni ens3 udp port 8472 -c2
      10.211.64.12.51384 > 10.211.64.11.8472: OTV, ... VXLAN, vni 1
10.244.2.11 > 10.244.1.8: ICMP echo request ...

      One line, two worlds: the outer packet is node02 → node01 on UDP 8472; tcpdump unwraps it and shows the inner pod-to-pod ICMP. That nesting is an overlay.

    3. Contrast: capture on flannel.1 (inside the tunnel) — only pod IPs, no UDP wrapper: 
      sudo tcpdump -ni flannel.1 -c2 icmp
kubectl delete pod pinger
      10.244.2.11 > 10.244.1.8: ICMP echo request
10.244.1.8 > 10.244.2.11: ICMP echo reply
    Verify

    You saw the same ping twice: wrapped in VXLAN on ens3, naked on flannel.1. Say the routed-mode difference out loud: on Calico-BGP, the ens3 capture would show the pod IPs directly.

    Gotchas

    Linux VXLAN defaults to port 8472, not the IANA-standard 4789 — firewalls opened for 4789 silently break flannel. And remember for 4.23: flannel does not implement NetworkPolicy at all — datapath family and policy enforcement are separate capabilities, both belonging to the CNI.

    4.8

    Break the CNI on purpose and read the signatures

    Why it matters

    This exact fault — CNI config missing/broken on one node — appears in troubleshooting tasks because it produces confusing, asymmetric symptoms. Plant it yourself once and the symptoms become a signature you read in ten seconds.

    Lab
    1. On node01, hide the CNI config and bounce the kubelet: 
      sudo mv /etc/cni/net.d/10-flannel.conflist /root/
sudo systemctl restart kubelet
    2. Watch the node condition flip (takes under a minute): 
      kubectl get node node01
kubectl describe node node01 | grep -A1 'Ready '
      NAME     STATUS     ROLES    AGE   VERSION
node01   NotReady   <none>   25d   v1.35.x
  Ready   False ... KubeletNotReady   container runtime network not ready:
  NetworkReady=false reason:NetworkPluginNotReady message:Network plugin
  returns error: cni plugin not initialized

      Memorize that message — it's verbatim what the exam shows you.

    3. Existing pods on node01 keep running (their veths are already wired). New pods pinned there stick: 
      kubectl run stuck --image=nginx --overrides='{"spec":{"nodeName":"node01"}}' --restart=Never
kubectl get pod stuck
kubectl describe pod stuck | tail -3
      stuck   0/1     ContainerCreating   0     30s
  Warning  FailedCreatePodSandBox ... failed to setup network for sandbox:
  plugin type="flannel" failed (add): open /run/flannel/subnet.env: no such file...
  — or: no valid networks found in /etc/cni/net.d

      Note the split: unscheduled pods would still land here (NotReady doesn't stop the scheduler by itself — the node gets node.kubernetes.io/not-ready:NoSchedule via taints), but sandbox creation fails at the CNI ADD.

    4. Fix it and watch everything self-heal: 
      sudo mv /root/10-flannel.conflist /etc/cni/net.d/
sudo systemctl restart kubelet
kubectl get node node01 -w
      node01   NotReady   <none>   25d   v1.35.x
node01   Ready      <none>   25d   v1.35.x
    5. Clean up: 
      kubectl delete pod stuck --force --grace-period 0
    Verify

    Node back to Ready, and a fresh kubectl run test --image=nginx pod reaches Running on node01. Delete it after.

    Gotchas

    Triage order for this family: kubectl describe node (the condition message names CNI explicitly) → ls /etc/cni/net.d/ on that node → ls /opt/cni/bin/ → flannel pod logs. Don't start at iptables or CoreDNS: node NotReady + ContainerCreating points below Services and DNS, at the CNI layer.

    Docs

    Docs search: "Troubleshooting Clusters" and "Network Plugins".

    Services & kube-proxy

    4.9

    A ClusterIP is a virtual IP — it exists only as rewrite rules

    Why it matters

    Once you truly believe "the service IP is not on any interface, anywhere", a whole class of confusion dissolves: why you can't ping services, why services fail differently than pods, why "it resolves but doesn't connect" points at kube-proxy.

    The concept

    When you create a Service, the apiserver picks a free IP from the service CIDR and stores it in etcd. Nothing else happens — no interface is created, no route is announced. On every node, kube-proxy watches Services and EndpointSlices and writes netfilter rules: "a packet to 10.101.x.x:80? Rewrite its destination (DNAT) to one of these pod IPs". The illusion of a stable service IP is manufactured independently on each node at the moment a packet leaves a socket. Kill kube-proxy and existing rules keep working (they live in the kernel) but drift as pods churn — a uniquely sneaky failure mode.

    Lab
    1. Create a deployment and a service for it: 
      kubectl create deploy web --image=nginx --replicas=2
kubectl expose deploy web --port 80
kubectl get svc web
      NAME   TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)
web    ClusterIP   10.101.44.7    <none>        80/TCP
    2. Hunt for that IP on a node — it isn't anywhere: 
      ip addr | grep 10.101.44.7 ; ip route get 10.101.44.7
      (no interface match)
10.101.44.7 via 10.211.64.1 dev ens3 src 10.211.64.11
    — the routing table would send it to the default gateway!
    3. And yet, from the same node, it works: 
      curl -s 10.101.44.7 | grep -o '<title>.*</title>'
ping -c1 -W1 10.101.44.7
      <title>Welcome to nginx!</title>
1 packets transmitted, 0 received, 100% packet loss

      curl works because the DNAT rule matches tcp dpt:80 and rewrites the packet before routing happens. ping fails because there is no rule for ICMP — nothing answers, because nothing owns the IP. An unpingable-but-curlable IP is the fingerprint of a virtual IP.

    Verify

    curl to the ClusterIP succeeds while ping to it gets 100% loss — and you can explain why in one sentence.

    Gotchas

    Never use ping to test a Service — use curl/wget to the actual port, or nc -zv. In the exam, "I can't ping the service" is expected behavior, not the fault.

    Docs

    Docs search: "Virtual IPs and Service Proxies" — the page that explains everything this group covers.

    4.10

    Walk the iptables chains behind one of your services

    Why it matters

    You'll likely never edit these rules — but reading them turns kube-proxy from a black box into three greppable chain names, and lets you prove "the rules for this service exist / don't exist" during triage.

    The concept

    kube-proxy in iptables mode (still the default; IPVS and the newer nftables backend — GA since 1.33 — exist for scale) is a rule compiler, not a proxy: no packet passes through the kube-proxy process. It renders Services + EndpointSlices into three layers of nat-table chains, and conntrack pins each TCP flow to the backend chosen for its first packet:

    Lab
    1. Give the walk three backends to find: 
      kubectl scale deploy web --replicas=3
    2. Layer 1 — find your service in KUBE-SERVICES (on any node; kube-proxy tags every rule with a comment): 
      sudo iptables -t nat -L KUBE-SERVICES -n | grep default/web
      KUBE-SVC-LOLE4ISW44XBNF3G  tcp -- 0.0.0.0/0  10.101.44.7  /* default/web cluster IP */  tcp dpt:80
    3. Layer 2 — the per-service chain and its probabilistic split: 
      sudo iptables -t nat -L KUBE-SVC-LOLE4ISW44XBNF3G -n
      KUBE-SEP-XVQ...  all -- ... /* default/web -> 10.244.1.8:80 */  statistic mode random probability 0.33333333349
KUBE-SEP-ZAB...  all -- ... /* default/web -> 10.244.2.4:80 */  statistic mode random probability 0.50000000000
KUBE-SEP-QRT...  all -- ... /* default/web -> 10.244.2.5:80 */

      Read the cascade: ⅓ chance of the first, else ½ of the remaining two, else the last — uniform ⅓ each. This is the load balancer: three lines of probability.

    4. Layer 3 — one endpoint chain, where the actual rewrite happens: 
      sudo iptables -t nat -L KUBE-SEP-XVQ... -n
      KUBE-MARK-MASQ  all  --  10.244.1.8  0.0.0.0/0   /* hairpin: pod dialing itself via the svc */
DNAT   tcp ... to:10.244.1.8:80
    5. See conntrack pinning a flow (install once: sudo apt-get install -y conntrack): 
      curl -s 10.101.44.7 >/dev/null; sudo conntrack -L 2>/dev/null | grep 10.101.44.7
      tcp ... src=10.211.64.11 dst=10.101.44.7 dport=80 ... src=10.244.2.4 sport=80  [ASSURED]

      Both truths in one line: the flow was addressed to the service IP, and conntrack knows it really talks to a pod.

    Verify

    You followed one service through all three layers, and the pod IPs in the KUBE-SEP chains match kubectl get pods -l app=web -o wide.

    Gotchas

    A service with no ready endpoints compiles to a REJECT rule — "connection refused" instantly. No rules at all for the IP (kube-proxy dead) means packets route out the default gateway and you get a timeout instead. Refused vs timeout is a diagnostic fork, not noise. Also don't hand-edit KUBE-* chains: kube-proxy owns and rewrites them.

    Docs

    Docs search: "Virtual IPs and Service Proxies" — includes the proxy-mode comparison (iptables / ipvs / nftables).

    4.11

    EndpointSlices: readiness decides who gets traffic

    Why it matters

    kubectl get endpointslices is the single highest-yield command for "service not working" tasks: it answers, in one look, whether the service selects anything, whether those pods are ready, and what IPs traffic actually goes to.

    The concept

    The EndpointSlice controller watches Services and Pods: for each Service it finds pods matching the selector and writes their IPs into EndpointSlice objects (sliced at 100 endpoints each, for scale), marking each endpoint with the pod's readiness. kube-proxy compiles only ready endpoints into forwarding rules. So a failing readinessProbe doesn't just decorate kubectl get pods — it's the mechanism that pulls a pod out of every service that selects it. The chain: selector → slice membership; readiness → traffic.

    Lab
    1. Look at the slice behind your service:

    4.13

    Headless services: DNS straight to pod IPs

    Why it matters

    Headless services are the machinery behind StatefulSet identity (W3) and the standard answer to "the client wants to pick its own backend" (databases, peer discovery). The exam likes them because they test whether you know what a ClusterIP actually does — by taking it away.

    The concept

    clusterIP: None means: no virtual IP, no kube-proxy rules, no load balancing. The service becomes only a DNS name plus endpoint tracking: CoreDNS answers the service name with one A record per ready pod. Clients get the whole list and connect to pod IPs directly. Behind a StatefulSet, each pod additionally gets a stable per-pod name: <pod>.<svc>.<ns>.svc.cluster.local.

    Lab
    1. Create a headless twin of your service: 
      kubectl expose deploy web --name=web-hl --cluster-ip=None --port=80

web-hl   ClusterIP   None          <none>   80/TCP
    2. Ask DNS for both and compare (dnsutils pod from 4.17, or any busybox): 
      kubectl run tmpq --image=busybox:1.36 --restart=Never --rm -it -- sh -c \
  'nslookup web; nslookup web-hl'
      Name: web.default.svc.cluster.local
Address: 10.101.44.7              ← one answer: the virtual IP

      web-hl-x2x9c   IPv4   80   10.244.1.8,10.244.2.4,10.244.2.5

    The normal service resolves to exactly one (virtual) IP; the headless one resolves to as many IPs as there are ready pods — and readiness (4.11) gates the DNS answers the same way it gates DNAT rules.

    Gotchas

    clusterIP: None must be set at creation — you can't edit a normal service into headless (the field is immutable; delete and recreate). And since there's no proxying, there's also no port remapping: clients hit the pods' real ports.

    4.14

    port vs targetPort vs nodePort — and the most-planted service bug

    Why it matters

    Almost every "service doesn't work" exam task is one of these three numbers being wrong. Knowing which is which — and which mistake produces which symptom — makes the fix a 30-second edit.

    The concept
    FieldWhose portSeen byDefault
    portthe Service's own port on its ClusterIPclients dialing the servicerequired
    targetPortthe container's port — where DNAT actually sends trafficthe poddefaults to port
    nodePortthe port opened on every node (NodePort/LB types)external clientsauto-assigned 30000–32767

    targetPort can be a name: if the container declares ports: [{name: http, containerPort: 8080}], then targetPort: http follows whatever each pod calls http — the indirection that lets you change container ports without touching services.

    Lab
    1. Plant the classic bug — targetPort pointing where nothing listens: 
      kubectl expose deploy web --name=web-broken --port=80 --target-port=8080
curl -s --max-time 3 $(kubectl get svc web-broken -o jsonpath='{.spec.clusterIP}')
      curl: (7) Failed to connect to 10.99.203.1 port 80: Connection refused
    2. Read the symptom precisely — endpoints exist, but carry the wrong port: 
      kubectl get endpointslices -l kubernetes.io/service-name=web-broken -o jsonpath='{.items[0].ports}'
      [{"name":"","port":8080,"protocol":"TCP"}]   ← pods are listed, DNAT goes to :8080, nginx listens on :80

      Distinguish the refusals: empty endpoints (selector bug, 4.11) vs present-but-wrong-port endpoints (this bug). Same "connection refused", different fix.

    3. Fix it live: 
      kubectl patch svc web-broken -p '{"spec":{"ports":[{"port":80,"targetPort":80}]}}'
curl -s $(kubectl get svc web-broken -o jsonpath='{.spec.clusterIP}') | grep -o 'Welcome to nginx'
      Welcome to nginx
    4. Clean up: 
      kubectl delete svc web-broken
    Verify

    You can recite the diagnostic: refused + endpoints listed with an unexpected port ⇒ targetPort ≠ containerPort; refused + no endpoints ⇒ selector; timeout ⇒ look lower (kube-proxy, NetworkPolicy, CNI).

    Gotchas

    The mirrored trap: containerPort in a pod spec is documentation only — nginx listens on 80 whether or not the manifest declares it. A wrong containerPort breaks nothing… unless a service uses a named targetPort resolved through it. Check what the process actually binds, not what the YAML claims.

    4.16

    A pod's resolv.conf, line by line — and the ndots:5 ladder

    Why it matters

    Every DNS behavior that surprises people — why bare service names work, why external lookups are slow from pods, why hostNetwork pods can't resolve services — is written in this three-line file.

    The concept

    The kubelet writes /etc/resolv.conf into every pod according to the pod's dnsPolicy. The default (ClusterFirst) yields:

    search default.svc.cluster.local svc.cluster.local cluster.local
nameserver 10.96.0.10
options ndots:5
    • nameserver — the kube-dns ClusterIP (4.4). All pod DNS goes to the service VIP, through the same DNAT machinery as any other service.
    • search — three suffixes, most-specific first, starting with the pod's own namespace.
    • ndots:5 — a name with fewer than 5 dots is tried against the search list before being tried as-is.

    The resolution ladder for a bare name from namespace default:

    db→db.default.svc.cluster.local→db.svc.cluster.local→db.cluster.local→db.

    First rung that answers wins — that's why db finds a service in your own namespace and db.other finds one in namespace other (second rung: db.other.svc.cluster.local). The cost: www.example.com has only 2 dots, so it climbs the whole ladder — three guaranteed-NXDOMAIN cluster queries — before the real lookup. Appending a trailing dot (www.example.com.) makes it absolute and skips the ladder.

    Lab
    1. Read a pod's file: 
      kubectl exec web4 -- cat /etc/resolv.conf
      search default.svc.cluster.local svc.cluster.local cluster.local
nameserver 10.96.0.10
options ndots:5
    2. Watch the ladder climb, live — turn on query logging and make a low-dot external lookup: 
      kubectl -n kube-system get cm coredns -o yaml | sed 's/    errors/    errors\n    log/' | kubectl apply -f -
kubectl -n kube-system rollout restart deploy coredns   # skip the ~2min reload wait
kubectl run tmpq --image=busybox:1.36 --restart=Never --rm -it -- nslookup example.com
kubectl -n kube-system logs deploy/coredns | grep example.com | head -4
      [INFO] ... "A IN example.com.default.svc.cluster.local." NXDOMAIN ...
[INFO] ... "A IN example.com.svc.cluster.local." NXDOMAIN ...
[INFO] ... "A IN example.com.cluster.local." NXDOMAIN ...
[INFO] ... "A IN example.com." NOERROR ...

      The ladder, verbatim, in the logs. Remove the log line the same way when done.

    3. Know the four dnsPolicy values (the names lie a little):
      dnsPolicyMeaning
      ClusterFirstthe default above — despite the name, it's not "try cluster first", it's this resolv.conf
      ClusterFirstWithHostNetsame file, for hostNetwork: true pods — which otherwise get the node's resolv.conf
      Defaultinherit the node's resolv.conf (confusingly named — it is not the default)
      Noneyou supply everything via dnsConfig
    Verify

    Predict, then confirm with the log trick: how many queries does resolving db.other take if the service exists? (Two-part name, first rung db.other.default.svc... fails, second rung db.other.svc.cluster.local answers.)

    Gotchas

    The hostNetwork trap is a planted-fault favorite: a hostNetwork: true pod with default policy can't resolve *.svc.cluster.local at all — fix is one line, dnsPolicy: ClusterFirstWithHostNet. Recognize the symptom: only host-network pods failing DNS while normal pods are fine.

    4.18

    Break DNS twice and drill the debug ladder

    Why it matters

    "Pods can't resolve names" is a guaranteed troubleshooting family. There are only about five places DNS can break; walking them in order beats every intuition.

    The concept

    The ladder, bottom of the stack to top:

    1 pod resolv.conf sane?→2 kube-dns svc + endpoints exist?→3 can I query 10.96.0.10 directly?→4 CoreDNS pods running? logs?→5 Corefile valid?

    Step 3 is the pivot: if a direct dig @10.96.0.10 works but normal resolution doesn't, the problem is the pod's resolv.conf (step 1 lied to you). If the direct query hangs, it's the service path (endpoints, kube-proxy, CNI) or CoreDNS itself (4–5).

    Lab

      kubectl -n kube-system scale deploy coredns --replicas=0

      kubectl exec dnsutils -- cat /etc/resolv.conf | head -2

      kubectl -n kube-system get cm coredns -o yaml | sed 's/    errors/    errors\n    bogus/' | kubectl apply -f -

      coredns-7c65d6cfc9-k2m4x   0/1   CrashLoopBackOff   2 (18s ago)

      kubectl -n kube-system logs -l k8s-app=kube-dns --tail=3

    Both faults produced different first symptoms (timeout vs CrashLoopBackOff) and you reached each root cause by ladder, not by flailing. Clean up: kubectl delete pod dnsutils web4 if you're done with them.

    A subtle third fault to know by signature: DNS works for cluster names but external names fail ⇒ the forward upstream is broken (node resolv.conf, or an edited forward line) — cluster answers come from API watches and never touch upstream. The reverse (external fine, cluster names dead) points at the kubernetes plugin's RBAC/zones or a wrong nameserver IP in pods.

    4.20

    Gateway API: the resource model and its role split

    Why it matters

    Gateway API is the newest thing on the CKA curriculum, which makes it predictable exam material — and most candidates have never touched it. Thirty minutes here is disproportionate points.

    The concept

    Gateway API splits "get traffic into the cluster" into three resources, deliberately matching three roles:

    Standard-channel resources at gateway.networking.k8s.io/v1: GatewayClass, Gateway, HTTPRoute, GRPCRoute (plus ReferenceGrant at v1beta1 for cross-namespace backend permissions). They're CRDs — installed separately, not shipped with Kubernetes; that's why a fresh cluster answers kubectl get gateway with "the server doesn't have a resource type".

    Lab
    1. Install the standard-channel CRDs (v1.6.0 current as of this writing): 
      kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.6.0/standard-install.yaml

gateways.gateway.networking.k8s.io          ...
grpcroutes.gateway.networking.k8s.io        ...
httproutes.gateway.networking.k8s.io        ...
referencegrants.gateway.networking.k8s.io   ...
    2. Turn on Traefik's Gateway provider — a Helm upgrade of your 4.19 release (--reuse-values keeps the NodePorts): 
      helm upgrade traefik traefik/traefik --namespace traefik --reuse-values \
  --set providers.kubernetesGateway.enabled=true \
  --set gateway.listeners.web.namespacePolicy.from=All

      The second flag sets the Gateway's allowedRoutes so routes from any namespace may attach — by default a Gateway only accepts routes from its own namespace, and forgetting that is the #1 "my HTTPRoute is ignored" cause.

    3. See what the chart created — one GatewayClass, one Gateway with a listener on port 80: 
      kubectl get gatewayclass

    GatewayClass shows ACCEPTED True, Gateway shows PROGRAMMED True, and you can name which role owns each of the three resources without looking.

    4.21

    Write an HTTPRoute from memory — including a weighted canary

    Why it matters

    The HTTPRoute is the object an exam task will actually make you write. Its killer feature over Ingress — first-class traffic weights — is also the most likely thing to be asked, because Ingress simply can't do it.

    The concept

    An HTTPRoute has three parts: parentRefs (which Gateway to attach to), hostnames, and rules — each rule being matches (path/header/query) plus backendRefs (services, each with an optional weight). Weights split traffic proportionally: 90/10 below means one request in ten goes to the canary.

    Lab
    1. A second backend that answers differently (Apache says "It works!", nginx says "Welcome to nginx"): 
      kubectl create deploy web-v2 --image=httpd
kubectl expose deploy web-v2 --port 80
    2. The route — write it by hand once, then check it against this: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: shop-canary
spec:
  parentRefs:
  - name: traefik-gateway
    namespace: traefik
  hostnames: ["gw.lab.test"]
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: web
      port: 80
      weight: 90
    - name: web-v2
      port: 80
      weight: 10
EOF
kubectl describe httproute shop-canary | grep -A4 Conditions:
      Conditions:
  Type: Accepted     Status: True
  Type: ResolvedRefs Status: True
    3. Prove the split — 50 requests through the Gateway's listener (the web NodePort from 4.19): 
      for i in $(seq 1 50); do curl -s http://node01:30080/ -H 'Host: gw.lab.test'; done \
  | grep -o 'Welcome to nginx\|It works' | sort | uniq -c
           5 It works
    45 Welcome to nginx   ← ≈90/10; exact counts wobble, the ratio doesn't lie
    4. Know that header matching exists — add a rule that routes testers to the canary regardless of weights: 
          - matches:
      - path: { type: PathPrefix, value: / }
        headers:
        - name: X-Canary
          value: "yes"
      backendRefs:
      - { name: web-v2, port: 80 }

      (Matches in one rule AND together; separate rules OR. More-specific rules win — the same AND/OR shape you'll meet again in NetworkPolicy, 4.25.)

    Verify

    The 50-request loop lands ~45/5, and describe httproute shows Accepted: True. Try breaking parentRefs.name on purpose and watch the condition flip — that's the debugging signature to remember.

    Gotchas

    Two silent-failure classics: parentRefs must name the Gateway's namespace when the route lives elsewhere (default is the route's own namespace); and cross-namespace backends additionally need a ReferenceGrant in the backend's namespace — without it, ResolvedRefs: False and 500s.

    Docs

    Docs search: "Gateway API" — the HTTPRoute example on that page is the template to memorize.

    4.22

    Say why Gateway API exists — the two-minute answer

    Why it matters

    Understanding-type questions ("which resource would you use to…") are cheap points if you hold the comparison crisply — and the reasoning tells you which tool a task wants even when it doesn't name one.

    The pitch
    • Ingress hit its ceiling. One object, one host/path → service vocabulary. Everything real — timeouts, rewrites, TLS options, canary weights — became controller-specific annotations: untyped strings, unvalidated, non-portable ("annotation soup"). Ingress is GA and frozen — it works forever, but gains nothing new.
    • Gateway API is typed and role-split. Weights, header matches, traffic filters are real schema'd fields; GatewayClass/Gateway/HTTPRoute separate infra, platform and app concerns (4.20); routes are namespaced and RBAC-able per team; every resource reports status conditions instead of failing silently.
    • The ecosystem has moved. ingress-nginx — the default choice for a decade — is retired (March 2026); its successors and every modern data plane (Envoy Gateway, Cilium, Traefik…) put Gateway API first. New clusters start there; Ingress remains for the installed base.
    • Exam angle: Gateway API entered the curriculum in the 2025 revision, killer.sh and the real exam both feature it, and most study material predates it — genuinely easy points for the prepared.
    Verify

    Deliver the four bullets out loud in under two minutes to an imaginary colleague, then answer their follow-up: "so should we migrate our existing Ingresses?" (No rush — Ingress keeps working; new routes and anything needing weights/typed config go to Gateway API.)

    Gotchas

    Don't say "Ingress is deprecated" — it isn't, and precision is the point of this checkpoint. It's frozen: stable, supported, not growing. The pressure to move comes from controller retirements and missing features, not an API removal.

    Docs

    Docs search: "Gateway API" and "Ingress" — the Ingress page itself points migrators at Gateway API.

    4.23

    The NetworkPolicy model: select, default-deny, additive allows

    Why it matters

    NetworkPolicy questions are pure model questions — the YAML is short, and every mistake comes from misunderstanding one of three rules below. Nail the model and the YAML writes itself.

    The concept

    Three rules carry the whole thing:

    • Policies select pods, not traffic. spec.podSelector picks the pods this policy protects, within the policy's namespace. podSelector: {} selects every pod in the namespace.
    • Selection flips the default. An unselected pod accepts everything (Kubernetes is open by default). The moment any policy selects a pod, that pod becomes default-deny for the directions listed in policyTypes — and only those directions.
    • Rules are additive allows. ingress:/egress: entries punch holes in that deny. There is no deny rule — you cannot write "block X"; you write "allow only Y" and let the default-deny do the blocking. Multiple policies on one pod union their allows.

    The precision trap lives in policyTypes: an empty rules list (ingress: [] — or the key absent) with policyTypes: [Ingress] allows nothing in — full ingress lockdown. But writing only egress rules and omitting policyTypes means it defaults to just [Egress] — ingress stays wide open, which may not be what the task asked. State your policyTypes explicitly, always.

    And the enforcement asterisk: the CNI plugin implements NetworkPolicy — not kube-proxy, not the kernel by default. Calico and Cilium enforce; flannel ignores policies entirely — they apply cleanly and do nothing. The API accepting your object says nothing about enforcement.

    Verify

    Answer without notes: a pod in namespace a is selected by one policy with policyTypes: [Ingress] and ingress: []. Can it still reach out to the internet? (Yes — egress was never restricted.) Can anything reach it? (No — not even pods in its own namespace.)

    Gotchas

    Policies affect connection establishment, not replies: allowing ingress to a pod implicitly allows its responses back out (connection tracking) — you don't need a matching egress rule on the server for return traffic. But a client pod with locked-down egress does need an egress allow to initiate. Think in connections, not packets.

    Docs

    Docs search: "Network Policies" — the concepts page; its "behavior of to/from selectors" section is 4.25's material.

    4.25

    The AND/OR trap in from/to rules — test both

    Why it matters

    This is the single most-tested NetworkPolicy subtlety, because two YAMLs differing by one dash mean opposite things — and both apply cleanly.

    The concept

    Inside one from: element, selectors combine with AND (all must match the same peer). Separate from: elements combine with OR:

    # A — ONE from element (no dash before podSelector): AND
ingress:
- from:
  - namespaceSelector:
      matchLabels: { team: qa }
    podSelector:
      matchLabels: { app: client }
# meaning: pods labeled app=client IN namespaces labeled team=qa. Only them.

# B — TWO from elements (note the extra dash): OR
ingress:
- from:
  - namespaceSelector:
      matchLabels: { team: qa }
  - podSelector:
      matchLabels: { app: client }
# meaning: ANY pod in team=qa namespaces, PLUS app=client pods
# from the policy's OWN namespace.

    Second subtlety inside variant B: a bare podSelector in a from rule always means "in this policy's namespace" — it does not scan other namespaces.

    Lab

    (Continues on the Calico cluster from 4.24, with default-deny-ingress still in place. Delete allow-client-to-web and allow-monitoring-ns first so they don't add extra holes: kubectl delete netpol allow-client-to-web allow-monitoring-ns.)

    1. Build the test matrix — three probers: 
      kubectl create ns qa && kubectl label ns qa team=qa
kubectl -n qa run qa-client --image=busybox:1.36 -l app=client --restart=Never -- sleep 1d
kubectl -n qa run qa-other  --image=busybox:1.36 --restart=Never -- sleep 1d
# plus 'client' (app=client, default ns) from 4.24
    2. Apply variant A (AND) targeting web, then probe all three: 
      kubectl -n qa exec qa-client -- wget -qO- -T2 http://web.default >/dev/null && echo qa-client:OK || echo qa-client:BLOCKED
kubectl -n qa exec qa-other  -- wget -qO- -T2 http://web.default >/dev/null && echo qa-other:OK  || echo qa-other:BLOCKED
kubectl exec client          -- wget -qO- -T2 http://web         >/dev/null && echo client:OK    || echo client:BLOCKED
      qa-client:OK        ← matches BOTH selectors
qa-other:BLOCKED     ← right namespace, wrong pod label
client:BLOCKED       ← right label, wrong namespace
    3. Swap in variant B (OR — add the dash), re-run the same three probes: 
      qa-client:OK         ← matches the namespace leg
qa-other:OK          ← matches the namespace leg too
client:OK            ← matches the podSelector leg (own namespace)
    4. Clean up and restore flannel (step 8 of 4.24): 
      kubectl delete ns qa monitoring; kubectl delete netpol --all; kubectl delete pod client
    Verify

    The probe matrix flipped exactly as predicted between A and B — 1/3 allowed vs 3/3 allowed — and you can point at the dash that did it.

    Gotchas

    In the exam, indentation decides this. When copying a policy from the docs, check whether podSelector under from has its own -. Reading the rule back out loud — "from peers that are (X and Y)" vs "from X-peers, or Y-peers" — catches the error faster than staring at YAML.

    Docs

    Docs search: "Network Policies" — the concepts page shows this exact AND/OR pair side by side; find it once now so you can find it in three seconds then.

    Week 4 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. The Kubernetes network model in two rules — and which common IP type is outside the model?
      Answer

      Every pod gets its own IP; all pods reach all pods without NAT. Service ClusterIPs are outside it — virtual, rule-manufactured, on no interface (4.1/4.9).

    2. What three Linux objects make up one pod's network plumbing, and who keeps the netns alive?
      Answer

      A network namespace, a veth pair (one end renamed eth0 inside, the other on the node bridge), and the bridge (cni0). The pause container holds the namespace open (4.2).

    3. Node NotReady, message "cni plugin not initialized". Which two directories do you check, and why do existing pods keep working?
      Answer

      /etc/cni/net.d (config) and /opt/cni/bin (binaries). Existing pods keep their already-wired veths — CNI is exec'd per pod creation, so only new sandboxes fail (4.5/4.8).

    4. You tcpdump the node NIC and can't see pod IPs, only node-to-node UDP on port 8472. What are you looking at?
      Answer

      A VXLAN overlay (flannel) — pod packets are encapsulated between nodes; Linux VXLAN defaults to 8472, not IANA's 4789. Capture on flannel.1 to see the inner pod traffic (4.7).

    5. Why does curl to a ClusterIP work while ping to it shows 100% loss?
      Answer

      The IP exists only as nat-table rules matching tcp dpt:80; DNAT rewrites the curl packet to a pod IP. ICMP matches no rule and nothing owns the IP, so ping dies. Never test services with ping (4.9).

    6. Walk the three iptables layers for a service, and name what pins a TCP flow to one backend.
      Answer

      KUBE-SERVICES (match ClusterIP:port) → KUBE-SVC-* (statistic-module random split) → KUBE-SEP-* (DNAT to podIP:port). conntrack pins the flow. kube-proxy compiles the rules; it proxies nothing (4.10).

    7. Service test: "connection refused" vs "timeout" — what does each point at?
      Answer

      Refused = something answered: no-ready-endpoints REJECT rule, or DNAT to a port nothing listens on (targetPort bug). Timeout = nothing answered: missing rules (kube-proxy), NetworkPolicy drop, or CNI/routing (4.10/4.14/4.24).

    8. What exact chain of events removes a pod from service load-balancing when its readinessProbe fails?
      Answer

      Probe fails → kubelet marks the pod not-ready → EndpointSlice controller sets that endpoint ready: false → kube-proxy recompiles rules without it (and CoreDNS drops it from headless answers). Selector picks members; readiness gates traffic (4.11/4.13).

    9. port, targetPort, nodePort — whose port is each, and which one is "the most-planted service bug"?
      Answer

      port = the service's own; targetPort = the container's (DNAT destination, defaults to port, may be a name); nodePort = 30000–32767 on every node. The classic bug: targetPort ≠ what the container listens on — endpoints present, port wrong, connection refused (4.14).

    10. Why does a bare name like db resolve, and why is ndots:5 a tax on external lookups?
      Answer

      Names with <5 dots try the search list first: db.<ns>.svc.cluster.local → db.svc.cluster.local → db.cluster.local → db.. External names like www.example.com (2 dots) climb the same ladder — three NXDOMAINs before the real query; a trailing dot skips it (4.16).

    11. Only hostNetwork pods can't resolve *.svc.cluster.local. Diagnosis and the one-line fix?
      Answer

      hostNetwork pods inherit the node's resolv.conf under default policy. Fix: dnsPolicy: ClusterFirstWithHostNet (4.16).

    12. An Ingress object exists but "does nothing", silently. Name the two most likely causes and your first command.
      Answer

      No ingress controller running, or wrong/missing ingressClassName. First: kubectl get ingressclass (then check controller pods). Ingress is only data — a controller must render it (4.19).

    13. Gateway API's three resources — who owns each, and how does an HTTPRoute attach?
      Answer

      GatewayClass (implementation — infra), Gateway (listeners/TLS — platform), HTTPRoute (matching + backendRefs — app team), attached via parentRefs and admitted by the Gateway's allowedRoutes. Status conditions (Accepted/Programmed/ResolvedRefs) replace Ingress's silence (4.20/4.21).

    14. One from: element with namespaceSelector + podSelector vs two elements — and what does a policy with policyTypes: [Ingress], ingress: [] do?
      Answer

      One element = AND (both must match the same peer); two elements = OR (either suffices; the bare podSelector leg means the policy's own namespace). ingress: [] + selection = full ingress default-deny for the selected pods — egress untouched. And remember flannel enforces none of it; policy enforcement belongs to the CNI (4.23–4.25).
