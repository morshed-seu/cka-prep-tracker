    Week 4 — Services & networking, from veth up

    Kubernetes networking has a bad reputation because most of it is invisible: a pod gets an IP from nowhere you can point to, a Service works even though it lives on no interface at all, and a name like web resolves to something without you ever configuring a DNS server. Networking is where most candidates guess, because guessing is what's left when the machinery is hidden. This week removes the hiding. You build a pod network by hand with plain Linux ip commands so you've personally done what the network plugin does automatically; you read the actual firewall rules a Service compiles down to; you walk a DNS lookup one step at a time instead of trusting that "DNS just works"; and you wire up both of the two ways to route outside traffic into the cluster — the older Ingress and the newer Gateway API — against a real, running controller. Each layer in this week sits directly on top of the previous one, so once you can name the exact layer a packet died in, every networking task on the exam becomes mechanical instead of mysterious. Ticking done here also ticks the tracker.

    Linux primitives→CNI→Services / kube-proxy→DNS→Ingress · Gateway API→NetworkPolicy

    Linux networking foundations

    4.1

    The network model: two rules, everything else is implementation

    Why it matters

    Kubernetes networking looks like a pile of unrelated topics — pods, Services, DNS, Ingress — but underneath all of it sits one small contract that never changes, and every networking exam task is really just "one of the two rules in that contract is being violated — find where." If you know what the model guarantees, you know exactly what a healthy cluster must be able to do, and every check this week is really testing one of those two guarantees.

    The concept

    Kubernetes imposes exactly two requirements on whatever network plugin you run — nothing more:

    • Every pod gets its own IP address. Containers inside one pod share that single IP (that's what it means for them to share the pod's network namespace — an isolated, private copy of the kernel's networking stack, covered in depth in 4.2) and talk to each other simply over localhost, exactly like two processes on the same machine.
    • All pods can reach all pods, on every node, without NAT (NAT, meaning no rewriting of addresses in flight). The IP a pod believes is its own is the exact same IP everyone else uses to reach it — nobody translates it into something else along the way. Node agents (kubelet's health probes, node-local daemons) can reach every pod on their node under that same rule.

    That's the entire contract — two sentences, and everything else is just implementation detail. Overlay tunnels, BGP-advertised routes, eBPF programs in the kernel — those are simply different engineering choices a network plugin can make to deliver the exact same two guarantees (compared side by side in 4.7). Everything built on top of the model — Services, DNS names, Ingress — is machinery layered over these flat, NAT-free pod IPs: a ClusterIP is a rewrite rule that ultimately points at a pod IP (4.9), and a DNS record is just a human-friendly name for one (4.17). NAT only shows up at the edges of this picture: traffic leaving the cluster for the outside world gets masqueraded (the node's IP is substituted in), and NodePort traffic gets rewritten again on the way in. Inside the pod network itself, addresses stay exactly as they are.

    Verify

    State from memory: what does a pod on node01 see as the source IP of a packet arriving from a pod on node02? (Answer: the other pod's real pod IP, untouched — no NAT anywhere in between. If you ever see the node's IP instead of the sending pod's own IP, that's a signal the CNI is misconfigured, or that the traffic somehow left the pod network partway through.)

    Gotchas

    Notice what the model deliberately says nothing about: Service IPs. Those are virtual — they don't live in any namespace, aren't attached to any interface, and very much do involve rewriting packets (4.9–4.10), which sounds like a contradiction of the "no NAT" rule until you realize it's a completely separate layer sitting above the model, not a violation of it. Keep two separate mental buckets — "pod IP = a real address somewhere" and "service IP = a set of rewrite rules with no address behind it" — because collapsing them into one bucket is the root of almost all DNS-versus-Service confusion later this week.

    Docs search: "Cluster Networking" — the model statement is at the top of that page.

    4.2

    Build a pod network by hand: netns, veth pair, bridge

    Why it matters

    Right now, "a pod gets its own IP" is a fact you're taking on faith. This 15-minute exercise turns it into something you built yourself with three plain Linux commands — because it is, almost verbatim, what the bridge CNI plugin does for every pod that gets created on a real cluster. After doing it once by hand, ip addr, ip route, and ip netns exec stop being incantations you copy from a cheat sheet and become debugging tools you reach for on instinct, because you know exactly what each piece of the picture is for.

    The concept

    Three Linux building blocks, combined, are the whole trick behind a pod network:

    • A network namespace (often shortened to netns) is an isolated, private copy of the kernel's entire networking stack: its own network interfaces, its own routing table, its own firewall rules — a process inside one netns cannot see or touch the networking of any other. A pod, underneath everything else Kubernetes adds on top, is a network namespace: it's created and kept alive by a special, otherwise-idle pause container, and every application container in the pod simply joins that same already-existing namespace rather than getting one of its own — which is exactly why containers in one pod can talk to each other over localhost, as noted in 4.1.
    • A veth pair (virtual ethernet pair) is best pictured as a virtual patch cable: two interfaces that are always created together as a linked set, where a packet sent into one end instantly comes out the other, and nothing else. One end of that cable gets moved into the pod's namespace and renamed eth0 (so it looks, from inside the pod, like an ordinary network card); the other end is left behind in the node's own default namespace (called the "root" namespace).
    • A bridge is a virtual Ethernet switch that lives in the node's root namespace. Every pod's veth pair has its host-side end plugged into this one bridge, the same way you'd plug several devices into a physical network switch — which is how pods on the same node reach each other (through the bridge), and how the node itself reaches its own pods (through the bridge's own IP address).
    Lab

    Run this on a scratch VM, or directly on node01 — the names chosen below don't clash with anything Kubernetes itself created, and the last step removes everything again so nothing is left behind.

    1. Create two pretend "pods" (really just two namespaces, one called red and one called blue) plus the "node bridge" they'll both plug into: 
      sudo ip netns add red
sudo ip netns add blue
sudo ip link add br-lab type bridge
sudo ip addr add 10.42.0.1/24 dev br-lab
sudo ip link set br-lab up
ip netns list
      blue
red
    2. Create the virtual patch cable for red — a linked pair of interfaces — then push one end of it into the red namespace and plug the other end into the bridge: 
      sudo ip link add veth-red type veth peer name ceth-red
sudo ip link set ceth-red netns red
sudo ip link set veth-red master br-lab
sudo ip link set veth-red up
    3. Now go inside the red namespace and finish setting up its end of the cable exactly the way a real CNI plugin would: rename the interface to eth0, give it an IP address, and bring it up: 
      sudo ip netns exec red ip link set ceth-red name eth0
sudo ip netns exec red ip addr add 10.42.0.2/24 dev eth0
sudo ip netns exec red ip link set eth0 up
sudo ip netns exec red ip link set lo up
sudo ip netns exec red ip route add default via 10.42.0.1
    4. Repeat the exact same recipe for blue, this time with the address 10.42.0.3: 
      sudo ip link add veth-blue type veth peer name ceth-blue
sudo ip link set ceth-blue netns blue
sudo ip link set veth-blue master br-lab
sudo ip link set veth-blue up
sudo ip netns exec blue ip link set ceth-blue name eth0
sudo ip netns exec blue ip addr add 10.42.0.3/24 dev eth0
sudo ip netns exec blue ip link set eth0 up
sudo ip netns exec blue ip link set lo up
sudo ip netns exec blue ip route add default via 10.42.0.1
    5. Prove the wiring works: ping "pod to pod" (from red to blue, both routed through the bridge) and "pod to node" (from red to the bridge's own IP, standing in for the node): 
      sudo ip netns exec red ping -c1 10.42.0.3
sudo ip netns exec red ping -c1 10.42.0.1
      64 bytes from 10.42.0.3: icmp_seq=1 ttl=64 time=0.08 ms
64 bytes from 10.42.0.1: icmp_seq=1 ttl=64 time=0.05 ms
    6. Look at what you just built, using the exact same commands you'll later use to inspect real pods: 
      ip link show master br-lab
sudo ip netns exec red ip addr show eth0
      5: veth-red@if2: <BROADCAST,MULTICAST,UP,LOWER_UP> ... master br-lab
7: veth-blue@if2: <BROADCAST,MULTICAST,UP,LOWER_UP> ... master br-lab
2: eth0@if5: <BROADCAST,MULTICAST,UP,LOWER_UP> ... link-netnsid 0
    inet 10.42.0.2/24 scope global eth0

      Read eth0@if5 as "my peer is host interface index 5" — that @ifN notation is how you'll match pods to veths in 4.3.

    7. Clean up (deleting a netns deletes the veth ends inside it; the pair dies with it): 
      sudo ip netns del red
sudo ip netns del blue
sudo ip link del br-lab
    Verify

    The first ping in step 5 succeeds — you have just routed a packet between two isolated namespaces through a bridge you built by hand, with your own commands, on a machine with no Kubernetes on it at all. If it fails: check that both veth host-ends show UP and master br-lab, and that the netns's eth0 really has its IP assigned.

    Gotchas

    If pings between namespaces fail even though everything shows UP, suspect the combination of br_netfilter (a kernel module) and ordinary firewall rules: with the sysctl net.bridge.bridge-nf-call-iptables=1 set (which Kubernetes requires, see W0.6), traffic crossing the bridge is run back through iptables just like any other traffic, and a restrictive FORWARD-chain policy can silently drop it. That interaction is exactly why the sysctl exists in the first place: without it, traffic between two pods on the same node would quietly bypass kube-proxy's rules and any NetworkPolicy entirely, since it would never touch iptables at all — a much scarier failure mode than an occasional dropped ping.

    Docs

    Not in the k8s docs — this is man ip-netns territory. The k8s side is docs search "Network Plugins".

    4.3

    Find a real pod's veth peer on the node

    Why it matters

    4.2 showed that every pod's eth0 is really just one end of a veth pair whose other end sits on the node. Answering "which host interface is the other end of this specific pod's cable?" is the move that lets you run tcpdump on the node and capture exactly one pod's traffic, in isolation from everything else on the machine — the sharpest tool you have when a pod's network "just doesn't work" and you need to see, packet by packet, what it's actually sending and receiving.

    The concept

    Every veth end carries a small piece of bookkeeping: the interface index (a plain integer ID the kernel assigns to each network interface) of its partner on the other end of the cable. Seen from inside the pod, an interface named eth0@if12 is the kernel telling you "my other end is host interface number 12"; on the node side, the matching interface shows up as something like vethXXXX@if2, pointing back at the pod's own interface number. Line the two numbers up and you've matched a specific pod to its specific veth on the node — that's the whole trick. Most container images ship with no ip binary at all, but the same peer index is also readable straight out of the filesystem at /sys/class/net/eth0/iflink, which needs no extra tools installed.

    Lab
    1. Run a throwaway pod and see which node the scheduler put it on: 
      kubectl run web4 --image=nginx
kubectl get pod web4 -o wide
      NAME   READY   STATUS    RESTARTS   AGE   IP           NODE
web4   1/1     Running   0          20s   10.244.1.8   node01
    2. Ask the pod, from inside its own filesystem, for the index of its veth's other end: 
      kubectl exec web4 -- cat /sys/class/net/eth0/iflink
      12
    3. Now go to node01 and find whichever host interface carries that same number, 12: 
      ip link | grep '^12:'
      12: veth1a2b3c4d@if2: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 ... master cni0

      Read @if2 as "my own other end points back at ifindex 2 inside the pod" — the pod's eth0; master cni0 tells you this veth is plugged into flannel's bridge, which is the real-world equivalent of the br-lab bridge you built by hand in 4.2.

    4. Prove you found the right interface by capturing the pod's traffic directly from the node — if this is the right veth, a ping from inside the pod should show up here: 
      sudo tcpdump -ni veth1a2b3c4d -c2 icmp &
kubectl exec web4 -- ping -c1 10.244.1.1
      10.244.1.8 > 10.244.1.1: ICMP echo request
10.244.1.1 > 10.244.1.8: ICMP echo reply
    Verify

    The iflink number read from inside the pod matches the ifindex of a real veth* interface you found on the node, and a tcpdump on that specific veth shows exactly the pod's ping traffic and nothing else's.

    Gotchas

    Interface indexes are assigned per-namespace and are not stable across pod restarts — a pod that gets recreated gets a fresh index, so never hardcode or script against these numbers; always look them up live. And if kubectl exec isn't an option (say, the pod has already crashed), you can still reach its network namespace through the container runtime directly: sudo crictl inspectp <sandbox-id> | grep netns gives you the namespace's filesystem path, and sudo nsenter --net=<path> ip addr lets you run ordinary commands inside it from the node.

    Docs

    Docs search: "Debug Running Pods" — the ephemeral-container variant (kubectl debug) for distroless images lives there.

    4.4

    Know your three CIDRs and where each is configured

    Why it matters

    A huge fraction of network triage boils down to one question: "which range is this IP address from?" — because the answer tells you which subsystem to go blame, before you've even run a single debugging command. A 10.244.x.x address that won't connect points at a CNI problem; a 10.96.x.x address points at a kube-proxy/Service problem; a plain node IP points at ordinary Linux networking, nothing Kubernetes-specific at all. Three ranges, three different failure domains — memorizing which is which turns "the network is broken" into "the network is broken here."

    The concept

    A cluster actually has three separate address ranges (CIDRs — Classless Inter-Domain Routing blocks, i.e. IP address ranges) layered on top of each other, each owned by a different component and used for a different purpose:

    RangeLab valueConfigured atWho consumes it
    Pod CIDR10.244.0.0/16kubeadm init --pod-network-cidr → controller-manager --cluster-cidrController-manager slices one /24 per node (node.spec.podCIDR); the CNI's IPAM hands out pod IPs from the node's slice
    Service CIDR10.96.0.0/12 (default)apiserver --service-cluster-ip-rangeapiserver assigns ClusterIPs; kube-proxy writes rules for them — no interface anywhere has these IPs
    Node network10.211.64.0/24 (your Multipass subnet)outside Kubernetes entirelyNode-to-node traffic, NodePorts, apiserver :6443, VXLAN tunnels
    Lab
    1. Read the pod CIDR straight out of the controller-manager's own manifest file, then check how it got sliced up per node: 
      sudo grep cluster-cidr /etc/kubernetes/manifests/kube-controller-manager.yaml
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.podCIDR}{"\n"}{end}'
          - --cluster-cidr=10.244.0.0/16
cp      10.244.0.0/24
node01  10.244.1.0/24
node02  10.244.2.0/24

      Now a pod IP alone tells you which node it lives on: anything in 10.244.1.x lives on node01, plain and simple. You will use that fact constantly during troubleshooting.

    2. Do the same for the service CIDR — read it from the apiserver's manifest, then look at the two well-known IPs it hands out first: 
      sudo grep service-cluster-ip-range /etc/kubernetes/manifests/kube-apiserver.yaml
kubectl get svc kubernetes kube-dns -A
          - --service-cluster-ip-range=10.96.0.0/12
NAMESPACE     NAME         TYPE        CLUSTER-IP    PORT(S)
default       kubernetes   ClusterIP   10.96.0.1    443/TCP
kube-system   kube-dns     ClusterIP   10.96.0.10   53/UDP,53/TCP,9153/TCP

      A convention worth memorizing because it holds on every kubeadm cluster you'll ever touch: the very first usable IP of the service range (.1) always goes to the apiserver's own Service, and the tenth IP (.10) always goes to cluster DNS.

    3. Finally, the node network — the one range that Kubernetes didn't create and doesn't control, just plain interfaces on plain Linux boxes: 
      ip -4 -brief addr show
      lo         UNKNOWN   127.0.0.1/8
ens3       UP        10.211.64.11/24
flannel.1  UNKNOWN   10.244.1.0/32
cni0       UP        10.244.1.1/24

      One node, and all three worlds show up side by side in one command's output: ens3 is the physical node network, cni0 is the pod bridge from 4.2/4.3 (sitting at .1 of this node's own slice), and flannel.1 is the VXLAN tunnel device that carries traffic to other nodes (explained fully in 4.7).

    Verify

    You should be able to answer both of these instantly, no lookup needed: which range is 10.96.130.5 from, and which component assigned it? (Service CIDR, assigned by the apiserver.) Which node hosts the pod at 10.244.2.9? (node02, because 10.244.2.0/24 is its slice.)

    Gotchas

    The three ranges must never overlap with each other — a genuinely exam-grade fault to know the signature of is a service CIDR that overlaps the node network, because then traffic addressed to some ClusterIPs gets routed straight out the physical NIC instead of ever hitting the DNAT rules that were supposed to catch it, since the routing table now has two competing ideas about where that address lives. Also worth knowing before you ever touch a real cluster: changing the pod CIDR after the cluster is already installed is effectively a full cluster rebuild, not a config edit — nodes keep whatever podCIDR slice they were originally allocated forever, with no supported way to reassign it live.

    Docs

    Docs search: "Cluster Networking" and "Virtual IPs and Service Proxies".

    CNI

    4.5

    The CNI contract: exec a binary, JSON in, JSON out

    Why it matters

    You've now built a pod network by hand (4.2) and traced a real pod down to its veth (4.3) — but something still has to do that wiring automatically, every time a pod is created, on whichever node the scheduler happens to pick. That something is CNI (Container Network Interface), and the 2025 exam curriculum added CNI understanding — not just "know it exists" — explicitly. More practically: "node NotReady, cni plugin not initialized" is a staple planted fault on troubleshooting tasks, and it only takes seconds to fix once you know the two directories involved and what each one is for.

    The concept

    CNI is deliberately dumb, and that's the whole point of it — it is not a daemon running in the background and not a network API you call over HTTP. It is a plain convention for executing a binary program once for every pod-networking event, the way a shell script calls another program and reads its output:

    • When containerd creates a new pod sandbox (the empty shell — namespace plus pause container — that the pod's containers will join), it reads the first config file it finds, in alphabetical order, from the directory /etc/cni/net.d/.
    • It then executes the plugin binary that config file names, out of the directory /opt/cni/bin/, telling it what to do via the verb in the CNI_COMMAND environment variable — ADD (wire this pod up), DEL (tear it back down), CHECK, VERSION — feeding it the network config as JSON text on its stdin, and passing extra details like the pod's netns filesystem path and container ID through more environment variables.
    • The plugin does the actual work (creating the veth pair, programming routes — exactly the primitives from 4.2) and then prints a JSON result back out on its stdout, describing the IP address it assigned.

    Plugins can also be chained together: a .conflist file holds an ordered list of them, where each plugin's result gets handed to the next one as additional input. The typical split of responsibilities is: one main plugin wires up the interface itself (either the generic bridge plugin from 4.2, or flannel/calico/cilium's own equivalent), a separate IPAM plugin (IP Address Management) is responsible only for picking which IP address the pod gets (host-local is the simplest one — it's just a directory holding one file per allocated IP), and small meta plugins bolt on extras (portmap for hostPort support, bandwidth for rate limits). Flannel's own plugin is famously thin — it does almost nothing itself: it reads the node's assigned subnet out of the file /run/flannel/subnet.env and then simply delegates the actual wiring work to the standard bridge plugin — the exact same plugin, doing the exact same thing, that you drove by hand in 4.2.

    containerd: create sandbox netns→read /etc/cni/net.d→exec plugin: CNI_COMMAND=ADD, JSON on stdin→plugin wires veth + routes→IPAM assigns IP→result JSON on stdout → pod IP
    Lab
    1. Read your cluster's actual CNI config file — this is the config that gets read first, on any node, every time a pod sandbox is created: 
      sudo cat /etc/cni/net.d/10-flannel.conflist
      {
  "name": "cbr0",
  "cniVersion": "0.3.1",
  "plugins": [
    { "type": "flannel", "delegate": { "hairpinMode": true, "isDefaultGateway": true } },
    { "type": "portmap", "capabilities": { "portMappings": true } }
  ]
}

      Two chained plugins run in order for every pod: flannel (which, as just explained, delegates the real work to bridge plus host-local), then portmap on top.

    2. List every binary the runtime is able to execute — this is the other half of the contract, the directory the plugin name in the config above actually points into: 
      ls /opt/cni/bin
      bandwidth  bridge  dhcp  firewall  flannel  host-device  host-local  ipvlan
loopback  macvlan  portmap  ptp  sbr  static  tuning  vlan  vrf
    3. Look at the IPAM plugin's entire "database" — it's nothing more exotic than one plain file per allocated IP, named after the IP itself, holding the ID of the container that owns it: 
      sudo ls /var/lib/cni/networks/cbr0/ | head -4
sudo cat /var/lib/cni/networks/cbr0/10.244.1.8
      10.244.1.7
10.244.1.8
last_reserved_ip.0
lock
3f9a1c…  (the pod sandbox's container ID)  eth0
    Verify

    You can name, without looking anything up: the config directory, the binary directory, the four verbs, and where the JSON goes in and where it comes back out. (/etc/cni/net.d, /opt/cni/bin, ADD/DEL/CHECK/VERSION, stdin in / stdout out.)

    Gotchas

    Both directories live per node, not cluster-wide — which means a CNI that's broken on one node breaks only that one node's pods, and even then only its new pods, because existing pods keep the veths they were already wired with and don't need the plugin to run again just to keep working. That asymmetry — old pods on the node keep humming along fine, while brand-new pods on that same node get stuck in ContainerCreating forever — is the exact signature of a CNI fault, and 4.8 has you produce it yourself so you recognize it instantly.

    Docs

    Docs search: "Network Plugins" (the CNI spec itself lives at github.com/containernetworking — not available in the exam, but the k8s page covers what you need).

    4.6

    Inspect your actual CNI: flannel's moving parts

    Why it matters

    4.5 gave you the generic shape of any CNI plugin. Knowledge like that only really sticks once you trace it through one concrete, real implementation end to end — and your lab cluster's flannel happens to be the simplest real-world CNI there is, which makes it the ideal one to fully understand rather than just memorize.

    The concept

    Flannel has exactly three moving parts on every node, and each has one job: the flanneld DaemonSet pod (a pod that runs on every node, watching the apiserver so it always knows every other node's slice of the pod address range, and using that knowledge to program the VXLAN tunnel device and the routing table); the subnet.env file flanneld writes to disk (a small handoff note so the CNI plugin can learn this node's own slice without needing to talk to the apiserver itself); and the CNI plugin binary (the thin delegator from 4.5 that hands the real interface-wiring work off to bridge plus host-local). The bridge cni0 and the VXLAN tunnel device flannel.1 — both mentioned already in 4.4 — are exactly what these three pieces, working together, create.

    Lab
    1. Start with the daemon itself — you should see exactly one flannel pod per node, since it's a DaemonSet: 
      kubectl -n kube-flannel get pods -o wide
      NAME                    READY   STATUS    NODE
kube-flannel-ds-7xk2p   1/1     Running   cp
kube-flannel-ds-9rl4m   1/1     Running   node01
kube-flannel-ds-tw8qn   1/1     Running   node02
    2. Now look at the handoff file on node01 — the one flanneld writes and the CNI plugin later reads: 
      cat /run/flannel/subnet.env
      FLANNEL_NETWORK=10.244.0.0/16
FLANNEL_SUBNET=10.244.1.1/24
FLANNEL_MTU=1450
FLANNEL_IPMASQ=true

      Notice FLANNEL_SUBNET matches node01's spec.podCIDR from 4.4 exactly — that's not a coincidence, it's the whole chain playing out: the controller-manager allocates the slice → flanneld reads it off the Node object in the apiserver → flanneld writes it into this file → the bridge plugin's IPAM reads the file and hands out IPs from it. The MTU of 1450, rather than the usual 1500, is 1500 minus the 50 bytes the VXLAN encapsulation header (explained in 4.7) adds to every packet.

    3. Look at the two interfaces flannel actually built on this node, in more detail than a plain ip addr shows: 
      ip -d link show flannel.1 | head -3
ip -d link show cni0 | head -3
      4: flannel.1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 ...
    vxlan id 1 local 10.211.64.11 dev ens3 srcport 0 0 dstport 8472 nolearning
5: cni0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 ...
    bridge forward_delay 1500 hello_time 200 ...
    4. Finally, look at the routing table entry that ties everything together into one working path: 
      ip route | grep 10.244
      10.244.0.0/24 via 10.244.0.0 dev flannel.1 onlink
10.244.1.0/24 dev cni0 proto kernel scope link src 10.244.1.1
10.244.2.0/24 via 10.244.2.0 dev flannel.1 onlink

      Read it aloud: "my own slice → local bridge; every other node's slice → into the VXLAN tunnel." That one routing table is flannel's whole datapath decision.

    Verify

    For a pod IP that lives on another node, you should be able to trace the full path from memory, hop by hop: pod's eth0 → its veth → the local cni0 bridge → routed via flannel.1 → carried as a VXLAN packet over ens3 to the other node → the peer's flannel.1 unwraps it → the peer's cni0 → the peer's veth → the destination pod.

    Gotchas

    The MTU chain matters more than it looks like it should: every pod inherits the MTU of 1450 from cni0, not the usual 1500. If something along the way resets an interface back to 1500 (or an outer network in between actually allows even less than 1450), large packets get silently dropped — "blackholed" — while small ones like pings sail through untouched, because pings are tiny and rarely hit the size limit. "Small requests work fine, but anything with a real response body just hangs" is the signature of an MTU mismatch, not of a NetworkPolicy or firewall blocking something — a distinction worth having memorized before you start troubleshooting.

    Docs

    Docs search: "Cluster Networking". Flannel specifics are on its GitHub (not exam-available — internalize the shape now).

    4.7

    Datapath families: overlay vs routed vs eBPF

    Why it matters

    The exam asks for genuine understanding here, not vendor trivia to memorize: given some cluster you've never seen before, can you tell how pod traffic actually crosses between its nodes, and what that specific choice implies for how you'd debug it?

    The concept

    Every CNI plugin has to solve the same problem — get a packet from a pod on one node to a pod on another node — but there are really only three different families of answer to how the packet physically gets across the wire between the two nodes:

    FamilyHow inter-node pod traffic movesOne-sentence tradeoffExample
    OverlayThe whole pod-to-pod packet gets wrapped inside a second, outer packet addressed node-to-node — VXLAN encapsulation over plain UDP (flannel uses UDP port 8472 for this)Works on literally any network that lets ordinary UDP through — the cost is about 50 bytes of MTU eaten by the extra wrapper, and moments where tcpdump on the physical NIC shows only the outer node-to-node packet, not the pod IPs hidden inside itFlannel (VXLAN), Calico in VXLAN/IPIP mode

    Prove with your own eyes that your lab cluster is an overlay, by catching the encapsulation happening live.

    1. Start a cross-node ping: recall web4 lives on node01 (from 4.3); ping it from a fresh pod deliberately pinned to node02, so the traffic has to cross between nodes: 
      kubectl run pinger --image=busybox:1.36 \
  --overrides='{"spec":{"nodeName":"node02"}}' \
  --restart=Never -- ping -i0.5 10.244.1.8
    2. On node01, capture traffic on the physical NIC itself, filtering just for the VXLAN port so the output stays readable: 
      sudo tcpdump -ni ens3 udp port 8472 -c2
      10.211.64.12.51384 > 10.211.64.11.8472: OTV, ... VXLAN, vni 1
10.244.2.11 > 10.244.1.8: ICMP echo request ...

      One line of output shows two nested worlds at once: the outer packet is plain node02-to-node01 traffic over UDP port 8472, and tcpdump is smart enough to unwrap the VXLAN header and show you the inner pod-to-pod ICMP ping riding inside it. That nesting — a packet inside a packet — is what an overlay network is.

    3. Now capture the same ping again, but this time on flannel.1 — the tunnel device itself, on the inside of the tunnel — to see the contrast: only pod IPs, no outer UDP wrapper at all: 
      sudo tcpdump -ni flannel.1 -c2 icmp
kubectl delete pod pinger
      10.244.2.11 > 10.244.1.8: ICMP echo request
10.244.1.8 > 10.244.2.11: ICMP echo reply
    Verify

    You captured the exact same ping twice, in two different disguises: wrapped inside VXLAN on ens3, and completely naked on flannel.1. Say the routed-mode contrast out loud to make sure it's landed: on Calico running in BGP mode, that same capture on ens3 would show the pod IPs directly, with no wrapping at all, because routed mode never encapsulates anything.

    Gotchas

    Linux's own VXLAN implementation defaults to UDP port 8472, not the official IANA-standard port 4789 that other VXLAN implementations often use — so a firewall rule that was carefully opened for 4789 will silently break flannel, because flannel was never using that port in the first place. And hold onto this for 4.23: flannel does not implement NetworkPolicy at all — which datapath family a CNI uses and whether it enforces NetworkPolicy are two entirely separate capabilities that happen to both belong to "the CNI plugin," and a plugin can have either one without the other.

    4.8

    Break the CNI on purpose and read the signatures

    Why it matters

    This exact fault — the CNI config missing or broken on just one node — shows up in troubleshooting tasks precisely because it produces confusing, lopsided symptoms: some things keep working, some things don't, and it's not obvious at first why. Plant this fault yourself once, deliberately, in a safe lab, and the symptoms turn into a signature you can read in ten seconds flat instead of a mystery you have to slowly reason your way through under exam pressure.

    Lab
    1. On node01, hide the CNI config file (simulating it going missing or getting corrupted) and restart the kubelet so it notices: 
      sudo mv /etc/cni/net.d/10-flannel.conflist /root/
sudo systemctl restart kubelet
    2. Watch the node's health condition flip within under a minute — this is the kubelet noticing it can no longer confirm the network is working: 
      kubectl get node node01
kubectl describe node node01 | grep -A1 'Ready '
      NAME     STATUS     ROLES    AGE   VERSION
node01   NotReady   <none>   25d   v1.35.x
  Ready   False ... KubeletNotReady   container runtime network not ready:
  NetworkReady=false reason:NetworkPluginNotReady message:Network plugin
  returns error: cni plugin not initialized

      Memorize that message — it's verbatim what the exam shows you.

    3. Existing pods on node01 just keep running undisturbed, because their veths are already wired and nothing needs the CNI plugin to run again for them. But any new pod pinned to node01 gets stuck, because creating it requires the CNI ADD step to succeed: 
      kubectl run stuck --image=nginx --overrides='{"spec":{"nodeName":"node01"}}' --restart=Never
kubectl get pod stuck
kubectl describe pod stuck | tail -3
      stuck   0/1     ContainerCreating   0     30s
  Warning  FailedCreatePodSandBox ... failed to setup network for sandbox:
  plugin type="flannel" failed (add): open /run/flannel/subnet.env: no such file...
  — or: no valid networks found in /etc/cni/net.d

      Notice the precise split this reveals: a pod would still get assigned to this node even though it's NotReady — the scheduler doesn't automatically avoid a NotReady node on its own; instead the node gets an automatic taint, node.kubernetes.io/not-ready:NoSchedule, which is what actually keeps new pods away in a real cluster. Here you forced the pod onto the node anyway, so you get to see the failure one layer deeper: scheduling succeeds, but sandbox creation then fails at the CNI ADD step itself.

    4. Fix the underlying cause and watch the whole node heal itself, with no further intervention needed: 
      sudo mv /root/10-flannel.conflist /etc/cni/net.d/

      node01   NotReady   <none>   25d   v1.35.x
node01   Ready      <none>   25d   v1.35.x
    5. Clean up: 
      kubectl delete pod stuck --force --grace-period 0
    Verify

    The node is back to Ready, and a fresh kubectl run test --image=nginx pod actually reaches Running on node01 rather than sticking. Delete it once you've confirmed that.

    Gotchas

    Memorize the triage order for this whole fault family, because it saves real time under pressure: kubectl describe node first (the condition message names CNI explicitly, so you don't have to guess) → then ls /etc/cni/net.d/ on that specific node → then ls /opt/cni/bin/ → then the flannel pod's own logs. Deliberately don't start by poking at iptables or CoreDNS here: node NotReady combined with pods stuck in ContainerCreating points to a layer below Services and DNS entirely — the CNI layer, which everything else in this week's networking stack sits on top of.

    Services & kube-proxy

    4.9

    A ClusterIP is a virtual IP — it exists only as rewrite rules

    Why it matters

    A Service is Kubernetes's answer to a problem 4.1 already implied: pod IPs are real, but pods are also disposable — they get replaced constantly as deployments roll out and scale — so nothing should ever be hardcoded to talk to a specific pod IP directly. A Service is meant to be a single, stable address that always points at whichever pods are currently healthy — the same role a phone book entry or a directory-assistance operator plays: you dial one memorable number, and the system behind it quietly connects you to whichever physical line happens to be free right now. But once you truly believe the mechanical fact behind that stability — that a service IP (called a ClusterIP) is not sitting on any network interface, anywhere, on any machine — a whole class of confusion just dissolves on its own: why you can't ping a service the way you'd ping a pod, why services fail in different ways than pods do, and why "the name resolves but the connection doesn't complete" always points at kube-proxy rather than at DNS.

    The concept

    When you create a Service, all that actually happens is: the apiserver picks one free IP address out of the service CIDR (4.4) and writes it into the Service object stored in etcd. That's it — nothing else happens at that moment. No network interface gets created anywhere, and no route gets announced to any router. The component that turns that inert stored IP into something that actually works is kube-proxy, a small process that runs on every single node, watching Services and EndpointSlices (4.11) and, in response, writing rules into the node's netfilter (the kernel's own packet-filtering and rewriting engine, which iptables is a frontend for): "does a packet's destination match 10.101.x.x on port 80? If so, rewrite its destination address" — that rewrite is called DNAT (Destination Network Address Translation) — "to one of these specific pod IPs instead." The illusion that there's one stable, reachable "service" sitting at that IP is manufactured completely independently, node by node, at the exact moment a packet happens to leave some process's socket — there is no central place that "is" the Service. One consequence worth sitting with: if you kill kube-proxy outright, the rules it already wrote keep working for a while, because they live in the kernel and don't need the process to stay alive — but they slowly go stale as pods are replaced and nothing is left to update them, which is a uniquely sneaky, delayed failure mode rather than an immediate outage.

    Lab
    1. Create an ordinary deployment and a Service in front of it: 
      kubectl create deploy web --image=nginx --replicas=2
kubectl expose deploy web --port 80
kubectl get svc web
      NAME   TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)
web    ClusterIP   10.101.44.7    <none>        80/TCP
    2. Now go hunting for that IP address anywhere on a node — you will not find it, because it genuinely isn't anywhere: 
      ip addr | grep 10.101.44.7 ; ip route get 10.101.44.7
      (no interface match)
10.101.44.7 via 10.211.64.1 dev ens3 src 10.211.64.11
    — the routing table would send it to the default gateway!
    3. And yet, from that exact same node, talking to that IP actually works — this is the paradox the DNAT rules resolve: 
      curl -s 10.101.44.7 | grep -o '<title>.*</title>'
ping -c1 -W1 10.101.44.7
      <title>Welcome to nginx!</title>
1 packets transmitted, 0 received, 100% packet loss

      curl succeeds because kube-proxy's DNAT rule specifically matches TCP traffic to destination port 80 (tcp dpt:80) and rewrites the packet's destination to a real pod IP before ordinary routing ever gets a chance to look at it. ping fails outright because there is no rule at all for ICMP (the ping protocol) — nothing on the network stack answers the ping, for the simple reason that nothing actually owns that IP address to answer on its behalf. An address you can curl but can never successfully ping is exactly the fingerprint of a virtual, rule-only IP like this one.

    Verify

    curl to the ClusterIP succeeds while ping to that same IP gets 100% packet loss — and you should be able to explain why in one sentence without hesitating.

    Gotchas

    Never use ping to test whether a Service is working — use curl/wget against the actual port instead, or nc -zv for a bare connectivity check. In the exam, "I can't ping the service" is completely expected, correct behavior, not evidence of a fault to go chase.

    4.10

    Walk the iptables chains behind one of your services

    Why it matters

    4.9 established that a Service is "just rules" — this lesson makes that concrete by actually reading those rules off a live node. You'll likely never hand-edit them, but being able to read them turns kube-proxy from an opaque black box into three plainly greppable chain names, and lets you prove during triage, with evidence rather than a guess, whether the rules for a given service exist at all.

    The concept

    Despite the name, kube-proxy running in its default iptables mode is best understood as a rule compiler, not a proxy — no packet ever actually passes through the kube-proxy process itself; the process only ever writes rules that the kernel then applies on its own. (IPVS and the newer nftables backend — GA, meaning stable and fully supported, since Kubernetes 1.33 — exist as faster alternatives at large scale, but iptables is still the default and what you'll see in the lab.) kube-proxy renders every Service plus its EndpointSlices (4.11) into three layers of chains in the kernel's nat table, and a separate kernel feature called conntrack (connection tracking — the kernel's memory of which backend a given ongoing connection was already routed to) pins each TCP flow to whichever backend pod was picked for its very first packet, so a long-lived connection doesn't get randomly reshuffled to a different pod partway through:

    Lab
    1. Scale up so there are three backend pods to find in the rules below, making the probabilistic split in step 3 worth actually looking at: 
      kubectl scale deploy web --replicas=3
    2. Layer 1 — find your specific service inside the KUBE-SERVICES chain, the entry point every packet destined for a ClusterIP passes through first (run this on any node; kube-proxy helpfully tags every rule it writes with a human-readable comment): 
      sudo iptables -t nat -L KUBE-SERVICES -n | grep default/web
      KUBE-SVC-LOLE4ISW44XBNF3G  tcp -- 0.0.0.0/0  10.101.44.7  /* default/web cluster IP */  tcp dpt:80
    3. Layer 2 — follow the jump into the per-service chain and look at how it splits traffic across backends: 
      sudo iptables -t nat -L KUBE-SVC-LOLE4ISW44XBNF3G -n
      KUBE-SEP-XVQ...  all -- ... /* default/web -> 10.244.1.8:80 */  statistic mode random probability 0.33333333349
KUBE-SEP-ZAB...  all -- ... /* default/web -> 10.244.2.4:80 */  statistic mode random probability 0.50000000000
KUBE-SEP-QRT...  all -- ... /* default/web -> 10.244.2.5:80 */

      Read the cascade the way the kernel evaluates it, top to bottom: a ⅓ chance of stopping at the first rule, otherwise a ½ chance of stopping at the second (which, combined, works out to a uniform ⅓ each across all three), otherwise the packet falls through to the last one unconditionally. This cascading-probability trick, however odd it looks written out, is the entire load balancer — there's no separate load-balancing component or algorithm running anywhere; it's just these three lines.

    4. Layer 3 — follow one more jump into a single endpoint chain, where the actual address rewrite finally happens: 
      sudo iptables -t nat -L KUBE-SEP-XVQ... -n
      KUBE-MARK-MASQ  all  --  10.244.1.8  0.0.0.0/0   /* hairpin: pod dialing itself via the svc */
DNAT   tcp ... to:10.244.1.8:80
    5. Finally, see conntrack actually pinning a flow to the backend it picked (install the tool once: sudo apt-get install -y conntrack): 
      curl -s 10.101.44.7 >/dev/null; sudo conntrack -L 2>/dev/null | grep 10.101.44.7
      tcp ... src=10.211.64.11 dst=10.101.44.7 dport=80 ... src=10.244.2.4 sport=80  [ASSURED]

      Both truths sit right there in one line of output: the connection was addressed, from the client's point of view, to the service IP the whole time — yet conntrack's own record shows it's really talking to one specific pod, confirming the rewrite actually took effect and stuck for the life of the connection.

    Verify

    You followed one service through all three layers by hand, and the pod IPs you found inside the KUBE-SEP chains match exactly what kubectl get pods -l app=web -o wide reports as the real, running backends.

    Gotchas

    Two failure modes that look similar but mean very different things: a service with no ready endpoints compiles down to a REJECT rule, which means something on the network actively refuses the connection — you get "connection refused" back almost instantly, because a rule fired and said no. But no rules at all existing for the IP (say, because kube-proxy itself has died) means the packet just falls through to ordinary routing and heads out the default gateway with nobody home — which produces a silent timeout instead, because nothing ever said no, it just never got an answer. Refused versus timeout is a genuine diagnostic fork you can act on, not background noise to ignore. And never hand-edit any KUBE-* chain directly — kube-proxy considers itself the sole owner of these rules and will overwrite anything you change the next time it reconciles.

    4.11

    EndpointSlices: readiness decides who gets traffic

    Why it matters

    4.9 and 4.10 both leaned on one fact without proving it yet: that kube-proxy's rules are built from a list of pod IPs somewhere. That list is the EndpointSlice, and kubectl get endpointslices is the single highest-yield command in your whole toolkit for "service isn't working" tasks — it tells you, in one look, whether the Service is selecting any pods at all, whether those pods are actually ready to receive traffic, and exactly which IPs traffic is really being sent to right now.

    The concept

    A separate controller (in the reconcile-loop sense from week 1 — something continuously watching the cluster's actual state and adjusting stored objects to match) watches every Service and every Pod: for each Service, it finds all the pods matching that Service's selector (a label query — "give me every pod labeled app: web") and writes their IPs into one or more EndpointSlice objects (split into chunks of 100 endpoints each, purely so a single object never gets too big to handle at large scale), marking each individual endpoint with whether that pod is currently ready. kube-proxy then compiles only the ready endpoints into its forwarding rules — not-ready ones are left out entirely, as if they didn't exist. That means a failing readinessProbe (W3) isn't just cosmetic decoration on kubectl get pods' READY column — it is the actual mechanism, the only mechanism, that pulls a pod out of every Service that selects it. Two separate jobs, one chain: the selector decides slice membership — who's a candidate at all — and readiness decides who actually receives traffic among those candidates.

    Lab
    1. Look at the EndpointSlice sitting behind your service right now: 
      kubectl get endpointslices -l kubernetes.io/service-name=web
      NAME        ADDRESSTYPE   PORTS   ENDPOINTS                            AGE
web-p7k2x   IPv4          80      10.244.1.8,10.244.2.4,10.244.2.5    9m
    2. Give the pods a genuine readiness probe, so there's something real for the controller to base "ready" on rather than the default (always ready the moment the container starts): 
      kubectl patch deploy web --type json -p '[{"op":"add",
 "path":"/spec/template/spec/containers/0/readinessProbe",
 "value":{"httpGet":{"path":"/","port":80},"periodSeconds":2}}]'
kubectl rollout status deploy web
    3. Deliberately break readiness in exactly one pod, leaving the other two untouched — deleting nginx's index page makes it return an HTTP 403, which the probe will treat as a failure: 
      POD=$(kubectl get pods -l app=web -o jsonpath='{.items[0].metadata.name}')
kubectl exec $POD -- rm /usr/share/nginx/html/index.html
kubectl get pods -l app=web -w
      web-6d4cf56db7-abcde   0/1     Running   0   3m   ← READY drops, pod still Running
    4. Watch the EndpointSlice flip that one endpoint to not-ready — and confirm, by hammering the Service with real requests, that traffic genuinely stops going to it: 
      kubectl get endpointslices -l kubernetes.io/service-name=web -o yaml | grep -B3 'ready:'
for i in $(seq 1 12); do curl -s --max-time 2 10.101.44.7 >/dev/null && echo ok; done | sort | uniq -c
        - addresses: [10.244.1.8]
    conditions:
      ready: false
     12 ok        ← every request served: the broken pod gets nothing
    5. Heal the pod and watch it get added back into rotation on its own, with no restart or manual step needed: 
      kubectl exec $POD -- sh -c 'echo back > /usr/share/nginx/html/index.html'
    Verify

    While the pod was not-ready, none of your curls actually failed — because traffic was successfully steered around it, which is the whole point; the EndpointSlice showed ready: false for exactly that one pod's IP and no other; and after healing it, ready: true comes back within a few seconds once the next probe succeeds.

    Gotchas

    The classic exam fault actually sits one step upstream of readiness entirely: a service whose selector matches no pods at all — in that case the EndpointSlice either exists but is completely empty, or doesn't get created at all, rather than showing not-ready endpoints. The triage ladder to follow: run kubectl get endpointslices first; if it's empty, compare the selector in kubectl get svc -o yaml against the actual labels on your pods, since a typo or a stale label is almost always the cause; if it's non-empty but shows not-ready endpoints, only then go chase the readinessProbe itself. (You may also encounter the older, legacy kubectl get endpoints command — it shows the same underlying information in an older format, and is being phased out in favor of EndpointSlices.)

    4.13

    Headless services: DNS straight to pod IPs

    Why it matters

    A headless service is the machinery hiding behind StatefulSet identity (W3) and the standard answer whenever a client actually needs to pick its own specific backend rather than being load-balanced to a random one (databases talking to each other, peer discovery between replicas). The exam likes testing this concept because it's really testing whether you understand what a normal ClusterIP does at all — by showing you what's left once you deliberately take it away.

    The concept

    Setting clusterIP: None on a Service means exactly this: no virtual IP gets assigned, no kube-proxy rules get written, and no load balancing happens anywhere. The Service becomes nothing but a DNS name plus ordinary endpoint tracking — CoreDNS answers a lookup of the service name with one A record per ready pod, instead of the single virtual IP a normal Service would return. The client gets the entire list of real pod IPs back from DNS and connects to one of them directly itself, with no rewriting or load-balancing layer in between at all. Behind a StatefulSet specifically, each individual pod additionally gets its own stable, per-pod DNS name of the shape <pod>.<svc>.<ns>.svc.cluster.local, which is what lets a StatefulSet's pod-0 always be reachable at the same name across restarts.

    Lab
    1. Create a headless twin sitting right next to your existing, normal service, so you can compare them directly: 
      kubectl expose deploy web --name=web-hl --cluster-ip=None --port=80
kubectl get svc web web-hl
      web      ClusterIP   10.101.44.7   <none>   80/TCP
web-hl   ClusterIP   None          <none>   80/TCP
    2. Ask DNS for both names and compare the answers directly (use the dnsutils pod from 4.17 if it's still running, or any throwaway busybox): 
      kubectl run tmpq --image=busybox:1.36 --restart=Never --rm -it -- sh -c \
  'nslookup web; nslookup web-hl'
      Name: web.default.svc.cluster.local
Address: 10.101.44.7              ← one answer: the virtual IP

Name: web-hl.default.svc.cluster.local
Address: 10.244.1.8
Address: 10.244.2.4
Address: 10.244.2.5               ← one answer per ready pod
    3. Confirm there's genuinely nothing standing in between the client and the pods to connect through — the pod IPs themselves are literally the same list DNS just handed back: 
      kubectl get endpointslices -l kubernetes.io/service-name=web-hl
      web-hl-x2x9c   IPv4   80   10.244.1.8,10.244.2.4,10.244.2.5
    Verify

    The normal service resolves to exactly one (virtual) IP address; the headless one resolves to as many real IPs as there are currently ready pods — and readiness (4.11) is gating these DNS answers in exactly the same way it gates the DNAT rules for a normal service, just showing up in a different place.

    Gotchas

    clusterIP: None has to be set at the moment of creation — you cannot edit an already-existing normal service into a headless one, because that field is immutable once set; the only path is to delete it and recreate it headless from scratch. And since there's no proxying layer at all here, there's also no port remapping happening anywhere: clients connect directly to whatever real port each pod actually listens on.

    4.14

    port vs targetPort vs nodePort — and the most-planted service bug

    Why it matters

    Almost every "the service doesn't work" exam task turns out to be one of these three port numbers being set wrong. Once you know which field is which — and, crucially, which specific mistake produces which specific symptom — the fix stops being a mystery and becomes a 30-second edit.

    The concept
    FieldWhose portSeen byDefault
    portthe Service's own port on its ClusterIPclients dialing the servicerequired
    targetPortthe container's port — where DNAT actually sends trafficthe poddefaults to port
    nodePortthe port opened on every node (NodePort/LB types)external clientsauto-assigned 30000–32767

    targetPort can also be given as a name instead of a number: if a container declares its port as ports: [{name: http, containerPort: 8080}], then setting targetPort: http makes the Service follow whatever number each individual pod happens to call http — a layer of indirection that lets you change a container's actual port later without ever having to touch the Service that points at it.

    Lab
    1. Plant the classic bug on purpose — a targetPort that points at a port nothing is actually listening on: 
      kubectl expose deploy web --name=web-broken --port=80 --target-port=8080

      curl: (7) Failed to connect to 10.99.203.1 port 80: Connection refused

      kubectl get endpointslices -l kubernetes.io/service-name=web-broken -o jsonpath='{.items[0].ports}'
      [{"name":"","port":8080,"protocol":"TCP"}]   ← pods are listed, DNAT goes to :8080, nginx listens on :80

      Learn to distinguish two refusals that look identical from the client's side but have completely different fixes: empty endpoints, meaning the selector bug from 4.11, versus present-but-wrong-port endpoints, meaning this bug. Both produce the exact same "connection refused" message, but reading the endpoint list itself tells you immediately which one you're facing.

    3. Fix it live and watch the Service start working immediately, with no restart needed anywhere: 
      kubectl patch svc web-broken -p '{"spec":{"ports":[{"port":80,"targetPort":80}]}}'
curl -s $(kubectl get svc web-broken -o jsonpath='{.spec.clusterIP}') | grep -o 'Welcome to nginx'
      Welcome to nginx
    4. Clean up: 
      kubectl delete svc web-broken
    Verify

    You should be able to recite this diagnostic without thinking hard about it: refused, with endpoints listed but carrying an unexpected port, means targetPort doesn't match the containerPort the process is actually listening on; refused, with no endpoints at all, means a selector mismatch (4.11); a plain timeout means the problem is somewhere lower in the stack entirely — kube-proxy, a NetworkPolicy, or the CNI (4.8).

    Gotchas

    Here's the trap mirrored the other way around: containerPort in a pod spec is documentation only to Kubernetes — it does not open, close, or configure anything. nginx listens on port 80 whether or not the manifest bothers to declare it, because the process itself decides what it binds to, not the YAML describing it. So a wrong containerPort value breaks nothing on its own… unless some Service is using a named targetPort that gets resolved through that exact declaration, in which case the wrong number quietly propagates into a real DNAT rule. The lesson generalizes: always check what a process actually binds to on the wire, never just what the YAML claims it does.

    DNS & CoreDNS

    4.15

    Read the Corefile plugin by plugin

    Why it matters

    CoreDNS is the DNS server that runs inside your cluster and answers every *.svc.cluster.local lookup — and its configuration file, the Corefile, lives in an ordinary ConfigMap you may well be asked to modify (adding a forward zone, changing an upstream server) and which is also a favorite place for exam faults to get planted.     Gotchas

    The loop story is a genuine exam-grade rabbit hole worth understanding fully rather than just memorizing. Many Linux hosts run systemd-resolved, a local stub DNS resolver, which makes /etc/resolv.conf point at the loopback address 127.0.0.53 instead of a real upstream server. If CoreDNS's forward plugin ever forwarded queries there, it would end up asking itself, indirectly, through the node — an infinite loop that would spin forever and eventually crash the process. That's exactly why the kubelet, when it configures CoreDNS's own pod, deliberately points it at /run/systemd/resolve/resolv.conf instead (the file holding the real upstream resolvers systemd-resolved itself uses) via its own resolvConf setting — sidestepping the loop entirely. So if you ever see CoreDNS crash-looping with the message plugin/loop: Loop ... detected, the fix is to inspect the node's resolv.conf and the kubelet's resolvConf setting, not to touch the Corefile itself.

    4.16

    A pod's resolv.conf, line by line — and the ndots:5 ladder

    Why it matters

    Nearly every piece of DNS behavior that surprises people the first time they see it — why a bare service name like web works at all without a fully qualified domain, why external lookups from inside a pod feel oddly slow, why pods with hostNetwork: true mysteriously can't resolve services — is written down, in full, in this one three-line file that every pod gets.

    The concept

    The kubelet writes a fresh /etc/resolv.conf (the standard Linux file that tells any program which DNS server to use, and how) into every pod, following whatever that pod's dnsPolicy field says. The default policy, called ClusterFirst, produces this:

    search default.svc.cluster.local svc.cluster.local cluster.local
nameserver 10.96.0.10
options ndots:5
    • nameserver — the kube-dns Service's own ClusterIP (4.4). Every DNS query a pod makes goes to this one virtual service IP, and gets there through the exact same DNAT machinery any other Service traffic uses (4.9/4.10) — DNS isn't special-cased at the networking layer at all.
    • search — a list of three suffixes to automatically try appending to a short name, ordered most-specific first, starting with the pod's own namespace.
    • ndots:5 — an option meaning: any name with fewer than 5 dots in it gets tried against every entry in the search list first, before it's ever tried as a plain, standalone name exactly as typed.

    Put together, here's the exact resolution ladder a bare name climbs, for a pod living in namespace default:

    db→db.default.svc.cluster.local→db.svc.cluster.local→db.cluster.local→db.

    The first rung of the ladder that gets an answer wins, and the climb stops there — that's the whole reason a bare db finds a service living in your own namespace, while db.other instead finds one in namespace other (matching on the ladder's second rung: db.other.svc.cluster.local). But this ladder has a real cost for names outside the cluster: www.example.com only has 2 dots in it, well under the ndots:5 threshold, so it has to climb the entire ladder first — three guaranteed-to-fail (NXDOMAIN) cluster-zone queries — before CoreDNS ever tries the real external lookup. Appending a trailing dot yourself (www.example.com.) tells the resolver the name is already absolute and complete, which skips the ladder and the wasted queries entirely.

    Lab
    1. Read a real pod's resolv.conf and confirm it matches exactly what's described above: 
      kubectl exec web4 -- cat /etc/resolv.conf
      search default.svc.cluster.local svc.cluster.local cluster.local
nameserver 10.96.0.10
options ndots:5
    2. Watch the ladder actually get climbed, live, in the logs — turn on CoreDNS's query logging, then make a lookup for a low-dot-count external name and see every rung it tries in order: 
      kubectl -n kube-system get cm coredns -o yaml | sed 's/    errors/    errors\n    log/' | kubectl apply -f -

      [INFO] ... "A IN example.com.default.svc.cluster.local." NXDOMAIN ...
[INFO] ... "A IN example.com.svc.cluster.local." NXDOMAIN ...
[INFO] ... "A IN example.com.cluster.local." NXDOMAIN ...
[INFO] ... "A IN example.com." NOERROR ...

      That's the ladder from earlier, playing out verbatim in the logs, in order, exactly as predicted. Remove the log line the same way you added it once you're done watching this.

    3. Know all four dnsPolicy values — worth noting up front that their names are a little misleading, so read the meaning column, not just the name:
      dnsPolicyMeaning
      ClusterFirstthe default above — despite the name, it's not "try cluster first", it's this resolv.conf
      ClusterFirstWithHostNetsame file, for hostNetwork: true pods — which otherwise get the node's resolv.conf

      Noneyou supply everything via dnsConfig
    Verify

    Predict the answer first, then confirm it with the same log trick: how many queries does resolving db.other take, assuming that service actually exists? (It's a two-part name, so the first rung, db.other.default.svc..., fails; the second rung, db.other.svc.cluster.local, succeeds — two queries total.)

    Gotchas

    The hostNetwork trap is a favorite for planted faults precisely because the symptom is so counterintuitive: a pod with hostNetwork: true under the default dnsPolicy inherits the node's own resolv.conf instead of the cluster-aware one described above, so it can't resolve *.svc.cluster.local names at all — while every ordinary pod on the same node resolves them fine. The fix is one line: dnsPolicy: ClusterFirstWithHostNet. Learn to recognize the symptom pattern itself — DNS failing only for host-network pods, working normally for everything else on the same node — since that pattern alone should point you straight at this fix before you've even opened the pod spec.

    4.17

    The record shapes: A, headless A, SRV — verify each with dig

    Why it matters

    An exam task might simply say "make the app reach the database at db.prod" and leave the rest to you — you need the full DNS record grammar memorized so you know instantly what that name expands to, which zone it's actually asking about, and whether it can even work as written.

    4.18

    Break DNS twice and drill the debug ladder

    Why it matters

    "Pods can't resolve names" is a guaranteed member of the troubleshooting-task family, precisely because there are only about five distinct places DNS can actually break, and walking them in a fixed order beats relying on intuition or guesswork every single time.

    The concept

    Here's the ladder, from the bottom of the stack up to the top of it:

    1 pod resolv.conf sane?→2 kube-dns svc + endpoints exist?→3 can I query 10.96.0.10 directly?→4 CoreDNS pods running? logs?→5 Corefile valid?

    Step 3 is the pivot point the whole ladder hinges on: if querying 10.96.0.10 directly with dig @10.96.0.10 works, but ordinary resolution through the pod's normal resolv.conf doesn't, then the problem has to be the pod's resolv.conf itself (step 1 was lying to you, or a different pod has a broken one). But if that same direct query just hangs with no answer at all, the problem is somewhere further along the service path — endpoints, kube-proxy, or the CNI — or in CoreDNS itself (steps 4–5).

    Lab
    1. Fault A — kill the backends. Scale CoreDNS down to zero replicas and watch resolution die from the client's point of view: 
      kubectl -n kube-system scale deploy coredns --replicas=0
kubectl exec dnsutils -- nslookup web
      ;; connection timed out; no servers could be reached
    2. Walk the ladder step by step and find the smoking gun waiting at rung 2: 
      kubectl exec dnsutils -- cat /etc/resolv.conf | head -2
kubectl -n kube-system get svc kube-dns
kubectl -n kube-system get endpointslices -l k8s-app=kube-dns
      search default.svc.cluster.local ...          ← rung 1: sane
kube-dns   ClusterIP   10.96.0.10   53/UDP... ← rung 2a: svc exists
kube-dns-x9k2p   IPv4   <unset>              ← rung 2b: no endpoints — nothing behind the VIP

      Timeout plus empty endpoints together mean exactly one thing: nobody's home to answer at all — there's no CoreDNS pod left standing behind the Service, so the request just times out waiting for a response that will never come. Heal it with: kubectl -n kube-system scale deploy coredns --replicas=2.

    3. Fault B — corrupt the Corefile itself. Insert a directive CoreDNS doesn't recognize: 
      kubectl -n kube-system get cm coredns -o yaml | sed 's/    errors/    errors\n    bogus/' | kubectl apply -f -
kubectl -n kube-system rollout restart deploy coredns
kubectl -n kube-system get pods -l k8s-app=kube-dns
      coredns-7c65d6cfc9-k2m4x   0/1   CrashLoopBackOff   2 (18s ago)
    4. The pod's own logs will name the exact broken line — this is rung 4 of the ladder handing you directly to rung 5: 
      kubectl -n kube-system logs -l k8s-app=kube-dns --tail=3
      plugin/kubernetes: Corefile:3 - Error during parsing: Unknown directive 'bogus'
    5. Fix the ConfigMap by removing the bad line, restart CoreDNS, and confirm the whole path works end to end again: 
      kubectl -n kube-system edit cm coredns        # delete the 'bogus' line
kubectl -n kube-system rollout restart deploy coredns
kubectl exec dnsutils -- nslookup web | tail -2
      Name: web.default.svc.cluster.local
Address: 10.101.44.7
    Verify

    Both faults produced visibly different first symptoms — a plain timeout for fault A versus a CrashLoopBackOff for fault B — and you reached each one's actual root cause by walking the ladder methodically, not by randomly poking at things until something worked.

    Gotchas

    Learn to recognize a subtle third fault by its signature alone, without needing to reproduce it: DNS works fine for cluster names but fails specifically for external ones ⇒ the forward plugin's upstream is broken somehow (a bad node resolv.conf, or an edited forward line in the Corefile) — because cluster-name answers come straight from the apiserver's own watches and never touch that upstream at all, so they keep working regardless. The reverse pattern — external names resolve fine, but cluster names are dead — instead points at the kubernetes plugin's own zone configuration or RBAC permissions, or simply a wrong nameserver IP baked into pods' resolv.conf. Clean up the practice pods when you're done with them: kubectl delete pod dnsutils web4.

    Ingress & Gateway API

    4.19

    Deploy an ingress controller; write an Ingress with host, path and TLS

    Why it matters

    Everything in this week so far has been about traffic already inside the cluster reaching a pod. An Ingress is about the opposite direction: letting HTTP/HTTPS traffic from outside the cluster in, routed by hostname and URL path to the right Service — the way a building's receptionist hears "I'm here for Acme Corp on the third floor" and sends the visitor to the right door, rather than the visitor needing to know which physical office to walk into. Writing one from a plain task description — the right host, the right pathType, the right backend, a TLS secret correctly attached — is a standing exam item. So is its most common trap: an Ingress object that "does nothing" at all, because no controller exists to actually serve it.

    The concept

    An Ingress object is only data sitting in etcd — a set of host/path routing rules, and nothing more; creating one, by itself, changes no actual network behavior whatsoever. The thing that turns that data into working traffic routing is the ingress controller: an ordinary reverse proxy running as regular pods, which watches Ingress objects the same way any controller watches anything, and renders them into its own live proxy configuration, forwarding matched requests directly to pod IPs pulled from EndpointSlices (4.11) — deliberately bypassing the Service's ClusterIP and DNAT machinery entirely, going straight to the pods. If no controller is running at all, every Ingress object in the cluster just sits there, inert, accepted by the API but doing absolutely nothing — and critically, silently, with no error anywhere telling you so. The ingressClassName field on an Ingress is what says which specific controller, if more than one is installed, should be the one to pick up and serve that particular set of rules.

    A fact worth knowing and verified fresh for this page: ingress-nginx is retired (archived March 2026, receiving no further security patches) — it's still perfectly fine as the Helm-mechanics example W3 used it for, but the wrong choice to build anything new on top of. This lab replaces it with Traefik instead, which happens to serve both Ingress and Gateway API (4.20) — meaning one single controller carries you through this entire lesson group, and gives you more real Helm practice along the way.

    Lab
    1. Retire the retiree first — W3's exercise release (3.25) is still sitting there running; use this as practice for a clean teardown before installing its replacement: 
      helm uninstall ingress-nginx -n ingress-nginx

    Gotchas

    pathType genuinely matters and is a required field, not optional: Prefix matches by whole path segment, so /app matches both /app itself and /app/x, but never something like /apple that merely starts with the same letters; Exact, by contrast, matches only the literal path given, so /app under Exact does not match /app/ with a trailing slash. An Ingress with no ingressClassName set at all only gets served if some other IngressClass in the cluster has been explicitly annotated as the default — so "my Ingress does nothing" is very often just a missing or wrong class, or no controller running at all, rather than a bug in the rules themselves. Always run kubectl get ingressclass first, before questioning anything else.

    4.20

    Gateway API: the resource model and its role split

    Why it matters

    Gateway API is the newest thing on the whole CKA curriculum, which makes it predictable exam material — exam writers love testing recently-added topics precisely because most candidates haven't gotten around to them yet. Thirty minutes spent here is disproportionately valuable in points earned per minute studied.

    The concept

    Gateway API takes the single job Ingress does — "get outside traffic routed into the cluster" — and deliberately splits it across three separate resources, one for each of three distinct roles that, in a real organization, are usually held by different people:

    The stable ("standard-channel") resources live under the API group gateway.networking.k8s.io/v1: GatewayClass, Gateway, HTTPRoute, GRPCRoute (plus ReferenceGrant, still at the earlier v1beta1 stability level, which grants explicit permission for routes to reach backends in a different namespace). All of these are CRDs (CustomResourceDefinitions, W3) — meaning they're installed separately as an add-on, and are not shipped as part of Kubernetes itself. That's exactly why a completely fresh cluster answers kubectl get gateway with "the server doesn't have a resource type" — the type genuinely doesn't exist yet, until you install it.

    Lab

  --set providers.kubernetesGateway.enabled=true \
  --set gateway.listeners.web.namespacePolicy.from=All

      That second flag sets the Gateway's own allowedRoutes field so routes coming from any namespace are allowed to attach to it — by default, a Gateway only accepts HTTPRoutes living in its own namespace, and forgetting that restriction is the single most common reason someone's freshly-written HTTPRoute appears to be silently ignored.

    3. See exactly what the Helm chart created on your behalf — one GatewayClass, and one Gateway with a listener open on port 80: 
      kubectl get gatewayclass
kubectl -n traefik get gateway
      NAME      CONTROLLER                      ACCEPTED   AGE
traefik   traefik.io/gateway-controller   True       30s
NAME              CLASS     ADDRESS   PROGRAMMED   AGE
traefik-gateway   traefik             True         30s

      The ACCEPTED and PROGRAMMED columns are Gateway API's biggest practical improvement over Ingress's total silence when something's wrong: every single resource carries explicit status conditions saying, in plain terms, whether some controller implementation actually picked it up and is acting on it. Seeing False, or the condition missing entirely, is always your very first debugging stop — before you go looking anywhere else.

    4. Read the Gateway object itself to anchor the model firmly: notice that listener configuration (which ports, which protocols) lives here, on the Gateway, not on any individual route: 
      kubectl -n traefik get gateway traefik-gateway -o yaml | grep -A6 listeners:
        listeners:
  - name: web
    port: 80

      namespaces: { from: All }
    Verify

    The GatewayClass shows ACCEPTED True, the Gateway shows PROGRAMMED True, and you can name, without looking anything up, which of the three roles owns each of the three resources.

    Gotchas

    Real exam environments may well pre-install the CRDs and a working Gateway for you already — in which case the entire task reduces to "write an HTTPRoute" (4.21), with none of this setup required. Always run kubectl get gatewayclass,gateway -A first, before writing anything, purely to learn the exact names you'll need to reference. A wrong parentRefs name doesn't error out when you apply it — it applies cleanly and then fails quietly as a status condition on the route itself, visible only via kubectl describe httproute.

    4.21

    Write an HTTPRoute from memory — including a weighted canary

    Why it matters

    Of the three Gateway API resources from 4.20, the HTTPRoute is the one an actual exam task will make you sit down and write yourself. Its single killer feature over Ingress — genuine, first-class traffic weights for splitting requests between backends — is also the single most likely thing to be asked about, precisely because Ingress has no built-in way to express it at all.

    The concept

    An HTTPRoute has three parts: parentRefs (which Gateway this route is asking to attach to — the arrow pointing back at 4.20's picture), hostnames it should match, and a list of rules — where each rule pairs some matches (on path, header, or query string) with one or more backendRefs (Services to send matching traffic to, each optionally carrying a weight). Weights split incoming traffic proportionally across backends: a 90/10 split below means, on average, one request out of every ten goes to the canary backend instead of the main one.

    Lab
    1. Create a second backend that answers with visibly different text, so a canary split is easy to eyeball in the output (Apache says "It works!", nginx says "Welcome to nginx"): 
      kubectl create deploy web-v2 --image=httpd
kubectl expose deploy web-v2 --port 80
    2. Write the route by hand once yourself first, then check what you wrote against this reference: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: shop-canary
spec:
  parentRefs:
  - name: traefik-gateway
    namespace: traefik
  hostnames: ["gw.lab.test"]
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: web
      port: 80
      weight: 90
    - name: web-v2
      port: 80
      weight: 10
EOF
kubectl describe httproute shop-canary | grep -A4 Conditions:
      Conditions:
  Type: Accepted     Status: True
  Type: ResolvedRefs Status: True
    3. Prove the weighted split actually happens by firing 50 requests through the Gateway's listener (reusing the same web NodePort from 4.19) and tallying which backend answered each one: 
      for i in $(seq 1 50); do curl -s http://node01:30080/ -H 'Host: gw.lab.test'; done \
  | grep -o 'Welcome to nginx\|It works' | sort | uniq -c
           5 It works
    45 Welcome to nginx   ← ≈90/10; exact counts wobble, the ratio doesn't lie
    4. Know that header-based matching also exists as an option — add a rule that deliberately routes any request carrying a specific test header straight to the canary, overriding the weights entirely: 
          - matches:
      - path: { type: PathPrefix, value: / }
        headers:
        - name: X-Canary
          value: "yes"
      backendRefs:
      - { name: web-v2, port: 80 }

      (Multiple match conditions listed inside one rule combine with AND; separate rules combine with OR, and the more specific rule wins when both could apply — this is the exact same AND/OR shape you'll meet again, spelled out fully, in NetworkPolicy's from rules in 4.25.)

    Verify

    The 50-request loop lands at roughly 45/5, matching the 90/10 weight, and describe httproute shows Accepted: True. Try breaking parentRefs.name on purpose, pointing it at a Gateway that doesn't exist, and watch that condition flip to false — that's the debugging signature worth remembering.

    Gotchas

    Two classic silent-failure traps to watch for: parentRefs must explicitly name the Gateway's namespace whenever the route itself lives somewhere else (the default, if you omit it, is to assume the route's own namespace); and routing to a backend Service in a different namespace additionally requires a ReferenceGrant object created in that backend's own namespace granting permission — without it, the route shows ResolvedRefs: False and clients get 500 errors, with no other indication of why.

    Docs

    Docs search: "Gateway API" — the HTTPRoute example on that page is the template to memorize.

    4.22

    Say why Gateway API exists — the two-minute answer

    Why it matters

    Understanding-type exam questions ("which resource would you reach for to…") are cheap, easy points to pick up — but only if you can hold this specific comparison crisply in your head, because the reasoning behind it is also what tells you which tool a task actually wants, even in cases where the task doesn't name one explicitly.

    The pitch
    • Ingress hit a real ceiling. It has exactly one object and one small vocabulary: host/path → service, nothing more. So everything people actually needed in practice — timeouts, URL rewrites, TLS options beyond the basics, canary traffic weights — had to be bolted on as controller-specific annotations: plain untyped strings, never validated by the API, and not portable from one controller to another (widely nicknamed "annotation soup" for exactly this reason). Ingress itself is GA (generally available) and deliberately frozen — it will keep working forever, but it will never gain a single new capability.
    • Gateway API is typed and split by role. Things like weights, header matches, and traffic filters are real, schema-validated fields on the object itself, not magic strings in an annotation. The three-way split from 4.20 — GatewayClass, Gateway, HTTPRoute — cleanly separates infrastructure, platform, and application concerns instead of cramming them into one object; routes are namespaced and can be governed by RBAC per team; and every resource reports explicit status conditions instead of Ingress's characteristic silent failure.
    • The whole ecosystem has already moved. ingress-nginx — the default choice for the better part of a decade — is now retired (as of March 2026); its intended successors, and essentially every modern data-plane project (Envoy Gateway, Cilium, Traefik among them), put Gateway API first in their own roadmaps. New clusters increasingly start on Gateway API from day one; Ingress mostly persists because of the huge installed base of clusters already using it.
    • The exam angle specifically: Gateway API entered the curriculum in the 2025 revision, both killer.sh (the official practice-exam simulator) and the real CKA exam now feature it, and most existing third-party study material predates all of this — which makes it genuinely easy points for anyone who actually prepared for it.
    Verify

    Deliver these four bullets out loud, from memory, in under two minutes, to an imaginary colleague — then answer their obvious follow-up question: "so should we go migrate all our existing Ingresses right now?" (No rush at all — Ingress keeps working exactly as it always has; new routes, and anything that actually needs weights or typed config, are what should go to Gateway API instead.)

    Gotchas

    Never say "Ingress is deprecated" — it isn't, and getting this precise distinction right is the entire point of this checkpoint. The correct word is frozen: stable, still fully supported, simply not growing any further. The pressure to move to Gateway API comes from individual controllers being retired and from missing features, not from the Ingress API itself being removed or deprecated by Kubernetes.

    Docs

    Docs search: "Gateway API" and "Ingress" — the Ingress page itself points migrators at Gateway API.

    4.23

    The NetworkPolicy model: select, default-deny, additive allows

    Why it matters

    Everywhere else this week has been about reachability — making sure traffic can get where it's going. A NetworkPolicy is the opposite instinct: a way to say "only these specific pods may talk to this pod," the same way a building might restrict a sensitive room to a named list of visitors instead of leaving every door open to anyone inside. NetworkPolicy questions on the exam are pure model questions — the YAML itself is genuinely short, and nearly every mistake traces back to misunderstanding one of the three rules below rather than a YAML syntax slip. Get the model itself right, in your head, and the YAML basically writes itself afterward.

    The concept

    Three rules carry the entire model:

    • Policies select pods, not traffic directly. The field spec.podSelector picks which pods this particular policy protects, scoped to the policy's own namespace only. Setting podSelector: {} (an empty selector) selects every pod in that namespace, with no exceptions.
    • Being selected flips a pod's default from open to closed. A pod that no policy selects at all simply accepts everything — Kubernetes ships open-by-default, with no NetworkPolicy meaning no restriction whatsoever. But the moment any policy selects a given pod, that pod flips to default-deny, but only for whichever directions are listed in that policy's policyTypes field — and nothing else changes for it.
    • Rules are additive allows, never denies. The ingress:/egress: entries punch specific holes in that default-deny wall. There is no such thing as a deny rule in NetworkPolicy at all — you cannot ever write "block traffic from X"; you can only write "allow traffic from Y," and let the default-deny you triggered by being selected do all the actual blocking implicitly. When more than one policy selects the same pod, their individual sets of allows simply get unioned together.

    The sharpest precision trap in the whole model lives in the policyTypes field: an empty rules list (writing ingress: [], or simply leaving the key out entirely) combined with policyTypes: [Ingress] allows absolutely nothing in — a complete ingress lockdown, full stop. But if you write only egress rules and omit policyTypes altogether, it silently defaults to just [Egress] — meaning ingress stays completely wide open the whole time, which is very often not what the task actually intended. The safe habit: always state policyTypes explicitly yourself, every single time, and never rely on the default to guess correctly.

    And one asterisk that changes everything about whether any of this actually works: it's the CNI plugin that implements NetworkPolicy enforcement — not kube-proxy, and not the Linux kernel by default on its own. Calico and Cilium do enforce policies; flannel, your lab's default CNI, ignores NetworkPolicy objects entirely — they apply to the API cleanly, with no error, and then simply do nothing at all. The apiserver happily accepting your NetworkPolicy object says absolutely nothing about whether any traffic is actually being enforced against it.

    Verify

    Answer this without checking any notes: a pod in namespace a is selected by exactly one policy with policyTypes: [Ingress] and ingress: []. Can that pod still reach out to the internet on its own? (Yes — egress was never touched or restricted by this policy.) Can anything at all reach into it? (No — not even another pod sitting right next to it in the same namespace.)

    Gotchas

    Policies govern connection establishment, not the individual packets that make up a reply: allowing ingress into a pod implicitly also allows that pod's responses back out, because the kernel's own connection tracking (conntrack, from 4.10) already knows those reply packets belong to an already-permitted connection — you never need a matching egress rule on the server side purely for return traffic. But a client pod whose own egress has been locked down by a separate policy does still need its own explicit egress allow in order to initiate a new connection in the first place. The mental model that keeps this straight: think in terms of whole connections, never in terms of individual packets.

    Docs

    Docs search: "Network Policies" — the concepts page; its "behavior of to/from selectors" section is 4.25's material.

    4.24

    The classic three policies, written and proven

    Why it matters

    Default-deny-all, allow-from-one-app, allow-from-another-namespace — nearly every NetworkPolicy exam task turns out to be one of these three patterns, or a two-line variation on one of them. The goal here is to write all three from memory, and then prove each one actually did what you intended with a real curl, because a policy you never tested against real traffic is exactly how you quietly lose points without realizing it.

    The concept

    Your lab's flannel doesn't enforce NetworkPolicy at all, as 4.23 already established, so this lab first swaps the whole cluster's CNI over to Calico, which does enforce it — itself excellent hands-on practice, since it's the exact 4.5–4.8 CNI knowledge applied for real, on a live cluster, rather than just discussed. The swap is protected by a VM snapshot taken first; you restore it afterward to bring the course back to its canonical flannel setup.

    Lab
    1. Take the safety snapshot first, from the host machine, before touching anything: 
      for vm in cp node01 node02; do multipass stop $vm; multipass snapshot $vm --name pre-calico; multipass start $vm; done
    2. Remove flannel and every leftover it left behind — delete its manifest cluster-wide, then clean up the interfaces it created on each individual node: 
      kubectl delete -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
# on cp, node01 AND node02:
sudo rm /etc/cni/net.d/10-flannel.conflist
sudo ip link del cni0; sudo ip link del flannel.1
    3. Install Calico (v3.32.1 is current at the time of authoring), and explicitly tell it your existing pod CIDR so it reuses the same address ranges every pod already has: 
      curl -sO https://raw.githubusercontent.com/projectcalico/calico/v3.32.1/manifests/calico.yaml
sed -i 's|# - name: CALICO_IPV4POOL_CIDR|- name: CALICO_IPV4POOL_CIDR|;
        s|#   value: "192.168.0.0/16"|  value: "10.244.0.0/16"|' calico.yaml
kubectl apply -f calico.yaml
kubectl -n kube-system get pods -l k8s-app=calico-node -w   # wait for 3× Running
kubectl delete pods -A --all   # re-wire every pod onto the new CNI
      calico-node-4x2lp   1/1   Running   ... (one per node)
    4. Build a small test rig — one server pod and one client pod — whose reachability you can check with a single line each time: 
      kubectl create deploy web --image=nginx 2>/dev/null; kubectl expose deploy web --port 80 2>/dev/null
kubectl run client --image=busybox:1.36 -l app=client --restart=Never -- sleep 1d
alias probe="kubectl exec client -- wget -qO- --timeout=2 http://web"
probe && echo OPEN
      <title>Welcome to nginx!</title> ... OPEN
    5. Policy 1 — namespace default-deny (ingress). Write it, apply it, and watch the door close on traffic that worked a moment ago: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
spec:
  podSelector: {}
  policyTypes: [Ingress]
EOF
probe || echo BLOCKED
      wget: download timed out
BLOCKED     ← and notice specifically: a NetworkPolicy block shows up as a TIMEOUT, never a "connection refused"
    6. Policy 2 — allow one specific app in, on one specific port. This policy's allow rule adds a hole on top of policy 1's default-deny, rather than replacing it: 
      cat <<'EOF' | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-client-to-web
spec:
  podSelector:
    matchLabels: { app: web }
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels: { app: client }
    ports:
    - protocol: TCP
      port: 80
EOF
probe && echo CLIENT-ALLOWED
      ... CLIENT-ALLOWED   ← a pod without the app=client label still times out — try it yourself and confirm
    7. Policy 3 — allow an entire other namespace. This one uses the automatic name label every namespace in the cluster carries by default: 
      kubectl create ns monitoring
kubectl -n monitoring run prober --image=busybox:1.36 --restart=Never -- sleep 1d
cat <<'EOF' | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-monitoring-ns
spec:
  podSelector:
    matchLabels: { app: web }
  policyTypes: [Ingress]
  ingress:
  - from:
    - namespaceSelector:
        matchLabels: { kubernetes.io/metadata.name: monitoring }
EOF
kubectl -n monitoring exec prober -- wget -qO- --timeout=2 http://web.default && echo NS-ALLOWED
      ... NS-ALLOWED
    8. Once you're done with 4.25 as well (which continues on this same Calico cluster), restore the original flannel cluster from the snapshot: 
      for vm in cp node01 node02; do multipass stop $vm; multipass restore $vm.pre-calico --destructive; multipass start $vm; done
    Verify

    You personally observed all three transitions happen: open turning into blocked (policy 1), blocked turning into allowed for exactly the one labeled client and nothing else (policy 2), and allowed for exactly the one labeled namespace and nothing else (policy 3) — each one proven with a real probe, never just assumed to be true.

    Gotchas

    Policy 2 quietly demonstrates an important scoping detail: it selects pods labeled app: web, which means the client pod itself was never selected by this policy at all, so its own egress traffic never needed to be explicitly allowed anywhere. If a task separately demands that the client's egress also be locked down, the client needs its own dedicated policy — and that policy must additionally carve out a DNS hole (allowing port 53, protocol UDP outbound to kube-system), or every single name lookup the client tries dies before the actual connection is even attempted. A forgotten DNS egress rule is the single most common self-inflicted NetworkPolicy wound — easy to forget, and it breaks everything downstream of it.

    4.25

    The AND/OR trap in from/to rules — test both

    Why it matters

    This is the single most-tested NetworkPolicy subtlety on the whole exam, precisely because two YAML files that differ by a single dash character mean completely opposite things in practice — and both versions apply to the apiserver cleanly, with no warning or error telling you which one you actually wrote.

    The concept

    Inside one single from: list item, multiple selectors combine with AND logic — meaning a peer pod must satisfy every one of them simultaneously to match. But separate, sibling from: list items instead combine with OR logic — meaning a peer only needs to satisfy any one of them to match. The difference between the two, in raw YAML, comes down entirely to where a dash is placed:

    # A — ONE from element (no dash before podSelector): AND
ingress:
- from:
  - namespaceSelector:
      matchLabels: { team: qa }
    podSelector:
      matchLabels: { app: client }
# meaning: pods labeled app=client IN namespaces labeled team=qa. Only them.

# B — TWO from elements (note the extra dash): OR
ingress:
- from:
  - namespaceSelector:
      matchLabels: { team: qa }
  - podSelector:
      matchLabels: { app: client }
# meaning: ANY pod in team=qa namespaces, PLUS app=client pods
# from the policy's OWN namespace.

    There's a second, easy-to-miss subtlety hiding inside variant B specifically: a bare podSelector appearing in a from rule always means "in this policy's own namespace" implicitly — it never scans across other namespaces on its own, even when it's sitting in an OR list next to a namespaceSelector that does reach other namespaces.

    Lab

    (This continues on the same Calico cluster from 4.24, with default-deny-ingress still in place. Delete allow-client-to-web and allow-monitoring-ns first, so their existing holes don't interfere with what you're about to test: kubectl delete netpol allow-client-to-web allow-monitoring-ns.)

    1. Build a small test matrix of three separate probers, deliberately covering every combination of namespace and label: 
      kubectl create ns qa && kubectl label ns qa team=qa
kubectl -n qa run qa-client --image=busybox:1.36 -l app=client --restart=Never -- sleep 1d
kubectl -n qa run qa-other  --image=busybox:1.36 --restart=Never -- sleep 1d
# plus 'client' (app=client, default ns) from 4.24
    2. Apply variant A (the AND form) targeting your web pods, then probe all three testers and see which ones get through: 
      kubectl -n qa exec qa-client -- wget -qO- -T2 http://web.default >/dev/null && echo qa-client:OK || echo qa-client:BLOCKED
kubectl -n qa exec qa-other  -- wget -qO- -T2 http://web.default >/dev/null && echo qa-other:OK  || echo qa-other:BLOCKED
kubectl exec client          -- wget -qO- -T2 http://web         >/dev/null && echo client:OK    || echo client:BLOCKED
      qa-client:OK        ← matches BOTH selectors at once, as AND requires
qa-other:BLOCKED     ← right namespace, but wrong pod label — fails the AND
client:BLOCKED       ← right label, but wrong namespace — fails the AND
    3. Now swap in variant B (the OR form — just add the extra dash) and re-run the exact same three probes, unchanged: 
      qa-client:OK         ← matches the namespace leg of the OR
qa-other:OK          ← also matches the namespace leg of the OR
client:OK            ← matches the podSelector leg instead (evaluated in its own namespace)
    4. Clean up your test objects, then restore flannel following step 8 of 4.24: 
      kubectl delete ns qa monitoring; kubectl delete netpol --all; kubectl delete pod client
    Verify

    The probe matrix flipped exactly as predicted between the two variants — 1 out of 3 allowed under the AND form, versus all 3 out of 3 allowed under the OR form — and you should be able to point directly at the one dash character responsible for that entire difference.

    Gotchas

    In the real exam, plain YAML indentation is what decides this, with nothing else marking the difference visually. Whenever you copy a policy snippet from the docs or from memory, deliberately check whether podSelector sitting under from has its own leading - or not. Reading the rule back out loud in plain English — "allow from peers that are (X and Y)" versus "allow from X-type peers, or separately from Y-type peers" — catches this kind of error faster than staring silently at the raw YAML ever will.

    Docs

    Docs search: "Network Policies" — the concepts page shows this exact AND/OR pair side by side; find it once now so you can find it in three seconds then.

    Week 4 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. The Kubernetes network model in two rules — and which common IP type is outside the model?
      Answer

      Every pod gets its own IP; all pods reach all pods without NAT. Service ClusterIPs are outside it — virtual, rule-manufactured, on no interface (4.1/4.9).

    2. What three Linux objects make up one pod's network plumbing, and who keeps the netns alive?
      Answer

      A network namespace, a veth pair (one end renamed eth0 inside, the other on the node bridge), and the bridge (cni0). The pause container holds the namespace open (4.2).

    3. Node NotReady, message "cni plugin not initialized". Which two directories do you check, and why do existing pods keep working?
      Answer

      /etc/cni/net.d (config) and /opt/cni/bin (binaries). Existing pods keep their already-wired veths — CNI is exec'd per pod creation, so only new sandboxes fail (4.5/4.8).

    4. You tcpdump the node NIC and can't see pod IPs, only node-to-node UDP on port 8472. What are you looking at?
      Answer

      A VXLAN overlay (flannel) — pod packets are encapsulated between nodes; Linux VXLAN defaults to 8472, not IANA's 4789. Capture on flannel.1 to see the inner pod traffic (4.7).

    5. Why does curl to a ClusterIP work while ping to it shows 100% loss?
      Answer

      The IP exists only as nat-table rules matching tcp dpt:80; DNAT rewrites the curl packet to a pod IP. ICMP matches no rule and nothing owns the IP, so ping dies. Never test services with ping (4.9).

    6. Walk the three iptables layers for a service, and name what pins a TCP flow to one backend.
      Answer

      KUBE-SERVICES (match ClusterIP:port) → KUBE-SVC-* (statistic-module random split) → KUBE-SEP-* (DNAT to podIP:port). conntrack pins the flow. kube-proxy compiles the rules; it proxies nothing (4.10).

    7. Service test: "connection refused" vs "timeout" — what does each point at?
      Answer

      Refused = something answered: no-ready-endpoints REJECT rule, or DNAT to a port nothing listens on (targetPort bug). Timeout = nothing answered: missing rules (kube-proxy), NetworkPolicy drop, or CNI/routing (4.10/4.14/4.24).

    8. What exact chain of events removes a pod from service load-balancing when its readinessProbe fails?
      Answer

      Probe fails → kubelet marks the pod not-ready → EndpointSlice controller sets that endpoint ready: false → kube-proxy recompiles rules without it (and CoreDNS drops it from headless answers). Selector picks members; readiness gates traffic (4.11/4.13).

    9. port, targetPort, nodePort — whose port is each, and which one is "the most-planted service bug"?
      Answer

      port = the service's own; targetPort = the container's (DNAT destination, defaults to port, may be a name); nodePort = 30000–32767 on every node. The classic bug: targetPort ≠ what the container listens on — endpoints present, port wrong, connection refused (4.14).

    10. Why does a bare name like db resolve, and why is ndots:5 a tax on external lookups?
      Answer

      Names with <5 dots try the search list first: db.<ns>.svc.cluster.local → db.svc.cluster.local → db.cluster.local → db.. External names like www.example.com (2 dots) climb the same ladder — three NXDOMAINs before the real query; a trailing dot skips it (4.16).

    11. Only hostNetwork pods can't resolve *.svc.cluster.local. Diagnosis and the one-line fix?
      Answer

      hostNetwork pods inherit the node's resolv.conf under default policy. Fix: dnsPolicy: ClusterFirstWithHostNet (4.16).

    12. An Ingress object exists but "does nothing", silently. Name the two most likely causes and your first command.
      Answer

      No ingress controller running, or wrong/missing ingressClassName. First: kubectl get ingressclass (then check controller pods). Ingress is only data — a controller must render it (4.19).

    13. Gateway API's three resources — who owns each, and how does an HTTPRoute attach?
      Answer

      GatewayClass (implementation — infra), Gateway (listeners/TLS — platform), HTTPRoute (matching + backendRefs — app team), attached via parentRefs and admitted by the Gateway's allowedRoutes. Status conditions (Accepted/Programmed/ResolvedRefs) replace Ingress's silence (4.20/4.21).

    14. One from: element with namespaceSelector + podSelector vs two elements — and what does a policy with policyTypes: [Ingress], ingress: [] do?
      Answer

      One element = AND (both must match the same peer); two elements = OR (either suffices; the bare podSelector leg means the policy's own namespace). ingress: [] + selection = full ingress default-deny for the selected pods — egress untouched. And remember flannel enforces none of it; policy enforcement belongs to the CNI (4.23–4.25).
