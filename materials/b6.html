    B6 — Isolation I: namespaces, chroot, pivot_root

    Everything you have run so far has shared one view of the machine: the same process list, the same hostname, the same filesystem, the same network. This module is where that stops being a law of nature. A Linux namespace lets one process — and its children — get a private answer to one specific question about the machine ("which processes exist?", "what is my hostname?", "which network interfaces are there?") while every other process keeps the old answer. Stack a handful of these together, wrap them around a private root filesystem, and you have built the thing the industry calls a container. Not a metaphor for one — an actual one, about twenty-five lines of shell, with no runtime installed. By the end of this module you will have done exactly that.

    The word "container" hides how humble the pieces are, so we take them one at a time. First the idea: a namespace is your own copy of one kernel resource, and you can see its identity as an inode number under /proc. Then one namespace at a time with the unshare command — a new hostname, a new process list where your shell is PID 1, a private mount table, an empty IPC world, a remapped user id, a re-rooted cgroup view — feeling exactly what each one hides. Then roots and entering: chroot and why it is a convenience rather than a wall, pivot_root which is the real thing runtimes use, and nsenter — which is, quite literally, what kubectl exec does. A lone network namespace gets created here and wired up in B9. Then you assemble container v0.1.

    B2 for processes, PIDs, and signals — a PID namespace is a private process list and killing PID 1 in it is a signal story; B3 because unshare, setns, and clone are system calls and the whole mechanism lives in the kernel you met there; B5 because a mount namespace is a private copy of the mount table, pivot_root is a mount operation with propagation preconditions, and container v0.1 boots on the rootfs you built in B5.17. Every lab runs on the throwaway sandbox VM (B0.1), never on the cluster — several of these commands hold or detach namespaces and can leave stray mounts if fumbled, so take a snapshot (B0.9) first. Everything here ships in util-linux (unshare, nsenter, lsns) and iproute2 (ip netns), already on the base image; the user-namespace lesson notes one Ubuntu 24.04 default you will have to understand before it will work unprivileged.

    The idea

    B6.1

    What a namespace actually is: your own copy of one kernel resource

    Why it matters

    "Container" sounds like it must be a small virtual machine, and almost every beginner carries that wrong picture for far too long. Getting the real idea early — before any command — saves you from that mistake and makes everything downstream obvious. A container is not a machine at all. It is an ordinary process on your ordinary kernel, which has simply been handed private answers to a few specific questions. The mechanism that hands out those private answers is the namespace, and it is much smaller and less magical than "virtual machine" suggests.

    The concept

    A namespace is a private copy of exactly one kernel-managed resource. Normally the kernel keeps one global version of each such resource — one list of all processes, one hostname, one set of network interfaces, one mount table — and every process sees the same one. A namespace splits one of those globals into an isolated instance: the processes inside it see and change their copy, and processes outside are untouched. Crucially, it is still the same kernel, the same CPU scheduler, the same memory manager, the same clock ticking. Nothing is being emulated or virtualised. Only the bookkeeping tables the kernel consults to answer a specific question have been forked in two.

    Back to the hotel from B2, where one hotel is one Linux machine: the building has one of many registers — a guest list, a directory of room phones, a hostname on the front sign. A namespace is the moment the front desk gives one room its own copy of one of those registers. The room writes in its guest list, reads from its guest list, and genuinely cannot tell that the rest of the hotel exists — yet it is the same building, the same plumbing, the same staff. It is not a separate hotel (that would be a virtual machine, with its own engineering crew and boiler). It is one room that has been handed a private ledger.

    Three things follow, and they are the whole reason the rest of this module works:

    • Each type is independent. You can give a process a private process list but leave it sharing the host's network, or a private hostname but the host's filesystem. A container is just a particular bundle of namespaces chosen together; there is nothing atomic about "a container".
    • Children inherit, by default. When a process in a namespace forks, the child is in the same namespaces — which is how one unshare at the top gives a whole tree of processes the same private view.
    • It is cheap. There is no second kernel to boot, no memory to pre-allocate. Creating a namespace is a single system call; this is exactly why containers start in milliseconds and a VM starts in seconds.
    one kernel· one globalprocess list · hostname · mounts · network→ namespace forksone of them·everything else stays shared

    Say, in one sentence, the difference between a namespace and a virtual machine. (A VM emulates a whole machine with its own kernel; a namespace is one real kernel giving a process a private copy of a single resource — same kernel, forked bookkeeping.) Then: you changed the hostname inside unshare --uts but the host's stayed sandbox. Which register was copied, and which were still shared? (Only the UTS/hostname register was private; the process list, mounts, and network were all still the host's.)

    Gotchas

    "Namespace" is one of the most overloaded words in computing — Kubernetes Namespaces, C++ namespaces, XML namespaces, DNS namespaces. This module means exactly one thing: a Linux kernel namespace, the process-isolation primitive. A Kubernetes Namespace (a label that groups API objects) is completely unrelated and does not create a single Linux namespace. Keep the two in separate mental drawers; the collision trips up nearly everyone once.

    Isolation is not security by itself. A namespace changes what a process can see, not automatically what it can do — a process with the right capabilities can often create, enter, or escape namespaces. Real container isolation stacks namespaces together with capabilities (B4), cgroups (B7), and seccomp/AppArmor. Treating a bare namespace as a security wall is a classic and dangerous mistake; we hit its sharpest edge in the chroot lesson (B6.10).

    This is the single most clarifying fact for reading Kubernetes: a pod is not a small VM, it is a set of processes sharing a chosen bundle of Linux namespaces. The containers in one pod share a network namespace (same IP, same localhost, same ports — which is why they reach each other on 127.0.0.1) and usually the IPC namespace, but each has its own mount and PID view. The near-empty pause container you will meet exists precisely to hold those shared namespaces open so individual containers can come and go without tearing them down. Everything about "what a pod shares" is a statement about the namespaces on this page.

    You will see the pause container and the pod's shared namespaces in W1.21, and pod-to-pod networking builds on the shared net namespace in W4.2.

    Docs

    man 7 namespaces is the master page — read its opening paragraphs now and the per-type detail as you meet each one. man 1 unshare for the command you just used, man 2 unshare and man 2 clone for the system calls underneath, and man 8 lsns to list namespaces on the system. In the exam docs, nothing indexes under "namespace" for this sense — the Linux primitive is assumed, not documented, on kubernetes.io.

    B6.2

    The eight types, and the one question each answers

    Why it matters

    There is not one kind of namespace; there are eight, one per resource the kernel was willing to make private. You do not need to memorise them as trivia, but you do need the mental table — because "what does a container isolate?" is answered type by type, and because when something leaks between containers that should not, the fix is always "which namespace was shared that shouldn't have been?". Learning them as "each one changes the answer to exactly one question" turns eight facts into one pattern.

    The concept

    Every namespace type isolates one specific kernel-maintained answer. Here is the full set, each framed as the question it privatises:

    Typeunshare flagThe question it changes the answer to
    mnt--mountWhat is mounted, and where? (a private copy of the mount table from B5)
    pid--pidWhich processes exist, and what are their PIDs? (your shell can become PID 1)
    net--netWhich interfaces, addresses, routes, and ports are there? (own lo, own port space)
    ipc--ipcWhich System V shared-memory segments and message queues exist?
    uts--utsWhat is my hostname (and NIS domain)? (UTS = Unix Time-sharing System)
    user--userWhich uid/gid am I, and what may that identity do? (uid mapping, rootless)
    cgroup--cgroupWhat does my cgroup tree look like — where do I sit in it? (B7's subject)
    time--timeWhat do the monotonic and boot-time clocks say? (rarely used by hand)

    Read that column top to bottom and it is a decent one-line summary of "what is a machine, from a process's point of view": its files, its neighbours, its network, its shared memory, its name, its identity, its resource accounting, its clock. A container is a process for which several of these answers have been quietly replaced. Two notes worth carrying forward: the user namespace is special because it is the one that changes not what you see but who you are, which is why rootless containers are built on it and why it has extra guardrails (B6.8); and the time namespace is the newest and least used — you will almost never create one by hand, but it exists so a checkpoint/restore can restore a process's sense of uptime.

    One more distinction that saves confusion later: creating a namespace with unshare puts the calling process (and its future children) into a fresh one; the flags simply select which of the eight to fork. You can combine them freely — unshare --uts --pid --mount forks three at once — and that combining is exactly what a container runtime does in a single clone() call.

    Without looking, name the question answered by net, by mnt, and by user. (net: which interfaces/addresses/routes/ports exist; mnt: what is mounted where; user: which uid/gid I am and what it may do.)     Gotchas

    The flag names and the CLONE_* constants do not always match the type name, and the mount one is the trap: the mount namespace's constant is CLONE_NEWNS (not CLONE_NEWMNT), because it was the first namespace Linux ever had — back in 2002 there was only one kind, so it was just "NEW NamespaceS". Every other type got a descriptive suffix later. When you read kernel or runtime source and see CLONE_NEWNS, that is the mount namespace.

    Not every type is equally load-bearing for containers. In practice mnt, pid, net, ipc, uts, and user do the heavy lifting; cgroup mostly tidies what a process sees of B7's hierarchy; and time is a specialist tool for checkpoint/restore. Do not over-index on time and cgroup when reasoning about "what isolates a container" — but know they exist so the count of eight is not a mystery.

    Kubernetes lets you selectively un-isolate some of these, and now you can read those knobs precisely. A pod's hostNetwork: true means "do not give me a net namespace, share the node's" — same interfaces, same ports, which is why a hostNetwork pod can clash with the node on a port. hostPID: true shares the pid namespace so the pod sees every process on the node; hostIPC: true shares System V IPC. Each of those fields toggles exactly one row of the table above. When a security review says "this pod runs with hostPID", the risk is the pid row: it can see and signal node processes.

    These host* pod fields appear in W1.21 (pod internals) and as a security concern in W7.

    man 7 namespaces "Namespace types" for the authoritative list and per-type cross-references (man 7 pid_namespaces, man 7 user_namespaces, man 7 mount_namespaces, man 7 network_namespaces). man 8 lsns for enumerating live namespaces.     B6.3

    Namespace identity: reading membership by inode under /proc

    Why it matters

    Namespaces feel abstract until you find the handle the kernel gives you for each one — and it does give you a handle, in a place you already know how to read. Every namespace shows up as a file under /proc/PID/ns/, and each file's inode number is that namespace's identity. This turns "are these two processes in the same namespace?" from a vague question into an exact one you answer by comparing two numbers. It is also the mechanism that makes nsenter (B6.12) possible, so it is worth seeing directly.

    The concept

    For every process, the kernel exposes a directory /proc/PID/ns/ containing one magic symlink per namespace type: uts, pid, net, mnt, and so on. These are not ordinary symlinks to a path — they point at a special nsfs node, and the text you read from each is type:[inode]. That inode number is the namespace's unique id. The rule is exact and worth stating flatly:

    Two processes are in the same namespace of some type ⇔ that type's inode number matches.

    So membership is not a name or a label — it is identity by inode, the same "an inode is the file, the name is just a pointer" idea you learned for the filesystem in B5.6, reused for namespaces. Holding one of these /proc/PID/ns/* files open also keeps the namespace alive even after every process in it has exited — a fact that is useful (it is how you pin a network namespace) and dangerous (it is how you leak one that will not go away, the second half of this module's debugging drill, B6.15).

    /proc/PID/ns/net→ readlink →net:[4026531992]· same inode ⇒same namespace

    1. Look at your shell's eight namespace links. Each names its type and an inode; these are the host's defaults, shared with PID 1. 
      ls -l /proc/$$/ns

lrwxrwxrwx 1 ubuntu ubuntu 0 Jul 24 10:00 ipc -> 'ipc:[4026531839]'

lrwxrwxrwx 1 ubuntu ubuntu 0 Jul 24 10:00 net -> 'net:[4026531992]'
lrwxrwxrwx 1 ubuntu ubuntu 0 Jul 24 10:00 pid -> 'pid:[4026531836]'
...
    2. Prove membership by matching inodes: compare your shell with PID 1. Every number is identical, because your shell inherited init's namespaces — you are all in the same eight. 
      readlink /proc/$$/ns/pid /proc/1/ns/pid
      pid:[4026531836]
pid:[4026531836]               ← identical ⇒ same PID namespace as init
    3. Now make one differ. Fork a new UTS namespace and compare only that link — the uts inode changes while net (which you did not unshare) still matches the host. 
      sudo unshare --uts bash
readlink /proc/self/ns/uts        # inside: a new number
readlink /proc/self/ns/net        # inside: still the host's
exit
      uts:[4026532210]                  ← different from the host's ...838
net:[4026531992]                  ← unchanged: net was never unshared
    Verify

    Given two shells, how do you decide — with certainty, not guessing — whether they share a network namespace? (Compare readlink /proc/PID/ns/net for each; identical inode ⇒ same namespace.) Then explain why, in the last step, the net inode still matched the host after unshare --uts. (You only forked the UTS namespace; every other type, including net, remained the inherited host one.)

    Gotchas

    These inode numbers are not stable or meaningful across reboots, and they are not sequential ids you should parse for meaning — they are allocations from the nsfs filesystem, reused over time. Compare them for equality only; never assume "4026531840 is always the root mount namespace" or that a smaller number is older. The identity is real within one running kernel; the specific value is not a constant.

    Reading /proc/PID/ns/* for a process you do not own can be restricted, and the more subtle trap is lifetime: ls -l is harmless, but a bind mount of one of these links (mount --bind /proc/PID/ns/net /somewhere) pins the namespace open forever, even after the process dies. That is the intended way to keep a network namespace around (it is exactly what ip netns does under the hood), but do it by accident and you have a namespace with no processes that lsns still lists and nothing will reap — the leak in B6.15.

    This inode-identity mechanism is how every node-level debugging tool finds "the pod's namespace". When you run crictl inspect on a container, the network namespace path it prints is one of these nsfs handles; when a CNI plugin is asked to wire up a pod, it is handed the pod's network-namespace path and it operates inside that inode. kubectl exec ultimately resolves to entering the container's namespaces by these handles (B6.12). Being able to readlink /proc/PID/ns/net and compare is the raw version of everything those tools automate.

    This underpins the container introspection you will do in W1.21 and the CNI wiring in W4.2.

    Docs

    man 7 namespaces under "The /proc/[pid]/ns/ directory" for the symlink semantics and lifetime rules, man 5 proc for /proc/[pid]/ns generally, and man 2 setns for how a program opens one of these files to join the namespace (the syscall behind nsenter). man 8 lsns lists namespaces by their inode ids. No exam-docs equivalent — this is pure Linux mechanism.
