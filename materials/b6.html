    B6 — Isolation I: namespaces, chroot, pivot_root

    Everything you have run so far has shared one view of the machine: the same process list, the same hostname, the same filesystem, the same network. This module is where that stops being a law of nature. A Linux namespace lets one process — and its children — get a private answer to one specific question about the machine ("which processes exist?", "what is my hostname?", "which network interfaces are there?") while every other process keeps the old answer. Stack a handful of these together, wrap them around a private root filesystem, and you have built the thing the industry calls a container. Not a metaphor for one — an actual one, about twenty-five lines of shell, with no runtime installed. By the end of this module you will have done exactly that.

    The word "container" hides how humble the pieces are, so we take them one at a time. First the idea: a namespace is your own copy of one kernel resource, and you can see its identity as an inode number under /proc. Then one namespace at a time with the unshare command — a new hostname, a new process list where your shell is PID 1, a private mount table, an empty IPC world, a remapped user id, a re-rooted cgroup view — feeling exactly what each one hides. Then roots and entering: chroot and why it is a convenience rather than a wall, pivot_root which is the real thing runtimes use, and nsenter — which is, quite literally, what kubectl exec does. A lone network namespace gets created here and wired up in B9. Then you assemble container v0.1.

    The idea

    B6.1

    What a namespace actually is: your own copy of one kernel resource

    Why it matters

    "Container" sounds like it must be a small virtual machine, and almost every beginner carries that wrong picture for far too long. Getting the real idea early — before any command — saves you from that mistake and makes everything downstream obvious. A container is not a machine at all. It is an ordinary process on your ordinary kernel, which has simply been handed private answers to a few specific questions. The mechanism that hands out those private answers is the namespace, and it is much smaller and less magical than "virtual machine" suggests.

    The concept

    A namespace is a private copy of exactly one kernel-managed resource. Normally the kernel keeps one global version of each such resource — one list of all processes, one hostname, one set of network interfaces, one mount table — and every process sees the same one. A namespace splits one of those globals into an isolated instance: the processes inside it see and change their copy, and processes outside are untouched. Crucially, it is still the same kernel, the same CPU scheduler, the same memory manager, the same clock ticking. Nothing is being emulated or virtualised. Only the bookkeeping tables the kernel consults to answer a specific question have been forked in two.

    Back to the hotel from B2, where one hotel is one Linux machine: the building has one of many registers — a guest list, a directory of room phones, a hostname on the front sign. A namespace is the moment the front desk gives one room its own copy of one of those registers. The room writes in its guest list, reads from its guest list, and genuinely cannot tell that the rest of the hotel exists — yet it is the same building, the same plumbing, the same staff. It is not a separate hotel (that would be a virtual machine, with its own engineering crew and boiler). It is one room that has been handed a private ledger.

    Three things follow, and they are the whole reason the rest of this module works:

    • Each type is independent. You can give a process a private process list but leave it sharing the host's network, or a private hostname but the host's filesystem. A container is just a particular bundle of namespaces chosen together; there is nothing atomic about "a container".
    • Children inherit, by default. When a process in a namespace forks, the child is in the same namespaces — which is how one unshare at the top gives a whole tree of processes the same private view.
    • It is cheap. There is no second kernel to boot, no memory to pre-allocate. Creating a namespace is a single system call; this is exactly why containers start in milliseconds and a VM starts in seconds.

    Gotchas

    "Namespace" is one of the most overloaded words in computing — Kubernetes Namespaces, C++ namespaces, XML namespaces, DNS namespaces. This module means exactly one thing: a Linux kernel namespace, the process-isolation primitive. A Kubernetes Namespace (a label that groups API objects) is completely unrelated and does not create a single Linux namespace. Keep the two in separate mental drawers; the collision trips up nearly everyone once.

    Isolation is not security by itself. A namespace changes what a process can see, not automatically what it can do — a process with the right capabilities can often create, enter, or escape namespaces. Real container isolation stacks namespaces together with capabilities (B4), cgroups (B7), and seccomp/AppArmor. Treating a bare namespace as a security wall is a classic and dangerous mistake; we hit its sharpest edge in the chroot lesson (B6.10).

    This is the single most clarifying fact for reading Kubernetes: a pod is not a small VM, it is a set of processes sharing a chosen bundle of Linux namespaces. The containers in one pod share a network namespace (same IP, same localhost, same ports — which is why they reach each other on 127.0.0.1) and usually the IPC namespace, but each has its own mount and PID view. The near-empty pause container you will meet exists precisely to hold those shared namespaces open so individual containers can come and go without tearing them down. Everything about "what a pod shares" is a statement about the namespaces on this page.

    You will see the pause container and the pod's shared namespaces in W1.21, and pod-to-pod networking builds on the shared net namespace in W4.2.

    B6.2

    The eight types, and the one question each answers

    Why it matters

    There is not one kind of namespace; there are eight, one per resource the kernel was willing to make private. You do not need to memorise them as trivia, but you do need the mental table — because "what does a container isolate?" is answered type by type, and because when something leaks between containers that should not, the fix is always "which namespace was shared that shouldn't have been?". Learning them as "each one changes the answer to exactly one question" turns eight facts into one pattern.

    The concept

    Every namespace type isolates one specific kernel-maintained answer. Here is the full set, each framed as the question it privatises:

    Typeunshare flagThe question it changes the answer to
    mnt--mountWhat is mounted, and where? (a private copy of the mount table from B5)
    pid--pidWhich processes exist, and what are their PIDs? (your shell can become PID 1)
    net--netWhich interfaces, addresses, routes, and ports are there? (own lo, own port space)
    ipc--ipcWhich System V shared-memory segments and message queues exist?
    uts--utsWhat is my hostname (and NIS domain)? (UTS = Unix Time-sharing System)
    user--userWhich uid/gid am I, and what may that identity do? (uid mapping, rootless)
    cgroup--cgroupWhat does my cgroup tree look like — where do I sit in it? (B7's subject)
    time--timeWhat do the monotonic and boot-time clocks say? (rarely used by hand)

    Read that column top to bottom and it is a decent one-line summary of "what is a machine, from a process's point of view": its files, its neighbours, its network, its shared memory, its name, its identity, its resource accounting, its clock. A container is a process for which several of these answers have been quietly replaced. Two notes worth carrying forward: the user namespace is special because it is the one that changes not what you see but who you are, which is why rootless containers are built on it and why it has extra guardrails (B6.8); and the time namespace is the newest and least used — you will almost never create one by hand, but it exists so a checkpoint/restore can restore a process's sense of uptime.

    One more distinction that saves confusion later: creating a namespace with unshare puts the calling process (and its future children) into a fresh one; the flags simply select which of the eight to fork. You can combine them freely — unshare --uts --pid --mount forks three at once — and that combining is exactly what a container runtime does in a single clone() call.

    Gotchas

    The flag names and the CLONE_* constants do not always match the type name, and the mount one is the trap: the mount namespace's constant is CLONE_NEWNS (not CLONE_NEWMNT), because it was the first namespace Linux ever had — back in 2002 there was only one kind, so it was just "NEW NamespaceS". Every other type got a descriptive suffix later. When you read kernel or runtime source and see CLONE_NEWNS, that is the mount namespace.

    Not every type is equally load-bearing for containers. In practice mnt, pid, net, ipc, uts, and user do the heavy lifting; cgroup mostly tidies what a process sees of B7's hierarchy; and time is a specialist tool for checkpoint/restore. Do not over-index on time and cgroup when reasoning about "what isolates a container" — but know they exist so the count of eight is not a mystery.

    Kubernetes lets you selectively un-isolate some of these, and now you can read those knobs precisely. A pod's hostNetwork: true means "do not give me a net namespace, share the node's" — same interfaces, same ports, which is why a hostNetwork pod can clash with the node on a port. hostPID: true shares the pid namespace so the pod sees every process on the node; hostIPC: true shares System V IPC. Each of those fields toggles exactly one row of the table above. When a security review says "this pod runs with hostPID", the risk is the pid row: it can see and signal node processes.

    These host* pod fields appear in W1.21 (pod internals) and as a security concern in W7.

    B6.3

    Namespace identity: reading membership by inode under /proc

    Why it matters

    Namespaces feel abstract until you find the handle the kernel gives you for each one — and it does give you a handle, in a place you already know how to read. Every namespace shows up as a file under /proc/PID/ns/, and each file's inode number is that namespace's identity. This turns "are these two processes in the same namespace?" from a vague question into an exact one you answer by comparing two numbers. It is also the mechanism that makes nsenter (B6.12) possible, so it is worth seeing directly.

    The concept

    For every process, the kernel exposes a directory /proc/PID/ns/ containing one magic symlink per namespace type: uts, pid, net, mnt, and so on. These are not ordinary symlinks to a path — they point at a special nsfs node, and the text you read from each is type:[inode]. That inode number is the namespace's unique id. The rule is exact and worth stating flatly:

    Two processes are in the same namespace of some type ⇔ that type's inode number matches.

    So membership is not a name or a label — it is identity by inode, the same "an inode is the file, the name is just a pointer" idea you learned for the filesystem in B5.6, reused for namespaces. Holding one of these /proc/PID/ns/* files open also keeps the namespace alive even after every process in it has exited — a fact that is useful (it is how you pin a network namespace) and dangerous (it is how you leak one that will not go away, the second half of this module's debugging drill, B6.15).

    Gotchas

    These inode numbers are not stable or meaningful across reboots, and they are not sequential ids you should parse for meaning — they are allocations from the nsfs filesystem, reused over time. Compare them for equality only; never assume "4026531840 is always the root mount namespace" or that a smaller number is older. The identity is real within one running kernel; the specific value is not a constant.

    Reading /proc/PID/ns/* for a process you do not own can be restricted, and the more subtle trap is lifetime: ls -l is harmless, but a bind mount of one of these links (mount --bind /proc/PID/ns/net /somewhere) pins the namespace open forever, even after the process dies. That is the intended way to keep a network namespace around (it is exactly what ip netns does under the hood), but do it by accident and you have a namespace with no processes that lsns still lists and nothing will reap — the leak in B6.15.

    This inode-identity mechanism is how every node-level debugging tool finds "the pod's namespace". When you run crictl inspect on a container, the network namespace path it prints is one of these nsfs handles; when a CNI plugin is asked to wire up a pod, it is handed the pod's network-namespace path and it operates inside that inode. kubectl exec ultimately resolves to entering the container's namespaces by these handles (B6.12). Being able to readlink /proc/PID/ns/net and compare is the raw version of everything those tools automate.

    This underpins the container introspection you will do in W1.21 and the CNI wiring in W4.2.

    Hands-on, one namespace at a time

    B6.4

    UTS: a process with its own hostname

    Why it matters

    We start with the gentlest namespace because it isolates the smallest, most harmless thing: the machine's name. Nothing depends on the hostname the way things depend on the process list or the network, so you can create a UTS namespace, change the name inside it, and see the effect without any risk of wedging your session. It is the perfect first proof that "a process can have a private answer to one question" is a real, mechanical thing and not a slogan.

    The concept

    The UTS namespace isolates exactly two strings: the hostname and the NIS domain name. (UTS stands for "Unix Time-sharing System", the name of the kernel struct that holds them — the name is historical baggage, not a hint about function.) A process in its own UTS namespace can call sethostname() and change what it and its children see, while every other process on the machine keeps the old name. That is the entire feature.

    What makes it a good teacher is that unshare --uts bash maps precisely onto one system call: the shell calls unshare(CLONE_NEWUTS), and from then on hostname newname is a sethostname() that lands only in the private copy. Recall from B3 that a command is a thin wrapper over a syscall — here you can see the whole stack in one line. Below, the same two-step (make a UTS namespace, set a hostname in it) is written once in C with the raw syscalls and once in Python, to make the wrapper explicit; the unshare command is doing nothing more than this.

    Gotchas

    The shell prompt may not update to the new hostname until you start a fresh prompt, because bash caches \h. Do not trust the prompt string; trust hostname and the /proc/self/ns/uts inode. This is a general lesson for the whole module — the prompt is decoration, the /proc evidence is truth.

    A UTS namespace changes the name a process reports, not any name a DNS server or another machine knows. Setting hostname container-01 inside a namespace does not make anything on the network reachable as container-01; name resolution is a separate system (B10). This is the same "the label is not the thing" separation you saw between filenames and inodes in B5.

    This is why a pod's hostname is its own. Each pod gets a UTS namespace, so a process inside sees hostname return the pod's name (by default the pod name), independent of the node it runs on and of every other pod. When you set hostname or subdomain on a pod spec, or read $HOSTNAME inside a container and get the pod name rather than the node name, this UTS namespace is the entire reason. A StatefulSet relies on it to give each replica a stable, predictable in-pod hostname.

    You will see per-pod hostnames when you meet pod identity in W3 (StatefulSets) and pod internals in W1.21.

    B6.5

    PID: your shell becomes PID 1

    Why it matters

    The PID namespace is the one that makes a container feel like a real, separate machine — because inside it, the process list is short and your shell is PID 1, the first process, the one every OS treats as special. This is where "a container thinks it is alone" stops being abstract: you run ps and you genuinely see three processes on what is otherwise a busy VM. It is also where the special responsibilities of PID 1 — which trip up real containers constantly — become something you can feel directly.

    The concept

    A PID namespace gives its members a private numbering of processes. The first process created in a new PID namespace is PID 1 inside it, and its children get 2, 3, 4… — a fresh sequence that has nothing to do with the host's PIDs. The same process has two PIDs at once: a small one inside its namespace and a large real one on the host (a PID namespace nests, so the outer view always still exists). Two consequences carry real weight:

    • PID 1 is special, and now it is you. The kernel gives PID 1 two jobs: it is the reaper that adopts orphaned processes so they do not become zombies, and it has special signal handling — the kernel will not deliver a default-action signal (like SIGTERM) to PID 1 unless the process installed a handler for it. A shell as PID 1 that ignores SIGTERM is why "my container won't stop" happens.
    • Killing PID 1 ends the namespace. When PID 1 exits, the kernel sends SIGKILL to every other process in that PID namespace and tears the namespace down. There is no "init respawns things" here — PID 1 is the namespace's life; end it and everything in it dies at once.

    One subtlety you will hit immediately: ps reads /proc, and /proc reflects whatever the mount namespace shows — not the PID namespace. So unshare --pid alone leaves ps still listing host processes, because it is reading the host's /proc. You need --fork --mount-proc so that a fresh /proc, reflecting the new PID namespace, is mounted. That coupling — a PID namespace only looks right once /proc is remounted — is the whole point of the next lesson (B6.6) and half of the debugging drill (B6.15).

      gone
    Verify

    Inside the namespace, echo $$ printed 1 but the host saw a PID like 4712 — explain how one process has two PIDs. (A PID namespace nests: the process is PID 1 in its own namespace and simultaneously an ordinary large PID in the parent namespace, which never stopped seeing it.) Then: why does killing PID 1 take the whole namespace with it? (The kernel SIGKILLs every remaining process in a PID namespace when its PID 1 exits and reaps the namespace — PID 1 is the namespace's anchor.)

    Gotchas

    Without --mount-proc, ps inside the new PID namespace still lists every host process, and beginners conclude the namespace "didn't work". It worked — ps is just reading the host's /proc, which is a mount-namespace artifact, not a PID-namespace one. This is the single most common namespace confusion, and it is exactly symptom 1 of the debugging drill (B6.15). The rule: a PID namespace is only legible once its own /proc is mounted.

    The "PID 1 ignores SIGTERM unless it handles it" rule bites real containers hard. If your container's PID 1 is a shell or an app that never installed a SIGTERM handler, docker stop / a pod deletion sends SIGTERM, nothing happens, and after the grace period the runtime resorts to SIGKILL — a slow, ungraceful stop. This is why production images use a tiny init (like tini) as PID 1 or ensure the main process forwards signals. You are feeling the mechanism now so the Kubernetes symptom later reads as obvious.

    Every container's main process is PID 1 in its own PID namespace — which is why your app sees itself as PID 1 and why Kubernetes' terminationGracePeriodSeconds exists: it is the window between the SIGTERM the kubelet sends to your PID 1 and the SIGKILL it follows up with. If your process does not handle SIGTERM, that whole grace period is wasted and the pod dies hard. shareProcessNamespace: true on a pod merges its containers into one PID namespace so they can see each other's processes — a direct application of this lesson. And the reaping duty of PID 1 is why a poorly-chosen entrypoint accumulates zombie processes in a container.

    Graceful shutdown and terminationGracePeriodSeconds appear in W3 (workload lifecycle); the pause container as the pod's stable PID 1 holder is in W1.21.

    B6.6

    mount: a private copy of the mount table

    Why it matters

    The mount namespace is what lets a container have its own filesystem — its own /, its own /proc, its own mounts — without disturbing the host's. It is also the namespace that finally explains the loose end from B6.5: why a new PID namespace needed --mount-proc. Everything about "the container sees a different root" traces back to this one namespace, and it ties directly to the propagation modes you learned in B5.

    The concept

    A mount namespace gives a process a private copy of the mount table — the kernel's list of what is mounted where (B5.3). After unshare --mount, mounts you make are visible only inside; mounts the host makes may or may not reach you depending on propagation (B5.12). This is the first namespace whose isolation is not automatic in both directions, and that is where propagation comes back:

    • By default a new mount namespace starts as a copy of the host's current mounts, but the copies are typically shared peers — so a mount the host makes can still propagate in, and vice versa, unless the runtime changes the mode.
    • Real runtimes immediately make their mounts rslave (recursive slave): host mount events flow into the container, but the container's own mounts do not leak out to the host. That asymmetry — hear the host, do not disturb the host — is exactly the propagation lesson from B5.12, now with a reason.

    And here is the resolution of B6.5's puzzle: --mount-proc is not a PID feature at all — it is a mount operation. When unshare --pid --fork --mount-proc runs, the --mount-proc implies --mount (a new mount namespace) and then mounts a fresh procfs over /proc inside it. Because /proc is a window onto the PID namespace, and the fresh mount is created inside the new PID namespace, ps finally shows the short list. No mount namespace, no place to hang a private /proc, no correct ps. The two namespaces are partners.

    Gotchas

    A new mount namespace is a snapshot-plus-propagation, not a blank slate. You start out seeing the host's current mounts (that is why /, /home, etc. are all there); the isolation is about future changes and their propagation, governed entirely by the private/shared/slave modes from B5.12. If you expected an empty mount table, re-read that lesson — the mental model "private mount namespace = private mount events, seeded from a copy" is the accurate one.

    Mounts made in a mount namespace vanish when the last process leaves it — there is nothing to clean up on the host, which is usually a feature. But combine a mount namespace with a leaked /proc/PID/ns/mnt bind mount and you get a namespace that persists with its mounts intact and no processes, the mount-flavoured version of the leak in B6.15. Unmount inside, or let the namespace die cleanly; do not pin its ns file unless you mean to.

    This is the namespace behind everything filesystem-shaped in a pod. Each container gets a mount namespace whose root is the overlay you built in B5 — its own /, its own /etc, its own mounted volumes — which is why one container's files are invisible to another in the same pod unless a shared volume is mounted into both. Kubernetes' mountPropagation field (None/HostToContainer/Bidirectional) is exactly the private/rslave/shared choice from B5.12 and this lesson, exposed on a volumeMount. A CSI driver mounting a volume on the node and expecting the pod to see it depends on HostToContainer propagation working — precisely the rslave direction described above.

    B6.7

    IPC: a private world of shared memory

    Why it matters

    The IPC namespace is small and easy to skip, but skipping it leaves a real hole in your mental model of isolation: without it, two containers could quietly share memory through a mechanism that has nothing to do with files or the network. Seeing it work — create a shared-memory segment on the host, then watch it be invisible inside a new IPC namespace — closes that hole and explains a class of "why can't these two containers see each other's shared memory?" questions.

    The concept

    The IPC namespace isolates IPC objects of the old System V kind and POSIX message queues: shared-memory segments, semaphore sets, and message queues. These are a way for unrelated processes to communicate by a shared key rather than through a file or a socket — process A creates a segment with key 0x1234, process B attaches to the same key, and now they share raw memory. Without an IPC namespace, that key space is global to the machine: any process can attach to any segment it has permission for. With one, each namespace has its own key space, so the same key in two namespaces refers to two different (or non-existent) segments. You list these objects with ipcs and create/remove them with ipcmk/ipcrm.

    It is worth being clear about what this namespace does not cover, because "IPC" is a broad word. Pipes, Unix domain sockets, and signals are not governed by the IPC namespace — pipes and sockets are filesystem/mount and network concerns, signals are a PID-namespace matter. The IPC namespace is specifically about the System V / POSIX shared-memory and queue APIs. It is narrow on purpose.

    Name one IPC mechanism the IPC namespace isolates and one it does not. (Isolates: System V shared memory / semaphores / message queues and POSIX message queues.     Gotchas

    The host's default /dev/shm (a tmpfs used for POSIX shared memory) is a mount, not an IPC-namespace object — so isolating /dev/shm is a mount-namespace job, while System V segments are the IPC-namespace job. Containers get both by default, which is why two containers do not accidentally share memory either way, but the two mechanisms are isolated by two different namespaces. Do not assume one flag covers all "shared memory".

    Leftover System V segments are a classic resource leak: unlike a file, a shm segment survives the death of the process that made it until someone runs ipcrm or the namespace is destroyed. On a shared host this accumulates; inside a container it is bounded because the whole IPC namespace is torn down when the container exits, which quietly reaps every segment. That automatic cleanup is a real, if invisible, benefit of the namespace.

    By default each pod has its own IPC namespace shared among its containers, so containers in one pod can use System V shared memory together while remaining isolated from other pods. hostIPC: true drops that isolation and puts the pod in the node's IPC namespace — occasionally needed for talking to a host daemon via shared memory, and a red flag in a security review for exactly that reason. Databases and some ML workloads that use large shared-memory buffers care whether their containers share an IPC namespace, and whether /dev/shm is big enough — both are decisions this lesson explains.

    B6.8

    user: becoming root without being root

    Why it matters

    Every namespace so far changed what a process can see. The user namespace is the one that changes who it is: inside it, your ordinary account can be uid 0 — root — while the host still sees you as your unprivileged self. This is the foundation of "rootless" containers, the mechanism that lets an unprivileged user create all the other namespaces, and the one place where Ubuntu 24.04 deliberately puts up a guardrail you must understand rather than blindly disable.

    The concept

    A user namespace maps user and group IDs between the inside and the outside. A uid map says, for example, "uid 0 inside = uid 1000 (your real account) outside". So a process in the namespace can see itself as root — id reports uid 0, it holds a full set of capabilities within the namespace — while every action it takes on host-owned resources is still checked against your real, unprivileged uid 1000. You are root over your own private world and a nobody everywhere else. This is genuinely powerful for two reasons:

    • It is the key that unlocks the others. Creating most namespaces normally needs CAP_SYS_ADMIN. But a process that first creates a user namespace becomes root inside it, and that in-namespace root is allowed to create the other namespace types nested underneath. This is how a completely unprivileged user runs containers with no sudo at all — "rootless" containers (Podman's headline feature) are built on exactly this.
    • The capabilities are scoped. Being uid 0 inside does not let you do uid-0 things to the host, because those capabilities only apply to resources owned by the namespace. You can, say, mount a filesystem inside your user+mount namespace, but you cannot suddenly read another user's files on the host.

    Now the guardrail. Historically, unprivileged user namespaces were a rich source of kernel privilege-escalation bugs: they hand an unprivileged user a path to code that previously required root, and any bug in that code becomes a root exploit. So Ubuntu 23.10+ (including 24.04) ships kernel.apparmor_restrict_unprivileged_userns=1, an AppArmor-backed restriction that blocks an unprivileged process from creating a user namespace unless it has an AppArmor profile permitting it (or you run it with privilege). This is why unshare --user as a plain user may fail with EPERM on your VM. Two honest ways forward, both of which you should understand before choosing:

    • Run it with privilege (sudo unshare --user ...): root is exempt from the restriction, so the lab works and you still see the mapping. This is what we do below.
    • Relax the sysctl on your throwaway VM only (echo 0 | sudo tee /proc/sys/kernel/apparmor_restrict_unprivileged_userns) to feel the genuinely-unprivileged, rootless path. Do this knowing you are re-opening the attack surface the default closes — acceptable on a disposable sandbox, never a habit on a real machine.
    Lab

    Gotchas

    Do not reflexively set apparmor_restrict_unprivileged_userns=0 and forget it. The default is a real, deliberate mitigation: unprivileged user namespaces have repeatedly been the entry point for kernel privilege-escalation exploits (fresh bypasses were still being published in 2025), so disabling the restriction genuinely widens the attack surface of the machine. On a disposable sandbox VM it is a reasonable, informed trade to see the rootless path; on anything you care about, leave it on and grant per-application AppArmor profiles instead. Understand the trade, then decide — that is the whole point of the checkpoint.

    "Root inside a user namespace" fools some tools into acting privileged when they are not, producing confusing errors. A program may see uid 0, attempt a host-level privileged operation, and fail with EPERM in a way that looks like a bug but is the mapping working correctly. When something "runs as root but can't do root things", check /proc/self/uid_map — you may be in a user namespace where that root is local.

    User namespaces are how Kubernetes is making containers that run as root inside the container map to an unprivileged uid on the node — the hostUsers: false pod field (user namespaces for pods, GA in recent releases) does exactly the mapping you just read in /proc/self/uid_map, so a container compromise as "root" is only an unprivileged user on the node. This is defence in depth on top of runAsNonRoot and the capability-dropping you learned in B4. Rootless Kubernetes distributions (running the kubelet and runtime without host root) are built on the same unprivileged-user-namespace mechanism this lesson gated.

    Pod user namespaces (hostUsers), runAsNonRoot, and dropped capabilities are security topics in W7; the capability model is B4.13.

    B6.9

    cgroup: re-rooting your view of the hierarchy

    Why it matters

    This is the last and subtlest of the eight, and it is a preview of the whole next module. A cgroup namespace does not limit any resource — B7 is about the limiting. What it does is hide where you sit in the machine's resource-control hierarchy, so that a containerised process sees its own cgroup as the root of the tree rather than as some deeply-nested branch of the host's. It is a cosmetic-looking namespace that turns out to matter for both tidiness and a little bit of security.

    The concept

    Recall (or preview) that the kernel arranges processes into a tree of cgroups — control groups — for accounting and limiting CPU, memory, and I/O (all of B7). On Ubuntu 24.04 this is the cgroup v2 unified hierarchy, one tree rooted at /sys/fs/cgroup, and a process's place in it shows in /proc/self/cgroup as a path like /system.slice/docker-abc123.scope. A cgroup namespace relativises that path: it makes the process's current cgroup appear to be /, so from inside, /proc/self/cgroup reads / and the process cannot see the host's system.slice, other containers' scopes, or where it really sits. The accounting and limits are unchanged — the process is still in the same real cgroup, still capped the same way — but its view of the hierarchy is re-rooted at its own node.

    Why bother? Two modest reasons. First, tidiness: software inside a container that reads /proc/self/cgroup or walks /sys/fs/cgroup (systemd, some runtimes, resource-aware apps) sees a clean tree rooted at itself instead of a confusing absolute path exposing the host's layout. Second, a little information hiding: the host's cgroup paths can leak details (other tenants' container ids, the orchestrator's structure), and re-rooting keeps those out of the container. It is the quietest namespace, but it completes the picture — a container's view of the resource hierarchy is private, just like its processes, mounts, and network.

    Inside the cgroup namespace /proc/self/cgroup read /, yet the process's real cgroup and its limits were unchanged — what exactly did the namespace change?     Gotchas

    Do not confuse the cgroup namespace (this lesson: hides your position in the tree) with cgroups themselves (B7: enforce CPU/memory/IO limits). Creating a cgroup namespace grants no resource control whatsoever — a process in a fresh cgroup namespace with no limits set can still use the whole machine. The namespace is about visibility; the limiting is a separate mechanism you configure in B7. Blurring the two leads to the false belief that "containerising" a process caps it, when capping requires actually writing limits into its cgroup.

    The re-rooting is relative to where the process was at unshare time. If you move the process into a different cgroup afterwards, the view can show a path above the new namespace root, which looks odd. In practice a runtime creates the cgroup namespace at the moment it places the container in its final cgroup, so the two line up; by hand, just know the root is pinned to your position when you ran unshare --cgroup.

    When you kubectl exec into a container and inspect /sys/fs/cgroup or /proc/self/cgroup, the clean tree you see — rooted at the container rather than at the node's kubepods.slice/… path — is this namespace. The kubelet places each pod's containers in a cgroup under kubepods (where the CPU/memory limits from B7 and your pod's resources requests/limits are enforced), and the cgroup namespace hides that placement so tools inside the container see themselves at the root. Understanding that the limits live in the real cgroup while the container only sees a re-rooted view is what stops "the container thinks it's at cgroup root, so it must be unlimited" — a dangerous misreading B7 will fully dismantle.

    Pod resource limits and their cgroup enforcement are the whole of W3's scheduling/limits material and the next module, B7.

    B6.10

    chroot: changing what / means — and why it is not a wall

    Why it matters

    chroot is the oldest and simplest way to give a process a different filesystem root, and it is the piece most people think a container is. Running it once — pointing a shell at the rootfs you built in B5 and watching / become that directory — makes "a container has its own filesystem" concrete. But the more important lesson is the opposite one: chroot is a convenience, not a security boundary, and knowing exactly why is what separates "I use containers" from "I understand them". This is where the "isolation ≠ security" warning from B6.1 gets its sharp edge.

    The concept

    chroot <dir> <cmd> changes the calling process's idea of the root directory to <dir>: from then on, an absolute path like /bin/sh is resolved relative to <dir>, so the process can only name files inside that subtree. Point it at a minimal rootfs (a directory containing bin/, lib/, etc. — exactly the debootstrap/busybox tree from B5.17) and you get a shell that believes <dir> is the whole filesystem. That is genuinely useful: rescue environments, build sandboxes, and the first stage of container startup all use it.

    Now the crucial caveat. chroot only changes path resolution starting from /; it does not sever the process from everything else on the machine. A process that keeps enough privilege — specifically CAP_SYS_CHROOT and the ability to hold an open file descriptor to a directory outside the new root — can climb straight back out. The classic escape is only a few lines: open a directory fd before or after chrooting, chroot into a subdirectory, then fchdir back to the saved fd and cd .. repeatedly until you are at the real /. The kernel does not stop this, because chroot was never designed as a containment boundary — it changes a lookup base, nothing more. It also does nothing about processes, network, users, or mounts: a chrooted process still shares the host's PID list, network, and everything else you spent this module learning to isolate. That is why a real container is chroot's job done properly by pivot_root (next lesson) inside a full stack of namespaces — not chroot alone.

    Gotchas

    Never rely on chroot to contain untrusted code. The escape is well-known, requires no kernel bug, and works whenever the process keeps CAP_SYS_CHROOT (which a root process inside the chroot has by default). If you need real containment you need namespaces + dropped capabilities + pivot_root, i.e. an actual container. Treating a chroot jail as a security sandbox for hostile code is a decades-old mistake that still causes breaches.

    A chroot is only as functional as the files you put in it: run a binary that needs a shared library not present under the new root and it fails with a confusing "not found" even though the binary exists, because the loader can no longer see /lib. This is why a rootfs must include the binary's libraries (or the binary must be static) — the same completeness requirement that makes container base images carry their own /lib, and exactly what B5.17 walked through.

    Container runtimes use the idea of chroot — give the container its own root filesystem — but implement it with pivot_root (next lesson) precisely because chroot is escapable. When you see a container's root filesystem in kubectl exec, that private / is the runtime's pivot_root into the image's overlay, not a bare chroot. The security lesson matters directly: this is why running a container as root, or granting it CAP_SYS_ADMIN/SYS_CHROOT, meaningfully weakens isolation, and why securityContext capability-dropping (B4) is defence-in-depth on top of the namespaces.

    B6.11

    pivot_root: the real way to swap a root filesystem

    Why it matters

    pivot_root is what container runtimes actually use where a beginner would reach for chroot, and the difference is exactly the escapability you just saw. Instead of merely changing where / points, pivot_root swaps the mount namespace's root and lets you unmount the old one entirely — so there is no old root left anywhere to climb back to. Doing it by hand, inside a mount namespace, is the last primitive you need before assembling a container in B6.14.

    The concept

    pivot_root <new_root> <put_old> does two things atomically: it makes <new_root> the root of the current mount namespace, and it moves the old root so it is now accessible at <put_old> (a directory inside the new root). Crucially, once the old root is relocated there, you can umount it — and after that unmount, the old filesystem is not reachable from anywhere in the namespace. That is the whole advantage over chroot: chroot leaves the old root mounted and merely redirects lookups, so an outside fd is a ladder back; pivot_root plus an unmount removes the ladder.

    Because it rewrites the mount namespace's root, pivot_root has real preconditions, and hitting them is most of the learning:

    • You must be in your own mount namespace (unshare --mount), or you would be swapping the host's root — the kernel refuses in many configurations, and you would not want to succeed.
    • <new_root> must be a mount point, not just a directory. The usual trick is to bind-mount the rootfs onto itself (mount --bind newroot newroot) so it qualifies.
    • The new root (and its parent mount) must not be shared — propagation must be private/slave, or the pivot would try to propagate a root change to the host. This is the B5.12 propagation rule showing up as a hard requirement.
    • <put_old> must be a directory underneath <new_root>.

    Meet those four and the pivot is clean; miss one and you get a terse EINVAL that only makes sense once you know the list. Every one of these preconditions is a mount-namespace/propagation fact from B5 and B6.6, which is why this lesson sits where it does.

    Lab
    1. Inside a fresh mount namespace, make the rootfs a private mount point (preconditions 1–3). The self-bind turns a directory into a mount; --make-private stops the pivot propagating to the host. 
      multipass shell sandbox
sudo unshare --mount bash
NEW=~/rootfs
mount --bind $NEW $NEW           # new_root must be a mount point
mount --make-private $NEW        # and not shared
mkdir -p $NEW/oldroot            # put_old, under new_root
      (no output — the setup succeeds silently)
    2. Pivot. After this, / is the rootfs and the old root hangs at /oldroot; cd / so nothing holds the old root. 
      cd $NEW
pivot_root . oldroot
cd /
ls /                             # the rootfs, as the true root
ls /oldroot | head -3            # the host's old root, still mounted here for now
      bin  dev  etc  lib  oldroot  usr
bin  boot  dev                   ← the old root, parked at /oldroot
    3. Remove the ladder: unmount the old root. Now there is no path to the host filesystem anywhere in this namespace — the thing chroot could never guarantee. 
      mount --make-private /oldroot 2>/dev/null
umount -l /oldroot
ls /oldroot                      # empty — the old root is gone from this namespace
exit
      (empty)                          ← host root no longer reachable from inside
    Verify

    State the one capability pivot_root gives you that chroot cannot. (After relocating and unmounting the old root, the previous filesystem is unreachable from anywhere in the mount namespace — there is no outside directory left to escape to, whereas chroot leaves the old root mounted and reachable.) Then: why must you run it inside unshare --mount with the new root made private? (It rewrites the mount namespace's root; without a private mount namespace you would be altering the host's root and the change could propagate out — the kernel refuses shared-propagation pivots, which is precisely the B5.12 rule.)

    Gotchas

    pivot_root failing with EINVAL almost always means a precondition, not a typo — most often that new_root is a plain directory rather than a mount point (fix: bind-mount it onto itself) or that it is a shared mount (fix: --make-private). Reading the error as "one of the four preconditions is unmet" and checking them in order is far faster than guessing. This is the payoff for learning propagation in B5: the fix is always a mount-mode adjustment.

    Leaving the old root mounted at /oldroot re-introduces exactly the escape route pivot_root exists to remove — a process could cd /oldroot straight back to the host tree. The unmount in step 3 is not optional cleanup; it is the security-relevant final step. Real runtimes always unmount (or never expose) the old root before handing the container to the workload.

    This is the exact sequence a container runtime (containerd/runc via the OCI runtime spec) performs to give a pod's container its root filesystem: create a mount namespace, mount the image's overlay (B5.16) as the new root, pivot_root into it, and unmount the old root so the container cannot reach the node's filesystem. When people say "the container is isolated from the host filesystem", this unmounted-old-root is the mechanism that makes the claim true — far more than chroot ever could. A hostPath volume is, in effect, deliberately mounting a piece of the node's filesystem back into that otherwise-severed root, which is why hostPath is powerful and security-sensitive.

    The overlay-as-root and runtime handoff continue in W1; hostPath's node-filesystem access is a security topic in W5 and W7.

    Docs

    man 2 pivot_root and man 8 pivot_root for the syscall/command and its exact preconditions (read the ERRORS section — every EINVAL cause is listed), and man 7 mount_namespaces for why a private mount namespace is required. Re-read B5.12 (propagation) and B5.16 (overlay root). In the exam docs, container root filesystem and hostPath volumes.

    B6.12

    nsenter: joining a namespace someone else made

    Why it matters

    So far you have created namespaces. The other half of the story is joining one that already exists — stepping into a running container's world to look around. That operation, nsenter, is not an obscure tool: it is, almost exactly, what kubectl exec and every "get me a shell in that container" command do under the hood. Doing it by hand — entering the container you will build in B6.14 — demystifies the single most-used debugging command in Kubernetes.

    The concept

    nsenter enters one or more of another process's namespaces by targeting that process's PID and naming which namespaces to join. It is the command-line face of the setns(2) syscall, which opens one of the /proc/PID/ns/* handles from B6.3 and switches the caller into that namespace. The usual invocation is nsenter -t <pid> -m -u -i -n -p <cmd>, where each flag selects a namespace type to join — -m mount, -u UTS, -i IPC, -n network, -p PID (there are flags for user and cgroup too). You give it the PID of any process already inside the target namespaces, and it runs your command inside them. Because you built those namespaces around a container's init process, entering that PID's namespaces drops you into the container.

    Two details make this click. First, you can join selectively: enter only the network namespace (-t PID -n) to debug a container's networking with host tools, while keeping your own mount and PID view — enormously useful when the container itself lacks ping or tcpdump. Second, joining a PID namespace has a wrinkle: setns into a PID namespace only affects future children, so nsenter -p forks your command as PID-something inside, rather than renumbering the running nsenter itself. This is why nsenter exists as a dedicated tool rather than a one-line wrapper — it handles that forking for you.

    Lab
    1. Start a long-lived namespaced process to enter, standing in for a container's init. Leave it running in one terminal. 
      multipass shell sandbox
sudo unshare --uts --pid --net --fork bash -c 'hostname box-01; sleep 3600'
      (this shell now holds a UTS+PID+net namespace open via the sleep)
    2. From a second terminal, find that process's PID and join its namespaces. Inside, hostname is the one it set and the process list is the container's — you are in its world. 
      multipass shell sandbox
PID=$(pgrep -f 'sleep 3600')
sudo nsenter -t $PID -u -p -n bash
hostname                         # box-01, not sandbox
ps aux | wc -l                   # a short list — the container's PID namespace
      box-01                          ← joined the container's UTS namespace
3                                ← its PID namespace: just a couple of processes
    3. Show the selective-entry superpower: join only the network namespace to inspect its interfaces with host tools, without leaving your own PID/mount view. 
      exit
sudo nsenter -t $PID -n ip addr show | grep -E 'inet|state' | head
          inet 127.0.0.1/8 scope host lo   ← the container's lone loopback (see B6.13)
    Verify

    What does nsenter -t PID -n (network only) let you do that entering all namespaces would not? (Debug the target's network using tools from your own environment — your shell keeps its own mount/PID view, so host tools like tcpdump or ip are available even if the container lacks them.) Then connect it to Kubernetes: what everyday command is nsenter essentially the internals of? (kubectl exec — it runs your command inside the target container's namespaces, exactly as nsenter does by PID.)

    Gotchas

    Entering another process's namespaces needs privilege (typically root, or the right capabilities in the target user namespace) — this is not an unprivileged operation, and rightly so: joining a namespace is joining someone's isolated world. If nsenter fails with EPERM, you are missing that privilege, not naming the wrong PID. On a real cluster this is why node-level debugging (entering a pod's namespaces from the host) requires node root access, which is itself a privilege boundary worth respecting.

    The PID-namespace forking quirk surprises people: after nsenter -t PID -p, tools inside see the container's PIDs, but if you also expected your own shell's $$ to become a small number it will not — nsenter forks your command into the namespace rather than moving the existing process. This is the same "setns into a PID namespace affects only children" rule from B6.5, and it is correct behaviour, not a bug. Match your mental model to it rather than fighting it.

    kubectl exec -it pod -- sh is, at the node level, the kubelet asking the runtime to do a setns/nsenter-equivalent into the container's namespaces and run your command there — which is why the shell you get sees the container's filesystem, hostname, and process list. The kubectl debug ephemeral-container feature and node-level troubleshooting (entering a pod's netns from the host to run tcpdump when the pod image is too minimal to contain it) are the selective-nsenter trick you just used, formalised. Understanding nsenter is understanding what "exec into the pod" physically means.

    kubectl exec/kubectl debug and container troubleshooting are covered in W6; the container introspection view is W1.21.

    Docs

    man 1 nsenter for the flags (-t, and -m -u -i -n -p -U -C for each type), man 2 setns for the syscall it wraps, and man 7 namespaces for the /proc/PID/ns/ handles it opens. Compare with man 1 unshare (create) — nsenter is its join-side counterpart. In the exam docs, kubectl exec and kubectl debug.

    Network namespace preview

    B6.13

    A network namespace: an empty machine with one dead loopback

    Why it matters

    The network namespace is the biggest of the eight and gets its own module (B9) to wire up — but you should meet it here, empty, so the container you build in B6.14 has a complete set of the isolations a real one has. Seeing a freshly-created network namespace — a lone loopback interface, switched off, no routes, no way to reach anything — makes vivid what "a container starts with no network" actually means, and sets up the central question B9 answers: how does anything ever get out of here?

    The concept

    A network namespace gives a process a completely private network stack: its own set of network interfaces, its own IP addresses, its own routing table, its own firewall rules, and — importantly — its own port space. A brand-new network namespace is almost entirely empty. It contains exactly one interface, the loopback (lo), and even that starts DOWN; there are no other interfaces, no routes, and therefore no path to anything outside itself. It is, at birth, a machine unplugged from the network — it cannot reach the host, the host cannot reach it, and it cannot reach the internet.

    Two facts to carry into B9. First, the private port space is why isolation is so useful for servers: two network namespaces can each bind port 80, with no conflict, because "port 80" means something different in each — this is exactly how ten pods can all run a web server on port 80 on one node. Second, connecting a network namespace to anything requires deliberately building a wire: the standard tool is a veth pair (a virtual cable with two ends — put one end in the namespace, leave the other on the host or in a bridge), which is the hotel-room-phone-wire-to-the-switchboard analogy and the entire subject of B9. For now the point is the starting condition: isolated and inert. The ip netns command manages named network namespaces, and it works by creating the namespace and bind-mounting its ns/net handle under /var/run/netns/ so it persists and can be named — the very pinning mechanism from B6.3.

    ip netns add demo→ contains onlylo (DOWN)· no routes · no way out · untila veth pair (B9)
    Lab
    1. Create a named network namespace and look inside. ip netns exec runs a command in it; all you find is a loopback, and it is down. 
      multipass shell sandbox
sudo ip netns add demo
sudo ip netns exec demo ip addr
      1: lo: <LOOPBACK> mtu 65536 state DOWN ...
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
                                     ← one interface, switched off, no address
    2. Confirm there is no way out — no routes at all, so nothing is reachable. 
      sudo ip netns exec demo ip route
sudo ip netns exec demo ping -c1 -W1 8.8.8.8 2>&1 | tail -1
                                       ← empty: no routing table entries
ping: connect: Network is unreachable   ← isolated and inert, exactly as expected
    3. See the private port space: bring loopback up inside, and note the namespace has its own 127.0.0.1 independent of the host's. (Wiring it to the outside world is B9.) 
      sudo ip netns exec demo ip link set lo up
sudo ip netns exec demo ip addr show lo | grep inet
sudo ip netns list
          inet 127.0.0.1/8 scope host lo   ← its own loopback, now up
demo                                 ← the named namespace persists (pinned under /var/run/netns)
    Verify

    A brand-new network namespace can't reach the internet even though the host can — name the two things it is missing. (Any interface other than a down lo, and any route — no wire out and no routing table, so there is literally no path.) Then: why can two pods both serve on port 80 on one node without colliding? (Each pod has its own network namespace with its own port space, so "port 80" is a separate binding in each namespace — they never contend for one shared port table.)

    Gotchas

    ip netns-created namespaces persist after the creating shell exits, because ip netns bind-mounts the namespace's handle under /var/run/netns/ to keep it alive and nameable — so you must sudo ip netns delete demo to clean up, or it lingers. That persistence is a feature (you can enter it later), but it is also the exact leak mechanism in B6.15's second symptom: a bind-mounted ns/net handle keeps a namespace alive with no processes in it. Delete named namespaces you are done with.

    Do not confuse "has a loopback" with "has networking". A fresh netns has lo but it starts DOWN and, even up, loopback only talks to itself — a process can reach 127.0.0.1 inside but nothing beyond. Real connectivity needs a veth pair and routes you have not built yet. Beginners see lo and assume the namespace is connected; it is the opposite — lo is the proof of how little is there.

    This empty network namespace is precisely the starting state of every pod before the CNI plugin runs. When the kubelet creates a pod, the pause container gets a fresh network namespace exactly like demo — one down loopback, no routes — and then the CNI plugin (Flannel, Calico, Cilium…) does the B9 work: creates a veth pair, puts one end in the pod's namespace, assigns the pod IP, and adds routes so the pod can reach the cluster. The private port space is why every pod can use the same container port without node-level conflict, and why a Service has to map to pod IPs rather than host ports. Everything in W4 begins from this inert namespace.

    Pod networking, CNI, and the veth-pair wiring are the whole of W4.2; you build the veth pair by hand in B9.

    Project & drill

    B6.14

    Mini project — container v0.1

    Why it matters

    This is the payoff for the whole module. You now hold every primitive a container is made of — a mount namespace and pivot_root for the filesystem, a PID namespace with a fresh /proc for the process list, UTS for the hostname, IPC for shared memory — and assembling them takes about twenty-five lines of shell and installs no runtime whatsoever. The result is a shell that genuinely believes it is alone on a machine: its own root filesystem, its own short process list where it is PID 1, its own hostname. When you finish this, "container" will never again feel like magic — it will feel like a script you wrote. B7 will add resource limits, B9 a network, and B14 will grow this into minibox.

    The concept

    Container v0.1 is the composition of everything above, in one command plus a little setup. The recipe:

    1. A rootfs — the minimal directory tree from B5.17 (busybox or debootstrap), the filesystem the container will call /.
    2. A bundle of namespaces — unshare --pid --mount --uts --ipc --fork: a private process list, mount table, hostname, and IPC world, with --fork so the shell becomes PID 1 of the new PID namespace.
    3. A pivoted root — bind-mount the rootfs onto itself, make it private, pivot_root into it, and unmount the old root (B6.11) so the host filesystem is unreachable.
    4. A mounted /proc — mount a fresh procfs inside so ps reflects the new PID namespace (B6.6). This is the --mount-proc job, done explicitly because we pivot first.

    That is the entire anatomy of a container's isolation. What we are deliberately not adding yet keeps the scope honest: no cgroup limits (B7 — so this container can still use the whole machine), no network namespace wiring (B9 — it shares the host network for now, or is isolated and inert if you add --net), no user namespace by default (so it runs as real root — fine on a throwaway VM, not how you would ship it), and no image format, registry, or runtime daemon. A production runtime adds those layers, but none of them change the core you are about to build. The container is the namespaces plus the pivoted root; everything else is management around it.

    Lab
    1. Write the script. It takes a rootfs path, enters the namespace bundle, pivots into the rootfs, mounts /proc, and runs a shell. Read each line against the lesson it came from. 
      cat > ~/container.sh <<'EOF'
#!/bin/sh
# container v0.1 — namespaces + pivot_root, no runtime. Usage: sudo ./container.sh ~/rootfs
set -eu
ROOTFS="$1"

# Re-exec ourselves inside a fresh bundle of namespaces (B6.2, B6.5).
# --fork makes the child PID 1 of the new PID namespace; --mount gives a
# private mount table so the pivot below can't disturb the host (B6.6).
if [ -z "${_IN_NS:-}" ]; then
    export _IN_NS=1
    exec unshare --pid --mount --uts --ipc --fork "$0" "$ROOTFS"
fi

hostname container-v0.1                       # UTS namespace (B6.4)

# Make the rootfs a private mount point so pivot_root's preconditions hold (B6.11).
mount --bind "$ROOTFS" "$ROOTFS"
mount --make-private "$ROOTFS"
mkdir -p "$ROOTFS/oldroot"

cd "$ROOTFS"
pivot_root . oldroot                          # swap the root (B6.11)
cd /

mount -t proc proc /proc                      # fresh /proc for the new PID ns (B6.6)
umount -l /oldroot                            # remove the ladder back to the host
rmdir /oldroot 2>/dev/null || true

echo "== container v0.1: I am PID $$ on host '$(hostname)' =="
exec /bin/sh
EOF
chmod +x ~/container.sh
      (script written — about 25 meaningful lines)
    2. Run it against the B5.17 rootfs. The banner shows you are PID 1 with the container's hostname. 
      sudo ~/container.sh ~/rootfs
      == container v0.1: I am PID 1 on host 'container-v0.1' ==
    3. Prove it is really isolated from inside: a short process list, its own root, and no view of the host filesystem. 
      ps -ef                        # a handful of processes; you are PID 1
ls /                          # the rootfs, not the host's /
ls /oldroot 2>/dev/null || echo 'host root unreachable'
exit
      PID   USER  ... COMMAND
    1 root  ... /bin/sh       ← PID 1, short list — its own PID namespace
    9 root  ... ps -ef
bin  dev  etc  lib  proc  usr  ← its own root filesystem
host root unreachable       ← old root was pivoted away and unmounted
    Verify

    From inside container v0.1, run echo $$ (expect 1), hostname (expect container-v0.1), and ls /oldroot (expect empty/unreachable). If all three hold, you have built a container: private PID namespace with you as PID 1, private UTS hostname, and a pivoted root with no path back to the host. Then, to prove the deliberate gaps, note that ip addr still shows the host's interfaces (no --net — B9's job) and nothing caps your CPU or memory (no cgroup — B7's job). Naming what is missing is as important as what works.

    Gotchas

    The self-re-exec (_IN_NS guard) trips people up: unshare --fork needs to run a program as the new PID 1, so the script relaunches itself inside the namespaces rather than trying to become PID 1 in place. Without the guard and the exec unshare … "$0", either nothing enters a PID namespace or you get an infinite fork loop. If you adapt this script, keep that two-phase structure — "set up namespaces, then re-enter myself" is the standard shell idiom for it.

    This container runs as real root with the host network and no limits — it is a teaching artifact, not a secure sandbox. Anything inside can exhaust the machine's CPU/memory (no cgroup) and, running as uid 0 without a user namespace, is far closer to host root than a real container would allow. That is fine on a disposable sandbox VM whose whole purpose is to be wrecked, but do not mistake v0.1 for something you would run untrusted code in. B7, B9, and B14 close these gaps one at a time.

    You have just built, by hand, the core of what containerd/runc do for every pod: a bundle of namespaces wrapped around a pivoted root filesystem. The pieces map one-to-one — the OCI runtime config lists exactly these namespaces, the runtime does the same pivot_root-and-unmount, and the pause container exists to hold the namespaces open (your --fork PID 1, generalised). Everything Kubernetes adds — the image that becomes your rootfs (an overlay from B5), the cgroup limits from a pod's resources (B7), the CNI-wired network (B9), the security hardening (user namespaces, dropped capabilities, seccomp) — is a layer around this core, not a replacement for it. When you read a pod spec now, you are reading configuration for the container you just wrote.

    The runtime/OCI handoff and the pause container are in W1.21; the layers you'll add map to W3 (limits), W4 (network), and W7 (security).

    man 1 unshare (the whole flag bundle), man 8 pivot_root, man 8 mount (bind/make-private/proc), and man 7 namespaces tie together everything this script uses.     B6.15

    Debugging drill — the invisible namespace and the leak that won't die

    Why it matters

    Two failures catch nearly everyone who works with namespaces, and both are cases where the loud symptom points at the wrong object — the same evidence-first discipline as the B2, B3, B4, and B5 drills. The first: a PID namespace that looks like it "didn't work" because ps still shows every host process. The second: a namespace that refuses to go away long after its last process exited, quietly holding resources. Reproducing both, and fixing them by reading state rather than guessing, is what turns the theory above into a debugging reflex.

    Symptom 1 — ps in a new PID namespace still lists every host process
    1. Reproduce it: create a PID namespace without --mount-proc, so /proc is still the host's. ps shows hundreds of processes and your shell is not PID 1 — it looks like the namespace was ignored. 
      multipass shell sandbox
sudo unshare --pid --fork bash        # note: no --mount-proc
echo "my PID says: $$"
ps aux | wc -l
      my PID says: 1                       ← you ARE PID 1...
247                                  ← ...yet ps lists 247 processes?!
    2. Diagnose from evidence, not vibes: $$ says 1, so the PID namespace is real — the lie must be in what ps reads. ps reads /proc, and /proc is a mount, still pointing at the host's. 
      readlink /proc/self/ns/pid            # a new inode — the PID ns is real
head -1 /proc/self/mountinfo | grep -o 'proc' | head -1
findmnt /proc                          # whose /proc is this?
      pid:[4026532xxx]                      ← genuinely a new PID namespace
TARGET SOURCE FSTYPE OPTIONS
/proc  proc   proc   ...              ← the HOST's procfs — never remounted
    Cause & fix

    The PID namespace worked; /proc was never remounted. ps derives the process list from /proc, which is an artifact of the mount namespace, not the PID namespace — and you did not create a mount namespace or mount a fresh procfs, so ps is reading the host's /proc and reporting host processes. The fix is to give the namespace its own /proc: add --mount-proc (which implies --mount) — sudo unshare --pid --fork --mount-proc bash — and ps immediately shows the short list. The reflex: when a PID namespace "shows host processes", the fault is /proc, not the namespace; check findmnt /proc before doubting unshare. This is the loose end from B6.5 and B6.6, made into a symptom.

    Symptom 2 — a network namespace that will not go away
    1. Reproduce it: create a network namespace, then bind-mount its ns/net handle somewhere — pinning it open (B6.3) — and let every process using it exit. The namespace has no processes yet still exists. 
      sudo ip netns add ghost
# simulate a leaked reference: pin the handle with a bind mount, then "lose" it
sudo touch /root/pin
sudo mount --bind /var/run/netns/ghost /root/pin
sudo ip netns delete ghost            # try to delete it
sudo lsns -t net | grep -c ghost || true
ls /var/run/netns/
                                            ← ip netns list no longer shows it, but...
(empty)                               ← the name is gone, yet the namespace lives on
    2. Diagnose: lsns still lists a net namespace with 0 processes — the tell-tale of a namespace kept alive by a reference, not a process. Find what is holding it: a mount. 
      sudo lsns -t net -o NS,NPROCS,PID,COMMAND | awk '$2==0'
findmnt /root/pin
      4026532xxx 0  -  -                    ← a net namespace with zero processes: leaked
TARGET     SOURCE                 FSTYPE OPTIONS
/root/pin  nsfs[net:[4026532xxx]] nsfs   rw   ← a bind mount is pinning it open
    Cause & fix

    A leaked reference, not a stuck kernel. A namespace lives as long as anything references it — a process, an open fd, or a bind mount of its /proc/PID/ns/* handle (B6.3). Here a bind mount of the nsfs handle keeps the network namespace alive even though ip netns delete removed the name and no process remains, so lsns shows it with NPROCS=0. The fix is to remove the holding reference: sudo umount /root/pin (and rm /root/pin), after which the namespace is finally reaped. The reflex: a namespace with zero processes that will not die is being pinned — hunt the reference with lsns (look for NPROCS 0) and findmnt/lsof for a bind mount or open fd, rather than rebooting. This is the dark side of the same pinning mechanism ip netns uses on purpose (B6.13).

    Gotchas

    Both symptoms share the module's recurring shape: the loud symptom names the wrong object. "The PID namespace didn't work" is really "/proc wasn't remounted"; "the kernel won't delete my namespace" is really "something still holds a reference." Reading the real state — findmnt /proc, lsns for zero-process namespaces, findmnt/lsof for the pinning reference — redirects you from the innocent-but-loud object to the guilty-but-quiet one, exactly as the earlier drills trained.

    The pinning that causes symptom 2 is not a bug — it is the mechanism that makes named network namespaces (and CNI-managed pod namespaces) persist on purpose. The lesson is not "never bind-mount an ns handle" but "know that you did": every deliberate pin is a reference you are responsible for releasing. Leaks happen when a tool or script pins a handle and forgets to unmount it on cleanup — the fix is always to find and drop the reference, never to force-kill or reboot.

    Both are real node incidents. Symptom 1 is why a debugging container that shows "all the node's processes" might simply be sharing the host PID namespace (hostPID) or lack its own /proc — read the mounts before concluding isolation failed. Symptom 2 is a genuine class of node bug: leaked network namespaces from pods whose CNI teardown failed to unpin the ns handle, accumulating as zero-process namespaces that lsns reveals and that slowly consume kernel resources until found and unmounted. When a node mysteriously accrues network namespaces or won't release pod IPs, this drill is the diagnosis — lsns, findmnt, drop the reference.

    These map onto the node-level troubleshooting in W6; hostPID and shared namespaces connect back to B6.2.

    B6 recall quiz

    Answer out loud before revealing. If you can explain what a pod's shared namespaces are and what the pause container holds, this module has done its job.

    1. In one sentence, how does a namespace differ from a virtual machine?
      Answer

      A VM emulates a whole machine with its own kernel; a namespace is one real kernel giving a process a private copy of a single resource (process list, hostname, mounts, network…). Same kernel, same scheduler — only the bookkeeping table for one specific question is forked. That is why containers start in milliseconds and a VM in seconds.

    2. Name four of the eight namespace types and the question each one changes the answer to.
      Answer

      Any four of: mnt — what is mounted where; pid — which processes exist (and their PIDs); net — which interfaces/addresses/routes/ports exist; ipc — which System V shared-memory/queues exist; uts — what is my hostname; user — which uid/gid I am and what it may do; cgroup — where I sit in the resource hierarchy; time — what the boot/monotonic clocks say.

    3. How do you decide, with certainty, whether two processes share a network namespace?
      Answer

      Compare readlink /proc/PID/ns/net for each. The number in net:[inode] is the namespace's identity; identical inode ⇒ same namespace, different inode ⇒ different namespace. Membership is by inode, exactly like a file is its inode not its name (B5.6).

    4. You run unshare --pid --fork bash, and inside echo $$ prints 1 but ps lists every host process. Explain, and give the fix.
      Answer

      The PID namespace is real (you are PID 1), but ps reads /proc, which is a mount-namespace artifact — and you never remounted it, so ps reads the host's /proc. Fix: add --mount-proc (which implies --mount and mounts a fresh procfs), i.e. unshare --pid --fork --mount-proc bash.

    5. Why does killing PID 1 inside a PID namespace end the whole namespace, and why does a container's app-as-PID-1 sometimes ignore SIGTERM?
      Answer

      When a PID namespace's PID 1 exits, the kernel SIGKILLs every other process in it and tears it down — PID 1 is the namespace's anchor. And the kernel will not deliver a default-action signal (like SIGTERM) to PID 1 unless the process installed a handler, so an app as PID 1 with no handler ignores SIGTERM — which is why containers need a signal-forwarding init or a handler, and why terminationGracePeriodSeconds exists.

    6. What does a user namespace change that no other namespace does, and why could you be uid 0 inside it yet unable to read /etc/shadow?
      Answer

      It changes who you are — it maps uids/gids, so you can be root (uid 0) inside while the host still sees your unprivileged uid. Your in-namespace root capabilities apply only to resources owned by the namespace; /etc/shadow is owned by the host's real root, against which you are still uid 1000, so the check fails. It is also the namespace that lets an unprivileged user create the others (rootless containers).

    7. Why is chroot not a security boundary, and what does pivot_root do that fixes it?
      Answer

      chroot only changes the base for absolute path lookups; it leaves the old root mounted and reachable, so a process with CAP_SYS_CHROOT and an open fd to an outside directory can climb back out, and it isolates nothing else (same PIDs, network, etc.). pivot_root swaps the mount namespace's root and lets you unmount the old root, so there is no old filesystem reachable from anywhere in the namespace — no ladder to escape on.

    8. What is nsenter -t PID -n doing, and which everyday Kubernetes command is it the internals of?
      Answer

      It joins only the network namespace of the target process (via setns), running your command inside that netns while keeping your own mount/PID view — so you can debug the target's network with host tools it may lack. It is essentially the internals of kubectl exec (which enters a container's namespaces to run a command) and node-level pod-network debugging.

    9. A freshly created network namespace can't reach the internet though the host can. What is it missing, and why can two of them both bind port 80?
      Answer

      It has only a loopback interface, which starts DOWN, and no routes at all — no wire out and no routing table, so there is no path anywhere. Connecting it needs a veth pair and routes (B9). Two netns each have their own private port space, so "port 80" is a separate binding in each — which is exactly how many pods run servers on the same port on one node.

    10. What are the minimum ingredients of "container v0.1", and what did you deliberately leave out?
      Answer

      Ingredients: a rootfs, a bundle of namespaces (unshare --pid --mount --uts --ipc --fork), a pivot_root into the rootfs with the old root unmounted, and a fresh /proc mounted for the PID namespace. Left out: cgroup resource limits (B7), network-namespace wiring (B9), a user namespace (so it runs as real root), and any image format/registry/runtime — those are layers around the core, not the core.

    11. A namespace won't go away long after its last process exited. What is happening and how do you find and fix it?
      Answer

      Something still references it — an open fd or, classically, a bind mount of its /proc/PID/ns/* handle — so the kernel keeps it alive with zero processes. Find it with lsns (look for NPROCS 0) and findmnt/lsof for the pin, then drop the reference (umount the handle / close the fd). Never reboot for it — it's a leaked reference, not a stuck kernel. This is the dark side of the same pinning ip netns uses on purpose.

    12. In one paragraph, what does "a pod's shared namespaces" mean, and what is the pause container for?
      Answer

      A pod is a set of containers that share a chosen bundle of Linux namespaces — always the network namespace (same IP, same localhost, same ports, reachable on 127.0.0.1) and usually IPC, while each container keeps its own mount and (by default) PID view. The pause container is a near-empty container that exists solely to hold those shared namespaces open, so individual app containers can crash and restart without tearing down the pod's network identity. It is the real-world version of the --fork PID 1 in container v0.1: the anchor that keeps the namespaces alive.
