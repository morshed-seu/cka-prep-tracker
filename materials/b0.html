    B0 — Your Linux sandbox

    Every module in this track is hands-on, and most of them are hands-on in a way that breaks the machine on purpose — you will delete a running program's files, unplug its network, fill its memory, and pull the filesystem out from under it. That is only a good way to learn if breaking things is free. So the first module builds one Linux computer that exists purely to be wrecked: a virtual machine called sandbox, living inside your real computer, that you can reset to a known-good state in about a minute.

    Nothing here assumes you have used a terminal before. By the end you will have that machine, the tools every later module reaches for, and — the part people skip — the reflex of restoring it without hesitating. Ticking done on a lesson here also ticks the matching checkpoint on the tracker.

    Checkpoints9 Hours~2 Prerequisitesnone
    • Explain why this track teaches the Linux primitive before it names the Kubernetes feature that wraps it.
    • Install a hypervisor front end and launch a specific Ubuntu release on purpose, not by accident.
    • Read a machine's vital signs — kernel, distribution, memory, CPUs, disk — and say what each number will constrain later.
    • Snapshot the sandbox, break it, and restore it in under a minute, without feeling nervous about it.
    • Look up any command, system call or concept offline, using man and its numbered sections.
    • Keep two small scripts of your own that make "reset the sandbox" a single word.

    Nothing. This is the front door of the whole site. You need a computer with roughly 8 GB of RAM and 25 GB of free disk, running Linux, macOS or Windows — the sandbox VM takes 4 GB and 20 GB of that, and only while it is running.

    How to use this track

    B0.1

    Why this track teaches Linux before it teaches Kubernetes

    Why it matters

    Kubernetes has a reputation for being enormous and arbitrary — a wall of new words you are expected to memorize. That reputation is mostly an accident of how it is usually taught. Almost nothing in Kubernetes is new; it is a thin coordination layer over operating-system machinery that has existed for decades. If you meet the machinery first, the Kubernetes layer stops being vocabulary and starts being obvious.

    The concept

    Take one concrete example. A pod is often introduced as "the smallest deployable unit in Kubernetes", which explains nothing. What a pod actually is, on the machine: a set of Linux processes that have been given their own private view of a few kernel resources — their own network interfaces, their own process list, their own filesystem root — while still running on the same kernel as everything else on that computer. Those private views are called namespaces, they are a Linux kernel feature from 2002–2013, and you can create one by hand with a single command. You will do exactly that in B6, long before anyone says "pod".

    That pattern repeats for essentially every Kubernetes concept worth knowing:

    Kubernetes thingWhat it actually isBuilt by hand in
    a pod's isolationLinux namespaces (clone, unshare)B6
    limits.cpu / limits.memorya number written into a cgroup fileB7
    a container imagestacked read-only directories (overlayfs) + a tar fileB5
    the pod networka software bridge, veth pairs, and a NAT ruleB8, B9
    a Service's cluster IPpacket-rewriting rules in the kernel's netfilter tablesB9
    cluster DNSa resolver, a search domain, and /etc/resolv.confB10
    the kubelet staying upa systemd unit with a restart policyB11
    every certificate in the clusterordinary PKI: a CA, a key, a signed requestB12
    etcd surviving a dead nodeRaft consensus over a strict majorityB13

    Hence the order of this track: one machine → one process → what a process may do → its files → isolating it → its network → naming and supervising it → securing it → many machines → assemble. Each module is a prerequisite for the next, which is why there is no calendar here and no suggestion to skip around.

    The device that keeps the three tracks stitched together is a block at the end of every conceptual lesson, headed "Where this shows up in Kubernetes". It names the feature that wraps the primitive you just built and links straight to the advanced lesson that covers it. You are not expected to click it yet — it is there so that when you eventually reach that lesson, it reads as recognition instead of new material.

    Steps
    1. Open the track outline and read the fifteen module titles in order. Do not try to understand them; just notice that the first nine are Linux, not Kubernetes.
    2. Read one Kubernetes-side lesson you have no background for — say requests vs limits — purely to notice how it feels right now. It should feel like a foreign language.
    3. Agree to the order. If you skip to the Advanced track anyway, that is allowed and nothing here breaks; you will simply be memorizing rather than recognizing.
    Gotchas

    The most common way this track fails a reader is impatience in the opposite direction — treating B0–B5 as "basics I already know" and skimming. The point is not the trivia (that ls lists files); it is the mechanism (that ls made an openat system call, which the kernel checked against a permission bitmask attached to an inode). If a module feels familiar, do its debugging drill first: if the drill is quick, skip the module honestly rather than skimming it.

    All of it. This lesson has no single Kubernetes counterpart because it is the map: each row in the table above is a primitive you will build with your own hands here and then meet again wearing a Kubernetes name.

    The clearest single example to look at now, with no expectation of following it: W3.16 — requests vs limits. It says a CPU limit becomes a value in cpu.max in a cgroup. After B7 you will have written that file yourself.

    Docs

    Nothing to look up yet. When you want the shape of the whole curriculum: the beginner tracker lists all 225 checkpoints, and Foundations is a ten-minute express version of the same cross-cutting ideas.

    B0.2

    How a checkpoint works, and where your progress is stored

    Why it matters

    This site has 225 checkpoints across fifteen modules and no accounts, no server and no sign-in. Knowing exactly where the tick marks live takes thirty seconds and saves you the experience of losing four modules of progress to a cleared browser cache.

    The concept

    A checkpoint is one skill, sized so that finishing it is unambiguous. Each one appears twice: as a line on the tracker, and as a lesson article on a module page like this one. They are the same checkpoint — tick either box and the other follows. The loop is always the same four moves:

    read the lesson→run the Lab→run Verify, compare output→tick done

    Only the third step decides anything. Verify is a command whose output you compare against what the lesson says you should see; if it disagrees, something on your machine is different and finding out why is the actual lesson. Reading alone never ticks a box — the whole design of the track assumes the commands were really run on a real machine that could really break.

    Two more blocks close most modules. The mini project builds one small artifact you keep and reuse later (B0's is a pair of scripts you will run before every destructive lab from B5 onward). The debugging drill hands you a broken machine and a symptom, with hints and the full diagnosis hidden behind progressive reveals — because in practice you will always meet a failure symptom-first, never cause-first. B0 has no drill: there is nothing built yet to break.

    Progress is stored in your browser's localStorage — a small key-value store that a page can write and read back on later visits, scoped to the site's origin and living on your disk, not on any server. This site uses a handful of keys:

    KeyHolds
    cka-prep-v1every ticked checkpoint, both tracks, one entry per data-id
    cka-themelight or dark
    cka-collapsed-v1, cka-grp-collapsed-v1which sections and groups you have collapsed

    The consequences are worth stating plainly. Both tracks share one checkpoint store, so beginner and CKA progress are always visible from either tracker, and each tracker's reset button only clears its own prefix. But because the store is per-browser and per-machine, your progress does not follow you to another browser, another computer, or a private window — and "clear site data" erases it. If that matters to you, the fix is a one-liner you will understand after B1: open the browser console and copy localStorage.getItem('cka-prep-v1') into a file.

    Steps
    1. Tick the done box at the top of this lesson.
    2. Open the tracker's B0 section in another tab and confirm checkpoint B0.2 is already ticked there.
    3. Untick it on the tracker, come back here, reload this page, and watch the box here follow.
    Verify

    Open your browser's developer tools (F12 on most browsers), go to the console, and run localStorage.getItem('cka-prep-v1'). You should see a small JSON object with an entry per ticked checkpoint, keyed by id:

    '{"b0-1":1,"b0-2":1}'
    Gotchas

    Private/incognito windows get their own throwaway localStorage that is discarded when the window closes — a whole evening's ticks can vanish that way. Opening the site as a file:// path rather than through a web server also gives you a different (and in some browsers, entirely blocked) storage origin, so ticks may not persist at all; if you are running this site locally, serve it with python3 -m http.server 8000 and visit http://localhost:8000.

    Docs

    Your browser's own devtools documentation, under "Storage" or "Application". Nothing to install; nothing exam-relevant here — this block is site mechanics only.

    The sandbox VM

    B0.3

    Install Multipass — and understand why the sandbox is a VM, not a container

    Why it matters

    You need a Linux machine you are allowed to destroy, and you almost certainly do not want it to be the computer you are reading this on. The cheapest way to get one is a virtual machine: a complete computer simulated in software, with its own kernel, its own disk and its own network stack, running as a process on your real computer. Multipass makes that a one-line command instead of an afternoon.

    The concept

    A hypervisor is the piece of software that convinces a guest operating system it is running on real hardware. Modern CPUs help it out with dedicated instructions, so a guest runs at close to native speed. Your computer already has one: KVM on Linux, the Hypervisor framework on macOS, Hyper-V on Windows. Multipass is a thin, uniform front end over whichever of those you have, plus a downloader for official Ubuntu cloud images — so the same three commands work on all three host operating systems.

    The choice of a VM rather than a container is deliberate, and it is worth understanding on day one because the distinction is the spine of this entire track. Use the hotel metaphor this track will keep returning to: one hotel is one Linux machine. The kernel is the engineering staff — the only ones who touch the wiring. Every program is a guest in a room.

    • A container is a room in your existing hotel. It gets its own closet and its own guest list, so it feels private, but it calls the same engineering staff as every other room — one kernel, shared. That is what "shares the host kernel" means.
    • A virtual machine is a whole separate hotel built inside a warehouse on your property. Its own engineering staff, its own wiring, its own everything.

    This track deliberately vandalizes the wiring: B6 replaces the filesystem root out from under a running process, B7 starves processes of memory until the kernel starts killing them, B9 rewrites the packet-filtering rules and then breaks routing on purpose. In a container, "the wiring" is your own computer's kernel state. In a VM, it is a warehouse you can bulldoze.

    Lab
    1. Install Multipass for your host operating system. Pick one line: 
      sudo snap install multipass          # Linux
brew install --cask multipass        # macOS
# Windows: download and run the installer from multipass.run
    2. Confirm the client and the background daemon are both alive. This is the only check that matters — an installed binary with a dead daemon looks fine and then hangs on the first real command. 
      multipass version
      multipass   1.17.0
multipassd  1.17.0

      Two lines. If multipassd is missing or reports an error, the service did not start — on Linux, snap services multipass will say so.

    3. Ask Multipass what it can build. This also proves image downloads work before you commit to a 20 GB VM. 
      multipass find
      Image             Aliases     Version    Description
20.04             focal       20250xxx   Ubuntu 20.04 LTS
22.04             jammy       20250xxx   Ubuntu 22.04 LTS
24.04             noble       20250xxx   Ubuntu 24.04 LTS
26.04             lts,resolute 2026xxxx  Ubuntu 26.04 LTS
    Verify

    multipass version prints two lines, both with a version number, and multipass find lists 24.04 with the alias noble. That is everything you need before the next lesson.

    Gotchas

    On Linux, hardware virtualization has to actually be enabled — it is switched off in the firmware of a surprising number of machines. lscpu | grep -i virtualization printing VT-x (Intel) or AMD-V means you are fine; printing nothing means the setting is off in your BIOS/UEFI, and Multipass will either fall back to painfully slow emulation or refuse outright.

    On Windows, Multipass uses Hyper-V, which is not available on Home editions; the installer offers VirtualBox as the fallback. On any host, running Multipass at the same time as VirtualBox or VMware can fail, because on some platforms only one hypervisor may hold the CPU's virtualization extensions at a time — shut the other one down rather than debugging it.

    A Kubernetes cluster is a set of machines, and this is what "a machine" will mean throughout. When you reach the advanced track's first practical lesson, W0.5 — create 3 Ubuntu VMs, it asks for exactly the same command you are about to learn, three times over, to build a control plane and two workers.

    Kubernetes calls each of those machines a Node, and a Node object in the API is little more than a record of one such machine's name, addresses and capacity. Everything you will learn about looking after this one sandbox — vital signs, snapshots, restores — is what looking after a node is.

    Docs

    Multipass command reference: multipass help, and multipass help launch for the flags used next. On Linux, man 1 lscpu for the virtualization check, and man 8 kvm-ok (in the cpu-checker package) for a blunter answer.

    B0.4

    Launch sandbox — and always name the release explicitly

    Why it matters

    Every command output pasted into this track was produced on Ubuntu 24.04 LTS. A different release means a different kernel, different defaults and different output — and when your screen disagrees with the lesson, you cannot tell whether you made a mistake or the lesson is stale. Naming the release explicitly costs five characters and removes that entire class of confusion.

    The concept

    Multipass identifies images by version number (24.04), by codename (noble), and by moving aliases like lts. That last kind is the trap: an alias is a pointer that Canonical repoints when a new release ships, so the same command produces a different machine depending on the day you ran it. As of July 2026, lts resolves to Ubuntu 26.04, not 24.04. Write multipass launch 24.04 and you get what this track was written against, today and in two years.

    Three 24.04 defaults matter enough to name now, because whole modules depend on them:

    • cgroups v2 only. The kernel's resource-limiting machinery has two generations, with completely different file layouts. 24.04 mounts only the v2 unified hierarchy at /sys/fs/cgroup. Every path in B7 assumes that; a v1 machine has a different tree and B7's labs will not find their files.
    • iptables is really nftables. The classic firewall command still exists, but on 24.04 it is a compatibility shim that translates into the newer nftables engine. B9 shows both faces of the same rule, which only makes sense on a machine where that shim is in place.
    • systemd-resolved answers on 127.0.0.53. Name lookups go to a local stub resolver on the loopback address rather than straight to a real DNS server, so /etc/resolv.conf shows one nameserver that is the machine itself. B10 spends a whole lesson on why.

    The resource numbers are chosen the same way: 2 vCPU and 4 GB is enough to run several isolated environments at once in B6 and B9 without swapping, and 20 GB leaves room for the image layers B5 unpacks. They are the VM's ceiling, not a reservation — an idle VM does not hold 4 GB of your host's memory hostage.

    Lab
    1. Launch the machine. The first run downloads an image (a few hundred MB) and takes a couple of minutes; later launches are fast. 
      multipass launch 24.04 --name sandbox --cpus 2 --memory 4G --disk 20G
      Launched: sandbox
    2. Confirm it is running and see the address it was given on your host's private VM network. 
      multipass list
      Name       State      IPv4         Image
sandbox    Running    10.87.4.112  Ubuntu 24.04 LTS
    3. Get a shell inside it. This is how every lab in every later module begins; the prompt changes to ubuntu@sandbox so you always know which machine you are typing into. 
      multipass shell sandbox
      Welcome to Ubuntu 24.04 LTS (GNU/Linux 6.8.0-xx-generic x86_64)
ubuntu@sandbox:~$
    4. Inside the VM, confirm you got the release you asked for, then leave again with exit. 
      lsb_release -cs
stat -fc %T /sys/fs/cgroup
      noble
cgroup2fs
    Verify

    Inside the VM, lsb_release -cs prints noble and stat -fc %T /sys/fs/cgroup prints cgroup2fs. The second one is the check that actually matters — it proves the cgroup v2 unified hierarchy that B7 depends on, rather than just trusting the label on the image.

    Gotchas

    Never write multipass launch lts. It is the single most likely way to end up on a machine whose output silently disagrees with this track — and the failure is nasty precisely because everything looks right: the VM boots, the commands exist, and only the details are different.

    A freshly launched VM keeps working for a minute or two after it says Launched: cloud-init is still finishing first-boot setup, and it holds the package manager's lock while it does. If apt in the next lesson complains that it cannot get a lock, you are not broken — wait, or watch it finish with cloud-init status --wait.

    If the name sandbox is already taken from an earlier attempt, multipass delete sandbox && multipass purge removes it for good. delete alone only marks it deleted; purge is what frees the disk.

    Pinning an exact version, rather than accepting whatever "latest" means today, is a habit Kubernetes will demand of you constantly. Cluster components have version skew rules — how far apart the control plane and the kubelets are allowed to drift — and an unplanned upgrade breaks them: see W2.12 — version skew rules.

    It is also why the advanced track's install lesson, W0.6, ends with apt-mark hold kubelet kubeadm kubectl: the same instinct as writing 24.04 instead of lts, applied to a cluster that a routine apt upgrade could otherwise wreck.

    Docs

    multipass help launch for the flag list and defaults. Inside the VM: man 1 lsb_release, man 1 stat (note -f asks about the filesystem, not the file), and man 7 cgroups — the primary source for all of B7, worth knowing exists even though it will be dense today.

    B0.5

    Take the machine's vital signs

    Why it matters

    Five commands describe any Linux machine well enough to reason about it: what kernel it runs, what distribution wraps that kernel, how much memory it has, how many CPUs, and how much disk. You will run these on the first day at any job, on any node that is misbehaving, and at the start of half the debugging drills in this track. Learn them now, on a machine whose answers you already know, so that later you are reading the numbers rather than the commands.

    The concept

    The distinction that trips people up first is kernel versus distribution. The kernel is the actual operating system core — the engineering staff in the hotel metaphor, the only code that touches hardware, manages memory and decides which program runs next. Linux is that kernel and nothing else. A distribution is everything assembled around it: the shell, the package manager, the default configuration, the release schedule. Ubuntu 24.04 and Debian 12 can run nearly the same kernel and still feel like different machines; that is why "which kernel" and "which distro" are two separate questions with two separate commands.

    Each number you are about to read becomes a constraint in a specific later module, which is the real reason to write them down:

    Vital signCommandWhere it binds later
    kernel versionuname -awhich namespace and cgroup features exist at all (B6, B7)
    distributionlsb_release -awhich init system, firewall backend and resolver you get (B9–B11)
    memoryfree -hthe ceiling the OOM killer enforces when B7 starves a cgroup
    CPUsnprocwhat "1 CPU" means as a cgroup quota, and how scheduling is shared (B2, B7)
    diskdf -h /room for the image layers B5 unpacks by hand

    One reading is worth explaining before you see it. free -h prints both free and available memory, and they are very different numbers. Free is memory holding nothing at all — on a healthy Linux machine this is always small, because the kernel spends idle memory on caching files it has read, which costs nothing and speeds everything up. Available is free memory plus all the cache the kernel would instantly give back if a program asked. Available is the number that answers "can I start something big"; a beginner who reads free concludes their machine is out of memory when it is perfectly healthy.

    Lab
    1. Get into the sandbox if you left it. 
      multipass shell sandbox
    2. Kernel first — version, build date and architecture. The number before the first dash is what matters: kernel features arrive in specific versions. 
      uname -a
      Linux sandbox 6.8.0-xx-generic #xx-Ubuntu SMP PREEMPT_DYNAMIC ... x86_64 GNU/Linux
    3. Now the distribution wrapped around that kernel. 
      lsb_release -a
      Distributor ID: Ubuntu
Description:    Ubuntu 24.04 LTS
Release:        24.04
Codename:       noble
    4. Memory. Read the available column, not free. 
      free -h
                     total        used        free      shared  buff/cache   available
Mem:           3.8Gi       380Mi       2.9Gi        1.0Mi       600Mi       3.4Gi
Swap:             0B          0B          0B

      Note total reads 3.8Gi rather than 4.0 — the kernel reserves a slice for itself before reporting. Note also that swap is 0: this machine has no overflow space, so memory pressure ends in a kill rather than a slowdown. That is exactly the behaviour B7 wants you to see, and it is also the default a Kubernetes node runs with.

    5. CPUs, then disk. 
      nproc
df -h /
      2
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        19G  2.1G   17G  12% /
    6. Write all five answers down somewhere you will still have in a month — a note file, a paper notebook, anywhere. Later modules ask you to compare against them.
    Verify

    You can say, without looking: this machine runs a 6.8 kernel, Ubuntu 24.04 "noble", has ~4 GB RAM with no swap, 2 CPUs, and ~17 GB of disk left. If any of those disagree with the numbers above, re-read B0.4 — the most likely cause is a different release or different launch flags.

    Gotchas

    uname -r reports the kernel the machine booted, not the newest one installed. After a kernel upgrade the two differ until you reboot, which is a genuinely common source of "but I installed the fix" confusion — a file in /var/run/reboot-required is Ubuntu's way of telling you.

    df shows a filesystem's free space, which is not the same as the space you can actually use: a running program holding an open handle to a deleted file keeps its blocks allocated until it exits, so a disk can be full with nothing visible to du. B5 comes back to this with the inode machinery that explains it.

    These same five numbers, gathered automatically, are most of what a Kubernetes Node object contains. The kubelet reports them to the API server as the node's capacity, then subtracts what the system itself needs to produce allocatable — the amount actually offered to workloads.

    That number is the whole basis of scheduling: the scheduler places a pod by checking whether its declared CPU and memory requests fit inside a node's remaining allocatable, which is why W3.16 — requests vs limits insists requests are bookkeeping rather than enforcement. And nproc is where "1 CPU" in a pod spec ultimately cashes out.

    Docs

    man 1 uname, man 1 free (read the description of available — it is the part everyone misses), man 1 nproc, man 1 df. For where these numbers come from underneath: man 5 proc, and look at /proc/meminfo and /proc/cpuinfo directly — B3 explains why a directory of fake files is how the kernel answers questions.

    Toolbelt & safety

    B0.6

    Install the toolbelt every later module assumes

    Why it matters

    An Ubuntu cloud image is deliberately minimal — it ships what a server needs to boot, not what a person needs to investigate one. Installing the whole toolbelt once, now, means no later lesson stops to install something mid-experiment, and it gives you a first look at what kind of tools this track is going to make you use.

    The concept

    apt is Ubuntu's package manager: it downloads software, its dependencies, and its man pages from repositories described in /etc/apt/sources.list.d/. apt update refreshes the local catalogue of what is available; apt install then acts on it. Running install without a recent update is the classic first-day failure — the machine looks for a package version that has since been replaced and reports a 404.

    Every package below is here because a specific module needs it:

    PackageWhat it gives youNeeded by
    build-essentialthe C compiler, linker and headersB2, B3 — some things (a signal handler, a raw write) cannot be shown honestly in a shell
    strace, ltracewatch every system call / library call a program makesB3 — the X-ray machines of the whole track
    iproute2ip: interfaces, addresses, routes, namespacesB8, B9
    tcpdumpcapture packets as they cross an interfaceB8, B9
    nftablesthe kernel's packet-filtering and NAT tablesB9 — where a Service's cluster IP really lives
    socatconnect anything to anything: TCP, UDP, unix socketsB9 — talking over a unix socket by hand
    dnsutilsdig, nslookupB10 — asking a specific resolver a specific question
    jq, treeread JSON; see a directory as a shapeeverywhere
    Lab
    1. Inside the sandbox, refresh the catalogue and install everything in one go. -y answers the confirmation prompt in advance. 
      sudo apt update
sudo apt install -y build-essential strace ltrace iproute2 tcpdump \
                    nftables jq tree dnsutils socat
    2. Check them the way a script would, rather than by eye. command -v prints a tool's path and exits non-zero if it is missing, so the loop names anything that failed. 
      for t in cc strace ltrace ip tcpdump nft jq tree dig socat; do
  command -v "$t" >/dev/null || echo "MISSING: $t"
done; echo done
      done

      A bare done with nothing above it is the pass. Anything printed before it names a package that did not install.

    Verify

    The loop above prints only done. As a second check, strace -c /bin/true should print a small table of system calls — proving not just that strace exists but that the kernel lets it attach, which is the part that actually matters in B3.

    Gotchas

    If apt reports it could not get lock, cloud-init is still finishing first-boot work and holds it. Wait it out with cloud-init status --wait rather than killing the lock holder — a half-finished package operation is a genuinely annoying state to repair.

    Note that the binary is often not named after the package: you install dnsutils and get dig; you install build-essential and get gcc and cc; you install nftables and get nft. That mismatch is why the verification loop tests commands rather than package names.

    This is the kit you debug a Kubernetes node with — the layer below anything kubectl can reach. When a pod cannot resolve a name or a Service refuses to answer, the ladder ends on the node with exactly these commands.

    W6.2 — where logs physically live is that ladder written down for the cluster case: when the API server itself is down, kubectl logs is useless and you are back to a shell, a journal and ip. Every rung of it is a tool you just installed.

    Docs

    man 8 apt, man 1 strace, man 8 ip (and the per-object pages it points to, like man 8 ip-netns), man 8 tcpdump, man 8 nft, man 1 jq. Do not read them yet — the point is knowing they are there, offline, on this machine.

    B0.7

    Snapshot the VM, break it on purpose, restore it

    Why it matters

    The single biggest thing that slows beginners down is not difficulty — it is the quiet fear of breaking the machine, which makes them skip exactly the destructive labs that teach the most. The cure is not courage; it is having already done a restore once, while the stakes were zero. Do it now, deliberately, on a machine holding nothing you care about.

    The concept

    A snapshot freezes the VM's entire disk at one instant. Restoring throws away everything that happened since and puts the disk back exactly as it was — not a merge, not a repair: a replacement. That bluntness is the feature. It means no amount of damage inside the VM is interesting, so you never have to weigh "is this experiment safe" against "will I have to rebuild".

    Two rules follow from how it works. First, the VM must be stopped to take or restore a snapshot: a running machine has state in RAM and half-written blocks on disk, and a frozen copy of that would be incoherent. Second, a snapshot is not a backup of your work — it is a rollback point for the machine. Anything you wrote inside the VM and want to keep should also exist outside it, which in this track means on your host, which is exactly what the mini project's scripts and your notes are for.

    stop→snapshot→start→experiment freely→stop→restore→start
    Lab
    1. From your host (type exit first if you are inside the VM), stop the machine and take the baseline snapshot. Call it clean — you will come back to this exact name for the rest of the track. 
      multipass stop sandbox
multipass snapshot sandbox --name clean
multipass start sandbox
      Snapshot taken: sandbox.clean
    2. Now break something. /etc/hostname is the file the system reads at boot to learn its own name — harmless to lose on a throwaway VM, and unmistakably gone. 
      multipass shell sandbox
sudo rm /etc/hostname
cat /etc/hostname
      cat: /etc/hostname: No such file or directory
    3. Leave the VM and put it back. restore also requires a stopped machine, and it will ask for confirmation because it is about to discard the current disk — that prompt is the whole safety mechanism, so read it once before answering. 
      exit
multipass stop sandbox
multipass restore sandbox.clean
multipass start sandbox
      Do you want to take a snapshot of sandbox before discarding its current state? (Yes/no): no
Snapshot restored: sandbox.clean
    4. Confirm the damage is gone. 
      multipass exec sandbox -- cat /etc/hostname
      sandbox
    5. Do the whole cycle once more and time it. Under a minute is the target; the number matters less than the fact that you now know it is small.
    Verify

    multipass list --snapshots shows a snapshot named clean on sandbox, and /etc/hostname exists again with the contents sandbox. If the restore silently did nothing, check you stopped the VM first — a running instance makes restore refuse rather than proceed.

    Gotchas

    Restoring is destructive in the direction people forget: it discards work done after the snapshot, including work you meant to keep. Answering Yes to that confirmation prompt takes a safety snapshot of the current state first, which is usually the right answer when you are unsure. In a script, -d (--destructive) skips the prompt and discards without asking — convenient and unforgiving, so only use it where discarding is the entire point.

    Each snapshot holds the blocks that have since changed, so they cost disk on your host and stack up if you take one per lab. Prune with multipass delete sandbox.oldname && multipass purge. And note that snapshots belong to the instance: multipass delete sandbox takes them all with it.

    The same "stop it, snapshot it, restore it" reflex is a marked CKA exam skill, applied one level up — to the cluster's database instead of a whole machine. Every object in a Kubernetes cluster lives in etcd, and etcdctl snapshot save takes exactly this kind of point-in-time copy of it.

    See W2.17 — take an etcd snapshot and W2.18 — restore it. The rules rhyme with the ones here: restoring replaces rather than merges, and the control plane must be stopped while it happens. The advanced track also snapshots its lab VMs for the same reason you just did — W0.8.

    Docs

    multipass help snapshot, multipass help restore, multipass help delete. Inside the VM, man 5 hostname explains the file you just deleted and man 1 hostnamectl the modern way to change it.

    B0.8

    Answer your own questions offline with man

    Why it matters

    Every command, configuration file, system call and concept on this machine already has authoritative documentation installed on it. Learning to read that instead of searching the web is faster, always matches your version, and works in the places this material is heading — a locked-down exam desktop, a production node with no internet, a shell inside a container.

    The concept

    The manual is split into numbered sections, and the numbers are not decoration: the same name legitimately appears in several sections meaning different things, and asking for the wrong one is why man sometimes seems to answer a question you did not ask.

    SectionContainsExample
    1commands you typeman 1 ls, man 1 printf
    2system calls — requests to the kernelman 2 mount, man 2 clone
    3library functions (C)man 3 printf
    5file formatsman 5 proc, man 5 resolv.conf
    7concepts and overviewsman 7 signal, man 7 namespaces
    8administration commandsman 8 mount, man 8 ip

    The pairs are the point. man 1 printf is the shell command; man 3 printf is the C function with the same name. man 8 mount is the tool you type; man 2 mount is the system call that tool makes — and B5 needs the second one. Running man mount without a number gives you whichever section wins a fixed priority order, which is usually 1 or 8 and almost never the one this track wants.

    Section 7 is the underrated one. It holds long conceptual essays rather than command syntax, and three of them — man 7 namespaces, man 7 capabilities, man 7 cgroups — are the primary sources for entire modules of this track. They are dense on a first read and that is fine; you are learning that they exist.

    Lab
    1. Read a section-7 concept page end to end — this one describes every signal a process can receive, and B2 is built on it. Navigate with the arrow keys, search with /, quit with q. 
      man 7 signal
    2. Prove the sections really differ. Open both and compare the first paragraph of each: one describes a command-line tool with flags, the other a kernel call with C arguments. 
      man 8 mount
man 2 mount
    3. Search when you do not know the name yet. apropos (identical to man -k) searches every page's one-line description. 
      apropos namespace | head -5
      namespaces (7)       - overview of Linux namespaces
unshare (1)          - run program in new namespaces
unshare (2)          - disassociate parts of the process execution context
setns (2)            - reassociate thread with a namespace
nsenter (1)          - run program in different namespaces

      Note unshare appearing as both (1) and (2) — the command and the system call it wraps. That pairing is the shape of most of this track.

    4. Know the faster path for a flag you half-remember. Almost every tool answers --help in a screen, without a pager. 
      ip --help 2>&1 | head -3
    Verify

    Without searching the web, answer: which section documents /etc/resolv.conf, and which documents the clone system call? Check yourself with man -f resolv.conf and man -f clone — -f prints just the sections a name appears in.

    resolv.conf (5)      - resolver configuration file
clone (2)            - create a child process
clone (3)            - ...
    Gotchas

    apropos depends on a search index that is built by a background job, so on a VM launched minutes ago it can report nothing appropriate for names that plainly exist. Building it by hand with sudo mandb takes a few seconds and fixes it permanently.

    Also, minimal images sometimes ship with man pages stripped out to save space. Ubuntu's Multipass image keeps them, but a container image usually does not — which is worth remembering before you plan to read documentation from inside one.

    Kubernetes has its own version of this, and it is the highest-value habit in the whole exam: kubectl explain. It reads the API schema out of your live cluster and describes any field of any object — kubectl explain pod.spec.containers.livenessProbe — with no browser and no guessing about which version's fields you are looking at.

    W0.11 — kubectl explain drills it. The instinct is identical to the one you are building here: the machine in front of you already knows, and asking it is faster and more accurate than asking the internet.

    Docs

    man 1 man (the manual's manual — read the SECTIONS part), man 1 apropos, man 8 mandb. Then browse man 7 man-pages for the conventions every page follows, which makes unfamiliar ones far quicker to skim.

    B0.9

    snapshot.sh / restore.sh — make resetting the sandbox one word

    Why it matters

    You just proved a restore is cheap. But "cheap" only changes your behaviour if it is also effortless — three commands with an instance-dot-snapshot name to remember is exactly enough friction to make you skip the interesting labs. Two small scripts remove it, and you will run them before every destructive lab from B5 onward.

    The concept

    These live on your host, not in the VM — they drive the VM from outside, and a script inside a machine cannot restore that machine. Three design decisions are worth naming, because they recur every time you write a small tool:

    • Fail loudly. set -euo pipefail makes the shell stop at the first failing command (-e), treat an unset variable as an error (-u), and notice a failure in the middle of a pipeline (-o pipefail). Without it, a failed stop is followed cheerfully by a snapshot that cannot work.
    • Defaults, not arguments. ${1:-sandbox} means "the first argument, or sandbox if there isn't one". You type snapshot.sh and it does the obvious thing; the argument is there for the day you have a second VM.
    • Do not parse output meant for humans. restore.sh needs the newest snapshot's name, and the tempting move is to scrape it from multipass list --snapshots. That output's shape is not a promise. Instead snapshot.sh records what it created in a small state file, and restore.sh reads that — one fact, written by the thing that knows it.
    Lab
    1. Create both files on your host, somewhere on your PATH (~/bin is conventional; mkdir -p ~/bin if it does not exist).

      Both listings are given in full in the next section of this lesson.

    2. Make them executable — a file is not a command until its execute bit is set. B1 and B4 explain exactly what that bit is. 
      chmod +x ~/bin/snapshot.sh ~/bin/restore.sh
    3. If snapshot.sh is not found when you type it, ~/bin is not on your PATH. Add it, then reload your shell config: 
      echo 'export PATH="$HOME/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
    4. Use them for real: take a snapshot, break the VM, restore it — this time in two words. 
      snapshot.sh
multipass exec sandbox -- sudo rm -rf /etc/apt
restore.sh
multipass exec sandbox -- ls /etc/apt
      snapshot: sandbox.20260723-141205
restored: sandbox.20260723-141205
apt.conf.d  sources.list.d  trusted.gpg.d
    Verify

    Running restore.sh with no arguments, from any directory, puts the sandbox back and prints the snapshot it used — and /etc/apt is present again after you deleted it. Keep the clean snapshot from B0.7 around as a floor to fall back to: restore.sh sandbox clean returns to the freshly-built machine at any point in the track.

    Gotchas

    restore.sh uses --destructive, which skips the confirmation prompt and throws the current disk away without asking. That is the right trade for a sandbox whose entire purpose is being disposable, and the wrong trade everywhere else — do not carry that flag into a script that touches anything you would miss.

    These scripts do exactly two things well and nothing else; resist growing them. Later modules add their own small tools, and the value of each is that you can read the whole thing in ten seconds and know what it will do to your machine.

    Docs

    man 1 bash — enormous, but the sections on Parameter Expansion (for ${1:-default}) and The Set Builtin (for -euo pipefail) are the two worth finding now. man 1 chmod, man 1 date for the timestamp format.
