    Week 3 — Workloads, scheduling & app config

    This is the 15% "Workloads & Scheduling" domain, but it earns more than its weight: pod lifecycle and rollout mechanics are the vocabulary every week-6 troubleshooting task is written in. The pattern of the week: each kubectl abstraction (probe, rollout, taint, limit) is a thin veneer over one concrete mechanism (a kubelet loop, two ReplicaSets, a scheduler filter, a cgroup file). Learn the mechanism and the veneer becomes obvious. Helm and Kustomize close the week — both are directly examinable since the 2025 revision.

    Pod lifecycle internals

    3.1

    Phases vs container states vs conditions — three different status vocabularies

    Why it matters

    "Why is this pod not working?" is the exam's favorite question, and the answer is always written in .status — if you know which of the three vocabularies to read. People who conflate them chase ghosts: a pod can be phase Running while crash-looping, and CrashLoopBackOff isn't a phase at all.

    The concept

    Pod status speaks three languages at once, at three levels:

    VocabularyValuesWhere in status
    Phase — one word for the whole podPending, Running, Succeeded, Failed, Unknown.status.phase
    Container state — per containerWaiting (with a reason), Running (with startedAt), Terminated (with exitCode + reason).status.containerStatuses[].state
    Conditions — checklist of booleansPodScheduled, Initialized, ContainersReady, Ready.status.conditions[]

    The famous names you see in the kubectl get pods STATUS column — CrashLoopBackOff, ImagePullBackOff, ErrImagePull, ContainerCreating — are none of these officially: they're Waiting reasons of a container, surfaced by kubectl as a convenience. Meanwhile the phase of a crash-looping pod stays Running (it's Running as long as at least one container is running or in the process of restarting). The Ready condition is the one with teeth: it's what the EndpointSlice controller reads to decide whether the pod receives Service traffic.

    Lab
    1. A healthy pod — read all three vocabularies: 
      k run web --image=nginx
k get pod web -o jsonpath='{.status.phase}{"\n"}'
k get pod web -o jsonpath='{.status.containerStatuses[0].state}{"\n"}'
k get pod web -o jsonpath='{range .status.conditions[*]}{.type}={.status}{"\n"}{end}'
      Running
{"running":{"startedAt":"2026-07-11T09:14:02Z"}}
PodReadyToStartContainers=True

ContainersReady=True
PodScheduled=True
    2. A crash-looper — watch the vocabularies diverge: 
      k run crash --image=busybox -- sh -c 'exit 1'
sleep 30
k get pod crash

k get pod crash -o jsonpath='{.status.containerStatuses[0].lastState.terminated.exitCode}{"\n"}'

crash   0/1     CrashLoopBackOff   2 (12s ago)   30s

      STATUS says CrashLoopBackOff (a Waiting reason); phase says Running. Both are correct — different vocabularies.

    3. A pod that finishes — the only route to Succeeded: 
      k run once --image=busybox --restart=Never -- true

      k delete pod web crash once $now
    Verify

    You can answer from memory: what's the phase of a crash-looping pod, and where exactly does the string CrashLoopBackOff live in the YAML? (Running; containerStatuses[].state.waiting.reason.)

    Gotchas

    Unknown phase means the apiserver lost contact with the kubelet — a node problem, not a pod problem. And a pod stuck Pending splits on the PodScheduled condition: False → scheduler can't place it (read the condition's message); True but still Pending → it's placed but the kubelet can't start it (image pull, volumes — read the events).

    3.2

    Init containers: serial, to completion, in order

    Why it matters

    Init containers are the standard "wait for X / prepare Y" tool, and a pod stuck in Init:0/2 is a classic troubleshooting prompt — you must know at a glance what that notation means and where to look next.

    The concept

    The kubelet runs initContainers strictly one at a time, in list order, and each must exit 0 before the next starts; app containers start only after the last init container succeeds. If an init container fails, the kubelet re-runs that init container per the pod's restartPolicy (with Never, the whole pod goes Failed). The STATUS column narrates progress: Init:0/2 → Init:1/2 → PodInitializing → Running. One scheduling subtlety: because init containers run before the app, the pod's effective resource request is max(highest init request, sum of app requests) — a fat init container inflates scheduling cost even though it exits.

    Lab
    1. Create a pod with two init containers slow enough to watch: 
      cat <<'EOF' | k apply -f -
apiVersion: v1
kind: Pod
metadata: {name: init-demo}
spec:
  initContainers:
  - name: step-a
    image: busybox
    command: [sh, -c, 'echo step-a running; sleep 8']
  - name: step-b
    image: busybox
    command: [sh, -c, 'echo step-b running; sleep 8']
  containers:
  - name: app
    image: nginx
EOF
k get pod init-demo -w

init-demo   0/1     Init:1/2          0          12s
init-demo   0/1     PodInitializing   0          21s
init-demo   1/1     Running           0          24s

      (Ctrl-C to stop the watch.) Serial and in order: step-b never started until step-a finished.

    2. Logs are per container — init containers included: 
      k logs init-demo -c step-a
      step-a running
    3. See how a failing init container behaves — change step-a to exit 1: 
      k delete pod init-demo $now

#  initContainers:
#  - {name: boom, image: busybox, command: [sh, -c, 'exit 1']}

      k delete pod init-fail $now

    During step 1's watch you saw Init:0/2 → Init:1/2 in that order, and k logs init-demo -c step-b works after the fact.

    Gotchas

    kubectl logs on an Init:… pod without -c errors or shows the wrong container — always name the init container. And init containers re-run from the top if the pod is recreated (rescheduled, node reboot), so they must be idempotent.

    3.3

    Native sidecars: an init container with restartPolicy: Always

    Why it matters

    Sidecars (log shippers, proxies) are everywhere, and the native mechanism is recent enough (GA in v1.33) to be attractive exam material — it's one magic field with several behavioral consequences.

    The concept

    A native sidecar is an entry in initContainers with restartPolicy: Always (the only value allowed there). That one field changes the contract in three ways: (1) at startup it keeps init ordering but doesn't need to exit — it just needs to have started (or pass its startup probe) before the next container begins; (2) it keeps running for the pod's whole life, restarting if it dies, and counts toward the pod's READY column; (3) at shutdown, sidecars are terminated after the main containers, in reverse start order — so a log shipper sees the app's final lines. Before this existed, sidecars were plain app containers: proxies raced their apps at startup, and Jobs never completed because the sidecar kept the pod alive. Both problems die with one field — note that Jobs treat sidecars specially: the Job completes when the main containers finish.

    Lab
    1. A log tailer sidecar sharing an emptyDir with the app: 
      cat <<'EOF' | k apply -f -
apiVersion: v1
kind: Pod
metadata: {name: sidecar-demo}
spec:
  volumes:
  - {name: logs, emptyDir: {}}
  initContainers:
  - name: tailer
    image: busybox
    restartPolicy: Always        # ← the whole feature
    command: [sh, -c, 'tail -F /logs/app.log']
    volumeMounts: [{name: logs, mountPath: /logs}]
  containers:
  - name: app
    image: busybox
    command: [sh, -c, 'while true; do date >> /logs/app.log; sleep 2; done']
    volumeMounts: [{name: logs, mountPath: /logs}]
EOF
sleep 10; k get pod sidecar-demo
      NAME           READY   STATUS    RESTARTS   AGE
sidecar-demo   2/2     Running   0          10s

      READY 2/2 — the "init" container is still running and counted. A normal init container would have shown 1/1.

    2. Prove the sidecar sees the app's output: 
      k logs sidecar-demo -c tailer --tail=3
      Sat Jul 11 09:30:04 UTC 2026
Sat Jul 11 09:30:06 UTC 2026
Sat Jul 11 09:30:08 UTC 2026
    3. Watch shutdown order — main container first, sidecar after: 
      k delete pod sidecar-demo &
sleep 1; k get pod sidecar-demo -o jsonpath='{range .status.containerStatuses[*]}{.name}: {.state}{"\n"}{end}{range .status.initContainerStatuses[*]}{.name}: {.state}{"\n"}{end}'
      app: {"terminated":{...}}
tailer: {"running":{...}}

      Mid-termination, the app is already gone while the tailer still runs — it gets its SIGTERM only after all main containers stop.

    Verify

    k get pod showed 2/2 with the sidecar under initContainers, and in step 3 the sidecar outlived the app container.

    Gotchas

    Setting restartPolicy: OnFailure/Never on an init container is rejected at admission — Always is the only legal value, and only in initContainers (app containers can't have per-container restartPolicy). If a sidecar has a readiness probe, its readiness gates the pod's Ready condition too.

    3.4

    The three probes and their three different consequences

    Why it matters

    "Add a probe to this deployment" is a straight exam task, and probe confusion is the most common self-inflicted production outage. The three probes differ not in how they check but in what happens on failure.

    The concept

    All probes are executed by the kubelet on the pod's node — an httpGet comes from the node's kubelet process, an exec runs inside the container via the CRI, tcpSocket just tries to connect. What differs is the consequence:

    ProbeOn failureFailure symptom
    livenesskubelet kills the container; restartPolicy restarts itRESTARTS climbing, container state Terminated → Running cycles
    readinesspod's Ready condition → False; EndpointSlice controller removes its IP from ServicesREADY 0/1, no restarts, Service has fewer endpoints
    startupsuspends the other two until it first succeeds; if it never does within failureThreshold × periodSeconds, kill like livenessslow-starting app killed before it's up (when missing)

    Tuning fields (same for all three): initialDelaySeconds (default 0), periodSeconds (10), timeoutSeconds (1), failureThreshold (3), successThreshold (1). So an unconfigured probe declares failure after roughly 30 seconds of bad checks.

    Lab
    1. Readiness: a deployment + service whose pod signals readiness via a file: 
      cat <<'EOF' | k apply -f -
apiVersion: apps/v1
kind: Deployment

spec:

  selector: {matchLabels: {app: probed}}
  template:
    metadata: {labels: {app: probed}}
    spec:
      containers:
      - name: web
        image: nginx
        readinessProbe:
          exec: {command: [cat, /tmp/ready]}

      k patch deployment probed --type=json -p='[{"op":"add","path":"/spec/template/spec/containers/0/livenessProbe","value":{"exec":{"command":["cat","/tmp/alive"]},"periodSeconds":5}}]'

    Gotchas

    Liveness probes on an overloaded app are a death spiral: slow responses fail the probe, the restart makes it slower, repeat. Overload is a readiness problem (shed traffic), not a liveness problem (restart). Also: a pod with no readiness probe is Ready the moment it starts — nginx serving 502s counts as "ready" unless you probe something meaningful.

    3.5

    Graceful termination: SIGTERM, the grace period, and SIGKILL

    Why it matters

    Explains three everyday mysteries: why deletes take 30 seconds, why exit code is 137, and why every serious deployment has a preStop sleep. Also why $now (--force --grace-period 0) exists.

    The concept

    Deleting a pod is a negotiation, not a kill. The apiserver only stamps metadata.deletionTimestamp plus the grace period (terminationGracePeriodSeconds, default 30). The kubelet sees that and, per container: runs the preStop hook if any (it counts against the grace period), then sends SIGTERM to PID 1. If the container is still alive when the grace period expires: SIGKILL, which reports as exit code 137 (128 + signal 9). In parallel — not before — the endpoints controller removes the pod from Services. That parallelism is a race: a proxy can still route a request to a pod that already got SIGTERM. The standard fix is a preStop hook of sleep 5: the pod keeps serving during the sleep while endpoint removal propagates, then gets SIGTERM. --force --grace-period 0 skips the negotiation: the API object is deleted immediately without waiting for kubelet confirmation — fine for lab pods, dangerous for stateful things.

    kubectl delete→deletionTimestamp set→preStop hook→SIGTERM → PID 1→grace period ticks (30s)→SIGKILL (exit 137)
    Lab
    1. A pod that handles SIGTERM exits fast: 
      k run polite --image=busybox -- sh -c 'trap "echo terminating; exit 0" TERM; while true; do sleep 1; done'
sleep 5; time k delete pod polite
      pod "polite" deleted

real    0m2.31s
    2. A pod that ignores SIGTERM burns the whole grace period: 
      k run rude --image=busybox -- sleep 3600
sleep 5; time k delete pod rude
      pod "rude" deleted

real    0m31.18s

      sleep as PID 1 has no TERM handler, and PID 1 ignores unhandled signals — so the kubelet waits 30s and SIGKILLs. This is why sh -c "…" wrapper entrypoints (PID 1 = sh, which doesn't forward signals) make apps take the full 30s to die.

    3. Tune the grace period and add a preStop hook: 
      cat <<'EOF' | k apply -f -
apiVersion: v1
kind: Pod
metadata: {name: hooked}
spec:
  terminationGracePeriodSeconds: 10
  containers:
  - name: c
    image: busybox
    command: [sleep, "3600"]
    lifecycle:
      preStop:
        exec: {command: [sh, -c, 'sleep 5']}
EOF
sleep 8; time k delete pod hooked
      pod "hooked" deleted

real    0m11.42s

      ~11s: 5s preStop + SIGTERM ignored + remaining grace → SIGKILL at the 10s mark.

    Verify

    The three timings ordered as predicted: ~2s (handler), ~31s (default grace), ~11s (10s grace incl. preStop).

    Gotchas

    terminationGracePeriodSeconds lives at pod spec level, not per container. A pod stuck in Terminating forever usually means its node's kubelet is unreachable — the API waits for a confirmation that never comes; that's when --force is legitimate. Exit code decoder: 137 = SIGKILL (OOM kill or grace expiry), 143 = SIGTERM handled and exited.

    Docs

    Docs search: "Pod Lifecycle" (§ Termination of Pods) and "Container Lifecycle Hooks".

    Deployments & rollouts

    3.6

    pod-template-hash: why every ReplicaSet name ends in gibberish

    Why it matters

    Once you see that a Deployment is just a manager of hashed ReplicaSets, every rollout behavior — surges, rollbacks, revision history — stops being magic and becomes something you can inspect with k get rs.

    The concept

    A Deployment never touches pods. It manages ReplicaSets, one per distinct pod template. The deployment controller hashes the pod template and injects pod-template-hash=<hash> into three places: the ReplicaSet's name suffix, its label selector, and the pods' labels. Change anything in the template → different hash → the controller creates a new ReplicaSet and shifts replicas across. The hash in the selector is what stops ReplicaSets from adopting each other's pods — without it, two RSes with selector app=web would fight over the same pods. Old ReplicaSets aren't deleted, just scaled to 0: they are the revision history (capped by revisionHistoryLimit, default 10). A rollback is not time travel — it copies an old RS's template back into the Deployment, which makes that old hash "current" again and scales that RS back up.

    Lab
    1. Create a deployment and find the hash in all three places: 
      k create deployment web --image=nginx:1.27 --replicas=3
k get rs -l app=web
k get pods -l app=web --show-labels | head -2

web-7c56dcdb95   3         3         3       20s
NAME                   READY   STATUS    ...   LABELS
web-7c56dcdb95-8xk2p   1/1     Running   ...   app=web,pod-template-hash=7c56dcdb95
{"app":"web","pod-template-hash":"7c56dcdb95"}
    2. Change the template → new hash, new ReplicaSet: 
      k set image deployment/web nginx=nginx:1.29

      NAME             DESIRED   CURRENT   READY   AGE
web-7c56dcdb95   0         0         0       2m

      The old RS is still there at 0 replicas — that's revision 1, retrievable.

    3. Roll back and watch the old RS scale up (no third RS appears):

    After the rollback, the RS from step 1 (same hash) is the one with 3 replicas, and rollout history shows the revision number advanced.

    Gotchas

    Scaling (k scale deployment web --replicas=5) does not create a new RS or revision — replicas lives outside the pod template. And never edit or delete the auto-created ReplicaSets directly: the deployment controller will fight you and win.

    3.7

    Watch a rolling update live: maxSurge and maxUnavailable in action

    Why it matters

    Exam tasks say things like "update the image ensuring at most one pod is unavailable" — that's a maxSurge/maxUnavailable sentence. You need to translate fluently in both directions.

    The concept

    During a rolling update two ReplicaSets scale in opposite directions under two constraints, both defaulting to 25% of desired replicas: maxSurge — how many pods may exist above desired (rounded up); maxUnavailable — how many may be missing from desired (rounded down). For 6 replicas: surge = ceil(1.5) = 2 → never more than 8 total; unavailable = floor(1.5) = 1 → never fewer than 5 ready. The controller repeatedly scales the new RS up to the surge ceiling, waits for readiness, scales the old RS down to the unavailable floor, repeat. Readiness probes are what make this safe: a pod counts as available only when Ready — bad image with no probe still "succeeds" the rollout. Corner cases worth knowing: maxUnavailable: 0 forces surge-only (needs headroom); both can't be 0 (rejected).

    Lab
    1. Six replicas, then trigger an update and watch both ReplicaSets (two terminals, or run the watch in the background): 
      k scale deployment web --replicas=6 && sleep 10

    2. Confirm the strategy numbers you were just watching: 
      k get deployment web -o jsonpath='{.spec.strategy}{"\n"}'

      k patch deployment web -p '{"spec":{"strategy":{"rollingUpdate":{"maxSurge":1,"maxUnavailable":0}}}}'

    In the step-1 transcript you can point at the moment each constraint bound: the 8-pod ceiling (surge) and the 5-ready floor (unavailable).

    Gotchas

    Absolute numbers and percentages are both legal. Percentages compute against spec.replicas, not current count. And a rollout that "hangs" is usually maxUnavailable: 0 + a new pod that never goes Ready — the controller is waiting forever for readiness; k rollout status + k describe pod on the new RS's pod shows why.

    3.8

    The rollout command family, drilled to reflex

    Why it matters

    Rollback tasks are pure speed points: the whole answer is two commands if they're reflexes. This is also where --record trips up people studying from old material — it's gone.

    The concept

    Everything operates on the revision list from 3.6 (retained ReplicaSets). --record was removed; its replacement is setting the kubernetes.io/change-cause annotation on the Deployment after each change — rollout history displays it. pause/resume exist to batch several template edits into one revision (each set image/set resources normally triggers its own rollout).

    CommandWhat it does
    k rollout status deploy/webblocks until done or stuck — your "did it work?" after any change
    k rollout history deploy/web [--revision=2]list revisions / dump one revision's full template
    k rollout undo deploy/web [--to-revision=1]previous revision, or a specific one
    k rollout pause/resume deploy/webbatch edits into one revision
    k rollout restart deploy/webrecreate all pods, same template (config reload trick — see 3.22)
    Lab
    1. Annotate each change so history stops saying <none>: 
      k set image deployment/web nginx=nginx:1.27
k annotate deployment/web kubernetes.io/change-cause="downgrade to 1.27 for bug X"
k rollout history deployment/web | tail -3
      4         <none>
5         downgrade to 1.27 for bug X
    2. Inspect a specific revision before rolling back to it (what image was rev 4?): 
      k rollout history deployment/web --revision=4 | grep -i image
          Image:      nginx:1.29
    3. Targeted rollback, then confirm: 
      k rollout undo deployment/web --to-revision=4
k rollout status deployment/web

      deployment "web" successfully rolled out
nginx:1.29
    4. Pause, stack two changes, resume — one revision, not two: 
      k rollout history deployment/web | tail -1        # note the latest revision number
k rollout pause deployment/web
k set image deployment/web nginx=nginx:1.29
k set resources deployment/web -c nginx --requests=cpu=100m
k get rs -l app=web                               # nothing rolling — paused
k rollout resume deployment/web
k rollout history deployment/web | tail -1
      6         <none>
...
7         <none>    ← both edits landed as a single new revision
    Verify

    From a cold prompt: image of revision N, roll back to it, confirm — under 60 seconds.

    Gotchas

    undo --to-revision to a revision whose RS was pruned (past revisionHistoryLimit) fails — history only reaches as far as retained ReplicaSets. And remember from 3.6: a successful undo consumes the old revision number and issues a new one; don't be confused when "revision 2" disappears from the list after you roll back to it.

    Docs search: "kubectl rollout" (reference) or the Deployments page § Rolling Back.

    3.9

    Recreate strategy, and why StatefulSets & DaemonSets roll differently

    Why it matters

    Not every workload can surge. Recognizing "this can't run two versions at once" or "this is one-per-node" tells you which rollout semantics — and which failure modes — apply.

    The concept

    Deployment, strategy Recreate: kill all old pods, wait, then create all new ones — full downtime window, but never two versions alive. You want it when versions can't coexist: a ReadWriteOnce volume both versions would mount, a DB schema migration, a singleton with a lock. StatefulSet (updateStrategy: RollingUpdate): pods are updated one at a time, highest ordinal first (web-2, then web-1, then web-0), each must be Ready before the next — identity-preserving, never a surge pod (there is no "web-1a"). Its partition field is a built-in canary: only ordinals ≥ partition update. DaemonSet: one pod per node, so surging is meaningless — the controller deletes the pod on a node and recreates it on the same node, maxUnavailable (default 1) nodes at a time. Both StatefulSet and DaemonSet also support updateStrategy: OnDelete: the controller updates nothing until you delete pods by hand.

    3.11

    Taints & tolerations: the three effects, and the one that evicts

    Why it matters

    "Why won't anything schedule on this node?" and "why did my pods suddenly disappear from node X?" are both taint questions. Taints are also how Kubernetes itself manages nodes — control-plane isolation and dead-node eviction are just taints.

    The concept

    First, the map this whole group hangs on. Week 1 gave you the scheduler framework; here's where each control you're about to learn plugs in:

    Now taints. A taint is a node-side repellent: key=value:effect. A toleration on a pod only neutralizes a taint — it never attracts the pod (attraction is affinity's job, 3.12). The three effects escalate: PreferNoSchedule — avoid if possible (a Score-time nudge); NoSchedule — hard Filter, but already-running pods stay; NoExecute — hard Filter plus eviction of running pods that don't tolerate it, enforced after scheduling by the taint manager. A toleration can carry tolerationSeconds for NoExecute: "I'll stay N seconds, then leave". Kubernetes runs on this: the control-plane node carries node-role.kubernetes.io/control-plane:NoSchedule, and dead-node handling is the node lifecycle controller adding node.kubernetes.io/unreachable:NoExecute — every pod silently gets a 300s toleration for it, which is exactly why pods on a dead node take 5 minutes to move.

    3.12

    nodeSelector vs node affinity: hard filter vs weighted score

    Why it matters

    "Schedule this pod on a node with label X" is one of the most literal recurring exam tasks. The skill being tested is writing the affinity YAML shape from memory — it's the most deeply nested spec in the exam.

    The concept

    Three tools, one decision (see the diagram in 3.11): nodeSelector — flat key:value match, pure Filter, three lines of YAML; use it whenever the task says nothing about "prefer". requiredDuringSchedulingIgnoredDuringExecution node affinity — also a hard Filter, but with expressions (In, NotIn, Exists, Gt…); multiple matchExpressions in one term AND together, multiple nodeSelectorTerms OR together. preferredDuringSchedulingIgnoredDuringExecution — a Score-time nudge with a weight (1–100); the pod schedules somewhere even if nothing matches. The suffix "IgnoredDuringExecution" is a promise worth reading literally: the rule is checked at scheduling only — later label changes don't move a running pod.

    3.13

    Pod anti-affinity: spreading replicas, and what topologyKey really means

    Why it matters

    "Ensure no two replicas run on the same node" is a classic task, and topologyKey is the field people wave their hands about. It has a precise meaning you can hold onto.

    The concept

    Pod (anti-)affinity differs from node affinity in what it matches: the labelSelector matches other pods, and topologyKey names a node label that defines what counts as "together". Mentally: group all nodes into buckets by their value of the topologyKey label; anti-affinity means "not in the same bucket as a matching pod". topologyKey: kubernetes.io/hostname → each node is its own bucket → "not on the same node". topology.kubernetes.io/zone → "not in the same zone". Like node affinity it comes in required (Filter) and preferred (Score) flavors. For the common "spread N replicas" need, modern Kubernetes also offers topologySpreadConstraints (maxSkew arithmetic) — know it exists; anti-affinity is what the exam phrasing usually points at.

    3.14

    Priority & preemption: how the scheduler chooses victims

    Why it matters

    Priority explains two things you'll meet in troubleshooting: why your pod was suddenly evicted with no node pressure, and how system components always find room. It's also a tidy "create a PriorityClass and use it" task.

    The concept

    A PriorityClass is a tiny cluster-scoped object: a name bound to an integer value (plus optional globalDefault and preemptionPolicy). Pods reference it by name; the admission controller stamps the integer into pod.spec.priority. It acts twice (see 3.11's diagram): in the queue — higher priority pods sort first — and in PostFilter: when no node passes the filters for a high-priority pod, the scheduler simulates removing lower-priority pods node by node until one would fit, then evicts that minimal victim set (respecting graceful termination, 3.5) and writes the node's name into the pod's status.nominatedNodeName. The nomination is not a reservation — the pod re-enters the queue and may land elsewhere if the world changes. Two built-ins tower over everything: system-cluster-critical (2000000000) and system-node-critical (2000001000) — that's how kube-proxy and CoreDNS never lose.

    3.15

    Why it matters

    You already ran drains inside week 2's upgrade drill; this lesson makes the mechanism explicit and the command a reflex, because node-maintenance tasks are near-guaranteed and worth easy points.

    The concept

    Cordon is one boolean. k cordon node02 sets spec.unschedulable: true on the Node object; the node controller mirrors it as a node.kubernetes.io/unschedulable:NoSchedule taint.     3.16

    Requests vs limits: scheduler bookkeeping vs cgroup enforcement

    Why it matters

    "CPU over limit = slow; memory over limit = dead." That asymmetry explains half of all container mysteries, including the exam's beloved exit code 137.

    The concept

    The two fields live in different worlds. Requests exist only in the scheduler's ledger: "requests fit free allocatable" is a Filter (diagram, 3.11) computed from other pods' requests, never from measured usage — a node full of idle pods with fat requests is "full". Limits are runtime enforcement, translated by the kubelet into cgroup v2 files on the node:

    Spec fieldcgroup fileEnforcement
    requests.cpu: 500mcpu.weightproportional share under contention only — no cap
    limits.cpu: 500mcpu.max = 50000 100000CFS quota: 50ms runtime per 100ms period → throttling, never killing
    limits.memory: 64Mimemory.maxexceed → kernel OOM killer → exit code 137, reason OOMKilled
    requests.memory(bookkeeping + eviction ranking, 3.18)no runtime enforcement

    3.17

    QoS classes: derived, not declared — and what they buy you at OOM time

    Why it matters

    When a node runs out of memory, QoS decides who dies first. Exam questions probe whether you know the classes are computed from requests/limits — there is no qos: field to set.

    The concept

    The apiserver derives one of three classes from the pod's resource spec and records it in status.qosClass:

    ClassRuleoom_score_adj (kernel OOM priority)
    Guaranteedevery container: requests == limits for both cpu and memory-997 — nearly untouchable
    Burstableat least one request or limit set, but not Guaranteed2–999, scaled down as memory request grows (≈ 1000 − 1000×memRequest/nodeMem)

The kubelet's own eviction ranking (3.18) is related but distinct: it ranks by whether usage exceeds requests, then priority, then usage-over-requests — QoS influences it indirectly, since BestEffort pods by definition exceed their zero requests.

    3.18

    Two evictions that share a name: node-pressure vs API-initiated

    Why it matters

    "Pod was evicted" has two completely different diagnoses depending on which machinery evicted it. Telling them apart in one look is a week-6 superpower.

    The concept

    Node-pressure eviction is the kubelet defending its node. API-initiated eviction is what drain (3.15) does: an Eviction object per pod — a polite delete that respects PodDisruptionBudgets and full graceful termination, refused with 429 when the budget is empty.

    3.20

    HPA: the control loop, the formula, and a real load test

    Why it matters

    "Create an HPA that scales deployment X between 1 and 10 replicas at 50% CPU" is a one-liner worth full points — if you also understand why it scales up fast, down slow, and never quite when you expect.

    The concept

    The HPA controller (in kube-controller-manager) wakes every 15 seconds, reads pod metrics from metrics.k8s.io (3.19), and computes: desired = ceil(current × actualUtilization / targetUtilization). Utilization is measured against requests — an HPA on a deployment without cpu requests fails with "missing request for cpu". A ±10% tolerance band suppresses jitter (configurable per-HPA since 1.33 via spec.behavior tolerance). The asymmetry is deliberate: scale-up acts immediately (stabilization window 0s); scale-down looks at the highest recommendation over a 300-second stabilization window — so load spikes are answered in seconds, but calm must persist five minutes before replicas drop. Use autoscaling/v2 for anything YAML (v1 only does CPU).

    Why it matters

    3.22

    Update propagation: volumes update, env never, subPath never

    Why it matters

    "I changed the ConfigMap, why does my app still use the old config?" is both a real-life classic and prime exam-trap material. The answer is mechanical, not mysterious.

    The concept

    Three rules, all consequences of 3.21's symlink discovery. Mounted ConfigMaps/Secrets update live: the kubelet periodically re-syncs, writes new contents into a fresh timestamped directory, and flips the ..data symlink atomically — consumers never see half-written files, and the change lands within about a minute (kubelet sync period + cache). Env vars never update: they were baked into the container's process environment at container creation; only a restart re-reads them. subPath mounts never update: subPath: COLOR bind-mounts the file directly, bypassing the ..data symlink — the bind mount keeps pointing at the old timestamped directory forever. The idiomatic escape hatches: kubectl rollout restart after config changes, or Kustomize's configMapGenerator (3.26), which renames the ConfigMap on every content change so a rollout happens by construction.

    3.23

    Secrets are base64, not encryption — and what encryption at rest actually is

    Why it matters

    The exam expects sober knowledge here: create and consume Secrets fast, know exactly what security they do not provide, and recognize the EncryptionConfiguration shape when a task or troubleshooting scenario shows it.

    The concept

    Base64 is an encoding — a reversible representation for arbitrary bytes, zero secrecy. By default a Secret sits in etcd as a plain protobuf record containing those bytes: anyone with etcd access (or a copy of your week-2 snapshot!) reads every secret in the cluster. Real protections are: RBAC on the Secrets API, filesystem protection of etcd, and encryption at rest — an apiserver-side layer configured by an EncryptionConfiguration file passed via --encryption-provider-config. It lists resources and an ordered list of providers (aescbc, secretbox, kms, or identity = plaintext): the first provider encrypts all new writes; the rest can only decrypt existing data. Existing objects stay in their old form until rewritten — which is why enabling encryption ends with a mass rewrite command.

    Why it matters

    The concept

    Helm 3 has no cluster-side component (Tiller died with v2): the client renders a chart's Go templates against values locally and applies the result.     3.26

    Kustomize: a base, an overlay, and kubectl apply -k

    Why it matters

    Kustomize is the curriculum's other config tool, and it's built into kubectl — no install, just -k. The exam shape: "apply this kustomization" or "make overlay X change image/replicas/namespace".

    The concept

    Where Helm templates text, Kustomize transforms objects: a kustomization.yaml names input resources and a pipeline of declarative transformations — no variables, no template syntax, plain YAML in and out. A base is a self-contained kustomization; an overlay is another kustomization that imports it (resources: [../../base]) and layers changes: cross-cutting fields (namespace, namePrefix, commonLabels), targeted patches (strategic-merge or JSON6902, auto-detected), and dedicated transformers like images (retag without touching manifests) and replicas. Note for old study material: patchesStrategicMerge: is deprecated since kustomize v5 — write patches:. The one generator worth knowing: configMapGenerator creates ConfigMaps with a content-hash suffix in the name and rewrites all references — changed config = new name = automatic rollout, the designed fix for 3.22's frozen env vars.

    Lab
    1. Build the base: 
      mkdir -p ~/kapp/base ~/kapp/overlays/prod
cat <<'EOF' > ~/kapp/base/deployment.yaml
apiVersion: apps/v1

metadata: {name: web}
spec:
  replicas: 2
  selector: {matchLabels: {app: web}}
  template:
    metadata: {labels: {app: web}}
    spec:
      containers: [{name: nginx, image: nginx:1.27}]
EOF
cat <<'EOF' > ~/kapp/base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
- deployment.yaml
EOF
    2. Build the prod overlay — namespace, name prefix, a strategic-merge patch, an image retag: 
      cat <<'EOF' > ~/kapp/overlays/prod/replicas.yaml
apiVersion: apps/v1
kind: Deployment
metadata: {name: web}
spec:
  replicas: 4
EOF

- ../../base

    Gotchas

    kubectl apply -k dir/ takes the directory (containing kustomization.yaml), not the file — -f with a kustomization.yaml is a different, wrong thing. namespace: in the kustomization doesn't create the namespace.     3.27

    CRDs & operators: install one, inspect it, recognize the pattern

    Why it matters

    "Custom Resource Definitions and operators" is a named 2025-revision curriculum bullet at the understand and use level: install an operator, find its CRDs, create a custom resource — not write controllers.

    The concept

    A CRD teaches the apiserver a new resource type: post one CustomResourceDefinition and the API instantly grows endpoints for it — storage in etcd, RBAC, kubectl get, even explain (CRDs carry an OpenAPI schema) — with zero behavior. Behavior comes from a controller watching those objects and reconciling the world toward them; an operator is just the bundle: CRDs + a controller Deployment, usually installed from one manifest or Helm chart. The punchline of the whole track: this is exactly the architecture you learned in week 1 — Deployment + deployment-controller and Certificate + cert-manager are the same watch/reconcile loop, one built-in, one installed. The API machinery is uniform; that's why kubectl works on resources that didn't exist yesterday.

    Week 3 recall quiz

    Answer out loud before revealing. If any answer surprises you, redo that lesson's lab.

    1. A pod shows CrashLoopBackOff. What is its phase, and where exactly does that string live in the status YAML?
      Answer

      Phase is Running (restarting counts). The string is a container Waiting reason: .status.containerStatuses[].state.waiting.reason. Phases, container states and conditions are three separate vocabularies (3.1).

    2. What single field turns an init container into a sidecar, and in what order do sidecars stop at pod shutdown?
      Answer

      restartPolicy: Always on an initContainers entry (the only value allowed there; GA in v1.33). At shutdown they stop after the main containers, in reverse start order — so a log shipper sees the app's final output (3.3).

    3. Failing readiness probe vs failing liveness probe — give the observable signature of each.
      Answer

      Readiness: READY 0/1, zero restarts, pod IP removed from Service EndpointSlices. Liveness: RESTARTS climbing (kubelet kills, restartPolicy restarts). Overloaded apps need readiness (shed traffic), never liveness (restart storm) (3.4).

    4. Why does a container whose entrypoint is sh -c "myapp" typically take exactly 30 seconds to terminate, and with what exit code?
      Answer

      PID 1 is sh, which doesn't forward SIGTERM and, as PID 1, ignores unhandled signals — so nothing reacts until the grace period (default 30s) expires and the kubelet sends SIGKILL: exit code 137 (128+9) (3.5).

    5. What three places carry the pod-template-hash, and what does a rollback actually do?
      Answer

      The ReplicaSet's name suffix, its selector, and the pods' labels — preventing RSes from adopting each other's pods. Rollback copies an old (retained, scaled-to-0) RS's template back into the Deployment, scaling that RS up again under a new revision number (3.6, 3.8).

    6. A 6-replica deployment rolls with default strategy. Maximum total pods, minimum ready pods, and the rounding rule?
      Answer

      maxSurge 25% of 6 = 1.5, rounded up → 2 → max 8 total. maxUnavailable 25% = 1.5, rounded down → 1 → min 5 ready (3.7).

    7. NoSchedule vs NoExecute — and why do pods on an unreachable node move only after ~5 minutes?
      Answer

      NoSchedule filters new placements only; NoExecute also evicts running pods that don't tolerate it. Dead nodes get node.kubernetes.io/unreachable:NoExecute, and every pod carries an automatic toleration for it with tolerationSeconds: 300 — hence the 5-minute delay (3.11).

    8. Define topologyKey in one sentence, and say what topologyKey: kubernetes.io/hostname means for anti-affinity.
      Answer

      The node label whose value defines what counts as one bucket. With hostname, every node is its own bucket, so anti-affinity means "never two matching pods on the same node" (3.13).

    9. Requests vs limits: name the cgroup mechanism behind each, and the two opposite failure symptoms.
      Answer

      requests.cpu → cpu.weight (share under contention) and scheduler bookkeeping only; limits.cpu → cpu.max CFS quota → throttling (slow, zero restarts, nr_throttled climbing); limits.memory → memory.max → kernel OOM kill (exit 137, OOMKilled, restarts). CPU over = slow; memory over = dead (3.16).

    10. Derive the QoS class: a pod with one container that sets requests.memory = limits.memory but no CPU values at all.
      Answer

      Burstable. Guaranteed requires requests == limits for both cpu and memory in every container. Derived, never declared: read it at .status.qosClass (3.17).

    11. You find pods with status Evicted and the node has a MemoryPressure condition. Which eviction machinery was it, and did PDBs apply?
      Answer

      Kubelet node-pressure eviction — a local decision against thresholds (memory.available < 100Mi etc.). PDBs did not apply; they only bind API-initiated eviction (drain). Hard-threshold kills also skip the grace period (3.18).

    12. An HPA targets 50% CPU; one pod currently averages 240%. How many replicas next, and why does scale-down take ~5 minutes?
      Answer

      desired = ceil(1 × 240/50) = 5, applied on the next 15-second loop. Scale-down uses the highest recommendation over a 300s stabilization window, so calm must persist five minutes (3.20).

    13. You edit a ConfigMap. Which consumers see the change, and which never will?
      Answer

      Volume mounts update within ~a minute via the atomic ..data symlink flip. Env vars never (baked at container creation) and subPath mounts never (bind-mounted past the symlink). Fixes: rollout restart, or Kustomize's configMapGenerator hash-rename (3.22, 3.26).

    14. Where does Helm store release history, and what's the classic helm upgrade value-loss trap?
      Answer

      As Secrets named sh.helm.release.v1.<release>.v<N> in the release namespace — one per revision; rollback re-applies one. Upgrading without repeating --set/-f reverts those values to chart defaults (fresh render, not a patch) — repeat them or use --reuse-values (3.24, 3.25).
