    Week 2 — Installation, PKI & cluster lifecycle

    Last week you learned what the control plane is; this week you learn how it gets built, trusted, upgraded, and recovered. Every lesson here maps to a guaranteed exam-task family: init/join, certificate surgery, the upgrade dance, and the etcd backup/restore drill. By the weekend, each should run without notes.

    kubeadm init, dissected

    2.1

    List the kubeadm init phases from memory

    Why it matters

    Exam tasks say things like "regenerate the apiserver certificate" or "the join is failing at preflight". If you know init as a pipeline of named phases, every such task becomes "re-run one phase" instead of "rebuild the cluster".

    The concept

    kubeadm init is not one monolithic action — it's a fixed pipeline, and every stage is independently re-runnable via kubeadm init phase <name>:

    preflight→certs→kubeconfig→etcd→control-plane→kubelet-start→wait-control-plane→upload-config→mark-control-plane→bootstrap-token→kubelet-finalize→addon (CoreDNS, kube-proxy)

    The load-bearing details:

    • certs and kubeconfig only write files under /etc/kubernetes. Nothing runs yet.
    • etcd and control-plane just drop static pod YAML into /etc/kubernetes/manifests — kubeadm never starts a process itself; the kubelet notices the files and starts the pods (week 1.2 machinery).
    • upload-config stores the cluster configuration as the kubeadm-config ConfigMap in kube-system. Later kubeadm join and kubeadm upgrade read that, not your original command line — it's the cluster's memory of how it was built.
    • bootstrap-token mints the join token; kubelet-finalize switches the kubelet onto its own rotated client cert (lesson 2.8).
    Lab
    1. On cp, list the pipeline straight from the binary: 
      kubeadm init --help | sed -n '/preflight/,/show-join-command/p'
      preflight                    Run pre-flight checks
certs                        Certificate generation
  /ca                          Generate the self-signed Kubernetes CA ...
kubeconfig                   Generate all kubeconfig files ...
etcd                         Generate static Pod manifest file for local etcd
control-plane                Generate all static Pod manifest files ...
kubelet-start                Write kubelet settings and (re)start the kubelet
...
addon                        Install required addons for passing conformance tests
    2. Re-run a single phase against the live cluster — regenerate just the apiserver serving cert: 
      sudo ls -l --time-style=+%H:%M /etc/kubernetes/pki/apiserver.crt
sudo kubeadm init phase certs apiserver
sudo ls -l --time-style=+%H:%M /etc/kubernetes/pki/apiserver.crt
      -rw-r--r-- 1 root root 1289 09:14 /etc/kubernetes/pki/apiserver.crt
[certs] Generating "apiserver" certificate and key
[certs] apiserver serving cert is signed for DNS names [cp kubernetes ...] and IPs [10.96.0.1 10.211.64.10]
-rw-r--r-- 1 root root 1289 10:02 /etc/kubernetes/pki/apiserver.crt   ← new timestamp

      Same CA, fresh leaf cert. This one command is the fix for half the certificate faults week 6 will throw at you.

    3. Read the cluster's memory of its own build: 
      kubectl -n kube-system get cm kubeadm-config -o yaml | head -20
      apiVersion: v1
data:
  ClusterConfiguration: |
    apiServer: {}
    ...
    kubernetesVersion: v1.35.x
    networking:
      podSubnet: 10.244.0.0/16
      serviceSubnet: 10.96.0.0/12

      There's your --pod-network-cidr from week 0 — persisted, so join and upgrade don't need you to remember it.

    Verify

    Close the page and recite the pipeline. Minimum bar: certs → kubeconfig → etcd/control-plane manifests → kubelet starts them → upload-config → token → addons, and why each early phase is "just files".

    Gotchas

    After regenerating a cert that a running component already loaded, the component keeps using the old one until restarted — certs are read at process start. That's why cert tasks pair with "restart the control-plane pods" (2.7). Also: older kubeadm ran kubelet-start earlier; don't be surprised by ordering differences in old blog posts — trust kubeadm init --help on the box.

    Docs

    Docs search: "kubeadm init phase" (every phase, independently documented) and "Creating a cluster with kubeadm".

    2.2

    Tear down and rebuild — and leave the rebuild one minor version old

    Why it matters

    Rebuilding from scratch once turns init from magic into a checklist. And there's a trick: rebuild the cluster one minor version behind (v1.34) on purpose — that gives you a real upgrade to perform in lessons 2.13–2.14 instead of a pretend one.

    The concept

    kubeadm reset undoes what kubeadm did: stops running control-plane containers, wipes /etc/kubernetes (manifests, PKI, kubeconfigs) and /var/lib/etcd. It deliberately does not undo what others did: CNI config in /etc/cni/net.d, iptables/IPVS rules, and your $HOME/.kube/config all survive — each a source of weird behavior on re-init if you forget.

    Lab
    1. Practice the small version first — remove one worker. On node02: 
      sudo kubeadm reset -f
sudo rm -rf /etc/cni/net.d
      [reset] Stopping the kubelet service
[reset] Deleting contents of directories: [/etc/kubernetes/manifests /var/lib/kubelet /etc/kubernetes/pki]
The reset process does not clean CNI configuration. To do so, you must remove /etc/cni/net.d
The reset process does not reset or clean up iptables rules or IPVS tables.

      Read that output — reset itself tells you what it leaves behind. Then on cp, remove the node object (reset can't reach the API to do it for you):

      kubectl delete node node02
kubectl get nodes
      node "node02" deleted
NAME     STATUS   ROLES           AGE   VERSION
cp       Ready    control-plane   7d    v1.35.x
node01   Ready    <none>          7d    v1.35.x
    2. Rejoin it (fresh token — the week-0 one expired long ago; details in 2.9): 
      kubeadm token create --print-join-command      # on cp; paste result on node02 with sudo
    3. Now the full rebuild, downgraded. Reset all three nodes as in step 1. Then on every node, repoint the package repo one minor back and downgrade the binaries (this is lesson 0.6's repo pattern in reverse): 
      sudo sed -i 's/v1\.35/v1.34/' /etc/apt/sources.list.d/kubernetes.list
sudo apt-get update
sudo apt-mark unhold kubelet kubeadm kubectl
apt-cache madison kubeadm | head -3            # pick the newest 1.34 patch
sudo apt-get install -y --allow-downgrades kubeadm=1.34.x-1.1 kubelet=1.34.x-1.1 kubectl=1.34.x-1.1
sudo apt-mark hold kubelet kubeadm kubectl
    4. Re-init on cp exactly as in week 0 — but this time read every phase line as it scrolls and tie it to the 2.1 pipeline: 
      sudo kubeadm init --pod-network-cidr=10.244.0.0/16
mkdir -p $HOME/.kube && sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config && sudo chown $(id -u):$(id -g) $HOME/.kube/config
kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
    5. Join both workers, then confirm the deliberate time-warp: 
      kubectl get nodes
      NAME     STATUS   ROLES           AGE   VERSION
cp       Ready    control-plane   4m    v1.34.x
node01   Ready    <none>          1m    v1.34.x
node02   Ready    <none>          1m    v1.34.x
    6. Snapshot this state (multipass snapshot per 0.8, name it pre-upgrade) — it's your reusable upgrade-practice fixture.
    Verify

    All three nodes Ready at v1.34, and a pre-upgrade snapshot exists for each VM.

    Gotchas

    If a re-init fails preflight with "port 6443 in use" or "manifests already exist", the previous reset didn't finish — run it again. If pods later can't reach each other on a rebuilt cluster, stale CNI state is the usual suspect: you skipped rm -rf /etc/cni/net.d or old flannel iptables rules linger (a reboot clears them).

    Docs

    Docs search: "kubeadm reset".

    2.3

    Know the kubeconfigs kubeadm writes — and whose identity is inside each

    Why it matters

    "Fix the broken kubeconfig" and "why is this component unauthorized?" tasks are solved by knowing that each file in /etc/kubernetes/*.conf is a wrapped client certificate — and by being able to unwrap one in ten seconds.

    The concept

    Kubernetes has no User objects. A "user" is whatever identity a request carries; for certificates the rule is: CN = username, O = group (Subject CommonName / Organization). kubeadm applies the pattern five times:

    FileCN (user)O (group)Power
    admin.confkubernetes-adminkubeadm:cluster-adminscluster-admin via an RBAC binding — revocable
    super-admin.confkubernetes-super-adminsystem:mastersbypasses RBAC entirely in the authorizer — break-glass only
    controller-manager.confsystem:kube-controller-manager—exactly the RBAC that controller needs
    scheduler.confsystem:kube-scheduler—ditto, scheduler-sized
    kubelet.confsystem:node:<name>system:nodesgated by the Node authorizer

    The admin/super-admin split exists because system:masters is hardwired into the apiserver — no RBAC change can lock it out. So day-to-day admin credentials live in a group you can govern, and the ungovernable one stays in a separate file you keep offline.

    Lab
    1. Unwrap the identity inside admin.conf — this two-step is worth memorizing: 
      sudo grep client-certificate-data /etc/kubernetes/admin.conf \
  | awk '{print $2}' | base64 -d | openssl x509 -noout -subject -issuer
      subject=O = kubeadm:cluster-admins, CN = kubernetes-admin
issuer=CN = kubernetes
    2. Compare with super-admin.conf and the scheduler: 
      sudo grep client-certificate-data /etc/kubernetes/super-admin.conf \
  | awk '{print $2}' | base64 -d | openssl x509 -noout -subject
sudo grep client-certificate-data /etc/kubernetes/scheduler.conf \
  | awk '{print $2}' | base64 -d | openssl x509 -noout -subject
      subject=O = system:masters, CN = kubernetes-super-admin
subject=CN = system:kube-scheduler
    3. Prove the admin binding is real RBAC, not magic: 
      kubectl get clusterrolebinding kubeadm:cluster-admins -o wide
      NAME                     ROLE                        SUBJECTS
kubeadm:cluster-admins   ClusterRole/cluster-admin   Group/kubeadm:cluster-admins
    4. Ask the API who you are — works with any credential: 
      kubectl auth whoami
      ATTRIBUTE   VALUE
Username    kubernetes-admin
Groups      [kubeadm:cluster-admins system:authenticated]
    Verify

    For each of the five files you can state the CN, whether it carries a group, and what would happen if you deleted the kubeadm:cluster-admins binding while holding each file.

    Gotchas

    The kubeconfigs embed the CA certificate too (certificate-authority-data) — a corrupted or wrong CA there gives "x509: certificate signed by unknown authority" from kubectl even though the server is fine. That's a favorite planted fault: diff the embedded CA against /etc/kubernetes/pki/ca.crt.

    Docs

    Docs search: "PKI certificates and requirements" — its table of certs and kubeconfigs is this lesson in official form.

    2.4

    The control-plane taint: why it exists, how to lift it

    Why it matters

    Exam clusters are often single-node — where this taint is already removed. Knowing the taint by exact name tells you instantly why a pod won't schedule on a control-plane node, and taint add/remove syntax is week-3 material you get to rehearse early.

    The concept

    kubeadm init's mark-control-plane phase (see 2.1) labels the node and taints it node-role.kubernetes.io/control-plane:NoSchedule. Taints repel pods that don't tolerate them — so ordinary workloads stay off the node that runs etcd and the apiserver, protecting them from noisy neighbors. Control-plane static pods don't care (they bypass the scheduler entirely), and DaemonSets like kube-proxy/Flannel carry a blanket toleration — which is why the node isn't empty.

    Lab
    1. See the taint, then watch it repel a pod: 
      kubectl describe node cp | grep -i taint
kubectl run probe --image=nginx --overrides='{"spec":{"nodeSelector":{"kubernetes.io/hostname":"cp"}}}'
kubectl get pod probe
      Taints:  node-role.kubernetes.io/control-plane:NoSchedule
pod/probe created
NAME    READY   STATUS    RESTARTS   AGE
probe   0/1     Pending   0          20s
      kubectl describe pod probe | tail -3
      Warning  FailedScheduling  ... 0/3 nodes are available: 1 node(s) had untolerated taint {node-role.kubernetes.io/control-plane: },
2 node(s) didn't match Pod's node affinity/selector.
    2. Lift the taint — note the trailing - means remove: 
      kubectl taint node cp node-role.kubernetes.io/control-plane:NoSchedule-
kubectl get pod probe -o wide
      node/cp untainted
NAME    READY   STATUS    NODE
probe   1/1     Running   cp
    3. Restore the guard rail (and clean up): 
      kubectl taint node cp node-role.kubernetes.io/control-plane:NoSchedule
kubectl delete pod probe --force --grace-period 0
      node/cp tainted
pod "probe" force deleted
    Verify

    kubectl describe node cp | grep -i taint shows the taint back in place, and the pending→running→gone story matches what you predicted at each step.

    Gotchas

    Removing a taint doesn't move anything: already-running pods stay put, and re-adding the taint doesn't evict pods either — NoSchedule only gates new scheduling decisions (NoExecute is the one that evicts; week 3). Also, the old taint name node-role.kubernetes.io/master is gone — muscle-memory from old tutorials will "succeed" at removing a taint that isn't there.

    Docs

    Docs search: "Taints and Tolerations".

    PKI — the certificate web

    2.5

    Map /etc/kubernetes/pki: three trust domains and one odd couple

    Why it matters

    Certificate faults are only confusing if the directory is a blur. Once you see it as three independent CAs plus one signing key, every filename explains itself — and week 6's cert faults become filename lookups.

    The concept

    kubeadm builds three separate trust domains. A client cert signed by one CA means nothing to a server trusting another — that isolation is the design, not an accident:

    Lab
    1. Walk the directory and sort every file into a domain: 
      sudo find /etc/kubernetes/pki -type f | sort
      /etc/kubernetes/pki/apiserver-etcd-client.crt      ← etcd domain (client)
/etc/kubernetes/pki/apiserver-etcd-client.key
/etc/kubernetes/pki/apiserver-kubelet-client.crt   ← cluster domain (client)
/etc/kubernetes/pki/apiserver-kubelet-client.key
/etc/kubernetes/pki/apiserver.crt                  ← cluster domain (serving)
/etc/kubernetes/pki/apiserver.key
/etc/kubernetes/pki/ca.crt                         ← cluster CA
/etc/kubernetes/pki/ca.key
/etc/kubernetes/pki/etcd/ca.crt                    ← etcd CA (independent!)
/etc/kubernetes/pki/etcd/ca.key
/etc/kubernetes/pki/etcd/healthcheck-client.crt
/etc/kubernetes/pki/etcd/peer.crt
/etc/kubernetes/pki/etcd/server.crt
/etc/kubernetes/pki/front-proxy-ca.crt             ← front-proxy CA
/etc/kubernetes/pki/front-proxy-client.crt
/etc/kubernetes/pki/sa.key                         ← JWT signing pair
/etc/kubernetes/pki/sa.pub

      (.key siblings abridged.)

    2. Prove the domains connect where the diagram says. Check issuers: 
      for c in apiserver apiserver-etcd-client etcd/server front-proxy-client; do
  echo -n "$c → "; sudo openssl x509 -in /etc/kubernetes/pki/$c.crt -noout -issuer
done
      apiserver → issuer=CN = kubernetes
apiserver-etcd-client → issuer=CN = etcd-ca
etcd/server → issuer=CN = etcd-ca
front-proxy-client → issuer=CN = front-proxy-ca
    3. Find where each file is consumed — the apiserver manifest is the map: 
      sudo grep -E 'client-ca|tls-cert|etcd-c|service-account-key' /etc/kubernetes/manifests/kube-apiserver.yaml
      - --client-ca-file=/etc/kubernetes/pki/ca.crt          ← "trust client certs signed by this"
- --etcd-cafile=/etc/kubernetes/pki/etcd/ca.crt
- --etcd-certfile=/etc/kubernetes/pki/apiserver-etcd-client.crt
- --service-account-key-file=/etc/kubernetes/pki/sa.pub
- --tls-cert-file=/etc/kubernetes/pki/apiserver.crt    ← "serve with this"
    Verify

    Cover the diagram and sort these from memory into their domains: apiserver-etcd-client, front-proxy-client, sa.key, etcd/peer, apiserver-kubelet-client. (Two are trick entries that live in the pki root but belong elsewhere.)

    Gotchas

    sa.key/sa.pub is the pair people mislabel in study groups: it never expires, has no CA, and openssl x509 chokes on it (it's a bare RSA key — inspect with openssl rsa -in sa.key -noout -text). If the apiserver and controller-manager somehow use different sa keys, every ServiceAccount token in the cluster is rejected — an outage signature worth filing away.

    Docs

    Docs search: "PKI certificates and requirements" — keep it open for 2.6 and 2.7 too.

    2.6

    Inspect a certificate by hand: subject, issuer, SANs, expiry

    Why it matters

    One openssl invocation answers four exam-relevant questions about any cert: who is this (Subject), who vouches for it (Issuer), which names is it valid for (SANs), and until when (Validity). Faults live in all four fields.

    Lab
    1. The one command to memorize: 
      sudo openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text -noout | \
  grep -A1 -E 'Issuer|Subject:|Not After|Alternative'
              Issuer: CN = kubernetes
        Validity
            Not Before: Jul 10 09:14:02 2026 GMT
            Not After : Jul 10 09:19:02 2027 GMT
        Subject: CN = kube-apiserver
            X509v3 Subject Alternative Name:
                DNS:cp, DNS:kubernetes, DNS:kubernetes.default, DNS:kubernetes.default.svc,
                DNS:kubernetes.default.svc.cluster.local, IP Address:10.96.0.1, IP Address:10.211.64.10
    2. Decode the SAN list — every entry is a name some client dials:
      SANWho uses it
      kubernetes.default.svc[.cluster.local]in-cluster clients via the kubernetes Service
      10.96.0.1same Service's ClusterIP — always the first IP of the service CIDR
      10.211.64.10 / cpkubelets and your kubectl hitting the node directly

      Add a load balancer or a second hostname without regenerating this cert, and every client using that name gets a TLS failure.

    3. Simulate the diagnosis without breaking anything — ask for a name that isn't in the SANs: 
      curl --cacert /etc/kubernetes/pki/ca.crt --resolve fake-lb:6443:127.0.0.1 https://fake-lb:6443/version 2>&1 | tail -2
      curl: (60) SSL: certificate subject name does not match target host name 'fake-lb'

      kubectl phrases the same failure as "x509: certificate is valid for cp, kubernetes, ... not fake-lb". Either wording means: SAN list vs dialed name mismatch. Fix: regenerate with the extra SAN (kubeadm init phase certs apiserver --apiserver-cert-extra-sans fake-lb after deleting the old pair) — or dial a name that's on the list.

    4. Sweep expiry across the whole directory in one go: 
      sudo find /etc/kubernetes/pki -name '*.crt' \
  -exec sh -c 'echo -n "{} → "; openssl x509 -in {} -noout -enddate' \;
      /etc/kubernetes/pki/apiserver.crt → notAfter=Jul 10 09:19:02 2027 GMT
/etc/kubernetes/pki/ca.crt → notAfter=Jul  8 09:14:02 2036 GMT
...leaf certs 1 year out, CAs 10 years out
    Verify

    Without notes: which four openssl output fields answer "who / vouched by whom / for which names / until when", and what error signature a SAN mismatch produces in kubectl.

    Gotchas

    "Certificate is valid for X, not Y" is a name problem, not an expiry problem — don't reflexively renew (renewal keeps the same SANs and fixes nothing). Conversely an expired cert says "certificate has expired or is not yet valid" — that one 2.7 fixes. Read the error before choosing the tool.

    Docs

    Docs search: "PKI certificates and requirements". openssl itself isn't in the k8s docs — the invocation above has to live in your fingers.

    2.7

    Check expiry and renew everything with kubeadm certs

    Why it matters

    "The cluster's certificates expire soon — renew them" is about as literal as exam tasks get. The trap is the second half everyone forgets: renewed files on disk change nothing until the control-plane processes restart.

    The concept

    kubeadm leaf certificates last 1 year; the CAs last 10 years (both tunable via certificateValidityPeriod / caCertificateValidityPeriod in the kubeadm config). kubeadm certs renew re-signs leaf certs with the existing CA — including the client certs embedded inside admin.conf, scheduler.conf and controller-manager.conf. Two facts explain the whole workflow: certs are loaded at process start (so: restart after renewing), and kubeadm upgrade apply renews everything as a side effect (so: clusters that upgrade regularly never hit the 1-year wall).

    Lab
    1. Read the expiry dashboard: 
      sudo kubeadm certs check-expiration
      CERTIFICATE                EXPIRES                  RESIDUAL TIME   CERTIFICATE AUTHORITY   EXTERNALLY MANAGED
admin.conf                 Jul 10, 2027 09:19 UTC   364d            ca                      no
apiserver                  Jul 10, 2027 09:19 UTC   364d            ca                      no
apiserver-etcd-client      Jul 10, 2027 09:19 UTC   364d            etcd-ca                 no
...
CERTIFICATE AUTHORITY   EXPIRES                  RESIDUAL TIME
ca                      Jul 08, 2036 09:14 UTC   9y
etcd-ca                 Jul 08, 2036 09:14 UTC   9y

      Notice what's missing: kubelet.conf. The kubelet renews its own client cert automatically (2.8), so kubeadm doesn't track it.

    2. Renew everything: 
      sudo kubeadm certs renew all
      certificate embedded in the kubeconfig file for the admin to use ... renewed
certificate for serving the Kubernetes API renewed
...
You must restart the kube-apiserver, kube-controller-manager, kube-scheduler and etcd, so that they can use the new certificates.
    3. Do what it says — bounce all four static pods at once by moving their manifests out for ~30s: 
      sudo mkdir -p /tmp/mk && sudo mv /etc/kubernetes/manifests/*.yaml /tmp/mk/
sleep 30 && sudo mv /tmp/mk/*.yaml /etc/kubernetes/manifests/
kubectl get nodes    # give it ~30s more; expect a refused connection first, then success
      The connection to the server 10.211.64.10:6443 was refused ...   ← normal mid-restart
NAME   STATUS   ROLES           AGE   VERSION
cp     Ready    control-plane   2h    v1.34.x
    4. Refresh your own credentials (your ~/.kube/config is a copy of the old admin.conf — renewal didn't touch it): 
      sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config && sudo chown $(id -u):$(id -g) $HOME/.kube/config
    Verify
    sudo kubeadm certs check-expiration | grep apiserver | head -1
    apiserver    Jul 10, 2027 10:41 UTC   364d   ca   no      ← timestamp moved to just now
    Gotchas

    Renewal renews leaf certs only — an expiring CA is a different, much heavier procedure (docs: "Manual Rotation of CA Certificates"; know it exists, don't memorize it). And if kubectl starts failing with "certificate signed by unknown authority" right after your renew+restart, that's your stale ~/.kube/config copy — step 4, not a broken cluster.

    Docs

    Docs search: "Certificate Management with kubeadm" — contains the check/renew commands and the restart warning verbatim.

    2.8

    Find the kubelet's own client cert and confirm rotation is on

    Why it matters

    The kubelet is the one client that manages its own certificate lifecycle. Knowing where that cert lives — and that kubelet.conf merely points at it — demystifies both the check-expiration gap from 2.7 and a class of node-auth failures.

    The concept

    At join time the kubelet gets a short-lived cert via TLS bootstrap (2.10). From then on it re-requests a fresh client cert via the CSR API before expiry — that's rotateCertificates: true in its config. Each new cert lands in /var/lib/kubelet/pki/, and a symlink kubelet-client-current.pem always names the newest one. kubelet.conf doesn't embed cert data like the other kubeconfigs — it references the symlink path, so rotation never has to rewrite it.

    Lab
    1. Look at the rotation trail on any node: 
      sudo ls -l /var/lib/kubelet/pki/
      -rw------- 1 root root 1113 Jul 10 09:20 kubelet-client-2026-07-10-09-20-15.pem
lrwxrwxrwx 1 root root   59 Jul 10 09:20 kubelet-client-current.pem -> /var/lib/kubelet/pki/kubelet-client-2026-07-10-09-20-15.pem
-rw-r--r-- 1 root root 2258 Jul 10 09:14 kubelet.crt      ← self-signed serving cert (unrelated)
-rw------- 1 root root 1675 Jul 10 09:14 kubelet.key
    2. Confirm kubelet.conf points at the symlink instead of embedding data: 
      sudo grep client-cert /etc/kubernetes/kubelet.conf
          client-certificate: /var/lib/kubelet/pki/kubelet-client-current.pem
    3. Check rotation is enabled in the kubelet's config file: 
      sudo grep -i rotate /var/lib/kubelet/config.yaml
      rotateCertificates: true
    4. Read the identity inside — it's the CN=user / O=group pattern again, node-flavored: 
      sudo openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -subject -enddate
      subject=O = system:nodes, CN = system:node:cp
notAfter=Jul 10 09:15:00 2027 GMT
    Verify

    You can answer: where does the kubelet's client cert live, what keeps it fresh, and why doesn't kubeadm certs check-expiration list kubelet.conf?

    Gotchas

    Rotation needs a working apiserver to file the CSR — a node that was powered off past its cert expiry can't rejoin by itself; its kubelet logs show "part of the existing bootstrap client certificate is expired". The repair is a manual re-bootstrap (docs: "kubelet client certificate rotation fails"). Also don't confuse the pair in that directory: kubelet-client-* is the rotated client identity; kubelet.crt/key is a self-signed serving cert for the kubelet's own :10250 API.

    Docs

    Docs search: "Configure Certificate Rotation for the Kubelet" and (troubleshooting) "Troubleshooting kubeadm".

    Joining nodes & TLS bootstrap

    2.9

    Mint a fresh join command and understand token lifecycle

    Why it matters

    "Join this node to the cluster" is a staple task, and the token from init is long gone — the exam expects you to mint one. Tokens are also just Secrets, which makes their whole lifecycle inspectable.

    The concept

    A bootstrap token is a credential of the form id.secret (e.g. abcdef.0123456789abcdef), stored as a Secret of type bootstrap.kubernetes.io/token in kube-system, valid for 24 hours by default. It authenticates a joining kubelet as a member of system:bootstrappers — a group whose powers are deliberately tiny (next lesson). Deleting the Secret revokes the token instantly.

    Lab

    kubeadm token create --ttl 0 makes a token that never expires — handy in a throwaway lab, an audit finding anywhere real.     2.10

    Watch the TLS bootstrap flow a join actually performs

    Why it matters

    Understanding the four-step handshake under kubeadm join turns node-onboarding failures from voodoo into a pipeline you can bisect — and it's the same CSR machinery week 7 uses to onboard humans.

    The concept
    token auth→group system:bootstrappers→kubelet creates CSR→csrapprover auto-approves→signed client cert→kubelet.conf written
    1. The kubelet connects with the bootstrap token — authenticating as group system:bootstrappers, whose RBAC grants exactly one meaningful power: create CertificateSigningRequests.
    2. It submits a CSR for identity CN=system:node:<name>, O=system:nodes with signer kubernetes.io/kube-apiserver-client-kubelet.
    3. The csrapproving controller in kube-controller-manager auto-approves CSRs that match the node-client pattern from a bootstrapper. (Only this signer is auto-approved — that's why human CSRs in week 7 need manual approval.)
    4. The kubelet downloads the signed cert into /var/lib/kubelet/pki/ and writes kubelet.conf pointing at it. The token's job is done; from here, 2.8's rotation takes over.

    And the flag mirror-image: the token proves the node to the cluster, while --discovery-token-ca-cert-hash proves the cluster to the node — it pins the CA public key, so a spoofed apiserver can't trick your kubelet into joining and shipping it credentials.

    Lab
    1. Redo the 2.9 join, but this time keep a watch running on cp before you paste the join command on node02: 
      kubectl get csr -w
      NAME        AGE   SIGNERNAME                                      REQUESTOR                  CONDITION
csr-x7k2p   0s    kubernetes.io/kube-apiserver-client-kubelet     system:bootstrap:dz54cd   Pending
csr-x7k2p   1s    kubernetes.io/kube-apiserver-client-kubelet     system:bootstrap:dz54cd    Approved,Issued

      Pending→Approved in about a second — that's step 3 happening live. The REQUESTOR column shows the token identity (system:bootstrap:<id>), not a node name: the node identity doesn't exist yet, it's what's being requested.

    2. Inspect what was requested and what was granted: 
      kubectl get csr csr-x7k2p -o jsonpath='{.spec.username}{"  →  "}{.status.conditions[0].type}{"\n"}'
kubectl get csr csr-x7k2p -o jsonpath='{.status.certificate}' | base64 -d | openssl x509 -noout -subject
      system:bootstrap:dz54cd  →  Approved
subject=O = system:nodes, CN = system:node:node02
    3. Verify the bootstrapper group's power is really that narrow: 
      kubectl auth can-i create certificatesigningrequests --as system:bootstrap:dz54cd

      yes

    Verify

    Narrate the four steps and name which component acts in each (kubelet, apiserver authn, csrapproving controller, kubelet again) — plus what each of the two join-command credentials protects against.

    Gotchas

    A join that hangs at "waiting for the kubelet to perform the TLS Bootstrap" means the CSR round-trip stalled: check kubectl get csr — a forever-Pending CSR points at the controller-manager (down, or its csrapprover misconfigured), while no CSR at all points earlier: token/connectivity/preflight on the node itself.

    2.11

    Know the two HA topologies and what the load balancer is for

    Why it matters

    You won't build an HA cluster in the exam, but topology questions hide inside other tasks ("why does the kubeconfig point at a name that isn't a node?"), and the concepts — quorum, controlPlaneEndpoint — explain flags you'll meet everywhere.

    The concept

    Two blessed layouts:

    Stacked etcdExternal etcd
    Where etcd runson each control-plane node, beside the apiserverits own dedicated hosts
    Minimum sensible size3 CP nodes3 etcd + 2+ CP nodes
    Failure couplinglosing a CP node loses an etcd member toodecoupled — pricier, more machines
    How kubeadm builds itinit --control-plane-endpoint lb:6443 --upload-certs, then join --control-planesame, plus an etcd cluster you built separately

    Why 3: etcd needs a Raft quorum (majority). 1 node → quorum 1; 2 nodes → quorum 2, so either failure halts writes (worse than 1!); 3 nodes → quorum 2, one node can die. That's week 1.9's math driving a purchasing decision.

    The load balancer is for the apiservers, not etcd: apiservers are stateless and hot-hot, so clients just need one stable name that spreads across them. That name — the controlPlaneEndpoint — gets baked into every kubeconfig and into the apiserver cert's SANs (2.6), which is why you must decide it at init time: retrofitting HA onto a cluster inited without it means cert and kubeconfig surgery everywhere. etcd needs no LB — its clients (the apiservers) know all member addresses natively.

    Lab
    1. Your lab cluster is the degenerate case — see what's absent: 
      kubectl -n kube-system get cm kubeadm-config -o yaml | grep -i -A1 controlplane
kubectl config view --minify | grep server
      controlPlaneEndpoint: ""        ← empty: no HA was planned
    server: https://10.211.64.10:6443   ← clients pinned to one node's IP

      On an HA cluster both would show the LB name. This empty string is the fork in the road.

    2. Thought-drill (write the answers down): (a) With 3 stacked CP nodes, how many can you lose and keep scheduling pods? (b) With 2? (c) Your kubeconfig points at lb.example.com but TLS fails with a SAN error — what was forgotten at init time?
    Verify

    Answers: (a) one — quorum 2 of 3 survives; (b) none — quorum 2 of 2 means any loss halts etcd writes; (c) lb.example.com wasn't in --apiserver-cert-extra-sans/controlPlaneEndpoint, so it's missing from the apiserver cert's SANs.

    Gotchas

    --upload-certs stashes the control-plane certs as an encrypted Secret so joining CP nodes can fetch them — it expires after 2 hours (re-run kubeadm init phase upload-certs --upload-certs to refresh). Don't confuse the two join flavors: plain join = worker, join --control-plane --certificate-key ... = another control-plane node.

    Docs

    Docs search: "Options for Highly Available Topology" and "Creating Highly Available Clusters with kubeadm".

    Cluster upgrades

    2.12

    Memorize the version skew rules

    Why it matters

    The skew rules dictate the order of every upgrade you'll ever run — control plane first, kubelets after, one minor at a time. Get the rules into memory and the procedure in 2.13/2.14 stops being a recipe and starts being obvious.

    The concept

    Everything is stated relative to kube-apiserver, the component nothing may be newer than:

    ComponentAllowed vs apiserverWith apiserver at 1.35
    kubeletup to 3 minors older, never newer1.32 – 1.35
    kube-proxyup to 3 minors older, never newer1.32 – 1.35
    controller-manager / schedulerup to 1 minor older, never newer1.34 – 1.35
    kubectl±1 minor (the only one allowed newer)1.34 – 1.36
    kubeadm (during upgrade)upgrade to kubeadm's own minor; cluster may be 1 behind itkubeadm 1.35 upgrades a 1.34 cluster

    And the iron rule: the apiserver moves one minor at a time — 1.33→1.35 means two full upgrade cycles, never one jump. All of this derives from one design constraint: during a rolling upgrade, old and new components coexist, so each pairing must be tested — and only adjacent (or near-adjacent) pairings are.

    Lab
    1. Audit your (deliberately old, if you did 2.2) lab against the table: 
      kubectl version | grep Server
kubectl get nodes -o custom-columns=NAME:.metadata.name,KUBELET:.status.nodeInfo.kubeletVersion
kubeadm version -o short
      Server Version: v1.34.x
NAME     KUBELET
cp       v1.34.x
node01   v1.34.x
node02   v1.34.x
v1.34.x
    2. Skew quiz — answer from the table, then check: (a) apiserver 1.35, kubelet 1.31: legal? (b) apiserver 1.34, kubectl 1.35: legal? (c) apiserver 1.33 upgrading to 1.35 directly: legal? (d) During 2.13's upgrade, cp's apiserver will be 1.35 while node01's kubelet is 1.34 — legal?
    Verify

    (a) no — 4 minors behind; (b) yes — kubectl may be 1 ahead; (c) no — one minor per hop; (d) yes — that window is exactly what the 3-minor kubelet allowance exists for.

    Gotchas

    The 3-minor kubelet allowance is a ceiling for emergencies, not a lifestyle: a kubelet 3 minors back can't use features its API objects may reference. In practice (and in the exam) you upgrade kubelets in the same session as the control plane. Also remember the kubelet is never allowed newer than the apiserver — upgrading node packages before the control plane is the classic ordering mistake.

    Docs

    Docs search: "Version Skew Policy" (it's under kubernetes.io/releases — the docs search finds it).

    2.13

    Upgrade the control plane for real: v1.34 → v1.35

    Why it matters

    The upgrade task appears on nearly every CKA. It's long, it's mechanical, and it's worth a lot of points to whoever has the sequence in muscle memory. You built the v1.34 fixture in 2.2 — now spend it.

    The concept

    The sequence, and why each step is where it is:

    repo file → new minor→upgrade kubeadm pkg→kubeadm upgrade plan→kubeadm upgrade apply v1.35.x→drain cp→upgrade kubelet+kubectl pkgs→restart kubelet→uncordon cp

    upgrade apply is mostly a manifest rewrite: it pulls new images, then replaces each static pod manifest in /etc/kubernetes/manifests one at a time, waiting for each new pod to go healthy before the next, rolling back automatically on failure. It also renews all certificates (2.7's side effect). What it never touches is the kubelet binary — that's yours, via apt, which is why the task has a second half at all. And the repo edit comes first because pkgs.k8s.io repos carry only their own minor (the 0.6 gotcha, now load-bearing).

    Lab
    1. On cp — repoint the repo and upgrade kubeadm only: 
      sudo sed -i 's/v1\.34/v1.35/' /etc/apt/sources.list.d/kubernetes.list

kubeadm version -o short
      v1.35.x
    2. Ask kubeadm what it would do: 
      sudo kubeadm upgrade plan
      [upgrade/versions] Cluster version: v1.34.x

Components that must be upgraded manually after you have upgraded the control plane with 'kubeadm upgrade apply':
COMPONENT   NODE     CURRENT    TARGET
kubelet     cp       v1.34.x    v1.35.x
kubelet     node01   v1.34.x    v1.35.x
...
You can now apply the upgrade by executing the following command:
        kubeadm upgrade apply v1.35.x

      Read the "manually" table — kubeadm is telling you, in writing, that the kubelet half is your job.

    3. Apply. Watch the one-manifest-at-a-time choreography in the output: 
      sudo kubeadm upgrade apply v1.35.x
      [upgrade/staticpods] Preparing for "kube-apiserver" upgrade

    4. Now the node half. Drain, upgrade packages, restart, uncordon: 
      kubectl drain cp --ignore-daemonsets

sudo apt-get install -y kubelet=1.35.x-1.1 kubectl=1.35.x-1.1

    Gotchas

    Three classic stumbles: forgetting the repo edit (apt then insists 1.34.x is the newest kubeadm — nothing to upgrade to); forgetting unhold (install silently keeps the held version); and running upgrade apply with the old kubeadm binary. If upgrade plan shows kubeadm version = cluster version, your package didn't actually upgrade — check those two causes before anything else.

    2.14

    Upgrade a worker — and time yourself

    Why it matters

    The worker procedure is the control-plane one minus the apply drama — but with a twist that trips people: where each command runs. Drain and uncordon talk to the API (run on cp); everything else changes the node itself (run on the worker).

    The concept

    kubeadm upgrade node on a worker is small but not skippable: it downloads the new cluster-wide kubelet configuration (from the kubelet-config ConfigMap the control-plane upgrade published) and rewrites the local config the kubelet will read on restart.     2.15

    What drain really does — and when it refuses

    Why it matters

    You've now typed drain four times on faith. It's also a standalone exam task ("make the node unschedulable and move workloads off it") — and its refusal messages are a mini-exam in themselves: each names its own fix.

    The concept

    drain = cordon (mark unschedulable — a taint-like flag on the node object) + evict every pod, via the Eviction API. Eviction is polite deletion: it respects PodDisruptionBudgets, retrying while a PDB says "not now". Three pod categories make drain balk, each with a flag that means "I understand the consequence":

    Pod typeWhy drain refusesOverrideConsequence
    DaemonSet podsthe DS controller would instantly recreate them (they tolerate the cordon)--ignore-daemonsetsthey simply stay — fine, they're per-node by design
    Pods with emptyDireviction destroys node-local data--delete-emptydir-datathat scratch data is gone
    Bare pods (no controller)nothing will recreate them anywhere--forcethe pod is gone, permanently
    Lab
    1. Stage all three refusals on node02 (check placement with -o wide; if something landed on node01, drain that instead): 
      kubectl run barepod --image=nginx --overrides='{"spec":{"nodeName":"node02"}}'
kubectl create deployment scratch --image=nginx -- sleep 1d
kubectl patch deployment scratch --patch '{"spec":{"template":{"spec":{
  "nodeSelector":{"kubernetes.io/hostname":"node02"},
  "volumes":[{"name":"scratch","emptyDir":{}}],
  "containers":[{"name":"nginx","image":"nginx","volumeMounts":[{"name":"scratch","mountPath":"/scratch"}]}]}}}}'
    2. Drain naked and collect the refusals: 
      kubectl drain node02
      node/node02 cordoned
error: unable to drain node "node02" due to error: [cannot delete DaemonSet-managed Pods (use --ignore-daemonsets to ignore): kube-system/kube-flannel-ds-..., kube-system/kube-proxy-...,
cannot delete cannot delete Pods that declare no controller (use --force to override): default/barepod,
cannot delete Pods with local storage (use --delete-emptydir-data to override): default/scratch-...]

      Note: it already cordoned! A failed drain leaves the node SchedulingDisabled — check kubectl get nodes.

    3. Acknowledge everything and drain for real: 
      kubectl drain node02 --ignore-daemonsets --delete-emptydir-data --force
kubectl get pods -o wide
      node/node02 drained
NAME             READY   STATUS    NODE
scratch-...      1/1     Pending   <none>    ← nodeSelector pins it to the cordoned node

      barepod is gone forever (that's what --force meant); the deployment pod was recreated by its controller but can't schedule until node02 returns — remove the nodeSelector or uncordon to release it.

    4. Watch a PDB stall a drain — the polite machinery visible: 
      kubectl uncordon node02
kubectl create deployment pdbdemo --image=nginx --replicas=2
kubectl create poddisruptionbudget pdbdemo --selector app=pdbdemo --min-available=2
kubectl drain node02 --ignore-daemonsets --delete-emptydir-data

      minAvailable=2 with replicas=2 allows zero disruptions — drain retries forever. Ctrl-C, then fix it like an operator: kubectl scale deployment pdbdemo --replicas=3 and watch the drain (rerun it) proceed. Clean up: delete deployment, PDB, and uncordon.

    Verify

    You can quote the three refusal categories with their flags from memory, and explain why a "stuck" drain with PDB retries isn't a bug.

    Gotchas

    uncordon does not move pods back — nothing does; rebalancing only happens as new scheduling decisions. And since a failed drain still cordons, an exam task that errored halfway leaves the node unschedulable — always end with a kubectl get nodes sanity check before moving on.

    etcd backup & restore — drill until reflexive

    2.16

    Take an etcd snapshot with the right certs, verify it

    Why it matters

    The etcd backup task is near-certain on the exam and pure mechanics: four flags, one endpoint, one file. All four flag values are readable off the etcd manifest, so there's nothing to memorize except where to look.

    The concept

    Two tools split the work: etcdctl talks to a live etcd over the network (that's why it needs endpoint + TLS client flags) and takes snapshots; etcdutl operates on files on disk (status, restore) and needs no certs at all. A snapshot is a point-in-time copy of etcd's bbolt keyspace — every API object in one file. etcd serves clients on :2379 over TLS from 2.5's etcd trust domain; its own server.crt/key pair doubles as a perfectly good client credential for our purposes, which is the standard exam trick.

    Lab
    1. Get matching tools. Check what your cluster runs, then pull that release's binaries (the etcd image tag tells you the version): 
      kubectl -n kube-system get pod etcd-cp -o jsonpath='{.spec.containers[0].image}{"\n"}'
ETCD_VER=v3.6.x    # ← match the tag you just saw
curl -L https://github.com/etcd-io/etcd/releases/download/${ETCD_VER}/etcd-${ETCD_VER}-linux-amd64.tar.gz \
  | tar xz --strip-components=1 -C /tmp etcd-${ETCD_VER}-linux-amd64/etcdctl etcd-${ETCD_VER}-linux-amd64/etcdutl
sudo mv /tmp/etcdctl /tmp/etcdutl /usr/local/bin/ && etcdctl version | head -1
      registry.k8s.io/etcd:3.6.x-0
etcdctl version: 3.6.x

      (On the exam the tools are pre-installed — this step is lab-only.)

    2. Read the four flag values off the manifest — this is the reflex to build: 
      sudo grep -E 'listen-client|trusted-ca|cert-file|key-file' /etc/kubernetes/manifests/etcd.yaml | grep -v peer
      - --cert-file=/etc/kubernetes/pki/etcd/server.crt
- --key-file=/etc/kubernetes/pki/etcd/server.key
- --listen-client-urls=https://127.0.0.1:2379,https://10.211.64.10:2379
- --trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt

      Mapping: --trusted-ca-file → your --cacert, --cert-file/--key-file → your --cert/--key, listen-client-urls → your --endpoints.

    3. Snapshot: 
      sudo mkdir -p /backup
sudo ETCDCTL_API=3 etcdctl snapshot save /backup/snap.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
      {"level":"info","msg":"created temporary db file","path":"/backup/snap.db.part"}
{"level":"info","msg":"fetching snapshot","endpoint":"https://127.0.0.1:2379"}
Snapshot saved at /backup/snap.db
    4. Verify with the offline tool — no certs needed, it's just a file now: 
      sudo etcdutl snapshot status /backup/snap.db -w table
      +----------+----------+------------+------------+
|   HASH   | REVISION | TOTAL KEYS | TOTAL SIZE |
+----------+----------+------------+------------+
| 8d2f45a1 |    28714 |       1152 |     4.9 MB |
+----------+----------+------------+------------+

      ~1000+ keys and a few MB is a healthy small cluster. A 0-key or KB-sized file means you snapshotted the wrong thing.

    Verify

    snapshot status shows a plausible key count — and you can name, without looking, where each of the four flag values comes from in the manifest.

    Gotchas

    Missing/wrong cert flags fail unhelpfully ("context deadline exceeded" — looks like a network problem, is an auth problem). Snapshot against exactly one endpoint — never a list. And note etcdctl snapshot status is deprecated in favor of etcdutl: using the offline tool for offline jobs is the habit the etcd project wants, and the one this site teaches.

    Docs

    Docs search: "Operating etcd clusters for Kubernetes" — the backup section has these exact flags; it's your copy-paste source in the exam.

    2.17

    Restore the snapshot and prove deleted objects come back

    Why it matters

    Restore is the half of the drill people fumble, because it's not the reverse of backup: no etcdctl, no endpoints, no certs — a file operation plus one manifest edit. Understanding that shape is what makes it fast.

    The concept

    Restore never touches the running etcd. etcdutl snapshot restore materializes a brand-new data directory on disk (with a fresh member identity) from the snapshot file. The running cluster only changes when you repoint the etcd static pod's hostPath volume at that new directory — then the kubelet restarts etcd on the restored data, the apiserver reconnects, and every controller reconciles the world back to the snapshot's state. Objects created after the snapshot vanish (their records don't exist in the restored keyspace); objects deleted after it return.

    Lab
    1. Plant evidence, snapshot, then diverge from it: 
      kubectl create deployment keepme --image=nginx
# take a fresh snapshot exactly as in 2.16, to /backup/snap.db
kubectl delete deployment keepme          # deleted AFTER the snapshot → restore brings it back
kubectl create deployment loseme --image=nginx   # created AFTER → restore erases it
kubectl get deploy
      NAME     READY   UP-TO-DATE   AVAILABLE
loseme   1/1     1            1
    2. Restore to a new directory (offline op — mind that it's etcdutl): 
      sudo etcdutl snapshot restore /backup/snap.db --data-dir /var/lib/etcd-restore
      ... "msg":"membership changed","added":[{"clusterID":"...","localMember":...}]
... snapshot restored at /var/lib/etcd-restore
    3. The one edit: repoint the hostPath volume in the etcd manifest. Find the block first: 
      sudo grep -B1 -A3 'hostPath' /etc/kubernetes/manifests/etcd.yaml | grep -A3 etcd-data
sudo vim /etc/kubernetes/manifests/etcd.yaml
        - hostPath:
      path: /var/lib/etcd        ← change this line to /var/lib/etcd-restore
      type: DirectoryOrCreate
    name: etcd-data

      Leave --data-dir=/var/lib/etcd and the volumeMount alone — inside the container nothing changes; only which host directory is mounted there.

    4. Wait out the restart (~60–90s; kubectl will refuse connections while etcd and then the apiserver bounce), then check the verdict: 
      kubectl get deploy
      NAME     READY   UP-TO-DATE   AVAILABLE
keepme   1/1     1            1          ← back from the dead
(loseme is gone)
    Verify

    keepme exists, loseme doesn't, and all kube-system pods are Running. That asymmetry is the proof the cluster is running on the restored keyspace.

    Gotchas

    The two classic fumbles: editing --data-dir instead of the hostPath (etcd then starts on an empty DirectoryOrCreate mount — cluster amnesia), and restoring into /var/lib/etcd over live data (etcdutl refuses if the dir exists; don't "fix" that with rm). If kubectl stays dead >2 minutes, kick the apiserver (move its manifest out/in per 2.7) — its etcd watches can wedge across a restore. Diagnosis for all of it: sudo crictl ps -a | grep etcd and journalctl -u kubelet.

    Docs

    Docs search: "Operating etcd clusters for Kubernetes" → "Restoring an etcd cluster". The manifest edit isn't spelled out there — that part is yours to own.

    2.18

    The full cycle from memory, twice more, under 10 minutes

    Why it matters

    Backup/restore is worth points only at speed and from memory — under exam adrenaline you'll have neither this page nor time to re-derive the flags. Spaced repetition is the entire lesson.

    Steps
    1. Print or save the etcd backup & restore card — but use it only to check your work, not to type from.
    2. Today, run the full cycle against a timer: plant a marker object → snapshot → delete marker + create decoy → restore → verify marker back, decoy gone (the 2.17 sequence). Everything from memory except the docs page you'd have in the exam. Log your time.
    3. Repeat once in ~3 days, and once more next week (put both in your calendar now). Same drill, cold start. Target: under 10 minutes end-to-end by the third run.
    4. Each run, also answer aloud: which tool for save vs restore? where do the cert flag values come from? which line in etcd.yaml changes? why does the decoy object vanish?
    Verify

    Three dated times in your notes, the last one under 10:00, with a marker-back/decoy-gone check passing each run.

    Gotchas

    Don't drill on a cluster state you care about — do it right after a snapshot of the VMs (0.8), or make the marker/decoy objects trivial. And vary the marker object type across runs (deployment, then configmap, then secret) so your verification habit doesn't overfit to one resource.

    Week 2 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. Recite the kubeadm init pipeline. Which phases only write files, and who actually starts the control plane?
      Answer

      preflight → certs → kubeconfig → etcd → control-plane → kubelet-start → wait → upload-config → mark-control-plane → bootstrap-token → kubelet-finalize → addons. certs/kubeconfig/etcd/control-plane only write files; the kubelet starts the pods from /etc/kubernetes/manifests.

    2. admin.conf vs super-admin.conf — what's the difference and why do both exist?
      Answer

      admin.conf is in group kubeadm:cluster-admins, powerful via an RBAC binding you can revoke. super-admin.conf is in system:masters, which the apiserver honors before RBAC — unrevokable, so it's split into a break-glass file.

    3. In a client certificate, what do CN and O map to in Kubernetes?
      Answer

      CN = username, O = group(s). There are no User objects — identity is whatever the cert (or token) asserts, and RBAC binds to those names.

    4. Name the three CAs kubeadm creates, and the one keypair in pki/ that isn't TLS at all.
      Answer

      Cluster CA, etcd CA, front-proxy CA — three isolated trust domains. sa.key/sa.pub is a bare RSA pair: sa.key signs ServiceAccount JWTs, sa.pub verifies them.

    5. kubectl fails with "x509: certificate is valid for cp, kubernetes…, not lb.example.com". Diagnosis and fix?
      Answer

      SAN mismatch — the apiserver cert doesn't include the name being dialed. Regenerate with the name (kubeadm init phase certs apiserver --apiserver-cert-extra-sans lb.example.com after removing the old pair) and restart the apiserver. Renewal alone fixes nothing — same SANs.

    6. You ran kubeadm certs renew all. Name the two follow-ups before the task is actually done.
      Answer

      Restart the four control-plane static pods (certs load at process start), and refresh any copies of admin.conf — your ~/.kube/config still holds the old client cert.

    7. Why doesn't kubeadm certs check-expiration list kubelet.conf?
      Answer

      The kubelet rotates its own client cert via the CSR API (rotateCertificates: true); kubelet.conf just points at the kubelet-client-current.pem symlink in /var/lib/kubelet/pki, so kubeadm has nothing to manage.

    8. What is the bootstrap token's one meaningful RBAC power, and what does --discovery-token-ca-cert-hash protect against?
      Answer

      As system:bootstrappers it can only create CSRs — the csrapprover then auto-approves node client certs. The CA hash pins the cluster's CA public key so the node can't be lured into joining (and sending credentials to) a rogue apiserver.

    9. apiserver is 1.35. Oldest legal kubelet? Legal kubectl range? And how do you get a 1.33 cluster to 1.35?
      Answer

      kubelet: 1.32 (3 minors older, never newer). kubectl: 1.34–1.36 (±1). 1.33→1.35 takes two full upgrade cycles — the apiserver moves one minor at a time.

    10. Recite the control-plane upgrade sequence, and name what kubeadm never upgrades.
      Answer

      Repo file to new minor → upgrade kubeadm pkg → upgrade plan → upgrade apply v1.x.y → drain → upgrade kubelet+kubectl pkgs → restart kubelet → uncordon. kubeadm never touches the kubelet binary — that's yours via apt.

    11. Drain refuses with three different complaints. Name the pod categories and their override flags.
      Answer

      DaemonSet pods → --ignore-daemonsets; pods with emptyDir → --delete-emptydir-data (data is destroyed); bare/controller-less pods → --force (gone forever). Bonus: a failed drain has already cordoned the node.

    12. etcdctl vs etcdutl — which does what, and why does restore need a manifest edit at all?
      Answer

      etcdctl = online, talks to live etcd (snapshot save, needs endpoint+certs). etcdutl = offline, operates on files (snapshot status/restore, no certs). Restore only writes a new data directory — the running cluster changes when you repoint the etcd pod's hostPath volume and the kubelet restarts it.
