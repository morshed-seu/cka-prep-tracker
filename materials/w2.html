    Week 2 — Installation, PKI & cluster lifecycle

    Last week you learned what the control plane is; this week you learn how it gets built, trusted, upgraded, and recovered. A lot of this week is certificates — files that prove "I am who I say I am" to another program — which sounds abstract until you've unwrapped a few by hand and realized they're just text. Every lesson here maps to a guaranteed exam-task family: init/join, certificate surgery, the upgrade dance, and the etcd backup/restore drill. By the weekend, each should run without notes.

    kubeadm init, dissected

    2.1

    List the kubeadm init phases from memory

    Why it matters

    kubeadm is the command-line tool that actually builds a Kubernetes control plane on a bare machine — instead of hand-writing certificates and config files yourself, you run one command and it works through a fixed, named checklist for you. Exam tasks say things like "regenerate the apiserver certificate" or "the join is failing at preflight". If you know init as a pipeline of named phases, every such task becomes "re-run one line of the checklist" instead of "rebuild the cluster".

    The concept

    kubeadm init looks like one big opaque action from the outside, but it isn't — it's a fixed sequence of named steps, and the genuinely useful part is that every single step can also be re-run on its own later, without redoing anything above it, via kubeadm init phase <name>:

    preflight→certs→kubeconfig→etcd→control-plane→kubelet-start→wait-control-plane→upload-config→mark-control-plane→bootstrap-token→kubelet-finalize→addon (CoreDNS, kube-proxy)

    Four details here are worth holding onto, because each one is the reason behind a real command you'll type later:

    • certs and kubeconfig only write files to disk, under /etc/kubernetes. Nothing is actually running yet — these two phases are pure paperwork.
    • etcd and control-plane drop static pod YAML files into /etc/kubernetes/manifests — a static pod is one the kubelet starts straight from a file on disk, with no apiserver involved (the full mechanics are week 1's lesson 1.2). kubeadm itself never launches a process; it just leaves the file there for the kubelet to notice and act on moments later.
    • upload-config saves the settings you gave kubeadm init — pod CIDR, version, and so on — as the kubeadm-config ConfigMap in kube-system. Later, kubeadm join and kubeadm upgrade read that stored copy, not your original command line — it's the cluster's own memory of how it was built, so you never have to retype the flags.
    • bootstrap-token mints the credential a new node uses to join (lesson 2.9); kubelet-finalize switches this node's own kubelet onto its rotated client certificate (lesson 2.8).
    Lab
    1. See the pipeline for yourself, straight from the binary's own help text — this is the same list as the diagram above, in kubeadm's own words: 
      kubeadm init --help | sed -n '/preflight/,/show-join-command/p'
      preflight                    Run pre-flight checks
certs                        Certificate generation
  /ca                          Generate the self-signed Kubernetes CA ...
kubeconfig                   Generate all kubeconfig files ...
etcd                         Generate static Pod manifest file for local etcd
control-plane                Generate all static Pod manifest files ...
kubelet-start                Write kubelet settings and (re)start the kubelet
...
addon                        Install required addons for passing conformance tests
    2. Now prove phases are independently re-runnable — regenerate just the apiserver's serving certificate (the cert it presents to prove its identity to clients) without touching anything else in the cluster: 
      sudo ls -l --time-style=+%H:%M /etc/kubernetes/pki/apiserver.crt
sudo kubeadm init phase certs apiserver
sudo ls -l --time-style=+%H:%M /etc/kubernetes/pki/apiserver.crt
      -rw-r--r-- 1 root root 1289 09:14 /etc/kubernetes/pki/apiserver.crt
[certs] Generating "apiserver" certificate and key
[certs] apiserver serving cert is signed for DNS names [cp kubernetes ...] and IPs [10.96.0.1 10.211.64.10]
-rw-r--r-- 1 root root 1289 10:02 /etc/kubernetes/pki/apiserver.crt   ← new timestamp

      Same CA, fresh leaf cert. This one command is the fix for half the certificate faults week 6 will throw at you.

    3. Read the cluster's memory of its own build: 
      kubectl -n kube-system get cm kubeadm-config -o yaml | head -20
      apiVersion: v1
data:
  ClusterConfiguration: |
    apiServer: {}
    ...
    kubernetesVersion: v1.35.x
    networking:
      podSubnet: 10.244.0.0/16

    Verify

    Close the page and recite the pipeline. Minimum bar: certs → kubeconfig → etcd/control-plane manifests → kubelet starts them → upload-config → token → addons, and why each early phase is "just files".

    Gotchas

    Here's the trap: a component reads its certificate files once, at process start, and then keeps that copy in memory — it never notices the file on disk changed underneath it. So regenerating a cert that a running component already loaded changes nothing until you restart that component. That's why cert tasks almost always pair with "restart the control-plane pods" (2.7) — regenerating alone silently does nothing. Also: older kubeadm versions ran kubelet-start at a different point in the sequence; don't be thrown by ordering differences in old blog posts — trust kubeadm init --help on the actual box you're working on.

    2.2

    Tear down and rebuild — and leave the rebuild one minor version old

    Why it matters

    Reading about kubeadm init is one thing; watching your own cluster disappear and come back under your own hands is what actually makes it stick. Rebuilding from scratch once turns init from magic into a checklist you've personally run. And there's a trick: rebuild the cluster one minor version behind (v1.34) on purpose — that gives you a real upgrade to perform in lessons 2.13–2.14 instead of a pretend one.

    The concept

    kubeadm reset is the undo button, but only for what kubeadm itself did: it stops the running control-plane containers and wipes /etc/kubernetes (manifests, PKI certificates, kubeconfigs) plus /var/lib/etcd, etcd's on-disk database. What it deliberately leaves alone is everything other tools set up on the node: the CNI (container network interface — the plugin that wires up pod networking) config in /etc/cni/net.d, the iptables/IPVS packet-forwarding rules that plugin installed, and your own $HOME/.kube/config credentials file all survive the reset untouched. Each one is a source of confusing, half-broken behavior on a fresh re-init if you forget it's still there.

    Lab
    1. Practice the small version first — remove just one worker and watch what reset actually says it's leaving behind. On node02: 
      sudo kubeadm reset -f
sudo rm -rf /etc/cni/net.d
      [reset] Stopping the kubelet service
[reset] Deleting contents of directories: [/etc/kubernetes/manifests /var/lib/kubelet /etc/kubernetes/pki]
The reset process does not clean CNI configuration. To do so, you must remove /etc/cni/net.d
The reset process does not reset or clean up iptables rules or IPVS tables.

      Read that output — reset itself tells you exactly what it leaves behind, in plain text, every time you run it. Then on cp, remove the node object (reset ran locally on node02, so it has no way to reach the apiserver and delete the node's entry for you):

      kubectl delete node node02
kubectl get nodes
      node "node02" deleted
NAME     STATUS   ROLES           AGE   VERSION
cp       Ready    control-plane   7d    v1.35.x
node01   Ready    <none>          7d    v1.35.x
    2. Rejoin it with a fresh token (the week-0 token expired long ago — the full lifecycle is 2.9's lesson): 
      kubeadm token create --print-join-command      # on cp; paste result on node02 with sudo
    3. Now the real exercise: a full rebuild, deliberately downgraded. Reset all three nodes as in step 1. Then on every node, point the package repo one minor version back and install the older binaries over the newer ones (this is lesson 0.6's repo pattern, run in reverse): 
      sudo sed -i 's/v1\.35/v1.34/' /etc/apt/sources.list.d/kubernetes.list
sudo apt-get update
sudo apt-mark unhold kubelet kubeadm kubectl
apt-cache madison kubeadm | head -3            # pick the newest 1.34 patch
sudo apt-get install -y --allow-downgrades kubeadm=1.34.x-1.1 kubelet=1.34.x-1.1 kubectl=1.34.x-1.1
sudo apt-mark hold kubelet kubeadm kubectl
    4. Re-init on cp exactly as in week 0 — but this time read every phase line as it scrolls past and tie each one back to the 2.1 pipeline you just memorized: 
      sudo kubeadm init --pod-network-cidr=10.244.0.0/16
mkdir -p $HOME/.kube && sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config && sudo chown $(id -u):$(id -g) $HOME/.kube/config
kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
    5. Join both workers, then confirm the deliberate time-warp: 
      kubectl get nodes
      NAME     STATUS   ROLES           AGE   VERSION
cp       Ready    control-plane   4m    v1.34.x
node01   Ready    <none>          1m    v1.34.x
node02   Ready    <none>          1m    v1.34.x
    6. Snapshot this state (multipass snapshot per 0.8, name it pre-upgrade) — it's your reusable upgrade-practice fixture.
    Verify

    All three nodes Ready at v1.34, and a pre-upgrade snapshot exists for each VM.

    Gotchas

    If a re-init fails preflight with "port 6443 in use" or "manifests already exist", read it literally: the previous reset didn't actually finish cleaning up, so just run it again. If pods on a freshly rebuilt cluster can't reach each other, stale CNI state is almost always the cause — either you skipped rm -rf /etc/cni/net.d, or old flannel iptables rules are still installed from before (a reboot clears those out if nothing else does).

    2.3

    Know the kubeconfigs kubeadm writes — and whose identity is inside each

    Why it matters

    A kubeconfig is just a file that bundles "which cluster to talk to" with "who I am when I talk to it" — kubectl reads one to know both. "Fix the broken kubeconfig" and "why is this component unauthorized?" tasks are solved by knowing that each file in /etc/kubernetes/*.conf is really a wrapped client certificate — a small proof-of-identity document — and by being able to unwrap one in ten seconds to see exactly whose identity it holds.

    The concept

    Here's the fact that makes the rest of this lesson make sense: Kubernetes has no User objects anywhere in its database. There's no table of accounts to look up. Instead, a "user" is simply whatever identity a request happens to carry proof of — and for a certificate, that proof follows one fixed rule: CN = username, O = group (the certificate's Subject field has a CommonName and an Organization, and Kubernetes reads those two as "who" and "which group they belong to"). Think of a client certificate as a notarized ID card: a trusted notary (the CA) has signed it, so anyone who trusts that notary trusts the name printed on the card without re-checking it themselves. kubeadm prints five of these ID cards when it builds the cluster, one per component:

    FileCN (user)O (group)Power
    admin.confkubernetes-adminkubeadm:cluster-adminscluster-admin via an RBAC binding — revocable
    super-admin.confkubernetes-super-adminsystem:mastersbypasses RBAC entirely in the authorizer — break-glass only
    controller-manager.confsystem:kube-controller-manager—exactly the RBAC that controller needs
    scheduler.confsystem:kube-scheduler—ditto, scheduler-sized
    kubelet.confsystem:node:<name>system:nodesgated by the Node authorizer

    The admin/super-admin split exists for a specific reason: system:masters is hardwired into the apiserver's authorization logic itself, not granted through the normal permissions system (RBAC, the keycard-style system week 7 covers in depth) — so no RBAC change can ever lock that group out, not even by an administrator trying to tighten security. That's a feature for a break-glass emergency credential and a liability for day-to-day use. So kubeadm splits it: day-to-day admin credentials live in a group you can govern through ordinary RBAC bindings, and the ungovernable one stays in a separate file meant to be kept offline, not used casually.

    Lab
    1. Unwrap the identity sitting inside admin.conf — the certificate data is base64-encoded text inside a YAML file, so this two-step decode-and-read is worth memorizing: 
      sudo grep client-certificate-data /etc/kubernetes/admin.conf \
  | awk '{print $2}' | base64 -d | openssl x509 -noout -subject -issuer
      subject=O = kubeadm:cluster-admins, CN = kubernetes-admin
issuer=CN = kubernetes
    2. Now compare it against super-admin.conf and the scheduler's kubeconfig — same decode, different files, to see the CN/O pattern repeat: 
      sudo grep client-certificate-data /etc/kubernetes/super-admin.conf \
  | awk '{print $2}' | base64 -d | openssl x509 -noout -subject
sudo grep client-certificate-data /etc/kubernetes/scheduler.conf \
  | awk '{print $2}' | base64 -d | openssl x509 -noout -subject
      subject=O = system:masters, CN = kubernetes-super-admin
subject=CN = system:kube-scheduler
    3. Prove the admin binding is ordinary, revocable RBAC and not some special hardcoded magic: 
      kubectl get clusterrolebinding kubeadm:cluster-admins -o wide
      NAME                     ROLE                        SUBJECTS
kubeadm:cluster-admins   ClusterRole/cluster-admin   Group/kubeadm:cluster-admins
    4. Finally, ask the API to tell you who it thinks you are — this works with any credential, and is the fastest sanity check when identity is in doubt: 
      kubectl auth whoami
      ATTRIBUTE   VALUE
Username    kubernetes-admin
Groups      [kubeadm:cluster-admins system:authenticated]
    Verify

    For each of the five files you can state the CN, whether it carries a group, and — the part that actually tests understanding, not memory — what would happen to each one's power if you deleted the kubeadm:cluster-admins binding while holding it.

    Gotchas

    Each kubeconfig also embeds a copy of the CA certificate itself (certificate-authority-data) — that's how kubectl knows which notary to trust when verifying the apiserver's own certificate. A corrupted or swapped-out CA there produces "x509: certificate signed by unknown authority" from kubectl even though the server is perfectly healthy — the client simply no longer trusts the notary that signed the server's cert. That's a favorite planted exam fault: diff the embedded CA against /etc/kubernetes/pki/ca.crt to catch it.

    Docs

    Docs search: "PKI certificates and requirements" — its table of certs and kubeconfigs is this lesson in official form.

    2.4

    The control-plane taint: why it exists, how to lift it

    Why it matters

    A taint is a marker on a node that says "don't schedule ordinary pods here unless they explicitly say they're okay with this" — it's a repel-by-default mechanism, the opposite of an attractive label. Exam clusters are often single-node, and on those the control-plane taint has usually already been removed so the one node can actually run workloads. Knowing this taint by its exact name tells you instantly why a pod refuses to schedule onto a control-plane node, and the add/remove syntax you'll use here is week-3 material you get to rehearse early, for free.

    The concept

    During kubeadm init's mark-control-plane phase (see 2.1), kubeadm labels the node and applies the taint node-role.kubernetes.io/control-plane:NoSchedule. A taint like this repels any pod that doesn't carry a matching toleration — a pod's explicit statement of "I'm fine running on a node with this taint." The point is to keep ordinary application workloads off the node running etcd and the apiserver, so a noisy pod can't starve the control plane of CPU or memory. Control-plane static pods aren't affected at all, because static pods bypass the scheduler entirely — nothing ever "schedules" them in the first place (1.2's mechanics). And DaemonSets like kube-proxy and Flannel carry a blanket toleration built in, which is why the control-plane node isn't actually empty even with the taint in place.

    Lab
    1. See the taint on the node, then watch it actually repel a pod in real time: 
      kubectl describe node cp | grep -i taint
kubectl run probe --image=nginx --overrides='{"spec":{"nodeSelector":{"kubernetes.io/hostname":"cp"}}}'
kubectl get pod probe
      Taints:  node-role.kubernetes.io/control-plane:NoSchedule
pod/probe created
NAME    READY   STATUS    RESTARTS   AGE
probe   0/1     Pending   0          20s
      kubectl describe pod probe | tail -3
      Warning  FailedScheduling  ... 0/3 nodes are available: 1 node(s) had untolerated taint {node-role.kubernetes.io/control-plane: },
2 node(s) didn't match Pod's node affinity/selector.
    2. Now lift the taint and watch the same pod finally schedule — note the trailing - on the command, which is kubectl's syntax for "remove this taint": 
      kubectl taint node cp node-role.kubernetes.io/control-plane:NoSchedule-
kubectl get pod probe -o wide
      node/cp untainted
NAME    READY   STATUS    NODE
probe   1/1     Running   cp
    3. Restore the guard rail (and clean up): 
      kubectl taint node cp node-role.kubernetes.io/control-plane:NoSchedule
kubectl delete pod probe --force --grace-period 0
      node/cp tainted
pod "probe" force deleted
    Verify

    kubectl describe node cp | grep -i taint shows the taint back in place, and the pending→running→gone story matches what you predicted before running each step.

    Gotchas

    Taints only ever affect new scheduling decisions, never pods that are already running — so removing a taint doesn't move anything onto the node, and re-adding it doesn't evict anything already there either. That's specific to this taint's effect, NoSchedule; there's a different effect, NoExecute, that does actively evict running pods, covered in week 3. Also worth knowing: the old taint name node-role.kubernetes.io/master no longer exists on current clusters — muscle-memory from old tutorials will "successfully" run a command that removes a taint that was never there, and quietly do nothing.

    Docs

    Docs search: "Taints and Tolerations".

    PKI — the certificate web

    2.5

    Map /etc/kubernetes/pki: three trust domains and one odd couple

    Why it matters

    /etc/kubernetes/pki is where every certificate kubeadm generated actually lives on disk — and it looks like a wall of similarly-named files until you know the organizing idea behind it. Certificate faults are only confusing if that directory is a blur. Once you see it as three independent notary offices (trust domains) plus one signing key that isn't a certificate at all, every filename explains itself — and week 6's cert faults become simple filename lookups instead of guesswork.

    The concept

    kubeadm sets up three separate trust domains — three independent notaries, each with its own signature, that don't recognize each other's signatures at all. A client certificate signed by one CA (Certificate Authority — the notary whose signature a trust domain honors) means nothing to a server that only trusts a different CA. That isolation between the three is deliberate design, not an accident: it means a leak or compromise in one domain doesn't automatically hand over the others.

    The diagram below sorts every certificate under pki/ into which of the three notaries signed it — plus a fourth box for the one file pair that isn't a certificate at all.

    Lab

      sudo grep -E 'client-ca|tls-cert|etcd-c|service-account-key' /etc/kubernetes/manifests/kube-apiserver.yaml

    Gotchas

    sa.key/sa.pub is the pair people mislabel in study groups, because it doesn't behave like the others: it never expires, has no CA, and openssl x509 simply refuses to read it — because it's not a certificate at all, just a bare RSA key pair (inspect it with openssl rsa -in sa.key -noout -text instead). If the apiserver and controller-manager somehow end up pointed at different sa keys, every ServiceAccount token in the entire cluster gets rejected at once — an outage signature specific enough to be worth filing away for week 6.

    2.6

    Inspect a certificate by hand: subject, issuer, SANs, expiry

    Why it matters

    A certificate is really just a small text document with four questions answered inside it, and openssl (the command-line tool for reading and manipulating certificates and keys) can print all four in one shot: who is this (Subject), who vouches for it (Issuer), which names is it allowed to be presented for (SANs — Subject Alternative Names), and until when is it valid (Validity). Every certificate fault you'll meet lives in one of those four fields, so being fluent in reading them is the actual skill here — memorizing specific error messages is not.

    Lab
    1. Here's the one command worth memorizing outright — it pulls all four fields at once: 
      sudo openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text -noout | \
  grep -A1 -E 'Issuer|Subject:|Not After|Alternative'
              Issuer: CN = kubernetes
        Validity
            Not Before: Jul 10 09:14:02 2026 GMT
            Not After : Jul 10 09:19:02 2027 GMT
        Subject: CN = kube-apiserver
            X509v3 Subject Alternative Name:
                DNS:cp, DNS:kubernetes, DNS:kubernetes.default, DNS:kubernetes.default.svc,
                DNS:kubernetes.default.svc.cluster.local, IP Address:10.96.0.1, IP Address:10.211.64.10
    2. Decode the SAN list — this is the part people skim past, but every single entry in it is a name some real client actually dials to reach this server:
      SANWho uses it
      kubernetes.default.svc[.cluster.local]in-cluster clients via the kubernetes Service
      10.96.0.1same Service's ClusterIP — always the first IP of the service CIDR
      10.211.64.10 / cpkubelets and your kubectl hitting the node directly

      The consequence follows directly: add a load balancer or a second hostname later without regenerating this cert to include it, and every client that tries to use that new name gets a TLS failure — the server's identity document simply doesn't cover that name.

    3. See that failure happen on purpose, without breaking anything real — dial a name that deliberately isn't in the SAN list: 
      curl --cacert /etc/kubernetes/pki/ca.crt --resolve fake-lb:6443:127.0.0.1 https://fake-lb:6443/version 2>&1 | tail -2
      curl: (60) SSL: certificate subject name does not match target host name 'fake-lb'

      kubectl phrases the exact same failure differently — as "x509: certificate is valid for cp, kubernetes, ... not fake-lb" — but it's the identical underlying problem either way: the SAN list and the name you dialed don't match. The fix follows directly from the diagnosis: either regenerate the cert with the extra name added (kubeadm init phase certs apiserver --apiserver-cert-extra-sans fake-lb, after deleting the old cert/key pair so kubeadm knows to make a new one), or simply dial a name that's already on the list.

    4. One more sweep worth doing: check expiry across the whole directory in one go, so you know at a glance which certs are close to their deadline: 
      sudo find /etc/kubernetes/pki -name '*.crt' \
  -exec sh -c 'echo -n "{} → "; openssl x509 -in {} -noout -enddate' \;
      /etc/kubernetes/pki/apiserver.crt → notAfter=Jul 10 09:19:02 2027 GMT
/etc/kubernetes/pki/ca.crt → notAfter=Jul  8 09:14:02 2036 GMT
...leaf certs 1 year out, CAs 10 years out
    Verify

    Without notes: which four openssl output fields answer "who / vouched by whom / for which names / until when", and what error signature a SAN mismatch produces in kubectl.

    Gotchas

    The single most useful habit this lesson builds: read the error message before reaching for a tool. "Certificate is valid for X, not Y" is a name problem, not an expiry problem — renewing it reflexively fixes nothing, because renewal keeps the exact same SAN list. An expired cert, by contrast, says something completely different: "certificate has expired or is not yet valid" — and that one actually is fixed by renewal, which is 2.7's lesson. Two different sentences, two different root causes, two different fixes.

    Docs

    Docs search: "PKI certificates and requirements". openssl itself isn't in the k8s docs — the invocation above has to live in your fingers.

    2.7

    Check expiry and renew everything with kubeadm certs

    Why it matters

    "The cluster's certificates expire soon — renew them" is about as literal as exam tasks get. The trap is the half everyone forgets: renewed files sitting on disk change absolutely nothing on their own, because — as you saw in 2.1's gotcha — a running process keeps using the certificate it loaded at start-up, not whatever's currently on disk. Renewal without a restart is renewal that does nothing.

    The concept

    kubeadm's day-to-day leaf certificates (the individual certs like apiserver.crt, as opposed to the CA certs that sign them) last 1 year by default; the CAs themselves last 10 years (both are tunable via certificateValidityPeriod / caCertificateValidityPeriod in the kubeadm config, if you ever need to change them). kubeadm certs renew re-signs every leaf cert using the existing CA — no new notary, just a fresh signature and a fresh expiry date — including the client certs that are embedded inside admin.conf, scheduler.conf, and controller-manager.conf, not just the standalone files under pki/. Two facts explain the entire workflow you're about to run: certificates are loaded once, at process start (so renewing always has to be followed by a restart), and kubeadm upgrade apply renews everything automatically as a side effect (so a cluster that gets upgraded regularly never actually hits the 1-year wall in practice).

    2.8

    Find the kubelet's own client cert and confirm rotation is on

    Why it matters

    Every other component in the cluster gets its certificate handed to it once by kubeadm and just keeps using that same file. The kubelet is the one exception — it's the single client that manages its own certificate lifecycle, quietly requesting itself a new one before the old one runs out. Knowing where that self-managed cert actually lives, and that kubelet.conf merely points at it rather than containing it, demystifies both the gap you noticed in 2.7's expiry dashboard and a whole class of node-authentication failures.

    The concept

    At join time the kubelet gets a short-lived starter certificate through the TLS bootstrap flow (2.10 walks through this step by step). From then on, before that certificate is due to expire, the kubelet requests itself a fresh one through the CSR (CertificateSigningRequest) API — that automatic behavior is what rotateCertificates: true in its config turns on. Each new certificate lands as a new file in /var/lib/kubelet/pki/, and a symlink named kubelet-client-current.pem always points at whichever one is newest. Here's the detail that makes rotation seamless: unlike the other kubeconfigs, kubelet.conf doesn't embed the certificate's actual bytes inline — it just references that symlink's path. So each time rotation happens and the symlink target changes, kubelet.conf itself never has to be rewritten at all.

    Lab
    1. Look at the rotation trail on any node — the directory holds every certificate the kubelet has ever issued itself, not just the current one: 
      sudo ls -l /var/lib/kubelet/pki/
      -rw------- 1 root root 1113 Jul 10 09:20 kubelet-client-2026-07-10-09-20-15.pem
lrwxrwxrwx 1 root root   59 Jul 10 09:20 kubelet-client-current.pem -> /var/lib/kubelet/pki/kubelet-client-2026-07-10-09-20-15.pem
-rw-r--r-- 1 root root 2258 Jul 10 09:14 kubelet.crt      ← self-signed serving cert (unrelated)
-rw------- 1 root root 1675 Jul 10 09:14 kubelet.key
    2. Confirm kubelet.conf really does just point at the symlink, rather than embedding certificate data the way admin.conf and the others do: 
      sudo grep client-cert /etc/kubernetes/kubelet.conf
          client-certificate: /var/lib/kubelet/pki/kubelet-client-current.pem
    3. Confirm rotation is actually switched on in the kubelet's own config file: 
      sudo grep -i rotate /var/lib/kubelet/config.yaml
      rotateCertificates: true
    4. Finally, read the identity baked into that certificate — it's the same CN=user / O=group pattern from 2.3, just node-flavored this time: 
      sudo openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -subject -enddate
      subject=O = system:nodes, CN = system:node:cp
notAfter=Jul 10 09:15:00 2027 GMT
    Verify

    You can answer: where does the kubelet's client cert live, what keeps it fresh, and why doesn't kubeadm certs check-expiration list kubelet.conf?

    Gotchas

    Rotation isn't magic — it still needs a working apiserver to actually file the CSR against. So a node that was powered off long enough for its certificate to expire past that point can't just quietly rejoin on its own; its kubelet logs show "part of the existing bootstrap client certificate is expired", and the fix is a manual re-bootstrap (docs: "kubelet client certificate rotation fails"). Also don't confuse the two similarly-named files in that directory: kubelet-client-* is the rotated client identity used when the kubelet talks to the apiserver, while kubelet.crt/kubelet.key is an unrelated, self-signed serving certificate the kubelet presents on its own :10250 API when something else (like the apiserver, for logs/exec) connects to it.

    Docs

    Docs search: "Configure Certificate Rotation for the Kubelet" and (troubleshooting) "Troubleshooting kubeadm".

    Joining nodes & TLS bootstrap

    2.9

    Mint a fresh join command and understand token lifecycle

    Why it matters

    A brand-new node has no certificate yet, so it can't authenticate the normal way — it needs a one-time, low-privilege credential just to get in the door and request its own certificate. That's a bootstrap token. "Join this node to the cluster" is a staple exam task, and the token minted back at init time is long gone by the time you sit the exam — you're expected to mint a fresh one from scratch. The reassuring part: a token is nothing exotic, it's stored as an ordinary Kubernetes Secret, which means its entire lifecycle — creation, inspection, expiry, deletion — is visible with plain kubectl.

    The concept

    A bootstrap token is a credential string shaped like id.secret (for example abcdef.0123456789abcdef), stored as a Secret of type bootstrap.kubernetes.io/token in the kube-system namespace, valid for 24 hours by default. Presenting it authenticates a joining kubelet as a member of the group system:bootstrappers — a group whose powers are deliberately kept tiny, as the next lesson shows in detail. Because it's just a Secret, deleting that Secret revokes the token instantly, with no separate revocation mechanism to learn.

    Lab

    Gotchas

    kubeadm token create --ttl 0 makes a token that never expires — convenient for a throwaway lab where you're joining nodes on and off all day, but an audit finding waiting to happen on anything real, since it means a permanently valid way for a new node to talk its way into the cluster. And if a join fails with "token is invalid or has expired", don't lose twenty minutes debugging the network first: minting a fresh token and retrying is a five-second test that rules out the single most common cause immediately.

    2.10

    Watch the TLS bootstrap flow a join actually performs

    Why it matters

    2.9 showed you the token; this lesson shows what actually happens with it, step by step, in the seconds between pasting the join command and the node turning Ready. Understanding this four-step handshake turns node-onboarding failures from voodoo into a pipeline you can bisect — check which step happened and which didn't. It's also the exact same CSR (CertificateSigningRequest) machinery week 7 reuses to onboard human users, so learning it once pays out twice.

    The concept
    token auth→group system:bootstrappers→kubelet creates CSR→csrapprover auto-approves→signed client cert→kubelet.conf written
    1. The joining kubelet connects and presents the bootstrap token — the apiserver authenticates it as a member of the group system:bootstrappers, a group whose RBAC grants exactly one meaningful power: create CertificateSigningRequests. Nothing more.
    2. Using that one power, it submits a CSR asking to be recognized as CN=system:node:<name>, O=system:nodes — the real node identity it wants — tagged with signer kubernetes.io/kube-apiserver-client-kubelet, which tells the apiserver what kind of certificate is being requested.
    3. The csrapproving controller, running inside kube-controller-manager, watches for exactly this pattern — a node-client CSR submitted by an identity that came in via a bootstrap token — and auto-approves it without a human involved. This narrow auto-approval is specific to that one signer and that one pattern; it's precisely why the human-onboarding CSRs in week 7 need someone to manually click approve instead.
    4. The kubelet downloads its newly-signed certificate into /var/lib/kubelet/pki/ and writes kubelet.conf to point at it. At this point the token's entire job is finished — from here on, 2.8's automatic rotation takes over maintaining this certificate for the rest of the node's life.

    One more piece worth noticing: the join command actually carries two credentials working in opposite directions. The token proves the node is legitimate to the cluster (steps 1–3 above); the --discovery-token-ca-cert-hash flag proves the cluster is legitimate to the node — it pins a fingerprint of the CA's public key, so a spoofed apiserver sitting on the network can't trick your kubelet into joining it and handing over credentials to an attacker.

    Lab
    1. Redo the 2.9 join, but this time keep a live watch running on cp before you paste the join command on node02, so you can see the CSR appear and get approved in real time: 
      kubectl get csr -w
      NAME        AGE   SIGNERNAME                                      REQUESTOR                  CONDITION
csr-x7k2p   0s    kubernetes.io/kube-apiserver-client-kubelet     system:bootstrap:dz54cd   Pending
csr-x7k2p   1s    kubernetes.io/kube-apiserver-client-kubelet     system:bootstrap:dz54cd    Approved,Issued

      Pending→Approved in about a second — that's step 3 from the concept above, happening live in front of you. Notice the REQUESTOR column shows the token's own identity (system:bootstrap:<id>), not a node name — that's not a display quirk, it's because the node identity doesn't exist yet at the moment of asking; it's literally the thing being requested.

    2. Inspect exactly what was requested versus what was actually granted: 
      kubectl get csr csr-x7k2p -o jsonpath='{.spec.username}{"  →  "}{.status.conditions[0].type}{"\n"}'

      system:bootstrap:dz54cd  →  Approved
subject=O = system:nodes, CN = system:node:node02
    3. Confirm the bootstrapper group's power really is as narrow as claimed — one power granted, everything else denied: 
      kubectl auth can-i create certificatesigningrequests --as system:bootstrap:dz54cd
kubectl auth can-i create pods --as system:bootstrap:dz54cd
      yes
no
    Verify

    Narrate the four steps and name which component acts in each (kubelet, apiserver authn, csrapproving controller, kubelet again) — plus what each of the two join-command credentials protects against.

    Gotchas

    A join that hangs at "waiting for the kubelet to perform the TLS Bootstrap" means the CSR round-trip stalled somewhere — and kubectl get csr tells you roughly where. A CSR that's stuck forever in Pending points at a problem later in the chain: the controller-manager is down, or its csrapprover isn't approving as expected. No CSR at all ever showing up points earlier: the token, network connectivity, or a preflight check failing on the node itself, before it ever got far enough to submit a request.

    2.11

    Know the two HA topologies and what the load balancer is for

    Why it matters

    HA (high availability) here means running multiple control-plane nodes instead of one, so the cluster survives losing any single machine. You won't build an HA cluster from scratch in the exam, but HA topology questions hide inside other tasks — "why does the kubeconfig point at a name that isn't any actual node?" — and the two ideas underneath it, quorum and controlPlaneEndpoint, explain flags and behaviors you'll run into throughout the exam even on a single-control-plane cluster.

    The concept

    kubeadm supports two blessed layouts, which differ in exactly one thing — where etcd runs relative to the apiservers:

    Stacked etcdExternal etcd
    Where etcd runson each control-plane node, beside the apiserverits own dedicated hosts
    Minimum sensible size3 CP nodes3 etcd + 2+ CP nodes
    Failure couplinglosing a CP node loses an etcd member toodecoupled — pricier, more machines
    How kubeadm builds itinit --control-plane-endpoint lb:6443 --upload-certs, then join --control-planesame, plus an etcd cluster you built separately

    Why the minimum sensible size is specifically 3, not 2: etcd needs a Raft quorum — a strict majority of members — to accept any write, the same consensus mechanics from week 1.9's "small committee" analogy. With 1 node, quorum is 1 (fine for learning, zero tolerance for failure). With 2 nodes, quorum is 2 — meaning either node failing halts every write, which is actually worse than running just 1. With 3 nodes, quorum is 2 — any single node can die and the cluster keeps accepting writes. That's week 1.9's arithmetic turning directly into a real purchasing and architecture decision here.

    The load balancer in the table exists for the apiservers, not for etcd. apiservers are stateless and can all serve the same request equally (often described as "hot-hot") — so clients just need one stable name that gets spread across whichever apiservers are healthy. That name is the controlPlaneEndpoint, and it gets baked into every kubeconfig and into the apiserver certificate's SAN list (the name-matching mechanism from 2.6) — which is exactly why you have to decide on it at init time: retrofitting HA onto a cluster that was inited without one means going back and doing certificate and kubeconfig surgery on every file, everywhere. etcd, by contrast, needs no load balancer at all — its only clients are the apiservers themselves, and they're configured with every member's address directly, so there's no "which one do I ask" ambiguity to solve.

    Lab
    1. Your lab cluster is the degenerate, non-HA case — single control-plane node, no load balancer — so look at what's simply absent compared to the HA picture above: 
      kubectl -n kube-system get cm kubeadm-config -o yaml | grep -i -A1 controlplane
kubectl config view --minify | grep server
      controlPlaneEndpoint: ""        ← empty: no HA was planned
    server: https://10.211.64.10:6443   ← clients pinned to one node's IP

      On an HA cluster, both of those lines would show the load balancer's name instead. This one empty string is the fork in the road that decides which of the two topologies you actually have.

    2. A short thought-drill to cement the quorum math — write the answers down before checking Verify: (a) With 3 stacked control-plane nodes, how many can you lose and still keep scheduling pods? (b) With 2? (c) Your kubeconfig points at lb.example.com, but TLS fails with a SAN mismatch error — what step was forgotten at init time?
    Verify

    Answers: (a) one — quorum 2 of 3 survives; (b) none — quorum 2 of 2 means any loss halts etcd writes; (c) lb.example.com wasn't in --apiserver-cert-extra-sans/controlPlaneEndpoint, so it's missing from the apiserver cert's SANs.

    Gotchas

    --upload-certs stashes the control-plane certificates as an encrypted Secret so that additional joining control-plane nodes can fetch them without you copying files by hand — but that Secret expires after 2 hours (re-run kubeadm init phase upload-certs --upload-certs to refresh it if the window closes). Don't confuse the two flavors of join: plain join brings up a worker, while join --control-plane --certificate-key ... brings up another control-plane node — same command, one extra flag, very different result.

    Docs

    Docs search: "Options for Highly Available Topology" and "Creating Highly Available Clusters with kubeadm".

    Cluster upgrades

    2.12

    Memorize the version skew rules

    Why it matters

    Version skew is the set of rules for how far out of sync different Kubernetes components are allowed to be with each other at the same time — because during any rolling upgrade, old and new versions of different components necessarily coexist for a while, and only certain combinations have actually been tested. These rules dictate the order of every upgrade you'll ever run: control plane first, kubelets after, one minor version at a time, never the reverse. Get the rules into memory now and the procedure in 2.13/2.14 stops feeling like a recipe to follow and starts feeling obvious.

    The concept

    Every rule below is stated relative to one anchor point: kube-apiserver, the one component nothing else in the cluster is ever allowed to be newer than:

    ComponentAllowed vs apiserverWith apiserver at 1.35
    kubeletup to 3 minors older, never newer1.32 – 1.35
    kube-proxyup to 3 minors older, never newer1.32 – 1.35
    controller-manager / schedulerup to 1 minor older, never newer1.34 – 1.35
    kubectl±1 minor (the only one allowed newer)1.34 – 1.36
    kubeadm (during upgrade)upgrade to kubeadm's own minor; cluster may be 1 behind itkubeadm 1.35 upgrades a 1.34 cluster

    And the one iron rule that ties everything together: the apiserver itself only ever moves one minor version at a time — going from 1.33 to 1.35 always means two separate, full upgrade cycles run back to back, never one jump straight across. All of this ultimately comes from a single design constraint mentioned above: during a rolling upgrade old and new components genuinely run side by side for a while, so every pairing that can occur has to have been tested by the Kubernetes project in advance — and only adjacent, or near-adjacent, version pairings ever are.

    Lab
    1. Audit your own lab cluster against the table above (it should be deliberately one minor behind, if you did the downgrade in 2.2): 
      kubectl version | grep Server

      Server Version: v1.34.x

    2. A short skew quiz — answer each from the table above before checking Verify: (a) apiserver 1.35, kubelet 1.31: legal? (b) apiserver 1.34, kubectl 1.35: legal? (c) apiserver 1.33 upgrading straight to 1.35 in one step: legal? (d) During 2.13's upgrade, cp's apiserver will briefly be 1.35 while node01's kubelet is still 1.34 — legal?
    Verify

    (a) no — 4 minors behind is one past the allowed 3; (b) yes — kubectl is the one component allowed to be a minor ahead; (c) no — the apiserver moves one minor per hop, so this needs two cycles; (d) yes — that in-between window is exactly the situation the 3-minor kubelet allowance exists to cover safely.

    Gotchas

    Don't mistake the 3-minor kubelet allowance for permission to leave kubelets behind indefinitely — it's a ceiling for emergencies and rolling-upgrade windows, not a lifestyle. A kubelet running 3 minors behind can't make use of features that newer API objects in the cluster might reference. In practice, and in the exam, you upgrade kubelets in the same session as the control plane, not weeks later. Also worth repeating because it's the classic ordering mistake: the kubelet is never allowed to be newer than the apiserver, so upgrading node packages before the control plane is the wrong order every time.

    Docs search: "Version Skew Policy" (it's under kubernetes.io/releases — the docs search finds it).

    2.13

    Upgrade the control plane for real: v1.34 → v1.35

    Why it matters

    The upgrade task appears on nearly every CKA exam. It's long, it's mechanical, and it's worth a lot of points specifically to whoever already has the sequence in muscle memory rather than reconstructing it under time pressure. You deliberately built a one-minor-behind fixture back in 2.2 — this is where you spend it on a real upgrade.

    The concept

    Here's the full sequence, and the reasoning for why each step has to be where it is:

    repo file → new minor→upgrade kubeadm pkg→kubeadm upgrade plan→kubeadm upgrade apply v1.35.x→drain cp→upgrade kubelet+kubectl pkgs→restart kubelet→uncordon cp

    At its core, upgrade apply is mostly a manifest rewrite: it pulls the new container images, then replaces each static pod manifest in /etc/kubernetes/manifests one at a time — waiting for each freshly-restarted pod to report healthy before moving to the next, and rolling everything back automatically if one fails. As a side effect it also renews every certificate, the same operation from 2.7. What it deliberately never touches is the kubelet binary itself on disk — that stays entirely your responsibility via apt, which is exactly why this task has a whole second half. And the repo edit has to come first, before anything else, because the pkgs.k8s.io package repositories only ever carry packages for one minor version each — the same fact from 0.6's setup lesson, now load-bearing in a real upgrade.

    Lab
    1. On cp — first repoint the package repo at the new minor version, then upgrade kubeadm only (not kubelet yet — that comes later, deliberately):

    Gotchas

    Three classic stumbles, all worth checking for by name if something feels stuck: forgetting the repo edit (apt then insists the old minor is still the newest kubeadm available — there's nothing for it to upgrade to); forgetting unhold (the install command runs but silently keeps the previously-held version, since apt-mark hold is designed to resist exactly this); and running upgrade apply with the old kubeadm binary because the package upgrade in step 1 didn't actually take. If upgrade plan shows the kubeadm version equal to the cluster version, that's your signal — the package didn't upgrade, so check those two causes before anything else.

    Docs

    Docs search: "Upgrading kubeadm clusters" — during the exam, keep this page open and copy each command; it includes the repo-edit step people forget.

    2.14

    Upgrade a worker — and time yourself

    Why it matters

    The worker upgrade procedure is essentially the control-plane one with the dramatic upgrade apply step removed — but it has its own twist that trips people up: where each command has to run. Drain and uncordon are API calls, so they only work from a machine with cluster credentials (cp); everything else changes files and packages on the node itself, so it has to run on the worker.

    The concept

    kubeadm upgrade node on a worker looks small, but it isn't skippable: it downloads the new cluster-wide kubelet configuration (published by the control-plane upgrade as the kubelet-config ConfigMap) and rewrites the local config file the kubelet will read the next time it restarts. There are no static pod manifests to rotate here at all, because workers don't run any control-plane components — that's the one piece of the control-plane procedure that simply doesn't apply.

    Lab
    1. Start a timer — the whole point of this lesson is speed, not just correctness. From cp, drain the worker so its pods move off before the kubelet restarts: 
      kubectl drain node01 --ignore-daemonsets
      node/node01 cordoned
evicting pod kube-system/coredns-...
node/node01 drained
    2. Switch to node01 itself for the rest of this — repoint the repo, upgrade kubeadm, then run the worker-specific upgrade step: 
      sudo sed -i 's/v1\.34/v1.35/' /etc/apt/sources.list.d/kubernetes.list
sudo apt-get update && sudo apt-mark unhold kubeadm
sudo apt-get install -y kubeadm=1.35.x-1.1 && sudo apt-mark hold kubeadm
sudo kubeadm upgrade node
      [upgrade] Reading configuration from the "kubeadm-config" ConfigMap
[upgrade] Backing up kubelet config file to /etc/kubernetes/tmp/...
[upgrade] The configuration for this node was successfully updated!
[upgrade] Now you should go ahead and upgrade the kubelet package
    3. Still on node01 — now the kubelet and kubectl packages themselves, then restart the kubelet so it picks up both the new binary and the config upgrade node just rewrote: 
      sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.35.x-1.1 kubectl=1.35.x-1.1
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet
    4. Back on cp — return the node to service by uncordoning it, and stop your timer: 
      kubectl uncordon node01
kubectl get nodes
      node/node01 uncordoned
NAME     STATUS   ROLES           AGE   VERSION
cp       Ready    control-plane   3h    v1.35.x
node01   Ready    <none>          3h    v1.35.x
node02   Ready    <none>          3h    v1.34.x
    5. Now repeat the entire thing for node02 without looking at this page — docs open if you need them, your notes closed. Target: under 10 minutes per worker. Log both times so you can see yourself improve.
    Verify

    All three nodes report v1.35.x, every pod in kube-system is Running, and your second worker was faster than your first — that improvement is the actual point of repeating this drill.

    Gotchas

    Wrong-terminal errors are the real time sink here, not the commands themselves: running drain on the worker fails outright (it has no kubeconfig with cluster credentials), while running kubeadm upgrade node on cp does something you didn't intend — it acts as a control-plane node upgrade instead. Say the rule out loud each time until it's automatic: API verbs run from cp, node surgery runs on the node itself. And forgetting uncordon at the end costs silent points on the real exam — grading checks whether the node is schedulable again, not just whether the version bumped.

    2.15

    What drain really does — and when it refuses

    Why it matters

    You've now typed drain four times on faith, trusting it to safely empty a node before an upgrade. It's also a standalone exam task in its own right ("make the node unschedulable and move workloads off it") — and its refusal messages are a mini-exam in themselves: each one names its own fix directly in the error text, if you read it.

    The concept

    drain is really two operations glued together: cordon (mark the node unschedulable — a flag on the node object, similar in spirit to a taint) plus evict every pod currently on it, through the Eviction API. Eviction is best understood as polite deletion — it respects any PodDisruptionBudget (a rule an application owner sets saying "never take down more than N of my pods at once"), retrying patiently while a PDB says "not right now" rather than forcing the issue. Three categories of pod make drain refuse outright rather than just wait, each with its own override flag that essentially means "yes, I understand the consequence, do it anyway":

    Pod typeWhy drain refusesOverrideConsequence
    DaemonSet podsthe DS controller would instantly recreate them (they tolerate the cordon)--ignore-daemonsetsthey simply stay — fine, they're per-node by design
    Pods with emptyDireviction destroys node-local data--delete-emptydir-datathat scratch data is gone
    Bare pods (no controller)nothing will recreate them anywhere--forcethe pod is gone, permanently
    Lab
    1. Stage all three refusal conditions on node02 at once, so you can see every error message in one drain attempt (check placement with -o wide; if something landed on node01 instead, drain that one): 
      kubectl run barepod --image=nginx --overrides='{"spec":{"nodeName":"node02"}}'
kubectl create deployment scratch --image=nginx -- sleep 1d
kubectl patch deployment scratch --patch '{"spec":{"template":{"spec":{
  "nodeSelector":{"kubernetes.io/hostname":"node02"},
  "volumes":[{"name":"scratch","emptyDir":{}}],
  "containers":[{"name":"nginx","image":"nginx","volumeMounts":[{"name":"scratch","mountPath":"/scratch"}]}]}}}}'
    2. Now drain with no flags at all, and read every refusal it hands back: 
      kubectl drain node02
      node/node02 cordoned
error: unable to drain node "node02" due to error: [cannot delete DaemonSet-managed Pods (use --ignore-daemonsets to ignore): kube-system/kube-flannel-ds-..., kube-system/kube-proxy-...,
cannot delete cannot delete Pods that declare no controller (use --force to override): default/barepod,
cannot delete Pods with local storage (use --delete-emptydir-data to override): default/scratch-...]

      Notice something important buried in that output: the node is already cordoned, even though the drain as a whole failed. Cordon and evict are separate steps run in sequence, and cordon succeeded before eviction hit its refusals — so a failed drain still leaves the node SchedulingDisabled behind. Always confirm with kubectl get nodes rather than assuming a failed drain changed nothing.

    3. Now acknowledge every consequence explicitly and drain for real: 
      kubectl drain node02 --ignore-daemonsets --delete-emptydir-data --force
kubectl get pods -o wide
      node/node02 drained
NAME             READY   STATUS    NODE
scratch-...      1/1     Pending   <none>    ← nodeSelector pins it to the cordoned node

      barepod is gone permanently — that's exactly what agreeing to --force meant, since nothing was going to recreate it. The deployment's pod, on the other hand, was immediately recreated by its controller (deployments always maintain their replica count), but it can't actually schedule anywhere until node02 comes back, because its nodeSelector pins it there specifically — remove the nodeSelector or uncordon the node to release it.

    4. One more thing worth seeing directly: watch a PDB actually stall a drain in progress, so the "polite" retry behavior described above becomes something you've watched happen rather than just read about: 
      kubectl uncordon node02
kubectl create deployment pdbdemo --image=nginx --replicas=2
kubectl create poddisruptionbudget pdbdemo --selector app=pdbdemo --min-available=2
kubectl drain node02 --ignore-daemonsets --delete-emptydir-data
      evicting pod default/pdbdemo-...
error when evicting pods/"pdbdemo-..." (will retry after 5s): Cannot evict pod as it would violate the pod's disruption budget.

      Setting minAvailable=2 on a deployment that only has 2 replicas means zero disruptions are ever permitted — so drain simply retries forever, exactly as the PDB instructs it to. Ctrl-C out of it, then fix the situation the way an operator actually would: give the PDB some slack by scaling up first (kubectl scale deployment pdbdemo --replicas=3), then re-run the drain and watch it proceed cleanly. Clean up afterward: delete the deployment and the PDB, and uncordon the node.

    Verify

    You can quote the three refusal categories with their override flags from memory, and explain in your own words why a "stuck" drain that's retrying against a PDB isn't a bug — it's the eviction API working exactly as designed.

    Gotchas

    uncordon does not move any pods back onto the node — nothing in Kubernetes actively rebalances existing workloads; a node only fills back up as new scheduling decisions happen to land there. And because a failed drain has already cordoned the node (as you saw above), an exam task that errors out halfway through still leaves that node unschedulable behind it — always finish with a kubectl get nodes sanity check before moving on to the next task.

    Docs

    Docs search: "Safely Drain a Node" and "Disruptions" (PDB concepts).

    2.16

    Take an etcd snapshot with the right certs, verify it

    Why it matters

    Recall etcd from week 1: it's the cluster's filing cabinet, the one place every object you've ever created actually lives. A snapshot is simply a photocopy of the entire cabinet's contents at one moment in time, saved to a single file. The etcd backup task is near-certain on the exam and is pure mechanics once you know it: four flags, one endpoint, one output file. All four flag values are readable directly off the etcd manifest already on disk, so there's genuinely nothing to memorize here except where to look.

    The concept

    Two different tools split this work between them: etcdctl talks to a live, running etcd over the network — which is exactly why it needs an endpoint address and TLS client credentials — and it's the tool that actually takes snapshots. etcdutl, by contrast, operates purely on files already sitting on disk (checking a snapshot's status, restoring one) and needs no network connection or certificates at all, because it never talks to anything running. A snapshot itself is a point-in-time copy of etcd's internal bbolt keyspace (bbolt being the on-disk B+tree structure etcd stores data in) — every API object in the whole cluster, frozen into one file. etcd serves its clients on port :2379 over TLS, using the separate etcd trust domain from 2.5's diagram; conveniently, its own server.crt/server.key pair also works as a valid client credential for talking to itself, which is the standard shortcut used both here and on the exam.

    Lab
    1. First, get matching tool versions. Check what version your cluster's etcd actually runs, then pull that exact release's binaries — the etcd container's image tag tells you the version to match: 
      kubectl -n kube-system get pod etcd-cp -o jsonpath='{.spec.containers[0].image}{"\n"}'
ETCD_VER=v3.6.x    # ← match the tag you just saw
curl -L https://github.com/etcd-io/etcd/releases/download/${ETCD_VER}/etcd-${ETCD_VER}-linux-amd64.tar.gz \
  | tar xz --strip-components=1 -C /tmp etcd-${ETCD_VER}-linux-amd64/etcdctl etcd-${ETCD_VER}-linux-amd64/etcdutl
sudo mv /tmp/etcdctl /tmp/etcdutl /usr/local/bin/ && etcdctl version | head -1
      registry.k8s.io/etcd:3.6.x-0
etcdctl version: 3.6.x

      (On the exam the tools are pre-installed — this step is lab-only.)

    2. Now read the four flag values you'll need straight off the manifest — building the reflex of looking here, instead of trying to remember values, is the actual skill this lesson teaches: 
      sudo grep -E 'listen-client|trusted-ca|cert-file|key-file' /etc/kubernetes/manifests/etcd.yaml | grep -v peer
      - --cert-file=/etc/kubernetes/pki/etcd/server.crt
- --key-file=/etc/kubernetes/pki/etcd/server.key
- --listen-client-urls=https://127.0.0.1:2379,https://10.211.64.10:2379
- --trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt

      The mapping in one line: --trusted-ca-file becomes your --cacert, --cert-file/--key-file become your --cert/--key, and listen-client-urls becomes your --endpoints. Same values, different flag names because they're two different tools.

    3. Take the snapshot: 
      sudo mkdir -p /backup
sudo ETCDCTL_API=3 etcdctl snapshot save /backup/snap.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
      {"level":"info","msg":"created temporary db file","path":"/backup/snap.db.part"}
{"level":"info","msg":"fetching snapshot","endpoint":"https://127.0.0.1:2379"}
Snapshot saved at /backup/snap.db
    4. Now verify the snapshot actually contains something real, using the offline tool — no certs needed here, because it's just a file on disk now, not a live connection: 
      sudo etcdutl snapshot status /backup/snap.db -w table
      +----------+----------+------------+------------+
|   HASH   | REVISION | TOTAL KEYS | TOTAL SIZE |
+----------+----------+------------+------------+
| 8d2f45a1 |    28714 |       1152 |     4.9 MB |
+----------+----------+------------+------------+

      ~1000+ keys and a few MB is a healthy small cluster. A 0-key or KB-sized file means you snapshotted the wrong thing.

    Verify

    snapshot status shows a plausible key count — and you can name, without looking, where each of the four flag values comes from in the manifest.

    Gotchas

    Missing or wrong certificate flags fail unhelpfully — you'll see "context deadline exceeded", which reads like a network timeout but is actually a TLS/auth failure in disguise (the connection is being silently rejected, not lost). Snapshot against exactly one endpoint, never a list of several — snapshotting more than one at once isn't meaningfully defined. And note that etcdctl snapshot status still technically works but is deprecated in favor of etcdutl: using the offline tool for offline jobs is the habit the etcd project wants you to build, and the one this lesson teaches.

    Docs

    Docs search: "Operating etcd clusters for Kubernetes" — the backup section has these exact flags; it's your copy-paste source in the exam.

    2.17

    Restore the snapshot and prove deleted objects come back

    Why it matters

    Restore is the half of the drill people fumble, precisely because it feels like it should be the reverse of backup and isn't. There's no etcdctl, no endpoint, no certs involved at all — it's a plain file operation followed by one manifest edit. Understanding that shape up front, before you touch a command, is what makes it fast instead of a guessing exercise.

    The concept

    The single fact that makes this whole lesson click: restore never touches the running etcd at all. etcdutl snapshot restore reads the snapshot file and materializes a brand-new data directory on disk from it — a fresh copy of the filing cabinet's contents, with a fresh internal member identity, sitting inert next to the live one. Nothing about the running cluster changes yet. The switch only happens when you repoint the etcd static pod's hostPath volume — the setting that says "which directory on this host's disk does the etcd container actually mount" — at that new directory. Once you do, the kubelet restarts etcd against the restored data, the apiserver reconnects to it, and every controller in the cluster reconciles the world back into line with whatever the snapshot said it should look like. The practical consequence: any object created after the snapshot was taken simply vanishes, because no record of it exists in the restored keyspace; any object deleted after the snapshot comes back, because the snapshot still remembers it.

    Lab
    1. Plant evidence first, take a snapshot, then deliberately diverge from it — this gives you something concrete to check restore against afterward: 
      kubectl create deployment keepme --image=nginx
# take a fresh snapshot exactly as in 2.16, to /backup/snap.db
kubectl delete deployment keepme          # deleted AFTER the snapshot → restore brings it back
kubectl create deployment loseme --image=nginx   # created AFTER → restore erases it
kubectl get deploy
      NAME     READY   UP-TO-DATE   AVAILABLE
loseme   1/1     1            1
    2. Restore into a new, empty directory — this is a pure offline file operation, so mind that the tool is etcdutl, not etcdctl: 
      sudo etcdutl snapshot restore /backup/snap.db --data-dir /var/lib/etcd-restore
      ... "msg":"membership changed","added":[{"clusterID":"...","localMember":...}]
... snapshot restored at /var/lib/etcd-restore
    3. Now the one edit that actually flips the switch: repoint the hostPath volume in the etcd manifest at the restored directory. Find the right block first, then edit it: 
      sudo grep -B1 -A3 'hostPath' /etc/kubernetes/manifests/etcd.yaml | grep -A3 etcd-data
sudo vim /etc/kubernetes/manifests/etcd.yaml
        - hostPath:
      path: /var/lib/etcd        ← change this line to /var/lib/etcd-restore
      type: DirectoryOrCreate
    name: etcd-data

      Leave --data-dir=/var/lib/etcd and the volumeMount itself alone — inside the container, nothing changes at all; only which directory on the host gets mounted into that same in-container path changes.

    4. Wait out the restart — expect roughly 60–90 seconds where kubectl refuses connections while etcd, and then the apiserver, both bounce — then check the verdict: 
      kubectl get deploy
      NAME     READY   UP-TO-DATE   AVAILABLE
keepme   1/1     1            1          ← back from the dead
(loseme is gone)
    Verify

    keepme exists again, loseme doesn't, and every kube-system pod is Running. That exact asymmetry — one back, one gone — is the proof the cluster is now genuinely running on the restored keyspace, not just that etcd restarted.

    Gotchas

    Two classic fumbles account for most restore failures: editing --data-dir instead of the hostPath (etcd then starts against an empty DirectoryOrCreate mount, since nothing actually points at your restored data — the cluster effectively forgets everything), and trying to restore directly into /var/lib/etcd on top of the live data (etcdutl refuses outright if the target directory already exists — resist the urge to "fix" that by deleting the live directory first). If kubectl stays unreachable for more than about 2 minutes, try kicking the apiserver by moving its manifest out and back in, the same trick from 2.7 — its watches against etcd can occasionally wedge across a restore. For diagnosing any of this: sudo crictl ps -a | grep etcd and journalctl -u kubelet.

    Docs

    Docs search: "Operating etcd clusters for Kubernetes" → "Restoring an etcd cluster". The manifest edit isn't spelled out there — that part is yours to own.

    Week 2 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. Recite the kubeadm init pipeline. Which phases only write files, and who actually starts the control plane?
      Answer

      preflight → certs → kubeconfig → etcd → control-plane → kubelet-start → wait → upload-config → mark-control-plane → bootstrap-token → kubelet-finalize → addons. certs/kubeconfig/etcd/control-plane only write files; the kubelet starts the pods from /etc/kubernetes/manifests.

    2. admin.conf vs super-admin.conf — what's the difference and why do both exist?
      Answer

      admin.conf is in group kubeadm:cluster-admins, powerful via an RBAC binding you can revoke. super-admin.conf is in system:masters, which the apiserver honors before RBAC — unrevokable, so it's split into a break-glass file.

    3. In a client certificate, what do CN and O map to in Kubernetes?
      Answer

      CN = username, O = group(s). There are no User objects — identity is whatever the cert (or token) asserts, and RBAC binds to those names.

    4. Name the three CAs kubeadm creates, and the one keypair in pki/ that isn't TLS at all.
      Answer

      Cluster CA, etcd CA, front-proxy CA — three isolated trust domains. sa.key/sa.pub is a bare RSA pair: sa.key signs ServiceAccount JWTs, sa.pub verifies them.

    5. kubectl fails with "x509: certificate is valid for cp, kubernetes…, not lb.example.com". Diagnosis and fix?
      Answer

      SAN mismatch — the apiserver cert doesn't include the name being dialed. Regenerate with the name (kubeadm init phase certs apiserver --apiserver-cert-extra-sans lb.example.com after removing the old pair) and restart the apiserver. Renewal alone fixes nothing — same SANs.

    6. You ran kubeadm certs renew all. Name the two follow-ups before the task is actually done.
      Answer

      Restart the four control-plane static pods (certs load at process start), and refresh any copies of admin.conf — your ~/.kube/config still holds the old client cert.

    7. Why doesn't kubeadm certs check-expiration list kubelet.conf?
      Answer

      The kubelet rotates its own client cert via the CSR API (rotateCertificates: true); kubelet.conf just points at the kubelet-client-current.pem symlink in /var/lib/kubelet/pki, so kubeadm has nothing to manage.

    8. What is the bootstrap token's one meaningful RBAC power, and what does --discovery-token-ca-cert-hash protect against?
      Answer

      As system:bootstrappers it can only create CSRs — the csrapprover then auto-approves node client certs. The CA hash pins the cluster's CA public key so the node can't be lured into joining (and sending credentials to) a rogue apiserver.

    9. apiserver is 1.35. Oldest legal kubelet? Legal kubectl range? And how do you get a 1.33 cluster to 1.35?
      Answer

      kubelet: 1.32 (3 minors older, never newer). kubectl: 1.34–1.36 (±1). 1.33→1.35 takes two full upgrade cycles — the apiserver moves one minor at a time.

    10. Recite the control-plane upgrade sequence, and name what kubeadm never upgrades.
      Answer

      Repo file to new minor → upgrade kubeadm pkg → upgrade plan → upgrade apply v1.x.y → drain → upgrade kubelet+kubectl pkgs → restart kubelet → uncordon. kubeadm never touches the kubelet binary — that's yours via apt.

    11. Drain refuses with three different complaints. Name the pod categories and their override flags.
      Answer

      DaemonSet pods → --ignore-daemonsets; pods with emptyDir → --delete-emptydir-data (data is destroyed); bare/controller-less pods → --force (gone forever). Bonus: a failed drain has already cordoned the node.

    12. etcdctl vs etcdutl — which does what, and why does restore need a manifest edit at all?
      Answer

      etcdctl = online, talks to live etcd (snapshot save, needs endpoint+certs). etcdutl = offline, operates on files (snapshot status/restore, no certs). Restore only writes a new data directory — the running cluster changes when you repoint the etcd pod's hostPath volume and the kubelet restarts it.
