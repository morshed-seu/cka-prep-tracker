    B7 — Isolation II: cgroups v2

    In B6 you gave a process a private view of the machine — its own process list, hostname, mounts, network. But a process with a private view can still burn every core and eat all the RAM: hiding the rest of the machine from it does nothing to stop it using the whole thing. That second half — putting a ceiling on how much CPU, memory, process slots and disk I/O a group of processes may consume — is the job of cgroups (control groups). Namespaces answer "what can this process see?"; cgroups answer "how much can this process use?". A container is both at once, and this module builds the "how much" half.

    We start with the model: why "limit" and "hide" are genuinely different powers, why Ubuntu 24.04 ships only cgroups v2 and what its single unified hierarchy under /sys/fs/cgroup looks like, and the two v2 rules that snag everyone. Then CPU: the difference between a relative share (cpu.weight, which only bites under contention) and a hard quota (cpu.max), and how to read the throttling counters that explain a service being slow on an idle box. Then memory, pids and io: the crucial gap between memory.high (throttle) and memory.max (kill), the OOM killer and the exit code 137 it leaves behind, plus fork-bomb containment. Finally, in practice: moving a live process into a cgroup by writing one number to one file, and seeing that systemd has been doing exactly this to every service all along. You finish by measuring a cage you built yourself.

    Checkpoints14 Hours~4 PrerequisitesB2, B6
    • State the difference that everything rests on: namespaces hide, cgroups limit — and why a container needs both.
    • Explain what cgroups v2 is, why Ubuntu 24.04 is v2-only, and how the unified hierarchy differs from v1's tree-per-controller.
    • Navigate /sys/fs/cgroup as a filesystem where a directory is a cgroup, and read its control files.
    • Apply the two v2 rules — hand a controller down with cgroup.subtree_control, and obey the no-internal-process rule — recognising each error on sight.
    • Distinguish cpu.weight (a proportional share under contention) from cpu.max (a hard quota per period), and set each.
    • Read throttling evidence in cpu.stat and explain why a throttled process is "slow for no reason" on an idle machine.
    • Distinguish memory.high (throttle and reclaim) from memory.max (kill), and read memory.current and memory.events.
    • Trigger an OOM kill deliberately and collect all three pieces of evidence: exit code 137, memory.events, and the dmesg verdict.
    • Contain a fork bomb with pids.max, and cap disk throughput with io.max.
    • Move a running process into a cgroup by writing its PID to cgroup.procs, and confirm membership from /proc/PID/cgroup.
    • Find a service's cgroup in systemctl status, explain slices and scopes, and set a live limit with systemctl set-property.
    • Measure a self-built CPU cage with time + cpu.stat, and capture an OOM kill in dmesg alongside the shell's 137.
    • Diagnose "slow but the CPU is idle" (throttling) and "dies with no error in its own logs" (the OOM killer) from evidence, not guesswork.
    • Connect all of this forward to Kubernetes requests, limits, QoS classes, and node-pressure eviction.

    B2 for processes, PIDs and signals — a cgroup is a set of processes, moving one means writing a PID, and the OOM killer works by sending SIGKILL (the un-catchable signal, hence exit 137); B6 because this module is the other half of what makes a container: B6 hid the machine, B7 rations it, and the mini project bolts a cgroup onto the container v0.1 script you wrote in B6.14. Everything runs on the throwaway sandbox VM (B0.1), never on the cluster — you will deliberately OOM-kill things and run a (contained) fork bomb, so take a snapshot (B0.9) first. Nothing extra to install: the cgroup filesystem is mounted at /sys/fs/cgroup by systemd on boot, and stress-ng (used for a couple of load tests) is one apt install away, noted where it is first needed.

    The model

    B7.1

    Namespaces hide, cgroups limit — the distinction everything rests on

    Why it matters

    If you take one idea from this whole module, take this one, because every later detail hangs off it. A namespace and a cgroup sound similar — both are "container plumbing" — but they grant two completely different powers, and confusing them is why people are surprised when a namespaced process still crashes the host. Get the distinction crisp now and the rest of the module is just filling in mechanism.

    The concept

    A namespace (B6) changes what a process can see. A cgroup changes what a process can use. Those are orthogonal:

    • A process in its own PID namespace cannot see the other processes on the machine — but nothing stops it spawning a thousand threads and pinning every core. Hiding the machine did not ration it.
    • A process in a CPU-capped cgroup cannot use more than, say, half a core — but if it shares the host's PID namespace it can still see and signal every process on the box. Rationing it did not hide anything.

    Neither implies the other, and a real container is deliberately both: a bundle of namespaces (the private view) wrapped around a cgroup (the resource ceiling). B6 built the first half; this module builds the second. Keep the pair as two independent dials — "what do I see?" and "how much may I use?" — and Kubernetes' resource model will read as obvious later.

    Back in the hotel (one hotel = one Linux machine): a namespace was giving a room its own private guest list, so it can't see the rest of the hotel. A cgroup is the room's metered power and water — the front desk sets a cap on how many kilowatts and litres that room may draw, no matter what it plugs in. A room can have a private guest list and no meter (sees little, can still overload the building's supply), or a meter and no privacy (capped draw, but reads everyone's guest list). A proper guest room has both. The meter is the cgroup.

    namespace= what you seeprocess list · hostname · net · mounts·cgroup= what you useCPU · memory · pids · io

    In one sentence each, state what a namespace controls and what a cgroup controls. (Namespace: what a process can see — its private view of one kernel resource. Cgroup: how much a set of processes can use — a metered cap on CPU/memory/pids/io.) Then: in the lab, why did being PID 1 in a private namespace do nothing to stop the CPU hog? (The PID namespace only hid the process list; it placed no limit on CPU, so the busy loop pinned a core regardless — isolation of view is not rationing of resources.)

    Gotchas

    The word "isolation" gets used for both, which is where the confusion breeds. Say "namespace isolation" (hiding) and "resource isolation" (limiting) and you will stay clear. Many real incidents are exactly this mix-up: a pod that is perfectly namespaced but has no memory limit will still trigger a node-wide out-of-memory event and take neighbours down with it — the isolation people assumed was there (resource) was never configured, only the isolation they weren't thinking about (namespace).

    A cgroup limits a group, not a single process — that is the whole point of the name. Every process in the cgroup, and every child they fork, counts against the same ceiling. This is why a CPU cap on a container applies to all the threads and subprocesses inside it together, not per-process, and why a fork bomb is contained by a pids cap on the group rather than a per-process limit.

    This is the exact split behind a pod spec. The namespaces you met in B6 are what make a pod's containers a private world; the resources block — requests and limits for cpu and memory — is what this module builds, and it becomes cgroup settings on the node. When you write resources.limits.memory: 256Mi, the kubelet creates a cgroup with memory.max set to 256Mi around your container. "Namespaces hide, cgroups limit" is literally "the pod's isolation vs. the pod's resources." A pod with no limits is namespaced but un-rationed — the noisy-neighbour and node-OOM problems trace straight back to this lesson.

    Requests, limits, and their cgroup enforcement are W3.16; the QoS classes that follow from them are W3.17.

    Docs

    man 7 cgroups is the master page — read its opening sections on what a cgroup and a controller are. man 7 namespaces for the other dial, to keep the two straight. The kernel's own Documentation/admin-guide/cgroup-v2.rst (online as docs.kernel.org/admin-guide/cgroup-v2.html) is the authoritative reference you will return to all module. In the exam docs, the Pod spec's resources field is the Kubernetes surface of the "limit" dial.

    B7.2

    cgroups v2: one unified hierarchy, and why 24.04 is v2-only

    Why it matters

    Search the web for "how to limit CPU with cgroups" and most of what you find will not work on your machine — the paths do not exist. That is not a broken tutorial; it is a version gap. cgroups exist in two incompatible generations, v1 and v2, and Ubuntu 24.04 ships v2 only. Knowing which one you are on, and why the file layout looks the way it does, saves you hours of chasing paths that were removed years ago.

    The concept

    cgroups v1 (2008) gave each controller its own separate hierarchy: one tree of directories for CPU, a different tree for memory, another for pids, and so on. A process could sit in one place in the CPU tree and a completely different place in the memory tree. It was flexible and, in practice, a confusing mess — a process's "cgroup" was really a dozen unrelated memberships, and controllers couldn't cooperate.

    cgroups v2 (stabilised around 2016) replaces all of that with one unified hierarchy: a single tree of directories under /sys/fs/cgroup, and a process lives at exactly one node in it. Every controller — cpu, memory, pids, io — acts at that same location. So "which cgroup is this process in?" finally has one answer, and controllers can coordinate (memory pressure can inform I/O decisions, etc.). Ubuntu 24.04, like every current mainstream distro, boots with v2 and no v1 hierarchy at all — this is called the "unified" setup. The single filesystem mounted at /sys/fs/cgroup is of type cgroup2fs, and that type string is how you confirm which world you are in.

    The practical upshot for reading tutorials: any instruction that mentions paths like /sys/fs/cgroup/cpu/ or /sys/fs/cgroup/memory/ (a controller name as a top-level directory) is v1 and simply has nowhere to land here. In v2 the controllers are not directories — they are features enabled within each directory, exposed as files named cpu.max, memory.max, and so on, right inside whatever cgroup you are looking at.

    Lab
    1. Confirm you are on v2. The filesystem type cgroup2fs is the definitive tell — if you see this, every v1 tutorial path is a dead end. 
      multipass shell sandbox
stat -fc %T /sys/fs/cgroup
      cgroup2fs                       ← v2 unified hierarchy (v1 would say tmpfs/cgroup)
    2. Look at the top of the single tree. There is no cpu/ or memory/ directory — instead every cgroup directory carries control files directly, and the top level lists which controllers exist. 
      ls /sys/fs/cgroup | head
cat /sys/fs/cgroup/cgroup.controllers
      cgroup.controllers  cgroup.procs  cgroup.subtree_control  ...
init.scope  system.slice  user.slice   ← systemd's carved-up tree (B7.12)
cpuset cpu io memory pids          ← the controllers available on this box
    3. See that there is exactly one hierarchy by reading your own membership. The line is short — one path — not one-per-controller as v1 would give. 
      cat /proc/self/cgroup
      0::/user.slice/user-1000.slice/session-3.scope
# the leading "0::" means v2 unified; a single path, one place in one tree
    Verify

    What single command proves you are on cgroups v2, and what output confirms it? (stat -fc %T /sys/fs/cgroup returning cgroup2fs.) Then: a blog says to write to /sys/fs/cgroup/cpu/mygroup/cpu.cfs_quota_us. Why will that fail on 24.04, and what is the v2 equivalent? (That is a v1 path — a per-controller top-level directory — which does not exist under the unified hierarchy; the v2 way is a cpu.max file inside a cgroup directory you make yourself.)

    Gotchas

    The 0:: prefix in /proc/self/cgroup is the v2 marker. On a v1 or hybrid system you would see many numbered lines, one per controller (1:name=systemd:…, 2:cpu:…). One 0:: line means pure unified v2 — which is what you have. If you ever see the multi-line form on some other machine, you are on a different (older) setup and this module's paths need translating.

    Do not go looking for a way to "switch back to v1" to follow an old guide — you would have to reboot with a kernel parameter (systemd.unified_cgroup_hierarchy=0) and you would be fighting the entire modern container stack, which assumes v2. Everything you actually care about (containerd, runc, Kubernetes) is v2-native now. Learn v2; treat v1 as history you only need to recognise in stale documentation.

    Kubernetes moved to cgroups v2 as the default and containerd/runc drive it directly, so the node your pods run on looks exactly like this sandbox: a single cgroup2fs hierarchy under /sys/fs/cgroup, with the kubelet carving out a subtree for pods. When you later inspect a running pod's limits on a node, you will cat the very same cpu.max / memory.max files this lesson introduces, just deeper in the tree under something like kubepods.slice. The v1-vs-v2 distinction also explains real upgrade notes: features like proper memory-limit accounting and pod-level PSI depend on the node being v2.

    The kubelet's cgroup layout underlies node-pressure eviction in W3.18 and the node internals in W6.

    Docs

    man 7 cgroups under "Cgroups version 2" contrasts the two generations. The kernel's cgroup-v2.rst (docs.kernel.org/admin-guide/cgroup-v2.html) is the reference for the unified hierarchy and every control file. man 5 proc documents /proc/[pid]/cgroup and the 0:: format. No exam-docs entry — the node's cgroup version is assumed, though the Kubernetes docs' "About cgroup v2" page is worth a skim.

    B7.3

    The tree is a filesystem: a directory is a cgroup

    Why it matters

    cgroups have a reputation for being arcane, but the interface is something you already know cold: it is just files and directories. There is no special cgroup command you must learn — you create a cgroup with mkdir, set a limit with echo > file, and read usage with cat. Seeing this once dissolves most of the mystery: the kernel exposes the entire mechanism as a filesystem, and everything else in this module is just knowing which file does what.

    The concept

    /sys/fs/cgroup is a pseudo-filesystem (like /proc from B3 — files backed by kernel data structures, not disk). In it, a directory is a cgroup. When you mkdir a new directory anywhere in the tree, the kernel instantly creates a cgroup and populates the directory with a standard set of control files — you do not create those files; they appear because the directory is a cgroup. The important ones you will meet:

    • cgroup.procs — the membership list: read it to see which PIDs are in this cgroup; write a PID to it to move a process in (B7.11).
    • cgroup.controllers — which controllers are available here (handed down from the parent).
    • cgroup.subtree_control — which controllers this cgroup passes down to its children (B7.4).
    • Per-controller files that appear once a controller is enabled: cpu.max, cpu.weight, cpu.stat, memory.current, memory.max, pids.max, and so on.

    Nesting is just subdirectories: a cgroup inside a cgroup is a child that inherits and can be limited more tightly than its parent, never more loosely. To destroy a cgroup you rmdir its directory — which only works once it is empty of processes and children. That "directories all the way down, control files at every level" shape is the unified hierarchy from B7.2; there is nothing else to it.

    mkdir /sys/fs/cgroup/demo→ kernel createsa cgroup· populated withcgroup.procs · cgroup.controllers · …
    Lab
    1. Create a cgroup by making a directory, and watch the control files appear on their own. You wrote none of these — the kernel did, because the directory is a cgroup. 
      multipass shell sandbox
sudo mkdir /sys/fs/cgroup/demo

cgroup.subtree_control  cgroup.threads  cpu.stat  ...   ← all auto-created
    2. Read the two files that matter most now: who is in it (nobody yet) and what controllers it has available (inherited from the root). 
      cat /sys/fs/cgroup/demo/cgroup.procs        # membership — empty
cat /sys/fs/cgroup/demo/cgroup.controllers  # what's available here
      cpuset cpu io memory pids                    ← available, not yet enabled for children
    3. Destroy it with rmdir — allowed because it is empty. (Try it while a process is inside and it refuses; that is the natural check.) 
      sudo rmdir /sys/fs/cgroup/demo
ls /sys/fs/cgroup/demo 2>&1 || echo 'gone'
      ls: cannot access '/sys/fs/cgroup/demo': No such file or directory
gone
    Verify

    How do you create a cgroup, and how do you delete one? (mkdir a directory under /sys/fs/cgroup to create; rmdir it to delete, which only succeeds when it holds no processes and no child cgroups.) Then: you never ran a command to create cgroup.procs inside demo — where did it come from? (The directory is a cgroup, so the kernel populates it with the standard control files automatically; the files are the kernel's interface, not something you author.)

    Gotchas

    These are not ordinary files and normal file intuitions can mislead. You cannot rm a control file, sizes are meaningless, and writes are validated by the kernel — write a malformed value and you get EINVAL ("write error: Invalid argument") rather than a file quietly holding junk. Treat every write as a syscall to the kernel (which it is), not as editing text: the file is a control knob with a strict accepted format, documented per-controller in cgroup-v2.rst.

    rmdir on a non-empty cgroup fails with EBUSY ("Device or resource busy"), and beginners read that as "permission problem" and reach for more sudo. It is not permissions — it is that the cgroup still contains processes or child cgroups. Move the processes out (or let them exit) and remove the children first; then rmdir succeeds. This is the same "a cgroup must be empty to delete" rule you will hit when a leftover container cgroup won't clean up.

    Every tool that manages container resources — runc, containerd, the kubelet, systemd — is ultimately doing mkdir/echo/rmdir against this filesystem on your behalf. When a pod starts, the kubelet (via the runtime) mkdirs a cgroup for it and writes its limits into these files; when the pod is deleted, the cgroup is rmdir'd — and the "cgroup busy, won't delete" failure you just saw is a real class of node bug where a leftover process keeps a pod's cgroup alive after the pod is gone. Being able to walk /sys/fs/cgroup by hand is exactly how you debug those on a node.

    You will read a live pod's cgroup files when inspecting node internals in W6; the kubelet's cgroup management sits under W3.16.

    Docs

    man 7 cgroups under "Creating cgroups and moving processes" for the filesystem interface. The kernel cgroup-v2.rst "Core Interface Files" section documents cgroup.procs, cgroup.controllers, and friends precisely. man 8 mount shows how cgroup2 is mounted (systemd does it at boot). No exam-docs entry — this is pure Linux mechanism the runtimes automate.

    B7.4

    The two v2 rules that trip everyone: subtree_control and no-internal-process

    Why it matters

    Two rules in cgroups v2 produce error messages that look like the kernel is broken, and everyone hits them the first time they try to set a limit on a nested cgroup. They are not bugs — they are the design that keeps the unified hierarchy coherent. Triggering both deliberately, and understanding why the kernel refuses, means you will recognise them instantly instead of losing an afternoon to "write error: No such file or directory" on a file that plainly exists.

    The concept

    Two rules govern how controllers and processes may be arranged in the v2 tree:

    • Controllers must be handed down explicitly. A controller being available in a cgroup (listed in cgroup.controllers) does not mean its files exist in the children. To give children a controller — so a child can have a cpu.max — the parent must enable it by writing +cpu to the parent's cgroup.subtree_control. Until you do, a child directory has no cpu.max file, and writing to the path fails with "No such file or directory" even though the cgroup exists. This is called delegation: a controller flows down the tree only where you route it.
    • The no-internal-process rule. A cgroup that has child cgroups and has controllers enabled for those children may not itself contain processes. Processes may live only in leaf cgroups (or in cgroups that have not enabled any child controllers). The reason is arithmetic fairness: a controller can't sanely divide CPU between "the processes sitting directly in this node" and "the child cgroups below it" at the same level — so v2 forbids the ambiguous arrangement. In practice you make a parent purely for structure and put processes only in its leaves.

    Put together, the normal shape is: enable controllers on a parent with +cpu +memory in its subtree_control, create leaf children, and place processes in the leaves. Fight either rule and the kernel stops you with a terse error — which, once you know these two rules, is self-explanatory.

    Lab
    1. Hit rule 1: make a parent and a child, then try to set a CPU limit on the child before delegating the controller. It fails because the child has no cpu.max yet. 
      multipass shell sandbox
sudo mkdir -p /sys/fs/cgroup/parent/child
ls /sys/fs/cgroup/parent/child | grep cpu.max || echo 'no cpu.max yet'
echo '50000 100000' | sudo tee /sys/fs/cgroup/parent/child/cpu.max
      no cpu.max yet
tee: /sys/fs/cgroup/parent/child/cpu.max: No such file or directory
    2. Fix it by delegating: enable cpu for the parent's children. Now the child grows a cpu.max and the same write succeeds. 
      echo '+cpu' | sudo tee /sys/fs/cgroup/parent/cgroup.subtree_control
ls /sys/fs/cgroup/parent/child | grep cpu.max        # it exists now
echo '50000 100000' | sudo tee /sys/fs/cgroup/parent/child/cpu.max
      cpu.max
50000 100000                    ← accepted, because the controller was handed down
    3. Hit rule 2: try to put a process directly in the parent, which now has a controller-enabled child. The kernel refuses — the parent may not hold processes. 
      echo $$ | sudo tee /sys/fs/cgroup/parent/cgroup.procs
      tee: /sys/fs/cgroup/parent/cgroup.procs: Device or resource busy
# processes may only sit in a leaf — move it into child instead:
# echo $$ | sudo tee /sys/fs/cgroup/parent/child/cgroup.procs   (would succeed)
    4. Clean up (empty the leaf first, then remove bottom-up). 
      sudo rmdir /sys/fs/cgroup/parent/child
sudo rmdir /sys/fs/cgroup/parent
      (both removed)
    Verify

    You made a child cgroup and writing to its cpu.max failed with "No such file or directory" — what is missing, and how do you fix it? (The parent has not delegated the cpu controller; write +cpu to the parent's cgroup.subtree_control and the child's cpu.max appears.) Then: why can't a cgroup that has controller-enabled children also hold processes directly? (The no-internal-process rule — a controller can't fairly split a resource between processes at a node and that node's child cgroups simultaneously, so processes are only allowed in leaves.)

    Gotchas

    The error for rule 1 — "No such file or directory" on cpu.max — is maddening because the cgroup directory clearly exists; it is the control file that doesn't, because the controller was never delegated. Train yourself to read that specific error as "the controller isn't enabled for children here" and check cat …/cgroup.subtree_control on the parent before anything else. It is the single most common first stumble with v2.

    The error for rule 2 — "Device or resource busy" when writing a PID to a non-leaf's cgroup.procs — is the same EBUSY you saw for a non-empty rmdir, which can mislead. Here it means "this cgroup has controller-enabled children, so it may not contain processes." The fix is always the same: put processes in a leaf. systemd's whole slice/scope layout (B7.12) exists precisely to obey these two rules automatically, which is why you rarely see them once systemd is doing the arranging.

    The kubelet builds its cgroup tree in exactly this shape and for exactly these reasons: a structural parent like kubepods.slice with controllers delegated down, and leaf cgroups per-pod and per-container where the processes actually live. The no-internal-process rule is why there is always that extra layer of nesting — you can't hang a container's processes directly off a node that also has child cgroups. When you read a node's cgroup tree and wonder why it is so deep, these two rules are the answer. Delegation is also how the kubelet hands controllers down to pods without letting a pod re-configure the node's.

    The pod/container cgroup nesting shows up when you trace resource enforcement in W3.16 and node internals in W6.

    Docs

    The kernel cgroup-v2.rst is essential here: "Enabling and Disabling" documents cgroup.subtree_control and the +cpu/-cpu syntax, and "No Internal Process Constraint" states rule 2 exactly. man 7 cgroups under "Cgroups v2 delegation" covers handing subtrees to a less-privileged manager. No exam-docs entry; this is the mechanism the kubelet automates.

    CPU

    B7.5

    cpu.weight: a proportional share that only bites under contention

    Why it matters

    There are two completely different ways to control CPU, and mixing them up leads to wrong expectations every time. The first, cpu.weight, is a relative priority: it decides how the CPU is divided when there is not enough to go around, and does nothing at all when there is plenty. Understanding "only matters under contention" up front stops you from setting a weight, seeing your process still use 100% of an idle core, and concluding the limit "didn't work" — it worked exactly as designed.

    The concept

    cpu.weight assigns a cgroup a share of CPU relative to its siblings, on a scale from 1 to 10000 with a default of 100. When several cgroups all want CPU at the same time and there isn't enough for everyone, the kernel's scheduler divides the available CPU time in proportion to their weights: two cgroups at weights 100 and 200 get roughly a 1:2 split — the 200 one gets twice as much. But this is purely a tie-break under scarcity. If only one cgroup wants the CPU, it gets all of it regardless of its weight; a weight of 1 will still use an entire idle machine if nothing else is competing. Weight sets the ratio of a contested resource, never a ceiling.

    This is the "fair share" model: nobody is capped, everybody is guaranteed a proportional slice when it's crowded. It maps onto the hotel meter loosely — think of it as priority on a shared circuit: when the circuit is near capacity, the room with the higher-priority allocation draws more, but when the circuit is quiet, any room can pull as much as it likes. Weight is about who yields to whom, not about a hard limit. The complementary tool — an actual ceiling that applies even on an idle box — is cpu.max, the very next lesson; the two are meant to be used together (a guaranteed share and a hard cap).

    2. Pin both to a single CPU so they must compete (otherwise each gets its own core and there is no contention to divide). Then launch one busy loop in each cgroup and let them fight. 
      # confine the whole demo to CPU 0 so the two must share one core
echo 0 | sudo tee /sys/fs/cgroup/cpudemo/cpuset.cpus 2>/dev/null || true
# start a hog in each leaf: move a subshell in, then spin
sudo bash -c 'echo $$ > /sys/fs/cgroup/cpudemo/a/cgroup.procs; exec yes >/dev/null' &
sudo bash -c 'echo $$ > /sys/fs/cgroup/cpudemo/b/cgroup.procs; exec yes >/dev/null' &
sleep 3
cat /sys/fs/cgroup/cpudemo/{a,b}/cpu.stat | grep usage_usec
      usage_usec 1010000               ← a: ~1.0s of CPU over the window
usage_usec 2020000               ← b: ~2.0s — twice as much, matching 100:200
    3. Now prove the "only under contention" half: kill a's hog and watch b take the whole core despite its weight being unchanged. Then clean up. 
      kill %1                            # stop a's hog
sleep 2
top -bn1 | grep -m1 yes           # b now near 100% of the shared core
kill %2
sudo rmdir /sys/fs/cgroup/cpudemo/{a,b} /sys/fs/cgroup/cpudemo
      ... 99.0 yes                 ← b uses everything now — weight caps nothing
    Verify

    Two cgroups have cpu.weight 100 and 300 and both run flat-out on a one-core box — roughly how is the CPU split? (1:3 — the 300-weight cgroup gets three times the CPU of the 100-weight one, about 75% vs 25%.) Then: you give a cgroup cpu.weight 1 and it still uses a whole idle CPU — is it broken? (No — weight only divides CPU under contention; with nothing else competing, even weight 1 gets everything. Weight is a ratio, not a ceiling.)

    Gotchas

    The number 1 vs 300 might read like "1% vs 300%," but weights are relative to the actual competitors, not percentages. A weight of 100 means nothing on its own; it only means something next to its siblings' weights. Two cgroups at 50 and 100 behave identically to two at 5000 and 10000 — same 1:2 ratio. Always reason about the ratio between competing cgroups, never a single weight in isolation.

    Because weight does nothing on an idle machine, it is the wrong tool when you need a guarantee that a process cannot exceed X — for that you need cpu.max (B7.6). A classic mistake is setting only a weight to "limit" a batch job, then being shocked when it consumes the whole node at 3am because nothing else happened to be running. Weight guarantees a floor under contention; it never imposes a ceiling.

    This is precisely what a Kubernetes CPU request becomes. When you set resources.requests.cpu: 500m, the kubelet translates it into a cpu.weight proportional to that request — so under node CPU pressure, pods get CPU in proportion to their requests, exactly the 1:2 behaviour you just measured. And it explains a frequent surprise: a pod with a small CPU request can still burst to use a whole idle node, because requests set weight (a share under contention), not a cap. The cap is the limit, which is B7.6's cpu.max.

    CPU requests-as-weight and limits-as-quota are the heart of W3.16; the QoS classes built on requests vs limits are W3.17.

    Docs

    The kernel cgroup-v2.rst "CPU" section documents cpu.weight (range 1–10000, default 100) and the weight-based proportional model. man 7 cgroups under "cpu" for the controller overview. man 1 chrt / man 7 sched for the underlying scheduler concepts. In the exam docs, a pod's resources.requests.cpu is the Kubernetes surface of weight.

    B7.6

    cpu.max: a hard quota per period, even on an idle box

    Why it matters

    This is the CPU control most people actually want: a genuine ceiling. cpu.max says "this cgroup may use at most half a CPU" and means it — even if the rest of the machine is completely idle, the process is held to its quota. It is the tool behind "limit this container to 0.5 cores," and unlike weight it produces a visible, measurable cap you can watch with top. Seeing it work turns the abstract idea of a CPU limit into a number you can point at.

    The concept

    cpu.max holds two numbers: QUOTA PERIOD, both in microseconds. It means "in every PERIOD microseconds of wall-clock time, this cgroup may run on the CPU for at most QUOTA microseconds." The default period is 100000 (100 ms). So:

    • 50000 100000 = 50ms of CPU per 100ms = half of one CPU (50%).
    • 100000 100000 = a full single CPU.
    • 200000 100000 = up to two CPUs' worth (quota can exceed one period on a multi-core box).
    • max 100000 = unlimited (the literal word max means no quota — the default).

    The enforcement is blunt and worth picturing: within each 100ms window, the cgroup's processes run until they have used their quota, and then the kernel throttles them — stops scheduling them on the CPU — until the next window begins. So a process capped at 50% that wants to run flat-out will run for ~50ms, sit frozen for ~50ms, run again, and so on. Averaged out it uses half a core; moment to moment it is being repeatedly paused. This is a hard ceiling and, crucially, it applies even when the machine is otherwise idle — the quota is absolute, not relative to demand. That "run, then get frozen till the window rolls over" behaviour is exactly what produces the throttling latency you will measure in B7.7.

    cpu.max = 50000 100000· each 100ms:run 50ms→throttled 50ms→ repeat
    Lab
    1. Create a capped cgroup and set it to half a CPU. Delegate cpu, make a leaf, and write the quota. 
      multipass shell sandbox
sudo mkdir -p /sys/fs/cgroup/cap/leaf
echo '+cpu' | sudo tee /sys/fs/cgroup/cap/cgroup.subtree_control
echo '50000 100000' | sudo tee /sys/fs/cgroup/cap/leaf/cpu.max
      +cpu
50000 100000                   ← half of one CPU per 100ms window
    2. Put a busy loop into the leaf and watch top pin it at ~50% — on an otherwise idle VM, which is the whole point. 
      sudo bash -c 'echo $$ > /sys/fs/cgroup/cap/leaf/cgroup.procs; exec yes >/dev/null' &
sleep 2
top -bn1 | grep -m1 yes
      ... 50.0 yes                 ← held to 50% even though the machine is idle
    3. Contrast with no cap. Move the same hog to a leaf with cpu.max = max (unlimited) and it jumps to a full core. Then clean up. 
      sudo mkdir /sys/fs/cgroup/cap/free
echo 'max 100000' | sudo tee /sys/fs/cgroup/cap/free/cpu.max
HOG=$(cat /sys/fs/cgroup/cap/leaf/cgroup.procs)
echo $HOG | sudo tee /sys/fs/cgroup/cap/free/cgroup.procs   # move it
sleep 2; top -bn1 | grep -m1 yes
kill $HOG
sudo rmdir /sys/fs/cgroup/cap/{leaf,free} /sys/fs/cgroup/cap
      ... 99.7 yes                 ← uncapped: a whole core
    Verify

    Translate cpu.max values: what fraction of a CPU is 25000 100000, and how do you express "up to 1.5 CPUs"? (25000/100000 = a quarter of one CPU; 1.5 CPUs is 150000 100000.) Then: why does a cpu.max-capped process stay at 50% even when the whole machine is idle, whereas a cpu.weight process would take everything? (cpu.max is an absolute quota per time window — throttled once spent regardless of demand — while weight only divides CPU under contention.)

    Gotchas

    The order is QUOTA PERIOD, quota first, and both are microseconds — a very common slip is writing them backwards or in milliseconds. 50000 100000 is 50ms per 100ms; 100000 50000 would be a full two CPUs (100ms of runtime allowed every 50ms), the opposite of what you probably meant. When in doubt, read it as "quota ÷ period = fraction of one CPU" and sanity-check the fraction.

    A too-small quota with a normal period causes severe latency even at modest average CPU, because the process keeps hitting the throttle wall and freezing until the next window. A latency-sensitive service capped at, say, 10% will feel far worse than "10% slow" — it stalls for up to 90ms at a stretch. This is the single most important cause of the "throttled and mysteriously slow" symptom in B7.7 and a real Kubernetes footgun with aggressive CPU limits. The cap is honest; the pauses are what hurt.

    cpu.max is exactly what a Kubernetes CPU limit sets. resources.limits.cpu: 500m becomes cpu.max = 50000 100000 on the container's cgroup — a hard half-core ceiling enforced by throttling. This is why CPU limits are famous for causing latency: a spiky, latency-sensitive app that occasionally needs a burst gets throttled at the limit and stalls until the next 100ms window, showing terrible p99 latency on a node whose CPU graphs look calm. Understanding cpu.max's window-and-throttle mechanic is what lets you reason about when to set a CPU limit and when to leave it off and rely on requests (weight) alone.

    CPU limits-as-quota and the throttling trade-off are core to W3.16; you'll see the latency symptom again in troubleshooting, W6.

    Docs

    The kernel cgroup-v2.rst "CPU" section documents cpu.max (the MAX PERIOD format, default period 100000, the literal max). man 7 cgroups under "cpu". For the throttling statistics this produces, read on to cpu.stat in B7.7. In the exam docs, resources.limits.cpu is the Kubernetes surface, and the "CPU limits and throttling" discussions in the docs are worth knowing exist.

    B7.7

    cpu.stat: reading throttling, the "slow on an idle box" evidence

    Why it matters

    The most baffling performance problem in a containerised world is a service that is slow while every host CPU graph says the machine is nearly idle. The cause is almost always CPU throttling — the process is capped and spends its time frozen against the quota wall — and the evidence is in one file, cpu.stat, that host-level tools never show you. Learning to read it turns an unfalsifiable "it feels slow" into a hard number, and it is the first half of this module's debugging drill.

    The concept

    Every cgroup with a CPU controller keeps a running tally in cpu.stat. Three lines tell the throttling story:

    • nr_periods — how many enforcement windows have elapsed for this cgroup.
    • nr_throttled — in how many of those windows the cgroup hit its quota and got throttled. A high ratio of nr_throttled to nr_periods means "this cgroup is constantly running out of quota."
    • throttled_usec — the total microseconds the cgroup spent frozen, waiting for a window to roll over. This is pure added latency: time the process wanted to run but wasn't allowed to.

    Here is why it explains the paradox. Host CPU metrics measure the machine's total utilisation. A single throttled cgroup can be pinned at its small quota — using very little of the machine — while its own processes stall for tens of milliseconds at a time waiting for quota. The machine is idle because the process is being held back, not despite it. So the symptom "low host CPU, terrible service latency" is not a contradiction; it is the exact signature of throttling, and throttled_usec climbing is the proof. Anyone watching only top or a node dashboard will never see it — you have to look inside the cgroup.

    host CPU: 5% idle· butnr_throttled climbing+throttled_usec rising→the app is starving on its own quota
    Lab
    1. Re-create a tightly capped cgroup — a small quota makes throttling obvious — and drop a busy loop in it. 
      multipass shell sandbox
sudo mkdir -p /sys/fs/cgroup/thr
echo '+cpu' | sudo tee /sys/fs/cgroup/thr/cgroup.subtree_control >/dev/null
sudo mkdir /sys/fs/cgroup/thr/leaf
echo '10000 100000' | sudo tee /sys/fs/cgroup/thr/leaf/cpu.max   # 10% cap — heavy throttling
sudo bash -c 'echo $$ > /sys/fs/cgroup/thr/leaf/cgroup.procs; exec yes >/dev/null' &
      10000 100000                     ← a 10% cap: the hog will spend ~90% of each window frozen
    2. Read cpu.stat twice, a few seconds apart, and watch the throttle counters climb while the host stays nearly idle. This is the whole lesson in two reads. 
      cat /sys/fs/cgroup/thr/leaf/cpu.stat
sleep 5
echo '--- 5s later ---'
cat /sys/fs/cgroup/thr/leaf/cpu.stat
top -bn1 | grep '%Cpu'
      nr_periods 40
nr_throttled 39                  ← throttled in nearly every window
throttled_usec 3510000
--- 5s later ---
nr_periods 90
nr_throttled 89              ← still throttled almost every window
throttled_usec 7960000      ← +4.4s of pure frozen time in 5s of wall-clock
%Cpu(s): 9.8 us, ...        ← host barely busy — yet the app is starving
    3. Confirm the fix: raise the cap and watch nr_throttled stop advancing. Then clean up. 
      echo '100000 100000' | sudo tee /sys/fs/cgroup/thr/leaf/cpu.max   # full core now
sleep 3
cat /sys/fs/cgroup/thr/leaf/cpu.stat | grep nr_throttled          # frozen count stops rising
kill %1
sudo rmdir /sys/fs/cgroup/thr/leaf /sys/fs/cgroup/thr
      nr_throttled 89                  ← unchanged from before the sleep: no new throttling
    Verify

    Which line of cpu.stat directly measures added latency from throttling, and what does it mean if it keeps climbing? (throttled_usec — the total time the cgroup's processes were frozen waiting for quota; a rising value is latency the app suffers with no host-CPU cost.) Then: explain in one sentence how a service can be slow while the node's CPU is idle. (It is CPU-throttled by its cgroup — pinned to a small quota and repeatedly frozen against the window wall — so it stalls without ever making the host busy.)

    Gotchas

    throttled_usec is cumulative since the cgroup was created — a big number is not automatically alarming; what matters is whether it is increasing, and how fast, relative to nr_periods. Always read it twice with a gap (as in the lab) and look at the delta. A cgroup that was briefly throttled at startup and is now fine has a large-but-static throttled_usec; a cgroup throttling right now has one that climbs every second.

    Some throttling can be perfectly acceptable — a batch job you deliberately capped should be throttled, that's the point. The problem is throttling on a latency-sensitive workload, where the freezes show up as tail latency. So cpu.stat is evidence, not a verdict: pair "is it throttled?" with "does this workload care about latency?" before deciding the cap is wrong. Blindly removing every CPU limit because nr_throttled > 0 is as much a mistake as ignoring it.

    This is one of the most valuable node-debugging skills in Kubernetes. When someone reports "the pod is slow but the node isn't busy," the answer is almost always CPU throttling from a resources.limits.cpu that is too tight — and you find it in exactly this file, at the container's cgroup under the node's kubepods tree, or via the container_cpu_cfs_throttled_periods_total metric that surfaces the same nr_throttled. Knowing that host CPU can be idle while a container starves on its quota is the insight that resolves these tickets. The fix is usually to raise or remove the CPU limit, keeping the request (weight) for fair scheduling.

    CPU-throttling latency is a headline troubleshooting scenario in W6, and it ties back to how limits are set in W3.16.

    Docs

    The kernel cgroup-v2.rst "CPU" section documents cpu.stat and each field (nr_periods, nr_throttled, throttled_usec). man 7 cgroups for the controller context. In practice you will also meet these as the container_cpu_cfs_* Prometheus metrics; in the exam docs, throttling is discussed under CPU limits and pod resource management.

    B7.8

    Memory: current, high (throttle) vs max (kill)

    Why it matters

    CPU pressure makes things slow; memory pressure makes things dead. That asymmetry is the single most important fact about resource limits, because you can wait out a slow process but a killed one is gone with an error you have to go hunting for. The memory controller has two different ceilings — one that throttles and one that kills — and knowing which is which is the difference between a process that crawls and a process that vanishes. This lesson lays out the controls; the next one pulls the trigger.

    The concept

    The memory controller exposes a few files, and three of them carry the story:

    • memory.current — how much memory the cgroup is using right now, in bytes. Read-only; this is your live gauge.
    • memory.high — the soft limit. When usage climbs past it, the kernel does not kill anything; instead it aggressively reclaims memory from the cgroup and throttles the processes (slows them down, stalling them in reclaim) to force usage back under the line. A process over memory.high keeps running but crawls. It is a back-pressure valve, not an executioner.
    • memory.max — the hard limit. When the cgroup cannot be kept under this even after reclaim, the kernel invokes the OOM killer (out-of-memory killer) and kills a process in the cgroup to bring usage down. Cross memory.max that the kernel can't reclaim its way out of, and something dies.

    The mental model: memory.high is "start braking hard" and memory.max is "hit the wall." Unlike CPU, there is no way to "throttle" a program that genuinely needs more memory than its cap forever — you can slow its allocation for a while (high), but if it truly needs more than max, the only options the kernel has are to reclaim (take back caches/pages) or kill. Memory is not compressible the way CPU time is: you cannot give a process "less memory more slowly" indefinitely. That is why the memory limit is the dangerous one, and why memory.max being reached ends in a fatality (B7.9).

    usage rises→ past memory.high:reclaim + throttle→ past memory.max:OOM kill
    Lab
    1. Install stress-ng (a load generator that can allocate memory on demand) and set up a memory-limited cgroup. Delegate the memory controller and set a soft and hard line. 
      multipass shell sandbox
sudo apt-get install -y stress-ng
sudo mkdir -p /sys/fs/cgroup/mem/leaf
echo '+memory' | sudo tee /sys/fs/cgroup/mem/cgroup.subtree_control >/dev/null
echo 100M | sudo tee /sys/fs/cgroup/mem/leaf/memory.high      # throttle line
echo 150M | sudo tee /sys/fs/cgroup/mem/leaf/memory.max       # kill line
cat /sys/fs/cgroup/mem/leaf/memory.current
      104857600                        ← memory.high = 100M (bytes)
157286400                        ← memory.max  = 150M
0                                ← nothing in the cgroup yet
    2. Watch memory.current track a real workload staying under both lines. Run a 60M allocator inside the cgroup and read the gauge — it rises to ~60M and holds, no throttling, no kill. 
      sudo bash -c 'echo $$ > /sys/fs/cgroup/mem/leaf/cgroup.procs
              exec stress-ng --vm 1 --vm-bytes 60M --vm-hang 0 -t 8s' &
sleep 3
cat /sys/fs/cgroup/mem/leaf/memory.current
      63102976                     ← ~60M, comfortably below high(100M) and max(150M)
    3. Now cross the soft line only: allocate 120M — over memory.high (100M) but under memory.max (150M). It survives but is throttled into heavy reclaim; note it does not die. 
      wait
sudo bash -c 'echo $$ > /sys/fs/cgroup/mem/leaf/cgroup.procs
              exec stress-ng --vm 1 --vm-bytes 120M --vm-hang 0 -t 6s' &
sleep 4
cat /sys/fs/cgroup/mem/leaf/memory.current   # pinned near high, not at 120M
wait
sudo rmdir /sys/fs/cgroup/mem/leaf /sys/fs/cgroup/mem
      104857600                    ← held around 100M by reclaim — throttled, but alive
    Verify

    What is the difference in the kernel's response when a cgroup exceeds memory.high versus memory.max? (memory.high: reclaim hard and throttle the processes — they slow but survive; memory.max: when reclaim can't keep it under, invoke the OOM killer and kill a process.) Then: why is there no memory equivalent of "just run slower forever" the way CPU throttling works? (Memory isn't compressible — a process that truly needs more than its cap can't be given "less, slower" indefinitely; the kernel can only reclaim reclaimable pages or, failing that, kill.)

    Gotchas

    Values accept suffixes (100M, 1G) on write but memory.current and friends always read back in bytes — 104857600, not 100M. Don't be thrown by the raw byte counts; divide by 1048576 for MiB. Also note memory.max can read as the literal max (unlimited) when unset, exactly like cpu.max.

    memory.current includes more than your program's own allocations — it counts page cache and other kernel memory charged to the cgroup, so it can sit higher than "what my app malloc'd." That is by design (the cgroup is charged for the memory it causes to be used), but it means memory.current near the limit isn't always "my app is about to OOM" — some of it may be reclaimable cache the kernel will drop under pressure before killing anything. This is why memory.high (reclaim first) exists as a gentler stage before memory.max.

    A Kubernetes memory limit (resources.limits.memory) becomes memory.max on the container's cgroup — the hard, kill line. Kubernetes does not by default use memory.high for the limit (there's no "throttle my memory" request the way there is for CPU), which is exactly why exceeding a memory limit kills the container rather than slowing it: the limit is max, and max means OOM. The memory request influences scheduling and eviction ordering rather than setting a hard cgroup cap. Understanding that memory limit = memory.max = "kill on breach" is what makes the next lesson's exit-code-137 story inevitable rather than surprising.

    Memory requests/limits and their enforcement are W3.16; node-pressure eviction, which uses memory usage to decide what to evict, is W3.18.

    Docs

In the exam docs, resources.limits.memory and resources.requests.memory.

    B7.9

    The OOM kill: exit 137, memory.events, and the dmesg verdict

    Why it matters

    "My container just disappeared and there's nothing in its logs" is one of the most common and most confusing failures in production, and this lesson makes it something you can diagnose in seconds. When a cgroup blows past its hard memory limit, the kernel kills a process — and because the kill is external and instant, the application never gets to log anything. The evidence lives in three specific places, none of them the app's own output. Trigger the kill on purpose once, learn where the fingerprints are, and this failure stops being a mystery forever.

    The concept

    When a cgroup exceeds memory.max and the kernel cannot reclaim enough to get it back under, the OOM killer picks a process in that cgroup and sends it SIGKILL — signal 9, the un-catchable, un-ignorable signal from B2. Because it is SIGKILL, the process cannot run a handler, cannot flush logs, cannot clean up; it is simply stopped. This is why the app's own logs are silent about its death. Three separate pieces of evidence record what happened:

    • Exit code 137. By Unix convention, a process killed by signal N reports exit status 128 + N. SIGKILL is signal 9, so 128 + 9 = 137. Whenever you see a process or container exit 137, read it as "killed by SIGKILL" — and in a memory-limited context that almost always means the OOM killer. (128+15 = 143 is the softer SIGTERM cousin.)
    • memory.events. The cgroup keeps a counter file; its oom_kill line increments each time the OOM killer fired inside this cgroup. A non-zero oom_kill is the cgroup itself telling you "I killed something for memory."
    • The kernel log (dmesg). The kernel writes a detailed verdict: which cgroup ran out, which PID and process name it chose, and a memory accounting table. This is the authoritative record, and it is in the kernel's log, never the application's.

    The throughline: the app can't tell you it was OOM-killed because SIGKILL gives it no chance to. So you never look in the app's logs for this — you look at the exit code, the cgroup's memory.events, and dmesg. That reflex is the entire debugging drill's second half (B7.14).

    usage > memory.max→ kernel sendsSIGKILL (9)→ exit137· logged inmemory.events + dmesg, not the app
    Lab
    1. Build a cgroup with a low hard limit and no soft valve, so an over-allocation goes straight to a kill. 50M max, then run a 200M hog inside it. 
      multipass shell sandbox
sudo mkdir -p /sys/fs/cgroup/oom/leaf
echo '+memory' | sudo tee /sys/fs/cgroup/oom/cgroup.subtree_control >/dev/null
echo 50M | sudo tee /sys/fs/cgroup/oom/leaf/memory.max
sudo dmesg -C                         # clear the kernel ring buffer for a clean read
      52428800                              ← 50M hard limit, no memory.high set
    2. Trigger the kill and capture the shell's exit code. Running the allocator as the cgrouped process, then reading $?, shows the 137. 
      sudo bash -c 'echo $$ > /sys/fs/cgroup/oom/leaf/cgroup.procs
              exec stress-ng --vm 1 --vm-bytes 200M --vm-hang 0 -t 10s'
echo "exit code: $?"
      stress-ng: info:  [xxxx] ... started
Killed
exit code: 137                    ← 128 + 9 (SIGKILL) — the OOM killer fired
    3. Collect the other two witnesses: the cgroup's own counter and the kernel's verdict. 
      cat /sys/fs/cgroup/oom/leaf/memory.events | grep oom
sudo dmesg -T | grep -iE 'killed process|out of memory|oom' | tail -3
sudo rmdir /sys/fs/cgroup/oom/leaf /sys/fs/cgroup/oom
      oom_kill 1                        ← the cgroup counted one OOM kill
[... noble] Memory cgroup out of memory: Killed process 4821 (stress-ng-vm)
             total-vm:206...kB, anon-rss:49...kB ...   ← the kernel's authoritative record
    Verify

    A container exits with code 137 and its own logs say nothing about why. What killed it, and how do you compute 137? (It was sent SIGKILL — signal 9 — and Unix reports "killed by signal N" as exit 128 + N, so 128 + 9 = 137; in a memory-limited context that means the OOM killer.) Then: name the two places other than the app's logs that record an OOM kill. (The cgroup's memory.events oom_kill counter, and the kernel log via dmesg, which names the cgroup and the chosen process.)

    Gotchas

    Exit 137 means "killed by SIGKILL," which is usually OOM in a memory-capped context but not always — anything that sends SIGKILL (a manual kill -9, a liveness-probe-triggered kill, a runtime forcibly stopping a container after a grace period) also yields 137. So 137 is "SIGKILL," and you confirm OOM specifically by checking memory.events/dmesg. Don't assume every 137 is memory — but do always check memory first, because it's the most common cause.

    The OOM killer chooses a process in the cgroup, not necessarily the one that allocated the most or the one you'd expect — it uses a heuristic (roughly, biggest memory user, adjustable via oom_score_adj). In a multi-process container this means a child can be killed while the parent survives in a broken state, or vice versa. This is why single-process-per-container is cleaner, and why dmesg naming the specific PID it killed matters: don't assume, read which one it actually chose.

    This is the mechanism behind OOMKilled in kubectl describe pod. When a container exceeds its memory limit (memory.max), the kernel OOM-kills it, the container exits 137, and the kubelet reports the container's lastState as Terminated with reason OOMKilled and exit code 137 — the exact chain you just reproduced. The fix is always one of: raise the memory limit, fix the leak, or right-size the workload. And because the app can't log its own OOM death, kubectl logs of the crashed container is often empty about the cause — you read the reason field and exit code instead, which is this lesson's evidence surfaced by Kubernetes. A tight limit that OOM-kills on every traffic spike is a classic production incident.

    OOMKilled, exit 137, and reading pod termination reasons are central to troubleshooting in W6; memory limits that cause it are set in W3.16.

    Docs

    The kernel cgroup-v2.rst "Memory" section documents memory.events (including oom_kill) and the OOM behaviour under memory.max. man 5 proc for oom_score_adj. man 1 dmesg for reading the kernel log (-T for human timestamps). B2 on the signals you cannot catch (B2.15) for why SIGKILL gives the app no chance to log, and the 128 + signal exit convention (B2.8).     B7.10

    pids.max and io.max: containing fork bombs and capping disk

    Why it matters

    CPU and memory get all the attention, but two other resources can wreck a machine just as thoroughly: the number of processes and the rate of disk I/O. A fork bomb — a process that endlessly spawns copies of itself — can exhaust the system's process table and make a box unloginnable, and a single greedy program can saturate a disk so nothing else can read or write. cgroups cap both, and seeing the fork bomb contained (safely, because you built the cage first) is one of the more satisfying demonstrations in the whole module.

    The concept

    Two more controllers, briefly:

    • pids.max caps the number of processes (and threads — each counts as a task) that may exist in the cgroup. Write a number; once the cgroup holds that many tasks, further fork()/clone() calls fail with EAGAIN ("Resource temporarily unavailable"). This is the fork-bomb containment: a runaway process trying to multiply simply cannot create the next child once the cap is hit, so the explosion is capped at pids.max processes instead of consuming the whole machine's process table. pids.current shows the live count.
    • io.max caps block-device I/O per device, keyed by the device's major:minor number. You can limit read/write bytes-per-second (rbps/wbps) and read/write operations-per-second (riops/wiops). The format is MAJ:MIN key=value …, e.g. 259:0 wbps=1048576 to cap writes to 1 MB/s on device 259:0. This throttles a noisy-neighbour that would otherwise starve the disk for everyone.

    The unifying idea is that "resource" is broader than CPU and RAM: the kernel meters process slots and I/O bandwidth through the same cgroup filesystem, the same echo > file interface. A container that can't be forced past N processes or M MB/s of disk is protected against two whole classes of denial-of-service — one accidental (a buggy loop), one adversarial (a fork bomb). Note that io.max throttling works cleanly on real block devices; on some setups (certain filesystems, or without the io controller fully active) it may not engage, which is why this lesson leans on pids.max for the hands-on proof.

    Gotchas

    Never test a fork bomb outside a pids.max cgroup — the classic :(){ :|:& };: will exhaust the host's process table in seconds and force you to reset the VM. The entire point of this lesson is that the cage must exist first; the fork bomb is only safe because it is born inside a cgroup that physically caps it. This is a good instinct generally: build the limit, then run the dangerous thing inside it.

    io.max is the fussiest controller to demonstrate: it acts on real block devices and its effect can be invisible on filesystems with heavy caching (writes hit page cache first and get flushed later, so a naive test seems unthrottled), or when the underlying storage stack doesn't cooperate with the cgroup accounting. Don't conclude "io.max is broken" from a quick dd test — it often needs oflag=direct to bypass cache and a real device to show clearly. The mechanism is genuine; it's just the least beginner-friendly to observe, which is why we prove pids.max hands-on and only demonstrate io.max's syntax.

    Kubernetes uses pids.max for pod and node PID limits — the kubelet's podPidsLimit and node-level PID reservation both land as pids.max on the relevant cgroup, protecting a node from a single pod's fork bomb or thread leak exhausting the node's process table (which would take down every pod, not just the offender). io.max underlies I/O isolation between pods, though Kubernetes' first-class I/O controls are less mature than CPU/memory. Both are the same "meter a resource through the cgroup filesystem" pattern you've now applied four times — cpu, memory, pids, io.

    Pod PID limits and resource isolation connect to node stability in W6 and to the resource model in W3.16.

    Answer out loud before revealing. If you can explain why a namespaced pod with no memory limit can still take down a node, this module has done its job.

    1. In one sentence each, what does a namespace control and what does a cgroup control?
      Answer

      A namespace controls what a process can see — a private copy of one kernel resource (process list, hostname, network, mounts). A cgroup controls how much a set of processes can use — a metered cap on CPU, memory, process slots, and disk I/O. Hiding is not rationing; a container is deliberately both.
