    Week 1 — How the control plane actually works

    Everything else in the exam sits on top of five binaries and a database. This week you learn each component's real job and — more valuable — its failure signature. Every lesson runs against the week-0 cluster: read a little, then go poke the live thing.

    The big picture

    1.1

    Draw the full architecture from memory

    Why it matters

    Troubleshooting (30% of the exam) is "which component failed?". You can only answer that fast if the map is in your head, not on a cheat sheet.

    The concept

    The one insight that organizes everything: every component talks only to the apiserver, and only the apiserver talks to etcd. The scheduler doesn't command kubelets. Controllers don't create containers. Each component runs its own loop against the API: watch desired state, converge, report back. Kubernetes is a set of independent control loops around a shared database — not a pipeline.

    Lab
    1. Close this page. On paper, draw every box and arrow above from memory, including who initiates each connection.
    2. Compare against the live cluster: 
      kubectl get pods -n kube-system -o wide
      NAME                          READY   STATUS    NODE
coredns-7db6d8ff4d-8x2mn      1/1     Running   node01
coredns-7db6d8ff4d-tp9qk      1/1     Running   node02
etcd-cp                       1/1     Running   cp
kube-apiserver-cp             1/1     Running   cp
kube-controller-manager-cp    1/1     Running   cp
kube-flannel-ds-*   (x3)      1/1     Running   all nodes
kube-proxy-*        (x3)      1/1     Running   all nodes
kube-scheduler-cp             1/1     Running   cp

      Map each pod to a box in your drawing. Note what's per-cluster (control plane, on cp) vs per-node (kube-proxy, CNI agent — they're DaemonSets).

    3. Redraw until you get it perfect twice in a row.
    Verify

    You can narrate, without notes: "kubectl talks to ___, the scheduler learns about new pods by ___, containers get started when ___."

    Gotchas

    The arrows' direction is the exam-relevant part: the apiserver does not push to kubelets (with one exception you'll meet later: it dials the kubelet for logs/exec). If the apiserver dies, kubelets keep running what they have — they just stop seeing changes. That asymmetry is the key to every control-plane-outage question.

    Docs

    Docs search: "Kubernetes Components" — the overview page with this same diagram, good for a 30-second exam refresher.

    1.2

    Find out how each component actually runs on the node

    Why it matters

    "Fix the broken apiserver" means editing a file, not running a systemctl command — but only if you know which components are static pods and which is the one real systemd service.

    The concept

    Chicken-and-egg: the control plane runs as pods, but pods need a control plane. The escape hatch is static pods: the kubelet watches a directory (staticPodPath: /etc/kubernetes/manifests in its config) and runs whatever pod YAML appears there — no apiserver involved. It then reports read-only mirror pods to the API so you can see them (named <name>-<node>, e.g. kube-apiserver-cp). So the boot order is: systemd starts kubelet (the only real service) → kubelet starts etcd + apiserver + scheduler + controller-manager from manifest files → everything else arrives via the API.

    Lab
    1. On cp, look at the static pod manifests: 
      ls /etc/kubernetes/manifests/
      etcd.yaml  kube-apiserver.yaml  kube-controller-manager.yaml  kube-scheduler.yaml
    2. Confirm kubelet is the only systemd citizen: 
      systemctl status kubelet --no-pager | head -3
systemctl list-units 'kube*'
      ● kubelet.service - kubelet: The Kubernetes Node Agent
     Loaded: loaded (/usr/lib/systemd/system/kubelet.service; enabled)
     Active: active (running)
only kubelet.service exists — no apiserver/scheduler units
    3. Prove mirror pods are read-only — try to delete one: 
      kubectl delete pod kube-apiserver-cp -n kube-system
kubectl get pod kube-apiserver-cp -n kube-system
      pod "kube-apiserver-cp" deleted
NAME                READY   STATUS    RESTARTS   AGE
kube-apiserver-cp   1/1     Running   0          5s

      You deleted the mirror object; the kubelet immediately re-created it because the manifest file still exists. The only way to actually stop a static pod is to move its file out of the manifests directory — try it:

      sudo mv /etc/kubernetes/manifests/kube-scheduler.yaml /tmp/
kubectl get pods -n kube-system | grep scheduler   # gone after ~20s
sudo mv /tmp/kube-scheduler.yaml /etc/kubernetes/manifests/
    Verify

    Scheduler pod disappeared when the file left the directory and returned when you put it back.

    Gotchas

    The kubelet rescans the directory every ~20 seconds — edits to these files feel slightly delayed; don't panic and re-edit. And a syntax error in a static pod manifest means the pod silently never comes back: nothing in kubectl will tell you why (the apiserver may be the thing that's down!) — the evidence is in journalctl -u kubelet. That's the week-6 bread and butter.

    Docs

    Docs search: "Create static Pods".

    1.3

    Trace one kubectl create deployment end-to-end

    Why it matters

    Every "why isn't my pod running?" question is answered by knowing which of the five actors in this chain stopped. This trace is the troubleshooting flowchart.

    The concept

    Five independent loops, no direct calls between them:

    1. kubectl POSTs a Deployment object; the apiserver validates and persists it to etcd. Nothing is running yet.
    2. The deployment controller (in controller-manager) sees a Deployment with no ReplicaSet → creates a ReplicaSet.
    3. The replicaset controller sees an RS with 0 of 2 pods → creates 2 Pod objects with spec.nodeName empty. Still nothing running — these are database rows.
    4. The scheduler watches for pods with empty nodeName, filters and scores nodes, then writes a Binding — which sets nodeName. Still nothing running.
    5. The kubelet on the chosen node sees "a pod scheduled to me" in its watch → pulls images via CRI, starts containers, posts status back. Now it's running.
    Lab
    1. Create a deployment and immediately capture the object chain: 
      kubectl create deployment trace --image=nginx --replicas=2
kubectl get deploy,rs,pods -l app=trace
      NAME                    READY   UP-TO-DATE   AVAILABLE
deployment.apps/trace   2/2     2            2
NAME                               DESIRED   CURRENT   READY
replicaset.apps/trace-66cc6c9b8b   2         2         2
NAME                         READY   STATUS    RESTARTS
pod/trace-66cc6c9b8b-2m9ql   1/1     Running   0
pod/trace-66cc6c9b8b-x7jwd   1/1     Running   0

      Note the naming: pod name = RS name + suffix, RS name = deployment name + pod-template hash. That chain is your ownership breadcrumb.

    2. Read the story the cluster wrote about it: 
      kubectl get events --sort-by=.metadata.creationTimestamp | grep trace | head -8
      ... deployment-controller  Scaled up replica set trace-66cc6c9b8b to 2
... replicaset-controller  Created pod: trace-66cc6c9b8b-2m9ql
... default-scheduler      Successfully assigned default/trace-66cc6c9b8b-2m9ql to node01
... kubelet                Pulling image "nginx"
... kubelet                Created container nginx
... kubelet                Started container nginx

      Every actor from the concept list appears in the SOURCE column, in order. Events are the trace log of the control loops.

    3. Narrate the whole chain out loud on paper — then clean up: kubectl delete deploy trace.
    Verify

    You can name which component acts at each step and — the useful inverse — what it looks like when each one is missing (deployment exists but no RS? controller-manager. Pod exists but Pending with no node? scheduler. Pod has a node but no containers? kubelet/runtime).

    Gotchas

    People say "the scheduler starts pods" — it never does; it only writes a name into a field. Keeping "database write" and "container start" separate in your head is what makes control-plane failures legible: with the scheduler dead, pods still get created, they just stay Pending forever.

    Docs

    Docs search: "Deployments" and "Kubernetes Scheduler".

    kube-apiserver

    1.4

    Learn the request pipeline in order

    Why it matters

    This one ordered list explains RBAC questions, admission questions, and a whole class of "permission denied vs rejected" confusions. It's the most quotable fact in the exam's 25% architecture domain.

    The concept
    authentication→authorization→mutating admission→schema validation→validating admission→persist to etcd
    • Authentication — who are you? Client certificate, bearer token (ServiceAccount JWT, bootstrap token), or OIDC. Modules are tried in turn; first success wins. Failure → 401 Unauthorized.
    • Authorization — may you do this verb on this resource? The configured authorizers (on kubeadm: Node,RBAC) are consulted in order; first to allow wins; default is deny. Failure → 403 Forbidden.
    • Mutating admission — webhooks may edit the object. This is how sidecar injectors and defaulters work.
    • Validation + validating admission — the object may be rejected but no longer changed.
    • Only then is it serialized into etcd. A watch event then fans out to whoever is watching.

    Reading errors backwards: 401 = authn (bad cert/token), 403 = authz (RBAC), admission webhook denied = admission. Three different fixes, and the status code tells you which lesson to apply.

    Lab
    1. See defaulting/mutation happen. Compare what you send with what gets stored: 
      kubectl run pipeline-demo --image=nginx $do > /tmp/sent.yaml
kubectl apply -f /tmp/sent.yaml
kubectl get pod pipeline-demo -o yaml | grep -E 'serviceAccountName|tolerations:|imagePullPolicy|node.kubernetes.io' 
          imagePullPolicy: Always
  serviceAccountName: default
  tolerations:
  - effect: NoExecute
    key: node.kubernetes.io/not-ready
...

      You never wrote a serviceAccount, tolerations, or an imagePullPolicy — admission plugins and defaulting added them between your POST and etcd.

    2. See authorization answer a question directly: 
      kubectl auth can-i delete nodes
kubectl auth can-i delete nodes --as=system:serviceaccount:default:default
      yes
no
    3. Clean up: kubectl delete pod pipeline-demo.
    Verify

    Recite the pipeline in order from memory, and state which stage produces 401, 403, and "webhook denied" errors.

    Gotchas

    Order matters and is testable: authentication always precedes authorization (a bad token gives 401 even for an action anyone may do), and mutating admission runs before validating — so a webhook can fix an object that would otherwise fail validation.

    Docs

    Docs search: "Controlling Access to the Kubernetes API".

    1.5

    Read your live kube-apiserver.yaml and explain 10 flags

    Why it matters

    Week-6 exam tasks plant faults in this exact file. Reading it fluently now means later you'll spot the wrong line in seconds.

    Lab
    1. Open it read-only (get in the habit — accidental edits restart the apiserver): 
      sudo less /etc/kubernetes/manifests/kube-apiserver.yaml
    2. Find each flag and say its job out loud before checking the table:
    FlagWhat it does / breaks
    --etcd-serversWhere state lives. Wrong URL → apiserver runs but every request 500s with connection-refused storms in its logs.
    --etcd-cafile / certfile / keyfileClient TLS to etcd (etcd has its own CA — see 2.5).
    --client-ca-fileThe CA that client certificates must chain to — this is what makes kubectl cert auth work.
    --tls-cert-file / --tls-private-key-fileThe apiserver's own serving certificate (SANs matter — see 2.6).
    --authorization-mode=Node,RBACAuthorizer chain, consulted in order (1.4).
    --enable-admission-pluginsCompiled-in admission plugins to enable (e.g. NodeRestriction).
    --service-cluster-ip-rangeCIDR that ClusterIPs are allocated from (week 4).
    --service-account-issuer / --service-account-key-fileIssuing and verifying ServiceAccount JWTs (sa.key/sa.pub — 2.5, 7.4).
    --kubelet-client-certificate / -keyThe apiserver's identity when it dials kubelets for logs/exec — the one reversed arrow from 1.1.
    --secure-port=6443The port everything connects to; the one in your kubeconfig.
    Verify

    Cover the right column and explain all ten from the flag name alone.

    Gotchas

    It's a pod manifest, so flags live under spec.containers[0].command — planted faults hide there, plus in volumeMounts/hostPath volumes (a wrong cert path is as fatal as a wrong cert). Also note livenessProbe: a misconfigured apiserver can get restart-looped by its own probe.

    Docs

    Docs search: "kube-apiserver" (the component reference lists every flag).

    1.6

    Understand watches — see the HTTP request kubectl actually makes

    Why it matters

    "Watch" is the verb that makes the whole architecture from 1.1 work. Understanding it explains resourceVersion, "object has been modified" errors, and why the control plane scales at all.

    The concept

    A watch is just a long-lived HTTP GET with ?watch=true&resourceVersion=N: "stream me every change to these objects since revision N". The apiserver keeps a watch cache per resource type, fed by a single etcd watch — so 5,000 kubelets watching pods cost etcd almost nothing. resourceVersion is literally etcd's MVCC revision counter (1.10): every write bumps it, and comparing versions is how optimistic concurrency ("the object has been modified; please apply your changes to the latest version") detects conflicting edits.

    Lab
    1. Watch kubectl's own HTTP calls with verbosity 6: 
      kubectl get pods -w -v6 2>&1 | head -5
      GET https://10.211.64.10:6443/api/v1/namespaces/default/pods?limit=500 200 OK in 12 ms
GET https://10.211.64.10:6443/api/v1/namespaces/default/pods?resourceVersion=48211&watch=true ...

      Two requests: a LIST to get current state + its resourceVersion, then a WATCH from that version. Every informer in every controller does exactly this pair.

    2. In a second terminal, create and delete a pod; watch events stream in the first terminal without polling. Ctrl-C when convinced.
    3. Trigger the concurrency error once on purpose: 
      kubectl create deployment cas-demo --image=nginx
kubectl get deploy cas-demo -o yaml > /tmp/a.yaml
kubectl scale deploy cas-demo --replicas=3     # someone else changes it
kubectl replace -f /tmp/a.yaml
      Error from server (Conflict): ... "cas-demo" ... the object has been
modified; please apply your changes to the latest version and try again

      Your file carries the old resourceVersion; the server refuses the stale write. Fix: re-get and re-edit (or use kubectl edit/apply, which handle it). Clean up: kubectl delete deploy cas-demo.

    Verify

    You can explain what the two -v6 requests were and why the Conflict error happened.

    Gotchas

    kubectl get -w is also your best "is the fix working?" tool: keep one terminal watching (k get pods -w) while you repair things in another. It's cheaper than re-running get twenty times.

    1.8

    Know API groups & versions — and where objects live in etcd

    Why it matters

    A typo'd apiVersion is one of the cheapest planted exam bugs, and kubectl api-resources is the tool that answers "what group is this thing in" in five seconds.

    The concept

    Resources are organized into groups: the original core group has no name (apiVersion: v1 — Pods, Services, ConfigMaps, PVCs, Namespaces); later resources live in named groups (apps/v1: Deployments, ReplicaSets, StatefulSets, DaemonSets; batch/v1: Jobs, CronJobs; networking.k8s.io/v1: Ingress, NetworkPolicy; rbac.authorization.k8s.io/v1: Roles, Bindings; storage.k8s.io/v1: StorageClasses). In etcd, every object is stored once, at a path like /registry/deployments/<namespace>/<name>, in one canonical storage version — the apiserver converts between served versions on the fly, which is why version skew mostly "just works".

    Lab
    1. Make the group table yourself: 
      kubectl api-resources | head -12
kubectl api-resources --api-group=apps
kubectl api-versions | head
      NAME          SHORTNAMES   APIVERSION   NAMESPACED   KIND
configmaps    cm           v1           true         ConfigMap
...
deployments   deploy       apps/v1      true         Deployment
daemonsets    ds           apps/v1      true         DaemonSet
...

      Note the two bonus columns you'll use constantly: SHORTNAMES (deploy, svc, cm, netpol, pvc…) and NAMESPACED (nodes, PVs, StorageClasses are cluster-scoped — relevant to RBAC in week 7).

    2. Reproduce the classic planted bug and read its error: 
      kubectl create deploy broken --image=nginx $do | sed 's|apps/v1|apps/v1beta1|' | kubectl apply -f -
      error: resource mapping not found for name: "broken" namespace: ""
from "STDIN": no matches for kind "Deployment" in version "apps/v1beta1"

      "No matches for kind X in version Y" always means: fix the apiVersion line; kubectl api-resources | grep -i <kind> tells you the right one.

    Verify

    Without looking: which group/version are Deployment, Ingress, Job, Role, PersistentVolume in? (apps/v1, networking.k8s.io/v1, batch/v1, rbac.authorization.k8s.io/v1, core v1.)

    Gotchas

    CronJob is batch/v1 and Ingress is networking.k8s.io/v1 — old tutorials show batch/v1beta1 and extensions/v1beta1, which no longer exist. When copying YAML from anywhere, the apiVersion line is the first thing to distrust.

    Docs

    Docs search: "API Overview"; in the exam, kubectl api-resources is usually faster than the docs.

    etcd

    1.9

    Learn Raft at the "explain to a colleague" level

    Why it matters

    You won't implement Raft in the exam — but quorum arithmetic explains etcd's failure behavior, and "what still works when etcd is down" is genuine exam material (6.8).

    The concept

    etcd members elect a leader; all writes go through it. The leader appends each write to its write-ahead log and replicates it to followers; the write commits once a quorum — a strict majority — has persisted it. If the leader dies, followers time out and elect a new one (elections take ~seconds; writes pause, reads from the apiserver's watch cache continue).

    MembersQuorumFailures toleratedNote
    110your lab; fine for learning
    220worse than 1 — either member down halts writes
    321the production minimum
    532bigger clusters slow writes; 5 is usually the ceiling

    Losing quorum makes etcd read-only: the apiserver still answers reads (its watch cache is warm) but every write hangs or errors. Pods already running keep running — the data plane doesn't depend on etcd at all.

    Lab
    1. Ask your (single-member) cluster who the leader is. Easiest path: exec etcdctl inside the etcd pod, where the certs and env are already set: 
      kubectl -n kube-system exec etcd-cp -- etcdctl \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  endpoint status -w table
      +------------------------+-----------+---------+---------+-----------+...
|        ENDPOINT        |    ID     | VERSION | DB SIZE | IS LEADER |
+------------------------+-----------+---------+---------+-----------+
| https://127.0.0.1:2379 | 8e9e05c52 | 3.5.x   | 6.1 MB  |   true    |
    2. Recite the quorum table for 3 and 5 members, and why 2 is worse than 1.
    Verify

    Explain to an imaginary colleague: where do writes go, when do they commit, what happens to reads and to running pods when quorum is lost.

    Gotchas

    "etcd down" ≠ "cluster down": workloads keep serving traffic. What stops is change — no new pods, no rescheduling, no scaling. That distinction is exactly what checkpoint 6.8 has you observe live.

    1.10

    Understand etcd's MVCC model: revisions, compaction, defrag

    Why it matters

    This is where resourceVersion (1.6) physically comes from, and compaction/defrag show up both in real operations and in etcd's own log messages you'll read during troubleshooting.

    The concept

    etcd never updates a key in place: every write creates a new revision of the key in a bbolt B+tree, and the global revision counter ticks up on every write to anything. Watches are literally "replay every event since revision N" — that's why a watch can reconnect and miss nothing. Old revisions pile up until compaction discards history below some revision (Kubernetes' apiserver requests compaction every 5 minutes). Compaction frees logical space only; defragmentation (etcdctl defrag) rewrites the DB file to return space to the OS. A neglected etcd hits its space quota and goes into a read-only alarm:NOSPACE state — the pathological version of "writes hang".

    Lab
    1. Watch the revision counter move. Create anything and check an object's resourceVersion before/after: 
      kubectl create cm mvcc-demo --from-literal=a=1
kubectl get cm mvcc-demo -o jsonpath='{.metadata.resourceVersion}'; echo
kubectl patch cm mvcc-demo -p '{"data":{"a":"2"}}'
kubectl get cm mvcc-demo -o jsonpath='{.metadata.resourceVersion}'; echo
      51284
51391

      The jump is bigger than 1 — the whole cluster shares one counter, and other components wrote in between. Clean up: kubectl delete cm mvcc-demo.

    2. Find compaction in etcd's own logs: 
      kubectl -n kube-system logs etcd-cp | grep -i compact | tail -2
      {"msg":"finished scheduled compaction","compact-revision":50812,"took":"12ms"}
    Verify

    You can define revision, compaction, defrag in one sentence each — and say which one returns disk space to the OS (only defrag).

    Gotchas

    When you restore an etcd snapshot (week 2), all watchers' resourceVersions suddenly point into a different history — that's why the standard procedure restarts the control plane afterwards. It'll make sense mechanically now: watches are revision streams, and you just swapped the book they were reading.

    1.11

    Extract the four etcdctl connection values from the manifest

    Why it matters

    The etcd backup/restore exam task is guaranteed points if you can assemble the etcdctl command quickly. All four values you need are sitting in one file — knowing where beats memorizing them.

    Lab

    Gotchas

    Using the server cert as the client cert (as above) works because kubeadm's etcd server cert allows client auth — standard exam practice. If a task provides dedicated healthcheck-client certs, use those. And remember it's --cacert, not --ca-cert — a 30-second typo hunt under exam pressure.

    1.13

    Check etcd health the way the exam expects

    Why it matters

    Two commands cover every "is etcd okay?" moment in troubleshooting tasks and in the backup/restore drill.

    Lab
    1. Health of the endpoint(s): 
      sudo etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  endpoint health
      https://127.0.0.1:2379 is healthy: successfully committed proposal: took = 5.9ms

      Note what it does: it commits a proposal — a real quorum write, not a ping. If quorum is lost, this hangs/fails even though the process is up. That's the difference between "running" and "healthy".

    2. Membership: 
      sudo etcdctl ... member list -w table
      +-----------+---------+------+---------------------------+---------------------------+
|    ID     | STATUS  | NAME |        PEER ADDRS         |       CLIENT ADDRS        |
+-----------+---------+------+---------------------------+---------------------------+
| 8e9e05c52 | started | cp   | https://10.211.64.10:2380 | https://10.211.64.10:2379 |

      Port map worth memorizing: 2379 = client traffic (apiserver→etcd), 2380 = peer traffic (etcd↔etcd replication).

    Verify

    Both commands from memory, and you can state what 2379 vs 2380 carry.

    Gotchas

    If etcdctl can't connect at all, check you're on the right node (etcd runs on the control plane; ssh cp first) and that you used sudo (the key files are root-readable only). "Permission denied" on the key file is the most common self-inflicted etcdctl failure.

    Scheduler & controller-manager

    1.14

    Learn the scheduler's two-phase cycle

    Why it matters

    Week 3's scheduling controls (taints, affinity, resources) all plug into this machinery, and "0/3 nodes are available" messages become readable once you know which phase produced them.

    The concept

    The scheduler watches for pods with empty spec.nodeName and runs each through two phases:

    PreFilter→Filter→PostFilter (preemption)→Score→Reserve→bind (async)
    • Filter drops nodes that can't host the pod: not enough free requests (not actual usage!), untolerated taints, failed nodeSelector/affinity, port conflicts, volume topology. Survivors are "feasible".
    • PostFilter only runs when zero nodes survive — this is where preemption lives: evict lower-priority pods to make room, if the pod has higher priority.
    • Score ranks feasible nodes: least-allocated resources, topology spreading, image already present, affinity preferences. Highest total wins.
    • The binding cycle then writes a Binding subresource asynchronously — the API write that sets nodeName. The scheduler never contacts any node.
    Lab
    1. Make Filter fail on every node and read the message: 
      kubectl run greedy --image=nginx --overrides='{"spec":{"containers":[{"name":"greedy","image":"nginx","resources":{"requests":{"cpu":"64"}}}]}}'
kubectl describe pod greedy | tail -4
      Warning  FailedScheduling  default-scheduler
  0/3 nodes are available: 3 Insufficient cpu.
  preemption: 0/3 nodes are available: 3 No preemption victims found
  for incoming pod.

      Read it precisely: Filter left 0 of 3 nodes (all failed the cpu check), then PostFilter/preemption also found nothing to evict. Every FailedScheduling message has this anatomy: counts + per-node filter reasons + preemption verdict.

    2. Fix by lowering the request and watch it schedule: kubectl delete pod greedy, re-run with "cpu":"100m", confirm Running, then delete it.
    Verify

    Name the two phases, three Filter examples, two Score examples, and where preemption happens.

    Gotchas

    Scheduling is about requests, not usage: a node running nothing but with all CPU requested filters out. And a Pending pod with no events at all means the scheduler never saw it — scheduler down (or a nonexistent schedulerName), a different signature from FailedScheduling. That distinction is week-6 gold.

    Docs

    Docs search: "Kubernetes Scheduler" and "Scheduler Configuration".

    1.15

    Prove it: bypass the scheduler with a preset nodeName

    Why it matters

    Ten minutes that permanently separates "the scheduler assigns pods" from "the kubelet runs pods". It also hands you a real repair tool: manual scheduling works even with the scheduler dead.

    Lab
    1. Stop the scheduler (static pod trick from 1.2): 
      sudo mv /etc/kubernetes/manifests/kube-scheduler.yaml /tmp/
    2. Show that normal pods now hang: 
      kubectl run waiting --image=nginx
kubectl get pod waiting
      NAME      READY   STATUS    RESTARTS   AGE
waiting   0/1     Pending   0          30s   ← and no events: nobody is scheduling
    3. Now cut the scheduler out of the loop — set nodeName yourself: 
      kubectl run direct --image=nginx --overrides='{"spec":{"nodeName":"node01"}}'
kubectl get pod direct -o wide
      NAME     READY   STATUS    NODE
direct   1/1     Running   node01

      The kubelet on node01 saw a pod bound to it and just ran it. nodeName preset = no scheduling, no Filter checks, not even taints are honored.

    4. Restore and clean up: 
      sudo mv /tmp/kube-scheduler.yaml /etc/kubernetes/manifests/
kubectl get pod waiting -o wide     # scheduled within seconds of scheduler return
kubectl delete pod waiting direct
    Verify

    direct ran with the scheduler dead; waiting sat Pending until the scheduler returned, then scheduled without you touching it (level-triggered recovery — see 1.16).

    Gotchas

    Because preset nodeName skips taint checks, it even lands pods on control-plane nodes. Exam angle: some troubleshooting tasks are fastest solved by pinning a pod to a specific node — and now you know why that always works.

    1.16

    Understand the controller pattern: informer → queue → reconcile

    Why it matters

    Every controller — built-in or operator/CRD (which the 2025 curriculum expects you to understand) — is this same shape. Learn it once, recognize it everywhere.

    The concept

    A controller has three parts: an informer (the LIST+WATCH pair from 1.6 plus an in-memory cache) that turns API changes into events; a work queue where events land as object keys (namespace/name, deduplicated); and worker loops that pop a key and run reconcile: read desired state (the spec), read observed state (the world), do one small step toward convergence, requeue with backoff on error. Crucially this is level-triggered, not edge-triggered: reconcile looks at current state, not at the event that woke it — so missed events don't matter, and every informer resyncs periodically anyway. A controller that was down catches up by itself the moment it's back (you watched that happen in 1.15 step 4).

    Lab
    1. Watch reconciliation beat you in real time: 
      kubectl create deployment recon --image=nginx --replicas=3
kubectl delete pod -l app=recon --wait=false && kubectl get pods -l app=recon
      NAME                     READY   STATUS              RESTARTS   AGE
recon-5b8f6d9c44-7xk2p   0/1     Terminating         0          40s
recon-5b8f6d9c44-9mtb4   0/1     ContainerCreating   0          1s
recon-5b8f6d9c44-cq8vw   0/1     Terminating         0          40s
recon-5b8f6d9c44-jw6ns   0/1     ContainerCreating   0          1s
...

      Replacements appear before the old pods finish terminating: the replicaset controller observed "desired 3, observed fewer", and acted. It doesn't care why pods vanished.

    2. Find the controller's identity in events: kubectl get events | grep replicaset-controller | tail -3. Clean up: kubectl delete deploy recon.
    Verify

    Define informer, work queue, reconcile, and level-triggered in one sentence each — and explain why a restarted controller needs no replay of missed events.

    Gotchas

    This is also the mental model for CRDs + operators: a CRD adds a new table to the API; an operator is just a controller reconciling that table. Exam-wise you must locate CRDs (kubectl get crd) and their objects (kubectl get <kind>) — week 3 covers the commands; the machinery is what you just learned.

    Docs

    Docs search: "Controllers" (Concepts → Architecture).

    1.17

    Follow an ownership chain live; understand cascade deletion

    Why it matters

    Explains what actually happens on kubectl delete deployment, why deleting an RS pod "doesn't work", and gives you the --cascade=orphan trick.

    The concept

    Child objects carry an ownerReferences entry pointing at their parent. Nothing "cascades" imperatively: deleting a Deployment just deletes the Deployment row. The garbage collector controller then notices ReplicaSets whose owner no longer exists and deletes them; the same wave then takes the pods. It's reconciliation all the way down.

    Lab
    1. Inspect the chain: 
      kubectl create deployment owners --image=nginx --replicas=2
kubectl get rs -l app=owners -o jsonpath='{.items[0].metadata.ownerReferences[0].kind}{" "}{.items[0].metadata.ownerReferences[0].name}'; echo
kubectl get pods -l app=owners -o jsonpath='{.items[0].metadata.ownerReferences[0].kind}'; echo
      Deployment owners
ReplicaSet
    2. Delete with orphaning and watch the pods survive: 
      kubectl delete deployment owners --cascade=orphan
kubectl get rs,pods -l app=owners
      NAME                                DESIRED   CURRENT   READY
replicaset.apps/owners-66d9c8b444   2         2         2
NAME                          READY   STATUS
pod/owners-66d9c8b444-9k2lp   1/1     Running
pod/owners-66d9c8b444-tv84q   1/1     Running

      The RS and pods live on — their ownerReferences were stripped/left pointing at nothing that the GC was told to ignore. This is how you replace a Deployment's management without an outage.

    3. Now delete the RS normally and watch the full cascade: kubectl delete rs -l app=owners → pods terminate.
    Verify

    Explain who deletes the pods when you delete a Deployment (the GC controller, via ownerReferences — not kubectl, not the apiserver).

    Gotchas

    A pod that keeps coming back after delete has a living owner — check ownerReferences (or just kubectl get rs,ds,sts). Deleting children of a controller is almost never the fix; delete or scale the owner. Conversely a pod stuck Terminating forever usually has a finalizer — a related but different mechanism you'll meet in 6.18.

    1.18

    Know leader election: why two controller-managers don't double-reconcile

    Why it matters

    Explains HA control planes (relevant to multi-control-plane questions) and introduces Lease objects, which reappear in node heartbeats (1.22).

    The concept

    Controller-manager and scheduler replicas would conflict if all were active — imagine three replicaset controllers each creating 3 pods. So all replicas run, but only the one holding a Lease object acts; the rest hot-standby. The holder renews the lease every few seconds; if it dies, another replica acquires the lease when it expires and takes over. Same watch-and-write machinery as everything else — coordination through the API, no direct communication.

    Lab
    1. Look at the leases: 
      kubectl get leases -n kube-system
      NAME                      HOLDER                                      AGE
kube-controller-manager   cp_a1b2c3d4-...                             1d
kube-scheduler            cp_e5f6a7b8-...                             1d
    2. Watch a renewal tick: 
      kubectl get lease kube-controller-manager -n kube-system \
  -o jsonpath='{.spec.renewTime}'; echo; sleep 5
kubectl get lease kube-controller-manager -n kube-system \
  -o jsonpath='{.spec.renewTime}'; echo
      2026-07-10T09:14:02.000000Z
2026-07-10T09:14:07.000000Z
    Verify

    You can explain why running two controller-manager replicas doesn't create duplicate pods, and name the object that guarantees it.

    Gotchas

    In your single-control-plane lab the election still runs — with one candidate. Symptom knowledge: if the controller-manager logs show it waiting to acquire lease, some other holder exists (or held it recently); that log line means "running but deliberately idle", not "broken".

    kubelet & the container runtime

    1.19

    Learn the kubelet's syncLoop and its three pod sources

    Why it matters

    The kubelet is the component that turns database rows into running processes. Knowing its inputs and its health signals ("PLEG is not healthy") turns several week-6 log messages into instant diagnoses.

    The concept

    The kubelet merges pod specs from three sources into one desired set: (1) the apiserver watch — normal pods bound to this node; (2) the static pod directory (staticPodPath) — how the control plane bootstraps (1.2); (3) an HTTP endpoint (rarely used). Per-pod workers then converge reality: create the sandbox, pull images, start containers, run probes. Coming back the other way, the PLEG (Pod Lifecycle Event Generator) polls the runtime for container state changes and feeds them into the sync loop — that's how the kubelet notices "container died" and restarts it per restartPolicy.

    Lab
    1. Find the sources in the live config: 
      sudo grep -E 'staticPodPath|syncFrequency' /var/lib/kubelet/config.yaml
      staticPodPath: /etc/kubernetes/manifests
    2. Watch a sync cycle narrate itself — restart a pod's container and follow kubelet's log on that node (find the node with kubectl get pod <pod> -o wide, then on it): 
      kubectl run pleg-demo --image=nginx    # then on its node:
sudo journalctl -u kubelet -f | grep -i pleg-demo &
kubectl exec pleg-demo -- kill 1       # kill nginx's PID 1
      ... SyncLoop (PLEG): "pleg-demo_default", event: ContainerDied
... Container nginx failed liveness/exited, restarting ...

      PLEG detected the death from the runtime side; the sync loop restarted the container. kubectl get pod pleg-demo now shows RESTARTS: 1. Clean up: kubectl delete pod pleg-demo.

    Verify

    Name the three pod sources and describe PLEG's job in one sentence.

    Gotchas

    Log signature to memorize: PLEG is not healthy in kubelet logs means the runtime is hung or desperately slow (often disk pressure or a containerd wedge) — the node soon flaps NotReady. The fix hunt starts at containerd, not at the kubelet.

    1.20

    Understand CRI: kubelet → containerd → shim → runc

    Why it matters

    crictl talks CRI directly to the runtime — no apiserver needed. It is the debugging tool for control-plane-down scenarios, so the chain it plugs into is worth knowing precisely.

    The concept

    The kubelet calls the runtime over gRPC on a Unix socket — two services: RuntimeService (sandboxes/containers) and ImageService (pulls). containerd receives the call and spawns a tiny shim process per pod, which invokes runc; runc assembles namespaces + cgroups per the OCI spec, starts the container's PID 1, and exits. The shim stays alive holding stdio and the exit code — which is why containerd itself can restart without killing your containers.

    Lab
    1. Find the socket the kubelet uses: 
      cat /var/lib/kubelet/kubeadm-flags.env
      KUBELET_KUBEADM_ARGS="--container-runtime-endpoint=unix:///run/containerd/containerd.sock --pod-infra-container-image=registry.k8s.io/pause:3.10"
    2. Point crictl at it permanently (on every node): 
      sudo crictl config --set runtime-endpoint=unix:///run/containerd/containerd.sock
sudo crictl ps | head -5
      CONTAINER      IMAGE          STATE     NAME                      POD ID
1a2b3c4d5e6f   a9dfdba8b719   Running   kube-apiserver            9f8e7d6c...
2b3c4d5e6f7a   014faa467e29   Running   etcd                      8e7d6c5b...

      Note: crictl sees containers on this node only — it's a node tool, not a cluster tool.

    3. See the shims: 
      ps -ef | grep containerd-shim | head -3
      root  1234  1  ... /usr/bin/containerd-shim-runc-v2 -namespace k8s.io -id 9f8e7d6c...

      One shim per pod, parented to PID 1 (not to containerd) — that's the restart-survival trick.

    Verify

    Draw the chain kubelet → CRI gRPC → containerd → shim → runc → container, and state which piece exits immediately (runc) and which holds the exit code (shim).

    Gotchas

    crictl without configuration may warn about deprecated endpoints or try multiple sockets — configure it once as above. Also memorize: images live under ImageService, so it's crictl images / crictl pull, and logs are crictl logs <container-id> (IDs from crictl ps -a — the -a matters, dead containers are the interesting ones).

    Docs

    Docs search: "crictl" (Debugging Kubernetes nodes with crictl).

    1.21

    Meet the pause container

    Why it matters

    The pause container is the physical answer to "why do containers in a pod share an IP?" — and it explains a whole family of crictl output you'll read in week 6.

    The concept

    Every pod starts with an invisible pause container (the "sandbox") whose only job is to sleep while owning the pod's namespaces — most importantly the network namespace. App containers join its namespaces. That's the whole mechanism behind "containers in a pod share an IP and localhost": they're in one netns, and the netns belongs to pause. When an app container crashes and restarts, the pod IP survives because pause never died. Only sandbox death (or pod deletion) releases the IP.

    Lab
    1. On any worker, compare the two views: 
      sudo crictl pods | head -3
sudo crictl ps | head -3
      POD ID         STATE   NAME                     NAMESPACE
9f8e7d6c5b4a   Ready   coredns-7db6d8ff4d-8x2   kube-system
...
CONTAINER      STATE     NAME      POD ID
5e6f7a8b9c0d   Running   coredns   9f8e7d6c5b4a

      pods lists sandboxes; ps lists app containers joined to them. The pause containers themselves are hidden by default — see the processes instead:

      ps -ef | grep -c '/pause'
      4     ← one per pod on this node
    2. Prove IP survival: kubectl run ip-demo --image=nginx, note kubectl get pod ip-demo -o wide IP, then kubectl exec ip-demo -- kill 1, wait for the restart, check the IP again — unchanged. Delete the pod; a recreated one gets a new IP. Clean up.
    Verify

    Explain what owns a pod's IP, and why a container restart keeps it but a pod recreation doesn't.

    Gotchas

    The pause image (registry.k8s.io/pause:3.x) is set in containerd's config and kubelet flags — if a task or broken node shows sandbox creation failures ("failed to get sandbox image"), that's a runtime config problem, not a pod problem. Pod IPs are ephemeral by design — which is exactly why Services exist (week 4).

    1.22

    Understand node heartbeats and the death timeline

    Why it matters

    "Node died — why isn't my pod moving?" has a precise, numeric answer. Exam troubleshooting tasks and real life both hinge on these four numbers.

    The concept

    Two heartbeat channels: the kubelet renews a Lease in namespace kube-node-lease every 10s (cheap), and patches NodeStatus every 5m (or on change — expensive, full conditions report). When leases stop, the node-lifecycle controller (in controller-manager) marks the node Unknown/NotReady after ~40s (node-monitor-grace-period) and adds the node.kubernetes.io/unreachable:NoExecute taint. Pods carry a default toleration for that taint of 300s — after which they're evicted and their controllers create replacements elsewhere. Timeline: death → ~40s NotReady → +300s pods evicted → replacements scheduled. Roughly five and a half minutes of "nothing seems to happen" — by design.

    Lab
    1. See the leases ticking: 
      kubectl get leases -n kube-node-lease
      NAME     HOLDER   AGE
cp       cp       1d
node01   node01   1d
node02   node02   1d
    2. Find the default tolerations on any pod you didn't write them into: 
      kubectl run tol-demo --image=nginx
kubectl get pod tol-demo -o yaml | grep -A4 tolerations
        tolerations:
  - effect: NoExecute
    key: node.kubernetes.io/not-ready
    operator: Exists
    tolerationSeconds: 300

      Admission injected these (1.4!) — every pod tolerates an unreachable node for 300s. Clean up: kubectl delete pod tol-demo.

    3. Save the full kill-a-node rehearsal for 6.13 — this week you just need the numbers and the objects.
    Verify

    Recite the timeline with numbers: 10s lease renewals, ~40s to NotReady, 300s toleration, then eviction.

    Gotchas

    Evicted-because-unreachable pods can show as lingering Terminating ghosts on a node that never comes back — the kubelet that must confirm the deletion is dead. Force-delete ($now) is the accepted cleanup for that specific case. Also note: DaemonSet pods and static pods never reschedule — they're node-bound by definition.

    1.23

    Explore /var/lib/kubelet — the kubelet's home directory

    Why it matters

    Week-6 kubelet faults hide in these files. Knowing the two config sources and what lives where makes "kubelet won't start" a two-minute fix.

    Lab
    1. Tour the directory on any node: 
      sudo ls /var/lib/kubelet/

memory_manager_state  pki  plugins  plugins_registry  pod-resources  pods

    After editing config.yaml you must systemctl restart kubelet — nothing watches that file. And the error for a wrong path in the systemd unit shows up in journalctl -u kubelet as the service failing before any Kubernetes logic runs: "no such file or directory" right at startup.

    1.24

    Weekend drill: stop the scheduler, then the controller-manager — narrate everything

    Why it matters

    This drill converts the week from knowledge into signatures. After it, "Pending, no events" and "deployment exists, no pods" are instant diagnoses — each worth minutes in the exam.

    Lab
    1. Scheduler outage. Stop it, create work, observe, restore: 
      sudo mv /etc/kubernetes/manifests/kube-scheduler.yaml /tmp/
kubectl create deployment drill --image=nginx --replicas=2
kubectl get pods -l app=drill
kubectl describe pod -l app=drill | grep -A2 Events
      NAME                     READY   STATUS    RESTARTS   AGE
drill-7c9d8b6f4-k2m9x    0/1     Pending   0          30s
drill-7c9d8b6f4-tq8vn    0/1     Pending   0          30s
Events:  <none>        ← the signature: Pending with NO events

      Narrate: deployment controller made an RS (controller-manager is fine), RS made pods, pods sit with empty nodeName because nobody is scheduling. Restore and watch them go:

      sudo mv /tmp/kube-scheduler.yaml /etc/kubernetes/manifests/
kubectl get pods -l app=drill -w    # Running within ~30s; Ctrl-C, delete deployment
    2. Controller-manager outage. Different layer, different signature: 
      sudo mv /etc/kubernetes/manifests/kube-controller-manager.yaml /tmp/
kubectl create deployment drill2 --image=nginx --replicas=2
kubectl get deploy,rs,pods -l app=drill2
      NAME                     READY   UP-TO-DATE   AVAILABLE
deployment.apps/drill2   0/2     0            0
No resources found (no ReplicaSet, no pods)

      Narrate: the Deployment row exists (apiserver fine) but no controller turned it into an RS. Existing pods elsewhere keep running — reconciliation is paused, not reversed. Restore and watch the RS + pods appear within seconds (level-triggered catch-up, 1.16):

      sudo mv /tmp/kube-controller-manager.yaml /etc/kubernetes/manifests/
kubectl get deploy,rs,pods -l app=drill2
    3. Clean up both deployments. Then write the two signatures into your notes: Pending + no events = scheduler; object exists + no children = controller-manager.
    Verify

    You performed both outages, narrated each chain aloud, and wrote the two signatures down.

    Gotchas

    Do the drill on the lab you can break (snapshot exists, right?). If a moved manifest doesn't come back, check you moved it back to exactly /etc/kubernetes/manifests/ and give the kubelet its ~20s rescan. That recovery-of-your-own-drill is itself week-6 practice.

    Week 1 recall quiz

    Answer out loud before revealing. Wrong or hesitant → redo that lesson's lab.

    1. Which component is the only one that talks to etcd, and which is the only one running as a systemd service?
      Answer

      The kube-apiserver talks to etcd; the kubelet is the only systemd service — everything else on the control plane is a static pod the kubelet starts from /etc/kubernetes/manifests.

    2. Recite the apiserver request pipeline in order. Which stages produce 401, 403, and "webhook denied"?
      Answer

      Authentication (401) → authorization (403) → mutating admission → validation → validating admission ("admission webhook … denied") → persist to etcd.

    3. kubectl says "the object has been modified; please apply your changes to the latest version". What mechanism fired?
      Answer

      Optimistic concurrency on resourceVersion (etcd's MVCC revision): your write carried a stale version, someone wrote in between. Re-get and re-apply.

    4. Why is a 2-member etcd cluster worse than 1 member?
      Answer

      Quorum of 2 is 2 — either member failing halts all writes. One member has quorum 1 and at least fails only when it fails.

    5. What still works when etcd is down? What stops?
      Answer

      Running pods keep running; apiserver serves (cached) reads. All writes hang: no new pods, no rescheduling, no scale operations.

    6. A pod is Pending with no events at all. A second pod is Pending with FailedScheduling: 0/3 nodes available. Diagnoses?
      Answer

      No events = the scheduler never saw it → scheduler down (or wrong schedulerName). FailedScheduling = scheduler alive, but Filter eliminated every node (resources/taints/affinity) and preemption found no victims.

    7. You delete a Deployment. Who actually deletes the pods?
      Answer

      The garbage collector controller, following ownerReferences: orphaned RS deleted first, then its pods. --cascade=orphan suppresses this.

    8. What does the pause container do, and what survives an app-container crash because of it?
      Answer

      It sleeps while owning the pod's namespaces (esp. network). The pod IP survives restarts because the netns owner (pause) never died.

    9. Node loses power. Walk the timeline until its pods run elsewhere.
      Answer

      Lease renewals (10s) stop → ~40s: node NotReady/Unknown + unreachable NoExecute taint → pods' default toleration runs out at 300s → eviction → controllers recreate, scheduler places on healthy nodes. ~5½ minutes total.

    10. Kubelet log says "PLEG is not healthy". Where do you look first?
      Answer

      At the container runtime — PLEG polls containerd for container states; unhealthy means the runtime is hung/slow (check systemctl status containerd, disk pressure, crictl ps responsiveness).
