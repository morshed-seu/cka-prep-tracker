      The machine under Kubernetes

      Kubernetes invented very little. It composes decades-old operating-system, networking and distributed-systems technology — namespaces, cgroups, overlay filesystems, virtual network interfaces, certificates, consensus. This track teaches those pieces directly, by building them with your own hands on one throwaway Linux VM, and only then names the Kubernetes feature that wraps each one.

      So you will meet CPU throttling as cpu.max in a cgroup file long before you meet it as limits.cpu in a pod spec, and you will build a bridge, a veth pair and a NAT rule by hand before anyone says the word "CNI". Every conceptual lesson ends with a “Where this shows up in Kubernetes” block that links forward to the exact CKA-track lesson where the same machinery reappears — so the advanced track reads as recognition rather than new material.

      There is deliberately no calendar here: the eight-week clock belongs to the exam track. Work through the modules in order — each is a prerequisite for the next — at whatever pace fits. Checkboxes save automatically in this browser, in the same store as the CKA tracker, so progress on either track is always visible from the other.

      B0 — Your Linux sandbox

      9 cp · ~2 h

      Everything in this track happens on one machine you are allowed to wreck. Build it, learn to put it back in under a minute, and install the tools every later module assumes are already there.

      Nothing. This is the front door — if you have never opened a terminal, you are in the right place.

      How to use this track

      • Under the hood

        Kubernetes invented very little. A pod is Linux namespaces; a CPU limit is a cgroup file; an image layer is overlayfs; the pod network is a bridge plus veth pairs plus a NAT rule; etcd is Raft consensus; every certificate is ordinary PKI. Learning the wrapper before the thing it wraps is why Kubernetes feels like memorization to so many people.

      • Under the hood

        Progress lives in your browser's localStorage, not on a server — there is no account and nothing to log into, which is also why progress does not follow you to another browser or a private window. The beginner and CKA trackers share one store, so a box ticked on a lesson page shows up here on reload, and vice versa. Each track's reset button clears only its own checkpoints.

      The sandbox VM

      • Under the hood

        Multipass is a thin front end over whatever hypervisor your machine already has — KVM on Linux, the Hypervisor framework on macOS, Hyper-V on Windows. A VM, not a container, is the right sandbox here for one blunt reason: this track deliberately breaks the kernel's networking, mount and cgroup state, and a container shares the host's kernel with you.

      • Under the hood

        Spell out 24.04 and never write multipass launch lts. The lts alias is a moving target — it now resolves to Ubuntu 26.04, which would quietly hand you a different kernel and different defaults from every command output pasted in this track. Every lab here is authored against 24.04 "noble": cgroups v2 only, iptables running on the nftables backend, and systemd-resolved answering on 127.0.0.53.

      Toolbelt & safety

      • Under the hood

        build-essential is the C compiler and headers — B2 and B3 write small C programs, because some things (a signal handler, a raw write(2)) cannot be shown honestly in a shell. strace/ltrace are the X-ray machines of B3; iproute2, tcpdump and nftables are all of B8–B9; socat is how B9 talks over a unix socket.

      • Under the hood

        Do this now, while the stakes are zero. The single biggest thing that slows beginners down is not difficulty — it is the quiet fear of breaking the machine, which makes them skip exactly the destructive labs that teach the most. A restore you have already done once stops being scary.

      • Under the hood

        The sections matter because the same name lives in several of them: man 1 printf is the shell command, man 3 printf is the C function; man 8 mount is the tool you type, man 2 mount is the system call it makes. Section 7 is the underrated one — man 7 namespaces, man 7 capabilities and man 7 cgroups are the primary sources for three whole modules of this track.

      B1 — Shell, files, and the filesystem tree

      16 cp · ~4 h

      The shell is not a lesser version of a graphical file manager — it is a language for composing small programs into an answer that did not exist before you typed it. This module is the vocabulary every other module speaks.

      B0 — you need the sandbox VM and a snapshot you trust before you start deleting things in it.

      Paths and the tree

      • Under the hood

        Linux has one tree with a single root /, not a drive letter per device. A second disk does not become D: — it gets grafted onto a directory somewhere in the same tree, which is what "mounting" means and what B5 takes apart. That single-tree design is also why a container can be handed a completely different / and never notice.

      • Under the hood

        This layout is a written standard (the FHS), which is why you can sit down at an unfamiliar Linux box and still know where to look. The distinction that pays off later: /etc is what you set, /var is what the machine accumulates — logs, spools, databases, and on a Kubernetes node, /var/lib/kubelet and /var/lib/containerd.

      • Under the hood

        "Everything is a file" is meant literally: a terminal (c, character device), a disk (b, block device), a socket (s) and a pipe (p) all appear in the same namespace with the same permission bits, and all answer the same read/write system calls. That uniformity is what lets B3's strace show one program talking to a file, a terminal and a network connection in identical syntax.

      Composing tools

      • Under the hood

        Those numbers are file descriptors, the small integers the kernel hands out for anything a process has open. Keeping errors on a different descriptor from results is the whole reason a pipeline can carry clean data while still telling you something went wrong — and it is why a script that merges the two with &> becomes so much harder to debug.

      • Under the hood

        Order matters and catches everybody once: cmd > f 2>&1 sends both streams to the file, but cmd 2>&1 > f sends errors to the old stdout (your terminal) because 2>&1 copies where stdout points at that moment, before the redirect moves it.

      • Under the hood

        A pipe is a kernel buffer between two processes that run at the same time, not one after the other. That is why … | head -5 can finish early: head exits, the writer's next write gets SIGPIPE, and the whole pipeline stops rather than computing output nobody will read.

      • Under the hood

        The same characters mean different things in each: glob * is "any run of characters", regex * is "zero or more of the previous item" — so the regex equivalent of glob * is .*. And because the shell expands globs first, grep *.log file silently searches for the name of the first matching file. Quote patterns you intend a program to read.

      • Under the hood

        uniq only collapses adjacent duplicates, so it is nearly always sort | uniq -c | sort -rn. find … -print0 | xargs -0 is the pairing that survives filenames with spaces. And awk '{print $2}' replaces most of what people reach for cut for, because it treats any run of whitespace as one separator.

      The shell itself

      • Under the hood

        The environment is a plain list of KEY=VALUE strings copied into a child process when it is created — that is the entire mechanism. It is a one-way copy at creation time, so a child can never change its parent's environment, and a running process's environment is frozen except by its own action. You can read any process's copy at /proc/PID/environ, which is exactly why B12 warns against putting secrets there.

      • Under the hood

        The rule that prevents most shell bugs: quote every variable expansion — "$f", not $f. Unquoted, the shell takes the variable's value, splits it on whitespace and glob-expands the pieces, so a filename with a space silently becomes two arguments and a value containing * becomes a directory listing.

      • Under the hood

        The shebang is not shell magic: the two bytes #! are read by the kernel in execve, which then runs the named interpreter with your file as its argument. That is why a script with Windows line endings fails with a baffling bad interpreter: No such file or directory — the carriage return is part of the interpreter's path. Exit codes come back in B2 with the 128 + signal convention that explains 137.

      B2 — Processes, threads, signals

      18 cp · ~6 h

      A container is a process. Everything Kubernetes does to a workload — start it, watch it, ask it to stop, give up and kill it, report why it died — is something Linux does to a process, and this module is where you learn to narrate all of it.

      B1 — the labs read /proc with the same tools you just learned, and the project is a shell script plus a small C program.

      What a process is

      • Under the hood

        Everything later in this track is a modification of that list. B4 changes the credentials, B5 and B6 change what the file descriptors and mount table can reach, B6 changes which PIDs are even visible, B7 caps how much memory and CPU the address space may use. There is no separate "container" object in the kernel — a container is a process with all four of those things adjusted.

      • Under the hood

        VSZ is address space the process has reserved; RSS is physical memory it is actually occupying. A process can map a huge file or reserve a large heap and touch almost none of it. Memory limits in B7 are enforced against something closer to RSS, which is why a scary VSZ is usually not the problem you are looking for.

      • Under the hood

        /proc is not a directory of files on a disk — it is a view of kernel data structures rendered as text on demand, which B3 takes apart properly. It is also the honest answer to "what is this process really doing": ls -l /proc/PID/fd shows every open file, socket and pipe, and it is how you find the process still holding a deleted 40 GB log file.

      Lifecycle

      • Under the hood

        The split looks redundant until you see what happens in between: the child is briefly a copy of the parent and can adjust itself — redirect its file descriptors, drop privileges, join a cgroup, enter namespaces — and only then exec the target program. Every shell redirect, and every container start, happens in that gap between fork and exec.

      • Under the hood

        D is the one that matters at 3 a.m.: the process is inside a kernel call that refuses to be interrupted — almost always storage or network I/O — and it will not respond to SIGKILL, because there is no safe point at which to deliver it. A pile of D-state processes means the layer underneath is wedged; killing them is not an option, so you go fix the disk or the mount.

      • Under the hood

        This is where 137 comes from — 128 + 9, killed by SIGKILL — and 143 is 128 + 15, killed by SIGTERM. You will see 137 constantly on Kubernetes nodes, and it means exactly what it means here: something outside the process decided it should stop existing. B7 shows you the something.

      • Under the hood

        A zombie is a receipt: the kernel keeps the exit status until someone reads it. Thousands of them mean a parent that has stopped reaping, and the real damage is PID exhaustion, not memory. This is precisely the "PID 1 problem" in containers — a process that was never written to be an init has no idea it inherited re-parented orphans, so they pile up. It is also why the pod spec has a shareProcessNamespace option and why runtimes offer to inject a tiny init.

      Threads

      • Under the hood

        The kernel does not have two different objects for "process" and "thread" — both are created by the same clone() call, and the flags decide how much is shared. That single call is also how B6's namespaces get created, so the machinery behind threading and the machinery behind containers is literally the same system call with different arguments.

      Signals

      • Under the hood

        A signal is a knock on the door with a standard meaning, not a message with a body — you get the number and nothing else. Almost all of the meaning is convention: nothing in the kernel forces HUP to mean "reload", it is simply what daemons agreed to do with it.

      • Under the hood

        They are uncatchable by design: the kernel needs at least one way to remove a process that has stopped cooperating, and it is handled entirely inside the kernel without ever running the process's code. That is also why a KILLed process gets no chance to flush buffers, remove its lock file, or finish a request — and why every graceful-shutdown design has to happen during the TERM that comes first.

      • Under the hood

        This is the exact contract Kubernetes uses to stop a pod: SIGTERM first, then a grace period (30 s by default), then SIGKILL. A container that ignores TERM is not "robust" — it just guarantees every one of its shutdowns takes the full grace period and then loses whatever it had in flight, showing up as exit code 137 in the pod's status.

      B3 — Kernel space, user space, system calls

      14 cp · ~4 h

      There is a hard line down the middle of a Linux machine. Your programs live on one side, the hardware lives on the other, and the only way across is a system call. Once you can see those crossings, you can find out what a program is really doing when its logs will not say.

      B2 — the syscalls you will read (clone, execve, wait4) are the lifecycle you just learned, now seen from the kernel's side.

      The boundary

      • Under the hood

        Think of the kernel as the hotel's engineering staff — the only people allowed to touch the wiring. A guest who wants hot water does not open the boiler; they file a work order. That work order is the system call, and the reason for the rule is that one buggy guest must not be able to burn the building down.

      • Under the hood

        There are only a few hundred system calls in total (man 2 syscalls) — a small, stable, documented interface, which is why a binary compiled a decade ago still runs. It is also what seccomp filters in B12: not "what may this program access" but "which of these few hundred doors may it even knock on".

      • Under the hood

        The gap between the two explains a classic confusion: a hundred printf calls may produce a single write syscall, because libc buffers. That is also why a program killed by SIGKILL loses output it "already printed" — the text was sitting in a userspace buffer that never reached the kernel.

      Reading syscalls

      • Under the hood

        Descriptors are per-process and allocated as the lowest free number, which is exactly why 0/1/2 are stdin/stdout/stderr and why the shell's redirection trick works: close 1, open the file, and the file lands in slot 1 — so the program writes to the file without knowing anything changed.

      Kernel interfaces

      • Under the hood

        This is why the standard tools have no special powers: ps reads /proc/*/stat, free reads /proc/meminfo, ss reads /proc/net/*. Anything they can tell you, you can read directly — which matters when you are inside a stripped-down container image that has none of them installed.

      • Under the hood

        Two of these show up on every Kubernetes node: net.ipv4.ip_forward=1, without which the node cannot route pod traffic at all (B9 builds exactly that by hand), and fs.inotify.max_user_instances, whose default is low enough that a busy node runs out of file watches and the kubelet starts failing in ways that look like anything but a sysctl.

      • Under the hood

        The OOM killer's verdict lands here and nowhere else. When B7 kills a memory hog, the application's own logs stop mid-sentence with no error, and the only record of what happened — which process, how much memory, which cgroup — is in dmesg. "The app died silently" almost always means "nobody read the ring buffer".

      B4 — Users, groups, permissions, capabilities

      15 cp · ~4 h

      "Permission denied" is the most common error on any Linux system, and it is never mysterious once you know which of three or four separate checks failed. This module also decomposes root, which is what makes "root inside a container" a survivable sentence.

      B1 for paths and ls -l; B2 helps, because permissions are checked against a process's credentials, not a person's.

      Identity

      • Under the hood

        There is also a real uid versus an effective uid: the real one is who you are, the effective one is who you are being checked as right now. The gap between them is the entire setuid mechanism later in this module — and it is why ps can show a process running as root that was started by you.

      • Under the hood

        Giving every service its own uid is blast-radius control: if nginx is compromised, the attacker inherits exactly what uid www-data can reach and nothing more. This is the same reasoning behind runAsUser and runAsNonRoot in a pod's securityContext — you are choosing which uid the compromised process would get.

      Permissions

      • Under the hood

        The consequence that generates most real "permission denied" tickets: to open /a/b/c/file you need x on /a, /a/b and /a/b/c, whatever the file's own mode says. One missing traverse bit three levels up denies a file that looks perfectly readable. Note also that deleting a file needs w on its directory, not on the file — which is why you can delete a file you cannot read.

      • Under the hood

        passwd has to update /etc/shadow, which is mode 640 root:shadow — so an unprivileged user genuinely cannot write their own password. Setuid solves it by letting a specific vetted program run with the owner's uid. That is also why every setuid binary is a security liability: a bug in one is a direct path to root, which is exactly what capabilities in this module and no_new_privs are designed to contain.

      Capabilities

      • Under the hood

        Think of a keyring of individually-issued permits rather than one master key. The practical value is subtraction: a web server needs CAP_NET_BIND_SERVICE and nothing else, so a compromise of it yields the ability to bind a low port — not the ability to mount filesystems, load kernel modules and read every file on the box.

      • Under the hood

        Container runtimes drop most of the bounding set by default, so uid 0 inside keeps a short list — enough to chown its own files, nowhere near enough to reconfigure the host's network or load a module. Combine that with the user namespace of B6, which maps container uid 0 to an unprivileged host uid, and "root" inside becomes a genuinely different, powerless identity outside.

      B5 — Files for real: VFS, storage, overlayfs

      18 cp · ~6 h

      By the end of this module you will have built a container image by hand — three stacked directories and one mount command — and found the physical file that a write inside a running container actually creates. Image layers stop being a diagram and become something you can ls.

      B1 for the tree and stat; B4, because half of "the mount failed" is really a permissions answer.

      Devices and filesystems

      • Under the hood

        Mounting hides rather than deletes: whatever was in the mount point is still there, inaccessible until you unmount. This is the mechanism behind every volume in Kubernetes — the container's image supplies /data, the kubelet mounts something else over it, and the image's version is simply covered up. It also explains "my ConfigMap replaced my directory".

      The VFS view

      • Under the hood

        Unlinking removes a name. The inode and its blocks are freed only when the link count reaches zero and no process still has it open — so deleting a 40 GB log that a daemon is writing frees nothing at all until you restart the daemon. This is the single most common "I deleted it and df did not change" incident, on nodes as much as anywhere.

      Special mounts

      • Under the hood

        This is exactly what a Kubernetes emptyDir with medium: Memory is, and the sting is in the last clause: tmpfs contents count towards the pod's memory limit, so filling one gets you OOM-killed rather than a disk-full error.

      • Under the hood

        Every hostPath volume and every file a container runtime injects (/etc/hosts, /etc/resolv.conf, a projected ServiceAccount token) is a bind mount. Once you have made one by hand, "the container sees the host's directory" stops being magic — it is this one command.

      • Under the hood

        This is why a node's CSI driver runs with mountPropagation: Bidirectional: it mounts volumes inside its own container and needs them to appear on the host so other pods can use them. Get it wrong and the mount exists in exactly one namespace and everything downstream sees an empty directory — a failure with no error message at all.

      Overlayfs

      • Under the hood

        A stack of transparent sheets: you read through the whole stack, top sheet winning where two disagree, but you only ever write on the top one. That single idea is why a hundred containers from the same image cost one copy of the image plus a hundred nearly-empty upper directories.

      • Under the hood

        This is why RUN rm -rf /something-secret in a Dockerfile does not shrink the image or remove the secret: the lower layer still contains the file and anyone with the image can read that layer directly. The deletion is a note on the top sheet, not an erasure.

      B6 — Isolation I: namespaces, chroot, pivot_root

      15 cp · ~6 h

      This is the module where you build something that is honestly a container, in about twenty-five lines, with no container runtime installed. A namespace gives a process its own private copy of one kernel resource — and that, repeated eight times, is the whole trick.

      B2 (processes and clone), B3 (syscalls and /proc) and B5 (mounts, bind mounts, overlayfs — the rootfs you built there is the one you boot here).

      The idea

      • Under the hood

        A hotel room is the metaphor to carry through the rest of this track: same building, same plumbing, same staff — but your own closet, your own phone line, and no view of the other rooms' guest lists. Nothing is duplicated; the kernel just consults a different table depending on which room the asking process is in.

      • Under the hood

        Those inode numbers are the whole reason "containers in a pod share a network namespace" is checkable rather than a claim: run readlink /proc/PID/ns/net for each container's PID and the numbers are identical. A namespace also lives exactly as long as something references it — a process inside it, or a bind mount of that symlink — which is how you keep an empty one alive, and how you leak one.

      Hands-on, one namespace at a time

      • Under the hood

        PID 1 in a namespace inherits init's duties — it reaps orphans, and signals sent to it have no default action, so a process that installed no SIGTERM handler simply ignores it. That is B2's PID 1 problem, and it is why "my container will not stop, it always takes the full 30 seconds" is so common.

      • Under the hood

        A new PID namespace does not change /proc, which is still the host's procfs mounted from the old mount namespace — so ps keeps listing host processes until you also unshare the mount namespace and remount /proc. Two namespaces cooperate to produce one apparently simple effect, and that is exactly the drill at the end of this module.

      • Under the hood

        User namespaces are how rootless containers work: uid 0 inside maps to some unprivileged uid outside, so "root" gets full authority over its own namespace and none over the host. They are also historically a rich source of privilege-escalation bugs, because they let unprivileged users reach kernel code that previously required root — which is precisely why 24.04 restricts them by default. That trade-off, not the workaround, is the lesson.

      Roots and entering

      • Under the hood

        This is what kubectl exec actually is — no remote shell daemon, no SSH: the kubelet asks the runtime to start a process that setns()-es into the target container's namespaces. It is also the node-level debugging move that works when everything else has failed, and the reason ephemeral debug containers can inspect a container that ships no shell of its own.

      B7 — Isolation II: cgroups v2

      14 cp · ~4 h

      Namespaces hide things; cgroups limit them. This module is where "requests and limits", QoS classes, throttling and exit code 137 stop being Kubernetes vocabulary and become files you can write a number into.

      B2 (processes, exit codes, signals) and B6 — cgroups and namespaces are the two halves of a container and are constantly confused with each other.

      The model

      • Under the hood

        The metered power and water of the hotel room, with a cap the front desk sets. It also explains a real-world surprise: a JVM or nproc inside a container often reports the host's full CPU count, because the cgroup limits how much CPU time it gets without changing what it can see. Sizing thread pools from that number is a classic outage.

      CPU

      • Under the hood

        This is a Kubernetes request. A request does not reserve or cap anything at runtime — it is used by the scheduler to decide where the pod fits, and translated into a CPU weight that only bites when the node is busy. "My pod requested 1 CPU but is using 3" is not a bug.

      • Under the hood

        This is a Kubernetes limit, and the units line up exactly: limits.cpu: "500m" becomes 50000 100000. The period is the part people miss — a process that would run 100 ms straight is instead stopped dead once it burns its quota and left idle until the next period starts.

      Memory, pids, io

      • Under the hood

        137 is 128 + 9 from B2.8: the process was SIGKILLed, so it got no chance to log anything, which is why its own logs just stop mid-sentence. Exactly this sequence produces OOMKilled in a pod's status — the kubelet is reading the same cgroup event you just read. Memory pressure kills; CPU pressure only slows.

      In practice

      • Under the hood

        On a Kubernetes node this same tree carries the pods: kubepods.slice splits into burstable and besteffort children, one cgroup per pod, one per container. QoS classes are not an abstraction layer — they are literally which sub-slice your pod's directory is created under, and therefore what gets squeezed first when the node runs short.

      B8 — Networking I: the real network

      18 cp · ~6 h

      Most "Kubernetes networking is hard" is really "ordinary networking was never learned". This module turns the layers into a fixed diagnostic ladder — link, address, route, filter, listener — that you climb the same way every time, on any machine, forever.

      B1 (you will live in ip, ss and grep output) and B3 — a socket is a file descriptor, and strace on curl is one of the labs.

      Layers and addresses

      • Under the hood

        Layering exists so each level can be replaced without disturbing the others — the same TCP works over Wi-Fi, fibre or a veth pair. For debugging, the payoff is that you can always bisect: prove one layer works and every layer below it is exonerated. That is why the ladder in B8.16 is climbed from the bottom, and why "it's DNS" is only a legitimate answer after the three layers under DNS have been cleared.

      • Under the hood

        ARP is a broadcast to the local segment only — nothing routes it. That single fact is what "on the same subnet" means physically, and it is why two machines with correct IPs but a wrong netmask can be mutually invisible: each believes the other is remote and hands the packet to a gateway instead of asking for its MAC. A FAILED or missing neighbour entry is a link-layer answer to what looked like an IP problem.

      • Under the hood

        The mask is just "how many leading bits are the neighbourhood, the rest is the house number". You will do this arithmetic constantly in Kubernetes without being asked to: the pod CIDR (10.244.0.0/16) is carved into a /24 per node, the Service CIDR is a separate range that exists only inside iptables rules, and "pods can't reach Services" is often just the two ranges overlapping something on the physical network.

      • Under the hood

        Two different "up"s catch people: UP is the administrative state you set, LOWER_UP means the driver sees an actual carrier. An interface can be administratively up with no link, which reads as a routing problem until you look. Note also that ifconfig is not merely old — it cannot see multiple addresses, network namespaces or many modern attributes, so it will lie to you by omission on exactly the machines this track builds.

      Routing

      • Under the hood

        "Most specific wins" is the whole algorithm: a /32 beats a /24 beats the /0 default. Nothing here has any idea whether the destination exists — routing only answers "out of which interface, to which next hop". This is exactly how a CNI plugin works in B9 and in Kubernetes: it does not "connect" anything, it adds routes so packets for another node's pod CIDR leave by the right door.

      • Under the hood

        ICMP is a different protocol from TCP and UDP, so firewalls treat it separately — a host that ignores ping may be serving traffic perfectly, and a host that answers ping may have every port closed. "Ping works but nothing else does" is therefore not a paradox, it is a filter, and it is the single most common false conclusion in network debugging.

      Transport

      • Under the hood

        The handshake is why a refused connection is instant (a RST comes straight back) while a filtered one hangs until timeout (the SYN vanished and nothing answers). Keep that pair: "connection refused" means you reached the machine and nothing was listening; "timed out" means something ate the packet. In Kubernetes those two map to a Service with no ready endpoints versus a NetworkPolicy or missing route.

      • Under the hood

        CLOSE-WAIT means the peer said "I'm done" and your side never answered — the fix is always in your code, never in the kernel. TIME-WAIT is the deliberate two-minute pause the closing side holds so late duplicate packets from the old connection cannot be mistaken for a new one on the same 4-tuple. Thousands of them are the expected state of a busy client, not a leak.

      • Under the hood

        The 4-tuple is why NAT in B9 must keep a table — rewriting an address means every reply has to be un-rewritten against the right entry. It is also the ceiling on outbound connections to one destination: a single source IP has ~28 000 ephemeral ports, which is exactly the limit busy egress NAT gateways hit and report as mysterious connection failures under load.

      • Under the hood

        The loopback-versus-wildcard bind is the fifth rung of the ladder and a permanent source of "the port is open but I can't reach it". It reappears verbatim in Kubernetes: a container that binds 127.0.0.1 is unreachable from any other pod no matter how correct the Service is, because the pod's loopback belongs to its own network namespace.

      Observing

      • Under the hood

        Always pass -nn: without it tcpdump resolves every address, which both slows the capture and can generate the very DNS traffic you are trying to study. The deeper skill is choosing the interface — in B9 the same connection looks completely different on the veth inside a namespace, on the bridge and on the uplink, and comparing those three captures is how you find which hop drops a packet.

      • Under the hood

        Doing it by hand kills the idea that HTTP is magic — it is text over a TCP connection, which is why Host: is mandatory (one IP serves many sites) and why a stray blank line ends your headers early. Every Kubernetes probe, every apiserver call and every kubectl command is this same exchange with TLS wrapped around it.

      • Under the hood

        The ladder's value is not that each rung is clever — it is that it stops you skipping to the most interesting hypothesis. Guessing feels faster and is reliably slower, because a wrong guess costs a full investigation. This is the same discipline as W6's triage template on the CKA track, one layer lower down.

      B9 — Networking II: virtual networking & packet filtering

      20 cp · ~8 h

      The heaviest module in the track, and the one that pays best: by the end you will have built a pod network and a load-balanced Service by hand, out of namespaces, veth pairs, a bridge and NAT rules — before either word has been used. Plan two sittings.

      B6 (network namespaces) and B8 (addresses, routes, TCP, tcpdump). Snapshot the VM first — this module deliberately breaks its own connectivity.

      Virtual wiring

      • Under the hood

        "Its own port space" is the sentence to hold on to: two processes in different network namespaces can both bind :80 without conflict, which is precisely why every pod gets a whole IP instead of a share of the node's ports. A namespace starts with only a down loopback, so the first surprise is that even ping 127.0.0.1 fails until you ip link set lo up.

      • Under the hood

        The hotel phone wire from the analogy: the room's handset is one end, the socket in the wiring closet is the other. This is exactly what a container runtime creates for every pod — one end named eth0 inside the pod's namespace, the other hanging on the node's bridge with a name like vethb3f1a2. Deleting either end destroys both; there is no such thing as half a veth pair.

      • Under the hood

        The switchboard the room wires plug into. On a Kubernetes node with a bridge-style CNI this device really exists and is usually called cni0 or docker0 — bridge link lists one port per running pod. Give the bridge itself an IP and it stops being only a switch and becomes the gateway those pods route through.

      • Under the hood

        You have just made three decisions a CNI plugin makes for every pod it creates: which subnet this node owns, which address this workload gets from it, and which gateway it points at. IPAM — the "IP address management" half of any CNI plugin — is nothing more grand than keeping a record of which addresses in that subnet are already handed out.

      • Under the hood

        Capturing at two points is the technique the rest of this module depends on. When a packet appears at hop 1 and not at hop 2, you have located the fault to a single device without a single guess — which is exactly how you will later find out whether a pod's traffic died inside the pod, on the bridge, or in a NAT rule.

      Getting out

      • Under the hood

        This is the same sysctl every Kubernetes install guide makes you set on every node, for the same reason: a node forwards traffic between pod veths and the outside world, so it is a router whether or not anyone calls it one. Forgetting it produces the classic signature — pods can reach each other on the local bridge but nothing beyond the node.

      • Under the hood

        Everything in a cluster's data path is one of these three. Pods reaching the internet is MASQUERADE; a Service's ClusterIP is DNAT to a real pod IP; a NodePort is DNAT on the way in. Once you can name which rewrite is supposed to happen, "the packet arrived with an address I didn't expect" becomes a specific question about a specific rule.

      • Under the hood

        The switchboard's log of calls currently in progress. Two consequences you will meet for real: a NAT rule changed mid-connection does not affect flows already in the table, which is why a fixed rule appears not to work until you retry with a fresh connection; and the table has a finite size, so a node under heavy churn logs nf_conntrack: table full, dropping packet and drops traffic that every rule says should pass.

      Netfilter

      • Under the hood

        Nearly every "my rule does nothing" is a hook mistake. Traffic destined for a container never traverses input on the host — it is forwarded — so a rule in the wrong chain is inspected by no packet you care about. The routing decision sits between prerouting and the split, which is why DNAT has to happen in prerouting: it must change the destination before the kernel decides where the packet goes.

      • Under the hood

        Also learn the difference in how traffic is refused: DROP discards silently and the client waits out its timeout, while REJECT sends an ICMP error and the client fails instantly. The symptom timing tells you which one you are behind before you have read a single rule — and it is the same reading that distinguishes a NetworkPolicy (drops, hangs) from a missing endpoint (refuses, instant).

      • Under the hood

        This matters the moment something appears to be missing: rules added with the nft syntax are invisible to a plain iptables -L and vice versa in places, so a half-view convinces you a rule is absent when it is merely in the other dialect. Kubernetes lives in exactly this transition — kube-proxy still writes iptables-style rules on most clusters, which land in nftables underneath, which is why nft list ruleset on a node shows thousands of KUBE- chains.

      • Under the hood

        Packet counters turn firewall debugging from argument into measurement: a rule that "should" match but whose counter never moves is being shadowed by something above it, or is in a chain this traffic never enters. This is the same instinct as reading a kube-proxy chain top-down to find which rule claimed the packet before the one you were looking at.

      Load balancing and sockets

      • Under the hood

        This is kube-proxy's iptables mode — not an analogy for it. A ClusterIP is an address that exists nowhere: no interface holds it, nothing answers ARP for it, and the only thing making it work is a DNAT rule on every node rewriting it to a real pod IP. The probabilities are chained (1/3, then 1/2, then 1) so N backends come out evenly, which is exactly what you will read in KUBE-SVC-* chains later.

      • Under the hood

        You have just built a NodePort by hand. The same rule shape, with a port from the 30000–32767 range and a jump into the Service's chain, is what type: NodePort produces on every node in a cluster — including nodes that run none of the Service's pods, which is why any node's IP works.

      • Under the hood

        No IP stack is involved at all — the kernel copies bytes between two processes on the same machine, which is both faster and, crucially, governed by file permissions rather than by firewall rules. That is why so much of the plumbing on a node talks over sockets in /run instead of over localhost ports.

      • Under the hood

        Now apply that reasoning to /run/containerd/containerd.sock: anyone who can write to that socket can start a container with the host's filesystem mounted and every capability granted, so access to the runtime socket is equivalent to root on the node. This is why "just mount the docker socket into the pod" is a full cluster compromise and not a convenience, and it is the concrete reason B12 spends time on trust boundaries.

      B10 — Names: DNS and resolution

      13 cp · ~4 h

      "It's always DNS" is a joke because DNS is the one layer most people never actually learned. Take it apart here — the client library, the resolver, the records, the caching — and the cluster's search-path ladder and ndots:5 will read as ordinary configuration rather than folklore.

      B8 (you need UDP, ports and tcpdump to watch a query leave). B9 is helpful — the project resolves names from inside a namespace you built there.

      The client side

      • Under the hood

        This explains the single most confusing DNS symptom: ping myhost works while dig myhost returns nothing. dig is a pure DNS tool that talks to a nameserver directly; ping goes through the resolver library, which may have answered from /etc/hosts, from mDNS, or from systemd-resolved without a DNS query ever leaving the box. They are not testing the same thing, and only one of them is testing what your application does.

      • Under the hood

        These three lines are the whole client configuration, and the pod-level version in Kubernetes is generated from exactly the same fields. A pod's file carries search default.svc.cluster.local svc.cluster.local cluster.local and options ndots:5 — meaning any name with fewer than five dots is tried against every search domain before being tried literally. That is why curl google.com from inside a pod fires four failing queries before the one that works.

      • Under the hood

        The trailing dot means "fully qualified, ask exactly this" — the one reliable way to take the search list out of an experiment. Keep it as a diagnostic: if a name resolves with the dot and fails without it, the fault is in the search path, not in the zone or the server.

      The system side

      • Under the hood

        Nobody holds the whole tree; every level only knows who to ask next. That design is why DNS survives at internet scale, and why caching matters so much — without it every lookup would restart at the root. CoreDNS in a cluster plays both roles at once: authoritative for cluster.local, and a recursive forwarder for everything else.

      • Under the hood

        The hotel operator who looks the name up before dialling. The practical consequences: tcpdump port 53 on the uplink shows nothing for a cached answer, because the query never left the box; and editing /etc/resolv.conf directly appears to work until something regenerates it. Configure systemd-resolved, or point the file elsewhere deliberately, as B10.12 does.

      Records and caching

      • Under the hood

        SRV is the one worth remembering: it is how a headless Service publishes both the address and the port of each backing pod, which is how StatefulSet members find each other without any load balancer in between. CNAME chains matter too — an ExternalName Service is literally just a CNAME answer from CoreDNS.

      • Under the hood

        The status field alone routes your investigation: NXDOMAIN means the server is healthy and the name genuinely does not exist (check the name and the search path), while SERVFAIL means the server failed trying (check the server and its upstreams). Two very different problems that both present to the user as "it doesn't resolve".

      • Under the hood

        TTL is why a DNS change is not instant no matter how confident the person who made it is: every resolver between you and the zone may serve the old answer until its copy expires. Lower the TTL before a planned change, not during it — by the time you need the short TTL, the long one is already cached everywhere.

      • Under the hood

        This is the real explanation behind "it started working on its own after a few minutes". A pod that starts before its Service exists can cache the failure and keep failing after the Service is created — which is why an application's startup retries should re-resolve rather than resolve once and give up.

      B11 — Service management: systemd, logs, boot

      14 cp · ~4 h

      Something has to start things at boot, restart them when they die, and collect what they printed. On a Kubernetes node that something is systemd — and the kubelet, the one component nobody can schedule as a pod, is an ordinary unit you will read with the same two commands you learn here.

      B2 (processes, signals, exit codes — a unit's status is those three, formatted) and B7, because every unit is placed in a cgroup and systemd is what puts it there.

      Units

      • Under the hood

        The building superintendent: they get the boiler running before any tenant office can open. Note the connection back to B2.9 — since PID 1 adopts orphans, it must reap them, which is exactly the job a container's PID 1 usually was not written to do. systemd is also a supervisor, a logger, a cgroup manager and a socket activator, which is why it is both powerful and controversial.

      • Under the hood

        Location decides precedence: /lib/systemd/system is the vendor's copy and gets replaced by package updates, while /etc/systemd/system is yours and wins. Editing the vendor file is the classic mistake that survives until the next apt upgrade quietly reverts it.

      • Under the hood

        Type= is the one that produces baffling failures: it tells systemd how to know the service is up. Declare simple for a daemon that forks into the background and systemd will supervise a parent that has already exited, then restart it forever. The restart-with-backoff behaviour is the same loop you wrote by hand in B2.17, and the same contract a kubelet applies to a container's restartPolicy.

      • Under the hood

        Nearly every "works on reboot, fails at boot" race is this pair used as if it were one setting. The kubelet unit is the working example: it wants the container runtime, and getting the ordering right is why a node comes up cleanly rather than logging runtime-connection errors for the first minute.

      Operating

      • Under the hood

        "I started it and it was gone after the reboot" is start without enable; "I enabled it and nothing happened" is the reverse. systemctl cat unit is the underrated one — it prints the effective unit including every drop-in, which is the only honest answer to "what is this service actually configured to do".

      • Under the hood

        Drop-ins merge over the vendor unit, so upgrades keep working and your change survives. One trap worth meeting now: to replace a list-valued key like ExecStart you must first clear it with an empty assignment (ExecStart=) or you end up with two. This is exactly the mechanism kubeadm uses to configure the kubelet — 10-kubeadm.conf in the kubelet unit's drop-in directory.

      • Under the hood

        Exit code means read the application's logs; a signal means look outside it — the OOM killer from B7.9, an operator, or a timeout systemd itself enforced. This is the same fork in the road as a pod's Error versus OOMKilled, read one layer down.

      Logs and boot

      • Under the hood

        The journal is a structured binary store, not a text file, which is why grep /var/log often finds nothing on a modern box. It captures a service's stdout and stderr automatically — so a program that just prints to the screen is already logging correctly, exactly like a container whose logs the kubelet collects from its stdout.

      • Under the hood

        Targets replaced runlevels with something more honest — multi-user.target is not a number, it is a set of units that must be running. And systemd-cgls on a Kubernetes node is the whole story in one screen: system.slice holding kubelet and containerd, next to kubepods.slice holding one cgroup per pod.

      B12 — Security fundamentals

      16 cp · ~5 h

      A Kubernetes cluster is held together by certificates: every component authenticates to the apiserver with one, and every confusing auth error is a PKI error underneath. Run your own certificate authority here, issue certificates, and break your own mutual TLS four ways until each error message means something specific.

      B4 (users, file permissions — key material is protected by nothing else) and B8, since the TLS labs are ordinary TCP connections you will watch with the tools from that module.

      Framing

      • Under the hood

        Security decisions are meaningless without the question "against whom?" — a control that stops an internet scanner may do nothing about a process already running on the box. Naming the boundary is the whole exercise: B9.18's runtime socket is a trust boundary, and once it is drawn on paper, "mount the docker socket into a pod" is obviously a boundary crossing rather than a convenience.

      • Under the hood

        This split is the shape of the apiserver's whole request pipeline: authn, then authz, then admission. It is also how you triage an error in one step — 401 Unauthorized means the certificate or token was not accepted at all (a PKI problem), while 403 Forbidden means you were recognized perfectly and RBAC said no. Reading the wrong one sends people to regenerate certificates when they needed a RoleBinding.

      Crypto in plain terms

      • Under the hood

        Base64 is encoding: it protects nothing, it exists so binary bytes survive a text field. This is the entire answer to "why are Kubernetes Secrets only base64?" — they are not encrypted at rest unless you configure an encryption provider, so anyone who can read the Secret object, or etcd, can read the value with one command. Treating base64 as security is the single most common beginner misreading in the whole ecosystem.

      • Under the hood

        Real protocols use both: TLS does an asymmetric handshake purely to agree on a symmetric session key, because asymmetric maths is far too slow for bulk traffic. That is the whole architecture of every HTTPS connection you have ever made — expensive introduction, cheap conversation.

      • Under the hood

        authorized_keys on the server is a list of public keys, and the login is a challenge the client answers by signing — your private key never crosses the network, which is the point. The agent is the same trade-off every credential cache makes: convenience in exchange for a decrypted secret sitting in a running process's memory.

      PKI hands-on

      • Under the hood

        The notary everyone already trusts. A CA is not special software — it is a keypair plus the willingness of others to trust its signature. kubeadm init does exactly what you just did, writing the cluster CA to /etc/kubernetes/pki/ca.crt and ca.key; possession of that key file is functionally cluster-admin forever, which is why it never leaves the control-plane node.

      • Under the hood

        CN-as-hostname was deprecated years ago and is now simply ignored for name matching. This is the direct explanation for the apiserver's long SAN list — the cluster IP, the node IP, kubernetes, kubernetes.default, kubernetes.default.svc.cluster.local — and for the classic failure after adding a load balancer in front of it: a new name nobody put in the SAN list.

      • Under the hood

        Certificate expiry is the outage that arrives with no deploy and no change — the machine was fine yesterday. On a kubeadm cluster the component certificates last one year, which is why kubeadm certs check-expiration exists and why a cluster left alone for thirteen months stops accepting its own kubelets.

      • Under the hood

        You have just built the cluster's authentication model. Every Kubernetes component authenticates with a client certificate, and the apiserver reads the identity straight out of it: the certificate's CN becomes the username and its O (organization) fields become the groups. That is why the admin kubeconfig's certificate says O=kubeadm:cluster-admins — the group name RBAC then binds to.

      Handling secrets

      • Under the hood

        Environment variables also get inherited by every child process and are routinely dumped by crash handlers and debug logging. That is the concrete argument for mounting a Kubernetes Secret as a file rather than as env: — same secret, far smaller blast radius, and the file can be rotated without restarting the pod.

      • Under the hood

        Both are defense in depth around B3 and B4: even a process that has fooled the permission check still has to get past a profile that says this program never opens that path, or a filter that says this container never calls mount. Kubernetes surfaces both through securityContext — seccompProfile: RuntimeDefault is the same mechanism, chosen by the runtime instead of by you.

      B13 — Distributed systems fundamentals

      15 cp · ~5 h

      Everything so far happened on one machine. Add a second and a category of problem appears that has no single-machine equivalent: you can no longer tell "it failed" apart from "it is slow". Kubernetes is a set of answers to that problem, so learn the problem first.

      B2 (processes and signals — the reconciler project supervises real ones), B8 (what a network can do to a message) and B11, so the reconciler can run as a unit.

      What changes

      • Under the hood

        This one fact generates most of the rest of the module. Because you cannot know, you must retry; because you retry, the operation must be safe to run twice (B13.10); because you cannot wait forever, you need a timeout; and because a timeout is a guess, you will sometimes declare a healthy peer dead. Every distributed system is built out of these compromises, including this one.

      • Under the hood

        This is why distributed systems order events with logical counters rather than wall-clock time. etcd's revision and every Kubernetes object's resourceVersion are exactly that: monotonically increasing numbers whose only job is to say what came after what. They are not timestamps and cannot be compared across clusters.

      State and agreement

      • Under the hood

        This distinction is why Kubernetes has both Deployments and StatefulSets. Deployment pods are interchangeable and get random names; StatefulSet pods get stable identities (db-0, db-1), stable storage and ordered startup, because their state makes them not interchangeable. Choosing the wrong one is a data-loss bug, not a style preference.

      • Under the hood

        Kubernetes makes both choices in different places, deliberately. etcd chooses consistency: lose quorum and writes stop rather than diverge. The controllers around it are eventually consistent by design — they act on a cached view that may be a moment out of date, and correct themselves on the next pass. That is why kubectl get can show something stale and be working exactly as intended.

      • Under the hood

        A small committee that can only act with a strict majority. Requiring a majority is what makes two halves of a partitioned cluster unable to both act — only one side can hold more than half. This is the entire reason etcd clusters are 3 or 5 members and never 4, and why losing 2 of 3 control-plane nodes makes the API read-only rather than degraded.

      • Under the hood

        This is etcd, and therefore the foundation the entire Kubernetes control plane stands on. Every object you will ever create is a committed entry in a replicated log — which is also why etcd is disk-latency-sensitive: a slow disk on one member delays commits for the whole cluster, and the symptom presents as a mysteriously slow apiserver.

      Patterns

      • Under the hood

        This is why the Kubernetes API is declarative. kubectl apply states the desired end state, so applying the same manifest ten times gives the same result as once — whereas an imperative "create" repeated is an error and a "scale up by one" repeated is an outage. Every controller in the system relies on its own actions being replayable.

      • Under the hood

        Without jitter, clients that failed together retry together — a thundering herd that keeps a recovering service down. You have already seen the naming of this pattern in Kubernetes: CrashLoopBackOff is a restart delay doubling to a five-minute cap, which is a feature protecting the node, not a diagnosis of the pod.

      • Under the hood

        The thermostat, and the pattern every Kubernetes controller implements without exception. It is also why the system recovers from things nobody wrote recovery code for: delete a pod a Deployment owns and no undo logic runs — the next pass simply observes one fewer pod than desired and creates one. Controllers watch for speed and re-list periodically for correctness, because a missed event must not mean permanent drift.

      • Under the hood

        A lease is a lock with an expiry, which is what makes it safe when the holder dies without releasing it. Kubernetes does this twice over: node health is a Lease object the kubelet renews every ten seconds (missing renewals eventually mark the node NotReady), and the scheduler and controller-manager elect their active instance by racing for a lease of their own.

      B14 — Capstone: build minibox, then the gauntlet

      10 cp · ~6 h

      No new concepts — everything here is assembly and proof. You will put the last thirteen modules into one script that runs a process in its own namespaces, on its own filesystem, inside its own limits, on its own network. Then you will fix eight planted faults against a clock.

      All of B0–B13. If any module still has unticked checkpoints, finish it first — the capstone assumes every layer is already familiar, and debugging two unfamiliar layers at once is how people conclude they are bad at this.

      Capstone — minibox, one layer at a time

      • Under the hood

        The stack of transparent sheets from B5: you write on the top one and read through the whole stack. You have just built the mechanism behind container images — the lower directories are the image's layers, shared read-only between every container that uses them, and the upper is the container's ephemeral writable layer that disappears when it does.

      • Under the hood

        That ps output is the moment the whole track pays off — the isolation is not a product feature, it is five flags to clone() plus a mount. Mounting /proc after the pivot is the step everyone forgets, and its signature is unmistakable: ps inside the "container" cheerfully lists the entire host, because it is reading the host's /proc.

      • Under the hood

        Compare what you built to what a real stack adds, because the differences are the entire Intermediate track: an image format and registry instead of a directory you assembled by hand, an OCI runtime spec instead of your argument parsing, a CRI so a kubelet can drive it, and a CNI plugin contract instead of your networking script. None of those replace the machinery underneath — they standardise it.

      The debugging gauntlet

      • Under the hood

        Speed on the CKA and on a real incident comes from having an order, not from knowing more commands. Almost every minute lost in run 1 will turn out to be a skipped rung on B8.16's ladder or a guess made before evidence — and the point of writing it down is that you cannot fix a habit you have not named.

      Track assessment

      • Under the hood

        The Advanced track is directly reachable from here and will now read as recognition: every "Under the hood" note there names something you have built by hand. The Intermediate track, when it lands, restarts from this exact capstone — minibox becomes the OCI runtime spec and runc, and B9's networking script becomes a plugin that speaks the CNI ADD/DEL contract.
