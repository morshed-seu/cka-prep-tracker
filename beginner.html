      The machine under Kubernetes

      Kubernetes invented very little. It composes decades-old operating-system, networking and distributed-systems technology — namespaces, cgroups, overlay filesystems, virtual network interfaces, certificates, consensus. This track teaches those pieces directly, by building them with your own hands on one throwaway Linux VM, and only then names the Kubernetes feature that wraps each one.

      So you will meet CPU throttling as cpu.max in a cgroup file long before you meet it as limits.cpu in a pod spec, and you will build a bridge, a veth pair and a NAT rule by hand before anyone says the word "CNI". Every conceptual lesson ends with a “Where this shows up in Kubernetes” block that links forward to the exact CKA-track lesson where the same machinery reappears — so the advanced track reads as recognition rather than new material.

      There is deliberately no calendar here: the eight-week clock belongs to the exam track. Work through the modules in order — each is a prerequisite for the next — at whatever pace fits. Checkboxes save automatically in this browser, in the same store as the CKA tracker, so progress on either track is always visible from the other.

      B0 — Your Linux sandbox

      9 cp · ~2 h

      Everything in this track happens on one machine you are allowed to wreck. Build it, learn to put it back in under a minute, and install the tools every later module assumes are already there.

      Nothing. This is the front door — if you have never opened a terminal, you are in the right place.

      How to use this track

      • Under the hood

        Kubernetes invented very little. A pod is Linux namespaces; a CPU limit is a cgroup file; an image layer is overlayfs; the pod network is a bridge plus veth pairs plus a NAT rule; etcd is Raft consensus; every certificate is ordinary PKI. Learning the wrapper before the thing it wraps is why Kubernetes feels like memorization to so many people.

      • Under the hood

        Progress lives in your browser's localStorage, not on a server — there is no account and nothing to log into, which is also why progress does not follow you to another browser or a private window. The beginner and CKA trackers share one store, so a box ticked on a lesson page shows up here on reload, and vice versa. Each track's reset button clears only its own checkpoints.

      The sandbox VM

      • Under the hood

        Multipass is a thin front end over whatever hypervisor your machine already has — KVM on Linux, the Hypervisor framework on macOS, Hyper-V on Windows. A VM, not a container, is the right sandbox here for one blunt reason: this track deliberately breaks the kernel's networking, mount and cgroup state, and a container shares the host's kernel with you.

      • Under the hood

        Spell out 24.04 and never write multipass launch lts. The lts alias is a moving target — it now resolves to Ubuntu 26.04, which would quietly hand you a different kernel and different defaults from every command output pasted in this track. Every lab here is authored against 24.04 "noble": cgroups v2 only, iptables running on the nftables backend, and systemd-resolved answering on 127.0.0.53.

      Toolbelt & safety

      • Under the hood

        build-essential is the C compiler and headers — B2 and B3 write small C programs, because some things (a signal handler, a raw write(2)) cannot be shown honestly in a shell. strace/ltrace are the X-ray machines of B3; iproute2, tcpdump and nftables are all of B8–B9; socat is how B9 talks over a unix socket.

      • Under the hood

        Do this now, while the stakes are zero. The single biggest thing that slows beginners down is not difficulty — it is the quiet fear of breaking the machine, which makes them skip exactly the destructive labs that teach the most. A restore you have already done once stops being scary.

      • Under the hood

        The sections matter because the same name lives in several of them: man 1 printf is the shell command, man 3 printf is the C function; man 8 mount is the tool you type, man 2 mount is the system call it makes. Section 7 is the underrated one — man 7 namespaces, man 7 capabilities and man 7 cgroups are the primary sources for three whole modules of this track.

      B1 — Shell, files, and the filesystem tree

      16 cp · ~4 h

      The shell is not a lesser version of a graphical file manager — it is a language for composing small programs into an answer that did not exist before you typed it. This module is the vocabulary every other module speaks.

      B0 — you need the sandbox VM and a snapshot you trust before you start deleting things in it.

      Paths and the tree

      • Under the hood

        Linux has one tree with a single root /, not a drive letter per device. A second disk does not become D: — it gets grafted onto a directory somewhere in the same tree, which is what "mounting" means and what B5 takes apart. That single-tree design is also why a container can be handed a completely different / and never notice.

      • Under the hood

        This layout is a written standard (the FHS), which is why you can sit down at an unfamiliar Linux box and still know where to look. The distinction that pays off later: /etc is what you set, /var is what the machine accumulates — logs, spools, databases, and on a Kubernetes node, /var/lib/kubelet and /var/lib/containerd.

      • Under the hood

        "Everything is a file" is meant literally: a terminal (c, character device), a disk (b, block device), a socket (s) and a pipe (p) all appear in the same namespace with the same permission bits, and all answer the same read/write system calls. That uniformity is what lets B3's strace show one program talking to a file, a terminal and a network connection in identical syntax.

      Composing tools

      • Under the hood

        Those numbers are file descriptors, the small integers the kernel hands out for anything a process has open. Keeping errors on a different descriptor from results is the whole reason a pipeline can carry clean data while still telling you something went wrong — and it is why a script that merges the two with &> becomes so much harder to debug.

      • Under the hood

        Order matters and catches everybody once: cmd > f 2>&1 sends both streams to the file, but cmd 2>&1 > f sends errors to the old stdout (your terminal) because 2>&1 copies where stdout points at that moment, before the redirect moves it.

      • Under the hood

        A pipe is a kernel buffer between two processes that run at the same time, not one after the other. That is why … | head -5 can finish early: head exits, the writer's next write gets SIGPIPE, and the whole pipeline stops rather than computing output nobody will read.

      • Under the hood

        The same characters mean different things in each: glob * is "any run of characters", regex * is "zero or more of the previous item" — so the regex equivalent of glob * is .*. And because the shell expands globs first, grep *.log file silently searches for the name of the first matching file. Quote patterns you intend a program to read.

      • Under the hood

        uniq only collapses adjacent duplicates, so it is nearly always sort | uniq -c | sort -rn. find … -print0 | xargs -0 is the pairing that survives filenames with spaces. And awk '{print $2}' replaces most of what people reach for cut for, because it treats any run of whitespace as one separator.

      The shell itself

      • Under the hood

        The environment is a plain list of KEY=VALUE strings copied into a child process when it is created — that is the entire mechanism. It is a one-way copy at creation time, so a child can never change its parent's environment, and a running process's environment is frozen except by its own action. You can read any process's copy at /proc/PID/environ, which is exactly why B12 warns against putting secrets there.

      • Under the hood

        The rule that prevents most shell bugs: quote every variable expansion — "$f", not $f. Unquoted, the shell takes the variable's value, splits it on whitespace and glob-expands the pieces, so a filename with a space silently becomes two arguments and a value containing * becomes a directory listing.

      • Under the hood

        The shebang is not shell magic: the two bytes #! are read by the kernel in execve, which then runs the named interpreter with your file as its argument. That is why a script with Windows line endings fails with a baffling bad interpreter: No such file or directory — the carriage return is part of the interpreter's path. Exit codes come back in B2 with the 128 + signal convention that explains 137.

      B2 — Processes, threads, signals

      18 cp · ~6 h

      A container is a process. Everything Kubernetes does to a workload — start it, watch it, ask it to stop, give up and kill it, report why it died — is something Linux does to a process, and this module is where you learn to narrate all of it.

      B1 — the labs read /proc with the same tools you just learned, and the project is a shell script plus a small C program.

      What a process is

      • Under the hood

        Everything later in this track is a modification of that list. B4 changes the credentials, B5 and B6 change what the file descriptors and mount table can reach, B6 changes which PIDs are even visible, B7 caps how much memory and CPU the address space may use. There is no separate "container" object in the kernel — a container is a process with all four of those things adjusted.

      • Under the hood

        VSZ is address space the process has reserved; RSS is physical memory it is actually occupying. A process can map a huge file or reserve a large heap and touch almost none of it. Memory limits in B7 are enforced against something closer to RSS, which is why a scary VSZ is usually not the problem you are looking for.

      • Under the hood

        /proc is not a directory of files on a disk — it is a view of kernel data structures rendered as text on demand, which B3 takes apart properly. It is also the honest answer to "what is this process really doing": ls -l /proc/PID/fd shows every open file, socket and pipe, and it is how you find the process still holding a deleted 40 GB log file.

      Lifecycle

      • Under the hood

        The split looks redundant until you see what happens in between: the child is briefly a copy of the parent and can adjust itself — redirect its file descriptors, drop privileges, join a cgroup, enter namespaces — and only then exec the target program. Every shell redirect, and every container start, happens in that gap between fork and exec.

      • Under the hood

        D is the one that matters at 3 a.m.: the process is inside a kernel call that refuses to be interrupted — almost always storage or network I/O — and it will not respond to SIGKILL, because there is no safe point at which to deliver it. A pile of D-state processes means the layer underneath is wedged; killing them is not an option, so you go fix the disk or the mount.

      • Under the hood

        This is where 137 comes from — 128 + 9, killed by SIGKILL — and 143 is 128 + 15, killed by SIGTERM. You will see 137 constantly on Kubernetes nodes, and it means exactly what it means here: something outside the process decided it should stop existing. B7 shows you the something.

      • Under the hood

        A zombie is a receipt: the kernel keeps the exit status until someone reads it. Thousands of them mean a parent that has stopped reaping, and the real damage is PID exhaustion, not memory. This is precisely the "PID 1 problem" in containers — a process that was never written to be an init has no idea it inherited re-parented orphans, so they pile up. It is also why the pod spec has a shareProcessNamespace option and why runtimes offer to inject a tiny init.

      Threads

      • Under the hood

        The kernel does not have two different objects for "process" and "thread" — both are created by the same clone() call, and the flags decide how much is shared. That single call is also how B6's namespaces get created, so the machinery behind threading and the machinery behind containers is literally the same system call with different arguments.

      Signals

      • Under the hood

        A signal is a knock on the door with a standard meaning, not a message with a body — you get the number and nothing else. Almost all of the meaning is convention: nothing in the kernel forces HUP to mean "reload", it is simply what daemons agreed to do with it.

      • Under the hood

        They are uncatchable by design: the kernel needs at least one way to remove a process that has stopped cooperating, and it is handled entirely inside the kernel without ever running the process's code. That is also why a KILLed process gets no chance to flush buffers, remove its lock file, or finish a request — and why every graceful-shutdown design has to happen during the TERM that comes first.

      • Under the hood

        This is the exact contract Kubernetes uses to stop a pod: SIGTERM first, then a grace period (30 s by default), then SIGKILL. A container that ignores TERM is not "robust" — it just guarantees every one of its shutdowns takes the full grace period and then loses whatever it had in flight, showing up as exit code 137 in the pod's status.

      B3 — Kernel space, user space, system calls

      14 cp · ~4 h

      There is a hard line down the middle of a Linux machine. Your programs live on one side, the hardware lives on the other, and the only way across is a system call. Once you can see those crossings, you can find out what a program is really doing when its logs will not say.

      B2 — the syscalls you will read (clone, execve, wait4) are the lifecycle you just learned, now seen from the kernel's side.

      The boundary

      • Under the hood

        Think of the kernel as the hotel's engineering staff — the only people allowed to touch the wiring. A guest who wants hot water does not open the boiler; they file a work order. That work order is the system call, and the reason for the rule is that one buggy guest must not be able to burn the building down.

      • Under the hood

        There are only a few hundred system calls in total (man 2 syscalls) — a small, stable, documented interface, which is why a binary compiled a decade ago still runs. It is also what seccomp filters in B12: not "what may this program access" but "which of these few hundred doors may it even knock on".

      • Under the hood

        The gap between the two explains a classic confusion: a hundred printf calls may produce a single write syscall, because libc buffers. That is also why a program killed by SIGKILL loses output it "already printed" — the text was sitting in a userspace buffer that never reached the kernel.

      Reading syscalls

      • Under the hood

        Descriptors are per-process and allocated as the lowest free number, which is exactly why 0/1/2 are stdin/stdout/stderr and why the shell's redirection trick works: close 1, open the file, and the file lands in slot 1 — so the program writes to the file without knowing anything changed.

      Kernel interfaces

      • Under the hood

        This is why the standard tools have no special powers: ps reads /proc/*/stat, free reads /proc/meminfo, ss reads /proc/net/*. Anything they can tell you, you can read directly — which matters when you are inside a stripped-down container image that has none of them installed.

      • Under the hood

        Two of these show up on every Kubernetes node: net.ipv4.ip_forward=1, without which the node cannot route pod traffic at all (B9 builds exactly that by hand), and fs.inotify.max_user_instances, whose default is low enough that a busy node runs out of file watches and the kubelet starts failing in ways that look like anything but a sysctl.

      • Under the hood

        The OOM killer's verdict lands here and nowhere else. When B7 kills a memory hog, the application's own logs stop mid-sentence with no error, and the only record of what happened — which process, how much memory, which cgroup — is in dmesg. "The app died silently" almost always means "nobody read the ring buffer".
