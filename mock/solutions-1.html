Mock exam 1 — solutions & grading

Grade like the real thing grades: by resulting state, verified with a command — not by intent. Each task lists the solution, the verification, and a point breakdown; award partial credit exactly as itemized, nothing for "I knew it, I just…". Pass line: 66/100. Every dropped point names the domain that goes on your 8.2 drill list.

Scorecard Troubleshooting: tasks 2, 6, 9, 12, 14 (30 pts) · Cluster architecture: 1, 7, 13, 15 (25 pts) · Networking: 5, 10, 16 (20 pts) · Workloads & scheduling: 4, 8 (15 pts) · Storage: 3, 11 (10 pts). Solutions show one good path; any route to the same verified end state scores full points.

Task 1 RBAC for a deployment bot

cluster architecture6 pts
Solution & rubric 
kubectl -n neptune create serviceaccount deploy-bot
kubectl -n neptune create role deployment-manager \
  --verb=get,list,create --resource=deployments
kubectl -n neptune create rolebinding deploy-bot-binding \
  --role=deployment-manager --serviceaccount=neptune:deploy-bot
Verify
kubectl auth can-i create deployments -n neptune \
  --as=system:serviceaccount:neptune:deploy-bot     # yes
kubectl auth can-i delete deployments -n neptune \
  --as=system:serviceaccount:neptune:deploy-bot     # no
kubectl auth can-i list deployments -n default \
  --as=system:serviceaccount:neptune:deploy-bot     # no
CriterionPts
ServiceAccount deploy-bot exists in neptune1
Role with exactly get/list/create on deployments (extra verbs = 0 here)2
RoleBinding connects the Role to the SA2
All three can-i checks answer as above1

Trap check: a ClusterRoleBinding instead of a RoleBinding grants cluster-wide access — that fails the "no access outside neptune" requirement and caps this task at 3.

Task 2 A deployment that won't schedule

troubleshooting6 pts
Solution & rubric

Diagnosis: pods are Pending; describe names the reason.

kubectl -n mercury get pods
kubectl -n mercury describe pod -l app=web-frontend | grep -A3 Events:
Warning  FailedScheduling ... 0/3 nodes are available: 3 Insufficient cpu.

Each replica requests cpu: "4" — more than any lab node has. Lower the request to something real (the exact value doesn't matter; schedulable does):

kubectl -n mercury edit deployment web-frontend
  # resources.requests.cpu: "4"  →  100m
Verify
kubectl -n mercury get deployment web-frontend    # 2/2 READY
CriterionPts
Root cause identified (Insufficient cpu from the request, stated from Events)2
Fixed via the resource request — image, replicas, deployment untouched otherwise3
2/2 Ready verified1

Deleting/recreating the deployment or dropping replicas to make it fit violates the task and caps at 2 (diagnosis only).

Task 3 Static PV, PVC and a writer pod

storage5 pts
Solution & rubric 
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolume
metadata:
  name: exam-pv
spec:
  capacity: {storage: 1Gi}
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  hostPath:
    path: /srv/exam-pv
    type: DirectoryOrCreate
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: exam-pvc
  namespace: pluto
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: manual
  resources:
    requests: {storage: 500Mi}
---
apiVersion: v1
kind: Pod
metadata:
  name: pv-writer
  namespace: pluto
spec:
  containers:
  - name: main
    image: busybox:1.36
    command: ["sh", "-c", "date > /data/out.txt; sleep 1d"]
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: exam-pvc
EOF

type: DirectoryOrCreate makes the kubelet create /srv/exam-pv on whichever node runs the pod — without it, a missing host directory fails the mount.

Verify
kubectl -n pluto get pvc exam-pvc      # STATUS Bound, VOLUME exam-pv
kubectl -n pluto exec pv-writer -- cat /data/out.txt   # a date
CriterionPts
PV with exact capacity/mode/class/hostPath2
PVC in pluto, 500Mi, class manual, Bound to exam-pv1
Pod Running with the claim mounted and /data/out.txt written2

Task 4 ConfigMap-driven deployment + rolling update

workloads & scheduling8 pts
Solution & rubric 
kubectl -n pluto create configmap app-config \
  --from-literal=APP_MODE=production --from-literal=APP_COLOR=blue
kubectl -n pluto create deployment webapp \
  --image=nginx:1.28-alpine --replicas=3 --dry-run=client -o yaml > webapp.yaml
vim webapp.yaml     # add under the container:
#        envFrom:
#        - configMapRef:
#            name: app-config
kubectl apply -f webapp.yaml
kubectl -n pluto rollout status deployment webapp
kubectl -n pluto set image deployment/webapp nginx=nginx:1.29-alpine
kubectl -n pluto rollout status deployment webapp

Building the YAML first (instead of create then set env --from=configmap/app-config) keeps the image change as revision 2 — but that's cosmetic: full credit as long as the newest revision is the image update and the end state matches. (set env --from is a fine, faster alternative; it just spends a revision.)

Verify
kubectl -n pluto rollout history deployment webapp        # image change = newest rev
kubectl -n pluto get deployment webapp \
  -o jsonpath='{.spec.template.spec.containers[0].image}'  # nginx:1.29-alpine
kubectl -n pluto exec deploy/webapp -- sh -c 'env | grep APP_'
APP_MODE=production
APP_COLOR=blue
CriterionPts
ConfigMap with both exact keys/values2
Deployment: right name/image/replicas, all CM keys in the env (envFrom or per-key)3
Rolling update performed to 1.29-alpine, rollout completed, visible as a new revision2
Env proven in a running pod1

Task 5 NodePort service + its DNS name

networking6 pts
Solution & rubric 
kubectl -n jupiter expose deployment web-portal --name=web-portal-svc \
  --port=80 --type=NodePort
kubectl -n jupiter patch service web-portal-svc --type='json' \

Verify
curl -s http://10.211.64.11:30300 | grep title    # Welcome to nginx!
cat /root/exam/dns.txt
CriterionPts
Service web-portal-svc: NodePort type, port 80, selects the web-portal pods3
nodePort is exactly 30300 (curl on that port answers)1
/root/exam/dns.txt = web-portal-svc.jupiter.svc.cluster.local1

Task 6 A service with no endpoints

troubleshooting6 pts
Solution & rubric

Two planted faults, both inside the Service — the no-endpoints signature (selector) plus a port that would fail even with endpoints:

kubectl -n venus get endpointslices                     # empty for db-service
kubectl -n venus get pods --show-labels                 # pods carry app=database
kubectl -n venus get svc db-service -o yaml | grep -A2 'selector\|targetPort'
selector: {app: db}          ← pods are labeled app=database
targetPort: 8080             ← nginx listens on 80
kubectl -n venus edit service db-service
  # selector: app: database
  # targetPort: 80
Verify
kubectl -n venus get endpointslices        # two endpoints appear
kubectl -n venus run check --rm -it --restart=Never \
  --image=busybox:1.36 -- wget -qO- -T2 http://db-service | grep title
CriterionPts
Selector fixed to match the pod labels (endpoints exist)2
targetPort fixed to 80 (traffic actually served)2
Deployment untouched1
wget verification returns the page1

Finding only the selector and declaring victory without the consumer-side wget is exactly the trap: endpoints appear, traffic still dies on 8080. Verification from the consumer's side catches it — that's the lesson this task exists for.

Task 8 Pin a pod to labeled hardware, with limits

workloads & scheduling7 pts
Solution & rubric 
kubectl label node node02 disk=fast
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: fast-worker
  namespace: pluto
spec:
  nodeSelector:
    disk: fast
  containers:
  - name: nginx
    image: nginx:1.29-alpine
    resources:
      requests: {cpu: 200m, memory: 128Mi}
      limits:   {cpu: 500m, memory: 256Mi}
EOF
Verify
kubectl -n pluto get pod fast-worker -o wide     # Running, NODE node02
CriterionPts
node02 labeled disk=fast1
Pod uses a nodeSelector on that label (not a nodeName pin)2
Requests and limits all four values exact3
Running on node02, verified with -o wide1

nodeName: node02 "works" but bypasses the scheduler and ignores the label — the task said only onto nodes labeled; it scores 0 on the selector row.

Task 9 A crash-looping deployment

troubleshooting6 pts
Solution & rubric 
kubectl -n mars get pods                       # CrashLoopBackOff
kubectl -n mars logs deploy/api
nginx: invalid option: "--with-turbo"

The pod template overrides the container command with a bogus nginx flag. Remove the flag (or the whole command: override — the image's default is correct):

kubectl -n mars edit deployment api
  # delete the line:  - --with-turbo     (or the entire command: block)
Verify
kubectl -n mars get pods    # 1/1 Running, RESTARTS not climbing
CriterionPts

Task 10 Route traffic through the Gateway

networking7 pts
Solution & rubric 
cat <<'EOF' | kubectl apply -f -
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: shop-route
  namespace: jupiter
spec:
  parentRefs:
  - name: traefik-gateway
    namespace: traefik
  hostnames: ["shop.exam.test"]
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: shop
      port: 80
EOF
Verify
kubectl -n jupiter describe httproute shop-route | grep -A2 Conditions:
curl -s -H 'Host: shop.exam.test' http://10.211.64.11:30080 | grep title
Type: Accepted      Status: True
<title>Welcome to nginx!</title>
CriterionPts
parentRefs names the Gateway and its namespace traefik2
Hostname shop.exam.test1
backendRef service shop port 80 (route in jupiter, same ns as backend)2
Accepted: True condition checked1
curl through the NodePort returns the page1

The classic zero here: omitting namespace: traefik in parentRefs — the route then looks for a Gateway in jupiter, finds none, and sits with Accepted: False. The condition names the problem; reading it is the skill.

Task 16 Lock down the database pod

networking7 pts
Solution & rubric 
cat <<'EOF' | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-guard
  namespace: jupiter
spec:
  podSelector:
    matchLabels:
      role: db
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          role: backend
    ports:
    - protocol: TCP
      port: 80
EOF
Verify

Spec-check the object (grading basis — flannel doesn't enforce policies, so traffic tests prove nothing here):

kubectl -n jupiter get networkpolicy db-guard -o yaml

Confirm: podSelector targets role: db only; one ingress rule; from is a bare podSelector (same-namespace pods); port TCP/80; policyTypes: [Ingress]. To watch it actually enforce, re-run the 4.24 Calico swap and wget from frontend (times out) vs backend (answers).

CriterionPts
podSelector: role=db (not empty — an empty selector would restrict every pod)2
from = podSelector role=backend alone (no namespaceSelector AND-combo, no extra peers)2
Ports limited to TCP 801
policyTypes = Ingress only (egress and other pods untouched)1
Object present in jupiter under the right name1

Indentation trap, one last time: - podSelector (one list item under from) means backend pods; adding a second - peer ORs another allowed source, while nesting a namespaceSelector inside the same item ANDs it. One dash decides the meaning (4.25).
