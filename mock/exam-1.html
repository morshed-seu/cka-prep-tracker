Mock exam 1 — 16 tasks · 120 minutes · pass ≥ 66/100

An original exam built for your lab cluster, weighted like the real curriculum: 30% troubleshooting, 25% cluster architecture, 20% networking, 15% workloads & scheduling, 10% storage. Treat it as the real thing: one extra browser tab (kubernetes.io docs only), no notes, no peeking at the solutions page until the timer rings.

120:00 The countdown survives page reloads — start it once and work.
Before you start
  1. Provision the environment (some objects are planted broken on purpose — don't investigate anything before the timer runs): 
    wget https://raw.githubusercontent.com/morshed-seu/cka-prep-tracker/main/mock/setup-exam-1.sh
sudo bash setup-exam-1.sh        # on cp; type 'begin' when asked
  2. Every task begins with its context line — your lab has one context, run it anyway; the ritual (lesson 8.6) is what you're rehearsing. Tasks that name a node expect you to shell into it (multipass shell node01) and exit when done.
  3. Answer files go exactly where each task says (under /root/exam/ on cp) — wrong path scores zero.
  4. First pass ≤6 minutes per task; flag and skip anything sticky (write its number down), second pass by points. Verify every task before calling it done.
  5. When the timer rings: hands off, then grade honestly with solutions-1 (per-task rubric, 66 points to pass). Clean up afterwards: sudo bash setup-exam-1.sh cleanup.

Task 1 RBAC for a deployment bot

cluster architecture6 pts
kubectl config use-context kubernetes-admin@kubernetes

In namespace neptune:

  • Create a ServiceAccount deploy-bot.
  • Create a Role deployment-manager that allows exactly get, list and create on deployments.
  • Bind it to the ServiceAccount with a RoleBinding deploy-bot-binding.

The ServiceAccount must be able to create deployments in neptune but must not be able to delete them, and must have no access outside neptune.

Task 2 A deployment that won't schedule

troubleshooting6 pts
kubectl config use-context kubernetes-admin@kubernetes

Deployment web-frontend in namespace mercury should run 2 replicas, but none are available. Diagnose why and fix it so the deployment reports 2/2 ready. You must not change the image, the replica count, or delete/recreate the deployment.

Task 3 Static PV, PVC and a writer pod

storage5 pts
kubectl config use-context kubernetes-admin@kubernetes
  • Create a PersistentVolume exam-pv: capacity 1Gi, access mode ReadWriteOnce, storageClassName: manual, hostPath /srv/exam-pv.
  • Create a PersistentVolumeClaim exam-pvc in namespace pluto requesting 500Mi from storage class manual.
  • Create a pod pv-writer in pluto (image busybox:1.36) that mounts the claim at /data and writes the current date to /data/out.txt at startup, then sleeps.

The PVC must end up Bound and the pod Running.

Task 4 ConfigMap-driven deployment + rolling update

workloads & scheduling8 pts
kubectl config use-context kubernetes-admin@kubernetes

In namespace pluto:

  • Create a ConfigMap app-config with keys APP_MODE=production and APP_COLOR=blue.
  • Create a deployment webapp, image nginx:1.28-alpine, 3 replicas, whose containers receive all keys of app-config as environment variables.
  • Once all replicas are ready, perform a rolling update of the image to nginx:1.29-alpine and wait for it to complete.

Afterwards: rollout history must show revision 2, all 3 replicas ready on the new image, and APP_MODE visible in the environment of a running pod.

Task 5 NodePort service + its DNS name

networking6 pts
kubectl config use-context kubernetes-admin@kubernetes

Expose deployment web-portal in namespace jupiter with a Service web-portal-svc: type NodePort, service port 80, node port 30300.

Then write the fully qualified DNS name by which pods in other namespaces reach this service (the full form ending in the cluster domain) to /root/exam/dns.txt on cp.

Verify: curl http://10.211.64.11:30300 from any lab VM returns the nginx welcome page.

Task 6 A service with no endpoints

troubleshooting6 pts
kubectl config use-context kubernetes-admin@kubernetes

Service db-service in namespace venus is supposed to route traffic on port 80 to the pods of deployment db, but requests hang and the service has no endpoints. Fix the Service only — the deployment is correct and must not be modified.

Verify: kubectl -n venus run check --rm -it --restart=Never --image=busybox:1.36 -- wget -qO- -T2 http://db-service returns the nginx welcome page.

Task 7 etcd backup, verified

cluster architecture7 pts
kubectl config use-context kubernetes-admin@kubernetes   # then work on cp

On cp:

  • Create a snapshot of the cluster's etcd at /root/exam/etcd-backup.db, authenticating with the etcd server certificates from the cluster's PKI directory.
  • Verify the snapshot's integrity offline and save the status table (the -w table output) to /root/exam/etcd-status.txt.

The snapshot must be restorable (non-zero keys); the running cluster must not be disturbed.

Task 8 Pin a pod to labeled hardware, with limits

workloads & scheduling7 pts
kubectl config use-context kubernetes-admin@kubernetes
  • Label node node02 with disk=fast.
  • Create pod fast-worker in namespace pluto: image nginx:1.29-alpine, scheduled only onto nodes labeled disk=fast (use a node selector), requesting 200m CPU / 128Mi memory with limits 500m CPU / 256Mi memory.

The pod must be Running on node02.

Task 9 A crash-looping deployment

troubleshooting6 pts
kubectl config use-context kubernetes-admin@kubernetes

Deployment api in namespace mars is crash-looping. Determine the cause from the pod's logs, then fix the deployment so its pod is Running and Ready. Do not change the image.

Task 10 Route traffic through the Gateway

networking7 pts
kubectl config use-context kubernetes-admin@kubernetes

The cluster runs a Gateway traefik-gateway in namespace traefik (HTTP listener on port 80, routes allowed from all namespaces). Namespace jupiter contains a deployment shop with a ClusterIP service shop on port 80.

Create an HTTPRoute shop-route in namespace jupiter that attaches to that Gateway and forwards all requests for hostname shop.exam.test to service shop on port 80.

Verify: curl -s -H 'Host: shop.exam.test' http://10.211.64.11:30080 returns the nginx welcome page.

Task 11 Dynamic provisioning with the default class

storage5 pts
kubectl config use-context kubernetes-admin@kubernetes
  • Create a PersistentVolumeClaim data-pvc in namespace pluto: 500Mi, ReadWriteOnce, storage class local-path.
  • Create a pod data-pod in pluto (image busybox:1.36, sleeping) that mounts the claim at /data.

End state: PVC Bound, pod Running. (If the PVC sits Pending before the pod exists, that is not an error — think about the storage class's binding mode.)

Task 12 Extract the error lines

troubleshooting6 pts
kubectl config use-context kubernetes-admin@kubernetes

Pod logger in namespace saturn writes a mixed application log. Save all log lines containing ERROR (the complete lines, nothing else) to /root/exam/errors.txt on cp.

Task 13 Kustomize overlay

cluster architecture6 pts
kubectl config use-context kubernetes-admin@kubernetes   # then work on cp

A Kustomize base at /root/exam/kustomize/base defines a deployment hello. Create an overlay at /root/exam/kustomize/overlays/prod that, without modifying the base:

  • deploys into namespace neptune,
  • prefixes resource names with prod-,
  • sets the hello deployment to 3 replicas.

Apply the overlay with kubectl -k. End state: deployment prod-hello in neptune with 3/3 ready.

Task 14 Find the CPU hog

troubleshooting6 pts
kubectl config use-context kubernetes-admin@kubernetes

Namespace uranus runs three pods. Using the cluster's resource-metrics pipeline, determine which pod consumes the most CPU and write only its name to /root/exam/high-cpu.txt on cp. (Metrics need a minute or two of pod runtime to appear — if kubectl top comes back empty, wait and retry.)

Task 15 A static pod on node01

cluster architecture6 pts
kubectl config use-context kubernetes-admin@kubernetes   # then: multipass shell node01

On node01, create a static pod named static-web: image nginx:1.29-alpine, container port 80, manifest placed in the kubelet's static-pod directory. Remember to exit the node when done.

End state, seen from cp: a pod static-web-node01 is Running in the default namespace.

Task 16 Lock down the database pod

networking7 pts
kubectl config use-context kubernetes-admin@kubernetes

Namespace jupiter contains pods frontend (label role=frontend), backend (label role=backend) and db (label role=db).

Create a NetworkPolicy db-guard in jupiter so that ingress to pods labeled role=db is allowed only from pods labeled role=backend in the same namespace, and only on TCP port 80. All other ingress to the db pods must be disallowed; other pods in the namespace must remain unrestricted.

Grading is on the policy object (as on the real exam). Note your lab's flannel CNI doesn't enforce NetworkPolicies — traffic still flowing is not a sign your policy is wrong; the solutions page shows how to check the spec precisely.

Timer rang? Hands off the keyboard. Count what you attempted and verified, then grade with solutions-1.html — per-task rubric, 66/100 to pass, partial credit exactly as specified. Afterwards: sudo bash setup-exam-1.sh cleanup, and put every dropped point's domain on your 8.2 drill list.
