      Week 0 — Set up your lab & logistics

      2–3 days

      Before Day 1 of studying: book the exam, build a real kubeadm cluster (the exam runs on kubeadm clusters on Ubuntu), and tune your shell so speed habits form from the start.

      Logistics

      Lab environment

      • Under the hood

        You want a cluster you can break. Managed clusters (EKS/GKE) and minikube hide the control plane from you; the exam expects you to SSH into nodes, read journalctl, and edit static pod manifests — only a kubeadm cluster gives you that.

      • Under the hood

        The manual steps teach you the node prerequisites the exam probes: swap off (kubelet refuses to start with swap by default), kernel modules overlay and br_netfilter, sysctls net.ipv4.ip_forward=1, and matching cgroup drivers — kubelet and containerd must both use systemd cgroups or pods will mysteriously fail.

      Week 1 — How the control plane actually works

      feeds the 25% + 30% domains

      Everything else in the exam sits on top of five binaries and a database. Learn each component's real job and its failure signature. This week is reading + tracing on your live cluster, not memorizing.

      The big picture

      • Under the hood

        The key insight: everything talks only to the apiserver, and only the apiserver talks to etcd. Scheduler and controllers don't command kubelets — they write desired state to the API, and kubelets watch and converge. Kubernetes is a set of independent control loops around a shared database, not a pipeline.

      • Under the hood

        Chicken-and-egg: the control plane runs as pods, but pods need a control plane. Solution: static pods — kubelet watches /etc/kubernetes/manifests and runs whatever YAML appears there, no apiserver needed. It then reports read-only mirror pods to the API (named kube-apiserver-<node>), which you can see but not delete via kubectl.

      • Under the hood

        kubectl POSTs the Deployment → apiserver persists it → deployment controller creates a ReplicaSet → replicaset controller creates Pods with empty nodeName → scheduler watches for unscheduled pods, picks a node, writes a Binding → the kubelet on that node sees the pod in its watch, calls CRI to pull the image and start containers → kubelet posts status back. Five independent loops, no direct calls between them.

      kube-apiserver

      • Under the hood

        Every request passes: authn (who are you? — client cert, bearer/ServiceAccount token, OIDC; first module to succeed wins), authz (may you? — Node, RBAC, webhook authorizers; first to allow wins, deny-by-default), mutating admission (webhooks can edit the object — this is how sidecar injectors work), schema validation, validating admission, then serialization to etcd. Memorize this order — it explains a whole class of "permission denied vs rejected" confusions.

      • Under the hood

        A watch is a long-lived HTTP GET with ?watch=true&resourceVersion=N, streaming change events since revision N. The apiserver maintains a watch cache per resource fed by one etcd watch, so thousands of kubelet watches don't hammer etcd. resourceVersion is literally etcd's MVCC revision — it's why optimistic concurrency ("the object has been modified") works.

      • Under the hood

        Objects are stored in etcd under /registry/<resource>/<namespace>/<name> in one canonical storage version; the apiserver converts between served versions on the fly. That's why apiVersion typos are a classic planted exam bug — the API literally has no such endpoint.

      etcd

      • Under the hood

        All writes go to the elected leader, which appends to its write-ahead log and replicates to followers; a write commits once a quorum (majority) has persisted it. That's why member counts are odd: 3 members tolerate 1 failure, 5 tolerate 2 — and why a 2-member cluster is worse than 1 (quorum of 2 means either failure halts writes). Losing quorum makes the cluster read-only: the apiserver still answers cached reads but every write hangs.

      • Under the hood

        etcd never updates in place — every change creates a new revision of the key in a bbolt B+tree, and watches are "replay all events since revision N". Old revisions accumulate until compaction discards them; compaction frees logical space but only defrag returns it to the OS. The resourceVersion you see on every Kubernetes object is this revision counter.

      Scheduler & controller-manager

      • Under the hood

        The scheduler watches for pods with empty spec.nodeName. The scheduling cycle runs plugins at fixed extension points: PreFilter → Filter (drop nodes failing resources, taints, affinity, ports, volume topology) → PostFilter (this is where preemption lives) → Score (rank survivors: least-allocated, spreading, image locality) → pick the winner. The async binding cycle then writes a Binding subresource, which sets nodeName. The scheduler never talks to nodes.

      • Under the hood

        Every controller is the same shape: an informer (a watch + in-memory cache, seeded by an initial LIST) delivers events into a work queue; workers pop keys and run reconcile: read desired state, read actual state, do one small step to converge, requeue on error with backoff. This is level-triggered, not edge-triggered — controllers recover from missed events by periodic resync. It's also exactly how operators/CRD controllers work, which the curriculum now expects you to understand.

      • Under the hood

        Nothing "cascades" imperatively. Deleting a Deployment leaves orphan-looking ReplicaSets whose ownerReferences now point to a missing object — the garbage collector controller notices and deletes them, then the same happens to pods. kubectl delete --cascade=orphan strips this behavior; it's how you can replace a Deployment without killing pods.

      kubelet & the container runtime

      • Under the hood

        The kubelet merges pod specs from all sources into one desired set, then per-pod workers converge reality: create sandbox, pull images, start containers, run probes. The PLEG (Pod Lifecycle Event Generator) detects container state changes from the runtime and triggers re-syncs — "PLEG is not healthy" in kubelet logs means the runtime is too slow/hung, a real troubleshooting signature.

      • Under the hood

        containerd spawns a tiny shim process per pod which calls runc; runc builds the namespaces + cgroups per the OCI spec, starts PID 1, and exits — the shim stays to hold stdio and the exit code, so containerd can restart without killing your containers. crictl speaks CRI directly to the runtime, which is why it works when the apiserver is down — your most important debugging tool for control plane failures.

      • Under the hood

        Every pod starts with an invisible pause container whose only job is to sleep and own the pod's network namespace. App containers join its namespaces — that's the entire mechanism behind "containers in a pod share an IP and localhost". If an app container crashes and restarts, the IP survives because pause never died.

      • Under the hood

        Kubelet renews a Lease every 10s (cheap) and NodeStatus every 5m (or on change). If leases stop, the node controller marks the node NotReady/Unknown after ~40s, then adds the node.kubernetes.io/unreachable:NoExecute taint; pods have a default toleration of 300s, after which they're evicted. This 5-minute delay before pods reschedule is a classic "why isn't my pod moving?" exam moment.

      Week 2 — Installation, PKI & cluster lifecycle

      Cluster Architecture · 25%

      kubeadm init/join/upgrade, the certificate machinery underneath it, and etcd backup/restore. These are guaranteed high-value exam tasks — practice each until it's muscle memory.

      kubeadm init, dissected

      • Under the hood

        Each phase is independently runnable: kubeadm init phase certs apiserver regenerates just one cert. control-plane and etcd phases simply write static pod YAML into /etc/kubernetes/manifests — kubeadm doesn't start anything itself; the already-running kubelet notices the files and starts the pods. upload-config stores the config as the kubeadm-config ConfigMap in kube-system, which later kubeadm upgrade and join read.

      • Under the hood

        Each is a client certificate identity: admin.conf is CN kubernetes-admin in group kubeadm:cluster-admins; controller-manager and scheduler have their own users bound to exactly the RBAC they need. This is the pattern for all cert-based auth in Kubernetes: CN = username, O = group. There is no "user" object in the API at all.

      PKI — the certificate web

      • Under the hood

        Three separate trust domains: the cluster CA signs the apiserver's serving cert and all client certs; the etcd CA is independent (apiserver holds an etcd-client cert signed by it); the front-proxy CA secures the aggregation layer (how metrics-server plugs in). sa.key is not TLS at all — it's the RSA key that signs ServiceAccount JWTs; sa.pub verifies them.

      • Under the hood

        The apiserver cert's SANs must include every name clients use: kubernetes.default.svc, the service ClusterIP (first IP of the service range), the node IP, and any LB name. "x509: certificate is valid for X, not Y" errors in kubelet or kubectl logs mean a SAN mismatch — a classic planted fault.

      Joining nodes & TLS bootstrap

      • Under the hood

        The bootstrap token (format id.secret, stored as a Secret in kube-system) lets the new kubelet authenticate as the system:bootstrappers group — which has exactly one RBAC power: create CertificateSigningRequests. The kubelet submits a CSR; the csrapproving controller auto-approves node client certs; the kubelet downloads its signed cert and writes kubelet.conf. The --discovery-token-ca-cert-hash pins the CA's public key hash so the node can't be tricked into joining a rogue cluster.

      Cluster upgrades

      • Under the hood

        upgrade apply is mostly a manifest rewrite: it pulls new images, then replaces each static pod manifest in /etc/kubernetes/manifests one at a time, waiting for the new pod to go healthy before the next (with automatic rollback on failure). The kubelet binary itself is never touched by kubeadm — that's your job via the package manager, which is why the exam task has those separate apt steps.

      etcd backup & restore — drill until reflexive

      • Under the hood

        A snapshot is a point-in-time copy of the bbolt keyspace. Restore never touches the running etcd — it materializes a brand-new data directory (with a fresh member identity) on disk; the cluster only changes when you repoint the etcd pod's volume and the kubelet restarts it. Afterwards, controllers reconcile the world back to the snapshot's state — pods created after the snapshot get deleted because their records no longer exist.

      Weekend checkpointYou can now rebuild, upgrade, back up, and restore a cluster without notes. If any of these still needs the docs open, repeat it before moving on — weeks 6–8 assume this fluency.

      Week 3 — Workloads, scheduling & app config

      Workloads & Scheduling · 15%

      Pod lifecycle mechanics, how Deployments really roll, how the scheduler decides, and how resources map to cgroups. Plus Helm and Kustomize, which the revised exam tests directly.

      Pod lifecycle internals

      • Under the hood

        Before 1.28, sidecars were plain extra containers with painful ordering (app could start before its proxy, Jobs never terminated because the sidecar kept running). The native mechanism reuses init ordering for startup but keeps the container running, and terminates it after app containers on shutdown — solving both problems with one field.

      • Under the hood

        Probes are executed by the kubelet on the node — httpGet comes from the node's kubelet, exec runs inside the container via CRI. A failing readiness probe doesn't restart anything; it flips the pod's Ready condition, and the EndpointSlice controller drops its IP from Services — the pod quietly stops receiving traffic. Confusing liveness with readiness is the most common self-inflicted outage: an overloaded pod that fails liveness gets restarted, making the overload worse.

      Deployments & rollouts

      • Under the hood

        The deployment controller hashes the pod template and injects pod-template-hash=<hash> into both the RS selector and pod labels. Change anything in the template → new hash → new ReplicaSet. A rollback isn't a revert of history: it just copies an old RS's template back into the Deployment, and that old RS scales up again (revision history is the set of retained old ReplicaSets, capped by revisionHistoryLimit).

      Scheduling controls

      • Under the hood

        Taints are a node-side repellent evaluated in the scheduler's Filter phase; tolerations only neutralize them — they never attract. NoExecute is special: it's enforced after scheduling too, by the taint-manager evicting non-tolerating pods. The node lifecycle controller uses exactly this to drain unreachable nodes (unreachable:NoExecute + the default 300s toleration every pod silently gets).

      • Under the hood

        When no node fits a high-priority pod, the PostFilter (preemption) plugin simulates removing lower-priority pods until one node would fit, evicts that minimal victim set (respecting graceful termination), and records the node in status.nominatedNodeName. The pod isn't guaranteed that node — it re-enters the queue and could land elsewhere if the world changed.

      Resources, QoS & autoscaling

      • Under the hood

        Requests translate to cpu.weight (proportional share under contention); CPU limits become a CFS quota (cpu.max: e.g. 50ms runtime per 100ms period for 500m) — exceeding it means throttling, never killing. Memory limits set memory.max; exceeding that triggers the kernel OOM killer → container exit code 137. CPU over limit = slow; memory over limit = dead. That asymmetry explains half of all container mysteries.

      App configuration

      • Under the hood

        The kubelet writes volume contents into a timestamped directory and flips a ..data symlink atomically, so consumers never see half-written files. Env vars are baked in at container creation — only a restart re-reads them. subPath bind-mounts a file directly, bypassing the symlink dance, hence frozen forever. Exam questions love "why didn't my app pick up the new config?"

      Helm & Kustomize

      • Under the hood

        Helm has no cluster-side component (Tiller died with v2). The client renders Go templates locally and applies the result; release state is stored as Secrets named sh.helm.release.v1.<name>.v<N> in the release namespace — each revision one Secret, which is what rollback re-applies. helm template renders without installing: your debugging tool for "what would this actually create?"

      • Under the hood

        Kustomize is a purely declarative transformer pipeline — no templates, no variables. configMapGenerator appends a content hash to the ConfigMap name and rewrites every reference to it, so changing config produces a new name and automatically triggers a Deployment rollout — the idiomatic fix for "env-var configs don't update".

      Week 4 — Services & networking, from veth up

      Services & Networking · 20%

      Build the network mental model bottom-up: Linux primitives → CNI → Services/kube-proxy → DNS → Ingress/Gateway API → NetworkPolicy. This is the week that makes network troubleshooting deterministic instead of guesswork.

      Linux networking foundations

      • Under the hood

        This 10-minute exercise is what the bridge CNI plugin does per pod: a network namespace (owned by the pause container), a veth pair as a virtual cable — one end becomes eth0 inside the pod, the other plugs into a node bridge — plus routes. Once you've built it manually, ip addr, ip route, and ip netns exec become debugging superpowers rather than incantations.

      CNI

      • Under the hood

        CNI is beautifully dumb: not a daemon protocol but exec-ing a binary per pod. When the kubelet (via containerd) creates a sandbox, it runs the configured plugin chain: main plugin (bridge/calico/cilium) wires the interface, an IPAM plugin (host-local, etc.) hands out the IP, meta plugins (portmap, bandwidth) add extras. A node with no config file in /etc/cni/net.d sits NotReady with "cni plugin not initialized" — a staple exam fault.

      • Under the hood

        Overlay: pod packets are wrapped in UDP (VXLAN port 8472) between nodes — works on any network, costs encapsulation overhead and gives "why doesn't tcpdump show my pod IP?" moments. Routed: each node advertises its pod CIDR via BGP; pod IPs are first-class on the network, no encap. eBPF (Cilium) replaces much of iptables/kube-proxy with programs attached in the kernel.

      Services & kube-proxy

      • Under the hood

        kube-proxy (iptables mode) renders Services+EndpointSlices into nat-table rules: KUBE-SERVICES matches clusterIP:port → KUBE-SVC-XXX picks a backend using the statistic module (random, weighted 1/n, 1/(n-1)…) → KUBE-SEP-XXX DNATs to podIP:port. conntrack remembers the choice so a TCP flow sticks with one backend. IPVS mode swaps the rule chains for kernel hash tables (O(1) at scale); nftables is the newest backend. kube-proxy is a rule compiler, not a proxy in the datapath.

      DNS & CoreDNS

      • Under the hood

        ndots:5 means any name with fewer than 5 dots is tried against the search list first: db → db.myns.svc.cluster.local → db.svc.cluster.local → … That's why bare service names resolve, and also why external lookups from pods generate several NXDOMAIN queries before succeeding. The kubelet writes this file per-pod based on dnsPolicy (ClusterFirst by default; ClusterFirstWithHostNet is the gotcha for hostNetwork pods).

      Ingress & Gateway API

      • Under the hood

        Ingress is only data — a rules object. The ingress controller (nginx, traefik…) watches Ingresses and renders them into its own proxy config, routing directly to pod IPs from EndpointSlices (bypassing the ClusterIP). No controller installed → Ingress objects sit there doing nothing, silently. That's both an exam fault and a real-world rite of passage.

      NetworkPolicy

      • Under the hood

        Enforcement is done by the CNI plugin, not kube-proxy or the kernel by default — Flannel famously ignores NetworkPolicies entirely. The subtle trap: an empty ingress: [] plus policyTypes: [Ingress] denies everything, while omitting policyTypes when you only wrote egress rules leaves ingress wide open. Also: podSelector: {} selects all pods in the namespace.

      Week 5 — Storage: PVs, PVCs & CSI

      Storage · 10%

      The smallest domain — one focused week. Learn the binding control loop and the CSI call sequence, and the exam's storage tasks become mechanical. Use the spare days to start week 6 early.

      Volume fundamentals

      Binding internals

      • Under the hood

        The PV controller runs the same reconcile pattern as everything else: watch unbound PVCs, find the smallest satisfying PV, stamp claimRef on the PV and volumeName on the PVC. The bind is exclusive even if the PV is far bigger than the request — a 100Gi PV bound to a 1Gi claim strands 99Gi. A PVC that finds no match just sits Pending, and so does any pod using it (with a clear event — always describe the PVC, not just the pod).

      StorageClasses & dynamic provisioning

      CSI internals

      • Under the hood

        Kubernetes core knows nothing about any storage vendor. The sidecars translate API objects into gRPC calls on the driver: external-provisioner watches PVCs → CreateVolume; external-attacher watches VolumeAttachment objects → ControllerPublishVolume (attach disk to node). Then the kubelet calls the node plugin directly: NodeStageVolume (format + mount once per node into a global dir) → NodePublishVolume (bind-mount into the pod's volumes dir). Provision → Attach → Stage → Publish: knowing that sequence tells you which log to read when a mount hangs.

      Week 6 — Troubleshooting: the 30% week

      Troubleshooting · 30%

      The highest-weighted domain, and where weeks 1–5 pay off. The method: for every failure class, break your own cluster, fix it, and write down the signature (symptom → first command → root cause). Repetition builds the pattern library that makes exam tasks 3-minute tasks.

      Toolbox & method

      • Under the hood

        kubectl logs is served by the kubelet reading the container's log file under /var/log/pods/… (the runtime writes each container's stdout there as JSON lines). So: apiserver down → use crictl logs on the node; kubelet down → read the files directly. Nothing about logs requires the control plane — that's your lifeline in control-plane-failure questions.

      Control plane failures

      • Under the hood

        Learn to read the failure mode: wrong image → pod stuck ImagePullBackOff in crictl; bad flag → container starts and exits instantly, log says unknown flag; wrong etcd endpoint → apiserver runs but logs connection-refused storms and kubectl gives 500s; bad cert path → instant exit with "no such file". Same file, four different fingerprints.

      Workload failures

      • Under the hood

        CrashLoopBackOff isn't an error — it's the kubelet saying "it keeps dying, so I'm waiting before the next restart" (backoff doubles 10s→20s→…→5m). The evidence is always in the previous container's logs or its lastState.terminated (reason: OOMKilled, exit code). Distinguish OOMKilled (kernel, over memory limit) from a liveness-probe kill (kubelet, events show "Liveness probe failed… Killing") — both restart the pod, different fixes.

      Network & DNS debugging

      • Under the hood

        The checklist works because it follows the machinery from 4.10–4.11: empty EndpointSlices mean kube-proxy programmed no DNAT targets (selector/readiness problem — the connection is refused or hangs); populated slices but bad targetPort means DNAT lands on a port nobody listens on. Each checklist step isolates one component of the chain: controller → proxy rules → pod.

      Authentication & authorization internals

      RBAC in practice

      • Under the hood

        The RBAC authorizer answers one question: does any binding connect this subject (user/group/SA) to a rule allowing this verb on this resource (apiGroup + resource + optional resourceNames) in this scope? Rules are purely additive — no deny exists, so "remove access" always means "delete/narrow a binding". Namespaced Role rules can never grant cluster-scoped things (like nodes) — a favorite trick question.
