etcd backup & restore — one-page card

Print this (it fits one page) and use it to check your drill runs, not to type from. The lessons behind it are 2.16–2.18.

0 · Read the flags off the manifest

sudo grep -E 'listen-client|trusted-ca|cert-file|key-file' \
  /etc/kubernetes/manifests/etcd.yaml | grep -v peer
Manifest flagYour flagkubeadm default
--trusted-ca-file--cacert/etc/kubernetes/pki/etcd/ca.crt
--cert-file--cert/etc/kubernetes/pki/etcd/server.crt
--key-file--key/etc/kubernetes/pki/etcd/server.key
--listen-client-urls--endpointshttps://127.0.0.1:2379

1 · Backup (etcdctl — online, needs certs)

sudo ETCDCTL_API=3 etcdctl snapshot save /backup/snap.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

Exactly one endpoint, never a list. "context deadline exceeded" = wrong/missing cert flags, not the network.

2 · Verify the snapshot (etcdutl — offline)

sudo etcdutl snapshot status /backup/snap.db -w table

Healthy small cluster ≈ 1000+ keys, a few MB. Tiny file or 0 keys → you saved the wrong thing. (etcdctl snapshot status is deprecated — offline jobs use etcdutl.)

3 · Restore (etcdutl — offline, no certs)

sudo etcdutl snapshot restore /backup/snap.db \
  --data-dir /var/lib/etcd-restore

Then edit /etc/kubernetes/manifests/etcd.yaml — change only the hostPath:

- hostPath:
    path: /var/lib/etcd-restore   # ← was /var/lib/etcd
    type: DirectoryOrCreate
  name: etcd-data

Leave --data-dir and the volumeMount alone. Kubelet restarts etcd in ~60–90s; kubectl refuses connections meanwhile — normal.

4 · Health checks

sudo ETCDCTL_API=3 etcdctl endpoint health \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
# same flags: endpoint status -w table · member list
sudo crictl ps -a | grep etcd      # is the container up/restarting?
kubectl -n kube-system logs etcd-cp | tail
journalctl -u kubelet | tail       # manifest syntax errors land here

⚠ Failure signatures

  • Edited --data-dir instead of hostPath → etcd boots on an empty dir: cluster amnesia.
  • Restore into existing /var/lib/etcd → etcdutl refuses. Use a new dir; never rm live data.
  • kubectl dead >2 min after restore → bounce the apiserver: move its manifest out of /etc/kubernetes/manifests, wait 30s, move back.
  • Objects created after the snapshot are gone → correct behavior, not a failure. Deleted-before = restored.

Mental model (why restore works this way)

etcdctl = online (talks to live etcd; snapshot save) · etcdutl = offline (operates on files; status + restore). Restore never touches the running etcd — it materializes a fresh data directory with a new member identity; the cluster switches over only when the manifest's hostPath points there and the kubelet restarts the pod. Controllers then reconcile the world back to the snapshot's state. Exam docs page: "Operating etcd clusters for Kubernetes".
